## Why This Matters

### Business Importance
AI features are often slow, expensive, and multi-step. If you do not understand when work should happen synchronously versus asynchronously, you will build APIs that feel broken, overload services, or make users wait for tasks that should run in the background.

### System Importance
This lesson teaches where to put long-running work:
- request path
- background task
- worker queue
- scheduled job

That decision affects latency, reliability, retries, observability, and cost.

### Interview Importance
This topic shows up indirectly in:
- FastAPI and backend screens
- ML system design
- AI production debugging
- LLM/RAG service design

Weak candidates talk only about "making it async." Strong candidates explain which work belongs on the request path and which work must leave it.

## Key Takeaways -- Read This Before Any Project

1. **Async does not mean background.** `async` is about efficient waiting on I/O. Background/queue decisions are about where work runs and when the user gets a response.
2. **Do not keep users waiting for jobs that should be queued.** OCR, embedding thousands of chunks, or full evaluation runs should not happen on the request path.
3. **Background tasks are not a substitute for real worker systems.** They are fine for small follow-up work, not durable long-running pipelines.
4. **Retries require idempotency.** If the same ingestion job runs twice, it must not silently duplicate chunks or corrupt state.
5. **Most "async" production incidents are really boundary mistakes.** The wrong work was left on the wrong path.
6. **Timeouts and failure classification matter more than raw concurrency jargon.**

## PART 1 -- CORE CONCEPT

### What It Is
Async programming lets a service handle I/O efficiently by not blocking while it waits for external systems such as:
- LLM providers
- databases
- caches
- object storage
- internal APIs

Background execution moves work off the request path so the user does not wait for long tasks to finish. Worker systems make that background work durable, retryable, and operationally visible.

### Why It Exists
AI systems mix:
- short, user-facing requests
- slow provider calls
- long-running ingestion and evaluation jobs
- retry-prone external operations

If you treat all of that as one execution model, the system becomes slow, fragile, and hard to operate.

### Mental Model

```text
Async I/O:
"while waiting on something external, do not block the whole server"

Background execution:
"return to the user now, do the rest afterward"

Worker queue:
"run long or retryable jobs outside the API process with status tracking"
```

### When To Use Which

| Pattern | Use When | Avoid When |
|---|---|---|
| Sync route | logic is CPU-light and dependency calls are small or legacy | provider call may hang or multiple external waits are involved |
| Async route | request path depends on I/O and user should wait for answer | work is long-running and should leave request path entirely |
| FastAPI background task | short post-response follow-up work | job must survive process restarts or needs retries/visibility |
| Queue / worker | long, expensive, retryable, or operationally important work | task is tiny and can safely finish immediately |
| Scheduled job | work runs on cadence, not from a user click | result is required immediately by the caller |

## PART 2 -- PRODUCTION REALITY

### Real Use Cases
- synchronous: classify a short ticket, fetch top documents, generate a short answer
- background task: write an audit event, send an analytics event, cache a result after response
- queued worker: ingest a PDF, OCR a scanned document, chunk and embed, run batch evals
- scheduled job: nightly regression suite, cost report generation, stale-document reindex checks

### Good Patterns

**Pattern 1 -- quick request, durable job**
- client uploads a document
- API validates and stores metadata
- API enqueues ingestion job
- API returns `202 Accepted` with `job_id`
- client polls `/jobs/{job_id}` or receives webhook later

**Pattern 2 -- async route for provider-bound latency**
- request is still user-facing
- route awaits provider/database/cache I/O
- timeout and fallback logic are explicit

**Pattern 3 -- idempotent ingestion**
- job can safely run twice
- duplicate chunks or duplicate embeddings do not get inserted blindly
- status transitions are explicit: `queued -> running -> completed/failed`

### Day-to-Day Team Reality
- product wants instant upload confirmation
- platform wants reliable retries and visibility into failures
- support wants to know whether a document is "uploaded", "processing", or "ready"
- engineers need logs, job IDs, and status tables to debug failures without reading application memory

### Common Mistakes
- using async routes but leaving blocking SDK calls inside them
- pushing long-running ingestion into FastAPI `BackgroundTasks` and assuming it is durable
- returning success before any job ID or status tracking exists
- retrying non-idempotent jobs and creating duplicate rows
- using the API process itself as the worker system

### Failure Modes
- LLM provider latency spikes cause endpoint saturation
- background task is lost because the process restarts
- queued job is retried and writes duplicate chunks
- user polls forever because no terminal failed state is recorded
- OCR or embedding jobs consume all worker capacity and starve other work

### Monitoring and Debugging
- track endpoint latency separately from queue latency
- track job state counts: queued, running, completed, failed
- track retry counts and dead-letter counts
- include `job_id`, `tenant_id`, `document_id`, and `run_id` in logs where relevant
- alert on stuck jobs with no status transition for too long

### Security and Reliability Concerns
- validate before enqueueing so bad jobs do not flood workers
- use access control on job-status endpoints
- rate limit expensive enqueue endpoints
- make job payloads minimal; fetch sensitive data server-side when workers run if possible
- do not trust in-memory background tasks for critical operations

## PART 3 -- ENGINEERING

### Async route with timeout

```python
import asyncio

from fastapi import APIRouter, HTTPException
from pydantic import BaseModel, Field


class AskRequest(BaseModel):
    question: str = Field(min_length=1, max_length=4000)


class AskResponse(BaseModel):
    answer: str
    model_name: str


router = APIRouter(prefix="/v1/ask", tags=["ask"])


async def call_provider(question: str) -> AskResponse:
    # Placeholder for async provider I/O
    await asyncio.sleep(0.1)
    return AskResponse(answer=f"Echo: {question}", model_name="demo-model")


@router.post("", response_model=AskResponse)
async def ask(request: AskRequest) -> AskResponse:
    try:
        return await asyncio.wait_for(call_provider(request.question), timeout=10.0)
    except asyncio.TimeoutError as exc:
        raise HTTPException(status_code=504, detail="provider timeout") from exc
```

### FastAPI background task for small follow-up work

```python
from fastapi import APIRouter, BackgroundTasks
from pydantic import BaseModel


class AuditEvent(BaseModel):
    event_type: str
    entity_id: str


def write_audit_log(event: AuditEvent) -> None:
    # Small post-response work only
    print(f"audit: {event.model_dump()}")


router = APIRouter(prefix="/v1/events", tags=["events"])


@router.post("")
async def create_event(background_tasks: BackgroundTasks) -> dict[str, str]:
    event = AuditEvent(event_type="document_uploaded", entity_id="doc-123")
    background_tasks.add_task(write_audit_log, event)
    return {"status": "accepted"}
```

### Queue-first API pattern

```python
from uuid import uuid4

from fastapi import APIRouter, HTTPException
from pydantic import BaseModel, Field


class IngestRequest(BaseModel):
    document_id: str
    source_path: str = Field(min_length=1)
    tenant_id: str


class JobAccepted(BaseModel):
    job_id: str
    status: str


router = APIRouter(prefix="/v1/ingest", tags=["ingestion"])


def enqueue_ingestion_job(payload: dict) -> str:
    # Placeholder for a real queue push
    return str(uuid4())


@router.post("", response_model=JobAccepted, status_code=202)
async def enqueue_ingestion(request: IngestRequest) -> JobAccepted:
    job_id = enqueue_ingestion_job(request.model_dump())
    return JobAccepted(job_id=job_id, status="queued")
```

### Job-status schema

from typing import Literal, Optional

from pydantic import BaseModel


class JobStatusResponse(BaseModel):

## PART 5 -- SENIORITY LAYER

### Seniority Story 1 -- Failure
"We originally processed uploaded documents inline in the API. Large scanned PDFs caused request timeouts, and users retried uploads, which duplicated work and doubled costs. The real issue was not OCR speed. It was that the entire ingestion path was left on the request boundary."

### Seniority Story 2 -- Design Decision
"We used FastAPI background tasks only for lightweight audit/log follow-up. Durable ingestion, OCR, and embedding went to a worker queue because those jobs needed retries, status tracking, and survival across process restarts."

### Tradeoff Language
"The tradeoff is not simply fast versus slow. It is user responsiveness versus operational control. Keeping work in the request path is simpler, but it couples user latency to infrastructure variance. Moving work to a queue adds complexity, but gives you retries, observability, and failure isolation."

### Failure Vocabulary
"The incident was caused by execution-boundary misuse. Long-running, retryable work was treated as request-path work, so transient dependency slowness became a user-facing outage."

### Interview Question and Strong Answer
**Q: "How would you handle document ingestion in an AI system?"**

Strong answer: "I would validate the upload request synchronously, persist metadata, enqueue an ingestion job, and return `202 Accepted` with a `job_id`. OCR, chunking, embedding, and indexing would run in workers because they are long-running and need retries. I would expose job status and make the worker idempotent so repeated retries do not duplicate results."
