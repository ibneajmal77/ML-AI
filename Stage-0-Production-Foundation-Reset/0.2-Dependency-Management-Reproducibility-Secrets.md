## CONCEPT 1: A Dependency Spec and a Lock File Are Not the Same Thing
<!-- cognitive job: teach the difference between intent and tested reality -->

| Dimension | Answer |
|---|---|
| **What it is** | A dependency spec declares what package versions you are willing to use; a lock file records the exact package set you actually tested. |
| **Why it matters** | Without both, the same project can resolve to different dependency trees on different machines or at different times. |
| **When to use it** | Use both from the start of any real project that will be shared, deployed, rebuilt, or debugged later. |
| **When NOT to use it** | Do not rely on a dependency spec alone if reproducibility matters beyond one local experiment. |
| **What juniors miss** | They often think the declared package list is enough and do not realize transitive dependency drift can still change behavior. |
| **Senior signal** | "The spec captures intent; the lock file captures the exact tested reality." |

### Mental Model

A dependency spec is like a shopping list; a lock file is the receipt for what you actually bought. The list says what kinds of things you want, but the receipt is what lets you reconstruct the exact purchase later.

```toml
[project]
dependencies = [
    "fastapi>=0.111.0",
    "openai>=1.30.0",
]
```

```text
uv.lock
  fastapi==0.111.1
  openai==1.37.0
  pydantic==2.8.2
  ...
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `dependencies = [...]` | Declares allowed dependency ranges. | This expresses intent, not the final fully resolved environment. |
| `fastapi>=...`, `openai>=...` | Leaves room for newer versions. | That flexibility is useful, but it also allows behavior to drift over time. |
| `uv.lock` listing exact versions | Pins the full resolved tree. | This is what lets CI, local, and prod run the same tested set. |
| transitive packages like `pydantic` | Shows packages you did not list directly. | A lot of real drift comes from these indirect dependencies, not just the top-level ones. |

**Practice -- apply the concept:**

> A teammate says, "We already have `pyproject.toml`, so why bother committing a lock file?" Answer in one practical sentence.

> **Real-world scenario:** A workflow using an orchestration SDK behaved differently in CI than on a laptop even though the project file looked unchanged. The mismatch came from resolved dependency versions, not from app logic. Once the lock file became part of the contract, those "ghost bugs" dropped sharply.

## CONCEPT 2: Reproducibility Comes From Isolated Environments Plus Controlled Updates
<!-- cognitive job: connect virtual environments, lock files, and update cadence into one reproducibility discipline -->

| Dimension | Answer |
|---|---|
| **What it is** | Reproducibility means the same project can be rebuilt into the same effective runtime by using isolated environments and controlled dependency updates. |
| **Why it matters** | If local, CI, and production do not share the same dependency reality, debugging becomes guesswork and rollbacks become harder. |
| **When to use it** | Use it for every team project, API service, or AI workflow that must survive rebuilds, handoffs, or incident response. |
| **When NOT to use it** | Do not install project dependencies into global Python and assume "works on my machine" is enough. |
| **What juniors miss** | They often think a virtual environment is just convenience, not part of runtime safety and incident prevention. |
| **Senior signal** | "Dependency updates in AI systems are behavior changes, so I roll them through explicit update PRs and verification." |

### Mental Model

Reproducibility is like freezing the stage before a performance. If the actors, props, lighting, and script version all keep changing between rehearsals, you cannot tell whether a bad scene came from the play or from the setup.

```bash
uv sync
uv tree
uv run pytest
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `uv sync` | Installs the project environment from the declared and locked state. | This keeps the environment aligned with the tested project definition. |
| `uv tree` | Shows the resolved dependency tree. | When environments drift, the tree helps you see where the difference lives. |
| `uv run pytest` | Runs tests inside the project environment. | Verification must happen in the same runtime you intend to trust. |

**Practice -- apply the concept:**

> A provider SDK update changes response parsing behavior. What process mistake made this feel like an application bug instead of an environment change?

> **Real-world scenario:** A team moved from ad hoc upgrades to explicit dependency-update PRs with smoke tests and eval checks. They did not stop updating; they stopped pretending updates were harmless housekeeping.

## CONCEPT 3: Secrets Hygiene Is Part of Runtime Design, Not Just Security Etiquette
<!-- cognitive job: teach that secret handling is a runtime and incident-management concern -->

| Dimension | Answer |
|---|---|
| **What it is** | Secrets hygiene means credentials stay out of committed code, logs, screenshots, and example files, and flow into the app through explicit runtime configuration. |
| **Why it matters** | A leaked secret is not fixed just because the latest commit no longer shows it; the credential may still exist in history or logs and usually must be rotated. |
| **When to use it** | Use it whenever the service depends on API keys, database URLs, tokens, or provider credentials. |
| **When NOT to use it** | Do not put real secrets in `.env.example`, docs, code samples, tests, or committed config files. |
| **What juniors miss** | They often think deleting a secret from a later commit is enough, or that local convenience justifies unsafe handling. |
| **Senior signal** | "If a secret touched version control or logs, I assume compromise and rotate it." |

### Mental Model

Secrets are like master keys to a building. The problem is not whether the key is still lying on the table right now; the problem is whether anyone copied it when it was exposed.

| Code line / block | What it does | Why it matters |
|---|---|---|
| `.env` vs `.env.example` | Splits real secret values from safe documentation. | Teams need both: one for runtime, one for clarity. |
| `(ignored)` on `.env` | Keeps secrets out of version control. | Local convenience should not become permanent exposure. |
| `detect-secrets` hook | Adds secret scanning to development workflow. | Prevention is cheaper than incident cleanup after a leak. |
| committed safe template | Documents required variables without exposing them. | New teammates and CI setup get clarity without credential leakage. |

> **Real-world scenario:** A service team caught a credential in a pre-commit scan before merge. That prevention step saved them from having to rotate a provider key, clean git history, and audit who might have copied the exposed token.

## DECISION RULES

### Reproducible AI Runtime Framework

1. Create an isolated project environment instead of using global Python.
2. Define dependencies in `pyproject.toml` or equivalent project metadata.
3. Commit the lock file so the tested dependency reality is rebuildable.
4. Install from the locked state in local development, CI, and container builds.
5. Treat dependency upgrades as explicit change events, not as silent drift.
6. Keep `.env.example` committed and `.env` ignored.
7. Add secret scanning so accidental exposure is caught before merge.
8. If a secret is exposed, rotate it instead of assuming deletion fixed the incident.

### Worked Example

| Step | AI API example |
|---|---|
| Environment | isolated project runtime created |
| Project intent | dependencies declared in `pyproject.toml` |
| Tested reality | lock file committed |
| Verification | tests run inside the same environment |
| Config docs | `.env.example` committed |
| Secret safety | `.env` ignored and scanning enabled |

### Red Flags

| What someone says / does | Why it should raise a flag |
|---|---|
| "I just pip install what I need globally." | That destroys isolation and makes behavior harder to reproduce. |
| "We don't need the lock file; the spec is enough." | The tested dependency tree can still drift. |
| "It only changed because of a minor SDK bump." | Minor AI-library upgrades can still alter behavior materially. |
| "The secret was deleted, so we’re fine now." | If it was exposed, rotation is usually required. |

### Use / Do Not Use

| Use this discipline when | Do NOT use this discipline when |
|---|---|
| you want local, CI, and production to share one tested runtime contract | you are assuming environment setup is just a convenience detail outside the real system |

## COMMON MISTAKES

### Mistake 1: Relying on the dependency spec alone

**Mistaken instinct:** "The declared package list already tells us what the project uses."

**Why it feels right:** The project file looks like the obvious source of truth.

**Why it fails:** The resolved dependency tree can still vary over time and across machines, especially through transitive dependencies.

**Better move:** Commit the lock file and treat it as part of the tested runtime contract.

### Mistake 2: Treating global Python as good enough

**Mistaken instinct:** "I already have the packages installed, so creating an environment is unnecessary."

**Why it feels right:** It feels faster and more convenient in the moment.

**Why it fails:** Global installs blur project boundaries and make cross-project drift much harder to reason about.

**Better move:** Use an isolated project environment from the start.

### Mistake 3: Letting AI-library updates drift in silently

**Mistaken instinct:** "We should stay current, so automatic or ad hoc upgrades are fine."

**Why it feels right:** Staying current sounds responsible and modern.

**Why it fails:** AI SDKs and orchestration libraries can change runtime behavior in ways that look like application bugs if the update was not controlled.

**Better move:** Upgrade through explicit PRs with tests, smoke checks, or eval gates.

### Mistake 4: Treating secret deletion as incident resolution

**Mistaken instinct:** "The key is no longer visible, so the problem is gone."

**Why it feels right:** The obvious exposed text has disappeared.

**Why it fails:** The secret may still live in history, logs, screenshots, or clones and should be treated as compromised.

**Better move:** Rotate exposed secrets and improve prevention.
