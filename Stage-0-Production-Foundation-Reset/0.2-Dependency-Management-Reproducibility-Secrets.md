## Why This Matters

### Business Importance
If environments drift, teams waste time debugging the wrong problem. A provider call, eval pipeline, or service route may fail not because the design is bad, but because dependency versions changed silently or a secret was mishandled.

### System Importance
AI systems are unusually sensitive to:
- SDK version changes
- orchestration library behavior changes
- provider API evolution
- environment mismatch between local, CI, and production

Reproducibility is not housekeeping. It is what makes debugging, rollback, and reliable deployment possible.

### Interview Importance
This topic is rarely asked in isolation, but it shows up as a strong maturity signal in:
- project review
- MLOps and production AI interviews
- “how do you keep AI systems stable?” questions
- incident and debugging discussions

Weak candidates say “I just install what I need.” Strong candidates explain version control, lock files, secret handling, and dependency update strategy.

## Key Takeaways -- Read This Before Any Project

1. **A dependency spec is not the same as a lock file.** One says what you want, the other records what you actually tested.
2. **AI libraries change behavior quickly.** Dependency updates are operational changes, not just package maintenance.
3. **Secrets are never “safe” because they were deleted from the latest commit.** If committed once, rotate them.
4. **A reproducible environment is part of incident prevention.**
5. **You want controlled updates, not either permanent freezing or uncontrolled drift.**

## PART 1 -- CORE CONCEPT

### What It Is
Dependency management is the practice of defining and controlling package versions so code behaves consistently across local development, CI, and production. Reproducibility means the same project can be rebuilt later with the same effective runtime. Secrets hygiene means sensitive credentials never live in committed code or logs.

### Why It Exists
AI projects are vulnerable to subtle dependency drift:
- provider SDKs change
- orchestration libraries change parsing behavior
- type handling changes across package versions
- local and CI environments stop matching

These changes often look like application bugs when the real issue is environment drift.

### Mental Model

```text
[Virtual Environment]
        |
[Dependency Spec]
        |
[Lock File]
        |
[Repeatable Runtime]
```

The virtual environment isolates.  
The spec defines intent.  
The lock file defines the exact tested state.

### When To Use It
- in every project from day one
- before adding real dependencies or secrets
- before sharing code with CI, teammates, or deployment systems

### When Not To Overdo It
- do not build an elaborate internal packaging platform before one clean project is working
- do not optimize for every possible environment if you cannot reproduce one environment well yet

## PART 2 -- PRODUCTION REALITY

### Real Use Cases
- CI must install the same packages every run
- a rebuilt container six months later must not silently change provider behavior
- a team needs to upgrade a library without breaking structured outputs or tool parsing
- secrets must be available safely in local development and production without being committed

### Good Patterns

**Pattern 1 -- dependency spec plus committed lock file**
- spec defines dependencies
- lock file captures exact tested versions
- same lock file is used in CI and container builds

**Pattern 2 -- controlled update cadence**
- dependency updates happen in explicit PRs
- tests and eval checks run before merge
- production changes are intentional, not accidental

**Pattern 3 -- committed `.env.example`, ignored `.env`**
- project documents required variables
- actual values stay out of version control

### Day-to-Day Team Reality
- one engineer upgrades an SDK
- CI or eval checks catch changed behavior
- reviewers decide whether the update is worth the risk
- security or platform teams care whether secrets were handled properly

### Common Mistakes
- no lock file at all
- very loose version ranges on unstable AI libraries
- installing into global Python
- copying real secrets into committed files or docs
- assuming “works locally” is a real deployment guarantee

### Failure Modes
- same code behaves differently in CI and local because the dependency tree changed
- provider SDK upgrade changes response shape handling
- secrets leak through git history, logs, or shared screenshots
- container build succeeds but runtime cannot find installed CLI tools because the environment path is wrong

### Monitoring and Debugging
- log service version and environment/build metadata at startup
- compare dependency trees when CI and local diverge
- treat dependency update PRs as behavior changes requiring tests or evals
- scan for secrets continuously, not only after an incident

### Security and Reliability Concerns
- secrets must be rotated if ever committed
- do not place secrets in `pyproject.toml`, tests, docs, or example commands
- keep runtime environments explicit in containers
- separate “bad config” failures from “application logic” failures operationally

## PART 3 -- ENGINEERING

### `pyproject.toml`

```toml
[project]
name = "ai-service"
version = "0.1.0"
requires-python = ">=3.11"
dependencies = [
    "fastapi>=0.111.0",
    "uvicorn[standard]>=0.29.0",
    "pydantic>=2.7.0",
    "pydantic-settings>=2.3.0",
    "openai>=1.30.0",
    "httpx>=0.27.0",
]

[project.optional-dependencies]
dev = [
    "pytest>=8.0.0",
    "pytest-asyncio>=0.23.0",
    "ruff>=0.4.0",
]
```

### Settings from environment

```python
# src/app/config.py
from pydantic_settings import BaseSettings, SettingsConfigDict


class Settings(BaseSettings):
    openai_api_key: str
    database_url: str
    redis_url: str = "redis://localhost:6379/0"

    model_config = SettingsConfigDict(
        env_file=".env",
        env_file_encoding="utf-8",
    )
```

### Pre-commit secret scanning

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.4.0
    hooks:
      - id: detect-secrets
        args: ["--baseline", ".secrets.baseline"]
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.4.0
    hooks:
      - id: ruff
      - id: ruff-format
```

### Dockerfile with explicit environment path

```dockerfile
FROM python:3.11-slim

WORKDIR /app

COPY --from=ghcr.io/astral-sh/uv:latest /uv /uvx /bin/

COPY pyproject.toml uv.lock ./
RUN uv sync --frozen --no-dev

COPY src/ ./src/

ENV PATH="/app/.venv/bin:$PATH"
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### Basic runtime checks

```bash
uv tree
uv run pytest
```

### Wrong Way Example

```text
Bad pattern:
- install packages globally
- no lock file
- commit `.env`
- update dependencies ad hoc

This creates an environment you cannot reproduce and incidents you cannot explain cleanly.
```

## PART 5 -- SENIORITY LAYER

### Seniority Story 1 -- Failure
"A minor orchestration-library upgrade changed tool-output parsing and our workflow started misrouting tasks. The bug looked like application logic at first, but the root cause was dependency drift. After that, every dependency update ran through smoke tests and eval gates."

### Seniority Story 2 -- Design Decision
"We chose scheduled dependency update PRs instead of constant ad hoc upgrades. The goal was to balance security and stability without letting behavior drift go unnoticed."

### Tradeoff Language
"The tradeoff is not stability versus change in the abstract. It is predictable behavior versus update speed. For AI systems, fast-moving dependencies make uncontrolled updates far more dangerous than they look."

### Failure Vocabulary
"The incident was caused by environment drift, not application logic. Local and deployed behavior diverged because the tested dependency set was not the deployed dependency set."

### Interview Question and Strong Answer
**Q: "How do you keep AI projects reproducible across environments?"**

Strong answer: "I use an isolated environment, a dependency spec plus a committed lock file, and config loaded from environment variables rather than code. For updates, I prefer explicit update PRs with tests or eval checks because AI libraries can change runtime behavior in ways that look like application bugs if you are not careful."
