## Key Takeaways -- Read This Before Any Project

1. **Use SQL for relational extraction, Pandas for investigation, and NumPy for dense numeric operations.**
2. **If you are looping over rows in Pandas, you are usually on the wrong path.**
3. **Dtypes matter.** `float32` vs `float64` affects memory and downstream compatibility.
4. **Null checks and shape/count checks should happen before you trust your metrics.**
5. **Averages hide bad cohorts.** Grouped slice analysis is often more useful than one overall score.
6. **Many “model issues” start as data-shape or data-quality issues.**

## PART 1 -- CORE CONCEPT

### What It Is
NumPy provides fast array operations. Pandas provides labeled tabular operations on top of that numeric foundation. For AI engineers, these tools are the working layer for evaluation data, feature computation, ingestion cleanup, and diagnostics.

### Why It Exists
Before an AI system produces value, you still need to:
- load and inspect data
- join structured records
- compute grouped metrics
- detect nulls and schema issues
- manipulate vectors such as embeddings or scores

### Mental Model

```text
SQL:
"pull and filter the right rows from the source"

Pandas:
"inspect, slice, reshape, aggregate, export"

NumPy:
"perform dense numeric work efficiently"
```

### When To Use It
- use SQL for heavy relational extraction close to the database
- use Pandas for local inspection, joining, slicing, and export
- use NumPy for numeric vector operations, similarity, and array math

### When Not To Overdo It
- do not pull huge relational workloads into Pandas unnecessarily
- do not use NumPy for work that is clearer in SQL or Pandas tables
- do not introduce large-scale distributed tools before you understand the local problem

## PART 2 -- PRODUCTION REALITY

### Real Use Cases
- evaluating a RAG run and slicing failures by source type or tenant
- joining user and event data before label generation
- checking nulls and schema drift before training or scoring
- manipulating embedding arrays before storage or comparison

### Good Patterns

**Pattern 1 -- extract in SQL, analyze in Pandas**
- let the database do filtering and joins where possible
- pull only the rows/columns needed for diagnosis or eval analysis

**Pattern 2 -- vectorized Pandas operations**
- use boolean masks, `.loc`, `groupby`, and aggregation
- avoid row-wise loops for common transformations

**Pattern 3 -- explicit memory discipline**
- use explicit dtypes
- use chunked reads for large CSVs
- prefer `float32` for embeddings when appropriate

### Day-to-Day Team Reality
- support says one market is performing worse
- you load eval data, slice by cohort, and inspect failures
- you discover the issue is concentrated in one source type or one missing metadata field
- the fix is often upstream data logic, not the model itself

### Common Mistakes
- using `iterrows()` for standard transformations
- blindly filling nulls with `0`
- forgetting dtype control on large data
- doing huge joins locally when SQL should have handled them
- reporting only the overall average and missing failure clusters

### Failure Modes
- hidden nulls distort metrics
- memory usage spikes because string/object columns were left uncontrolled
- duplicate or dropped rows after a merge go unnoticed
- embedding arrays use the wrong dtype and waste memory or break downstream assumptions

### Monitoring and Debugging
- run `df.info()` and `df.isnull().sum()` early
- log row counts before and after joins or filters
- inspect grouped performance, not just overall mean
- check dtypes before numeric operations

### Security and Reliability Concerns
- avoid exporting sensitive raw rows carelessly when generating failure-review files
- be explicit about tenant and cohort columns when analyzing multi-tenant data
- do not let local exploratory code become production logic without cleanup

## PART 3 -- ENGINEERING

### NumPy patterns AI engineers actually use

```python
import numpy as np


embedding = np.array([0.12, -0.04, 0.88], dtype=np.float32)


def cosine_similarity(a: np.ndarray, b: np.ndarray) -> float:
    return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)))


def batch_cosine_similarity(query: np.ndarray, corpus: np.ndarray) -> np.ndarray:
    query_norm = query / np.linalg.norm(query)
    corpus_norm = corpus / np.linalg.norm(corpus, axis=1, keepdims=True)
    return corpus_norm @ query_norm
```

### Pandas patterns

```python
import pandas as pd


results = pd.read_csv("eval_results.csv")

print(results.info())
print(results.isnull().sum())

results.loc[
    (results["faithfulness"] < 0.5) & (results["relevance"] < 0.6),
    "needs_review",
] = True

failing = results[results["needs_review"] == True].sort_values("faithfulness")
```

### Pulling data from PostgreSQL

```python
import pandas as pd
from sqlalchemy import create_engine


engine = create_engine("postgresql+psycopg://user:pass@localhost:5432/aidb")

query = """
select
    question_id,
    category,
    faithfulness,
    relevance
from rag_eval_results
where run_id = :run_id
"""

df = pd.read_sql(query, engine, params={"run_id": "eval_run_2026_04_03"})
```

### Memory-conscious CSV reading

```python
import pandas as pd


dtype_map = {
    "question_id": "string",
    "faithfulness": "float32",
    "relevance": "float32",
}

for chunk in pd.read_csv("large_eval_results.csv", dtype=dtype_map, chunksize=50_000):
    print(chunk.shape)
```

### Wrong Way Example

```python
# Bad pattern:
for _, row in results.iterrows():
    if row["faithfulness"] < 0.5 and row["relevance"] < 0.6:
        ...

# This is usually slower, harder to reason about, and unnecessary for common Pandas work.
```

## PART 5 -- SENIORITY LAYER

### Seniority Story 1 -- Failure
"A retrieval-quality dashboard looked acceptable overall, but one market was failing badly. Slice analysis in Pandas showed the failures were concentrated in one document type. The global mean had hidden the problem."

### Seniority Story 2 -- Design Decision
"We kept heavy extraction and large joins in SQL because moving tens of millions of rows into Pandas was pointless. Pandas stayed the diagnostic layer, not the warehouse."

### Tradeoff Language
"If the work is relational and large, SQL is usually the right first tool. If the work is exploratory and cohort-driven, Pandas is often faster for human diagnosis. If the work is dense numeric math, NumPy is the right lower layer."

### Failure Vocabulary
"The issue was not model quality first. It was data-shape drift and weak slice analysis."

### Interview Question and Strong Answer
**Q: "You have a Pandas workflow taking 45 minutes on 10 million rows. What do you do?"**

Strong answer: "First I check whether the logic belongs in SQL instead of Pandas. If Pandas is still the right layer, I reduce columns, set explicit dtypes, chunk the read, and replace row-wise operations with vectorized ones. If that still is not enough, I consider a different execution layer such as Polars or a distributed approach."
