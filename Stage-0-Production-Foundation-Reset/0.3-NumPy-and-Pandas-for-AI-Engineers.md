## CONCEPT 1: SQL, Pandas, and NumPy Are Different Layers, Not Competing Replacements
<!-- cognitive job: teach tool choice by job type -->

| Dimension | Answer |
|---|---|
| **What it is** | SQL is usually the extraction layer, Pandas is the inspection and tabular-analysis layer, and NumPy is the dense numeric array layer. |
| **Why it matters** | Using the wrong layer makes workflows slower, harder to reason about, and easier to break. |
| **When to use it** | Use SQL for relational filtering and joins, Pandas for local table analysis and export, and NumPy for vector math and array operations. |
| **When NOT to use it** | Do not pull huge relational workloads into Pandas unnecessarily, and do not use NumPy where labeled table operations are clearer. |
| **What juniors miss** | They often reach for whichever tool they know first instead of choosing based on the shape of the task. |
| **Senior signal** | "I let the database do relational work, Pandas do diagnostic table work, and NumPy do dense numeric work." |

### Mental Model

These tools are like three workbenches in the same shop. SQL is the warehouse loading dock, Pandas is the labeled inspection table, and NumPy is the precision math bench.

tool_choice = {

| Code line / block | What it does | Why it matters |
|---|---|---|
| `tool_choice = {...}` | Maps tool to its natural role. | Good data work starts with choosing the right layer, not with memorizing APIs blindly. |
| `"sql": "extract and join"` | Assigns relational work to the database. | Pushing relational heavy lifting closer to the data source is often cleaner and cheaper. |
| `"pandas": "inspect and slice"` | Gives Pandas the diagnostic role. | Pandas shines when you need local, human-readable analysis and cohort slicing. |
| `"numpy": "dense numeric math"` | Gives NumPy the lower-level math role. | NumPy is the right tool when labels and table semantics matter less than vector operations. |

## CONCEPT 2: Vectorized Operations Beat Row Loops for Most Pandas Work
<!-- cognitive job: teach the habit of masks, loc, groupby, and aggregations instead of row iteration -->

| Dimension | Answer |
|---|---|
| **What it is** | Vectorized Pandas means operating on whole columns, masks, or groups at once instead of iterating row by row. |
| **Why it matters** | Vectorized code is usually faster, clearer, and easier to verify than ad hoc row loops. |
| **When to use it** | Use boolean masks, `.loc`, `groupby`, and aggregation for most filtering, flagging, and cohort analysis. |
| **When NOT to use it** | Do not default to `iterrows()` for normal transformations or diagnostics unless the logic truly cannot be expressed column-wise. |
| **What juniors miss** | They often translate Python-for-loop thinking directly into Pandas and create slow, brittle code. |
| **Senior signal** | "If I’m reaching for row iteration in Pandas, I first ask whether a mask, groupby, or vectorized assignment would express the same logic better." |

### Mental Model

Vectorized Pandas is like using a stencil over the whole sheet instead of drawing one mark at a time. You describe the pattern once and let the dataframe apply it across the relevant rows.

---

> **Before reading on:** Why is `iterrows()` usually a warning sign rather than a sign of mastery?

---

```python
import pandas as pd

results = pd.DataFrame({
    "faithfulness": [0.92, 0.41, 0.33],
    "relevance": [0.88, 0.52, 0.40],
})

results.loc[
    (results["faithfulness"] < 0.5) & (results["relevance"] < 0.6),
    "needs_review",
] = True
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `results = pd.DataFrame(...)` | Creates labeled tabular data. | Pandas is most useful when you need named columns and human-readable inspection. |
| `(results["faithfulness"] < 0.5) & ...` | Builds a boolean mask across rows. | This is the core vectorized pattern for filtering or flagging. |
| `results.loc[..., "needs_review"] = True` | Assigns a value to all matching rows at once. | One vectorized statement replaces many row-wise condition checks. |
| `"needs_review"` flag | Adds a diagnostic label. | This is how real eval-analysis workflows often mark failure cohorts for review. |

**Practice -- apply the concept:**

> A dataframe has 100,000 rows. You need to flag all rows where `score < 0.4` and `latency > 3000`. Describe the vectorized approach without writing a loop.

> **Real-world scenario:** An evaluation workflow was taking far too long because it used row iteration to label failures. Rewriting the logic with masks and `.loc` made the code shorter, easier to test, and much faster.

## CONCEPT 3: Shape, Nulls, Dtypes, and Cohorts Should Be Checked Before You Trust the Result
<!-- cognitive job: teach pre-trust inspection habits for data quality and evaluation honesty -->

| Dimension | Answer |
|---|---|
| **What it is** | Before trusting model metrics or data outputs, you should inspect shape, nulls, dtypes, and grouped slices. |
| **Why it matters** | Many "model problems" are actually data-shape drift, hidden nulls, merge mistakes, or one bad cohort hidden by the overall average. |
| **When to use it** | Use these checks at the start of analysis, after joins, before exports, and before declaring any metric trustworthy. |
| **When NOT to use it** | Do not jump straight to global means or charts before checking whether the table itself is structurally sane. |
| **What juniors miss** | They often trust one aggregate metric before checking whether the data is incomplete, mis-typed, or hiding a broken segment. |
| **Senior signal** | "Before I trust the metric, I check row counts, nulls, dtypes, and which cohorts the average may be hiding." |

### Mental Model

These checks are like taking vital signs before diagnosis. If the table is the wrong shape, full of nulls, or hiding one broken subgroup, the fancy downstream analysis is built on a bad physical exam.

---

> **Before reading on:** Why can one overall average be dangerous even when it looks stable?

---

```python
print(results.info())
print(results.isnull().sum())

summary = results.groupby("needs_review", dropna=False).size()
print(summary)
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `results.info()` | Shows row counts, dtypes, and memory summary. | This is one of the fastest ways to catch structural problems early. |
| `results.isnull().sum()` | Counts nulls per column. | Hidden nulls can quietly break metrics, joins, or downstream assumptions. |
| `groupby("needs_review")` | Slices the table by a meaningful cohort. | Grouped inspection often reveals patterns the global average hides. |
| `dropna=False` | Keeps missing groups visible. | Missingness itself can be an important diagnostic signal. |

**Practice -- apply the concept:**

> You merged two tables and your metric changed sharply. Name three inspection checks you should run before blaming the model.

> **Real-world scenario:** A retrieval system’s aggregate quality score looked acceptable, but grouped slicing showed one document type was failing badly because a metadata field was missing for that cohort. The "model issue" was really an upstream data-quality issue.

## DECISION RULES

### Data Tooling Framework

1. Ask whether the task is relational extraction, labeled table analysis, or dense numeric math.
2. Use SQL for large relational filtering and joins close to the database.
3. Use Pandas for local inspection, cohort slicing, aggregation, and export.
4. Use NumPy for embedding math, similarity, and low-level array operations.
5. Prefer vectorized Pandas patterns over row loops.
6. Check `.info()`, nulls, row counts, and dtypes before trusting results.
7. Slice by cohort before declaring success from an overall average.
8. If the dataframe is too large or slow, reduce columns, set dtypes, and reconsider whether SQL should have handled more of the workload.

### Worked Example

| Step | RAG evaluation example |
|---|---|
| Extract | pull run-specific rows and joins in SQL |
| Inspect | load into Pandas for null checks and grouped analysis |
| Flag | use vectorized masks to mark failures |
| Slice | group by source type or tenant to find weak cohorts |
| Numeric step | use NumPy for embedding or score-array math if needed |

### Red Flags

| What someone says / does | Why it should raise a flag |
|---|---|
| "I loaded the whole warehouse table into Pandas first." | Relational heavy lifting may be happening in the wrong layer. |
| "I used `iterrows()` for all transformations." | The workflow is probably slower and less clear than it needs to be. |
| "The overall average looks fine, so we’re done." | One broken cohort may still be hiding underneath. |
| "The dataframe worked, so I didn’t check dtypes or nulls." | Structural issues can silently distort downstream results. |

### Use / Do Not Use

| Use this framing when | Do NOT use this framing when |
|---|---|
| you need to choose the right data layer and inspect outputs honestly in AI/ML workflows | you are treating SQL, Pandas, and NumPy as interchangeable tools with no boundary logic |

## COMMON MISTAKES

### Mistake 1: Using the wrong layer for the job

**Mistaken instinct:** "I already know Pandas, so I’ll do everything there."

**Why it feels right:** Familiar tools feel faster than switching layers.

**Why it fails:** Large relational work and dense numeric work often become slower or harder to reason about when forced into the wrong tool.

**Better move:** Decide first whether the task is relational, tabular-analytic, or numeric, then choose SQL, Pandas, or NumPy accordingly.

### Mistake 2: Writing row-wise loops for ordinary Pandas logic

**Mistaken instinct:** "A for loop makes the condition easier to think about."

**Why it feels right:** Plain Python control flow is familiar and explicit.

**Why it fails:** Row loops are usually slower, noisier, and harder to verify than vectorized masks or grouped operations.

**Better move:** Reach for masks, `.loc`, `groupby`, and aggregations first.

### Mistake 3: Trusting aggregate metrics before structural checks

**Mistaken instinct:** "The mean score looks okay, so the data must be okay too."

**Why it feels right:** One summary number feels like a clean starting point.

**Why it fails:** Nulls, join errors, dtype drift, and bad cohorts can all hide behind a plausible-looking average.

**Better move:** Check shape, nulls, dtypes, and grouped slices before trusting the headline result.

### Mistake 4: Ignoring dtype and memory discipline

**Mistaken instinct:** "The dataframe loaded, so dtype details can wait."

**Why it feels right:** Early correctness seems more urgent than efficiency.

**Why it fails:** Object-heavy or oversized dtypes waste memory, slow workflows, and can break assumptions for downstream numeric code.

**Better move:** Use explicit dtypes where practical and inspect memory behavior early on larger data.

## PRACTICAL LANDSCAPE

### Technique Landscape

| Concept family | Main technique families | What they are for |
|---|---|---|
| relational work | SQL, `read_sql` | extraction and joins close to the source |
| tabular analysis | Pandas dataframe operations | inspection, slicing, aggregation, export |
| dense numeric work | NumPy arrays and linalg | vector math, similarity, numeric transforms |
| trust checks | null checks, row counts, dtypes, grouped summaries | catch bad data before trusting metrics |

### When to Use What

| If your real need is... | Usually start with... | Not with... |
|---|---|---|
| large relational extraction | SQL | local dataframe joins first |
| local diagnostic slicing | Pandas | raw NumPy arrays |
| embedding similarity or vector math | NumPy | table operations alone |
| finding broken cohorts | Pandas `groupby` | overall average only |

### Coding Surface

| Concept family | Common libraries / tools | Canonical API shape | Typical output |
|---|---|---|---|
| SQL to dataframe | `pandas`, SQLAlchemy | `pd.read_sql(query, engine)` | dataframe |
| dataframe inspection | `pandas` | `df.info()`, `df.isnull().sum()` | structure summary |
| vectorized flagging | `pandas` | `df.loc[mask, "col"] = value` | updated dataframe |
| grouped analysis | `pandas` | `df.groupby("key").agg(...)` | cohort summary table |
| embedding math | `numpy` | `np.dot`, `np.linalg.norm` | numeric score or array |

### Output Shapes to Remember

| Need | Typical output |
|---|---|
| extracted data | dataframe |
| cohort summary | grouped dataframe |
| mask-based labeling | updated column or filtered subset |
