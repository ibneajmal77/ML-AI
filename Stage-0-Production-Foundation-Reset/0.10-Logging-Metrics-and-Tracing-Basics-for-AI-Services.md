## Why This Matters

### Business Importance
When an AI feature is slow, expensive, or wrong, the business does not care that the code looked clean. It cares whether the team can detect the issue, diagnose it quickly, and fix it without guessing. Observability is what makes that possible.

### System Importance
AI services are harder to observe than normal APIs because failures can come from:
- provider latency
- malformed structured output
- retrieval misses
- background job drift
- prompt regressions
- cost spikes

Without logs, metrics, and traces, all of those failures blur together.

### Interview Importance
This topic shows up in:
- production AI interviews
- system design discussions
- LLMOps/MLOps conversations
- debugging and incident-response questions

Weak candidates say “I would add logs.” Strong candidates explain what to log, what to measure, what to trace, and how those signals differ.

## PART 1 -- CORE CONCEPT

### What It Is
Observability is the set of signals that help you understand what a system is doing after it is running. The three core signal types are:
- logs: detailed event records
- metrics: aggregated quantitative signals
- traces: end-to-end timing and path information for a request or job

### Why It Exists
AI systems have more ambiguous failures than ordinary CRUD apps. A user complaint like “the assistant is slow and wrong” could mean:
- provider latency
- retrieval miss
- invalid structured output
- prompt regression
- downstream timeout

Observability makes those failure modes separable.

### Mental Model

```text
Logs:
"what happened?"

Metrics:
"how often / how much / how long?"

Traces:
"where in the request path did the time or failure happen?"
```

### When To Use It
- in every production AI service
- in every request path that touches providers, retrieval, or background jobs
- in every system where cost or latency matters

### When Not To Overdo It
- do not log huge raw payloads by default
- do not emit metrics with uncontrolled high-cardinality labels
- do not add tracing detail so deep that nobody can read or afford it

## PART 2 -- PRODUCTION REALITY

### Real Use Cases
- logging provider timeout details and request IDs
- measuring endpoint latency and error rate
- tracking token usage and cost over time
- tracing one request across API -> retrieval -> provider -> response
- debugging stuck ingestion or evaluation jobs

### Good Patterns

**Pattern 1 -- structured logs with identifiers**
- route name
- request ID / trace ID
- tenant ID when appropriate
- model/provider
- job/document IDs where relevant

**Pattern 2 -- metrics for health, not raw detail**
- request count
- request latency
- error count
- queue depth / job state counts
- token or cost totals when available

**Pattern 3 -- traces for boundary-heavy paths**
- route span
- provider call span
- retrieval span
- database span
- worker job span

### Day-to-Day Team Reality
- support says a request failed
- logs show the request ID and error class
- metrics show whether this is a one-off or a broad incident
- traces show where time was spent or where the failure started

### Common Mistakes
- using unstructured print logs
- logging without IDs that let you correlate events
- measuring only technical latency and ignoring AI-specific fields like provider/model/prompt version
- putting user IDs or raw prompts into metrics labels, causing cardinality explosions
- assuming traces replace logs or metrics

### Failure Modes
- cannot reconstruct a failure because logs have no request ID
- metrics say latency is high, but nothing indicates whether provider, retrieval, or DB caused it
- traces exist but carry no useful metadata
- logs contain sensitive payloads that should not have been recorded
- alerting fires too late because the wrong metric was chosen

### Monitoring and Debugging
- logs for detail
- metrics for alerting and dashboards
- traces for path-level diagnosis
- separate user-visible latency from queue/job latency
- track AI-specific fields carefully: provider, model, prompt version, token count, retrieval source count

### Security and Reliability Concerns
- do not log secrets or raw sensitive payloads carelessly
- be cautious with tenant/user identifiers in telemetry
- preserve enough information to debug without violating privacy or exploding telemetry cost

## PART 3 -- ENGINEERING

### Structured logging example

```python
import json
import logging


class JSONFormatter(logging.Formatter):
    def format(self, record: logging.LogRecord) -> str:
        payload = {
            "timestamp": self.formatTime(record, self.datefmt),
            "level": record.levelname,
            "logger": record.name,
            "message": record.getMessage(),
        }

        for key, value in record.__dict__.items():
            if key not in {
                "msg", "args", "levelname", "name", "pathname", "filename",
                "module", "exc_info", "exc_text", "stack_info", "lineno",
                "funcName", "created", "msecs", "relativeCreated", "thread",
                "threadName", "processName", "process", "message", "taskName",
            }:
                payload[key] = value

        return json.dumps(payload)
```

### Request logging pattern

```python
import logging
import time
from uuid import uuid4

from fastapi import Request


logger = logging.getLogger(__name__)


async def log_request_boundary(request: Request, route_name: str):
    request_id = str(uuid4())
    start = time.perf_counter()

    logger.info(
        "request_started",
        extra={
            "request_id": request_id,
            "route": route_name,
            "method": request.method,
            "path": request.url.path,
        },
    )

    return request_id, start
```

### Simple metrics mental model

```text
Counter:
number of requests, failures, retries

Histogram / summary:
latency, provider duration, token count

Gauge:
queue depth, active jobs, in-flight requests
```

### AI-specific log fields to include

```text
request_id
trace_id
route
tenant_id (if appropriate)
provider
model_name
prompt_version
job_id / document_id
token_input / token_output
latency_ms
error_class
```

### Trace mental model

```text
Trace: one end-to-end request
  Span 1: API route
  Span 2: retrieval
  Span 3: provider call
  Span 4: response serialization
```

### Wrong Way Example

```text
Bad pattern:
- print("calling model...")
- print("got response")
- no request ID
- no model/provider field
- no latency
- no error categorization

This is not observability. It is noise.
```

## PART 5 -- SENIORITY LAYER

### Seniority Story 1 -- Failure
"We had a user complaint that the assistant was 'slow and wrong.' Latency metrics alone showed the system was slow, but not why. Once we added request IDs, provider/model fields, retrieval source counts, and span timing, it became obvious the slowdown was mostly retrieval, while the wrongness was caused by missing context from one source type."

### Seniority Story 2 -- Design Decision
"We kept metrics low-cardinality and pushed detailed context into logs and traces. The goal was to keep dashboards useful and affordable while still preserving enough detail to debug individual failures."

### Tradeoff Language
"Logs are detailed but noisy. Metrics are great for alerting but weak for root cause. Traces show the path but do not replace event detail. You need all three, but each one should be used for what it is good at."

### Failure Vocabulary
"The incident was not just elevated latency. It was an observability gap. We had enough signal to know something was wrong, but not enough context to isolate whether the issue was provider slowness, retrieval failure, or output validation."

### Interview Question and Strong Answer
**Q: "How would you monitor an AI service in production?"**

Strong answer: "I would start with structured logs carrying request IDs, model/provider info, and key business identifiers; metrics for request count, error rate, latency, queue depth, and token/cost trends; and traces across route, retrieval, provider, and storage boundaries. The goal is to separate one-off failures from systemic ones and isolate where time or bad behavior originates."
