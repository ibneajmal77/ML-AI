## Why This Matters

### Business Importance
AI systems are already probabilistic and failure-prone. If the surrounding service logic is also untested, incidents multiply. Tests do not make an LLM deterministic, but they do make the system around it far less fragile.

### System Importance
Testing gives you confidence in:
- request/response contracts
- config loading
- route behavior
- timeout and error handling
- schema validation
- worker/job logic
- integrations around AI providers, not just the providers themselves

### Interview Importance
This topic appears in:
- production AI interviews
- MLOps/LLMOps discussions
- code-review and portfolio evaluation
- “how do you keep AI systems reliable?” questions

Weak candidates say “testing AI is hard.” Strong candidates explain what is deterministic enough to test, what should be mocked, and what needs regression or eval-style validation instead of classic unit tests.

## Key Takeaways -- Read This Before Any Project

1. **You usually do not unit-test the model. You unit-test the system around the model.**
2. **Most Stage 0 AI-service tests should be deterministic.**
3. **Mock external providers in baseline tests.**
4. **Test failure paths, not just happy paths.**
5. **Validation tests and route-contract tests are high-value, low-cost.**
6. **Eval-style quality testing is different from unit/integration testing.**

## PART 1 -- CORE CONCEPT

### What It Is
Testing for AI services means verifying the deterministic parts of the system so the probabilistic parts do not create unnecessary chaos. This includes route behavior, schema validation, config loading, timeout handling, job-status logic, and parsing behavior.

### Why It Exists
AI systems have enough uncertainty already. Tests are there to make sure uncertainty only comes from the model or retrieval behavior where expected, not from broken app logic, missing config, bad schema handling, or sloppy error behavior.

### Mental Model

```text
Unit tests:
"does this function/component behave correctly?"

Integration tests:
"do these pieces work together?"

Contract tests:
"does the system still accept and return the expected shapes?"

Eval tests:
"is the AI behavior good enough?"
```

### When To Use It
- on every route contract
- on every validation boundary
- on every config boundary
- on critical failure paths and previously fixed bugs

### When Not To Overdo It
- do not try to prove prompt/model quality with only unit tests
- do not make tests depend on live providers for basic correctness
- do not test framework internals instead of your own logic

## PART 2 -- PRODUCTION REALITY

### Real Use Cases
- test that missing env vars fail at startup
- test that invalid requests return `422`
- test that route timeouts map to the right error behavior
- test that malformed provider output is rejected before storage
- test that worker/job status logic handles failed and completed states

### Good Patterns

**Pattern 1 -- deterministic outer layer tests**
- config loading
- request validation
- route wiring
- serialization
- retry/timeout/error handling

**Pattern 2 -- provider mocking**
- replace live provider call with a fake function
- force success, timeout, and malformed-output scenarios

**Pattern 3 -- regression tests after incidents**
- when a bug happens, write the smallest test that would have caught it
- keep that test forever unless the behavior is intentionally removed

### Day-to-Day Team Reality
- a provider behavior changes
- your parser or route contract starts failing
- if tests cover only happy paths, the issue reaches production
- if schema/failure-path tests exist, the issue is caught before deployment

### Common Mistakes
- depending on live provider calls in ordinary tests
- testing only success scenarios
- writing broad integration tests but skipping small deterministic unit tests
- assuming “AI is nondeterministic” means “testing is mostly pointless”

### Failure Modes
- route returns the wrong status code for a known failure
- invalid model output reaches downstream storage
- config drift breaks startup silently
- request schema evolves and callers break without detection
- a fixed bug reappears because no regression test was added

### Monitoring and Debugging
- test failures should be understandable and local
- use test names that describe the behavioral contract
- keep failure-path tests close to the bug or boundary they protect
- if production incidents repeat, your test set is missing important contracts

### Security and Reliability Concerns
- test that bad or oversized requests are rejected
- test that sensitive failures do not leak internal details unnecessarily
- test access-control and tenant-shape boundaries where relevant
- keep secrets and live provider keys out of normal test execution

## PART 3 -- ENGINEERING

### Unit test for config loading

```python
from pydantic import ValidationError

from app.config import Settings


def test_settings_require_openai_key(monkeypatch):
    monkeypatch.delenv("OPENAI_API_KEY", raising=False)
    monkeypatch.setenv("DATABASE_URL", "postgresql://example")

    try:
        Settings()
        assert False, "expected validation failure"
    except ValidationError:
        pass
```

### Route contract test with `TestClient`

```python
from fastapi.testclient import TestClient

from app.main import app


client = TestClient(app)


def test_classify_route_returns_expected_shape():
    response = client.post("/v1/classify", json={"text": "Where is my invoice?"})
    assert response.status_code == 200
    body = response.json()
    assert set(body.keys()) == {"label", "confidence", "model_name"}
```

### Validation failure test

```python
def test_classify_route_rejects_empty_text():
    response = client.post("/v1/classify", json={"text": ""})
    assert response.status_code == 422
```

### Mocking a provider call

```python
import asyncio

from app.models.inference import ClassifyResponse


async def fake_provider_success(text: str) -> ClassifyResponse:
    await asyncio.sleep(0)
    return ClassifyResponse(label="billing", confidence=0.9, model_name="fake-model")
```

### Timeout-path test idea

```python
import pytest


@pytest.mark.asyncio
async def test_timeout_wrapper_raises_timeout():
    async def slow():
        await asyncio.sleep(10)

    with pytest.raises(asyncio.TimeoutError):
        await asyncio.wait_for(slow(), timeout=0.01)
```

### Structured-output parsing test

```python
import pytest


def test_parse_llm_classification_rejects_invalid_json():
    with pytest.raises(ValueError, match="invalid JSON"):
        parse_llm_classification("not json")
```

### Wrong Way Example

```text
Bad pattern:
- every test calls the live LLM provider
- failures depend on network, quota, or provider instability
- no tests for invalid input or malformed output

This creates expensive, flaky tests that still miss the most important service contracts.
```

## PART 5 -- SENIORITY LAYER

### Seniority Story 1 -- Failure
"We had a provider output shape change that broke downstream scoring. The service had plenty of happy-path tests but no malformed-output regression test. After the incident, we added one parser failure test and one route-level error-path test so the issue could never return silently."

### Seniority Story 2 -- Design Decision
"We kept baseline tests deterministic by mocking providers. Live-provider checks existed separately as smoke or eval workflows, not as the default correctness suite. That kept the main test suite fast, cheap, and reliable."

### Tradeoff Language
"The tradeoff is not realism versus fakeness. It is fast deterministic feedback versus slow flaky dependence on external systems. You need both kinds of confidence, but they belong in different layers of the testing strategy."

### Failure Vocabulary
"The incident was not caused by missing model quality checks. It was caused by a missing service-boundary regression test."

### Interview Question and Strong Answer
**Q: "How do you test an AI service?"**

Strong answer: "I test the deterministic outer layers directly: config, validation, route contracts, timeout/error handling, and parsing. I mock live provider calls in the baseline suite so tests stay reliable. Then I handle behavior-quality confidence separately through eval or regression datasets, because that is a different problem from ordinary software correctness."
