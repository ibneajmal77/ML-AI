## Key Takeaways

- A strong AI API is a system boundary, not just an endpoint.
- Route handlers should stay thin; orchestration and business logic belong in services.
- Contracts, auth, timeouts, retries, observability, and rollback are part of API design.
- Long-running AI work often needs asynchronous job patterns, not a blocking request.
- Production readiness means another engineer can deploy, debug, and operate the service safely.

## PART 1 -- CORE CONCEPT

### What a production AI API actually includes

A production API is not only:
- an HTTP route
- request parsing
- and a JSON response

It also includes:
- contract definition
- authentication and authorization
- service orchestration
- provider and database boundaries
- failure handling
- async job strategy
- monitoring and tracing
- deployment model
- rollback and operational documentation

For AI systems, the architecture must also account for:
- slow or variable provider latency
- token/cost sensitivity
- schema drift
- unstructured input
- fallback paths
- partial failures
- evaluation and behavior monitoring

### Recommended architecture

Use a layered architecture like this:

```text
api/
  routes/
  schemas/
service/
  use_cases/
  orchestration/
domain/
  entities/
  contracts/
infra/
  llm_clients/
  db/
  cache/
  queue/
  logging/
tests/
```

#### Layer responsibilities

**API layer**
- HTTP transport only
- request/response schema binding
- auth dependency hookup
- correlation ID extraction
- response code mapping

**Service/orchestration layer**
- business workflow
- prompt/model selection
- retries/timeouts/fallback decisions
- calls into infra clients
- converts infra failures into application-level outcomes

**Domain layer**
- core concepts and contracts
- job status enum
- request policies
- evaluation result semantics

**Infrastructure layer**
- actual OpenAI/Azure/provider clients
- Postgres access
- Redis/cache
- message queue / background tasks
- telemetry implementation

This separation matters because AI services change often. If everything lives inside route functions, the codebase becomes fragile quickly.

### API styles you should know

**1. Synchronous request-response**

Use when:
- work completes quickly
- user expects immediate answer
- timeout budget is manageable

Examples:
- short summarization
- classification
- metadata extraction on small input

**2. Asynchronous job-based API**

Use when:
- request may exceed normal HTTP timeout budgets
- work fans out across multiple steps
- user can poll or receive callback later

Examples:
- large document processing
- batch enrichment
- evaluation runs
- indexing pipelines

**3. Streaming response**

Use when:
- user experience benefits from progressive output
- token generation is long
- client can handle partial chunks

Examples:
- chat
- long-form generation
- agent reasoning visibility when appropriate

### What "best architecture" means here

There is no one universal best architecture. The right architecture is the one that:
- makes common paths simple
- isolates unstable dependencies
- fails clearly
- scales operationally
- is easy for other engineers to reason about
- supports safe deployment and rollback

For Stage 0, "best" means:
- modular
- typed
- observable
- testable
- versionable
- cost-aware
- secure by default

## PART 2 -- PRODUCTION REALITY

### Real business use cases

- a document summarization API with synchronous requests for small inputs and async jobs for large files
- a retrieval-backed question-answering service with auth, per-tenant rate limits, and audit logs
- a classification API that must be deterministic enough for downstream automation
- an internal AI service shared by multiple products, each needing stable contracts and versioned changes

### Production design patterns

**1. Thin route, thick service**

The route should:
- parse request
- call service
- map result to response

The service should:
- enforce workflow
- call infra clients
- implement retries, fallbacks, and domain decisions

**2. Explicit request classes**

Not all requests are equal. Separate:
- immediate sync request
- async submission request
- job status lookup
- admin/debug endpoint

Trying to force everything into one endpoint usually produces muddy contracts.

**3. Health model with liveness and readiness**

Liveness:
- process is alive

Readiness:
- service can actually handle requests because critical dependencies are reachable enough

**4. Stable external contracts, flexible internal composition**

External APIs should change carefully. Internal service logic can evolve more often.

**5. Version contracts, not chaos**

When contract changes are breaking:
- use versioning
- support migration window
- document old vs new behavior

Do not silently break clients.

**6. Protect capacity**

AI endpoints are often expensive and slow relative to CRUD APIs. Use:
- request size limits
- concurrency limits
- rate limits
- timeout budgets
- queue backpressure

**7. Separate operational endpoints**

Examples:
- `/health`
- `/ready`
- `/metrics`
- `/jobs/{id}`
- `/admin/reload-config` only if secured and justified

Avoid mixing operational control with public user endpoints casually.

### Common mistakes

- putting provider calls, prompt building, DB writes, and response formatting all in one route
- using synchronous endpoints for inherently long-running workloads
- no request size limits
- no rate limiting or tenant isolation
- no versioning strategy
- no idempotency for retried write-like operations
- logging raw sensitive user payloads
- no correlation ID
- no runbook
- assuming "works locally" means production-ready

### Failure modes

**Failure mode: route-layer spaghetti**

Cause:
- all logic is kept in one endpoint

What happens:
- testing is painful
- changes are risky
- retries and error mapping become inconsistent

**Failure mode: timeout mismatch**

Cause:
- endpoint is synchronous but real work takes 45-120 seconds

What happens:
- user sees timeouts
- duplicate retries occur
- backend work may continue without client awareness

**Failure mode: silent contract drift**

Cause:
- response shape changes
- no versioning or compatibility testing

What happens:
- downstream client breaks
- internal team claims "API is still up" while consumers fail

**Failure mode: observability without context**

Cause:
- logs exist, but no request ID, job ID, tenant ID, model name, or status code grouping

What happens:
- incidents take too long to diagnose

**Failure mode: cost runaway**

Cause:
- no throttling, no payload limits, no model routing discipline

What happens:
- token spend spikes
- latency and concurrency collapse together

### Monitoring and operational ownership

At minimum, monitor:
- request volume
- success/error rate
- p50/p95 latency
- timeout rate
- queue depth for async work
- provider failure rate
- validation failure rate
- parse failure rate
- token and cost metrics

Add dimensions where useful:
- route
- tenant
- model/provider
- prompt version
- response status family

### Security and governance

A production API should account for:
- authentication
- authorization
- secret management
- request size limits
- rate limiting
- auditability
- PII handling
- safe logging
- dependency and image scanning

Validation helps, but validation is not your security model.

### Day-to-day in a real team

Senior engineers working on AI APIs spend time on:
- endpoint contract changes
- debugging latency spikes
- deciding sync vs async behavior
- release coordination with downstream teams
- adding metrics for new failure modes
- reviewing auth and data exposure concerns
- writing or improving runbooks

That is why API architecture is not "just backend basics." In AI teams, this is core product infrastructure.

### Route and service separation example

```python
from fastapi import APIRouter, Depends, HTTPException, status

from app.api.schemas.summarize import SummarizeRequest, SummarizeResponse
from app.service.summarize_service import SummarizeService

router = APIRouter(prefix="/v1")


def get_summarize_service() -> SummarizeService:
    return SummarizeService()


@router.post("/summarize", response_model=SummarizeResponse)
async def summarize(
    request: SummarizeRequest,
    service: SummarizeService = Depends(get_summarize_service),
) -> SummarizeResponse:
    try:
        result = await service.summarize(request)
        return SummarizeResponse.model_validate(result)
    except ValueError as exc:
        raise HTTPException(
            status_code=status.HTTP_400_BAD_REQUEST,
            detail=str(exc),
        ) from exc
```

```python
from app.api.schemas.summarize import SummarizeRequest


class SummarizeService:
    async def summarize(self, request: SummarizeRequest) -> dict:
        if len(request.text) > 20_000:
            raise ValueError("Use the async document endpoint for large inputs.")

        # Business policy lives here, not in the route.
        normalized_text = request.text.strip()

        # Provider calls, retries, metrics, and fallback logic would live here.
        summary = normalized_text[:200]

        return {
            "summary": summary,
            "model_name": "gpt-4.1-mini",
            "processing_mode": "sync",
        }
```

### Async submission and status endpoint example

```python
from uuid import uuid4

from fastapi import APIRouter, status

from app.api.schemas.job import JobAcceptedResponse, JobStatusResponse

router = APIRouter(prefix="/v1/documents")

_FAKE_JOBS = {}


@router.post("/process", response_model=JobAcceptedResponse, status_code=status.HTTP_202_ACCEPTED)
async def submit_document() -> JobAcceptedResponse:
    job_id = str(uuid4())
    _FAKE_JOBS[job_id] = {"status": "queued"}
    return JobAcceptedResponse(job_id=job_id, status="queued")


@router.get("/jobs/{job_id}", response_model=JobStatusResponse)
async def get_job_status(job_id: str) -> JobStatusResponse:
    job = _FAKE_JOBS.get(job_id)
    if job is None:
        return JobStatusResponse(job_id=job_id, status="not_found")
    return JobStatusResponse(job_id=job_id, status=job["status"])
```

### Middleware-style correlation ID example

```python
from uuid import uuid4

from fastapi import Request


async def correlation_id_middleware(request: Request, call_next):
    correlation_id = request.headers.get("x-correlation-id", str(uuid4()))
    request.state.correlation_id = correlation_id

    response = await call_next(request)
    response.headers["x-correlation-id"] = correlation_id
    return response
```

### API versioning guidance

Use a simple versioning rule first:
- `/v1/...` for stable current contract
- introduce `/v2/...` only for real breaking changes

Do not version for every small internal change. Version when client expectations break.

### Auth and rate limiting expectations

Your Stage 0 API should at least define where these belong:
- auth dependency or gateway layer
- per-tenant or per-key throttling
- admin-only operational endpoints
- request size and file-type checks

Even if you do not fully implement them yet, the architecture must reserve space for them.

## PART 5 -- SENIORITY LAYER

### Failure story

"We had an endpoint that looked healthy in infra dashboards, but users kept retrying because requests timed out around the 60-second mark. The underlying background work continued, so retries created duplicate load and duplicate records. The real fix was not 'increase timeout.' We split the workflow into submit-and-poll, added idempotency for retried submissions, and exposed job status separately."

### Design decision story

"We kept the route handlers intentionally thin and moved provider selection, retry logic, and business rules into service classes. That felt slightly heavier at first, but it paid off when we added a second provider and had to test fallback behavior. The transport layer stayed stable while the orchestration changed."

### Tradeoff language

Use language like:
- "I design the API around workload shape, not just coding convenience."
- "I want stable external contracts and flexible internal composition."
- "For expensive AI endpoints, capacity protection is part of the API design."
- "If the work is naturally long-running, I prefer explicit async job semantics over pretending sync is enough."
