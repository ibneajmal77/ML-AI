## CONCEPT 1: Good Project Structure Is Boundary Design, Not Folder Decoration
<!-- cognitive job: establish that structure is about separating runtimes and responsibilities, not cosmetic neatness -->

| Dimension | Answer |
|---|---|
| **What it is** | Project structure is the deliberate separation of serving code, business logic, prompts, config, scripts, and tests into boundaries that change and fail differently. |
| **Why it matters** | When these concerns are mixed together, debugging, rollback, testing, and deployment become much harder. |
| **When to use it** | Use it from the first moment an AI project is more than a quick experiment or notebook. |
| **When NOT to use it** | Do not add platform-level complexity too early, but do not stay in one-file script mode once the project becomes a service. |
| **What juniors miss** | They often think structure is just style, when it is really runtime separation and operational control. |
| **Senior signal** | "These folders exist because these concerns change at different rates and should not share the same failure path." |

### Mental Model

Project structure is like labeling and separating the rooms in a workshop. If welding tools, paperwork, paint, and spare parts all live on one table, every change becomes risky and messy even if the workshop technically still functions.

```text
project-root/
  src/
    app/
      routes/
      services/
      models/
      prompts/
  scripts/
  tests/
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `src/app/` | Holds the actual application package. | This keeps importable serving code separate from repo-root noise. |
| `routes/` and `services/` | Splits transport from business logic. | API wiring and core logic should not be forced to change together. |
| `prompts/` | Isolates prompt assets from Python logic. | Prompt changes become reviewable and revertible without hiding inside code. |
| `scripts/` and `tests/` | Separates operational jobs and verification from the app package. | Jobs and tests have different runtime assumptions than the web app. |

**Practice -- apply the concept:**

> **Real-world scenario:** A small RAG service started as one script and became painful once ingestion, prompt iteration, and API serving all shared the same file. Once the team separated routes, services, and scripts, failures stopped spilling across unrelated workflows.

## CONCEPT 2: Settings and Prompts Are Operational Assets, Not Hidden Implementation Details
<!-- cognitive job: teach why settings must fail fast and prompts must be versioned as assets -->

| Dimension | Answer |
|---|---|
| **What it is** | Settings define the runtime contract and prompts define model behavior, so both should live in explicit, reviewable places instead of being buried in route code. |
| **Why it matters** | Missing config should fail at startup, and prompt changes should be easy to review, test, and roll back independently. |
| **When to use it** | Use this whenever your service depends on provider keys, model names, database URLs, or prompt text that changes over time. |
| **When NOT to use it** | Do not hardcode secrets or hide prompt text inside business logic once the prompt is part of the product behavior. |
| **What juniors miss** | They often treat prompts like string literals and settings like scattered constants instead of runtime contracts. |
| **Senior signal** | "If a prompt change or missing env var can change production behavior, it deserves an explicit boundary and startup validation." |

### Mental Model

Settings are like the power and water connections to a building; prompts are like the operating instructions on the machines inside. Both are critical to how the system behaves, and neither should be improvised in random places.

```python
from functools import lru_cache

from pydantic_settings import BaseSettings


class Settings(BaseSettings):
    openai_api_key: str
    app_name: str = "ai-service"


@lru_cache
def get_settings() -> Settings:
    return Settings()
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `class Settings(BaseSettings)` | Declares typed runtime configuration. | Config stops being ad hoc and becomes a validated contract. |
| `openai_api_key: str` | Marks a required environment variable. | Missing critical config fails loudly instead of becoming a runtime surprise. |
| `app_name: str = "ai-service"` | Gives a safe default for non-critical settings. | Not every setting needs to be mandatory, but the contract stays explicit. |
| `@lru_cache` + `get_settings()` | Loads settings once and reuses them. | Startup config becomes stable and cheap to access across the app. |

**Practice -- apply the concept:**

> A prompt is embedded directly inside an API route. What future change becomes harder because of that choice?

> **Real-world scenario:** A team buried prompt text in Python modules, so prompt rollback required code rollback and redeploy. After moving prompts into files, they could review and revert prompt changes without pretending every prompt edit was an application-code change.

## CONCEPT 3: A Minimal Production Repo Shape Beats a Clever but Blurry Repo Shape
<!-- cognitive job: teach the minimum useful structure and why thin routes and reusable services matter -->

| Dimension | Answer |
|---|---|
| **What it is** | A good starter AI service repo has one clear application package, thin routes, reusable services, separate scripts, tests, and deployment/config files. |
| **Why it matters** | This is enough structure to support growth without introducing platform complexity too early. |
| **When to use it** | Use it for almost any new AI API, RAG service, tool backend, or evaluation service that is beyond toy scope. |
| **When NOT to use it** | Do not split into many packages or platform abstractions before one service boundary is working cleanly. |
| **What juniors miss** | They often confuse "minimal" with "flat" and end up with one big file instead of one small, clean package. |
| **Senior signal** | "I want the smallest structure that preserves route thinness, service reuse, prompt versioning, startup validation, and testability." |

### Mental Model

Minimal production structure is like building a house with proper walls and doors before adding extra floors. You do not need a skyscraper, but you do need rooms that keep different kinds of work from crashing into one another.

```python
from fastapi import APIRouter

from app.services.answering import generate_answer

router = APIRouter()


@router.post("/answer")
async def answer(question: str) -> dict[str, str]:
    result = await generate_answer(question)
    return {"answer": result}
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `from app.services.answering import generate_answer` | Pulls business logic from a service layer. | Route files stay thin and avoid becoming the place where all application logic accumulates. |
| `router = APIRouter()` | Defines the transport boundary. | Routes should own HTTP concerns, not the whole product behavior. |
| `@router.post("/answer")` | Exposes the endpoint. | The route should mainly translate request/response, not hide every internal detail. |
| `await generate_answer(question)` | Delegates the actual work. | The same service logic can now be reused by jobs, tests, or other interfaces. |

**Practice -- apply the concept:**

> **Real-world scenario:** An internal copilot backend kept routes thin and pushed model-calling logic into services. That made it possible to run evaluation jobs against the same service layer without importing the whole API stack or duplicating business behavior.

## DECISION RULES

### Starter AI Service Structure Framework

1. Start with one application package under `src/app/`.
2. Put HTTP transport in `routes/`, core reusable logic in `services/`, and typed request/response or internal schemas in `models/`.
3. Put prompt files in `prompts/` so prompt behavior is reviewable and versioned.
4. Put one-off or operational jobs in `scripts/`, not inside route modules.
5. Put typed startup config in `config.py` and validate required settings early.
6. Keep `.env.example`, `.gitignore`, and `pyproject.toml` at repo root as project contract files.
7. Add tests early so config loading and core services can be checked without hitting the whole stack.
8. Stop there until the service boundary is actually under control; do not add architecture theater.

### Worked Example

| Step | RAG service example |
|---|---|
| App package | `src/app/` holds serving code |
| Route boundary | `/ask` route lives under `routes/` |
| Core logic | retrieval + answer generation live under `services/` |
| Prompt asset | answer prompt file lives under `prompts/` |
| Operations | indexing job lives under `scripts/ingest.py` |
| Safety | `config.py` validates provider key and DB URL at startup |

### Red Flags

| What someone says / does | Why it should raise a flag |
|---|---|
| "Everything is still in `main.py`, but we’ll refactor later." | That usually means the wrong boundaries are becoming harder to untangle. |
| "Prompts are just strings, so keeping them in route files is fine." | Prompt behavior becomes harder to review, test, and roll back. |
| "Scripts import half the API app because it was convenient." | Runtime concerns are coupling in the wrong direction. |
| "We’ll add tests once the structure settles." | Weak structure often becomes the reason tests keep being delayed. |

### Use / Do Not Use

| Use this structure when | Do NOT use this structure when |
|---|---|
| you are building a real AI service that needs routes, prompts, config, scripts, and tests to evolve safely | you are trying to invent platform complexity before one service has clean boundaries |

## COMMON MISTAKES

### Mistake 1: Treating project structure as a style preference

**Mistaken instinct:** "We can organize the repo later once the product idea is working."

**Why it feels right:** Early velocity feels more important than folder decisions.

**Why it fails:** Once routes, prompts, scripts, and config all depend on one another in messy ways, the cleanup cost rises sharply and slows every next change.

**Better move:** Add the minimum good boundaries early so later growth has somewhere clean to attach.

### Mistake 2: Hiding prompts inside application logic

**Mistaken instinct:** "It’s just one string, so keeping it inline is simpler."

**Why it feels right:** Inline prompts are fast to start with and avoid another file.

**Why it fails:** Prompt behavior becomes harder to review, diff, version, test, and roll back independently.

**Better move:** Treat prompts as explicit assets once they are part of real service behavior.

### Mistake 3: Letting routes become the service layer

**Mistaken instinct:** "The route already has the request, so it might as well do the work too."

**Why it feels right:** It feels like fewer layers and faster implementation.

**Why it fails:** Reuse drops, tests become harder, and operational jobs start importing transport logic just to access business behavior.

**Better move:** Keep routes thin and push reusable work into services.

### Mistake 4: Discovering config problems only during live requests

**Mistaken instinct:** "We’ll know if a key is missing when the provider call fails."

**Why it feels right:** It defers setup work and keeps the code moving.

**Why it fails:** The first real failure becomes a runtime incident instead of a clean startup failure with a clear message.

**Better move:** Validate required settings at startup through typed config.

## SUMMARY

**Five things to carry out of this lesson:**

1. Good AI service structure is about separating concerns that change and fail differently, not about making folders look elegant.
2. Routes, services, prompts, scripts, settings, and tests should not live in the same runtime boundary.
3. Typed settings turn configuration into a startup contract instead of a runtime surprise.
4. Prompt files should be treated as versioned assets once prompt behavior matters to the product.
5. A minimal clean structure beats both one-file chaos and premature architecture complexity.

```text
settings -> startup contract
routes   -> transport boundary
services -> reusable logic
prompts  -> versioned assets
scripts  -> operational jobs
tests    -> verification
```

## PRACTICAL LANDSCAPE

### Technique Landscape

| Concept family | Main technique families | What they are for |
|---|---|---|
| application boundary | `src/app`, routes, services, models | organize importable serving code |
| operational boundary | scripts, tests, deployment files | separate jobs and verification from app runtime |
| runtime contract | typed settings, `.env.example`, `.gitignore` | safe configuration and startup behavior |
| behavior assets | prompt files, static templates | versioned non-code behavior inputs |

### When to Use What

| If your real need is... | Usually start with... | Not with... |
|---|---|---|
| a clean service repo | one `src/app` package plus scripts/tests | many top-level ad hoc Python files |
| reusable business logic | `services/` | piling logic into route handlers |
| prompt review and rollback | `prompts/` as files | hiding prompts in Python strings forever |
| safe runtime config | typed settings + `.env.example` | scattered constants and secret literals |

### Coding Surface

| Concept family | Common libraries / tools | Canonical API shape | Typical output |
|---|---|---|---|
| settings | `pydantic-settings` | `class Settings(BaseSettings)` | validated config object |
| API routes | `fastapi` | `APIRouter()` + route functions | HTTP boundary |
| services | plain Python modules | `service_fn(...)` | reusable domain logic |
| prompts | `pathlib` / text files | `Path(...).read_text()` | prompt string |
| tests | `pytest` | `def test_...():` | verification result |

### Output Shapes to Remember

| Need | Typical output |
|---|---|
| config contract | typed settings object |
| route layer | request/response handlers |
| prompt store | versioned `.txt` files |
| reusable logic | service functions or classes |
| operational entrypoint | script file under `scripts/` |

### Interview Q&A

**Q: Why separate route logic from service logic?**

> "Because the route is just the transport boundary. The real business behavior often needs to be reused by tests, jobs, or other interfaces, and that becomes awkward fast if the logic lives inside the HTTP layer. Thin routes make the system easier to test and evolve."

**Q: Why should prompts live outside Python code?**

> "Because once prompt behavior matters to the product, the prompt is no longer just an implementation detail. It should be reviewable, versioned, and revertible like other important assets, and keeping it outside route code makes that much easier."
