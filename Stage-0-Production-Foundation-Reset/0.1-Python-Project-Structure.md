## Why This Matters

### Business Importance
A badly structured AI project slows everything down:
- prompt changes become risky
- debugging gets harder
- deployment becomes fragile
- experiments and evals stop being reproducible

Good structure is not a cosmetic choice. It directly affects how fast and safely a team can ship AI features.

### System Importance
AI services have more runtime concerns than a typical backend app:
- prompt files
- model/provider configs
- ingestion scripts
- eval jobs
- API routes
- deployment assets

If those concerns are not separated early, the codebase becomes hard to test, hard to reuse, and hard to operate.

### Interview Importance
This topic is rarely asked as a standalone theory question, but it shows up immediately in:
- portfolio review
- code review discussions
- system design conversations
- “walk me through one of your AI projects” interviews

Weak candidates show a repo full of mixed scripts and notebooks. Strong candidates show a project shape that already reflects production thinking.

## Key Takeaways -- Read This Before Any Project

1. **Separate runtime responsibilities early.** API serving, ingestion, evaluation, and config do not belong in one file.
2. **Prompt files should be versioned like code.** If prompts change independently of logic, the project structure should reflect that.
3. **Settings should fail loudly at startup.** Missing config should not become a mid-request surprise.
4. **Routes are not the place for all logic.** Route code should stay thin; services should own the work.
5. **A clean repo is an operational tool, not just a style preference.**

## PART 1 -- CORE CONCEPT

### What It Is
A Python project structure for AI services is the deliberate organization of code, configuration, prompts, tests, scripts, and deployment assets so that the same system can run cleanly in local development, CI/CD, and production.

### Why It Exists
AI projects have more moving parts than generic scripts:
- prompts evolve independently of Python code
- configuration changes by environment
- ingestion and evaluation often run as scripts or jobs, not API requests
- service logic needs reuse across routes, jobs, and tests

### Mental Model

```text
[Config + Secrets]        environment-specific
      |
[Application Package]     routes, services, models, prompts
      |
[Operational Scripts]     ingestion, evals, migrations
      |
[Tests + Deployment]      CI, Docker, verification
```

Each layer changes at a different rate and fails in a different way.

### When To Use It
- from the first non-trivial AI project
- whenever a notebook starts becoming a service
- whenever prompts, routes, or scripts are no longer one-file experiments

### When Not To Overdo It
- do not introduce monorepo/platform complexity too early
- do not split into many packages before you have stable boundaries
- do not add “architecture” that only creates ceremony without solving real separation problems

## PART 2 -- PRODUCTION REALITY

### Real Use Cases
- a RAG service with API routes, ingestion scripts, and eval scripts
- an internal copilot with versioned prompts and typed request/response models
- a workflow agent service where serving, tools, and background jobs must stay separate

### Good Patterns

**Pattern 1 -- thin transport, reusable service logic**
- routes call services
- jobs call services
- tests target services directly when possible

**Pattern 2 -- prompts as assets, not hidden literals**
- prompts live in files
- prompt changes are reviewed and revertible

**Pattern 3 -- scripts are not the app package**
- ingestion/eval scripts should not pretend to be route handlers
- app code and one-off code should be clearly separated

### Day-to-Day Team Reality
- one engineer changes prompt behavior
- another updates route logic
- another adds eval scripts
- platform or DevOps engineers touch deployment/config concerns

If structure is weak, these changes collide constantly.

### Common Mistakes
- all code in `main.py`
- prompts embedded in route functions
- scripts importing half the app accidentally
- `.env` committed or copied carelessly
- tests added late because code was not written with separable boundaries

### Failure Modes
- deployment works locally but fails in CI because imports are messy
- prompt rollback requires code rollback because prompt text is buried in logic
- ingestion scripts break API startup because runtime responsibilities are coupled
- secrets leak because config handling was improvised

### Monitoring and Debugging
- startup should fail if required settings are missing
- logs should include safe app metadata like app name or provider/model
- CI should validate imports, settings loading, and route wiring
- the repo should make it obvious where to look when a prompt, route, or job fails

### Security and Reliability Concerns
- ignore `.env` before the first commit
- keep `.env.example` committed as documentation
- isolate prompt files and scripts so risky changes are reviewable
- avoid hardcoded secrets, provider IDs, or database strings in application code

## PART 3 -- ENGINEERING

### Recommended project layout

```text
project-root/
  src/
    app/
      __init__.py
      main.py
      config.py
      routes/
      services/
      models/
      prompts/
        classify_v1.txt
        summarize_v1.txt
  scripts/
    ingest.py
    evaluate.py
    migrate.py
  tests/
    unit/
    integration/
  docker/
    Dockerfile
  .env.example
  .gitignore
  pyproject.toml
  README.md
```

### Settings model

```python
# src/app/config.py
from functools import lru_cache

from pydantic_settings import BaseSettings, SettingsConfigDict


class Settings(BaseSettings):
    openai_api_key: str
    openai_model: str = "gpt-4o"
    database_url: str
    app_name: str = "ai-service"
    debug: bool = False

    model_config = SettingsConfigDict(
        env_file=".env",
        env_file_encoding="utf-8",
        case_sensitive=False,
    )


@lru_cache
def get_settings() -> Settings:
    return Settings()
```

### Prompt file loading

```python
# src/app/services/prompts.py
from pathlib import Path


PROMPTS_DIR = Path(__file__).resolve().parent.parent / "prompts"


def load_prompt(name: str) -> str:
    return (PROMPTS_DIR / name).read_text(encoding="utf-8")
```

### FastAPI app wiring

```python
# src/app/main.py
import logging
from contextlib import asynccontextmanager

from fastapi import FastAPI

from app.config import get_settings

logger = logging.getLogger(__name__)


@asynccontextmanager
async def lifespan(app: FastAPI):
    settings = get_settings()
    logger.info("service_starting", extra={"app_name": settings.app_name, "model": settings.openai_model})
    yield
    logger.info("service_stopping", extra={"app_name": settings.app_name})


def create_app() -> FastAPI:
    settings = get_settings()
    return FastAPI(
        title=settings.app_name,
        debug=settings.debug,
        lifespan=lifespan,
    )


app = create_app()
```

### `.env.example`

```bash
OPENAI_API_KEY=your-key-here
OPENAI_MODEL=gpt-4o
DATABASE_URL=postgresql+asyncpg://user:pass@localhost:5432/aidb
APP_NAME=ai-service
DEBUG=false
```

### Minimal config test

```python
# tests/unit/test_config.py
from app.config import Settings


def test_settings_from_env(monkeypatch):
    monkeypatch.setenv("OPENAI_API_KEY", "test-key")
    monkeypatch.setenv("DATABASE_URL", "postgresql://example")
    settings = Settings()
    assert settings.openai_api_key == "test-key"
    assert settings.database_url == "postgresql://example"
```

## PART 5 -- SENIORITY LAYER

### Seniority Story 1 -- Failure
"We had prompt text embedded in Python modules. A prompt regression meant a code rollback and redeploy, even though the application logic itself was fine. After that, we moved prompts into versioned files so they could be reviewed and reverted independently."

### Seniority Story 2 -- Design Decision
"We separated scripts from the app package because ingestion, evals, and serving had different runtime assumptions. That paid off the first time CI needed to run eval logic without pulling in server-only dependencies."

### Tradeoff Language
"The tradeoff is not just neatness versus speed. A flat repo gets you to first demo faster. A structured repo gets you to stable iteration, testing, and rollback faster once the feature becomes real."

### Failure Vocabulary
"The incident was caused by weak runtime separation. Prompt logic, route logic, and operational scripts were coupled tightly enough that one change affected unrelated execution paths."

### Interview Question and Strong Answer
**Q: "How would you structure a new AI service project?"**

Strong answer: "I create a clear application package with routes, services, models, and prompt files, plus separate scripts for ingestion and evaluation. Settings are typed and loaded from environment variables, and prompts are versioned as files rather than buried in code. The goal is to keep serving, jobs, config, and testing separable from day one."
