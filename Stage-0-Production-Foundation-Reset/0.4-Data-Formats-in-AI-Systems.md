## Why This Matters

### Business Importance
AI systems rarely fail only because of the model. They fail because ingestion, parsing, serialization, and logging are weak. If a document was uploaded but never extracted cleanly, the best retriever in the world cannot save the system.

### System Importance
You will encounter multiple data shapes in real AI systems:
- JSON from APIs and models
- CSV and Parquet in analytics and evaluation
- PDF and DOCX in enterprise retrieval
- HTML from crawled/web content
- structured logs for observability

### Interview Importance
This topic often appears indirectly in:
- RAG design interviews
- production debugging scenarios
- ingestion pipeline discussions
- “why did the system miss this document?” questions

Weak candidates assume extraction and parsing work. Strong candidates treat these as failure-prone boundaries that must be validated and monitored.

## PART 1 -- CORE CONCEPT

### What It Is
AI systems ingest and emit multiple data formats: JSON for APIs and model outputs, CSV and Parquet for analytics, PDF and DOCX for enterprise documents, HTML for crawled content, and structured logs for operational visibility.

### Why It Exists
Production AI systems are only as reliable as their ingestion and serialization layers. Real failures often happen before retrieval or generation:
- empty text extracted from scanned PDFs
- malformed JSON from model outputs
- schema drift in CSVs
- noisy HTML embedded without cleanup

### Mental Model

```text
Human-readable text:
JSON, CSV, Markdown, HTML, logs

Rich documents:
PDF, DOCX, XLSX

Columnar analytics:
Parquet
```

### When To Use It
- JSON for API boundaries and structured outputs
- CSV for simple ad hoc exchange
- Parquet for typed analytics and evaluation storage
- PDF/DOCX/HTML extraction for document ingestion pipelines
- structured logs for production debugging and incident analysis

### When Not To Overdo It
- do not build custom parsers before you understand the basic failure modes
- do not use CSV as a universal default if schema and type stability matter
- do not assume text extraction preserves meaning or layout automatically

## PART 2 -- PRODUCTION REALITY

### Real Use Cases
- parsing LLM JSON outputs into schemas
- extracting enterprise documents for RAG
- writing evaluation results to analytics-friendly storage
- debugging ingestion and retrieval failures via structured logs

### Good Patterns

**Pattern 1 -- validate before trust**
- JSON is parsed and validated
- CSV columns are checked
- extracted document text is quality-checked

**Pattern 2 -- keep ingestion observable**
- track extraction success
- track character count and chunk count
- track source/document identifiers through the pipeline

**Pattern 3 -- match format to responsibility**
- JSON for API payloads
- Parquet for typed analytics
- structured logs for operations

### Day-to-Day Team Reality
- support says a document exists but the assistant cannot answer from it
- engineers inspect extraction output, chunk counts, metadata, and logs
- the fix often happens before retrieval or prompting is touched

### Common Mistakes
- trusting PDF extraction output without validation
- using CSV as a durable multi-service contract
- embedding raw HTML without stripping noise
- logging unstructured free text that is impossible to query later

### Failure Modes
- scanned PDFs produce near-empty text
- malformed JSON is treated as valid enough and fails later
- CSV schema drift silently changes downstream logic
- important tables in PDFs are flattened or lost in plain text extraction
- logs contain too little structured context to reconstruct the incident

### Monitoring and Debugging
- track extraction success rate, extracted text length, and chunk count
- log document IDs, source type, tenant, and ingestion status transitions
- sample malformed JSON and schema failures safely
- alert on spikes in empty extraction or invalid structured-output parsing

### Security and Reliability Concerns
- do not log raw payloads carelessly if they may contain sensitive content
- validate document/source metadata before deeper processing
- keep inter-service contracts structured and explicit
- do not let ingestion failures become silent partial success

## PART 3 -- ENGINEERING

### JSON parsing with validation

```python
import json
from typing import Optional

from pydantic import BaseModel, ValidationError


class LLMResponseSchema(BaseModel):
    answer: str
    confidence: float
    sources: list[str] = []
    metadata: Optional[dict] = None


def parse_llm_json_response(raw_text: str) -> LLMResponseSchema:
    text = raw_text.strip()
    if text.startswith("```json"):
        text = text[7:]
    if text.startswith("```"):
        text = text[3:]
    if text.endswith("```"):
        text = text[:-3]
    text = text.strip()

    try:
        data = json.loads(text)
        return LLMResponseSchema(**data)
    except json.JSONDecodeError as exc:
        raise ValueError(f"LLM did not return valid JSON: {exc}") from exc
    except ValidationError as exc:
        raise ValueError(f"LLM JSON missing required fields: {exc}") from exc
```

### CSV validation

```python
from pathlib import Path

import pandas as pd


def load_eval_csv(path: Path) -> pd.DataFrame:
    required_columns = {"question_id", "answer", "faithfulness", "relevance"}
    df = pd.read_csv(path, encoding="utf-8")

    missing = required_columns - set(df.columns)
    if missing:
        raise ValueError(f"CSV missing required columns: {missing}")

    return df
```

### PDF and DOCX extraction

```python
from pathlib import Path

import fitz
from docx import Document


def extract_text_from_pdf(pdf_path: Path) -> str:
    doc = fitz.open(pdf_path)
    pages_text = [page.get_text("text") for page in doc]
    full_text = "\n\n".join(pages_text).strip()
    if len(full_text) < 100:
        raise ValueError(f"Extraction too short for {pdf_path}; OCR may be required")
    return full_text


def extract_text_from_docx(docx_path: Path) -> str:
    doc = Document(docx_path)
    text = "\n".join(paragraph.text for paragraph in doc.paragraphs).strip()
    if len(text) < 50:
        raise ValueError(f"DOCX extraction too short for {docx_path}")
    return text
```

### HTML cleanup

```python
import re

from bs4 import BeautifulSoup


def extract_text_from_html(html_content: str) -> str:
    soup = BeautifulSoup(html_content, "html.parser")

    for tag in soup(["script", "style", "nav", "footer", "header"]):
        tag.decompose()

    text = soup.get_text(separator="\n")
    return re.sub(r"\n{3,}", "\n\n", text).strip()
```

### Structured logging

```python
import json
import logging


class JSONFormatter(logging.Formatter):
    def format(self, record: logging.LogRecord) -> str:
        log_data = {
            "timestamp": self.formatTime(record, self.datefmt),
            "level": record.levelname,
            "logger": record.name,
            "message": record.getMessage(),
        }

        for key, value in record.__dict__.items():
            if key not in {
                "msg", "args", "levelname", "name", "pathname", "filename",
                "module", "exc_info", "exc_text", "stack_info", "lineno",
                "funcName", "created", "msecs", "relativeCreated", "thread",
                "threadName", "processName", "process", "message", "taskName",
            }:
                log_data[key] = value

        return json.dumps(log_data)
```

## PART 5 -- SENIORITY LAYER

### Seniority Story 1 -- Failure
"We were embedding empty strings from image-only PDFs because extraction technically 'succeeded.' The retriever looked broken, but the real issue was silent extraction degradation upstream."

### Seniority Story 2 -- Design Decision
"We moved larger analytics datasets from CSV to Parquet because typed schemas and column reads mattered more than spreadsheet convenience. CSV stayed for simple ad hoc exchange, not for durable analytics contracts."

### Tradeoff Language
"CSV is easy for ad hoc sharing, but weak as a durable contract. Parquet is better when schema stability, compression, and selective reads matter. JSON is the right API boundary, not the default answer for every analytics problem."

### Failure Vocabulary
"The incident was caused by ingestion degradation, not hallucination. The model had no useful context because the extraction layer had already failed."

### Interview Question and Strong Answer
**Q: "A document was uploaded, but the RAG system says it has no information. How do you debug it?"**

Strong answer: "I check ingestion success, extracted text length, chunk count, metadata presence, and retrieval for that source ID. If extraction is empty or too short, the problem is upstream. If chunks exist but retrieval still misses, I inspect metadata filters, chunking, and ranking."
