## Why This Matters

### Business Importance
Many AI/ML failures are blamed on the model when the real issue is structured data: wrong joins, stale rows, weak filters, missing labels, or broken metadata. SQL is how you inspect and control those problems before they become model or product failures.

### System Importance
Even LLM- and RAG-heavy systems still depend on structured data for:
- metadata
- access control
- evaluation tables
- ingestion state
- audit trails
- prompt/model version tracking

### Interview Importance
SQL remains one of the highest-signal parts of AI/ML interviews because it shows whether a candidate can reason about data concretely rather than just talk about models abstractly.

## Key Takeaways -- Read This Before Any Project

1. **SQL is still core AI/ML infrastructure.** Models and LLMs depend on structured metadata, labels, evaluation records, and filters.
2. **A bad join can look like a model-quality problem.** Missing rows, duplicated rows, or stale rows distort everything downstream.
3. **Use SQL for relational work, Pandas for local analysis, and vector stores for semantic retrieval.** Do not confuse those responsibilities.
4. **Tenant filters, freshness rules, and audit trails belong in structured systems, not in vague application assumptions.**
5. **Parameterize queries always.** Unsafe string formatting is not acceptable in production systems.
6. **Most retrieval and evaluation debugging starts with metadata tables and SQL, not with the model itself.**

## PART 1 -- CORE CONCEPT

### What It Is
SQL is the language used to query, transform, validate, and analyze structured data. PostgreSQL is a relational database commonly used in AI applications for metadata, ingestion status, evaluation records, audit trails, session state, and often moderate-scale vector search via `pgvector`.

### Why It Exists
Even if your application uses LLMs or a vector database, the system still needs structure:
- which document belongs to which tenant
- which prompt version was used in an eval run
- which documents were ingested successfully
- which cohort or source type is underperforming
- which records are stale or duplicated

### Mental Model

```text
SQL:
"what rows are we actually working with?"

PostgreSQL:
"where does the structured state of the AI system live?"

Vector retrieval:
"which records are semantically similar?"
```

### When To Use It
- metadata storage
- evaluation analysis
- label generation
- data quality checks
- debugging by tenant, cohort, or source
- structured filters paired with retrieval systems

### When Not To Overdo It
- do not force high-volume semantic retrieval logic to become complex relational logic
- do not move clearly interactive ad hoc slicing into giant static SQL if Pandas is faster for local diagnosis
- do not assume Postgres must do everything just because it can

## PART 2 -- PRODUCTION REALITY

### Real Use Cases
- document metadata for RAG filtering by tenant, source type, freshness, and active state
- eval runs and eval cases for comparing prompt/model versions
- ingestion status tracking for uploaded documents
- ML label generation from user/event/activity tables
- audit trails for who uploaded, processed, or queried what

### Good Patterns

**Pattern 1 -- metadata table next to unstructured content**
- raw documents live in object storage
- structured metadata lives in PostgreSQL
- application and retrieval filters depend on SQL

**Pattern 2 -- eval runs as relational records**
- one table for run metadata
- one table for case-level scores
- one table or view for failures requiring review

**Pattern 3 -- relational filtering before or around retrieval**
- filter by tenant and active state first
- then perform retrieval or scoped ranking
- avoid vague application-side filtering after too much work is already done

### Day-to-Day Team Reality
- support says one customer is seeing poor results
- product says one source type seems worse than others
- engineering checks eval tables, prompt versions, document freshness, and chunk counts
- SQL is usually the first debugging tool, not the last

### Common Mistakes
- using the wrong join type and silently dropping rows
- mixing historical and current data without time filters
- relying on `select *` in pipelines where schemas evolve
- applying tenant or status filtering too late
- leaving key columns unindexed, making debugging and reporting painfully slow

### Failure Modes
- eval averages improve because a bad join removed failing rows
- retrieval appears broken but ingestion state shows documents were never chunked
- one tenant gets another tenant’s metadata because filters were weak
- stale rows remain active and make the system look inconsistent
- duplicate chunks or duplicate eval rows distort quality reporting

### Monitoring and Debugging
- track counts before and after joins or pipeline stages
- check null rates on key fields used in evals and filtering
- track freshness lag between upload, ingestion, and ready states
- inspect slow queries and missing indexes on frequent debugging paths

### Security and Reliability Concerns
- always parameterize queries
- enforce tenant isolation in query logic and schema design
- control access to metadata and audit tables
- avoid storing sensitive raw data when metadata is enough
- keep status transitions explicit so operators know whether a job is queued, running, failed, or done

## PART 3 -- ENGINEERING

### Example schema for AI application metadata

```sql
create table documents (
    document_id uuid primary key,
    tenant_id uuid not null,
    source_path text not null,
    source_type text not null,
    title text,
    uploaded_at timestamptz not null default now(),
    ingested_at timestamptz,
    is_active boolean not null default true
);

create table document_chunks (
    chunk_id uuid primary key,
    document_id uuid not null references documents(document_id),
    tenant_id uuid not null,
    chunk_index int not null,
    content text not null,
    token_count int,
    created_at timestamptz not null default now()
);

create index idx_documents_tenant_uploaded_at
    on documents (tenant_id, uploaded_at desc);

create index idx_chunks_document_id
    on document_chunks (document_id);
```

### Parameterized query from Python

```python
from sqlalchemy import create_engine, text


engine = create_engine("postgresql+psycopg://user:pass@localhost:5432/aidb")

query = text("""
select
    d.document_id,
    d.title,
    d.source_type,
    count(c.chunk_id) as chunk_count
from documents d
left join document_chunks c
    on d.document_id = c.document_id
where d.tenant_id = :tenant_id
  and d.is_active = true
group by d.document_id, d.title, d.source_type
order by chunk_count desc
""")

with engine.connect() as conn:
    rows = conn.execute(query, {"tenant_id": "tenant-a"}).fetchall()
```

### Pulling eval data into Pandas

```python
import pandas as pd

query = """
select
    tenant_id,
    source_type,
    prompt_version,
    faithfulness,
    relevance
from rag_eval_cases
where run_id = %(run_id)s
"""

df = pd.read_sql(query, engine, params={"run_id": "eval_run_2026_04_03"})
```

### Label-generation query pattern

```sql
with user_events as (
    select
        user_id,
        event_time,
        event_type
    from app_events
    where event_time >= current_date - interval '90 days'
),
churn_labels as (
    select
        user_id,
        max(case when event_time >= current_date - interval '30 days' then 1 else 0 end) as active_last_30d
    from user_events
    group by user_id
)
select
    user_id,
    case when active_last_30d = 0 then 1 else 0 end as churn_label
from churn_labels;
```

### Useful debugging checks

```sql
-- Null-rate check
select
    count(*) as total_rows,
    count(*) filter (where faithfulness is null) as null_faithfulness,
    count(*) filter (where relevance is null) as null_relevance
from rag_eval_cases
where run_id = 'eval_run_2026_04_03';

-- Duplicate chunk detection
select
    document_id,
    chunk_index,
    count(*) as duplicate_count
from document_chunks
group by document_id, chunk_index
having count(*) > 1;
```

### `pgvector` mental model

```text
PostgreSQL + pgvector is often enough when:
- you already use Postgres heavily
- scale is moderate
- metadata filtering and transactional consistency matter

Move to a dedicated vector store when:
- semantic retrieval is the main workload
- latency/scale requirements outgrow the current setup
- vector-specific operational features become dominant
```

### Wrong Way Example

```text
Bad pattern:
"The vector DB handles everything."

Why it fails:
- tenant filters become fragile
- eval/debug analysis has no relational home
- ingestion state and auditability become harder to reason about
```

## PART 5 -- SENIORITY LAYER

### Seniority Story 1 -- Failure
"We thought retrieval quality had regressed, but the real issue was a bad inner join in the evaluation pipeline that dropped an entire failing source type. The model looked better only because the worst rows disappeared."

### Seniority Story 2 -- Design Decision
"We kept document metadata and ingestion state in PostgreSQL even though document bodies lived elsewhere. That gave us reliable tenant filtering, auditability, and fast operational debugging without making the retrieval layer own every relational concern."

### Tradeoff Language
"SQL is the right tool when the problem is relational, auditable, and slice-heavy. Pandas is better for interactive local analysis. A vector store solves semantic similarity, not relational correctness."

### Failure Vocabulary
"The issue was not model drift first. It was data-shape drift caused by a query change that altered which rows were eligible for evaluation."

### Interview Question and Strong Answer
**Q: "How does SQL matter in an LLM or RAG system?"**

Strong answer: "SQL handles the structured side of the system: document metadata, tenant filters, eval records, ingestion state, and audit trails. In practice I use it to debug why one tenant, source type, or prompt version is failing before I assume the model or retriever is the issue. It is also where I enforce a lot of the filtering and control logic that should not be left to vague application assumptions."
