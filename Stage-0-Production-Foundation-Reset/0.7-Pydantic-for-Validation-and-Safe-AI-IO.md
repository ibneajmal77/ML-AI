## Why This Matters

### Business Importance
AI systems fail in expensive ways when bad input or malformed model output is allowed to move deeper into the stack. Validation is one of the cheapest ways to stop bad data early and keep incidents smaller.

### System Importance
Pydantic matters at two high-risk boundaries:
- inbound requests, jobs, and metadata entering the system
- outbound structured responses from LLMs, providers, and internal components

It turns raw payloads into typed objects or rejects them early with useful errors.

### Interview Importance
This topic shows up in:
- FastAPI and backend-for-AI interviews
- structured-output discussions
- production debugging and reliability interviews
- contract and schema-drift discussions

Weak candidates say “I ask the model for JSON.” Strong candidates explain validation boundaries, strictness tradeoffs, and error categorization.

## Key Takeaways -- Read This Before Any Project

1. **LLM output is still untrusted input until validated.**
2. **Validation should happen at boundaries, not randomly deep inside business logic.**
3. **Default Pydantic behavior may coerce values.** If you need `"0.85"` to fail instead of becoming `0.85`, use stricter validation.
4. **User validation failures and provider-output validation failures are not the same operational event.**
5. **Validation reduces malformed-input risk, but it is not a substitute for auth, policy, or sanitization.**
6. **Schema strictness is a tradeoff, not a religion.**

## PART 1 -- CORE CONCEPT

### What It Is
Pydantic is a Python data-validation library that converts raw, untrusted input into typed Python objects. In AI systems, that matters for API requests, ingestion metadata, evaluation records, and structured outputs returned by LLMs or internal services.

### Why It Exists
Without explicit validation:
- bad data moves deeper into the system
- runtime failures occur far from the real boundary problem
- provider output drift becomes invisible until storage, scoring, or response logic breaks
- incidents get misclassified as “model problems” when they were actually contract problems

### Mental Model

```text
Untrusted payload
      |
      v
Validation boundary
      |
      +--> fail early with useful detail
      |
      v
Trusted typed object
```

### When To Use It
- every request entering your service
- every structured output returned by an LLM or ML provider
- internal data contracts where shape drift is dangerous
- eval rows and ingestion metadata before they affect metrics or storage

### When Not To Overdo It
- do not make every schema maximally strict without considering contract ownership and change cadence
- do not confuse validation with full application security
- do not create overly abstract schema layers before you have real boundary complexity

## PART 2 -- PRODUCTION REALITY

### Real Use Cases
- validating an API request before a provider call spends tokens
- parsing LLM output into a typed schema before it touches a database
- validating retrieval metadata before passing it into ranking or storage
- rejecting malformed eval rows before metrics are computed

### Good Patterns

**Pattern 1 -- validate at the boundary, pass typed objects inside**
- convert once
- stop passing raw dicts around

**Pattern 2 -- separate request validation from provider-response validation**
- client sent bad data -> request validation failure
- provider returned malformed/invalid shape -> upstream contract failure

**Pattern 3 -- choose schema strictness intentionally**
- strict schemas for drift-sensitive internal contracts
- flexible or migration-friendly schemas where callers/providers are still evolving

### Day-to-Day Team Reality
- provider output looks “close enough” but one field changed shape
- eval job starts failing or silently producing wrong results
- validated schemas expose the issue immediately and keep the incident localized

### Common Mistakes
- using raw dicts throughout the service layer
- assuming JSON-looking provider output is safe enough
- treating coercion as validation when stricter guarantees are actually needed
- returning all validation issues as generic 500s
- relying on validation alone as a security model

### Failure Modes
- provider output changes type or omits fields
- hidden schema drift corrupts eval data
- overly strict schemas break normal iteration unnecessarily
- overly loose schemas let bad data through

### Monitoring and Debugging
- count request validation failures separately from provider-output validation failures
- track which fields fail most often for structured model outputs
- sample invalid payloads safely with redaction
- monitor validation-failure spikes after prompt/provider updates

### Security and Reliability Concerns
- validation helps reject malformed input shapes
- length and type constraints reduce abuse opportunities
- validation does not replace auth, policy checks, sanitization, or access control
- be careful logging invalid payloads if they may contain sensitive data

## PART 3 -- ENGINEERING

### API request and response models

```python
from pydantic import BaseModel, Field


class SummarizeRequest(BaseModel):
    document_text: str = Field(min_length=20, max_length=20_000)
    max_bullets: int = Field(default=5, ge=1, le=20)


class SummarizeResponse(BaseModel):
    summary: list[str]
    model_name: str
```

### Internal constrained model

```python
from typing import Literal

from pydantic import BaseModel, Field


class RetrievalContext(BaseModel):
    document_id: str
    source_type: Literal["pdf", "docx", "html", "markdown"]
    chunk_count: int = Field(ge=1)
    tenant_id: str
```

### Structured provider output parsing

```python
import json

from pydantic import BaseModel, Field, ValidationError


class TicketClassification(BaseModel):
    label: str
    confidence: float = Field(ge=0.0, le=1.0)
    reason: str


def parse_llm_classification(raw_text: str) -> TicketClassification:
    text = raw_text.strip()
    if text.startswith("```json"):
        text = text[7:]
    if text.startswith("```"):
        text = text[3:]
    if text.endswith("```"):
        text = text[:-3]
    text = text.strip()

    try:
        data = json.loads(text)
        return TicketClassification.model_validate(data)
    except json.JSONDecodeError as exc:
        raise ValueError(f"provider returned invalid JSON: {exc}") from exc
    except ValidationError as exc:
        raise ValueError(f"provider returned schema-invalid output: {exc}") from exc
```

### Strict validation when coercion is unsafe

```python
from pydantic import BaseModel, StrictFloat


class StrictTicketClassification(BaseModel):
    label: str
    confidence: StrictFloat
    reason: str
```

Use this pattern when you do **not** want `"0.85"` to be silently coerced into `0.85`.

### Strict internal eval contract

```python
from pydantic import BaseModel, ConfigDict, Field


class EvalCase(BaseModel):
    model_config = ConfigDict(extra="forbid")

    question_id: str
    prompt_version: str
    faithfulness: float = Field(ge=0.0, le=1.0)
    relevance: float = Field(ge=0.0, le=1.0)
```

### Custom validator example

```python
from pydantic import BaseModel, Field, field_validator


class RankedResult(BaseModel):
    rank: int = Field(ge=1)
    score: float

    @field_validator("score")
    @classmethod
    def check_score(cls, value: float) -> float:
        if value < 0.0:
            raise ValueError("score must be non-negative")
        return value
```

### FastAPI integration

```python
from fastapi import APIRouter, HTTPException

from app.models.summarize import SummarizeRequest, SummarizeResponse
from app.services.summarizer import summarize_document


router = APIRouter(prefix="/v1/summarize", tags=["summarization"])


@router.post("", response_model=SummarizeResponse)
async def summarize(request: SummarizeRequest) -> SummarizeResponse:
    try:
        return await summarize_document(request)
    except ValueError as exc:
        raise HTTPException(status_code=422, detail=str(exc)) from exc
```

Note:
- bad client request -> `422` is appropriate
- schema-invalid provider output is typically an internal/upstream failure, not a client validation problem

### Wrong Way Example

```text
Bad pattern:
- prompt says "return JSON"
- call json.loads()
- assume the result is safe
- let downstream code discover missing fields or wrong types later
```

## PART 5 -- SENIORITY LAYER

### Seniority Story 1 -- Failure
"A provider output changed shape and our downstream scoring logic started producing wrong results. The real issue was missing output validation. We were blaming model quality for a contract problem."

### Seniority Story 2 -- Design Decision
"We kept internal eval schemas strict because silent drift there was dangerous, but made some client-facing fields optional during active rollout to avoid breaking every caller immediately."

### Tradeoff Language
"Validation strictness is not an ideology. It depends on who owns the contract, how quickly it evolves, and how dangerous silent drift would be if the schema accepted too much."

### Failure Vocabulary
"The incident was a contract-boundary failure, not a hallucination failure. The model output shape changed enough to violate downstream assumptions."

### Interview Question and Strong Answer
**Q: "Why use Pydantic in an AI system if the model already returns structured JSON?"**

Strong answer: "Because provider output is still untrusted application input. JSON mode reduces malformed-output risk, but it does not guarantee the right fields, right types, or safe values. Pydantic gives me a validation boundary where I can reject or categorize bad outputs before they affect storage, scoring, or user-facing responses."
