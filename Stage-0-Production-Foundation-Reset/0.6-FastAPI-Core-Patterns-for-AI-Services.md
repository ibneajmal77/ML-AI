## Key Takeaways -- Read This Before Any Project

1. **FastAPI routes are contract boundaries, not dumping grounds for business logic.**
2. **Thin route, thick service is the default pattern for production AI APIs.**
3. **Typed request and response models are not optional in serious AI services.**
4. **Health and readiness are operational tools, not cosmetic endpoints.**
5. **Async helps when you are waiting on I/O. It does not magically fix bad system boundaries.**
6. **Provider timeouts and downstream failures should be handled deliberately, not left as generic crashes.**

## PART 1 -- CORE CONCEPT

### What It Is
FastAPI is a Python web framework built around typed request/response contracts, validation, async-friendly request handling, and automatic API schema generation. For AI systems, it commonly serves as the entry point to ML inference, LLM features, retrieval workflows, and internal operations.

### Why It Exists
AI systems need a reliable application boundary:
- callers must know what data to send
- invalid requests should fail before they spend compute or tokens
- service logic should be reusable outside the API layer
- responses should be predictable enough for clients, tests, and observability

### Mental Model

```text
Route layer:
"accept, validate, classify errors, call service"

Service layer:
"do the actual AI/ML work"

Response layer:
"return a typed, stable contract"
```

### When To Use It
- when exposing model or LLM logic over HTTP
- when you need typed contracts and predictable responses
- when you want a clean service boundary between callers and AI logic

### When Not To Overdo It
- do not put all business logic in routes
- do not treat every background, batch, or worker operation as a direct request/response API problem
- do not build a large middleware/dependency stack before the service needs it

## PART 2 -- PRODUCTION REALITY

### Real Use Cases
- text classification endpoint
- document summarization endpoint
- RAG question-answering API
- document ingestion trigger endpoint
- admin or evaluation endpoints for internal teams

### Good Patterns

**Pattern 1 -- thin route, thick service**
- route validates request
- route handles status codes and contract boundaries
- service layer owns model/provider/retrieval logic

**Pattern 2 -- explicit request/response schemas**
- routes accept typed request objects
- routes return typed response objects
- clients and tests get stable contracts

**Pattern 3 -- health and readiness split**
- `/health` checks process liveness
- `/ready` checks whether key dependencies are ready enough to serve

**Pattern 4 -- controlled downstream failure handling**
- external provider timeout should not become a random stack trace
- known failure classes should be mapped to useful API behavior

### Day-to-Day Team Reality
- product asks for a new AI feature
- engineers first define the request and response contract
- service code evolves behind that contract
- SRE/platform rely on endpoint health, status codes, and logs during rollout
- support teams often debug via request IDs, timestamps, and error categories, not by reading code

### Common Mistakes
- putting retrieval/model/provider logic directly inside the route
- returning untyped dicts with inconsistent shapes
- exposing internal exceptions directly to clients
- using async syntax without understanding whether downstream calls are actually async
- having no health/readiness endpoints for deployment platforms

### Failure Modes
- provider latency spike causes endpoint timeouts
- response shape drifts and breaks a caller
- route returns `200` with an application-level error embedded in the body
- blocking I/O inside async handlers degrades concurrency
- readiness says “ok” even when required dependencies are unavailable

### Monitoring and Debugging
- log request ID, route name, status code, latency, provider/model when applicable
- track endpoint P50/P95/P99 latency
- separate client errors (`4xx`) from service or dependency failures (`5xx`)
- trace route -> service -> provider/database path for slow requests

### Security and Reliability Concerns
- validate inputs before expensive work starts
- authenticate sensitive endpoints
- limit request sizes for costly AI routes
- return safe error messages to clients, keep internals in logs
- avoid using the user-facing API process for work that belongs in worker systems

## PART 3 -- ENGINEERING

### Typed request and response models

```python
# src/app/models/inference.py
from pydantic import BaseModel, Field


class ClassifyRequest(BaseModel):
    text: str = Field(min_length=1, max_length=5000)


class ClassifyResponse(BaseModel):
    label: str
    confidence: float
    model_name: str
```

### Thin route, service-driven design

```python
# src/app/services/classifier.py
from app.models.inference import ClassifyResponse


async def classify_text(text: str) -> ClassifyResponse:
    label = "billing" if "invoice" in text.lower() else "general"
    confidence = 0.91 if label == "billing" else 0.74
    return ClassifyResponse(
        label=label,
        confidence=confidence,
        model_name="rule-baseline-v1",
    )
```

```python
# src/app/routes/classification.py
from fastapi import APIRouter, HTTPException

from app.models.inference import ClassifyRequest, ClassifyResponse
from app.services.classifier import classify_text


router = APIRouter(prefix="/v1/classify", tags=["classification"])


@router.post("", response_model=ClassifyResponse)
async def classify(request: ClassifyRequest) -> ClassifyResponse:
    try:
        return await classify_text(request.text)
    except ValueError as exc:
        raise HTTPException(status_code=400, detail=str(exc)) from exc
    except Exception as exc:
        raise HTTPException(status_code=500, detail="internal classification error") from exc
```

### Health and readiness endpoints

```python
# src/app/routes/health.py
from fastapi import APIRouter


router = APIRouter(tags=["health"])


@router.get("/health")
async def health() -> dict[str, str]:
    return {"status": "ok"}


@router.get("/ready")
async def ready() -> dict[str, str]:
    return {"status": "ready"}
```

### App wiring

```python
# src/app/main.py
from fastapi import FastAPI

from app.routes.classification import router as classification_router
from app.routes.health import router as health_router


def create_app() -> FastAPI:
    app = FastAPI(title="ai-service")
    app.include_router(health_router)
    app.include_router(classification_router)
    return app


app = create_app()
```

### Timeout wrapper for downstream calls

```python
import asyncio


async def with_timeout(awaitable, timeout_seconds: float):
    return await asyncio.wait_for(awaitable, timeout=timeout_seconds)
```

### Minimal integration test

```python
# tests/integration/test_classification.py
from fastapi.testclient import TestClient

from app.main import app


client = TestClient(app)


def test_classify_route():
    response = client.post("/v1/classify", json={"text": "Where is my invoice?"})
    assert response.status_code == 200
    assert response.json()["label"] == "billing"
```

### Wrong Way Example

```python
# Bad route design:
# - validates input
# - talks to provider
# - reads the database
# - computes business logic
# - formats analytics event
# - writes audit log
# all inside one endpoint function
#
# This is hard to test, hard to reuse, and painful to debug.
```

## PART 5 -- SENIORITY LAYER

### Seniority Story 1 -- Failure
"We had a route calling provider logic directly with no timeout boundary. When the provider slowed down, request latency spiked and the service looked unhealthy. The issue was not just provider slowness. It was that the route owned too much of the downstream execution logic."

### Seniority Story 2 -- Design Decision
"We kept routes thin because the same classification logic had to be reused later from both the API and a background batch job. That reuse would have been painful if the route had become the implementation."

### Tradeoff Language
"The tradeoff is not just speed of implementation versus cleanliness. It is short-term convenience versus long-term operability. A fat route gets you a demo faster. A thin route plus service layer gives you testing, reuse, and clearer failure boundaries."

### Failure Vocabulary
"The incident was caused by a weak service boundary. Route code, downstream dependency handling, and business logic were coupled tightly enough that a provider slowdown became a user-facing API outage."

### Interview Question and Strong Answer
**Q: "How would you structure a FastAPI service for an AI feature?"**

Strong answer: "I treat the route as the contract boundary. It validates input, maps known errors, and returns a typed response. The service layer owns the AI logic so it can be tested and reused outside HTTP. I add health/readiness endpoints for deployment operations and explicit timeout/failure handling around downstream provider calls."
