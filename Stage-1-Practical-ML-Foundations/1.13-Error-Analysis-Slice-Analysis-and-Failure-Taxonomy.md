## CONCEPT 1: Aggregate Metrics Hide Failure Clusters
<!-- cognitive job: establish that overall metrics are often too coarse to guide fixes -->

| Dimension | Answer |
|---|---|
| **What it is** | Aggregate metrics summarize the whole model, but real model pain is often concentrated in specific slices, cohorts, or input types. |
| **Why it matters** | If you only look at overall performance, you can miss the exact segment where the model is failing hardest or hurting the business most. |
| **When to use it** | Use slice analysis whenever one global metric feels inconsistent with user complaints, operational pain, or known business-critical groups. |
| **When NOT to use it** | Do not assume a healthy average means all important subgroups are healthy. |
| **What juniors miss** | They often stop at the global metric and treat it as the final truth instead of a starting summary. |
| **Senior signal** | "The global metric is only the headline; the real diagnosis starts when we ask which slices are carrying the failures." |

### Mental Model

Aggregate metrics are like the average temperature of a building. The average may look fine while one room is freezing and another is overheating. Slice analysis is how you find the room people are actually suffering in.

---

> **Before reading on:** Why can a model with acceptable overall recall still feel broken to a business team?

---

```python
import pandas as pd

df = pd.DataFrame({
    "segment": ["enterprise", "startup", "enterprise", "startup"],
    "correct": [1, 0, 1, 0],
})

overall_accuracy = df["correct"].mean()
by_segment = df.groupby("segment")["correct"].mean()

print(overall_accuracy)
print(by_segment)
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `overall_accuracy = ...` | Computes one global metric. | This is the number teams often stop at too early. |
| `groupby("segment")` | Breaks performance by subgroup. | This is where hidden failure concentration often becomes visible. |
| `by_segment` output | Shows uneven performance behind one average. | Slice analysis reveals whether one cohort is carrying most of the error burden. |
| `segment` as a grouping key | Uses business structure, not just random categories. | Good error analysis groups by real operational meaning. |

**Practice -- apply the concept:**

> A model’s average metric looks stable, but one customer tier complains loudly. What is the first analysis move you should make?

> **Real-world scenario:** A support classifier looked fine overall, but grouping errors by ticket source showed that one integration was feeding a noisier text pattern the model had not learned well. The issue was invisible in the overall average.

## CONCEPT 2: Failure Taxonomy Turns Random Error Examples into Debuggable Patterns
<!-- cognitive job: teach that errors should be organized into reusable categories rather than inspected as isolated anecdotes -->

| Dimension | Answer |
|---|---|
| **What it is** | A failure taxonomy is a structured set of error categories that explains how the model fails, not just how often it fails. |
| **Why it matters** | Without categories, teams collect random bad examples and never turn them into a repeatable debugging workflow. |
| **When to use it** | Use it when reviewing false positives, false negatives, large residuals, or any repeated error pattern that needs systematic improvement. |
| **When NOT to use it** | Do not treat each bad example as unique if multiple examples clearly share a common cause. |
| **What juniors miss** | They often collect error screenshots or rows without labeling the type of failure, so the analysis stays anecdotal. |
| **Senior signal** | "I want a failure taxonomy because a pile of examples is not yet a debugging system." |

### Mental Model

Failure taxonomy is like sorting damaged products into bins: cracked, bent, mislabeled, missing part. Once you sort the failures, the fix path becomes much clearer than if all broken items stay in one pile.

---

> **Before reading on:** If you review 50 mispredictions and never categorize them, what important thing are you likely to miss?

---

```python
errors = pd.DataFrame({
    "example_id": [101, 102, 103],
    "error_type": [
        "missing_context",
        "label_noise",
        "ambiguous_input",
    ],
})

taxonomy_counts = errors["error_type"].value_counts()
print(taxonomy_counts)
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `error_type` column | Assigns a category to each failure. | Categorization is what turns raw examples into a reusable debugging asset. |
| `"missing_context"` / `"label_noise"` | Names likely root causes, not just example IDs. | Good taxonomies describe why the failure happened or what kind of failure it is. |
| `value_counts()` | Counts repeated failure modes. | You can now prioritize fixes by error type frequency or business severity. |
| taxonomy summary | Compresses many examples into a pattern view. | This is how teams avoid chasing noisy individual anecdotes. |

**Practice -- apply the concept:**

> You review 30 false negatives and notice many come from short, ambiguous inputs. What should you do with that observation besides "note it down"?

> **Real-world scenario:** A ticket triage team stopped improving once they only reviewed random mistakes. Progress resumed after they grouped failures into categories such as sparse text, wrong label, edge-case intent, and missing metadata.

## CONCEPT 3: Good Error Analysis Ends in an Action Map, Not Just Better Understanding
<!-- cognitive job: connect observed failures to concrete next interventions -->

| Dimension | Answer |
|---|---|
| **What it is** | Strong error analysis maps each major failure pattern to a likely intervention: better labels, better features, threshold change, more representative data, or model change. |
| **Why it matters** | Understanding failure is useful only if it helps you choose the next intervention more accurately than blind retuning. |
| **When to use it** | Use it after slice analysis and taxonomy work, when you must decide what to change next. |
| **When NOT to use it** | Do not stop at "interesting insight" without asking what concrete change that insight suggests. |
| **What juniors miss** | They often end error analysis with observations rather than decisions. |
| **Senior signal** | "Each failure cluster should point to a likely fix path; otherwise the analysis is informative but not operational." |

### Mental Model

Error analysis should end like triage in an operations room: each issue gets routed to the team or intervention that can actually address it. If everything ends in "maybe tune the model," the analysis was too shallow.

---

> **Before reading on:** If a large share of failures come from missing metadata rather than weak model reasoning, what kind of intervention is probably wrong as the first step?

---

```python
action_map = {
    "label_noise": "review labeling policy or relabel examples",
    "missing_context": "add features or upstream context",
    "rare_slice_failure": "collect more representative data",
    "threshold_error": "adjust operating point, not model family",
}

for failure_type, action in action_map.items():
    print(f"{failure_type} -> {action}")
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `action_map = {...}` | Connects failure type to intervention. | This is the step that turns analysis into engineering progress. |
| `"label_noise"` | Routes some errors away from model tuning. | Not every model failure is a model-capacity problem. |
| `"missing_context"` | Points toward feature or data improvements. | Some model misses come from missing information rather than poor learning. |
| `"threshold_error"` | Recognizes decision-policy failure. | Sometimes the model is fine and only the operating point is wrong. |

**Practice -- apply the concept:**

> A slice analysis shows the model is strong overall but weak on one small but important segment with very different input patterns. Name two possible intervention directions before you default to "bigger model."

> **Real-world scenario:** A model’s worst failures were traced not to algorithm choice but to one segment with stale metadata and weak labels. The winning fix was data cleanup and new features, not another round of boosting or regularization tuning.

## DECISION RULES

### Error Analysis Framework

1. Start with the overall metric, but treat it only as the headline.
2. Slice performance by meaningful business groups, input types, time windows, and other relevant cohorts.
3. Pull real examples from the worst slices.
4. Build a failure taxonomy instead of keeping a loose pile of anecdotes.
5. Count the error types and rank them by frequency and business severity.
6. Map each major failure type to the most likely intervention class.
7. Change one meaningful thing at a time and re-check the same slices.
8. Keep the taxonomy and slice definitions stable enough to compare improvements over time.

### Worked Example

| Step | Customer-support classifier example |
|---|---|
| Overall metric | looks acceptable |
| Slice check | weak recall on enterprise billing tickets |
| Example review | many failures involve short ambiguous text and missing account metadata |
| Taxonomy | sparse text, missing metadata, label ambiguity |
| Action map | enrich metadata, review labels, add text features |
| Re-evaluation | compare the same enterprise billing slice after the fix |

### Red Flags

| What someone says / does | Why it should raise a flag |
|---|---|
| "The average metric is good, so we don’t need to dig deeper." | One high-impact slice may still be failing badly. |
| "We reviewed some bad examples." | Example review without taxonomy often stays anecdotal. |
| "Let’s retune the model first." | That may skip easier fixes in labels, data, features, or thresholds. |
| "This failure type is only qualitative." | If it matters repeatedly, it can usually be categorized and tracked. |

### Use / Do Not Use

| Use this framing when | Do NOT use this framing when |
|---|---|
| you need to turn weak or uneven model behavior into targeted debugging action | you are treating model debugging as only a search for a better global metric |

## COMMON MISTAKES

### Mistake 1: Trusting the global metric too much

**Mistaken instinct:** "The average score is good enough, so the model is good enough."

**Why it feels right:** A clean headline metric feels like the official answer.

**Why it fails:** Important cohorts, inputs, or time windows can still fail badly while the overall average stays acceptable.

**Better move:** Slice performance before declaring the model healthy.

### Mistake 2: Reviewing bad examples without building categories

**Mistaken instinct:** "Reading a sample of failures is already error analysis."

**Why it feels right:** Example review feels concrete and insightful.

**Why it fails:** Without categories, the insights stay scattered and are hard to prioritize or track over time.

**Better move:** Build a small, stable failure taxonomy and count categories.

### Mistake 3: Assuming all failures should be fixed by changing the model

**Mistaken instinct:** "If the prediction is wrong, the algorithm must be the main problem."

**Why it feels right:** The model is the most visible moving part.

**Why it fails:** Many failures are driven by labels, missing context, threshold policy, or weak data slices rather than model family.

**Better move:** Route each failure pattern to the right intervention class.

### Mistake 4: Changing many things before rechecking the same slices

**Mistaken instinct:** "We found several problems, so let’s fix everything together."

**Why it feels right:** It feels efficient and ambitious.

**Why it fails:** You lose the ability to tell which intervention actually improved the known failure cluster.

**Better move:** Preserve the slice/taxonomy view and test changes incrementally.

## PRACTICAL LANDSCAPE

### Technique Landscape

| Concept family | Main technique families | What they are for |
|---|---|---|
| slice analysis | cohort/grouped metrics by tenant, source, segment, time | find where failure concentrates |
| example review | worst cases, false positives, false negatives, large residuals | inspect concrete model mistakes |
| failure taxonomy | category labels for common error types | organize and prioritize debugging |
| intervention routing | labels, features, thresholds, data, model family | map failure patterns to next actions |

### When to Use What

| If your real need is... | Usually start with... | Not with... |
|---|---|---|
| understand who is being hurt most | slice analysis | overall average only |
| understand what kind of mistakes dominate | failure taxonomy | random screenshot review |
| decide what to change next | action map by failure type | generic hyperparameter search |
| validate improvement | re-measure the same slices and categories | only a new global headline metric |

### Coding Surface

| Concept family | Common libraries / tools | Canonical API shape | Typical output |
|---|---|---|---|
| slice metrics | `pandas`, `sklearn.metrics` | `groupby(...).agg(...)` | grouped performance table |
| example extraction | dataframe filtering/sorting | filter worst errors | review dataset |
| taxonomy counts | `value_counts()` | count error labels | ranked failure types |
| action mapping | plain Python / tables | dict or dataframe of category -> next step | prioritized debugging plan |

### Output Shapes to Remember

| Need | Typical output |
|---|---|
| worst cohort view | grouped metric table |
| error review set | dataframe of failed examples |
| taxonomy summary | category counts |
| debugging plan | intervention table by category |
