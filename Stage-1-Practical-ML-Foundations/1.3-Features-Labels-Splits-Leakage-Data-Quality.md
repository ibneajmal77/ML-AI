## Why This Matters

### Business Importance
Data quality and correct evaluation setup determine whether a model that looks good in development actually works in production. More ML projects fail because of leakage, bad splits, or label noise than because of wrong algorithm choice. A model trained on leaked data will pass every offline evaluation and fail the moment it touches real traffic. The business cost of discovering this post-launch is orders of magnitude higher than catching it during data preparation.

### System Importance
The split strategy, feature engineering decisions, and label quality directly determine:
- whether your offline evaluation results are trustworthy
- whether the model generalizes to production inputs
- what your monitoring strategy needs to look for after deployment
- how quickly the model will degrade as the world changes

### Interview Importance
This is one of the highest-signal topics in senior ML interviews. Any experienced interviewer will probe for leakage awareness. They want to know whether you have shipped a model that looked great offline and then failed in production -- or whether you have the discipline to prevent that from happening. Candidates who answer "I check the test set accuracy" without mentioning leakage, temporal splits, or label quality are immediately flagged as lacking production experience.

## Key Takeaways -- Read Before Any Project

1. **Data leakage is the single most dangerous failure mode in supervised ML.** A leaked model will pass every offline test and fail on real traffic. Learn to audit for it before trusting any evaluation result.
2. **The test set is sacred.** It is touched exactly once -- at the very end. If you use it to make any decision during development, it becomes a second validation set and your evaluation is no longer trustworthy.
3. **Random splits are wrong for time-ordered data.** Using random splits on time-series or event-driven data causes severe leakage: the model sees future information during training.
4. **Label quality matters more than model complexity.** A simple model trained on clean, correctly-labeled data consistently outperforms a complex model trained on noisy or misaligned labels.
5. **Fit preprocessing on train only; transform train, val, and test.** Fitting a scaler or encoder on the full dataset before splitting is one of the most common leakage sources and is almost universally taught incorrectly in tutorials.
6. **Missing values in production are a signal, not just noise.** How a value went missing often carries information. Removing or imputing blindly can destroy that signal.
7. **Your split must match how the model will be used.** If production predictions are always made about future events, your evaluation must also be: train on past, evaluate on future.

## PART 1 -- CORE CONCEPT

### Features

A feature is any input variable the model uses to make a prediction.

```text
Predicting customer churn:

  Features (inputs):                Target (output):
  - account_age_days                 churn_in_30_days
  - num_support_tickets_30d          (1 = churned, 0 = retained)
  - last_login_days_ago
  - monthly_spend
  - plan_type
  - num_seats
  - payment_failures_90d
```

**Feature types:**
| Type | Example | Common issues |
|---|---|---|
| Numeric continuous | `monthly_spend`, `age` | Scale differences, outliers, missing values |
| Numeric discrete | `num_seats`, `ticket_count` | Treat as numeric or categorical depending on range |
| Categorical nominal | `plan_type`, `country` | Encoding required; high-cardinality is expensive |
| Categorical ordinal | `satisfaction_score (1-5)` | Encode carefully -- order matters |
| Temporal | `last_login_date`, `account_created_at` | Requires feature extraction; raw timestamps are useless |
| Text | `support_ticket_body` | Requires NLP preprocessing |
| Binary | `is_enterprise`, `has_mfa_enabled` | Often the most reliable features |

**What makes a good feature:**
- It is available at inference time (when you actually need to make the prediction)
- It does not contain information about the future relative to the prediction time
- It is stable (does not have uncontrolled missingness or constant change)
- It has some relationship with the target (verified, not assumed)

**The most important test for any feature:**
*"Will this feature be available in the same form when the model makes a prediction in production?"*
If the answer is no or "maybe," the feature is either a leakage risk or an operational problem.

---

### Labels

A label is the correct answer for each training example -- the value the model is trained to predict.

**Label types:**
| Task | Label example |
|---|---|
| Binary classification | `churned = 1`, `not_churned = 0` |
| Multi-class classification | `ticket_category = "billing"` |
| Regression | `predicted_ltv = 420.50` |
| Ranking | `relevance_score = 3` (scale 1-5) |
| Sequence tagging | `["O", "ORG", "O", "PER"]` per token |

**Label quality is the single most underrated factor in production ML.**

Label problems and their effects:

| Problem | Effect on model | How to detect |
|---|---|---|
| Label noise (random errors) | Adds irreducible error; accuracy ceiling drops | Sample review, inter-annotator agreement |
| Systematic label bias | Model learns biased decision boundary | Disaggregated evaluation by annotator or source |
| Label leakage (labels derived from features) | Inflated metrics that collapse in production | Causal audit of label derivation |
| Inconsistent definitions | Model learns conflicting signal | Check annotation guidelines; audit label distribution over time |
| Stale labels | Model trained on outdated behavior | Check label collection date vs prediction date gap |

**Inter-annotator agreement:**
If multiple humans label the same data and frequently disagree, your label quality ceiling is already low. No model will exceed the consistency of its training labels.

### Train / Val / Test Split

The three partitions serve completely different purposes:

```text
Full Dataset
     |
     +-----------> Training set (60-80%)
     |              Model sees this. Parameters are fitted here.
     |
     +-----------> Validation set (10-20%)
     |              Used during development for:
     |              - hyperparameter tuning
     |              - model selection (which architecture works best)
     |              - early stopping
     |              Cannot be used for final reported performance.
     |
     +-----------> Test set (10-20%)
                   Held out completely.
                   Touched ONCE at the very end.
                   Your honest estimate of production performance.
                   If you look at it to make decisions: it becomes val.
```

**The sacred rule of the test set:**
Every time you use the test set to make a decision (tuning a threshold, selecting a model, fixing a bug you noticed in error analysis), you are implicitly fitting to the test set. The more times you do this, the more optimistic your final metric becomes. The honest answer is: the test set is checked once, after all decisions are locked.

**How to decide split sizes:**
| Dataset size | Typical split |
|---|---|
| < 1,000 examples | 60/20/20 or use cross-validation instead |
| 1,000 - 10,000 | 70/15/15 |
| 10,000 - 100,000 | 75/12.5/12.5 |
| > 100,000 | 80/10/10 or even 90/5/5 -- absolute numbers matter more than percentages |

**Key insight on large datasets:** With 1 million examples, a 1% test set is 10,000 examples -- more than enough. You do not need a 20% test set when 2% gives you statistical reliability.

---

### Stratified Split

For classification problems with imbalanced classes, random splitting can leave you with too few examples of minority classes in val/test sets.

```text
Dataset: 1,000 examples
  900 not_churned (90%)
  100 churned (10%)

Random 80/20 split (unlucky):
  Train: 900 total, 88 churned (9.8%)   <- acceptable
  Test:  100 total, 7 churned (7%)      <- too few to evaluate well

Stratified 80/20 split (guaranteed):
  Train: 800 total, 80 churned (10%)    <- mirrors dataset
  Test:  200 total, 20 churned (10%)    <- same ratio as full dataset
```

Always use stratified splits for classification. The cost is zero and the benefit is consistent evaluation across runs.

---

### Temporal Split -- The One Most Often Done Wrong

**The rule:** For any data where the order of time matters -- financial transactions, user behavior, logs, events -- you must split by time, not randomly.

**Why random splits cause leakage on temporal data:**

```text
Transaction data: Jan 1 through Dec 31

Random split:
  Training set:  Jan 3, Jan 7, Mar 12, Apr 2, Jun 15, Sep 20, Nov 5 ...
  Test set:      Jan 5, Feb 18, Apr 4, Jul 3, Oct 11, Dec 28 ...

Problem: The model sees November data during training,
         then is "tested" on February data.
         It has already seen the future.
         Its patterns include information from after the prediction date.
         This is guaranteed leakage on temporal data.

Correct temporal split:
  Training set:  Jan 1 -- Sep 30
  Validation:    Oct 1 -- Oct 31
  Test set:      Nov 1 -- Dec 31

Now the model can only use past data to predict.
This matches exactly how it will be used in production.
```

**Walk-forward validation for time series:**
For datasets where you want to evaluate across multiple time windows rather than one holdout:

```text
Window 1: Train Jan-Mar, Evaluate Apr
Window 2: Train Jan-Apr, Evaluate May
Window 3: Train Jan-May, Evaluate Jun
...

Average performance across windows = more reliable estimate
than a single temporal holdout.
```

### Data Leakage

**Definition:** Information that would not be available at prediction time leaks into the training features, causing the model to appear far more accurate than it actually is.

**Why it is the most dangerous failure mode:**
- It passes all offline tests with flying colors
- It is invisible until deployment
- The gap between offline and production performance can be enormous (95% → 55%)
- It can take weeks or months to diagnose after launch

#### The Four Most Common Leakage Forms

**1. Target leakage -- a feature encodes the label**

```text
Predicting whether a patient will be diagnosed with diabetes:

Feature included: "prescribed_metformin = True"
                  (metformin is the primary diabetes medication)

The model learns: if prescribed_metformin -> diabetes
This is not prediction. It is tautology.
The prescription happens AFTER (or simultaneously with) the diagnosis.

Rule: Features must represent information available BEFORE the label is determined.
```

**2. Train-test contamination -- preprocessing on the full dataset**

```text
# WRONG (leakage):
scaler.fit(full_dataset)     # sees test data during fit
X_train_scaled = scaler.transform(X_train)
X_test_scaled  = scaler.transform(X_test)

# The scaler's mean and std were computed using test set values.
# Test distribution is now baked into the training preprocessing.

# CORRECT (no leakage):
scaler.fit(X_train)           # sees only training data
X_train_scaled = scaler.transform(X_train)
X_test_scaled  = scaler.transform(X_test)  # transform only, never fit
```

**3. Temporal leakage -- future information in a historical feature**

```text
Predicting loan default at time of application:

Feature: "total_defaults_in_applicant_zip_code_next_12_months"

This feature uses data from after the loan was issued.
It cannot be known at application time.
But it correlates strongly with default -> model looks great.
In production: this feature doesn't exist. Model is useless.

Temporal leakage variants:
- Rolling features computed on the wrong window
  (e.g., 30-day average computed past AND future around event)
- Label derived from activity that happens after the prediction point
- Join between events at prediction time and outcomes from future events
```

**4. Group leakage -- test examples share context with training examples**

```text
Medical imaging: train/test split on scans.
10 scans per patient, 100 patients.
Random split gives the same patient's scans in both train and test.

Model learns patient-specific artifacts (lighting, equipment, positioning)
rather than the actual clinical finding.
Reports 97% accuracy. Works only on patients it has already seen.

Fix: Split by patient ID (or user_id, session_id, document_id)
     so all examples from one entity are in one partition only.
```

### Data Quality at Production Level

**Before training any model, run this audit:**

| Check | What to look for |
|---|---|
| Missing values | Which features? How often? Is missingness random or meaningful? |
| Cardinality | How many unique values per categorical feature? >100 needs special handling |
| Distribution | Are any features heavily skewed? Outliers present? |
| Temporal coverage | Does training data span the same seasons / events as production will? |
| Label distribution | How imbalanced? How was labeling done? When were labels collected? |
| Feature availability at inference | Will all features exist when the model runs in production? |
| Schema stability | Have feature definitions changed over the data history? |
| Leakage audit | Does any feature encode future or target information? |

**The missing value question is not just statistical -- it is causal:**

```text
"last_login_days_ago" is NULL for 15% of users.

What does the NULL mean?
  Option A: System error, random.  -> impute with median, no information lost.
  Option B: User has never logged in after signup.  -> NULL IS the signal.
             Imputing destroys the most predictive information.

Always ask: WHY is this value missing?
            before deciding HOW to handle it.
```

## PART 2 -- PRODUCTION REALITY

### The Production Data Gap

The most common cause of online/offline metric mismatch:

```text
Training data characteristics          Production data characteristics
──────────────────────────────         ────────────────────────────────
Collected 12 months ago                Current
Clean batch export                     Live streaming events
Fully populated features               Frequent missing values
Labels verified by QA                  No labels (ground truth delayed)
Balanced sampling for training         Heavily imbalanced real distribution
Static schema                          Schema evolves with product changes
```

Every one of these gaps is a potential degradation source. Senior engineers explicitly audit these gaps before deployment.

### Design Patterns

**Pattern 1 -- Feature store with point-in-time correctness**
A feature store ensures that when you train on historical data, the feature values used for training are exactly what was available at the time of each historical prediction -- not values that were updated after the fact.

Without this: if customer lifetime value is updated daily and you train on current LTV values for historical transactions, you have temporal leakage baked into every training example.

**Pattern 2 -- Held-out time window as the true test set**
Rather than a random split, always hold out a trailing time window as test. This mirrors the production scenario: model is trained on past, predicts future.

**Pattern 3 -- Separate preprocessing pipeline tracked alongside model**
The scaler, encoder, and imputer fitted on training data must be serialized and deployed alongside the model. If you refit preprocessing in production, the model receives different feature distributions than it was trained on.

**Pattern 4 -- Shadow labeling for production ground truth**
After deployment, you need to collect ground-truth labels for production predictions. This is harder than it sounds. For churn: you need to wait 30 days to know if a customer churned. Plan the ground-truth collection pipeline before you deploy, not after.

### Day-to-Day Team Reality

A data science team delivers a churn model at 87% AUC. The engineering team integrates it. After three weeks in production, the business says churn is not actually going down. Investigation reveals:

1. The training data used `plan_cancel_initiated = True` as a feature -- this event fires at the same time as churn, not before it. **Target leakage.**
2. The scaler was fitted on the full dataset before splitting. The test set performance was artificially inflated. **Train-test contamination.**
3. The model was evaluated with random splits on 18 months of data. The seasonal patterns of Q4 and Q1 appear in both train and test. **Temporal leakage.**

All three problems would have been caught with a 2-hour data audit before training began.

### Common Mistakes

- **Fitting preprocessing before splitting.** The single most common leakage source in tutorial-driven code.
- **Using test set for multiple decisions.** Each look at the test set costs you some of its honesty.
- **Random split on user-behavior data.** Same user's behavior in both train and test is group leakage.
- **Ignoring label quality.** Accepting labels from a third-party annotation service without auditing agreement rates.
- **Treating features as static.** Features computed from aggregate windows (30-day averages) must be computed with the correct historical window, not the current window.
- **Missing value imputation without causation.** Imputing a missing value that was informative (user has never logged in) destroys a predictive signal.

### Failure Modes

| Failure | Root cause | Diagnostic |
|---|---|---|
| Model excellent offline, fails in production | Data leakage (any form) | Audit feature availability, split methodology, preprocessing order |
| Model degrades 3 months post-launch | Concept drift | Monitor input feature distributions against training baseline |
| Model excellent on test, poor on new user segment | Train/test from same distribution; new segment is OOD | Evaluate disaggregated by cohort before launch |
| Model learns sensitive proxy features | Proxy encoding protected attributes | Fairness audit; test for disparate impact by demographic slice |
| Retraining makes model worse | Training data quality degraded; label drift | Audit new labels; compare new training data distribution to baseline |

### Monitoring Considerations

After deployment, production data quality monitoring should track:
- **Feature distribution drift:** mean, variance, and quartile distributions for each feature vs training baseline
- **Missing value rates per feature:** sudden spikes indicate upstream data pipeline issues
- **Prediction score distribution:** if the distribution of model output scores shifts, something upstream has changed
- **Ground-truth label lag:** track how long it takes to collect production ground truth and whether it is actually being collected
- **Schema violations:** features arriving with unexpected types or ranges

### Building a Correct Split Pipeline

```python
import pandas as pd
from sklearn.model_selection import train_test_split
from sklearn.preprocessing import StandardScaler
from sklearn.pipeline import Pipeline


def split_tabular_classification(
    df: pd.DataFrame,
    feature_cols: list[str],
    label_col: str,
    val_size: float = 0.15,
    test_size: float = 0.15,
    random_state: int = 42,
    stratify: bool = True,
) -> dict[str, pd.DataFrame]:
    """
    Produce train/val/test splits with stratification and no leakage.
    Returns dict of DataFrames, not scaled arrays -- scaling happens in the pipeline.
    """
    X = df[feature_cols]
    y = df[label_col]

    stratify_col = y if stratify else None

    # First split: separate test set completely
    X_trainval, X_test, y_trainval, y_test = train_test_split(
        X, y,
        test_size=test_size,
        random_state=random_state,
        stratify=stratify_col,
    )

    # Second split: separate val from train
    val_fraction_of_trainval = val_size / (1 - test_size)
    stratify_trainval = y_trainval if stratify else None

    X_train, X_val, y_train, y_val = train_test_split(
        X_trainval, y_trainval,
        test_size=val_fraction_of_trainval,
        random_state=random_state,
        stratify=stratify_trainval,
    )

    return {
        "X_train": X_train, "y_train": y_train,
        "X_val":   X_val,   "y_val":   y_val,
        "X_test":  X_test,  "y_test":  y_test,
    }

# > Analogy: Test set separation first, then train/val split -- same reason
#   you seal an exam paper before marking it. Once opened, its value changes.

# > Common mistake: Passing the full DataFrame to train_test_split once
#   and getting only train + test. Then using the test set for validation
#   during development. This contaminates your final evaluation.

# > Used in Stage 1.9 (Evaluation Metrics): The test set produced here is
#   what you report on. Every metric in Lesson 1.9 assumes a clean, unseen
#   test set produced by this exact process.
```

### Temporal Split for Event-Driven Data

```python
import pandas as pd


def split_temporal(
    df: pd.DataFrame,
    timestamp_col: str,
    val_cutoff: str,   # ISO date string: "2024-10-01"
    test_cutoff: str,  # ISO date string: "2024-11-01"
) -> dict[str, pd.DataFrame]:
    """
    Split by time. Training = past. Val = recent past. Test = most recent.
    This mirrors production: model is trained on past, predicts future.
    """
    df = df.sort_values(timestamp_col)

    val_date  = pd.Timestamp(val_cutoff)
    test_date = pd.Timestamp(test_cutoff)

    train = df[df[timestamp_col] < val_date]
    val   = df[(df[timestamp_col] >= val_date) & (df[timestamp_col] < test_date)]
    test  = df[df[timestamp_col] >= test_date]

    print(f"Train: {len(train):,} rows | {train[timestamp_col].min()} -> {train[timestamp_col].max()}")
    print(f"Val:   {len(val):,} rows   | {val[timestamp_col].min()} -> {val[timestamp_col].max()}")
    print(f"Test:  {len(test):,} rows  | {test[timestamp_col].min()} -> {test[timestamp_col].max()}")

    return {"train": train, "val": val, "test": test}

# > Analogy: You would not let a stock trader train on 2024 prices and then
#   test on 2022 prices -- that is not how prediction works. Temporal splits
#   enforce the same reality on any event-driven ML system.

# > Common mistake: Using train_test_split (random) on transaction, log, or
#   clickstream data. The sklearn default is random. You must override it.

# > Used in Lesson 1.15 (Offline vs Online Mismatch): Temporal splits are
#   the main mechanism for measuring the offline/online gap before deployment.
```

### Correct Preprocessing Pipeline (No Leakage)

```python
from sklearn.compose import ColumnTransformer
from sklearn.impute import SimpleImputer
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import OneHotEncoder, StandardScaler
from sklearn.linear_model import LogisticRegression


def build_pipeline(
    numeric_features: list[str],
    categorical_features: list[str],
) -> Pipeline:
    """
    All preprocessing is inside the pipeline.
    Pipeline.fit() on train data only -- no leakage possible.
    Pipeline.transform() on val and test data uses train-fitted parameters.
    """
    numeric_transformer = Pipeline([
        ("imputer", SimpleImputer(strategy="median")),    # fit on train only
        ("scaler",  StandardScaler()),                    # fit on train only
    ])

    categorical_transformer = Pipeline([
        ("imputer", SimpleImputer(strategy="most_frequent")),  # fit on train only
        ("encoder", OneHotEncoder(handle_unknown="ignore")),   # fit on train only
    ])

    preprocessor = ColumnTransformer([
        ("num", numeric_transformer,  numeric_features),
        ("cat", categorical_transformer, categorical_features),
    ])

    full_pipeline = Pipeline([
        ("preprocessor", preprocessor),
        ("classifier",   LogisticRegression(max_iter=1000)),
    ])

    return full_pipeline


# Usage -- leakage-free:
pipeline = build_pipeline(numeric_features, categorical_features)
pipeline.fit(X_train, y_train)           # ALL fitting on train only
val_score  = pipeline.score(X_val,  y_val)    # transform only
test_score = pipeline.score(X_test, y_test)   # transform only -- once, at the end

# > Analogy: The sklearn Pipeline is the same as saying "my recipe is sealed --
#   I measure all ingredients using the proportions I learned from the training
#   kitchen, and I never let the test ingredients influence those proportions."

# > Common mistake (extremely common in tutorials):
#   scaler = StandardScaler()
#   X_all_scaled = scaler.fit_transform(X)   # leakage
#   X_train, X_test = train_test_split(X_all_scaled)
#   -- The scaler has already seen the test data. This is wrong.

# > Used in Lesson 1.18 (scikit-learn Pipelines): This pattern becomes the
#   standard for all model training -- preprocessing + model in one serializable unit.
```

### Data Quality Audit Function

```python
import pandas as pd
import numpy as np


def data_quality_report(df: pd.DataFrame, label_col: str) -> None:
    """
    Run before any model training. Surface the issues that will degrade
    model quality or validity if left unaddressed.
    """
    print(f"{'='*60}")
    print(f"DATASET: {len(df):,} rows x {len(df.columns)} columns")
    print(f"{'='*60}\n")

    # Label distribution
    print("LABEL DISTRIBUTION:")
    label_counts = df[label_col].value_counts(normalize=True)
    for label, pct in label_counts.items():
        bar = "█" * int(pct * 40)
        print(f"  {str(label):20s} {pct:.1%}  {bar}")
    print()

    # Missing values
    print("MISSING VALUES (features with any nulls):")
    missing = df.drop(columns=[label_col]).isnull().mean()
    missing_nonzero = missing[missing > 0].sort_values(ascending=False)
    if missing_nonzero.empty:
        print("  None found.")
    else:
        for col, rate in missing_nonzero.items():
            severity = "HIGH" if rate > 0.2 else "MED" if rate > 0.05 else "LOW"
            print(f"  [{severity}] {col}: {rate:.1%} missing")
    print()

    # Cardinality for categorical-like columns
    print("HIGH-CARDINALITY FEATURES (>50 unique values, object dtype):")
    for col in df.select_dtypes(include="object").columns:
        if col == label_col:
            continue
        n_unique = df[col].nunique()
        if n_unique > 50:
            print(f"  {col}: {n_unique:,} unique values -- consider hashing or embedding")
    print()

    # Constant or near-constant features
    print("NEAR-CONSTANT FEATURES (>95% single value -- low predictive value):")
    for col in df.drop(columns=[label_col]).columns:
        top_freq = df[col].value_counts(normalize=True).iloc[0]
        if top_freq > 0.95:
            print(f"  {col}: {top_freq:.1%} is '{df[col].mode()[0]}' -- likely useless")
    print()

# > Analogy: This is the pre-flight checklist. A pilot does not skip it
#   because the flight looks routine. You do not skip it because the data
#   looks clean at first glance.

# > Common mistake: Treating data quality as a post-hoc step to run if
#   model performance is disappointing. Run it first, always, before any
#   model training begins.

# > Used in Project P1: This function (or equivalent) is the first thing
#   you run on your ticket classification or churn dataset.
```

## PART 5 -- SENIORITY LAYER

### Seniority Story 1 -- The Leakage Incident

> "We built a fraud detection model for a payments client. It reached AUC 0.97 in development and the team was elated. I ran a leakage audit the night before the launch review and found two problems. First, we had a feature called `chargeback_filed = True/False` -- a chargeback is what the bank files after fraud is confirmed. It is literally the label, just from a different data source, with a one-day lag. Second, the StandardScaler had been fitted on the full dataset before splitting. We removed the chargeback feature, refitted preprocessing on train only, and reran evaluation. AUC dropped to 0.81. Still good. Still deployable. But if we had shipped the 0.97 model, we would have deployed a model that required future information to work and looked completely broken on day one of production."

**Lesson:** High offline performance is the warning sign, not the success signal. Audit harder when the metrics look too good.

### Seniority Story 2 -- The Label Quality Decision

> "We were building a ticket prioritization model and had 40,000 labeled examples from our CRM. When I sampled 200 labels and compared them, I found that two different support agents labeled the same ticket type differently about 30% of the time. Our best theoretical accuracy was capped at 70% -- no model complexity could overcome inconsistent ground truth. Instead of trying to build a more complex model, we spent two days writing a clearer annotation guide and re-labeling 5,000 of the most ambiguous examples. On the new labels, our simple logistic regression went from 68% to 84% accuracy. Cleaning the labels did 10x more than any model change could have."

**Lesson:** When model accuracy is stuck, audit the labels before tuning the model.

### Tradeoff Language

> "The split strategy is an operational decision, not a statistical one. The split must mirror the deployment scenario. If you will always predict future events from past data, random splits are wrong by definition regardless of how the textbook presents them. The mathematical validity of the evaluation is secondary to whether the evaluation scenario matches the production scenario."

### Failure Vocabulary

| Phrase | Operational meaning |
|---|---|
| "the model saw the future" | Temporal leakage; model used post-prediction-time features during training |
| "evaluation theater" | Excellent test set metrics that do not reflect production performance due to leakage or distribution mismatch |
| "the test set is burned" | Used the test set to make decisions; it is now a second validation set |
| "the labels were the problem" | Performance ceiling imposed by label inconsistency, not model quality |
| "train/test came from the same user" | Group leakage; model memorized user-specific patterns not generalizable to new users |
| "preprocessing contamination" | Scaler/encoder fitted on full dataset, leaking test distribution into training |

### Interview Q&A

**Q: "What is data leakage and how do you detect it?"**

Strong answer:
> "Data leakage is any mechanism by which information that would not be available at prediction time influences the model during training. There are four main forms. Target leakage: a feature directly encodes or is causally downstream of the label -- for example, using 'chargeback filed' as a feature to predict fraud when the chargeback is the fraud confirmation itself. Train-test contamination: preprocessing is fitted on the full dataset before splitting, so the scaler or encoder has seen the test distribution. Temporal leakage: random splits on time-ordered data, where the model sees future events during training. Group leakage: the same user, patient, or document appears in both train and test sets. Detection: run a feature-level causal audit before training; check that preprocessing is inside the pipeline and fitted only on train; ensure splits match the deployment scenario; be suspicious of any feature that dramatically improves performance -- high importance is a leakage indicator, not an all-clear."

---

**Q: "How do you split time-series data correctly?"**

Strong answer:
> "For any time-ordered data, I split chronologically, not randomly. The training set is everything before a cutoff date, the validation set is the next window, and the test set is the trailing window -- always the most recent. This mirrors the production scenario exactly: the model is trained on the past and predicts the future. For a more robust estimate, I use walk-forward validation: train on months 1-3, evaluate on month 4, then train on 1-4 and evaluate on 5, and so on, averaging performance across windows. Random splits on temporal data are leakage: a model trained on November data and tested on March data has already seen future patterns in its training set."

---

**Q: "Your model is stuck at 72% accuracy. What do you try first?"**

Strong answer:
> "Before changing the model, I audit the data. First, I check label quality: sample 100-200 examples and manually verify the labels. If two humans disagree on 25% of them, my accuracy ceiling is already 75% and no model change will help. Second, I check for leakage: are there features that encode or are correlated with the label by construction? Third, I look at the confusion matrix by class and slice: is performance uniformly bad or catastrophically bad on one class or segment? If it's one class, I have an imbalance or label problem for that class specifically. Only after ruling out data problems do I consider model complexity, feature engineering, or hyperparameter tuning."
