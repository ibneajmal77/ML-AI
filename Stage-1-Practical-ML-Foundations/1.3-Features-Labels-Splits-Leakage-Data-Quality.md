## CONCEPT 1: Features and Labels Define the Learning Problem
<!-- cognitive job: anchor the learner on what information goes in, what target comes out, and what makes either one invalid -->

| Dimension | Answer |
|---|---|
| **What it is** | Features are the inputs the model is allowed to use; labels are the correct outputs it is trained to predict. |
| **Why it matters** | If features are unavailable or leaking, or labels are noisy or inconsistent, the model learns the wrong problem no matter how good the algorithm is. |
| **When to use it** | At the start of every supervised ML project, before splitting or training. |
| **When NOT to use it** | Do not finalize features by convenience alone, and do not accept labels just because they exist in a table. |
| **What juniors miss** | The key question is not "does this column correlate with the target?" but "will this feature exist in the same form at prediction time?" |
| **Senior signal** | "I treat feature availability and label quality as deployment questions, not spreadsheet questions." |

### Mental Model

Features are the evidence the model is allowed to inspect. Labels are the answer key you are asking it to reproduce. If either side is wrong, the model becomes a student preparing for the wrong exam with the wrong answer sheet.

### First Concrete Artifact: Building `X` and `y` Correctly

```python
import pandas as pd

df = pd.DataFrame({
    "account_age_days": [100, 220, 35],
    "num_support_tickets_30d": [0, 4, 2],
    "monthly_spend": [120.0, 80.0, 40.0],
    "plan_type": ["pro", "basic", "basic"],
    "account_closed_date": [None, "2025-03-02", None],
    "churned_in_30_days": [0, 1, 0],
})

feature_cols = [
    "account_age_days",
    "num_support_tickets_30d",
    "monthly_spend",
    "plan_type",
]

X = df[feature_cols]
y = df["churned_in_30_days"]
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `df = pd.DataFrame(...)` | Creates raw supervised data with candidate features and a target | Real datasets mix valid features with dangerous ones; selection is part of the ML work. |
| `"account_closed_date"` | Shows a tempting but invalid feature | This column is downstream of churn and would leak future information into training. |
| `feature_cols = [...]` | Explicitly names allowed inputs | Good ML pipelines do not "just use every column except target." Feature choice must be intentional. |
| `X = df[feature_cols]` | Builds the model input matrix | `X` should contain only information available at inference time. |
| `y = df["churned_in_30_days"]` | Builds the target vector | Label definition is part of problem design; if the label is inconsistent or stale, the model ceiling drops. |

### Technique / Method Map

| Technique / Family | What it does | When to use it | Typical tools | Typical output |
|---|---|---|---|---|
| Raw numeric features | Use measured counts, values, or rates directly | When the number already has business meaning | pandas, NumPy | numeric column |
| Encoded categorical features | Convert categories into model-usable form | Plan type, country, channel, device, segment | `OneHotEncoder`, `CatBoost`, target-safe encoders | indicator columns or encoded values |
| Derived temporal features | Turn timestamps into usable signals | When raw dates need recency, age, weekday, seasonality | pandas datetime ops, feature scripts | age/lag/period columns |
| Text features / embeddings | Convert text into representations | Ticket text, notes, reviews, logs | `TfidfVectorizer`, `sentence-transformers`, `transformers` | sparse features or embeddings |
| Binary labels | Define yes/no outcomes | Churn, fraud, spam, approval | pandas, label pipelines | `0/1` target |
| Multi-class labels | Define one-of-many outcomes | Routing, diagnosis, product type | pandas, annotation systems | category target |
| Continuous labels | Define numeric targets | Price, demand, LTV, time to resolution | pandas, warehouse metrics | numeric target |

### When to Pick Which Technique

| If your real need is... | Usually start with... | Not with... |
|---|---|---|
| use tabular account/product data | raw numeric + categorical features | raw IDs without thought |
| use timestamps | derived temporal features | raw timestamp strings |
| use free text | TF-IDF or embeddings | leaving text unused because it is messy |
| predict yes/no | binary label design | forcing it into clustering |
| predict one of many buckets | multi-class label design | many ad hoc binary labels unless needed |
| predict a value | continuous labels | rounding it into arbitrary classes too early |

## CONCEPT 2: The Split Must Mirror Reality
<!-- cognitive job: teach the learner that train/val/test design is not a default API call but an operational simulation of production -->

| Dimension | Answer |
|---|---|
| **What it is** | The train, validation, and test split is the structure that separates learning, tuning, and honest final evaluation. |
| **Why it matters** | If the split does not match production reality, your offline metrics will be optimistic, misleading, or useless. |
| **When to use it** | Before any preprocessing fit or model training. |
| **When NOT to use it** | Do not use a random split by habit when time, groups, or class imbalance change what "fair evaluation" means. |
| **What juniors miss** | The split is not just statistical hygiene; it is a simulation of how the model will actually be used. |
| **Senior signal** | "My split strategy is chosen from the deployment scenario: random, stratified, grouped, or temporal." |

### Mental Model

Your split is a rehearsal. Training is the studying phase, validation is the practice exam, and test is the sealed final exam. If the rehearsal does not match the real performance setting, the score tells you very little.

---

### First Concrete Artifact: Random vs Temporal vs Stratified

```python
from sklearn.model_selection import train_test_split

# Classification with imbalance -> stratify
X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.2, stratify=y, random_state=42
)

# Time-ordered data -> split by time, not randomly
train_df = df[df["event_date"] < "2025-01-01"]
val_df = df[(df["event_date"] >= "2025-01-01") & (df["event_date"] < "2025-02-01")]
test_df = df[df["event_date"] >= "2025-02-01"]
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `train_test_split(... stratify=y ...)` | Preserves class balance across splits | For imbalanced classification, this keeps evaluation stable and comparable. |
| `random_state=42` | Makes the split reproducible | Reproducibility matters when debugging, comparing models, or reviewing experiments. |
| `df["event_date"] < "2025-01-01"` | Creates a past-only training window | This matches the real production condition: predict future cases from past information. |
| `val_df` and `test_df` as later windows | Separates tuning from final evaluation chronologically | Validation helps iterate; test stays the last honest check. |

### Technique / Method Map

| Technique / Family | What it does | When to use it | Typical tools | Typical output |
|---|---|---|---|---|
| Random split | Shuffles and partitions examples | IID-style data with no time or group dependency | `train_test_split` | train/test partitions |
| Stratified split | Preserves class distribution across partitions | Classification with imbalance | `train_test_split(..., stratify=y)` | balanced partitions |
| Group split | Keeps all rows from one entity in one partition | Multiple rows per user, patient, device, account, document | `GroupShuffleSplit`, `GroupKFold` | entity-safe partitions |
| Temporal split | Uses past for training and future for evaluation | Time-series, event logs, transactions, behavioral data | pandas filtering, time-based split logic | chronological partitions |
| Walk-forward validation | Repeats temporal evaluation across rolling windows | Stronger reliability for time-ordered data | custom loops, `TimeSeriesSplit` | multiple temporal scores |

### When to Pick Which Technique

| If your real need is... | Usually start with... | Not with... |
|---|---|---|
| a standard balanced tabular classification task | random or stratified split | temporal split if time is irrelevant |
| a class-imbalanced classifier | stratified split | plain random split by default |
| many rows per person/account/device | group split | row-level random split |
| future prediction from historical events | temporal split | random split |
| a more robust time-based estimate | walk-forward validation | a single random holdout |

**Practice -- apply the concept:**

The model had learned patient-specific artifacts, not disease generalization. Switching to a patient-level group split dropped the score, but the new score finally meant something.

## CONCEPT 3: Leakage and Data Quality Decide Whether the Evaluation Can Be Trusted
<!-- cognitive job: make leakage and quality feel like a pre-training audit system, not a debugging afterthought -->

| Dimension | Answer |
|---|---|
| **What it is** | Leakage is any path by which forbidden information reaches training; data quality is the condition of the features and labels the model learns from. |
| **Why it matters** | Leakage creates fake performance; poor data quality creates hard performance ceilings and unstable behavior. |
| **When to use it** | Before training, during feature review, and whenever results look suspiciously good or unexpectedly bad. |
| **When NOT to use it** | Do not treat leakage audits or data-quality checks as optional cleanup after model training. |
| **What juniors miss** | A high offline score is not proof the setup is good; sometimes it is the strongest warning sign that the setup is bad. |
| **Senior signal** | "I audit leakage before I trust accuracy, and I audit labels before I blame the model." |

### Mental Model

Leakage is like getting tomorrow's exam answers hidden inside today's study guide. Data quality is the condition of the guide itself: missing pages, wrong answers, inconsistent notation. If either one is compromised, the final score stops meaning what you think it means.

---

> **Before reading on:** A model jumps from 0.81 AUC to 0.97 AUC after one new feature is added. Should you celebrate first or audit first?

---

### First Concrete Artifact: Leakage-Proof Preprocessing Pipeline

```python
from sklearn.compose import ColumnTransformer
from sklearn.impute import SimpleImputer
from sklearn.linear_model import LogisticRegression
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import OneHotEncoder, StandardScaler

numeric_features = ["account_age_days", "monthly_spend"]
categorical_features = ["plan_type"]

numeric_transformer = Pipeline([
    ("imputer", SimpleImputer(strategy="median")),
    ("scaler", StandardScaler()),
])

categorical_transformer = Pipeline([
    ("imputer", SimpleImputer(strategy="most_frequent")),
    ("encoder", OneHotEncoder(handle_unknown="ignore")),
])

preprocessor = ColumnTransformer([
    ("num", numeric_transformer, numeric_features),
    ("cat", categorical_transformer, categorical_features),
])

pipeline = Pipeline([
    ("preprocessor", preprocessor),
    ("classifier", LogisticRegression(max_iter=1000)),
])

pipeline.fit(X_train, y_train)    # fit on train only
val_score = pipeline.score(X_val, y_val)
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `SimpleImputer(...)` inside `Pipeline` | Learns imputation values only from training data | Fitting imputers on all data leaks information from validation/test into training. |
| `OneHotEncoder(handle_unknown="ignore")` | Encodes categories safely for unseen values | Production data will contain category drift; this prevents brittle failures. |
| `ColumnTransformer([...])` | Applies separate preprocessing by feature family | Numeric and categorical data need different treatment; combining them carelessly breaks the setup. |
| `pipeline.fit(X_train, y_train)` | Fits preprocessing and model in one sealed train-only unit | This is the safest default pattern for supervised ML in sklearn. |
| `pipeline.score(X_val, y_val)` | Evaluates using train-fitted preprocessing only | Validation now reflects a clean development scenario instead of contaminated preprocessing. |

### Technique / Method Map

| Technique / Family | What it does | When to use it | Typical tools | Typical output |
|---|---|---|---|---|
| Missing-value audit | Measures where and why nulls happen | Before imputation or dropping columns | pandas, profiling scripts | missingness report |
| Imputation | Fills missing values systematically | When missingness is manageable and meaning is understood | `SimpleImputer`, custom logic | completed feature matrix |
| Leakage audit | Checks whether features use future or target info | Before trusting any metric | feature review, SQL lineage, manual checklist | approved / rejected features |
| Label audit | Samples and verifies label consistency | When performance is noisy, capped, or suspicious | manual review, annotation tools | label-quality findings |
| Schema/cardinality audit | Checks data types, uniqueness, stability | Before encoding or deployment | pandas, warehouse queries | feature-risk report |
| Pipeline-based preprocessing | Fits transforms on train only and reuses them safely | Standard supervised training workflow | sklearn `Pipeline`, `ColumnTransformer` | leakage-safe training artifact |

### When to Pick Which Technique

| If your real need is... | Usually start with... | Not with... |
|---|---|---|
| understand nulls | missing-value audit | immediate blind imputation |
| clean train/val/test preprocessing | pipeline-based preprocessing | fitting transforms on the full dataset |
| explain suspiciously strong metrics | leakage audit | immediate celebration |
| explain a low performance ceiling | label audit | only hyperparameter tuning |
| prepare categorical columns for modeling | schema/cardinality audit plus safe encoding | dumping raw IDs directly into the model |

## DECISION RULES

### Trustworthy Dataset Setup Framework

1. Define the prediction point: exactly when will the model make its prediction?
2. Define the label: what outcome are you predicting, and how is it collected?
3. Build candidate features, then reject anything unavailable or invalid at prediction time.
4. Choose the split from production reality: random, stratified, grouped, or temporal.
5. Separate train, validation, and test before fitting any preprocessing.
6. Put preprocessing inside a pipeline so all fitting happens on train only.
7. Run a leakage audit and a data-quality audit before trusting any metric.
8. Touch the test set only once, after decisions are locked.

```text
STEP 1: When does prediction happen?
        |
        v
STEP 2: What is the target at that moment?
        |
        v
STEP 3: Which features exist BEFORE that moment?
        |
        v
STEP 4: What split mirrors production?
        |
        +--> imbalanced classes? -> stratify
        +--> repeated entities?  -> group split
        +--> future prediction?  -> temporal split
        +--> none of the above?  -> random split
        |
        v
STEP 5: Seal test set first
        |
        v
STEP 6: Fit preprocessing on train only
        |
        v
STEP 7: Audit leakage + quality
        |
        v
TRUST METRICS ONLY AFTER THIS
```

### Worked Example

**Scenario:** You want to predict 30-day churn for a SaaS product using account and behavioral data.

| Step | Question | Answer in this case |
|---|---|---|
| 1 | Prediction point? | End of each customer week. |
| 2 | Label? | `churned_in_next_30_days`, verified from account status after that week. |
| 3 | Safe features? | Account age, support tickets in trailing 30 days, spend, plan type, login recency. |
| 4 | Suspicious feature? | `account_closed_date` is invalid because it only exists after churn happens. |
| 5 | Split strategy? | Temporal split, because production predicts future churn from past behavior. |
| 6 | Preprocessing setup? | Imputation, scaling, and encoding inside one sklearn pipeline fitted on train only. |
| 7 | Quality checks? | Missing-value audit, label review, schema check, leakage review. |
| 8 | Final evaluation? | Test set once, after feature and threshold decisions are complete. |

### Red Flags

| What someone says / does | Why it should raise a flag |
|---|---|
| "Let's just use every column except the target" | Some columns may be leakage, IDs, or unavailable at inference time. |
| "I scaled the whole dataset before splitting" | This is train-test contamination. |
| "The same user can appear in both train and test, that's fine" | Group leakage can create fake generalization. |
| "We used a random split on event data from the whole year" | Temporal leakage risk is high. |
| "The model improved massively after adding a post-outcome field" | That is likely target leakage, not genuine improvement. |
| "The labels are whatever the CRM had" | Label definition and consistency have not been audited. |

### When to Use / When Not to Use

| Use this setup choice when | Do NOT use it when |
|---|---|
| `Stratified split` when minority classes must remain represented | when the main issue is time order or entity leakage |
| `Group split` when multiple rows come from one entity | when rows are truly independent and no grouped context exists |
| `Temporal split` when future prediction is the production scenario | when the problem is IID and time has no predictive deployment role |
| `Pipeline-based preprocessing` for nearly all sklearn supervised work | never fit transforms on full data before splitting |
| `Label audit` when the performance ceiling looks suspiciously low or noisy | do not assume more model complexity will fix inconsistent labels |

## COMMON MISTAKES

### Mistake 1: Treating Feature Selection as "Use Every Column"

**Mistaken instinct:** "If a column improves prediction, keep it."

**Why it feels right:** ML feels empirical, so keeping every predictive signal seems rational.

**Why it fails:** Some signals are illegal at prediction time because they use future, downstream, or target-adjacent information. The model then learns a problem it cannot solve in production.

**Better move:** Ask of every feature: is it available at prediction time, in the same form, with the same latency, for every production case?

---

### Mistake 2: Using the Wrong Split Because It Is the Default

**Mistaken instinct:** "I'll just call `train_test_split` and move on."

**Why it feels right:** The API is easy, tutorials use it everywhere, and the split feels like plumbing rather than modeling.

**Why it fails:** The split is part of the evaluation design. Random splitting on temporal or grouped data can create severe leakage and misleading metrics.

**Better move:** Choose the split from the deployment scenario: stratified for imbalanced classification, group-based for repeated entities, temporal for future prediction.

---

### Mistake 3: Fitting Preprocessing Before Splitting

**Mistaken instinct:** "Let me clean and scale the full dataset first so everything is ready."

**Why it feels right:** It feels tidy and efficient to preprocess once up front.

**Why it fails:** The preprocessing step learns from validation/test data and leaks distribution information into training. The evaluation stops being honest.

**Better move:** Put preprocessing inside a pipeline and call `.fit()` only on the training partition.

---

### Mistake 4: Blaming the Model Before Auditing the Labels

**Mistaken instinct:** "Accuracy is stuck, so we need a more powerful model."

**Why it feels right:** Algorithm upgrades are visible and familiar; label review feels slower and less glamorous.

**Why it fails:** If labels are noisy, stale, biased, or inconsistent, model complexity cannot recover the missing truth. You are optimizing against a broken target.

**Better move:** Sample and inspect labels first. If humans disagree heavily, improve the labeling process before changing the algorithm.

## FAST RECALL

### What This Lesson Is Really About

A supervised ML model is only trustworthy when the inputs are valid, the labels are meaningful, the split matches reality, and no forbidden information leaks into training.

### 30-Second Recall

- Features are the evidence the model can use; labels are the answer key.
- A feature is only valid if it exists at prediction time in the same usable form.
- Split strategy must match deployment reality: stratified, group, temporal, or random.
- Leakage usually enters through features, preprocessing, or bad splits.
- Audit data quality and leakage before blaming the model.

### Core Mental Model

```text
safe features + valid labels
            |
            v
correct split for reality
            |
            v
fit preprocessing on train only
            |
            v
audit leakage + quality
            |
            v
trust the metric
```

### Must Know Cold

- Ask of every feature: will it exist at prediction time?
- Test set is touched once, at the end.
- Fit scalers, imputers, and encoders on train only.
- Future prediction problems usually need temporal splits.
- Suspiciously high offline scores should trigger leakage review.
- Weak label quality can cap performance no matter which model you use.

### Code Patterns to Remember

```python
# build supervised inputs
X = df[feature_cols]
y = df[target_col]

# stratified split
train_test_split(X, y, stratify=y)

# temporal split
train_df = df[df["date"] < cutoff]
test_df = df[df["date"] >= cutoff]

# leakage-safe preprocessing
pipeline.fit(X_train, y_train)
pipeline.score(X_val, y_val)
```

### Decision Triggers

- If classes are imbalanced, think `stratified split`.
- If one entity has many rows, think `group split`.
- If production predicts the future, think `temporal split`.
- If one feature causes a huge jump, think `leakage audit`.
- If performance ceiling is low or unstable, think `label audit`.
- If preprocessing was done before splitting, think `contamination`.

### Output Shapes to Remember

| Need | Typical output |
|---|---|
| Feature matrix | `X` table / array |
| Target vector | `y` labels / values |
| Split strategy | train/val/test partitions |
| Leakage-safe pipeline | fitted preprocessing + model artifact |
| Quality audit | report of missingness, labels, schema, risks |

### Confusions to Avoid

| Easy confusion | Correct distinction |
|---|---|
| leakage vs overfitting | leakage = forbidden info; overfitting = memorizing train patterns |
| validation vs test | validation for development decisions; test for final untouched evaluation |
| missing value vs bad value | missingness itself can be signal |
| feature correlation vs valid feature | high correlation does not make a feature safe |
| random split vs correct split | default API choice is not always the right evaluation design |

### Top Traps

- Using every column except the target.
- Scaling or imputing before splitting.
- Random splitting event-driven or time-ordered data.
- Letting the same user/patient/account appear in multiple partitions.
- Tuning the model before checking labels.
- Trusting a surprisingly high metric without auditing leakage.

### One Good Example to Keep

A fraud model looked excellent offline because a feature reflected information created only after fraud confirmation. Removing that leaked feature made the score lower, but honest. The lesson is simple: a lower trustworthy metric is better than a high fake one.

### When to Pick Which Technique

| If your real need is... | Usually start with... | Not with... |
|---|---|---|
| preserve minority classes in evaluation | stratified split | naive random split |
| avoid same-entity contamination | group split | row-level random split |
| predict future outcomes | temporal split | shuffling across the whole history |
| safely scale, impute, and encode | sklearn pipeline | precomputing transforms on full data |
| understand poor model ceiling | label audit | immediate model-family changes |
| understand suspiciously high metrics | leakage audit | immediate celebration |

### Interview Q&A

**Q: What is data leakage and how do you detect it in practice?**

> "Data leakage is any path by which information unavailable at prediction time influences training. The common cases are target leakage through downstream features, preprocessing contamination from fitting on the full dataset, temporal leakage from random splits on time-ordered data, and group leakage where the same entity appears in multiple partitions. In practice, I detect it by reviewing feature lineage, checking whether every feature exists before the prediction point, sealing preprocessing inside a train-only pipeline, and making sure the split mirrors deployment. If a metric looks unbelievably good, I audit harder, not less."

---

**Q: How do you split time-ordered data correctly?**

> "I split chronologically: train on the past, validate on a later window, and test on the most recent holdout. That mirrors production because the model will always predict future outcomes from past information. If I want a more reliable estimate, I use walk-forward validation so I can evaluate across multiple future windows. What I do not do is random shuffle across the entire history, because that lets the model learn future patterns that would not be available at deployment time."

---

**Q: Your model is stuck at 72% accuracy. What do you try first?**

> "Before reaching for a more complex model, I audit the data. I sample labels to check consistency, because noisy labels can cap achievable performance. I review the hardest and most important features for leakage or point-in-time correctness issues. I also verify the split and preprocessing setup, because bad evaluation design can hide the real bottleneck. Only after the data and setup look trustworthy do I spend effort on model family or hyperparameters."
