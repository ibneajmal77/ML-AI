## Why This Matters

### Business Importance
XGBoost and LightGBM win Kaggle competitions and dominate production tabular ML not by accident but because of specific engineering decisions that directly address the limitations of sklearn's gradient boosting implementation: speed, memory efficiency, regularization, and native handling of missing values and categorical features. On a dataset of 500,000 rows and 50 features, sklearn GBM might take 40 minutes to train one model. LightGBM on the same dataset: under 2 minutes. This difference changes what is operationally feasible -- daily retraining, hyperparameter search, cross-validation at scale. Beyond speed, XGBoost and LightGBM have built-in regularization terms, native missing value handling, and categorical feature support that sklearn GBM lacks. These are not convenience features; they are production requirements.

## Key Takeaways -- Read Before Any Project

1. **XGBoost and LightGBM are the production standard for tabular ML.** sklearn GBM is for learning. These two are for production.
2. **LightGBM is typically 5-10x faster than XGBoost with comparable accuracy.** On large datasets this difference is the determining factor. On small datasets the difference is irrelevant.
3. **Always use early stopping.** Set `num_boost_round=1000` and let early stopping decide the actual count. Never pick a fixed number of rounds.
4. **XGBoost requires explicit categorical encoding. LightGBM has native categorical support.** Using LightGBM's native encoding often outperforms manual one-hot encoding on high-cardinality features.
5. **Both handle missing values natively.** Do not impute before feeding to these models unless you have a specific reason. They learn the optimal direction for missing values during training.
6. **On structured tabular data, tree-based models consistently beat deep learning.** This is an empirical fact that has held across benchmarks. The reason is structural: real-world tabular data has irregular patterns, mixed types, and feature interactions that trees handle naturally and that NNs require extensive architecture engineering to match.
7. **`num_leaves` in LightGBM is the primary capacity control, not `max_depth`.** LightGBM grows leaf-wise (finding the best leaf to split globally) rather than level-wise (splitting all nodes at the same depth). `num_leaves` controls complexity more directly than depth in this growth strategy.

### Why sklearn GBM Is Not Enough for Production

On a dataset with 500,000 rows and 50 features, 100 rounds:

### XGBoost: The Regularized Gradient Boosting Standard

XGBoost (Extreme Gradient Boosting) improves on standard GBM in three key ways:

**1. Built-in L1 and L2 regularization on tree weights:**

```text
XGBoost loss = sum of errors + L1 penalty on leaf weights + L2 penalty on leaf weights

This directly prevents the model from placing extreme weights in any leaf.
In sklearn GBM, regularization is indirect (min_samples_leaf, max_depth only).
XGBoost's explicit weight regularization is more precise and tunable.
```

**2. Approximate split finding (faster on large datasets):**

Instead of evaluating every possible split point, XGBoost uses quantile sketching to find approximate optimal splits. The result is nearly identical accuracy with substantially lower compute.

**3. Native missing value handling:**

```text
During training, for each feature with missing values, XGBoost learns
whether examples with missing values should go LEFT or RIGHT at each split.
It tries both directions and picks the one that reduces the loss more.

This means:
- NaN values in features are not a problem
- The model learns to use missingness as a signal if it is one
- Do NOT impute before passing to XGBoost unless you have a specific reason
```

**4. Second-order gradient information:**

Standard GBM uses only the first derivative of the loss function. XGBoost uses both first and second derivatives (the Hessian). This more precise gradient information leads to better split decisions and faster convergence.

---

### LightGBM: The Speed-First Gradient Boosting Library

LightGBM's primary innovation is the histogram-based approach and leaf-wise tree growth.

**Histogram-based splitting:**

```text
Standard GBM: for each feature, sort all values and try every threshold.
  Feature with 1M unique values -> 1M split candidates to evaluate.

LightGBM: bin each feature into 256 discrete buckets (histograms).
  Feature with 1M unique values -> 255 split candidates to evaluate.
  Memory: 1M floats -> 255 integers per feature.
  Speed: 4000x fewer split candidates.
  Accuracy loss: minimal (most information is preserved in 256 bins).
```

**Leaf-wise vs level-wise tree growth:**

```text
Level-wise (sklearn, XGBoost):
  Grow all nodes at depth 1 -> then all at depth 2 -> etc.
  Same max_depth across all branches.
  More symmetric trees.

         ┌──────┐
    ┌────┤ Root ├────┐
    │    └──────┘    │
  ┌─┴─┐           ┌─┴─┐
  │ L │           │ R │    <- all depth-1 nodes split together
  └───┘           └───┘

Leaf-wise (LightGBM):
  Find the leaf with highest loss reduction globally and split it.
  Other leaves may not be split at this round.
  Can create asymmetric, deep branches where they help most.

         ┌──────┐
    ┌────┤ Root ├────┐
    │    └──────┘    │
  ┌─┴─┐             │     <- left leaf split (high gain)
  │   ├──┐          │
  │   │  │          └─    <- right leaf NOT split (low gain)
  └───┘  └─

Result: leaf-wise trees find the highest-loss regions first.
More efficient use of tree capacity -> better accuracy per tree.
Risk: more prone to overfit on small datasets -> control with num_leaves.
```

**Native categorical feature support:**

```text
Standard approach (required for XGBoost):
  Plan type: ["basic", "pro", "enterprise"]
  -> One-hot encode -> three binary columns
  -> 100-category feature -> 100 binary columns (memory explosion)

LightGBM native:
  Plan type: ["basic", "pro", "enterprise"]
  -> LightGBM groups categories by target mean, finds optimal binary splits
  -> 100-category feature -> still one column
  -> No one-hot encoding needed
  -> Often outperforms one-hot encoding on high-cardinality features
```

### Why Trees Beat Neural Networks on Tabular Data

This is one of the most important senior-level framing questions in the field.

```text
"Why not just use a neural network for tabular data?"
```

**The empirical answer:** every major benchmark comparing gradient boosting to deep learning on tabular structured data shows that tree-based models match or outperform neural networks in the majority of settings. A 2022 paper surveyed 45 real-world tabular datasets -- XGBoost/LightGBM won or tied in 80% of cases.

**The structural reasons:**

**1. Tabular data has irregular patterns, not smooth manifolds.**

Neural networks excel on data where nearby inputs have similar outputs (images, audio, text) -- the data lives on a smooth manifold. Tabular data often has sharp, irregular decision boundaries: `if (customer_tier == "enterprise") AND (tenure > 24 months): low churn` regardless of what other features look like. Trees find these exact boundaries naturally. NNs must learn to approximate them with smooth functions, which is inefficient.

**2. Feature scales and types are mixed.**

Tabular data mixes binary flags, counts, ratios, categorical strings, and continuous values. NNs are sensitive to scale and require careful preprocessing for each type. Trees are completely scale-invariant and handle mixed types natively.

**3. Missing values and informative sparsity.**

Real tabular data has meaningful missing values. `no_login_in_90d` (NULL) means something fundamentally different from `last_login = 85 days ago`. Trees handle this natively. NNs require explicit imputation.

**4. Small datasets.**

Most business tabular problems have thousands to hundreds of thousands of examples, not millions. Neural networks need large datasets to generalize well from their overparameterized structure. Tree models with explicit regularization generalize efficiently from smaller datasets.

**5. Feature interactions are the signal.**

Business tabular data has critical feature interactions: `low_usage AND high_support_tickets -> churn`. Trees discover these automatically. NNs require explicit architecture decisions (cross-product layers, attention over features) to find them.

```text
WHEN NEURAL NETWORKS DO BEAT TREES ON TABULAR DATA:
  - Very large datasets (millions of rows) with complex continuous patterns
  - Data with spatial or temporal structure (images embedded in tables, time-series)
  - High-dimensional sparse inputs (raw text bags, very large one-hot encodings)
  - When you can leverage pretrained representations from deep learning

WHEN TREES WIN (most production tabular ML):
  - Mixed feature types
  - Meaningful missing values
  - Sharp non-linear boundaries
  - Moderate dataset sizes (10k - 1M rows)
  - Feature interactions are the primary signal
  - Speed and latency requirements
```

## PART 2 -- PRODUCTION REALITY

### XGBoost vs LightGBM: The Production Decision

| Factor | Prefer XGBoost | Prefer LightGBM |
|---|---|---|
| Dataset size | < 100k rows (speed not critical) | > 100k rows (speed matters significantly) |
| Training time budget | Not a constraint | Tight (daily retraining, rapid iteration) |
| Categorical features | Many, handle with encoding | Many, use native categorical support |
| Ecosystem maturity | Slightly more mature, wider integrations | Nearly equivalent now |
| Memory constraints | Less critical | LGB uses ~3x less memory |
| Default safe behavior | Slightly more conservative | Leaf-wise needs `num_leaves` attention |

**In most production settings: start with LightGBM.** Speed and native categorical support make it the default choice. Switch to XGBoost when you need more mature ecosystem integrations or are dealing with a specific edge case.

### Design Patterns

**Pattern 1 -- Production training with early stopping and eval set**

Always pass an evaluation set and use early stopping. Never commit to a fixed number of rounds. The early stopping round count is the model's true capacity for this dataset.

**Pattern 2 -- Parameter grid by dataset size**

Use principled starting parameters based on dataset size rather than random search:

```text
Small dataset (< 10k rows):
  learning_rate: 0.05
  num_leaves: 15        (LGB) / max_depth: 3 (XGB)
  min_child_weight: 20  (more regularization)
  early_stopping: 50 rounds

Medium dataset (10k - 200k rows):
  learning_rate: 0.05
  num_leaves: 31        (LGB default) / max_depth: 4-5 (XGB)
  min_child_weight: 10
  early_stopping: 50 rounds

Large dataset (> 200k rows):
  learning_rate: 0.1    (can afford faster steps)
  num_leaves: 63-127    (LGB) / max_depth: 6 (XGB)
  min_child_weight: 5
  early_stopping: 100 rounds
```

**Pattern 3 -- Native missing value passthrough**

Do not impute before passing to XGBoost or LightGBM. Pass NaN values directly. The models learn optimal missing value directions during training. Imputing before training destroys information and adds a preprocessing step that must also be applied identically at inference time.

**Pattern 4 -- Model artifact tracking**

Always save both the model and the feature list used to train it. At inference time, verify the incoming features match the training feature list exactly. Feature mismatch between training and serving is a silent production failure.

### Day-to-Day Team Reality

An ML engineer is building a daily churn prediction pipeline that needs to retrain on 800,000 rows of behavioral data each night within a 30-minute compute window. They prototype with sklearn GBM. A single training run takes 55 minutes -- already over budget before any hyperparameter search. They switch to LightGBM. Single run: 4 minutes. Hyperparameter search with 20 combinations: 80 minutes. They schedule a weekly search run and a nightly fixed-parameter retrain. Both fit within SLA. The model quality is indistinguishable from sklearn GBM and actually slightly better because they can now afford 5-fold cross-validation to get more stable AUC estimates.

The library choice was an operational decision, not a quality decision.

### Common Mistakes

- **Not using early stopping.** The most common mistake. Fixed `n_estimators=100` either undertrains or overtrains. Early stopping is non-negotiable.
- **Setting `num_leaves` too high in LightGBM without compensating regularization.** LightGBM's leaf-wise growth can overfit aggressively with high `num_leaves` on small datasets. Start with default 31; increase only if the data justifies it.
- **One-hot encoding categoricals before LightGBM.** LightGBM's native categorical encoding often outperforms one-hot. Always try native first.
- **Not passing NaN values natively.** Imputing before XGBoost/LightGBM is usually wrong. These models handle missing values better than any imputation strategy.
- **Training and inference preprocessing mismatch.** If you apply any transformation (scaling, encoding) before the model, that transformation must be serialized and applied identically at inference time.
- **Forgetting `verbose=-1` in production training.** Both libraries print extensive training logs by default. Production training jobs need silent mode or structured logging.

### Failure Modes

| Failure | Root cause | Fix |
|---|---|---|
| Model degrades immediately after launch | Training/serving feature mismatch | Log and validate feature names at inference |
| Very long training time at retraining | Not using LightGBM for large datasets; no GPU | Switch library; enable GPU |
| Overfitting on small dataset with LightGBM | `num_leaves` too high | Reduce `num_leaves`; increase `min_data_in_leaf` |
| Categorical features producing worse AUC than one-hot | Wrong dtype passed to LightGBM native | Explicitly cast categorical columns to pandas `category` dtype |
| Missing values cause error at inference | NaN handling different between training and serving | Use model's native NaN handling; never impute before model |
| Memory error on large dataset | XGBoost on very large data | Switch to LightGBM or use `tree_method='hist'` in XGBoost |

## PART 3 -- ENGINEERING

### Production LightGBM Training Pattern

```python
import lightgbm as lgb
import numpy as np
import pandas as pd
from sklearn.metrics import roc_auc_score


def train_lightgbm(
    X_train: pd.DataFrame,
    y_train: pd.Series,
    X_val: pd.DataFrame,
    y_val: pd.Series,
    categorical_features: list[str] | None = None,
    dataset_size: str = "medium",   # "small" | "medium" | "large"
) -> lgb.Booster:
    """
    Production LightGBM training pattern with early stopping.
    categorical_features: list of column names that are categorical.
    LightGBM handles them natively -- do NOT one-hot encode these first.
    """
    # Cast categoricals to the correct dtype so LightGBM recognizes them
    X_train = X_train.copy()
    X_val   = X_val.copy()
    if categorical_features:
        for col in categorical_features:
            X_train[col] = X_train[col].astype("category")
            X_val[col]   = X_val[col].astype("category")

    train_data = lgb.Dataset(X_train, label=y_train,
                              categorical_feature=categorical_features or "auto",
                              free_raw_data=False)
    val_data   = lgb.Dataset(X_val, label=y_val, reference=train_data,
                              free_raw_data=False)

    # Size-guided starting parameters
    size_params = {
        "small":  {"num_leaves": 15, "min_data_in_leaf": 30, "learning_rate": 0.05},
        "medium": {"num_leaves": 31, "min_data_in_leaf": 20, "learning_rate": 0.05},
        "large":  {"num_leaves": 63, "min_data_in_leaf": 10, "learning_rate": 0.1},
    }
    params = {
        "objective":       "binary",
        "metric":          "auc",
        "boosting_type":   "gbdt",
        "feature_fraction": 0.8,        # sample 80% of features per tree
        "bagging_fraction": 0.8,        # sample 80% of rows per tree
        "bagging_freq":    5,
        "lambda_l1":       0.1,         # L1 regularization
        "lambda_l2":       1.0,         # L2 regularization
        "verbose":         -1,          # silent -- required for production pipelines
        "seed":            42,
        **size_params[dataset_size],
    }

    callbacks = [
        lgb.early_stopping(stopping_rounds=50, verbose=True),
        lgb.log_evaluation(period=100),   # print every 100 rounds only
    ]

    model = lgb.train(
        params,
        train_data,
        num_boost_round=2000,      # upper bound -- early stopping reduces this
        valid_sets=[val_data],
        callbacks=callbacks,
    )

    val_preds = model.predict(X_val)
    val_auc   = roc_auc_score(y_val, val_preds)
    train_preds = model.predict(X_train)
    train_auc   = roc_auc_score(y_train, train_preds)

    print(f"\nLightGBM training complete:")
    print(f"  Best round:     {model.best_iteration}")
    print(f"  Train AUC:      {train_auc:.4f}")
    print(f"  Val AUC:        {val_auc:.4f}")
    print(f"  Train-val gap:  {train_auc - val_auc:.4f}")
    if train_auc - val_auc > 0.08:
        print("  WARNING: significant overfit -- reduce num_leaves or increase regularization")

    return model

# > Analogy: Setting num_boost_round=2000 with early_stopping(50) is like
#   telling a runner "run as far as you can, but stop if you haven't
#   improved your pace in the last 50 metres." The early stop is the
#   actual finish line, not the 2000 mark.

# > Common mistake: Setting num_boost_round=100 as a fixed number.
#   This may stop the model before convergence (underfit) or may be the
#   right number by coincidence, but you have no way to know.
#   Always let early stopping decide.

# > Used in Project P1: This function is your primary model training call.
#   Replace the sklearn GBM from the Lesson 1.7 three-model protocol with
#   this pattern for production-quality results.
```

### Production XGBoost Training Pattern

```python
import xgboost as xgb
import numpy as np
import pandas as pd
from sklearn.metrics import roc_auc_score
from sklearn.preprocessing import OrdinalEncoder


def train_xgboost(
    X_train: pd.DataFrame,
    y_train: pd.Series,
    X_val: pd.DataFrame,
    y_val: pd.Series,
    categorical_features: list[str] | None = None,
) -> xgb.Booster:
    """
    Production XGBoost training pattern.
    XGBoost does NOT have native categorical support in the same way as LightGBM.
    Use enable_categorical=True with pandas category dtype (XGBoost >= 1.6).
    For older versions or complex categoricals: use ordinal encoding.
    NaN values are handled natively -- do NOT impute.
    """
    X_train = X_train.copy()
    X_val   = X_val.copy()

    if categorical_features:
        # XGBoost 1.6+ native categorical (still experimental)
        for col in categorical_features:
            X_train[col] = X_train[col].astype("category")
            X_val[col]   = X_val[col].astype("category")

    dtrain = xgb.DMatrix(X_train, label=y_train,
                          enable_categorical=bool(categorical_features))
    dval   = xgb.DMatrix(X_val,   label=y_val,
                          enable_categorical=bool(categorical_features))

    params = {
        "objective":        "binary:logistic",
        "eval_metric":      "auc",
        "eta":              0.05,           # learning rate
        "max_depth":        5,
        "min_child_weight": 10,             # min sum of instance weights in leaf
        "subsample":        0.8,            # row sampling
        "colsample_bytree": 0.8,            # feature sampling
        "reg_alpha":        0.1,            # L1
        "reg_lambda":       1.0,            # L2
        "tree_method":      "hist",         # histogram-based (faster; use for large data)
        "seed":             42,
        "verbosity":        0,              # silent
    }

    evals = [(dtrain, "train"), (dval, "val")]
    evals_result = {}

    model = xgb.train(
        params,
        dtrain,
        num_boost_round=2000,
        evals=evals,
        early_stopping_rounds=50,
        evals_result=evals_result,
        verbose_eval=100,
    )

    val_preds   = model.predict(dval)
    train_preds = model.predict(dtrain)
    val_auc     = roc_auc_score(y_val, val_preds)
    train_auc   = roc_auc_score(y_train, train_preds)

    print(f"\nXGBoost training complete:")
    print(f"  Best round:     {model.best_iteration}")
    print(f"  Train AUC:      {train_auc:.4f}")
    print(f"  Val AUC:        {val_auc:.4f}")
    print(f"  Train-val gap:  {train_auc - val_auc:.4f}")

    return model

# > Analogy: XGBoost's explicit L1 and L2 parameters (reg_alpha, reg_lambda)
#   are the shock absorbers of the model. Without them the car (model) handles
#   well on the road it was tested on but vibrates on any new surface.

# > Common mistake: Using xgb.XGBClassifier (sklearn API) with default
#   n_estimators=100 and no early_stopping_rounds. This is the single most
#   common way to train an undertrained or overtrained XGBoost model.
#   Always use the native API or set early_stopping_rounds explicitly.

# > Used in Lesson 1.12 (Overfitting): The reg_alpha and reg_lambda parameters
#   introduced here are the XGBoost-specific tools in the overfitting toolkit.
```

### Feature Importance and SHAP Values

```python
import lightgbm as lgb
import numpy as np
import pandas as pd


def feature_analysis(
    model: lgb.Booster,
    X_val: pd.DataFrame,
    feature_names: list[str],
    top_n: int = 15,
) -> pd.DataFrame:
    """
    Two levels of feature analysis:
    1. Built-in importance (split count or gain) -- fast, direction-less
    2. SHAP values -- slower, shows direction and magnitude per prediction

    Use built-in importance for quick audits.
    Use SHAP for explanation and debugging.
    """
    # Built-in importance
    importance_split = model.feature_importance(importance_type="split")
    importance_gain  = model.feature_importance(importance_type="gain")

    df = pd.DataFrame({
        "feature":    feature_names,
        "split_count": importance_split,
        "gain":        importance_gain,
    })
    df["gain_normalized"] = df["gain"] / df["gain"].sum()
    df = df.sort_values("gain_normalized", ascending=False)

    print(f"\nTOP {top_n} FEATURES BY GAIN:")
    print(f"{'Feature':30s} {'Gain %':>8} {'Splits':>8}")
    print("-" * 55)
    for _, row in df.head(top_n).iterrows():
        bar = "█" * int(row["gain_normalized"] * 50)
        flag = "  <- AUDIT" if row["gain_normalized"] > 0.25 else ""
        print(f"  {row['feature']:28s} {row['gain_normalized']:>7.1%} {row['split_count']:>7}{flag}")

    # SHAP values (requires shap package: pip install shap)
    try:
        import shap
        explainer = shap.TreeExplainer(model)
        shap_values = explainer.shap_values(X_val.head(1000))  # sample for speed

        mean_abs_shap = np.abs(shap_values).mean(axis=0)
        shap_df = pd.DataFrame({
            "feature": feature_names,
            "mean_abs_shap": mean_abs_shap,
        }).sort_values("mean_abs_shap", ascending=False)

        print(f"\nTOP {top_n} FEATURES BY MEAN |SHAP| (direction-aware importance):")
        for _, row in shap_df.head(top_n).iterrows():
            print(f"  {row['feature']:30s} {row['mean_abs_shap']:.4f}")

    except ImportError:
        print("\n(SHAP not installed -- pip install shap for direction-aware importance)")

    return df

# > Analogy: Built-in feature importance is like knowing how often each witness
#   testified in a court case. SHAP values tell you whether each witness
#   argued for or against the verdict, and by how much, for each specific case.
#   Both are useful. Built-in is fast. SHAP is precise.

# > Common mistake: Using 'split' importance instead of 'gain' importance.
#   A feature that appears in 500 splits of 1-sample leaves contributes less
#   than a feature that appears in 10 splits of 50,000-sample leaves.
#   'gain' measures actual loss reduction -- the correct measure.

# > Used in Lesson 1.14 (Error Analysis): SHAP values on the misclassified
#   subset tell you which features were most responsible for wrong predictions.
#   This is the production debugging workflow for gradient boosting models.
```

### Handling Class Imbalance in XGBoost/LightGBM

```python
import lightgbm as lgb
import numpy as np


def compute_class_weight_params(y_train: np.ndarray) -> dict:
    """
    Both libraries support class imbalance natively via scale_pos_weight (XGB)
    or is_unbalance / class_weight (LGB).
    Compute the correct values from the training labels.
    """
    n_negative = (y_train == 0).sum()
    n_positive = (y_train == 1).sum()
    scale_pos_weight = n_negative / n_positive

    print(f"Class distribution:")
    print(f"  Negative: {n_negative:,}  ({n_negative / len(y_train):.1%})")
    print(f"  Positive: {n_positive:,}  ({n_positive / len(y_train):.1%})")
    print(f"  scale_pos_weight (XGBoost): {scale_pos_weight:.1f}")
    print(f"  Equivalent LightGBM: set 'is_unbalance': True OR")
    print(f"                            'class_weight': 'balanced'")

    return {
        "xgboost": {"scale_pos_weight": scale_pos_weight},
        "lightgbm_is_unbalance": {"is_unbalance": True},
        "lightgbm_weight": {"class_weight": "balanced"},
    }

# NOTE: These params tell the model to weight positive examples more heavily
# during training. This shifts precision/recall in the same direction as
# lowering the threshold. They interact with your threshold choice.
# After using them, re-evaluate your precision/recall operating point.

# > Analogy: scale_pos_weight is like hiring more fraud investigators
#   for rare fraud cases: you're telling the model "pay 20x more attention
#   to fraud examples because you see 20 legitimate ones for each fraud one."

# > Common mistake: Using scale_pos_weight AND also setting a low threshold.
#   Both push toward higher recall. Using both together can cause precision
#   to collapse. Set one primary imbalance mechanism and tune the threshold
#   on top of it. Don't stack imbalance corrections.

# > Used in Lesson 1.10 (Class Imbalance): This is one of three techniques
#   covered in depth there (along with resampling and threshold tuning).
#   Lesson 1.10 explains when each is most appropriate.
```

### Model Persistence and Feature Validation

```python
import json
import lightgbm as lgb
import pandas as pd
from pathlib import Path


def save_model_with_metadata(
    model: lgb.Booster,
    feature_names: list[str],
    val_auc: float,
    model_dir: str = "models",
) -> None:
    """
    Save model artifact + feature list + evaluation metrics together.
    The feature list is critical: at inference time, verify that incoming
    features match the training feature list exactly.
    """
    path = Path(model_dir)
    path.mkdir(exist_ok=True)

    model.save_model(str(path / "model.lgb"))

    metadata = {
        "feature_names": feature_names,
        "n_features": len(feature_names),
        "best_iteration": model.best_iteration,
        "val_auc": round(val_auc, 5),
    }
    with open(path / "metadata.json", "w") as f:
        json.dump(metadata, f, indent=2)

    print(f"Model saved to {model_dir}/")
    print(f"  model.lgb         <- model artifact")
    print(f"  metadata.json     <- feature list + evaluation metrics")


def load_and_validate(
    model_dir: str,
    X_inference: pd.DataFrame,
) -> tuple[lgb.Booster, pd.DataFrame]:
    """
    Load model and validate that inference features match training features.
    Feature mismatch is a silent production failure -- this catches it explicitly.
    """
    path = Path(model_dir)
    model = lgb.Booster(model_file=str(path / "model.lgb"))

    with open(path / "metadata.json") as f:
        metadata = json.load(f)

    trained_features = metadata["feature_names"]
    incoming_features = list(X_inference.columns)

    missing  = set(trained_features) - set(incoming_features)
    extra    = set(incoming_features) - set(trained_features)

    if missing:
        raise ValueError(f"Features missing from inference data: {missing}")
    if extra:
        print(f"WARNING: extra features in inference data (will be ignored): {extra}")

    # Reorder to match training order exactly
    X_aligned = X_inference[trained_features]
    return model, X_aligned

# > Analogy: Saving the feature list alongside the model is like keeping the
#   instruction manual with the machine. Without it, the next engineer
#   (or the inference server) has no way to verify they're feeding the
#   machine the right inputs in the right order.

# > Common mistake: Saving only the model artifact without the feature list.
#   LightGBM will silently use features in the order they appear in the
#   inference DataFrame, which may differ from training order. This produces
#   wrong predictions with no error.

# > Used in Lesson 1.19 (Model Packaging): Feature validation is the first
#   step in the model serving health check. A production inference service
#   runs this validation on every request batch, not just at deployment.
```

## PART 5 -- SENIORITY LAYER

### Seniority Story 1 -- The Categorical Encoding Trap

> "We had a customer segmentation feature with 340 unique values -- company industry codes. One-hot encoding produced 340 binary columns. Our XGBoost model used these heavily but the one-hot vectors were sparse and the model had to discover the clustering of similar industries through 340 separate features. We switched to LightGBM with native categorical support for that single column. LightGBM internally groups the industry codes by their relationship with the target -- creating binary splits that capture 'tech vs non-tech' or 'financial services vs everything else' without us having to define those groups manually. Val AUC improved by 0.02 and training time dropped because we removed 340 columns. The model was also easier to interpret: feature importance now showed 'industry_code' as a single feature rather than scattered across 340 binary columns. One dtype change -- `df['industry_code'] = df['industry_code'].astype('category')` -- produced better results than three days of manual category grouping engineering."

**Lesson:** Native categorical encoding in LightGBM is not a convenience feature -- it is often a better algorithm than manual one-hot encoding for high-cardinality features.

### Seniority Story 2 -- The Missing Value Imputation Mistake

> "A new ML engineer on our team was preprocessing churn data before LightGBM training. They imputed `last_login_days_ago` with the median (85 days) for users who had never logged in at all. Their reasoning was sound from a classical statistics perspective: missing values should be imputed before modeling. The problem: users who had never logged in after signup were the most likely to churn, and their NULL value was the exact signal. The median imputation assigned them an 85-day gap -- indistinguishable from active users who hadn't logged in recently. The model lost one of its most predictive signals entirely. After removing the imputation and passing NaN directly to LightGBM, the model learned a large negative weight for the native-missing direction of that feature. Val AUC went from 0.81 to 0.86. The lesson was: for tree-based models, especially LightGBM and XGBoost, NaN is data. Imputing before the model is treating a signal as noise."

**Lesson:** The question for missing values is not "how should I fill this?" but "what does this missing value mean, and is the model better equipped than I am to learn from it?"

### Tradeoff Language

> "The choice between XGBoost and LightGBM is primarily operational, not quality-driven. If my retraining SLA requires daily runs on 500k+ rows, LightGBM is the only library that fits the window. If my dataset is small, both are equivalent and I might use XGBoost for its more mature integrations with some inference infrastructure. For categorical-heavy datasets, LightGBM's native encoding is usually the right default. The more interesting tradeoff question is tree-based models versus neural networks for tabular data -- and there the evidence is consistent: trees win on most structured data problems until the dataset exceeds several million rows and the patterns become genuinely continuous in character."

| Phrase | Operational meaning |
|---|---|
| "leaf-wise overfitting" | LightGBM with high `num_leaves` overfit to training noise on small dataset |

### Interview Q&A

**Q: "Why do gradient boosting models like XGBoost often beat neural networks on tabular data?"**

Strong answer:
> "Several structural reasons that consistently hold empirically. First, real-world tabular data has sharp, irregular decision boundaries -- 'if enterprise tier AND tenure > 24 months, then low churn' -- regardless of all other features. Trees find these boundaries exactly. Neural networks must approximate them with smooth functions, which requires more data and architecture engineering to match. Second, tabular data mixes binary flags, counts, ratios, and categorical strings. Trees are completely scale-invariant and handle all types natively. Neural networks are scale-sensitive and require careful preprocessing for each type. Third, meaningful missing values. Trees learn optimal split directions for NaN values. Neural networks require explicit imputation. Fourth, dataset size: most production tabular problems have thousands to hundreds of thousands of examples. Neural networks are overparameterized relative to this scale and require more regularization engineering to generalize well. Trees with explicit depth and leaf constraints generalize efficiently. The empirical benchmark evidence is consistent -- XGBoost and LightGBM outperform deep learning on the majority of tabular benchmarks. Neural networks close the gap or win when the dataset is very large, when there's inherent spatial or temporal structure, or when pretrained representations are available."

---

**Q: "How does LightGBM's leaf-wise growth differ from standard level-wise growth, and what are the implications?"**

Strong answer:
> "Standard gradient boosting and most decision tree algorithms grow level-wise: they split all nodes at the current depth before going deeper, producing symmetric trees where all leaves are at the same depth. LightGBM grows leaf-wise: it finds the single leaf across the entire tree that would produce the greatest loss reduction if split, and splits only that leaf. This creates asymmetric trees where some branches are deep and others are shallow, concentrating the model's capacity where the data actually needs it. The benefit is better accuracy per tree for the same number of leaves, which translates to faster convergence. The risk is overfitting on small datasets -- leaf-wise growth can create very deep, narrow branches that memorize training noise. The primary control parameter is `num_leaves`, which limits the total number of terminal nodes. The default of 31 is conservative. Increasing it increases capacity. For small datasets I reduce it to 15 or even 7, with higher `min_data_in_leaf` to ensure leaves represent real population segments, not noise."

  ├── Use for: large datasets, rapid iteration, categorical-heavy data

  ├── Leaf-wise growth: control with num_leaves (default 31)
  └── Missing values: handled natively -- pass NaN directly

  DEFAULT CHOICE: LightGBM
  SWITCH TO XGB: mature ecosystem needed, small dataset, preference


THE MANDATORY SETTINGS:
  num_boost_round = 1000-2000   (upper bound only)
  early_stopping  = 50 rounds   (actual stop condition)
  verbose / verbosity = -1 / 0  (silent in production)
  feature_fraction = 0.8        (regularization)
  bagging_fraction = 0.8        (regularization)


TABULAR DATA RULE:
  Trees win over NNs when:
  ├── Mixed feature types
  ├── Meaningful missing values
  ├── Sharp non-linear boundaries
  ├── Moderate dataset size (10k-1M rows)
  └── Feature interactions are the signal

  NNs win over trees when:
  ├── Very large datasets (10M+ rows)
  ├── Spatial or temporal structure
  ├── Pretrained representations available
  └── Dense continuous feature spaces
