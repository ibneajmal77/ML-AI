## CONCEPT 1: A Pipeline Treats Preprocessing as Part of the Model, Not as Notebook Setup
<!-- cognitive job: establish that transforms are part of the learned system -->

| Dimension | Answer |
|---|---|
| **What it is** | A scikit-learn pipeline packages preprocessing steps and the model into one fit/predict object. |
| **Why it matters** | If the model depends on the transform, then separating them loosely creates leakage risk, reproducibility risk, and train/serve mismatch. |
| **When to use it** | Use pipelines whenever any learned transform, encoding, scaling, or feature step must happen before modeling. |
| **When NOT to use it** | Do not rely on manual preprocessing sequences in notebooks once the workflow needs repeatability or deployment. |
| **What juniors miss** | They often think of preprocessing as setup code instead of part of the actual trained system. |
| **Senior signal** | "If the model would fail without the transform, the transform is part of the model artifact." |

### Mental Model

A pipeline is like sealing the recipe and the oven settings together instead of keeping them on different sticky notes. If the recipe and the oven settings get separated, you can no longer trust that the next batch is the same product.

> **Before reading on:** Why is saving only the trained classifier but not the preprocessing logic a production risk?

```python
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression

pipe = Pipeline([
    ("scaler", StandardScaler()),
    ("model", LogisticRegression(max_iter=1000)),
])
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `Pipeline([...])` | Creates one ordered preprocessing-plus-model object. | This is the basic pattern that keeps transforms and prediction logic together. |
| `("scaler", StandardScaler())` | Adds a learned preprocessing step. | The transform now fits on training data inside the same unit as the model. |
| `("model", LogisticRegression(...))` | Adds the final estimator. | The model no longer assumes that preprocessing happened somewhere else correctly. |
| one object `pipe` | Encapsulates the whole path. | Fitting, cross-validation, persistence, and inference can now operate on one consistent unit. |

**Practice -- apply the concept:**

> A teammate says, "We can just remember to run the scaler before inference." Why is that weaker than using a pipeline?

> **Real-world scenario:** A churn baseline worked offline because the notebook scaled inputs manually before prediction. The API skipped one preprocessing detail and quietly fed the model mismatched features. Converting the workflow into a pipeline removed the human memory requirement.

## CONCEPT 2: Pipelines Reduce Leakage and Make Evaluation More Honest
<!-- cognitive job: connect pipelines to train-only fitting and safer cross-validation -->

| Dimension | Answer |
|---|---|
| **What it is** | Pipelines ensure that fit-time transforms are learned only from the training portion of each split or fold. |
| **Why it matters** | Manual preprocessing often leaks validation or test information into the training path, which inflates metrics and hides real generalization risk. |
| **When to use it** | Use pipelines in train/validation workflows, cross-validation, model comparison, and any setup with learned transforms. |
| **When NOT to use it** | Do not fit scalers, encoders, or vectorizers outside the training boundary and then pretend the evaluation is still clean. |
| **What juniors miss** | They often think "preprocessing first, split later" is tidy, not realizing the transform has already seen future information. |
| **Senior signal** | "A pipeline is one of the easiest ways to turn preprocessing from leakage risk into evaluation discipline." |

### Mental Model

Using a pipeline for evaluation is like making sure each student takes the test with only the study notes they had before the exam. Manual preprocessing often sneaks answer hints into the study material without anyone noticing.

---

> **Before reading on:** Why is fitting a `TfidfVectorizer` on the full dataset before cross-validation a leakage problem?

---

```python
from sklearn.model_selection import cross_val_score
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.pipeline import Pipeline
from sklearn.linear_model import LogisticRegression

text_pipe = Pipeline([
    ("tfidf", TfidfVectorizer(max_features=500)),
    ("clf", LogisticRegression(max_iter=1000)),
])

scores = cross_val_score(text_pipe, texts, labels, cv=5)
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `TfidfVectorizer(...)` inside the pipeline | Moves text fitting into the fold-safe path. | The vocabulary is learned separately inside each training fold rather than from the whole dataset. |
| `cross_val_score(text_pipe, ...)` | Cross-validates the whole pipeline, not just the classifier. | This makes the evaluation reflect the real training process. |
| `scores = ...` | Produces honest fold scores. | Manual preprocessing outside the pipeline would often make these scores too optimistic. |
| one object for CV | Treats transforms and model as one evaluable unit. | This is the right mental model for reproducible model comparison. |

**Practice -- apply the concept:**

> You want to compare logistic regression and random forest on the same encoded data. Why is comparing two full pipelines safer than encoding once globally and passing the matrix around?

> **Real-world scenario:** A team thought one model was clearly better than another because the shared preprocessing had already seen the full dataset. Once each model was evaluated inside its own leakage-safe pipeline, the gap narrowed and the ranking changed.

## CONCEPT 3: ColumnTransformer Pipelines Are the Practical Way to Handle Mixed-Type Features
<!-- cognitive job: teach the deployable pattern for numeric, categorical, and text features together -->

| Dimension | Answer |
|---|---|
| **What it is** | `ColumnTransformer` lets you apply different preprocessing logic to different feature groups inside one pipeline. |
| **Why it matters** | Real ML tables mix numeric, categorical, time-derived, and text features, and each type needs its own preprocessing path. |
| **When to use it** | Use it whenever a dataset has mixed feature types and you want one reproducible train/serve workflow. |
| **When NOT to use it** | Do not hand-roll parallel preprocessing branches outside the model object if the workflow needs repeatability, comparison, or deployment. |
| **What juniors miss** | They often preprocess feature types separately in scattered code and later lose track of the exact feature contract. |
| **Senior signal** | "ColumnTransformer is the practical bridge from mixed raw columns to one reproducible model-ready feature space." |

### Mental Model

`ColumnTransformer` is like an assembly line with dedicated stations for each material. Numeric columns go through one station, categories through another, text through another, and the final product exits as one consistent assembled feature set.

---

> **Before reading on:** Why is a mixed-type dataset especially risky to preprocess manually across several notebook cells or helper functions?

---

```python
from sklearn.compose import ColumnTransformer
from sklearn.preprocessing import OneHotEncoder, StandardScaler
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.pipeline import Pipeline
from sklearn.linear_model import LogisticRegression

preprocess = ColumnTransformer(
    transformers=[
        ("num", StandardScaler(), ["monthly_spend", "days_since_signup"]),
        ("cat", OneHotEncoder(handle_unknown="ignore"), ["plan_type"]),
        ("txt", TfidfVectorizer(max_features=300), "support_note"),
    ]
)

model = Pipeline([
    ("preprocess", preprocess),
    ("clf", LogisticRegression(max_iter=1000)),
])
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `ColumnTransformer(...)` | Declares type-specific preprocessing branches. | Mixed-feature tables need explicit, reproducible treatment by type. |
| `StandardScaler()` numeric branch | Handles continuous columns. | Some models need scaled numeric inputs to behave well. |
| `OneHotEncoder(handle_unknown="ignore")` | Handles categorical columns safely. | Unknown categories should not crash inference. |
| `TfidfVectorizer(...)` text branch | Handles raw text in the same feature pipeline. | Text can be part of a classical mixed-feature model without living outside the pipeline. |
| `Pipeline([...])` wrapping it all | Couples heterogeneous preprocessing to the estimator. | The full feature contract is now deployable and evaluable as one unit. |

**Practice -- apply the concept:**

> Your dataset has numeric account features, plan categories, and support-ticket text. Why is `ColumnTransformer` a better deployment pattern than three separate manual preprocessing steps?

> **Real-world scenario:** A team’s customer-health model mixed spend, plan type, and notes. Manual preprocessing worked until a new plan type appeared in production. The pipeline version handled unseen categories cleanly and kept the text and numeric paths aligned.

## DECISION RULES

### Pipeline Design Framework

1. Ask whether the model depends on any learned preprocessing or multi-step feature transformation.
2. If yes, build a pipeline instead of a loose sequence of manual steps.
3. Group features by type and define the preprocessing path for each type explicitly.
4. Keep learned transforms inside the pipeline so train-only fitting happens automatically.
5. Evaluate whole pipelines, not estimators detached from their transforms.
6. Persist the whole pipeline when the transform path is part of the model behavior.
7. Prefer `handle_unknown="ignore"` or other safe inference settings for categories.
8. Treat the pipeline as the deployable unit, not just as notebook convenience.

### Worked Example

| Step | Ticket-routing example |
|---|---|
| Raw features | metadata counts, category fields, ticket text |
| Preprocessing needs | scaling, one-hot encoding, TF-IDF |
| Safe structure | one `ColumnTransformer` + classifier pipeline |
| Evaluation | cross-validate the full pipeline |
| Deployment | save the whole pipeline artifact |
| Benefit | train, validation, and inference see the same feature logic |

### Red Flags

| What someone says / does | Why it should raise a flag |
|---|---|
| "We save the model, and preprocessing lives in the notebook." | Train/serve mismatch is now a likely future incident. |
| "We fit the scaler first so the data is ready for all models." | That often leaks information across comparisons or splits. |
| "We manually one-hot categories before every training run." | Reproducibility and feature ordering are now fragile. |
| "The model works as long as we remember the preprocessing steps." | Memory is not a deployment strategy. |

### Use / Do Not Use

| Use pipelines this way when | Do NOT use pipelines this way when |
|---|---|
| your feature logic must be repeatable, evaluable, and deployable with the model | you are treating preprocessing as temporary setup code detached from the trained system |

## COMMON MISTAKES

### Mistake 1: Saving only the estimator

**Mistaken instinct:** "The model weights are the important part; the preprocessing is easy to recreate."

**Why it feels right:** The estimator feels like the "real" ML object.

**Why it fails:** If the model was trained on transformed features, then inference without the same transform path is a different model in practice.

**Better move:** Save the full pipeline whenever preprocessing matters to the prediction behavior.

### Mistake 2: Preprocessing globally before splitting or cross-validation

**Mistaken instinct:** "It is cleaner to prepare the data once up front."

**Why it feels right:** It produces a neat matrix that seems reusable for all later steps.

**Why it fails:** Learned transforms can leak validation or test information into the training path and inflate the reported performance.

**Better move:** Put learned preprocessing inside the pipeline and let each split fit it safely.

### Mistake 3: Hand-managing mixed-type preprocessing

**Mistaken instinct:** "Numeric, categorical, and text transforms are separate anyway, so manual code is fine."

**Why it feels right:** Each branch can look straightforward in isolation.

**Why it fails:** Feature order, unknown categories, and transform consistency become harder to guarantee across retrains and deployments.

**Better move:** Use `ColumnTransformer` so the mixed-type contract lives in one explicit place.

### Mistake 4: Treating pipelines as only a scikit-learn convenience feature

**Mistaken instinct:** "Pipelines are just for nicer code style."

**Why it feels right:** The workflow may still run without them in development.

**Why it fails:** Pipelines are really about leakage control, reproducibility, and deployable feature consistency.

**Better move:** Think of pipelines as system-safety tools, not only syntax sugar.
