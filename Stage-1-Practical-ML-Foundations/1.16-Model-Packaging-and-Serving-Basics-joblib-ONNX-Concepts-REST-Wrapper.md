## CONCEPT 1: The Real Deliverable Is the Model Artifact, Not Just the Training Script
<!-- cognitive job: shift the learner from "I trained a model" to "I produced a reusable, loadable artifact" -->

| Dimension | Answer |
|---|---|
| **What it is** | A model artifact is the saved object and metadata needed to reproduce prediction behavior outside the training notebook. |
| **Why it matters** | If you cannot reload the exact evaluated model state, you do not really have a deployable system. |
| **When to use it** | Use artifact packaging whenever a trained model must be reused by another script, service, job, or teammate. |
| **When NOT to use it** | Do not rely on ad hoc notebook state or manual re-training as the way to recreate inference behavior. |
| **What juniors miss** | They often think the code that trained the model is enough, even though serving needs the exact fitted object and its feature contract. |
| **Senior signal** | "The deployable unit is the evaluated artifact plus its loading contract, not the notebook that happened to create it." |

### Mental Model

A model artifact is like a compiled binary, not like the source code that produced it. The training code explains how it was built, but the artifact is what another system can actually run.

```python
from pathlib import Path
import joblib

artifact_dir = Path("artifacts/churn_baseline")
artifact_dir.mkdir(parents=True, exist_ok=True)

joblib.dump(model_pipeline, artifact_dir / "model.joblib")
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `artifact_dir = Path(...)` | Defines a stable artifact location. | Packaging needs an intentional boundary, not a random notebook output cell. |
| `mkdir(..., exist_ok=True)` | Ensures the artifact directory exists. | A repeatable path is part of making saving and loading operational. |
| `joblib.dump(model_pipeline, ...)` | Serializes the fitted pipeline object to disk. | Saving the full pipeline preserves preprocessing and model behavior together. |

## CONCEPT 2: Serialization Choice Is a Tradeoff Between Simplicity, Portability, and Runtime Target
<!-- cognitive job: teach the learner to choose between Python-native persistence and cross-runtime export without cargo-culting tools -->

| Dimension | Answer |
|---|---|
| **What it is** | Serialization is the process of turning a fitted model artifact into a form that can be stored and loaded later in a target runtime. |
| **Why it matters** | Different serialization formats optimize for different needs: easiest Python reload, safer portability, or serving outside Python. |
| **When to use it** | Use `joblib` or similar Python-native persistence for scikit-learn-first workflows, and consider ONNX when cross-runtime portability or non-Python inference matters. |
| **When NOT to use it** | Do not choose a format only because it sounds more production-like if the serving environment does not benefit from it. |
| **What juniors miss** | They often confuse artifact persistence with universal portability and assume one saved file works equally well everywhere. |
| **Senior signal** | "Pick the serialization format based on the inference runtime you actually need, not on abstract deployment prestige." |

### Mental Model

`joblib` is like saving a Python object for another Python process. ONNX is more like exporting a model into a neutral interchange format so another runtime can execute it without depending on the original training stack in the same way.

---

> **Before reading on:** If your inference service is already a Python service loading a scikit-learn pipeline, what problem does ONNX solve that `joblib` may not need to solve?

```python
import joblib

loaded_model = joblib.load("artifacts/churn_baseline/model.joblib")
predictions = loaded_model.predict(batch_df)
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `joblib.load(...)` | Reloads the saved Python artifact. | This is the simplest and most common reload path for sklearn-style systems. |
| `loaded_model.predict(batch_df)` | Runs inference through the loaded pipeline. | Packaging only matters if the loaded artifact behaves like the evaluated one. |
| one Python-native path | Keeps training and serving in the same ecosystem. | This lowers complexity when portability is not the main constraint. |

> **Real-world scenario:** A scoring job ran entirely in Python and only needed nightly batch predictions. The team initially planned an ONNX conversion because it "felt more production," but a versioned `joblib` artifact solved the real need with far less complexity.

## CONCEPT 3: Serving Is an Interface Design Problem, Not Just "Put the Model Behind an Endpoint"
<!-- cognitive job: teach the boundary between model artifact and inference contract -->

| Dimension | Answer |
|---|---|
| **What it is** | Model serving is the system boundary that loads an artifact, accepts inputs, validates them, runs inference, and returns outputs in a stable contract. |
| **Why it matters** | A good model can still fail operationally if the serving layer sends the wrong schema, hides version drift, or returns unusable outputs. |
| **When to use it** | Use a REST wrapper or batch scoring entrypoint when another system needs a stable way to call the model. |
| **When NOT to use it** | Do not build a real-time API if the use case is naturally offline batch scoring or if the surrounding system does not need synchronous predictions. |
| **What juniors miss** | They often think serving starts at `predict()` and forget schema validation, versioning, response shape, and failure handling. |
| **Senior signal** | "The serving layer owns the inference contract: inputs, artifact version, output schema, and operational failure modes." |

### Mental Model

Serving is like a receptionist plus a translator in front of the model. The model only knows how to score valid feature inputs; the serving layer checks what arrived, routes it to the right artifact, and returns an answer the caller can actually use.

---

> **Before reading on:** Why is a `predict()` call inside a notebook not the same thing as a usable serving interface?

---

```python
from fastapi import FastAPI

app = FastAPI()
model = joblib.load("artifacts/churn_baseline/model.joblib")

@app.post("/predict")
def predict(payload: list[dict]):
    rows = pd.DataFrame(payload)
    scores = model.predict_proba(rows)[:, 1]
    return {"model_version": "churn_baseline_v1", "scores": scores.tolist()}
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `model = joblib.load(...)` | Loads the packaged artifact at service startup. | The endpoint must serve a known artifact, not hidden notebook state. |
| `@app.post("/predict")` | Defines a stable inference entrypoint. | Other systems need a consistent call boundary, not direct notebook access. |
| `rows = pd.DataFrame(payload)` | Converts incoming records into the model's expected tabular shape. | Input-shape mismatches are one of the most common serving failures. |
| `predict_proba(rows)[:, 1]` | Produces class probabilities rather than only labels. | The returned output should match the downstream decision need. |
| `model_version` in the response | Returns explicit artifact identity. | Version visibility makes debugging and rollback much easier. |

**Practice -- apply the concept:**

> A business workflow only needs nightly churn scores written to a warehouse table. Why might a batch scoring job be a better serving shape than a synchronous REST endpoint?

> **Real-world scenario:** A support-risk model was exposed as a bare endpoint with no version field and no input checks. When predictions shifted after a deploy, nobody could tell whether the issue came from the artifact, the payload schema, or the caller.

## DECISION RULES

### Packaging and Serving Framework

1. First define the deployable unit: usually the full fitted pipeline, not only the estimator.
2. Save the exact evaluated artifact before thinking about endpoints or containers.
3. Choose the serialization format based on the target runtime and portability need.
4. Start with Python-native persistence when training and inference both live in Python.
5. Consider ONNX when you need broader runtime portability or a non-Python scoring path.
6. Decide whether the use case is batch scoring, synchronous API scoring, or event-driven scoring.
7. Design the serving contract explicitly: input schema, artifact version, output shape, and error behavior.
8. Treat loading, validation, prediction, and response formatting as part of one serving boundary.

### Worked Example

| Step | Churn scoring example |
|---|---|
| Trained unit | sklearn pipeline with preprocessing + classifier |
| Packaging choice | save full pipeline artifact with `joblib` |
| Runtime need | Python batch job and one internal API |
| Serving shape | start with batch scoring; expose REST only if another system needs synchronous calls |
| Response contract | probability score, model version, prediction timestamp |
| Benefit | reproducible loading path and debuggable inference behavior |

### Red Flags

| What someone says / does | Why it should raise a flag |
|---|---|
| "The notebook can rebuild the model if we need it." | Rebuilding is not the same as serving the exact evaluated artifact. |
| "We shipped an endpoint, so the model is production-ready." | An endpoint without schema control and artifact discipline is not enough. |
| "Let us export everything to ONNX because that sounds more serious." | Portability tools should solve a real runtime need, not just add status. |
| "The response only needs the score; version metadata is extra." | Missing version identity makes debugging, rollback, and audit much harder. |

### Use / Do Not Use

| Use this approach when | Do NOT use this approach when |
|---|---|
| you need a reproducible artifact and a clear inference boundary | you are treating deployment as an afterthought bolted onto notebook state |

## COMMON MISTAKES

### Mistake 1: Treating the training script as the deployable output

**Mistaken instinct:** "If we can rerun the notebook, we effectively have the model."

**Why it feels right:** The training code looks like the source of truth.

**Why it fails:** Deployment needs the exact fitted state that was evaluated, not a hope that re-training later will recreate it precisely.

**Better move:** Save and version the evaluated artifact directly.

### Mistake 2: Choosing a serialization format for prestige instead of need

**Mistaken instinct:** "ONNX sounds more production, so we should use it by default."

**Why it feels right:** More portable tools sound more advanced.

**Why it fails:** Extra conversion complexity only pays off if the target runtime actually needs that portability.

**Better move:** Start with the simplest format that matches the real serving environment.

### Mistake 3: Confusing a prediction function with a serving system

**Mistaken instinct:** "If `predict()` works locally, serving is basically done."

**Why it feels right:** The core model behavior appears correct.

**Why it fails:** Real serving also needs schema handling, version identity, response design, and failure behavior.

**Better move:** Design the input/output contract and loading path as first-class parts of the system.

### Mistake 4: Defaulting to real-time serving when batch scoring is enough

**Mistaken instinct:** "Production means API."

**Why it feels right:** APIs feel more serious and more interactive.

**Why it fails:** Real-time endpoints add latency, reliability, and ops constraints that many business workflows do not need.

**Better move:** Choose the serving shape based on business timing, not on prestige.

## SUMMARY

- The deployable unit is usually the full evaluated pipeline artifact, not only the model class or training notebook.
- Serialization is a runtime choice: use the simplest format that fits the real inference environment.
- `joblib` is often the right first step for scikit-learn systems that train and serve in Python.
- ONNX is valuable when portability across runtimes is a real requirement, not when it is only fashionable.
- Serving is an interface problem: artifact loading, schema validation, version identity, output design, and failure handling all matter.
- Many business workflows should start with batch scoring before they pay the cost of real-time API serving.

### Must Know Cold

- A model artifact is the saved fitted state another system can reload.

### Code Patterns to Remember

```python

```

```python

```

### Decision Triggers

- If training and inference are both in Python, start simple with Python-native artifact persistence.
- If another runtime must execute the model, portability concerns become more important.
- If the workflow scores nightly or hourly in bulk, batch is often better than an API.
- If downstream systems need synchronous answers, design an explicit online serving contract.

### Output Shapes to Remember

- saved artifact -> file or object bundle that can be reloaded
- batch scoring -> table or file of predictions
- REST serving -> JSON response with score, label, or both
- classification API -> often probability plus model version

### Confusions to Avoid

- training script vs fitted artifact
- serialization format vs serving method
- `predict()` working locally vs production-ready serving
- real-time inference need vs batch scoring need

### Top Traps

- saving only model weights and forgetting preprocessing
- choosing tools for prestige instead of deployment need
- exposing an endpoint with no version field or schema checks
- assuming every production use case needs online inference

### One Good Example to Keep

A churn model saved as one `joblib` pipeline artifact can be reloaded consistently by a Python batch job or a small FastAPI wrapper. That is often more valuable than a more complex export path chosen without a runtime reason.

## PRACTICAL LANDSCAPE

### Technique Landscape

| Need | Usual first practical move | Why |
|---|---|---|
| save a sklearn pipeline for Python reuse | `joblib.dump` / `joblib.load` | simple and aligned with the Python runtime |
| expose predictions to another service | small REST wrapper around the saved artifact | creates a stable call boundary |
| score large datasets on a schedule | batch scoring job | lower operational complexity than an always-on API |
| support non-Python runtime execution | evaluate ONNX-style export path | portability may outweigh Python-native simplicity |

### When to Use What

| If your real need is... | Usually start with... | Not with... |
|---|---|---|
| internal Python inference | saved Python artifact | extra export complexity you do not need |
| nightly warehouse scoring | batch pipeline | a synchronous endpoint by default |
| cross-runtime portability | export evaluation path | assuming `joblib` alone solves non-Python inference |
| debuggable API scoring | versioned REST wrapper | bare `predict()` with no schema contract |

### Coding Surface

| Practical area | Common tools / shapes | What you usually touch |
|---|---|---|
| artifact saving | `joblib.dump`, filesystem path | fitted pipeline object, versioned artifact path |
| artifact loading | `joblib.load` | startup load logic, artifact version selection |
| REST serving | FastAPI / Flask style route | request schema, dataframe conversion, response JSON |
| portable export concept | ONNX-related conversion/runtime tools | export step, runtime compatibility check |
| batch scoring | scheduled Python job | read inputs, load artifact, write predictions |

### Output Shapes to Remember

| Pattern | Typical output |
|---|---|
| classifier artifact | saved pipeline file |
| batch scoring | rows with `id`, `score`, `prediction_time`, `model_version` |
| REST endpoint | JSON with score, label, version, optional explanation fields |
| portability export | alternate runtime-readable model artifact |
