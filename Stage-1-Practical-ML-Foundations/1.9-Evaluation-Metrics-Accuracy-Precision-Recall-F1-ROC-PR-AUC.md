## CONCEPT 1: A Metric Is Not Just a Number, It Is a Choice About Which Mistakes Matter
<!-- cognitive job: establish that metric choice shapes model judgment and that accuracy/F1 can mislead -->

| Dimension | Answer |
|---|---|
| **What it is** | An evaluation metric is a compressed view of model behavior, and each metric emphasizes some kinds of mistakes while hiding others. |
| **Why it matters** | If you choose the wrong metric, you can optimize, compare, or report a model in a way that flatters the wrong behavior. |
| **When to use it** | Use this framing whenever you evaluate a classifier, especially before training and before presenting results. |
| **When NOT to use it** | Do not treat any single metric as a universal truth across all class balances and cost structures. |
| **What juniors miss** | They often ask "what metric is standard?" instead of "what metric reflects the real failure cost here?" |
| **Senior signal** | "Metrics are a business decision layer on top of the confusion matrix, not a neutral scoreboard." |

### Mental Model

A metric is like a camera angle on the same scene. The scene is the confusion matrix. The metric is just the viewpoint. If you pick the wrong angle, you can hide the exact part of the model that is failing.

### First Concrete Artifact: Accuracy Can Look Great While the Model Does Nothing Useful

```python
total = 100_000
positives = 300
negatives = total - positives

always_negative_accuracy = negatives / total
print(round(always_negative_accuracy, 4))
# 0.997
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `positives = 300` | Defines a rare-event dataset | Imbalance is where accuracy becomes most deceptive. |
| `negatives = total - positives` | Makes the majority class dominant | A trivial model can look strong simply by predicting the majority class. |
| `negatives / total` | Computes accuracy of an "always negative" model | This shows why a high accuracy number can hide total failure on the positive class. |
| `0.997` | Looks excellent at a glance | A polished-looking number can still represent a useless model if it ignores the cases you care about. |

**Practice -- apply the concept:**

> A fraud model reports `99.7%` accuracy on a dataset where fraud is `0.3%`. Explain in plain language why that number is not enough.

> **Real-world scenario:** A manufacturing team celebrated high accuracy on a defect model until someone converted the misses into actual defective items still reaching customers. The model had been rewarded for predicting the non-defect majority class, not for catching enough defects.

## CONCEPT 2: Precision, Recall, and F-Scores Describe One Threshold, While ROC-AUC and PR-AUC Describe the Whole Ranking
<!-- cognitive job: separate threshold-specific metrics from threshold-free metrics and clarify ROC vs PR -->

| Dimension | Answer |
|---|---|
| **What it is** | Precision, recall, and F-scores evaluate one chosen operating point, while ROC-AUC and PR-AUC summarize ranking quality across all thresholds. |
| **Why it matters** | You need threshold-free metrics for model comparison and threshold-specific metrics for deployment reporting. |
| **When to use it** | Use AUC-style metrics when comparing models broadly, and use precision/recall-style metrics when judging the chosen deployment threshold. |
| **When NOT to use it** | Do not report ROC-AUC alone on rare-event problems and assume it tells the full truth about deployed behavior. |
| **What juniors miss** | They often mix up "good ranking quality" with "good threshold behavior" and report whichever number looks best. |
| **Senior signal** | "AUC tells me whether the model sorts cases well; precision and recall tell me whether the deployed operating point is acceptable." |

### Mental Model

Think of ROC-AUC and PR-AUC as maps of the whole mountain, while precision and recall are the checkpoint you actually chose to stand on. The map matters for comparing routes. The checkpoint matters for the reality you deploy.

---

### First Concrete Artifact: Precision, Recall, and F1 at One Threshold

```python
precision = 72 / (72 + 18)
recall = 72 / (72 + 9)
f1 = 2 * precision * recall / (precision + recall)

print(round(precision, 3), round(recall, 3), round(f1, 3))
# 0.8 0.889 0.842
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `precision = ...` | Measures correctness of positive predictions | Precision matters when false alarms are expensive. |
| `recall = ...` | Measures coverage of actual positives | Recall matters when missed positives are expensive. |
| `f1 = ...` | Combines precision and recall into one threshold-specific score | F1 rewards balance, but that balance assumes false positives and false negatives are roughly equally costly. |
| `0.8`, `0.889`, `0.842` | Shows a concrete operating-point summary | These describe one threshold, not the full ranking behavior across all thresholds. |

**Practice -- apply the concept:**

> Your rare-event model has strong ROC-AUC but weak PR-AUC. What does that suggest about how well it handles the minority positive class?

> **Real-world scenario:** A churn model had a respectable ROC-AUC, but once the team checked the PR curve they saw the model was much weaker at the rare positive cases than the headline number implied. That changed which model they selected before deployment.

## CONCEPT 3: Good Evaluation Ends in a Scorecard and a Business Translation, Not a Single Headline Metric
<!-- cognitive job: teach complete reporting and metric selection by class balance and business cost -->

| Dimension | Answer |
|---|---|
| **What it is** | A production evaluation should combine the right comparison metric, threshold metrics, baseline comparison, and business-cost interpretation. |
| **Why it matters** | A single number can mislead; a scorecard shows both model quality and deployment reality. |
| **When to use it** | Use it whenever a model is being compared, approved, monitored, or explained to stakeholders. |
| **When NOT to use it** | Do not stop after reporting one metric that makes the model look strongest. |
| **What juniors miss** | They often report a metric but not the baseline, threshold, class balance, or business consequence. |
| **Senior signal** | "I report the metric that fits class balance, then precision and recall at deployment threshold, then I translate the result into operational impact." |

### Mental Model

A good evaluation report is like a medical chart, not a single vital sign. One reading can be useful, but the real judgment comes from seeing the whole set together and relating it to actual risk.

### First Concrete Artifact: A Multi-Metric Scorecard

```python
scorecard = {
    "roc_auc": 0.89,
    "pr_auc": 0.58,
    "precision_at_threshold": 0.64,
    "recall_at_threshold": 0.84,
}

for metric, value in scorecard.items():
    print(f"{metric}: {value}")
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `scorecard = {...}` | Stores multiple evaluation views together | Real evaluation needs more than one metric because each metric answers a different question. |
| `"roc_auc"` and `"pr_auc"` | Summarize model quality across thresholds | These help compare models before locking the operating point. |
| `"precision_at_threshold"` and `"recall_at_threshold"` | Describe the chosen deployment behavior | This is the operational reality the business will actually live with. |
| iterating through the scorecard | Treats metrics as a set, not a single winner | Strong evaluation is a reporting system, not a hunt for one flattering number. |

## DECISION RULES

### Metric Selection Framework

1. Start with the positive-class rate.
2. If the classes are strongly imbalanced, do not lead with accuracy.
3. Use ROC-AUC as a general ranking comparison metric mainly when class balance is not severely skewed.
4. Use PR-AUC as the primary comparison metric when the positive class is rare.
5. For the deployed threshold, always report precision and recall directly.
6. Use F1 only when false positives and false negatives are roughly equally costly.
7. Use F-beta when you still want one combined number but one error type matters more.
8. Translate the deployed threshold behavior into business impact before presentation.

### Worked Example

| Step | Fraud example |
|---|---|
| Positive rate | very low |
| Accuracy choice | reject as headline metric |
| Comparison metric | PR-AUC first, ROC-AUC second |
| Deployment metric | precision and recall at chosen threshold |
| Combined metric | maybe F2 if recall deserves heavier weight |
| Business translation | fraud caught, fraud missed, false alarm load, cost impact |

### Red Flags

| What someone says / does | Why it should raise a flag |
|---|---|
| "Accuracy is 99%, so we're done." | That can hide total failure on the rare class. |
| "ROC-AUC is high, so the fraud model is excellent." | ROC-AUC can flatter rare-event problems by giving credit for true negatives. |
| "F1 dropped, so the new threshold is worse." | F1 may be the wrong metric if the business cost is asymmetric. |
| "We picked the metric after seeing which one looked best." | That is evaluation theater, not disciplined model selection. |

### Use / Do Not Use

| Use this framing when | Do NOT use this framing when |
|---|---|
| you need to choose and defend the right classification metric for the real problem | you are treating all classification problems as if one universal metric should always win |

## COMMON MISTAKES

### Mistake 1: Leading With Accuracy on Imbalanced Data

**Mistaken instinct:** "Accuracy is the clearest metric, so it should be the main number."

**Why it feels right:** It is easy to understand and looks like a fair overall summary.

**Why it fails:** When one class dominates, accuracy mostly rewards correct majority-class predictions and can hide collapse on the minority class.

**Better move:** Start with PR-AUC or precision/recall if the positive class is rare.

---

### Mistake 2: Treating ROC-AUC as the Full Answer for Rare-Event Problems

**Mistaken instinct:** "AUC is threshold-free and widely used, so it must be the safest comparison metric."

**Why it feels right:** It sounds more advanced and more complete than threshold-specific metrics.

**Why it fails:** ROC-AUC includes behavior on the abundant negative class, which can make rare-event performance look better than it really is.

**Better move:** Use PR-AUC as the primary comparison metric when positives are rare, and still report precision/recall at the chosen threshold.

---

### Mistake 3: Using F1 When the Business Costs Are Asymmetric

**Mistaken instinct:** "F1 balances precision and recall, so it must be the fair compromise."

**Why it feels right:** One clean combined number feels easier to compare and defend.

**Why it fails:** F1 assumes false positives and false negatives matter equally, which is rarely true in production.

**Better move:** Use F-beta or direct precision/recall at the business threshold when the costs differ.

---

### Mistake 4: Reporting Metrics Without Business Translation

**Mistaken instinct:** "If I report enough metrics, the business will infer the right conclusion."

**Why it feels right:** The technical report feels complete.

**Why it fails:** Stakeholders decide from consequences, not from metric names. Without translation, the wrong metric often dominates the conversation.

**Better move:** Convert threshold behavior into missed events, false alarms, workload, or money before presenting.

## PRACTICAL LANDSCAPE

### Technique Landscape

| Concept family | Main technique families | What they are for |
|---|---|---|
| threshold metrics | accuracy, precision, recall, F1, F-beta | evaluate one chosen operating point |
| ranking metrics | ROC-AUC, PR-AUC | compare model ranking quality across thresholds |
| probability metrics | log-loss | judge probabilistic quality when score meaning matters |
| reporting structure | scorecard, segmented metrics, business translation | make evaluation decision-ready |

### When to Use What

| If your real need is... | Usually start with... | Not with... |
|---|---|---|
| balanced-class quick summary | accuracy plus better supporting metrics | accuracy alone in all cases |
| rare-event model comparison | PR-AUC | accuracy as the headline |
| deployed threshold judgment | precision and recall at threshold | ROC-AUC alone |
| one-number summary with unequal costs | F-beta or direct business tradeoff | plain F1 by habit |

### Coding Surface

| Concept family | Common libraries / tools | Canonical API shape | Typical output |
|---|---|---|---|
| threshold metrics | `sklearn.metrics` | `precision_score(y, y_pred)` | one number at chosen threshold |
| ranking metrics | `sklearn.metrics` | `roc_auc_score(y, y_score)` / `average_precision_score(...)` | threshold-free comparison score |
| curves | `sklearn.metrics` | `roc_curve(...)`, `precision_recall_curve(...)` | arrays across thresholds |
| scorecard reporting | `pandas` + metrics | compare multiple metrics together | evaluation table |
| segmented evaluation | grouped reporting | compute metrics by slice | worst segments surfaced |

### Output Shapes to Remember

| Need | Typical output |
|---|---|
| operating point | precision / recall / F-score |
| model comparison | ROC-AUC / PR-AUC |
| rare-event honesty | PR curve and PR-AUC |
| stakeholder report | evaluation scorecard plus business translation |
