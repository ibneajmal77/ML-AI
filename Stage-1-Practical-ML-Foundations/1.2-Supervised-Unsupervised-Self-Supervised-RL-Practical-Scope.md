## CONCEPT 1: Supervised Learning Is the Default for Business Prediction
<!-- cognitive job: establish the dominant paradigm and anchor it to the presence of labels -->

| Dimension | Answer |
|---|---|
| **What it is** | Learning from labeled input-output pairs so the model can predict the correct output for new inputs. |
| **Why it matters** | Most business ML problems are predictive: spam or not, churn or not, price value, fraud risk, ticket category. |
| **When to use it** | When you have historical examples with verified outcomes and you need future predictions at the case level. |
| **When NOT to use it** | When you do not have labels, or when the task is really discovery or grouping rather than prediction. |
| **What juniors miss** | "We have data" is not enough. Supervised learning needs labeled outcomes, not just records. |
| **Senior signal** | "If the business question is predictive and labeled outcomes exist, supervised is the baseline assumption until proven otherwise." |

### Mental Model

Supervised learning is like learning from an answer key: each training example already includes the correct answer. Unlike clustering, the goal is not to discover structure but to reproduce a known target on new cases. If the business asks "what will happen for this next customer, ticket, or transaction?", supervised should be your first mental route.

### First Concrete Artifact: Ticket Routing as a Supervised Problem

```python
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.linear_model import LogisticRegression
from sklearn.pipeline import Pipeline

train_texts = [
    "I was charged twice for my subscription",
    "The app crashes when I upload a file",
    "Please refund my last invoice",
    "Login page keeps timing out",
]
train_labels = [
    "billing",
    "technical",
    "billing",
    "technical",
]

model = Pipeline([
    ("vectorizer", TfidfVectorizer(ngram_range=(1, 2))),
    ("classifier", LogisticRegression(max_iter=1000)),
])

model.fit(train_texts, train_labels)
prediction = model.predict(["My payment failed and I need a refund"])[0]
print(prediction)  # billing
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `train_labels = [...]` | Supplies the correct answer for each example | This is the defining feature of supervision. Without labels, this training setup cannot exist. |
| `Pipeline([...])` | Chains text featurization and classification together | In production, supervised ML is almost always a transformation step plus a predictor, not "raw text straight into a label." |
| `TfidfVectorizer(ngram_range=(1, 2))` | Converts text into weighted unigram and bigram features | The model needs numeric representations. The label alone is not enough; representation quality still matters. |
| `model.fit(train_texts, train_labels)` | Learns the mapping from text to routing label | This is where the answer key is consumed. The model is optimizing against known outcomes. |
| `model.predict(...)` | Produces a label for a previously unseen case | This is the business payoff: prediction on new cases, not just description of old ones. |

**Practice -- apply the concept:**

> A bank has 3 years of transactions and a verified `fraud` / `not_fraud` decision for each investigated case. The goal is to score new transactions in real time. Is this supervised learning? Name the input, the label, and one reason this is a good fit.

> **Real-world scenario:** A SaaS support team had 40,000 historical tickets already labeled by destination team. They considered clustering first because it sounded "more AI." The right move was a supervised classifier: the business already knew the target categories, so discovery was not the problem. The resulting model cut manual routing time because it was trained directly against the outcome the team cared about.

## CONCEPT 2: Unsupervised and Self-Supervised Solve Different Problems
<!-- cognitive job: prevent the common confusion between no labels, discovered structure, and labels created from data itself -->

| Dimension | Answer |
|---|---|
| **What it is** | Unsupervised learning finds structure without labels; self-supervised learning creates a training signal from the data itself so a model can learn representations at scale. |
| **Why it matters** | These two are often confused, especially in NLP and LLM discussions, but they support very different decisions and system designs. |
| **When to use it** | Use unsupervised when you need grouping, anomaly detection, or exploratory structure; use self-supervised when you want general representations from large unlabeled corpora. |
| **When NOT to use it** | Do not use clustering as a substitute for labels when the actual goal is prediction, and do not call embeddings "unsupervised" if they came from pretraining. |
| **What juniors miss** | "No human labels" does not mean the paradigm is unsupervised. Self-supervised systems still have a training signal; it is generated from the data structure. |
| **Senior signal** | "Clustering gives me hypotheses about structure; self-supervised pretraining gives me representations I can fine-tune or reuse." |

### Mental Model

Unsupervised learning is like sorting a pile of mixed documents into stacks by similarity without knowing what any stack should be called. Self-supervised learning is like taking a damaged sentence and learning by filling in the missing word; the data itself provides the teaching signal. The first discovers structure. The second builds reusable internal representations.

### First Concrete Artifact: Contrast the Two Cases

```python
# UNSUPERVISED: discover structure in unlabeled customer behavior
from sklearn.cluster import KMeans
from sklearn.preprocessing import StandardScaler
import numpy as np

customer_features = np.array([
    [120, 5, 0.30],
    [95, 3, 0.10],
    [450, 12, 0.05],
    [380, 9, 0.02],
])

X_scaled = StandardScaler().fit_transform(customer_features)
clusters = KMeans(n_clusters=2, random_state=42).fit_predict(X_scaled)
# Example output: [0, 0, 1, 1]


# SELF-SUPERVISED: reuse a model pretrained from structure in raw text
from sentence_transformers import SentenceTransformer

embedder = SentenceTransformer("all-MiniLM-L6-v2")
vector = embedder.encode("Refund requested after duplicate charge")
# vector is a dense semantic representation
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `KMeans(...).fit_predict(X_scaled)` | Groups similar examples without any label column | Clustering gives you group IDs, not business meanings. Humans still have to interpret what each cluster means. |
| `clusters = ...` | Produces discovered group membership | This is descriptive structure, not a trained predictor of a known business target. |
| `SentenceTransformer("all-MiniLM-L6-v2")` | Loads a pretrained embedding model | The useful representation came from self-supervised pretraining done elsewhere at scale. |
| `embedder.encode(...)` | Converts text into a semantic vector | Embeddings are a reusable representation learned from data structure, not a cluster assignment. |
| `vector is a dense semantic representation` | Shows the output type is a representation, not a label | This is what self-supervised learning buys you: a strong feature space you can reuse for search, ranking, or later supervised fine-tuning. |

**Practice -- apply the concept:**

> You have 50,000 unlabeled customer records and want to "predict churn." What is the wrong shortcut people often take with clustering, and what should you do instead if the business really needs churn prediction?

> **Real-world scenario:** A legal-documents team had very few labels and initially wanted to cluster documents into case types. That would only have given them groups to inspect. The better route was to start from a domain-pretrained language model, use its self-supervised representations, and then fine-tune on the small labeled set they did have. Accuracy jumped because they stopped confusing structure discovery with prediction.

## CONCEPT 3: Reinforcement Learning Is for Sequential Decisions with Reward, Not Ordinary Classification
<!-- cognitive job: give the learner a practical boundary for RL and explain why RLHF matters without over-expanding its scope -->

| Dimension | Answer |
|---|---|
| **What it is** | Learning by acting in an environment, receiving reward feedback, and improving a policy over time. |
| **Why it matters** | RL is important conceptually because it powers some robotics, control, game agents, and the alignment layer in modern LLM systems. |
| **When to use it** | When actions affect future states and success is defined by cumulative reward rather than a single correct label per example. |
| **When NOT to use it** | Do not use RL for ordinary one-shot prediction tasks like ticket classification, fraud classification, or house-price prediction. |
| **What juniors miss** | RL sounds advanced, so people over-apply it; most business ML problems are not sequential decision problems. |
| **Senior signal** | "If there is no state-action-reward loop with delayed consequences, I probably do not have an RL problem." |

### Mental Model

Reinforcement learning is like training a game player: each move changes the next situation, and the payoff may arrive later. Unlike supervised learning, there is no answer key saying "this exact action is correct" for every state. The system has to explore, receive reward, and learn a policy that works over a sequence.

### First Concrete Artifact: The RL Loop and RLHF

```python
# Conceptual RL loop
state = environment.observe()
action = policy.choose_action(state)
next_state, reward = environment.step(action)
policy.update(state, action, reward, next_state)


# RLHF mental model for LLMs
# 1. Start with a pretrained language model
# 2. Fine-tune on instruction-response examples (supervised)
# 3. Train a reward model from human preference comparisons
# 4. Update the language model to maximize reward-model score
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `state = environment.observe()` | Captures the current situation | RL begins from state, not from a static row in a table. |
| `action = policy.choose_action(state)` | Chooses what to do next | The output is an action policy, not a one-off label. |
| `next_state, reward = environment.step(action)` | Shows the environment responds to the action | This feedback loop is the defining shape of RL. Actions change what happens next. |
| `policy.update(...)` | Improves behavior using reward | The model learns from scalar feedback rather than a direct correct answer for every case. |
| `# 2. Fine-tune ... (supervised)` and `# 4. Update ... maximize reward-model score` | Separates supervised fine-tuning from the later RLHF step | This avoids the common mistake of saying LLMs are "just RL." Base capability is largely from self-supervised pretraining, then supervised tuning, then alignment. |

**Practice -- apply the concept:**

> A recommendation system chooses which item to show first, and that choice changes what the user clicks next and what feedback is observed later. Why is this closer to RL than ordinary classification? Name the state, the action, and the reward in one sentence each.

> **Real-world scenario:** A robotics team training a warehouse arm to pick and place packages has a real RL problem: each movement changes the next state, and the system is rewarded for successful placement without damage. By contrast, a support-ticket team does not have an RL problem just because routing decisions have downstream business impact. The routing task still has a direct label, so supervised is the right paradigm.

## DECISION RULES

### Paradigm Routing Framework

1. State the task as `input -> output`.
2. Ask: do verified output labels already exist for past examples?
3. If no labels exist, ask whether the goal is discovery/structure or strong reusable representations.
4. If the system acts over time and receives reward from consequences, ask whether this is a state-action-reward loop.
5. For text, image, and audio problems, ask whether a relevant pretrained model already exists before training from scratch.
6. Choose the simplest paradigm that matches the data and feedback you actually have.

```text
STEP 1: What feedback exists?
        |
        +--> Verified labels for past examples?
        |        |
        |        +--> Yes -> SUPERVISED
        |
        +--> No labels
                 |
                 +--> Need groups, anomalies, or structure?
                 |        |
                 |        +--> Yes -> UNSUPERVISED
                 |
                 +--> Need reusable representations from raw data at scale?
                 |        |
                 |        +--> Yes -> SELF-SUPERVISED
                 |
                 +--> Agent acts, world responds, reward arrives?
                          |
                          +--> Yes -> REINFORCEMENT LEARNING
```

### Worked Example

**Scenario:** A company wants to build an internal document assistant for employees. They have 900 labeled question-answer pairs and 80,000 unlabeled internal documents.

| Step | Question | Answer in this case |
|---|---|---|
| 1 | Input -> output clear? | Yes. Input: employee question + relevant context. Output: useful answer or ranked relevant docs. |
| 2 | Verified labels exist? | Yes, but only 900 QA pairs. That supports supervised fine-tuning or evaluation, not training from scratch at scale. |
| 3 | Valuable unlabeled data exists? | Yes. The 80,000 docs are ideal for embeddings and retrieval built on self-supervised pretrained models. |
| 4 | Need groups/anomalies only? | No. The goal is answer quality, not clustering documents into piles. |
| 5 | State-action-reward loop? | Not primarily. This is not an RL-first problem. |
| 6 | Best route | Use self-supervised pretrained embeddings for retrieval, then supervised fine-tuning or evaluation on the labeled QA set. |

### Red Flags

| What someone says / does | Why it should raise a flag |
|---|---|
| "We have lots of data, so supervised should work" | Data volume is meaningless if the outcomes are not labeled. |
| "We don't have labels, so let's just cluster and call the clusters our classes" | Clusters are hypotheses about structure, not verified business targets. |
| "Embeddings are unsupervised" | Most modern embeddings come from self-supervised pretraining, which is a different training setup. |
| "Let's use RL because the problem is important" | Importance does not make a problem sequential or reward-driven. |
| "We'll train our text model from scratch on 800 examples" | In text tasks, not checking for a strong pretrained model first is usually a scoping error. |

### When to Use / When Not to Use

| Use this when | Do NOT use this when |
|---|---|
| `Supervised` when you have labeled historical outcomes and need future predictions | when you only have raw data and no target variable |
| `Unsupervised` when you need clusters, anomalies, or exploratory structure | when the real goal is predicting a known business label |
| `Self-supervised` when you need strong representations from unlabeled corpora or pretrained models | when you are describing pure clustering or anomaly discovery |
| `Reinforcement learning` when actions change future states and reward arrives over time | when the task is a one-shot classification or regression problem |

## COMMON MISTAKES

### Mistake 1: Starting with Algorithms Instead of Feedback Type

**Mistaken instinct:** "Should we use XGBoost, BERT, or PPO for this?"

**Why it feels right:** Algorithms are concrete and familiar, while paradigm selection feels abstract.

**Why it fails:** The available feedback type eliminates most of the option space immediately. If you do not have labels, asking which supervised model to use is already the wrong conversation.

**Better move:** Route the problem by data and feedback first: labels, no labels, self-generated signal, or reward loop. Choose the model family only after that.

---

### Mistake 2: Using Clustering to Avoid Labeling

**Mistaken instinct:** "We don't have labels, so clustering will solve the prediction problem."

**Why it feels right:** Clustering produces visible groups quickly, which feels like progress without the cost of annotation.

**Why it fails:** Cluster IDs are not ground truth business outcomes. They describe structure in past data but do not automatically answer predictive questions like churn, fraud, or escalation risk.

**Better move:** Use clustering only to explore structure or guide labeling. If the business needs prediction, create or recover labels and then train supervised.

---

### Mistake 3: Calling Self-Supervised Models "Unsupervised"

**Mistaken instinct:** "BERT and embedding models are unsupervised because nobody labeled the data."

**Why it feels right:** Both paradigms use unlabeled raw data, so the distinction sounds pedantic.

**Why it fails:** Self-supervised learning still has a training signal generated from the data itself, such as next-token prediction or contrastive pairs. That difference explains why pretrained models are so useful for downstream tasks.

**Better move:** Say exactly what happened: the model was pretrained self-supervised on unlabeled data, then reused or fine-tuned for a downstream task.

---

### Mistake 4: Reaching for RL on Ordinary Prediction Problems

**Mistaken instinct:** "This is a high-value decision problem, so RL should be more powerful."

**Why it feels right:** RL sounds like the most advanced part of ML, and downstream consequences make the problem feel sequential even when it is not.

**Why it fails:** If each case has a direct label and there is no true state-action-reward loop, RL only adds complexity. Most tabular, NLP classification, and forecasting tasks are not RL problems.

**Better move:** Ask whether actions change future states and whether reward arrives over time. If not, stay in supervised or another simpler paradigm.

## PRACTICAL LANDSCAPE

### Technique Landscape

| Paradigm | Main technique families | What they are for |
|---|---|---|
| Supervised | classification, regression, ranking, sequence labeling | predicting known business targets from labeled examples |
| Unsupervised | clustering, dimensionality reduction, anomaly detection | finding structure without labels |
| Self-supervised | next-token prediction, masked modeling, contrastive learning, reconstruction | learning reusable representations from raw data |
| Reinforcement | value-based, policy-based, actor-critic, RLHF | sequential decision-making with reward |

### When to Use What

| If your real need is... | Usually start with... | Not with... |
|---|---|---|
| predict a category | supervised classification | clustering |
| predict a number | supervised regression | anomaly detection |
| order results by likely relevance | ranking | plain classification if order matters more than label |
| label each token in text | sequence labeling | document-level classification |
| group similar items | K-Means or hierarchical clustering | supervised classification without labels |
| find irregular dense groups and noise | DBSCAN | K-Means if cluster shape is highly irregular |
| reduce many features to a smaller representation | PCA | RL |
| flag rare weird behavior | anomaly detection | ordinary clustering alone |
| get reusable text/image vectors | contrastive or other self-supervised embeddings | hand-labeling every item first |
| pretrain a language model | next-token or masked-token objectives | clustering text documents |
| discrete action decisions in a controlled environment | value-based RL | ordinary supervised classification if actions affect future state |
| continuous control | policy-based or actor-critic | value-only methods by default |
| stable general RL training | actor-critic methods like PPO | ad hoc hand-rolled loops |
| align model behavior to human preferences | RLHF / preference optimization | plain next-token pretraining alone |

### Coding Surface

| Paradigm | Common libraries / tools | Canonical API shape | Typical output |
|---|---|---|---|
| Supervised | `scikit-learn`, `xgboost`, `lightgbm`, `catboost`, `transformers` | `.fit(X, y)` -> `.predict(...)` / `.predict_proba(...)` | labels, values, probabilities |
| Unsupervised | `KMeans`, `DBSCAN`, `AgglomerativeClustering`, `PCA`, `IsolationForest`, `UMAP` | `.fit_predict(X)` / `.fit_transform(X)` | cluster IDs, anomaly flags, reduced vectors |
| Self-supervised reuse | `sentence-transformers`, `transformers`, embedding APIs | load model -> `.encode(...)` or fine-tune | embeddings or adapted checkpoints |
| Reinforcement | `gymnasium`, `stable-baselines3`, RLHF stacks | `reset/step`, agent `.learn()` | learned policy or aligned behavior |

|---|---|
| Classification | label, probability |
| Regression | number |
| Clustering | cluster ID |
| PCA / reduction | reduced vector |
| Embeddings | dense vector |

**Weak vs strong answer:**

Q: "What is self-supervised learning and why does it matter?"

- **Weak:** "It is like unsupervised learning where the model labels things itself."
- **Strong:** "Self-supervised learning creates a training signal from the structure of raw data itself, such as next-token prediction or contrastive pairs. That removes the human-label bottleneck and is why language and embedding models can be pretrained on massive corpora, then fine-tuned or reused effectively on much smaller downstream tasks."

### Interview Q&A

**Q: You have a new NLP classification problem and only 800 labeled examples. What is your approach?**

> "First I ask whether a relevant pretrained model already exists for the domain, because with only 800 labels I do not want to start from scratch unless I have a very strong reason. I would still build a simple supervised baseline, maybe TF-IDF plus logistic regression, to set a floor. Then I would compare that against fine-tuning or using embeddings from a pretrained language model, ideally a domain-specific checkpoint if one exists. If the classes are imbalanced, I would fix that in evaluation before drawing conclusions. The key judgment is that the task is supervised, but the learning setup should probably leverage self-supervised pretraining rather than ignore it."

---

**Q: What is the difference between unsupervised and self-supervised learning?**

> "Unsupervised learning finds structure in unlabeled data without a target, so the outputs are things like clusters, anomalies, or lower-dimensional representations that still need interpretation. Self-supervised learning also starts from unlabeled data, but it manufactures a training signal from the data structure itself, such as predicting the next token in text. That difference matters because self-supervised pretraining gives you reusable representations that transfer well to downstream tasks, while plain clustering does not solve a predictive business task by itself."

---

**Q: What is RLHF and why does it matter for LLMs?**

> "RLHF means reinforcement learning from human feedback. In practice, a language model is first pretrained self-supervised on massive text, then usually instruction-tuned with supervised examples, and then aligned using human preference feedback as a reward signal. The reason it matters is that next-token prediction gives you language capability, but not necessarily helpful assistant behavior. RLHF is part of what makes an LLM prefer responses humans judge as useful and safe, rather than just statistically likely."
