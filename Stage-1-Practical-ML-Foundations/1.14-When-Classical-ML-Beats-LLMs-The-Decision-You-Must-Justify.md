## CONCEPT 1: Classical ML Wins When the Task Is Bounded, Repetitive, and Measurable
<!-- cognitive job: establish the kind of problem where classical ML is naturally stronger -->

| Dimension | Answer |
|---|---|
| **What it is** | Classical ML is often the better choice when the task is narrow, the output space is fixed, the signal is structured, and success can be measured cleanly. |
| **Why it matters** | Many teams reach for LLMs too early and pay unnecessary cost, latency, and controllability penalties for tasks that do not need generative flexibility. |
| **When to use it** | Use classical ML for bounded classification, regression, ranking, anomaly detection, and structured scoring tasks with stable inputs and outputs. |
| **When NOT to use it** | Do not force classical ML onto open-ended reasoning, generation, tool-using dialogue, or tasks where the output space is not cleanly bounded. |
| **What juniors miss** | They often confuse "more powerful model class exists" with "that model class is the better engineering choice for this task." |
| **Senior signal** | "If the task has a fixed label set, stable features, measurable success, and tight cost or latency constraints, classical ML gets the first right of refusal." |

### Mental Model

Classical ML is like building a specialized machine for one factory step. If the job is narrow and repeated at scale, the specialized machine often beats a general-purpose worker who can do many things but costs more and is harder to control.

## CONCEPT 2: LLMs Win Only When the Task Actually Needs Generative or Reasoning Flexibility
<!-- cognitive job: prevent overuse of both classical ML and LLMs by drawing the real boundary -->

| Dimension | Answer |
|---|---|
| **What it is** | LLMs are the better choice when the task needs open-ended language understanding, generation, instruction following, flexible composition, or reasoning over variable contexts. |
| **Why it matters** | The question is not "are LLMs powerful?" but "does this task need that kind of power badly enough to justify its tradeoffs?" |
| **When to use it** | Use LLMs when the output is not a small fixed set, when natural-language generation matters, or when instructions and context vary too much for stable feature engineering alone. |
| **When NOT to use it** | Do not use an LLM just because the input contains text; many text tasks are still bounded enough for classical ML or embeddings plus simple classifiers. |
| **What juniors miss** | They often treat any text task as automatically LLM-shaped, even when the business output is still fixed and repetitive. |
| **Senior signal** | "The presence of text is not the deciding factor; the deciding factor is whether the task needs flexible generation or can be reduced to a bounded decision." |

### Mental Model

An LLM is like a skilled generalist consultant. It earns its keep when the problem keeps changing shape. If the task is the same decision over and over, a narrower system is often the better hire.

## CONCEPT 3: The Real Decision Is an Engineering Tradeoff Across Quality, Cost, Control, Latency, and Evaluation
<!-- cognitive job: teach the multi-factor decision instead of one-dimensional model hype -->

| Dimension | Answer |
|---|---|
| **What it is** | Choosing between classical ML and an LLM is a tradeoff across task fit, controllability, evaluation clarity, latency, recurring cost, and operational stability. |
| **Why it matters** | Teams often compare only headline capability and ignore the production system consequences of the choice. |
| **When to use it** | Use this tradeoff framing whenever someone proposes replacing a bounded model with an LLM or vice versa. |
| **When NOT to use it** | Do not make the decision from trend pressure, vendor excitement, or model prestige alone. |
| **What juniors miss** | They often optimize for model impressiveness instead of total system quality. |
| **Senior signal** | "I compare model choices as systems: not only what they can do, but how they are evaluated, controlled, served, and paid for." |

### Mental Model

This decision is like choosing between a custom tool and a highly capable human operator. The better choice depends not only on what each can do, but also on speed, repeatability, training cost, supervision burden, and how often the job changes.

| Code line / block | What it does | Why it matters |
|---|---|---|
| `quality` | Keeps task success central. | Better capability still matters, but it is not the only axis. |
| `latency` and `cost` | Add operational reality. | LLMs often lose on these axes when the task is repetitive and high-volume. |
| `control` | Introduces output-constraining and reliability concerns. | Classical ML often wins when outputs must be fixed and predictable. |
| `evaluation` | Asks whether success is measurable cleanly. | Bounded tasks with clear labels often favor classical ML because the eval loop is tighter. |

## DECISION RULES

### Classical ML vs LLM Framework

1. Define the output space first: fixed labels/score or open-ended generation.
2. Ask whether the task can be measured cleanly with bounded evaluation.
3. Check whether the inputs can be represented as stable features or embeddings plus a simple predictor.
4. Compare the latency and per-request cost requirements to what an LLM would impose.
5. Evaluate how much output control, determinism, and auditability the system needs.
6. If the task is bounded, repetitive, and measurable, give classical ML the first pass.
7. If the task needs flexible reasoning, generation, or instruction following, give the LLM stronger consideration.
8. Choose the smallest system that meets the real task, not the most fashionable system available.

### Worked Example

| Step | Support routing example |
|---|---|
| Output shape | one of six queues |
| Evaluation | clean labeled accuracy/PR-AUC possible |
| Inputs | structured metadata + short text |
| Cost/latency | tight for high-volume traffic |
| Control need | high, because routes must be deterministic enough |
| Decision | classical ML or embeddings + classical head before a full LLM router |

### Red Flags

| What someone says / does | Why it should raise a flag |
|---|---|
| "It has text, so it needs an LLM." | Text input alone does not determine the right model family. |
| "The LLM feels smarter in demos." | Demos often ignore cost, latency, control, and large-scale bounded evaluation. |
| "We can always prompt it into the label set." | Prompting into a fixed label space may still be less controllable and more expensive than a bounded classifier. |
| "Classical ML is too old-fashioned." | That treats model choice as status instead of engineering. |

### Use / Do Not Use

| Use classical ML first when | Do NOT default to classical ML when |
|---|---|
| the task is bounded, repetitive, measurable, and operationally constrained | the task requires open-ended generation, flexible instruction following, or reasoning over variable context that a bounded predictor cannot express well |

## COMMON MISTAKES

### Mistake 1: Equating text input with LLM necessity

**Mistaken instinct:** "If the input is natural language, the right answer must be an LLM."

**Why it feels right:** LLMs are strongly associated with text and appear more flexible.

**Why it fails:** Many text tasks still produce fixed labels, scores, or rankings and can be handled more cheaply and predictably by embeddings plus classical ML or other bounded approaches.

**Better move:** Decide from the output shape and task flexibility, not from the mere presence of text.

### Mistake 2: Comparing only demo quality

**Mistaken instinct:** "The LLM gave nicer-looking outputs in a few examples, so it is the better system."

**Why it feels right:** LLM demos are often more impressive qualitatively.

**Why it fails:** Demo quality can hide latency, cost, instability, prompt brittleness, and weaker bounded evaluation performance at scale.

**Better move:** Compare whole-system behavior under real constraints, not only the most impressive examples.

### Mistake 3: Using an LLM for a fixed-label problem without first trying the bounded baseline

**Mistaken instinct:** "The flexible system should dominate the narrow one."

**Why it feels right:** More expressive systems seem like they should subsume simpler tasks automatically.

**Why it fails:** The extra flexibility may add cost and unpredictability without improving the actual business metric.

**Better move:** Give classical ML the first pass on narrow structured tasks and make the LLM earn its complexity.

### Mistake 4: Treating classical ML as incapable of working with text

**Mistaken instinct:** "If there is text, classical ML is out."

**Why it feels right:** People often picture classical ML as only numeric-tabular.

**Why it fails:** Text can often be represented through TF-IDF, embeddings, counts, or metadata fusion, and the final task can remain bounded and classical.

**Better move:** Separate text representation from output decision. A task can use text features and still be a classical ML problem.

## PRACTICAL LANDSCAPE

### Technique Landscape

| Concept family | Main technique families | What they are for |
|---|---|---|
| bounded predictive tasks | logistic regression, trees, boosting, calibrated scorers | fixed labels, scores, rankings |
| text-understanding but bounded output | TF-IDF, embeddings + classical classifier | text input with fixed decision outputs |
| generative language tasks | LLM prompting, structured generation, instruction-following systems | open-ended or flexible text outputs |
| system tradeoff evaluation | latency, cost, control, evaluation, reliability | choose the right production approach |

### When to Use What

| If your real need is... | Usually start with... | Not with... |
|---|---|---|
| fixed-label prediction | classical ML | a full generative stack by default |
| bounded text routing or scoring | embeddings + classical head | assuming text forces an LLM |
| open-ended generation or rewriting | LLM | a fixed-label classifier |
| tight latency/cost constraints at high volume | classical ML first | an expensive flexible model without justification |

### Coding Surface

| Concept family | Common libraries / tools | Canonical API shape | Typical output |
|---|---|---|---|
| classical bounded model | `sklearn`, `lightgbm`, `xgboost` | `fit(X, y)` then `predict/predict_proba` | class or score |
| text + bounded decision | embeddings / vectorizers + classifier | `encode(text)` then `clf.predict(...)` | label or risk score |
| LLM task | provider SDK / prompt call | `client.responses.create(...)` or equivalent | generated text / structured output |
| decision review | metrics + cost/latency table | compare alternatives by system axes | justified model choice |

### Output Shapes to Remember

|---|---|
| bounded classical ML | label, probability, numeric score |
| bounded text ML | label or score from text features |

**Q: What would make you choose classical ML over an LLM for a production task?**

> "I choose classical ML when the task is bounded, the evaluation is clean, the output must be tightly controlled, and the system has meaningful latency or cost constraints.
