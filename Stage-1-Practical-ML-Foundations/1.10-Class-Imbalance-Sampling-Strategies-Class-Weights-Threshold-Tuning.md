## Why This Matters

### Business Importance
The most economically important ML problems are almost always imbalanced. Fraud is 0.1-1% of transactions. Disease is 0.01-5% of screened patients. Equipment failure is 0.5-3% of operating hours. Churn in healthy SaaS is 2-8% per month. The rare class is exactly what the business cares about -- and it is the class the model will learn to ignore unless you explicitly address the imbalance. Getting this wrong means shipping a model that is technically trained but operationally useless for its entire purpose.

### System Importance
Class imbalance affects models at three distinct points: during training (the loss function sees many more negative examples and optimizes for them), at threshold selection (0.5 default is calibrated for balanced classes), and at evaluation (accuracy and ROC-AUC flatter models that learn the majority class). The three techniques in this lesson address each of these points respectively. Knowing which problem each technique solves -- and which problems it does not -- is what distinguishes principled handling from trial-and-error.

### Interview Importance
Class imbalance is one of the most frequently asked production ML topics. Interviewers expect you to name at least three approaches, explain the mechanism behind each, and know when each is appropriate. The strongest answers include specific failure modes (SMOTE generating interpolated examples that do not reflect real production inputs, random oversampling causing memorization of duplicates) and an explicit ranking of preference (threshold tuning and class weights first; resampling as a last resort with careful validation).

## Key Takeaways -- Read Before Any Project

1. **Threshold tuning is the first and most cost-free imbalance fix.** Try it before any data modification. A model trained on imbalanced data can still produce good minority-class recall if the threshold is set correctly.
2. **Class weights are cleaner than resampling in almost every case.** They modify what the model optimizes -- the loss -- without touching the training data. No data duplication, no synthetic examples, no risk of introducing artifacts.
3. **Resampling should be your last resort, not your first.** It modifies the data distribution, adds complexity to the pipeline, and introduces failure modes that class weights and threshold tuning do not.
4. **SMOTE has real failure modes in production.** It generates synthetic examples by interpolating between real examples in feature space. On categorical features, this interpolation is meaningless. On real production data with messy distributions, SMOTE neighbors are often not semantically similar. Always validate on real (non-SMOTE) data.
5. **Apply resampling only to the training set. Never to validation or test.** Resampling the validation or test set inflates performance metrics and destroys the honest evaluation baseline.
6. **Always evaluate with PR-AUC after applying any imbalance technique.** Accuracy and ROC-AUC will not tell you whether the technique helped the rare class.
7. **The three techniques are not stacked.** Using class weights AND SMOTE AND threshold tuning simultaneously is over-correction. Start with one, measure its effect on PR-AUC, then decide if more is needed.

## PART 1 -- CORE CONCEPT

### Why Imbalance Breaks Standard Training

```text
Dataset: 10,000 examples
  Fraud (positive):      50   (0.5%)
  Legitimate (negative): 9,950 (99.5%)

What the model's loss function sees during training:
  For every 1 fraud example, it sees 199 legitimate examples.
  Minimizing cross-entropy loss: predicting "legitimate" for everything
  gives a loss of ~0.005 per example (very small).
  Actually learning fraud patterns costs more error initially.

The model takes the easy path:
  -> Learns to predict "legitimate" for everything
  -> Training loss is very low
  -> Recall for fraud: 0%
  -> Accuracy: 99.5%  <- looks great
  -> Business value: zero
```

The three techniques address this at different points:

```text
TIMELINE OF A TRAINING RUN:

  Data preparation         Training              Deployment
  ─────────────────────────────────────────────────────────
  
  [Resampling]             [Class weights]       [Threshold tuning]
  Modify the data          Modify the loss       Modify the decision
  before training          during training       boundary after training
  
  Most invasive            Middle ground         Least invasive
  Most failure modes       Clean and principled  Always available
  Sometimes necessary      Almost always better  Should always be tried
```

---

### Technique 1: Threshold Tuning

*(Covered in depth in Lesson 1.5 -- reviewed and extended here in the imbalance context)*

On a severely imbalanced dataset, the model's raw probability outputs are often well-calibrated -- it outputs low probabilities for the minority class because the minority class IS genuinely rare. The default threshold of 0.5 therefore flags very few cases. Lowering the threshold directly increases recall.

```text
Model output on fraud dataset (1% positive rate):

  Legitimate transactions: score distribution centered around 0.02
  Fraud transactions:      score distribution centered around 0.15-0.30

At threshold = 0.50: almost nothing flagged (all scores below 0.5)
At threshold = 0.10: fraud recall increases significantly
At threshold = 0.05: very high recall, more false positives

Threshold tuning does NOT change the model.
It changes which side of the decision boundary each example lands on.
It is operationally free: no retraining required.
```

**When threshold tuning is enough:**
- The model's probability scores have reasonable separation between classes
- PR-AUC is acceptable (model is ranking well) but operating point is wrong
- You only need to shift the recall-precision tradeoff, not improve raw model quality

**When threshold tuning is NOT enough:**
- The model has learned to output near-zero probabilities for ALL examples (total collapse)
- PR-AUC is at baseline (model cannot rank positives above negatives at all)
- The model has never seen enough minority examples to learn their patterns

In the second case, the model needs to learn differently -- which requires class weights or resampling during training.

---

### Technique 2: Class Weights

Class weights multiply the per-example loss contribution by a factor based on class membership. Minority examples contribute more to the total loss, forcing the model to pay more attention to getting them right.

```text
Without class weights:
  Loss = mean(loss_per_example)
  Dominated by the 99.5% majority class.

With class_weight = {0: 1, 1: 199}  (n_negative / n_positive = 9950/50):
  Loss = mean(weight_per_example * loss_per_example)
  A single fraud example contributes 199x as much to the loss.
  The model cannot ignore fraud anymore -- misclassifying it is too expensive.
```

**How to set class weights:**

```text
Balanced class weight:
  weight_class_i = n_total / (n_classes * n_class_i)

  For fraud (1%):
    weight_fraud = 10,000 / (2 * 50) = 100
    weight_legit = 10,000 / (2 * 9,950) = 0.503

  Balanced weights make each class contribute equally to the total loss,
  regardless of frequency.

Custom class weight:
  Based on FN/FP cost ratio (from Lesson 1.5):
  FN cost / FP cost = $400 / $14 ≈ 29
  weight_positive = 29, weight_negative = 1
  This directly mirrors the business cost structure.
```

**Why class weights are preferred over resampling:**

```text
Class weights:
  - No data modification (training data remains real examples only)
  - No pipeline complexity (one parameter in the model call)
  - Works with sklearn, XGBoost, LightGBM natively
  - Preserves original data distribution for all other features
  - Cannot generate artifacts or unrealistic examples

Resampling (oversampling):
  - Modifies training data (introduces duplicates or synthetics)
  - Requires additional preprocessing step in the pipeline
  - Duplicates can cause memorization
  - SMOTE synthetics can be unrealistic
  - Pipeline must correctly apply resampling only to train fold

In most cases: class weights are strictly better. Use them first.
```

---

### Technique 3: Resampling

Resampling changes the class distribution in the training data directly. There are three forms:

#### Random Oversampling

```text
Minority class: 50 fraud examples
Target ratio: 1:10 (fraud to legitimate)
Random oversampling: duplicate fraud examples until 995 copies exist

Before: 50 fraud + 9,950 legit
After:  995 fraud + 9,950 legit

Risk: the model sees the same 50 fraud examples repeated 20 times each.
It may memorize those specific patterns rather than generalizing.
New fraud examples that look slightly different may be missed.
```

#### Random Undersampling

```text
Majority class: 9,950 legitimate examples
Target ratio: 1:10 (fraud to legitimate)
Random undersampling: randomly drop legitimate examples until 500 remain

Before: 50 fraud + 9,950 legit
After:  50 fraud + 500 legit (total: 550 from 10,000)

Risk: you discarded 9,450 legitimate examples with real information.
The model sees less data overall.
On small datasets this can significantly hurt performance.
Best used when majority class is very large and redundant.
```

#### SMOTE (Synthetic Minority Oversampling Technique)

```text
SMOTE mechanism:
  For each minority example:
    1. Find its k nearest neighbors in feature space
    2. Randomly select one neighbor
    3. Create a synthetic example at a random point between
       the original and the neighbor

  Example (2 features):
    Real fraud: [0.8, 0.6]
    Neighbor:   [0.9, 0.7]
    Synthetic:  [0.8 + 0.5*(0.9-0.8), 0.6 + 0.5*(0.7-0.6)]
              = [0.85, 0.65]
```

**SMOTE failure modes:**

```text
SMOTE failure 1: Categorical features
  Interpolating between "enterprise" and "basic" plan produces
  a value like 0.5 * "enterprise" + 0.5 * "basic"  <- undefined.
  SMOTE cannot generate meaningful synthetics for categorical features.
  Solution: encode categoricals carefully or use SMOTE variants
  that handle mixed types (e.g., SMOTENC for numeric+categorical).

SMOTE failure 2: Noisy minority examples
  If fraud examples include some mislabeled legitimate transactions,
  SMOTE will interpolate between genuine fraud and noise.
  The resulting synthetics populate the decision boundary with
  noise-derived examples.

SMOTE failure 3: Overconfident boundary
  SMOTE fills the space between minority examples.
  If the real minority class has subpopulations (different fraud types)
  with empty space between them, SMOTE generates unrealistic examples
  in that empty space, forcing the model to classify that space as fraud
  when it should not be.

SMOTE failure 4: Validation contamination
  If SMOTE is applied before splitting, synthetic examples may be
  neighbors of validation-set real examples -- leakage.
  Always split FIRST, then apply SMOTE only to the training fold.
```

### Choosing the Right Technique

```text
IMBALANCE HANDLING DECISION TREE:

Step 1: Check PR-AUC on an unmodified model.
  Is PR-AUC significantly above the random baseline (= positive rate)?
    YES: Model can rank. Try threshold tuning first.
    NO:  Model has collapsed. Need class weights or resampling.

Step 2: Try threshold tuning.
  Lower threshold until recall reaches business target.
  Check if precision is still acceptable.
  If yes: DONE. No data modification needed.

Step 3: Add class weights.
  Use balanced weights or FN/FP cost-ratio weights.
  Retrain. Evaluate PR-AUC on validation set.
  Check improvement vs unweighted model.
  If PR-AUC improves and operating point is acceptable: DONE.

Step 4: Try undersampling (if majority class is very large).
  Apply only to training data after stratified split.
  Check if PR-AUC improves further vs class weights alone.
  Improvement is unlikely on moderate imbalance.
  On extreme imbalance (1000:1+) it can help.

Step 5: Try SMOTE (last resort, numeric features only).
  Apply only to training data after splitting.
  Validate on REAL (non-synthetic) validation data only.
  If features include high-cardinality categoricals: use SMOTENC or skip.
  If PR-AUC improves beyond class weights: keep it. Otherwise: revert.

NEVER:
  - Apply resampling before splitting
  - Apply resampling to validation or test data
  - Stack class weights + SMOTE + threshold all together without measuring each
```

---

### The Stacking Problem

A common mistake is applying multiple techniques simultaneously:

```text
OVER-CORRECTION EXAMPLE:
  scale_pos_weight = 199           <- shifts training loss
  + SMOTE oversampling             <- further shifts data distribution
  + threshold = 0.05               <- then shifts decision boundary

All three push in the same direction (more recall, less precision).
Combined: model flags everything as positive.
Precision collapses to the base rate (0.5%).

CORRECT APPROACH:
  Apply one technique at a time.
  Measure PR-AUC at each step.
  Stop when PR-AUC at your business-acceptable precision point is satisfied.
```

## PART 2 -- PRODUCTION REALITY

### The End-to-End Imbalance Protocol

```text
1. ALWAYS split first (stratified), then apply imbalance handling to train only.

2. Run baseline model (no imbalance handling). Record PR-AUC.

3. Apply class weights (balanced or cost-ratio). Retrain. Record PR-AUC.
   -> If PR-AUC improved: compare operating points. If good: done.

4. Threshold sweep on validation set. Find cost-minimizing threshold.
   -> Report precision and recall at that threshold.
   -> If operating point is acceptable: done.

5. If still insufficient (severe imbalance, very rare events):
   a. Try random undersampling of majority class.
   b. If features are mostly numeric: try SMOTE on training fold only.
   c. Evaluate on REAL validation data only (never on synthetic examples).

6. Evaluate final model on test set using PR-AUC + full evaluation scorecard.
   Never use accuracy. Never use ROC-AUC alone.
```

### Real Use Cases by Imbalance Regime

| Domain | Positive rate | Typical technique | Why |
|---|---|---|---|
| SaaS churn | 5-15% | Threshold tuning | Mild imbalance; model learns well; threshold fixes operating point |
| Credit card fraud | 0.1-0.5% | Class weights + threshold | Severe but workable with weights; threshold for operating point |
| Medical rare disease | 0.01-1% | Class weights + possibly SMOTE | Very severe; model needs strong weighting; SMOTE helps if features are numeric |
| Defect detection (manufacturing) | 0.5-3% | Class weights | Moderate; weights usually sufficient |
| Spam detection | 5-20% | Threshold tuning or slight weights | Mild; threshold usually enough |
| Anomaly detection (0.01%) | < 0.1% | Specialized anomaly detection | Standard classification may not be the right framing at this extreme |

### Design Patterns

**Pattern 1 -- imbalanced-learn Pipeline Integration**
When using SMOTE, integrate it into an `imblearn.pipeline.Pipeline`, not a standalone step. This ensures resampling is applied correctly inside each cross-validation fold, not before the split.

**Pattern 2 -- Measure PR-AUC before and after every technique change**
Any imbalance technique that does not improve PR-AUC on the real validation set (not synthetic) is not helping. Remove it and do not add complexity without demonstrated improvement.

**Pattern 3 -- Keep the threshold as a config value**
After training with class weights, the calibrated probability outputs may still need threshold adjustment. Implement threshold as a deployment config value that can be updated without retraining when business requirements change.

**Pattern 4 -- Report the positive rate alongside every metric**
When presenting results, always state the positive rate. "PR-AUC of 0.62 on a 3% positive rate dataset" is a very different statement from "PR-AUC of 0.62 on a 40% positive rate dataset."

### Day-to-Day Team Reality

An ML engineer is building a fraud detection model. Positive rate: 0.8%. The unweighted GBM achieves PR-AUC of 0.31 (random baseline: 0.008 -- so it's 38x better than random, but absolute value is low). At threshold 0.5, recall is 0.23 -- missing 77% of fraud.

Step 1: Add `scale_pos_weight = 124` (n_negative / n_positive). PR-AUC: 0.51. Recall at 0.5 threshold: 0.61.

Step 2: Threshold sweep on validation. At threshold 0.12, recall is 0.81 at precision 0.44. Business cost analysis: $400 FN, $14 FP. Cost-minimizing threshold: 0.15. PR-AUC and operating point are now acceptable.

Step 3: Team suggests adding SMOTE. Engineer runs the experiment: SMOTE + class weights produces PR-AUC 0.49 -- worse than class weights alone (the categorical features in the dataset made SMOTE generate noisy synthetics). SMOTE is dropped.

Final model: class weights + threshold tuning. No data modification. Clean pipeline. Defensible to compliance.

### Common Mistakes

- **Applying SMOTE before the train/val split.** Synthetic examples derived from validation-set neighbors leak validation information into training. Always split, then resample.
- **Applying SMOTE or oversampling to the validation set.** Performance on synthetic examples is meaningless. Evaluate only on real examples.
- **Using SMOTE when features include high-cardinality categoricals.** Interpolation is undefined for categorical values. Use SMOTENC or avoid SMOTE entirely.
- **Stacking all three techniques.** Class weights + SMOTE + low threshold = everything flagged as positive. Apply one at a time.
- **Not measuring PR-AUC after applying the technique.** The technique may not have helped -- or may have hurt. Always measure with the correct metric.
- **Treating class weights as a replacement for threshold tuning.** They solve different problems. Class weights improve learning. Threshold tuning sets the operating point. Both may be needed.

### Failure Modes

| Failure | Root cause | Diagnostic | Fix |
|---|---|---|---|
| Recall high but precision near base rate | Over-correction; too low threshold or too high class weight | Check precision at deployed threshold; PR curve collapse | Raise threshold; reduce class weight magnitude |
| SMOTE made things worse | High-cardinality categoricals; noisy minority class | Compare PR-AUC with vs without SMOTE on real val data | Remove SMOTE; use class weights only |
| Model still ignores minority class after class weights | Extreme imbalance (> 500:1) | Check probability score distribution -- all scores near 0 | Add undersampling to bring ratio to ~50:1; or try anomaly detection framing |
| Performance looks great on val, fails on real deployment | Validation set included synthetic examples | Re-run evaluation on real examples only | Fix pipeline: resample only inside training fold |
| Threshold from validation doesn't hold in production | Distribution shift since val set was collected | Monitor positive rate in production vs training | Retrain with more recent data; update threshold |

## PART 3 -- ENGINEERING

### Complete Imbalance Handling Protocol in Code

```python
import numpy as np
import pandas as pd
from sklearn.metrics import average_precision_score, roc_auc_score
from sklearn.model_selection import train_test_split
import lightgbm as lgb


def imbalance_protocol(
    X: pd.DataFrame,
    y: pd.Series,
    cost_fp: float,
    cost_fn: float,
    test_size: float = 0.2,
    val_size: float = 0.15,
) -> dict:
    """
    End-to-end imbalance handling protocol.
    Step 1: Baseline (no handling)
    Step 2: Class weights
    Step 3: Threshold tuning on weighted model
    Reports PR-AUC at each step to confirm improvement.
    """
    pos_rate = y.mean()
    print(f"Positive rate: {pos_rate:.3%}  (random PR-AUC baseline: {pos_rate:.4f})")

    # Stratified split -- always stratify on imbalanced data
    X_trainval, X_test, y_trainval, y_test = train_test_split(
        X, y, test_size=test_size, stratify=y, random_state=42
    )
    val_frac = val_size / (1 - test_size)
    X_train, X_val, y_train, y_val = train_test_split(
        X_trainval, y_trainval, test_size=val_frac, stratify=y_trainval, random_state=42
    )

    results = {}
    scale_pos_weight = (y_train == 0).sum() / (y_train == 1).sum()

    for label, params_extra in [
        ("Baseline (no weights)", {"scale_pos_weight": 1}),
        ("Class weights (balanced)", {"scale_pos_weight": scale_pos_weight}),
    ]:
        params = {
            "objective":        "binary",
            "metric":           "auc",
            "learning_rate":    0.05,
            "num_leaves":       31,
            "feature_fraction": 0.8,
            "bagging_fraction": 0.8,
            "bagging_freq":     5,
            "verbose":          -1,
            **params_extra,
        }
        train_ds = lgb.Dataset(X_train, label=y_train)
        val_ds   = lgb.Dataset(X_val,   label=y_val, reference=train_ds)

        model = lgb.train(
            params, train_ds, num_boost_round=1000,
            valid_sets=[val_ds],
            callbacks=[lgb.early_stopping(50, verbose=False),
                       lgb.log_evaluation(-1)],
        )
        val_scores = model.predict(X_val)
        pr_auc = average_precision_score(y_val, val_scores)
        roc_auc = roc_auc_score(y_val, val_scores)

        results[label] = {"model": model, "val_scores": val_scores,
                          "pr_auc": pr_auc, "roc_auc": roc_auc}
        print(f"\n{label}")
        print(f"  Val PR-AUC:  {pr_auc:.4f}  (random: {pos_rate:.4f})")
        print(f"  Val ROC-AUC: {roc_auc:.4f}  (random: 0.5000)")

    return results, X_test, y_test, X_val, y_val

# > Analogy: This protocol is the before/after experiment.
#   You measure PR-AUC at each step the same way a doctor measures blood pressure
#   before and after each medication change -- one change at a time,
#   measured each time, no stacking until you know the individual effects.

# > Common mistake: Applying class weights and checking only ROC-AUC.
#   On a 1% positive rate dataset, ROC-AUC can improve trivially while
#   PR-AUC (the honest metric) stays flat. Always measure PR-AUC.

# > Used in Project P1: If your churn dataset has < 10% positive rate,
#   run this protocol on it. Report the PR-AUC at each step to show
#   the class weight effect before choosing the final training configuration.
```

### Threshold Sweep with Cost Optimization

```python
import numpy as np
import pandas as pd
from sklearn.metrics import precision_recall_curve, average_precision_score


def find_cost_optimal_threshold(
    y_true: np.ndarray,
    y_scores: np.ndarray,
    cost_fp: float,
    cost_fn: float,
) -> tuple[float, pd.DataFrame]:
    """
    Sweep all thresholds. Find the one that minimizes total business cost.
    This is the correct way to set the deployment threshold after handling imbalance.
    Evaluate on REAL validation data (never on SMOTE-augmented data).
    """
    precision_vals, recall_vals, thresholds = precision_recall_curve(y_true, y_scores)
    precision_vals = precision_vals[:-1]
    recall_vals    = recall_vals[:-1]
    n_pos          = y_true.sum()

    rows = []
    for t, p, r in zip(thresholds, precision_vals, recall_vals):
        tp = r * n_pos
        fp = (tp / p - tp) if p > 0 else 0
        fn = n_pos - tp
        cost = fp * cost_fp + fn * cost_fn
        rows.append({"threshold": round(t, 3), "precision": round(p, 3),
                     "recall": round(r, 3), "est_cost": round(cost)})

    df = pd.DataFrame(rows)
    best_idx = df["est_cost"].idxmin()
    best = df.loc[best_idx]

    print(f"\nCost-optimal threshold: {best['threshold']:.3f}")
    print(f"  Precision: {best['precision']:.3f}")
    print(f"  Recall:    {best['recall']:.3f}")
    print(f"  Est. cost: ${best['est_cost']:,.0f}")

    # Show the table around the optimal point
    window = df.iloc[max(0, best_idx-3):best_idx+4]
    print(f"\nThreshold sweep (around optimal):")
    print(window.to_string(index=False))

    return best["threshold"], df

# > Analogy: This sweep is the same as adjusting the dial on a security
#   camera sensitivity. Too sensitive: constant false alarms (high FP cost).
#   Too insensitive: misses real intrusions (high FN cost).
#   The optimal setting minimizes total alarm cost, not just one type.

# > Common mistake: Running this threshold sweep on validation data that
#   includes SMOTE-generated examples. The optimal threshold on synthetic data
#   will not transfer correctly to real production data.

# > Used in Lesson 1.5 (already introduced) and now extended here
#   specifically for post-imbalance-handling deployment decisions.
```

### SMOTE -- Correct Implementation with Pipeline Integration

```python
import numpy as np
import pandas as pd
from imblearn.over_sampling import SMOTE, SMOTENC
from imblearn.pipeline import Pipeline as ImbPipeline
from sklearn.preprocessing import StandardScaler
from sklearn.linear_model import LogisticRegression
from sklearn.metrics import average_precision_score


def smote_correct_usage(
    X_train: pd.DataFrame,
    y_train: pd.Series,
    X_val: pd.DataFrame,
    y_val: pd.Series,
    categorical_indices: list[int] | None = None,
) -> None:
    """
    CORRECT: SMOTE applied inside pipeline, only on training data.
    SMOTE is never applied to validation or test data.
    If categorical features exist, use SMOTENC.

    categorical_indices: list of column positions for categorical features.
    If None: treat all features as numeric (standard SMOTE).
    """
    if categorical_indices:
        sampler = SMOTENC(
            categorical_features=categorical_indices,
            random_state=42,
            k_neighbors=5,
        )
        print("Using SMOTENC (handles mixed numeric/categorical)")
    else:
        sampler = SMOTE(random_state=42, k_neighbors=5)
        print("Using SMOTE (numeric features only)")

    # CRITICAL: SMOTE is a step in the TRAINING pipeline only.
    # It is NOT applied to X_val or X_test.
    pipeline = ImbPipeline([
        ("smote",      sampler),              # resamples training data only
        ("scaler",     StandardScaler()),
        ("classifier", LogisticRegression(max_iter=1000)),
    ])

    pipeline.fit(X_train, y_train)

    # Evaluate on REAL validation data -- never on synthetic examples
    val_scores = pipeline.predict_proba(X_val)[:, 1]
    pr_auc = average_precision_score(y_val, val_scores)

    # Compare to class weights baseline
    lr_weighted = LogisticRegression(class_weight="balanced", max_iter=1000)
    from sklearn.preprocessing import StandardScaler as SS
    sc = SS()
    X_train_s = sc.fit_transform(X_train)
    X_val_s   = sc.transform(X_val)
    lr_weighted.fit(X_train_s, y_train)
    val_scores_weighted = lr_weighted.predict_proba(X_val_s)[:, 1]
    pr_auc_weighted = average_precision_score(y_val, val_scores_weighted)

    print(f"\nPR-AUC comparison:")
    print(f"  SMOTE + LR:             {pr_auc:.4f}")
    print(f"  Class weights + LR:     {pr_auc_weighted:.4f}")
    print(f"\nIf SMOTE is not better: use class weights. Simpler, safer.")


# WRONG PATTERNS -- never do these:
def wrong_smote_usage(X, y):
    # WRONG 1: SMOTE before split
    smote = SMOTE(random_state=42)
    X_resampled, y_resampled = smote.fit_resample(X, y)
    # Now synthetic examples are neighbors of val/test examples -> leakage

    # WRONG 2: SMOTE on validation data
    X_val_synthetic, y_val_synthetic = smote.fit_resample(X_val, y_val)
    # Performance on synthetic val data is meaningless

    # WRONG 3: SMOTE with high-cardinality categoricals using plain SMOTE
    # smote.fit_resample(X_with_categories, y)
    # Produces undefined interpolations for categorical columns

# > Analogy: SMOTE inside a Pipeline is like a sous vide machine that
#   only activates when cooking the training meal, not when the validation
#   inspector arrives. The inspector always sees real food, not the
#   augmented version you practiced on.

# > Common mistake: Calling smote.fit_resample(X_train, y_train) as a
#   standalone step before model training. This looks correct but fails
#   inside cross-validation: the same synthetic examples appear in
#   multiple folds, inflating CV estimates.

# > Used in Lesson 1.12 (Overfitting): When resampling causes duplicate
```

### SMOTE vs Class Weights Empirical Comparison

```python
import numpy as np
from sklearn.datasets import make_classification
from sklearn.metrics import average_precision_score
from sklearn.model_selection import train_test_split
from sklearn.linear_model import LogisticRegression
from sklearn.preprocessing import StandardScaler
from imblearn.over_sampling import SMOTE
from imblearn.pipeline import Pipeline as ImbPipeline


def compare_imbalance_techniques(
    pos_rate: float = 0.05,
    n_samples: int = 20_000,
) -> None:
    """
    Empirical comparison of three techniques on synthetic data.
    Shows PR-AUC improvement for each approach.
    """
    X, y = make_classification(
        n_samples=n_samples,
        n_features=20,
        n_informative=10,
        weights=[1 - pos_rate, pos_rate],
        random_state=42,
    )
    X_train, X_val, y_train, y_val = train_test_split(
        X, y, test_size=0.2, stratify=y, random_state=42
    )

    scaler = StandardScaler()
    X_tr_s = scaler.fit_transform(X_train)
    X_val_s = scaler.transform(X_val)

    results = {}

    # Baseline
    lr = LogisticRegression(max_iter=1000)
    lr.fit(X_tr_s, y_train)
    results["1. Baseline (no handling)"] = average_precision_score(
        y_val, lr.predict_proba(X_val_s)[:, 1]
    )

    # Class weights
    lr_w = LogisticRegression(class_weight="balanced", max_iter=1000)
    lr_w.fit(X_tr_s, y_train)
    results["2. Class weights (balanced)"] = average_precision_score(
        y_val, lr_w.predict_proba(X_val_s)[:, 1]
    )

    # SMOTE
    pipe = ImbPipeline([
        ("smote", SMOTE(random_state=42)),
        ("scaler", StandardScaler()),
        ("clf", LogisticRegression(max_iter=1000)),
    ])
    pipe.fit(X_train, y_train)
    results["3. SMOTE oversampling"] = average_precision_score(
        y_val, pipe.predict_proba(X_val)[:, 1]
    )

    print(f"\nIMBALANCE TECHNIQUE COMPARISON  (pos_rate={pos_rate:.1%})")
    print(f"Random PR-AUC baseline: {pos_rate:.4f}")
    print()
    for name, pr in results.items():
        improvement = pr / pos_rate
        bar = "█" * int(improvement * 3)
        print(f"  {name:40s}  PR-AUC={pr:.4f}  ({improvement:.1f}x random)  {bar}")

# > Analogy: This comparison is the A/B test for your imbalance strategy.
#   You would never recommend a treatment without testing it. You should
#   never recommend SMOTE over class weights without measuring the difference.

# > Common mistake: Using SMOTE by default because "more data is better."
#   On many datasets, class weights alone match or beat SMOTE while
#   being simpler, faster, and free of categorical feature artifacts.

# > Used in Project P1: Run this comparison on your actual dataset
#   before committing to any specific imbalance technique.
```

### Must Know Cold
- The three techniques and their mechanisms: threshold tuning (post-training), class weights (loss modification), resampling (data modification)

### Weak Answer vs Strong Answer

**Q: "How do you handle class imbalance in a production ML model?"**

- **Weak:** "I use SMOTE to oversample the minority class, then train the model on the balanced dataset."
- **Strong:** "My approach depends on the severity of the imbalance and what the data looks like. I start with the least invasive techniques first. First, I try threshold tuning: lower the decision boundary on the validation set to the cost-minimizing point -- this requires no data modification and is free. If the model has learned nothing about the minority class and PR-AUC is at baseline, I move to class weights: the balanced class weight or a weight derived from the FN/FP cost ratio. Class weights modify the loss function without touching the training data, so they're clean and auditable. I reserve resampling for cases where the imbalance is so severe (greater than 200:1) that even class weights aren't enough. If I use SMOTE, I apply it only inside the training fold of a cross-validation or after the train/test split, never before. And I avoid SMOTE when features include high-cardinality categorical values -- it generates meaningless interpolations there. I always evaluate improvement with PR-AUC on real (not synthetic) validation data, because that's the only honest signal of whether the technique actually helped."

## PART 5 -- SENIORITY LAYER

### Seniority Story 1 -- The SMOTE Disaster

> "We had a medical screening model for a rare autoimmune condition. Positive rate was 1.2%. A contractor applied SMOTE to balance the dataset before training. The model's cross-validation AUC looked great: 0.94. When we evaluated it on a fully held-out real patient cohort, AUC was 0.71. The SMOTE-generated synthetics had been derived from real patients' continuous lab values, which made sense, but our feature set included three categorical variables: diagnosis_code (ICD-10, 800+ unique values), insurance_type (5 categories), and referring_specialist (ordinal, 4 levels). SMOTE had interpolated between these: a synthetic patient had insurance_type = 2.7 and referring_specialist = 1.4 -- neither of which is a valid state. The model learned patterns from an impossible population. The synthetic-versus-real distribution gap was invisible in cross-validation because the CV folds were drawn from the same synthetic pool. The fix was class weights only. AUC on the real cohort: 0.89. Two weeks of SMOTE work produced a worse, untrustworthy model."

**Lesson:** SMOTE on categorical features generates patients (or customers, or transactions) that cannot exist. The model learns from a phantom population and generalizes to the real one poorly.

### Seniority Story 2 -- The Over-Correction

> "A data scientist on my team was handling a fraud dataset with 0.3% positive rate. They applied: scale_pos_weight = 333 (full balance) + SMOTE with 5x oversampling + threshold = 0.05. Validation PR-AUC was 0.67. They were happy. I ran each technique separately. scale_pos_weight alone: PR-AUC 0.61. Threshold tuning on that model: PR-AUC unchanged (it's a threshold, not a model quality metric). SMOTE on top of class weights: PR-AUC 0.58 -- actually worse, because the SMOTE synthetics were noisy. Threshold 0.05 on top of class weights only: operating point was precision 0.38, recall 0.82 -- which was fine. The three-technique stack produced a model that flagged 21% of all transactions as fraud (precision = 0.014 at threshold 0.05 with all three corrections stacked). The single-technique model with tuned threshold flagged 4.2% as fraud at precision 0.38. The over-corrected model was useless for operations. The single-technique model was deployable. Three corrections, each pushing in the same direction, destroyed precision without improving recall."

**Lesson:** Imbalance handling is not additive. Each technique shifts the model in the same direction. Applying all three simultaneously produces a model optimized to recall everything -- and precision collapses.

### Tradeoff Language

> "Class imbalance handling is a one-dial problem disguised as a three-dial problem. All three techniques turn the same dial: toward more recall at the cost of precision. The question is not which combination to apply -- it's how far to turn the dial, measured by PR-AUC and the precision-recall operating point on real data. Class weights are the cleanest dial. Threshold is the deployment adjustment. Resampling is a bigger turn that comes with artifacts. Most production imbalanced problems are solved by class weights plus threshold tuning. SMOTE is reaching for a more powerful dial with more side effects."

### Failure Vocabulary

| Phrase | Operational meaning |
|---|---|
| "all three corrections stacked" | Class weights + SMOTE + low threshold applied simultaneously -- precision collapses |
| "evaluated on synthetic data" | PR-AUC computed on SMOTE-augmented val set -- not a real signal |
| "SMOTE on categoricals" | Meaningless interpolations generated for categorical features -- model learns phantom population |
| "the model collapsed" | Extreme imbalance; model predicts majority class for everything; PR-AUC at baseline |
| "the threshold was set at 0.5 on imbalanced data" | No operating point optimization; most minority class examples below 0.5 |
| "oversampled before splitting" | SMOTE or random oversampling applied to full dataset; leakage of synthetic examples into val/test |

### Interview Q&A

**Q: "What are SMOTE's failure modes and when would you avoid it?"**

Strong answer:
> "SMOTE has three main production failure modes. First, categorical features: SMOTE generates synthetic examples by interpolating between real minority examples in feature space. For continuous numeric features this produces plausible values. For categorical features it produces nonsense -- an insurance type of 2.7, a product category of 1.4. These impossible synthetic examples create a training population that doesn't exist in the real world, and the model generalizes poorly to actual production inputs. I avoid SMOTE when more than a few features are categorical, and use SMOTENC or class weights instead. Second, noisy minority labels: if some minority examples are mislabeled, SMOTE interpolates between real minority examples and those noise points, generating synthetics that populate the decision boundary with noise-derived examples. The model learns from a corrupted population. Third, overfit when applied before cross-validation folds are constructed: if SMOTE runs before the train/test split, synthetic examples derived from validation-set neighbors appear in training folds, inflating cross-validation estimates. The model looks better than it is. I always apply SMOTE inside the training pipeline using imbalanced-learn's Pipeline integration, not as a standalone preprocessing step before the split."
