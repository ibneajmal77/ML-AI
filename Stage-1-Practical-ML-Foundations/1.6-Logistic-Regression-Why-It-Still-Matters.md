## CONCEPT 1: Logistic Regression Is a Linear Scoring Model for Classification
<!-- cognitive job: establish what logistic regression actually does and why it is a classification model despite its name -->

| Dimension | Answer |
|---|---|
| **What it is** | Logistic regression is a linear model that turns a weighted feature score into a probability for a binary class. |
| **Why it matters** | It gives you a fast, interpretable, well-behaved classification baseline before you reach for more complex models. |
| **When to use it** | Use it first for binary classification when you want a strong baseline, interpretable coefficients, or cheap inference. |
| **When NOT to use it** | Do not expect it to automatically discover complex non-linear patterns or hidden feature interactions. |
| **What juniors miss** | They hear "regression" in the name and assume it is a value-prediction model instead of a classifier. |
| **Senior signal** | "Logistic regression is a linear classifier with a probability link, not a fancy version of linear regression." |

### Mental Model

Logistic regression is like a weighted checklist that ends in a probability dial. Each feature pushes the score up or down. Then the sigmoid function turns that running score into a number between `0` and `1`. It is still classification because the model is estimating the chance of a class, not a continuous target value.

---

> **Before reading on:** If the linear score gets larger and larger, what should happen to the predicted probability?

---

### First Concrete Artifact: Linear Score First, Probability Second

```python
import numpy as np

z_values = np.array([-3.0, -1.0, 0.0, 1.0, 3.0])
probabilities = 1 / (1 + np.exp(-z_values))

for z, p in zip(z_values, probabilities):
    print(f"linear_score={z:>4}: probability={p:.3f}")

# linear_score=-3.0: probability=0.047
# linear_score=-1.0: probability=0.269
# linear_score= 0.0: probability=0.500
# linear_score= 1.0: probability=0.731
# linear_score= 3.0: probability=0.953
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `z_values = ...` | Represents raw linear model scores before probability conversion | Logistic regression first builds a linear score, then converts it into a probability. |
| `1 / (1 + np.exp(-z_values))` | Applies the sigmoid function | This is what turns an unbounded score into a usable probability-like output. |
| `z = 0 -> p = 0.5` | Shows the midpoint | The sign and size of the linear score directly affect class confidence and threshold behavior. |
| larger `z` means larger `p` | Demonstrates monotonic movement | If features push the score up, the model becomes more confident in the positive class. |

**Practice -- apply the concept:**

> **Real-world scenario:** An ad-tech team used logistic regression for click prediction because they needed billions of fast decisions per day. The model was not chosen because it was glamorous. It was chosen because a simple linear score plus sigmoid was operationally cheap and good enough.

## CONCEPT 2: Coefficients Make the Model Readable, but Only if You Treat Them Carefully
<!-- cognitive job: teach coefficient interpretation and the importance of scaling and sign -->

| Dimension | Answer |
|---|---|
| **What it is** | Each coefficient shows how a feature pushes the linear score toward or away from the positive class. |
| **Why it matters** | Coefficients can turn the model into an explanation tool, not just a predictor. |
| **When to use it** | Use coefficient reading when stakeholders need to know what signals the model is relying on. |
| **When NOT to use it** | Do not compare raw coefficient magnitudes across features if the features were not scaled comparably. |
| **What juniors miss** | They often print coefficients and immediately rank them without checking scale, correlation, or encoding details. |
| **Senior signal** | "The coefficient direction is useful immediately; magnitude comparisons only make sense after disciplined preprocessing." |

### Mental Model

Coefficients are like votes in a weighted committee. Positive coefficients vote toward the positive class. Negative coefficients vote against it. But the size of the vote only becomes meaningfully comparable if everyone is speaking on roughly the same scale.

---

### First Concrete Artifact: Turning Coefficients Into Plain Language

```python
coefficients = {
    "payment_failures_30d": 0.82,
    "account_age_months": -0.34,
    "support_tickets_90d": 0.51,
    "plan_type_enterprise": -0.95,
}

for feature, coef in coefficients.items():
    direction = "pushes toward churn" if coef > 0 else "pushes away from churn"
    print(f"{feature:25s} {coef:+.2f} -> {direction}")

# payment_failures_30d       +0.82 -> pushes toward churn
# account_age_months         -0.34 -> pushes away from churn
# support_tickets_90d        +0.51 -> pushes toward churn
# plan_type_enterprise       -0.95 -> pushes away from churn
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `coefficients = {...}` | Represents learned feature weights | This is the simplest window into what the model learned. |
| `coef > 0` | Uses sign to interpret direction | Positive means more evidence toward the positive class; negative means the opposite. |
| `print(... direction ...)` | Converts coefficient signs into plain language | Stakeholders usually need the story, not raw math vocabulary. |
| `plan_type_enterprise: -0.95` | Shows a strong negative signal | Logistic regression can highlight protective factors as clearly as risk factors. |

## CONCEPT 3: Logistic Regression Wins When Simplicity, Speed, or Explainability Matter More Than Model Flexibility
<!-- cognitive job: teach when logistic regression is the correct engineering choice and introduce regularization as practical control -->

| Dimension | Answer |
|---|---|
| **What it is** | Logistic regression is often the best baseline and sometimes the best final model because it is interpretable, fast, and easy to regularize. |
| **Why it matters** | Real ML work is constrained by latency, auditability, maintenance cost, and data size, not just leaderboard score. |
| **When to use it** | Use it as the default first classifier, in regulated settings, at huge inference scale, or when labeled data is limited and you need a stable baseline. |
| **When NOT to use it** | Do not rely on it alone when the signal is strongly non-linear or depends on complex interactions you are not engineering explicitly. |
| **What juniors miss** | They often compare models only by benchmark score and ignore the operational cost of complexity. |
| **Senior signal** | "I use logistic regression first to establish the floor and to measure whether extra complexity actually buys enough value." |

### Mental Model

Choosing logistic regression is like choosing a reliable wrench before reaching for a full workshop. If the wrench already solves the problem cleanly, heavier machinery may add cost without adding enough value. In ML, that extra cost can be latency, maintenance burden, explanation risk, or deployment complexity.

### First Concrete Artifact: A Disciplined Logistic Regression Baseline

```python
from sklearn.linear_model import LogisticRegression
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler

baseline = Pipeline([
    ("scaler", StandardScaler()),
    ("model", LogisticRegression(C=1.0, penalty="l2", max_iter=1000)),
])

print(baseline)
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `StandardScaler()` | Scales features before fitting the model | Logistic regression depends on disciplined feature scale for stable coefficients and regularization behavior. |
| `LogisticRegression(C=1.0, penalty="l2", ...)` | Builds a regularized baseline classifier | `L2` is the usual stable default, and `C` controls how strongly coefficients are constrained. |
| `Pipeline([...])` | Packages preprocessing and model together | This prevents train/serve mismatch and makes the baseline reusable and audit-friendly. |
| `baseline` as the first model | Establishes the reference point | You should know what this model achieves before claiming a complex model is necessary. |

## DECISION RULES

### Logistic Regression First Framework

1. Confirm the task is binary classification.
2. Train logistic regression as the baseline before any complex classifier.
3. Scale numeric features before fitting and before reading coefficients.
4. Read the top positive and negative coefficients in business language.
5. Check whether the deployment context values explainability, low latency, or easy auditability.
6. Compare any more complex model against logistic regression on the same validation or test setup.
7. Keep the simpler model if the performance gain from complexity is small relative to its operational cost.
8. Add or tune regularization when coefficients are unstable or the feature set is large.

### Worked Example

| Step | Churn model example |
|---|---|
| Frame task | Binary classification: churn vs retain |
| Baseline | Train logistic regression first |
| Preprocessing | Scale account age, spend, usage, and ticket counts |
| Read model | Payment failures and support tickets push toward churn; tenure pushes away |
| Compare complexity | Gradient boosting beats logistic regression only slightly |
| Deployment choice | Ship logistic regression if interpretability and simplicity matter more than the small score gain |

### Red Flags

| What someone says / does | Why it should raise a flag |
|---|---|
| "We skipped logistic regression and went straight to boosting." | You have no baseline floor and may be paying complexity without evidence. |
| "Here are the top coefficients," without mentioning scaling | Coefficient comparisons may be misleading. |
| "AUC is higher, so the more complex model is automatically better." | That ignores latency, explainability, artifact size, auditability, and maintenance cost. |
| "L1 removed half the features, so that must be best." | Sparse does not automatically mean stable or correct, especially with correlated features. |

### Use / Do Not Use

| Use logistic regression when | Do NOT stop at logistic regression when |
|---|---|
| you need a baseline, explainability, low latency, or a simple deployable classifier | the problem clearly depends on strong non-linear structure or feature interactions that the linear model cannot capture |

## COMMON MISTAKES

### Mistake 1: Treating Logistic Regression as Obsolete

**Mistaken instinct:** "Tree models exist, so logistic regression is only for toy problems."

**Why it feels right:** Modern model discussions often focus on what is most powerful, not what is most appropriate.

**Why it fails:** In production, interpretability, latency, maintenance cost, and regulation often matter enough that logistic regression is still the right model.

**Better move:** Treat logistic regression as the required baseline and sometimes the correct final answer, not as a historical relic.

---

### Mistake 2: Interpreting Coefficients Without Scaling

**Mistaken instinct:** "The biggest absolute coefficient is the most important feature."

**Why it feels right:** The model outputs a clear numeric weight, so ranking those weights feels straightforward.

**Why it fails:** Feature scale can distort coefficient magnitude badly, so raw comparisons may tell the wrong story.

**Better move:** Scale features first, then interpret signs and relative magnitude more carefully.

---

### Mistake 3: Forgetting That Linear Models Miss Interactions

**Mistaken instinct:** "If the model has these features, it will automatically discover how they combine."

**Why it feels right:** Complex relationships in the data can feel like something any model should detect.

**Why it fails:** Logistic regression only sees the linear terms you give it unless you add interaction or transformed features explicitly.

**Better move:** Use feature engineering or move to tree-based models when interaction structure is part of the signal.

---

### Mistake 4: Using Regularization Without Understanding the Tradeoff

**Mistaken instinct:** "L1 is better because it removes features."

**Why it feels right:** Sparse models look cleaner and easier to explain.

**Why it fails:** L1 can behave unstably with correlated features and may drop one useful signal from a group arbitrarily.

**Better move:** Default to L2 for stability, try L1 when sparsity is a real goal, and compare both with validation.

### Decision Triggers

- If you need a strong baseline fast, start with logistic regression.
- If you need readable feature effects, logistic regression is a strong candidate.
- If a complex model only improves a little, question whether the complexity premium is worth it.
- If the signal is strongly non-linear, logistic regression may become the floor, not the finish.

### Output Shapes to Remember

| Need | Typical output |
|---|---|
| class probability | one score between `0` and `1` per row |
| feature effect direction | positive or negative coefficient |
| sparse feature selection | many zero coefficients with L1 |
| baseline comparison | validation metrics vs more complex models |

### Confusions to Avoid

| Easy confusion | Correct distinction |
|---|---|
| logistic regression vs linear regression | logistic regression is used for class probability, not continuous value prediction |
| coefficient size vs real importance | size comparison is unreliable without proper scaling and context |
| simple vs weak | a simple model can still be the best engineering choice |

### Top Traps

- skipping the baseline
- reading raw coefficients too literally
- assuming higher AUC automatically justifies complexity
- expecting a linear model to discover interactions on its own

### One Good Example to Keep

A regulated credit model may accept a slightly worse AUC from logistic regression because the model can generate clear reason codes and survive audit review. A more accurate black-box model is not better if it cannot be deployed lawfully or explained operationally.

### Technique Landscape

| Concept family | Main technique families | What they are for |

| linear classifiers | logistic regression | fast, interpretable binary classification |
| regularization choices | L1, L2 | control overfitting and coefficient behavior |
| feature shaping | scaling, interaction features | make coefficients stable and linear models more useful |

### When to Use What

| If your real need is... | Usually start with... | Not with... |
|---|---|---|
| interpretable binary classification | logistic regression + scaling | a complex black box by default |
| sparse coefficient set | L1 logistic regression | assuming L2 will zero features |
| stable default classifier | L2 logistic regression | skipping the baseline |
| non-linear automatic interactions | tree-based models after the logistic regression baseline | expecting plain logistic regression to discover them alone |

### Coding Surface

| Concept family | Common libraries / tools | Canonical API shape | Typical output |

| sparse feature selection | L1 penalty | `LogisticRegression(penalty="l1", solver="liblinear")` | many zero coefficients |
