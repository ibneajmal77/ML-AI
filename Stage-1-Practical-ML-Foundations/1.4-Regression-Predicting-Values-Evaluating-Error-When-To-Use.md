## CONCEPT 1: Regression Predicts Continuous Values
<!-- cognitive job: establish what regression is, what it is not, and how to detect when the framing itself is wrong -->

| Dimension | Answer |
|---|---|
| **What it is** | Regression is supervised learning where the target is a continuous numeric value. |
| **Why it matters** | Many business problems are about predicting amounts, not categories: price, revenue, demand, latency, LTV. |
| **When to use it** | When the output is a meaningful number on a continuous scale and the size of the error matters. |
| **When NOT to use it** | Do not use regression when the real decision is above/below threshold, category selection, or ranking order. |
| **What juniors miss** | A numeric-looking problem is not always a regression problem; sometimes the actual business action is classification or ranking. |
| **Senior signal** | "Before choosing the model, I confirm that the business really needs the number itself, not just a threshold decision or an ordering." |

### Mental Model

Regression is like estimating the dial reading on a gauge, not choosing a bucket. If the business needs "how much" or "how many," regression is often right. If the business really needs yes/no or first/second/third, the framing may be wrong even if the label is numeric.

---

> **Before reading on:** A stakeholder asks, "Will this customer spend more than $100 next month?" Is that naturally regression, or is that actually a different framing?

### First Concrete Artifact: Continuous Target vs Wrong Framing

```python
import pandas as pd

df = pd.DataFrame({
    "account_age_days": [120, 400, 40],
    "support_tickets_30d": [0, 3, 1],
    "monthly_spend": [45.0, 190.0, 75.0],
    "predicted_ltv": [220.0, 1400.0, 380.0],
})

X = df[["account_age_days", "support_tickets_30d", "monthly_spend"]]
y = df["predicted_ltv"]   # regression target: continuous value
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `predicted_ltv` | Defines a numeric target | This is a true regression target because the business wants the value itself. |
| `X = df[[...]]` | Selects model inputs | Regression uses the same supervised shape as classification: `X` features and `y` target. |
| `y = df["predicted_ltv"]` | Builds the target vector | The key difference from classification is the output type and how error is judged. |

**Practice -- apply the concept:**

> Decide whether each is better framed as regression, classification, or ranking: predicted house price, whether a loan will default, and which 10 products should appear first in search.

> **Real-world scenario:** A marketplace team first framed pricing as classification: "cheap, fair, expensive." That was too coarse for actually setting prices. Reframing it as regression let them estimate the number directly, then apply business rules on top of the prediction.

## CONCEPT 2: The Right Metric Depends on the Cost of Being Wrong
<!-- cognitive job: teach that regression metrics are business choices, not defaults -->

| Dimension | Answer |
|---|---|
| **What it is** | Regression metrics measure numeric prediction error, but they penalize mistakes in different ways. |
| **Why it matters** | Optimizing the wrong metric trains the model to care about the wrong kinds of mistakes. |
| **When to use it** | Every time you evaluate or compare regression models. |
| **When NOT to use it** | Do not report a metric just because it is popular; it must match business cost. |
| **What juniors miss** | `R²` sounds impressive but does not tell you whether the absolute error is acceptable in business terms. |
| **Senior signal** | "I choose MAE or RMSE based on the cost of large misses, and I never report R² alone." |

### Mental Model

Metrics are scoring rules for pain. MAE says all errors matter in proportion to size. RMSE says large misses hurt much more than small ones. `R²` is a relative explanation score, not a direct business-error score.

---

> **Before reading on:** Which is worse for the business: being wrong by 1,000 once, or being wrong by 100 ten times? Your answer determines whether MAE or RMSE is more appropriate.

---

### First Concrete Artifact: MAE vs RMSE vs R²

```python
import numpy as np
from sklearn.metrics import mean_absolute_error, mean_squared_error, r2_score

y_true = np.array([100, 200, 300, 400])
y_pred = np.array([90, 220, 280, 430])

mae = mean_absolute_error(y_true, y_pred)
rmse = np.sqrt(mean_squared_error(y_true, y_pred))
r2 = r2_score(y_true, y_pred)

print(mae)   # 20.0
print(rmse)  # 21.2...
print(r2)    # close to 0.97 on this tiny example
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `mean_absolute_error(...)` | Computes average absolute miss | MAE is the easiest metric to explain in business units: "we are off by X on average." |
| `np.sqrt(mean_squared_error(...))` | Computes RMSE in original units | RMSE punishes large misses more heavily, which is useful when tails matter. |
| `r2_score(...)` | Measures variance explained relative to predicting the mean | `R²` is informative, but not enough to decide if the model is usable. |
| `print(...)` values | Shows metrics can all look "good" at once | Different metrics answer different questions; none should be read blindly. |

**Practice -- apply the concept:**

> Pick the better metric for each case: delivery-time prediction where average miss matters, and inventory forecasting where one huge miss is extremely costly.

> **Real-world scenario:** A demand-forecasting team optimized MAE and got a decent average error, but operations still faced stockouts because a few very large misses were disastrous. RMSE was the more faithful business metric because it exposed the pain of the tail.

## CONCEPT 3: A Good Regression Model Must Beat the Baseline and Survive Residual Analysis
<!-- cognitive job: teach the learner that regression evaluation needs baseline comparison plus pattern inspection, not one headline number -->

| Dimension | Answer |
|---|---|
| **What it is** | Baseline comparison shows whether the model learned anything; residual analysis shows where it still fails. |
| **Why it matters** | A model can beat one metric and still be systematically wrong for important ranges or segments. |
| **When to use it** | Every time you evaluate a regression model before calling it production-ready. |
| **When NOT to use it** | Do not stop at aggregate error alone; that hides structure in the failure. |
| **What juniors miss** | A model that does not beat the mean predictor has learned nothing, and a model with decent average error can still have patterned residual failures. |
| **Senior signal** | "First beat the mean baseline, then check residuals and segment-level error before trusting the model." |

### Mental Model

The baseline tells you whether the model is better than saying "everything is average." Residuals are the model's error fingerprint. A healthy regression model beats the baseline and leaves behind mostly unstructured noise, not obvious patterns.

---

> **Before reading on:** If your model cannot beat "predict the training mean for everyone," what has it actually learned?

---

### First Concrete Artifact: Baseline and Residuals

```python
import numpy as np
from sklearn.metrics import mean_absolute_error

y_train = np.array([120, 350, 80, 620, 200])
y_test = np.array([100, 400, 150])
y_pred = np.array([130, 360, 220])

baseline_pred = np.full_like(y_test, y_train.mean(), dtype=float)

baseline_mae = mean_absolute_error(y_test, baseline_pred)
model_mae = mean_absolute_error(y_test, y_pred)
residuals = y_test - y_pred
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `np.full_like(..., y_train.mean())` | Creates the mean-predictor baseline | If your model cannot beat this, it is not useful. |
| `baseline_mae` vs `model_mae` | Compares model against the simplest possible predictor | The improvement over baseline is more meaningful than the raw score alone. |
| `residuals = y_test - y_pred` | Computes actual minus predicted error | Residuals are the starting point for diagnosing bias, outliers, and heteroscedasticity. |

**Practice -- apply the concept:**

> Your model has MAE 180, and the mean baseline has MAE 210. Is that a real improvement? What else would you inspect before deployment?

> **Real-world scenario:** A pricing model beat the baseline comfortably, but residual plots showed the model consistently underpredicted high-end properties. The average metric looked acceptable, but the failure pattern would have directly hurt premium listings. The residual pattern mattered more than the headline score.

## DECISION RULES

### Regression Framing and Evaluation Framework

1. Confirm the target is truly continuous and the business needs the number itself.
2. Choose the metric before training, based on what kinds of errors hurt most.
3. Build the simplest baseline: predict the training mean for every case.
4. Train the model and compare it to baseline in business units.
5. Check residuals and segment-level errors before calling the model good.
6. Only then decide whether the model is fit for deployment.

```text
STEP 1: Is the output really continuous?
        |
        +--> no -> maybe classification or ranking
        |
        +--> yes
               |
               v
STEP 2: What error hurts most?
        |
        +--> all misses similar -> MAE
        +--> large misses much worse -> RMSE
        +--> scale-relative view needed -> MAPE
        |
        v
STEP 3: Beat the mean baseline
        |
        v
STEP 4: Check residuals + key segments
        |
        v
DECIDE IF MODEL IS GOOD ENOUGH
```

### Worked Example

**Scenario:** A retailer wants to predict next-week unit demand for each SKU.

| Step | Question | Answer in this case |
|---|---|---|
| 1 | Continuous target? | Yes. Units sold is a numeric target. |
| 2 | Best framing? | Regression, because the actual number matters for ordering inventory. |
| 3 | Best metric? | RMSE, because one large miss can create a costly stockout. |
| 4 | Baseline? | Predict the training mean demand for every SKU. |
| 5 | Extra checks? | Residual plot and error by SKU velocity tier. |
| 6 | Deployment decision? | Only if the model beats baseline and is not badly wrong on high-velocity SKUs. |

### Red Flags

| What someone says / does | Why it should raise a flag |
|---|---|
| "R² is 0.9, so we are done" | `R²` alone does not tell you if business error is acceptable. |
| "We use RMSE because it is standard" | Metric choice should reflect business cost, not convention. |
| "The model has MAE 180" | Without baseline or context, that number is meaningless. |
| "Overall error looks fine" | Aggregate metrics may hide important segment failures. |
| "The target is numeric, so it must be regression" | The real decision may actually be thresholding or ranking. |

### When to Use / When Not to Use

| Use this when | Do NOT use this when |
|---|---|
| `Regression` when the business needs a continuous estimate | when the real task is above/below threshold or ordered retrieval |
| `MAE` when all errors matter roughly equally | when large misses are disproportionately costly |
| `RMSE` when large misses should be penalized hard | when outliers should not dominate the score |
| `MAPE` when relative percentage error matters and values are not near zero | when zeros or near-zeros are common |
| `R²` as supporting context | never as the only deployment metric |

## COMMON MISTAKES

### Mistake 1: Framing a Threshold Problem as Regression

**Mistaken instinct:** "The output is a number, so this must be regression."

**Why it feels right:** Numeric outputs look naturally continuous.

**Why it fails:** Sometimes the business does not care about the exact number; it cares about whether a value crosses a cutoff or how options should be ordered.

**Better move:** Ask what action the business takes from the prediction. If it is threshold-based, it may be classification. If it is ordering-based, it may be ranking.

---

### Mistake 2: Using a Metric Because It Is Familiar

**Mistaken instinct:** "Let's use RMSE because everyone uses it."

**Why it feels right:** RMSE is common in tutorials and benchmark tables.

**Why it fails:** Different metrics teach the model to care about different error patterns. A familiar metric can still be the wrong business metric.

**Better move:** Ask whether one big miss is worse than many small misses. That answer should drive MAE vs RMSE.

---

### Mistake 3: Reporting Error Without a Baseline

**Mistaken instinct:** "The model has MAE 180, so that is our result."

**Why it feels right:** The number sounds concrete and precise.

**Why it fails:** MAE 180 could be excellent or terrible depending on the baseline. Without a naive predictor for comparison, the score is not anchored.

**Better move:** Always report model error next to the mean-predictor baseline.

---

### Mistake 4: Trusting Aggregate Metrics Without Residuals or Slices

**Mistaken instinct:** "The average error is acceptable, so the model is fine."

**Why it feels right:** A single number is easy to communicate and compare.

**Why it fails:** Aggregate metrics can hide systematic bias in certain value ranges or business segments.

**Better move:** Plot residuals and check errors by segment before deployment.

### Interview Q&A

**Q: When would you use RMSE over MAE?**

> "I use RMSE when large mistakes are disproportionately painful. Because RMSE squares each error before averaging, it punishes a single very large miss much more than MAE does. That matches businesses like inventory forecasting or cost prediction, where one huge miss can be worse than many small ones. If errors are more uniformly painful and I need something easy to explain in business units, I prefer MAE."

---

**Q: Why is R-squared not enough on its own?**

> "`R²` tells me how much variance the model explains relative to predicting the mean, but it does not tell me whether the absolute error is acceptable for the business. A model can have strong `R²` and still be unusable if the average miss is too large in dollars, units, or minutes. I use `R²` as context, not as the deployment decision by itself."

---

**Q: How do you know a regression model is good enough to deploy?**

> "First it has to beat the mean baseline. Then I want the chosen metric in business units, not just a relative score. After that I inspect residuals and important segments, because aggregate performance can hide structured failures. Only when the model beats baseline, the error is operationally acceptable, and the failure pattern looks manageable do I consider deployment."
