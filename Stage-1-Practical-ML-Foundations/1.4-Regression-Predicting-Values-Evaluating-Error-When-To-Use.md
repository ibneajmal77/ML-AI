## Why This Matters

### Business Importance
Regression is the ML task behind price prediction, demand forecasting, revenue projection, latency estimation, and customer lifetime value scoring. These are the problems where getting the number wrong has a direct, measurable dollar cost. Understanding how to evaluate and communicate regression error in business terms -- not just MSE on a leaderboard -- is one of the most practical skills in applied ML.

### System Importance
Regression outputs are numeric scores that flow into downstream decisions: budget allocation, inventory orders, threshold-based routing, pricing engines. A regression model that is technically "good" but evaluated with the wrong metric can make business decisions systematically worse. The evaluation metric is a design decision with operational consequences, not a default to accept without thought.

### Interview Importance
Regression questions appear in ML design interviews as sub-tasks inside larger problems ("estimate demand for this product," "predict LTV for this user cohort"). Interviewers care less about algorithm selection and more about whether you can choose the right error metric for the business problem, diagnose model failures from a residuals plot, and know when a regression framing is actually wrong for the problem at hand.

## Key Takeaways -- Read Before Any Project

1. **Regression predicts a continuous number. The metric must match the business cost of being wrong.** RMSE and MAE penalize errors differently. Choosing the wrong one optimizes for the wrong thing.
2. **RMSE penalizes large errors heavily. MAE does not.** If a single large error (a $10,000 forecasting miss) is far more costly than ten small errors ($1,000 each), RMSE is the right optimization target.
3. **R² is not a business metric.** An R² of 0.90 tells you nothing about whether your prediction errors are operationally acceptable. Always pair it with an absolute error metric in business units.
4. **Always beat the baseline before reporting a model.** The baseline for regression is predicting the training mean for every input. If your model cannot beat this, it has learned nothing.
5. **Residual plots reveal failures that summary metrics hide.** A model with acceptable RMSE can still have systematic bias in specific ranges -- this shows up in residuals, not in the headline number.
6. **Not every regression problem should be regression.** Predicting whether a value will be above or below a threshold is classification. Predicting a rank ordering is a ranking problem. Frame first.
7. **Log-transform the target when it spans orders of magnitude.** Revenue, prices, and counts often violate linearity assumptions and are better modeled in log space.

## PART 1 -- CORE CONCEPT

### What Regression Is

Regression is the supervised learning task of predicting a continuous numeric output.

```text
Input features  ->  Model  ->  Continuous output

User behavior features  ->  LTV model  ->  $420.50 (predicted LTV)
Past sales data         ->  Demand model ->  847 units (predicted demand)
Property features       ->  Price model  ->  $612,000 (predicted price)
Request features        ->  Latency model -> 145ms (predicted latency)
```

The output is a number on a continuous scale -- not a category, not a probability, not a rank.

### When Regression Is the Right Framing

| Business question | Regression? |
|---|---|
| How much will this customer spend? | Yes |
| Will this customer spend more than $100? | No -- classification (above/below threshold) |
| What price should we charge for this listing? | Yes |
| Is this price too high or too low? | No -- classification |
| How many units will we sell next week? | Yes (or Poisson regression for counts) |
| Which product should we recommend first? | No -- ranking |
| Will this transaction be fraudulent? | No -- classification |
| How much does fraud cost us per transaction? | Yes (conditional regression) |

**The key question:** Is the output a number on a continuous scale where the difference between 100 and 200 is the same as the difference between 500 and 600? If yes: regression. If you care about above/below a threshold: classification. If you care about ordering: ranking.

### The Baseline -- Always Build This First

Before any regression model, compute the mean predictor:

```text
Training set: LTV values = [120, 350, 80, 620, 200, ...]
Training mean LTV = $274

Baseline model: predict $274 for every single customer

Baseline MAE = mean(|actual - $274|) for every test example

If your model cannot beat this baseline, it has learned nothing.
```

A model with MAE of $180 that beats a baseline MAE of $210 is a genuine improvement. A model with MAE of $180 from a baseline MAE of $150 is worse than predicting the mean.

### The Four Core Regression Metrics

#### Mean Absolute Error (MAE)

```text
MAE = mean(|actual - predicted|)

Example:
  Actual:    [100, 200, 300, 400]
  Predicted: [90,  220, 280, 430]
  Errors:    [10,  20,  20,  30]
  MAE = (10 + 20 + 20 + 30) / 4 = 20

Business interpretation: "On average, our predictions are off by $20."
```

**Use MAE when:**
- All errors matter equally regardless of size
- Outliers in predictions or actuals should not disproportionately influence the metric
- The business interpretation "on average we're off by X" is what stakeholders care about

#### Root Mean Squared Error (RMSE)

```text
RMSE = sqrt(mean((actual - predicted)²))

Same example:
  Squared errors: [100, 400, 400, 900]
  MSE = (100 + 400 + 400 + 900) / 4 = 450
  RMSE = sqrt(450) ≈ 21.2

Now add one large outlier:
  Actual:    [100, 200, 300, 400, 100]
  Predicted: [90,  220, 280, 430, 500]  <- one very wrong prediction
  RMSE ≈ 183.7   <- blown up by the outlier
  MAE  ≈ 88.0    <- less affected
```

**Use RMSE when:**
- Large errors are disproportionately costly (one $10,000 miss is worse than ten $1,000 misses)
- Optimization: gradient-based optimizers work naturally with squared loss
- Comparing models: RMSE is in the same units as the target, making it interpretable

**The practical difference:**

```text
Inventory forecasting:
  Being off by 1,000 units once = stockout = $50,000 lost revenue
  Being off by 100 units ten times = minor adjustments = $5,000 cost

-> RMSE is the right metric here. Penalize the big miss heavily.

Customer satisfaction score prediction (1-5 scale):
  Off by 1.0 is equally bad whether the actual score is 2.0 or 4.0

-> MAE is the right metric here. All errors are equally undesirable.
```

#### Mean Absolute Percentage Error (MAPE)

```text
MAPE = mean(|actual - predicted| / |actual|) * 100%

Example:
  Actual:    [100, 200, 1000]
  Predicted: [90,  220,  950]
  Errors:    [10,  20,    50]
  MAPE = mean([10%, 10%, 5%]) = 8.3%
```

**Use MAPE when:**
- Scale independence matters (comparing forecast error across products with very different price ranges)
- Relative error is what stakeholders understand ("we're off by X% on average")

**MAPE failure cases:**
- When actual values can be zero or near-zero (divides by zero)
- When negative values are possible
- When a 10% error on a $10 item and a 10% error on a $10,000 item should not be weighted equally

#### R-Squared (R²)

```text
R² = 1 - (sum of squared residuals / total sum of squares)
   = 1 - (variance unexplained / total variance)

R² = 1.0: model perfectly explains all variance
R² = 0.0: model explains nothing more than predicting the mean
R² < 0:   model is worse than predicting the mean (possible)
```

**Why R² is dangerous as a standalone metric:**

```text
Scenario A:
  Model predicts house prices with R² = 0.92
  Average prediction error: $180,000
  Business verdict: useless for real estate pricing

Scenario B:
  Model predicts daily coffee sales with R² = 0.72
  Average prediction error: 3 cups
  Business verdict: operationally excellent
```

R² tells you what fraction of variance is explained relative to the mean. It says nothing about whether your absolute errors are acceptable in business terms. Always pair R² with MAE or RMSE.

### Residual Analysis -- Where Summary Metrics Fail

A residual is the difference between actual and predicted value: `residual = actual - predicted`

A well-behaved regression model has residuals that are:
- centered around zero (no systematic bias)
- uniformly distributed across the range of predictions
- not showing any pattern

```text
RESIDUAL PLOT SHAPES AND THEIR MEANING:

Good (random scatter around zero):
  residual
     |    .  .
     |  .    .  .
  0 -+------------- predicted value
     |  .  .   .
     |    .
  -> model is working correctly

Systematic bias (curved pattern):
  residual
     |.
     |  .
  0 -+-------.---
              .
               .
  -> model underestimates at high values (missing non-linearity)
  -> add polynomial features or use a non-linear model

Heteroscedasticity (funnel shape):
  residual
     |           . .
     |         .    .
  0 -+--.---.-------
     |
     |           . .
  -> error variance grows with predicted value
  -> common in revenue/price data -> try log-transforming the target

Outlier cluster:
  residual
     |
  0 -+---.--.------
     |
     |           *  <- outlier
  -> may indicate a data segment with a fundamentally different pattern
  -> investigate: different product category, different customer segment?
```

### Log-Transforming the Target

Many real business targets are right-skewed and span orders of magnitude:
- Revenue: a few customers spend $100,000+; most spend $50-200
- House prices: from $80,000 to $15 million
- API latency: mostly 50-200ms but occasional 10,000ms spikes

```text
Raw revenue distribution:
  0    500    1000   5000   50000   500000
  ████████████████▓▓▒▒░          .      .

Log revenue distribution:
  1.7  2.7   3.0   3.7   4.7    5.7
  ████████████████████████████████

Linear regression on log(target) + exp(prediction) = predicted target
```

Training a linear model on log-transformed target improves performance when the relationship between features and target is multiplicative rather than additive. It also reduces the influence of extreme outliers on the loss.

## PART 2 -- PRODUCTION REALITY

### Real Use Cases by Metric Choice

| Domain | Problem | Right metric | Reason |
|---|---|---|---|
| Retail | Weekly demand forecasting | RMSE | Stockout from under-forecast is catastrophic |
| Real estate | Property price estimation | Median AE or MAPE | Outliers (mansions) should not dominate evaluation |
| SaaS | Customer LTV prediction | MAE in dollars | Equal weight on all errors; stakeholders want dollar figure |
| Cloud | Resource cost forecasting | MAPE | Scale-independent; comparing across services |
| Operations | Delivery time estimation | MAE | Consistent accuracy regardless of distance |
| Finance | Revenue forecast | RMSE | Large misses have outsized business consequences |

### Design Patterns

**Pattern 1 -- Predict a transformed target, report in original units**
Train on `log(price)`, evaluate on `exp(predicted_log_price)` vs actual price. Report errors in dollars, not in log-dollars. Stakeholders need interpretable numbers.

**Pattern 2 -- Pair a summary metric with a distribution**
Never report only MAE = $180. Report: "MAE = $180, but the 95th percentile error is $620, meaning 5% of predictions are off by more than $620." The distribution of errors is often more actionable than the average.

**Pattern 3 -- Segment error analysis before launch**
Compute MAE separately for high-value, mid-value, and low-value targets. A model with good overall MAE can still be systematically wrong for one segment. This is the regression equivalent of the accuracy-on-balanced-class problem.

**Pattern 4 -- Establish prediction intervals, not just point estimates**
A point estimate says "LTV = $420." A prediction interval says "LTV is between $280 and $620 with 90% confidence." For decisions like budget allocation, the interval matters more than the point.

### Day-to-Day Team Reality

A data science team builds a demand forecasting model with RMSE of 120 units. The operations team integrates it into their inventory system. Three months later, operations reports that stockouts are actually worse than before the model. Investigation reveals:

1. The model has RMSE 120 on the test set, but the 90th percentile error is 480 units -- the model is occasionally very wrong on the exact items that drive stockouts (high-velocity SKUs).
2. The evaluation was done on a random sample of all SKUs, equally weighted. High-velocity SKUs account for 15% of the dataset but 80% of stockout risk.
3. The correct evaluation should have weighted errors by SKU velocity, not averaged them uniformly.

The headline metric looked fine. The slice that mattered was broken.

### Common Mistakes

- **Using R² as the business metric without absolute error context.** R² of 0.85 on a revenue model that's off by $500,000 per prediction is not good.
- **Reporting mean without distribution.** A model with MAE $180 that has 10% of predictions off by $2,000 will cause real problems that the mean hides.
- **Optimizing RMSE when MAE is the right business metric.** The model learns to minimize squared error, which biases predictions toward the mean to avoid large penalties -- this can be wrong for your use case.
- **Not evaluating on the segment that matters most.** Aggregate metrics mask systematic failures on the most critical slices.
- **Forgetting to beat the baseline.** Reporting MAE without comparing it to the mean-predictor baseline makes the number uninterpretable.

### Failure Modes

| Failure | Cause | Detection |
|---|---|---|
| Predictions correct on average, wrong in tails | Model optimized MAE, high-value tail is underfit | Plot predicted vs actual by value decile |
| Revenue forecast consistently biased low | Target right-skewed; linear model pulled toward median | Plot residuals vs predicted; try log-transform |
| Model excellent in evaluation, noisy in production | Temporal distribution shift in key features | Monitor feature means and model output distribution after launch |
| Model worse than predicting the mean | Insufficient data, wrong features, or leakage removed | Check baseline performance first |
| Confidence interval too narrow | Prediction interval not calibrated | Calibrate intervals with held-out data |

## PART 3 -- ENGINEERING

### Regression Pipeline with Correct Evaluation

```python
import numpy as np
import pandas as pd
from sklearn.ensemble import GradientBoostingRegressor
from sklearn.linear_model import LinearRegression
from sklearn.metrics import mean_absolute_error, mean_squared_error, r2_score
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler


def evaluate_regression(
    y_true: np.ndarray,
    y_pred: np.ndarray,
    label: str = "Model",
) -> dict[str, float]:
    """
    Report the full set of regression metrics with business-readable output.
    Never report just one metric.
    """
    mae  = mean_absolute_error(y_true, y_pred)
    rmse = np.sqrt(mean_squared_error(y_true, y_pred))
    r2   = r2_score(y_true, y_pred)
    p95  = np.percentile(np.abs(y_true - y_pred), 95)
    pct_within_10 = np.mean(np.abs(y_true - y_pred) / np.maximum(np.abs(y_true), 1e-8) < 0.10) * 100

    print(f"\n{'='*50}")
    print(f"  {label}")
    print(f"{'='*50}")
    print(f"  MAE:              {mae:>10.2f}")
    print(f"  RMSE:             {rmse:>10.2f}")
    print(f"  R²:               {r2:>10.3f}")
    print(f"  95th pct error:   {p95:>10.2f}  (5% of predictions worse than this)")
    print(f"  Within 10%:       {pct_within_10:>9.1f}%  (predictions within 10% of actual)")
    print(f"{'='*50}")

    return {"mae": mae, "rmse": rmse, "r2": r2, "p95": p95, "pct_within_10": pct_within_10}


def baseline_mean_predictor(y_train: np.ndarray, y_test: np.ndarray) -> dict[str, float]:
    """
    The baseline every regression model must beat.
    Predicts training mean for every test example.
    If your model cannot beat this: it has learned nothing.
    """
    y_pred_baseline = np.full_like(y_test, fill_value=y_train.mean(), dtype=float)
    return evaluate_regression(y_test, y_pred_baseline, label="Baseline (mean predictor)")

# > Analogy: The baseline is the "random guessing" equivalent for regression.
#   A model that cannot beat it has no predictive signal -- just a fancier
#   way of predicting the average.

# > Common mistake: Reporting a model's MAE of $180 without noting that the
#   baseline MAE is $210. The $30 improvement over baseline is the real number.
#   Without the baseline, $180 is meaningless.

# > Used in Project P1: Every metric you report in your churn or ticket
#   classifier also needs a baseline comparison. For any regression component,
#   the mean predictor is that baseline.
```

### Residual Analysis in Code

```python
import matplotlib
matplotlib.use("Agg")  # non-interactive backend for server/CI use
import matplotlib.pyplot as plt
import numpy as np


def plot_residuals(
    y_true: np.ndarray,
    y_pred: np.ndarray,
    title: str = "Residual Analysis",
    save_path: str | None = None,
) -> None:
    """
    Two essential residual plots:
    1. Residuals vs predicted value -- reveals systematic bias and heteroscedasticity
    2. Distribution of residuals -- reveals skew and outliers
    """
    residuals = y_true - y_pred

    fig, axes = plt.subplots(1, 2, figsize=(12, 5))
    fig.suptitle(title)

    # Plot 1: residuals vs predicted
    axes[0].scatter(y_pred, residuals, alpha=0.4, s=15)
    axes[0].axhline(0, color="red", linewidth=1.5, linestyle="--")
    axes[0].set_xlabel("Predicted value")
    axes[0].set_ylabel("Residual (actual - predicted)")
    axes[0].set_title("Residuals vs Predicted")

    # What to look for:
    # - Random scatter around zero line: good
    # - Curve: model missing non-linearity
    # - Funnel shape: heteroscedasticity -> try log-transform
    # - Cluster far from zero: outlier segment needing separate model

    # Plot 2: residual distribution
    axes[1].hist(residuals, bins=50, edgecolor="white", linewidth=0.5)
    axes[1].axvline(0, color="red", linewidth=1.5, linestyle="--")
    axes[1].axvline(residuals.mean(), color="orange", linewidth=1.5, label=f"mean={residuals.mean():.1f}")
    axes[1].set_xlabel("Residual")
    axes[1].set_ylabel("Count")
    axes[1].set_title("Residual Distribution")
    axes[1].legend()

    # What to look for:
    # - Centered at zero: no systematic bias
    # - Roughly symmetric: errors are balanced
    # - Heavy right tail: model underestimates high values
    # - Heavy left tail: model overestimates high values

    plt.tight_layout()
    if save_path:
        plt.savefig(save_path, dpi=150, bbox_inches="tight")
    plt.close()

# > Analogy: The residual plot is the EKG of your regression model.
#   The headline number (MAE) is the heart rate. The plot shows the rhythm.
#   A "normal" heart rate can still have an abnormal rhythm.

# > Common mistake: Reporting only MAE/RMSE and never plotting residuals.
#   Heteroscedasticity, systematic bias, and outlier clusters are
#   invisible in summary statistics but immediately obvious in plots.

# > Used in Lesson 1.14 (Error Analysis): Residual analysis by segment
#   is the regression version of slice-based debugging. The same discipline
#   of "what slice is the model failing on?" applies here.
```

### Log-Transform Pattern

```python
import numpy as np
import pandas as pd
from sklearn.metrics import mean_absolute_error


def train_with_log_target(
    X_train: pd.DataFrame,
    y_train: pd.Series,
    X_test: pd.DataFrame,
    y_test: pd.Series,
    model,
) -> dict[str, float]:
    """
    Train on log(target), evaluate in original units.
    Use when target is right-skewed or spans orders of magnitude.
    """
    # Fit on log-transformed target
    log_y_train = np.log1p(y_train)  # log1p handles zero values safely
    model.fit(X_train, log_y_train)

    # Predict in log space, convert back to original units
    log_y_pred = model.predict(X_test)
    y_pred = np.expm1(log_y_pred)    # inverse of log1p

    # Evaluate in ORIGINAL units -- not log units
    mae_original = mean_absolute_error(y_test, y_pred)

    # Compare to training on raw target
    model_raw = type(model)(**model.get_params())
    model_raw.fit(X_train, y_train)
    y_pred_raw = model_raw.predict(X_test)
    mae_raw = mean_absolute_error(y_test, y_pred_raw)

    print(f"MAE with log-transform:  {mae_original:.2f}")
    print(f"MAE without transform:   {mae_raw:.2f}")

    return {"log_transform": mae_original, "raw": mae_raw}

# > Analogy: Log-transforming is like switching from a linear ruler to a
#   logarithmic scale. When values span from $10 to $10,000,000, a $10
#   error and a $10,000 error are both "small" in percentage terms.
#   Linear regression in raw space cannot represent this correctly.

# > Common mistake: Evaluating log-transformed predictions in log units
#   and reporting log-MAE to stakeholders. Always convert back and report
#   in the original business units. Log-MAE of 0.3 means nothing to a PM.

# > Used in Lesson 1.7 (Trees and Gradient Boosting): Tree-based models
#   are less sensitive to target scale, but log-transforming the target
#   still improves calibration for skewed distributions.
```

### Segmented Error Analysis

```python
import numpy as np
import pandas as pd


def segmented_error_report(
    y_true: np.ndarray,
    y_pred: np.ndarray,
    segment_labels: np.ndarray,
    metric_name: str = "MAE",
) -> pd.DataFrame:
    """
    Compute error metrics per segment.
    A model with good aggregate MAE can still fail badly on key segments.
    Run this before declaring a model production-ready.
    """
    results = []
    unique_segments = np.unique(segment_labels)

    for seg in unique_segments:
        mask = segment_labels == seg
        seg_true = y_true[mask]
        seg_pred = y_pred[mask]

        mae  = np.mean(np.abs(seg_true - seg_pred))
        rmse = np.sqrt(np.mean((seg_true - seg_pred) ** 2))
        p95  = np.percentile(np.abs(seg_true - seg_pred), 95)
        n    = mask.sum()

        results.append({
            "segment": seg,
            "n_samples": n,
            "pct_of_total": f"{n / len(y_true):.1%}",
            "mae": round(mae, 2),
            "rmse": round(rmse, 2),
            "p95_error": round(p95, 2),
        })

    df = pd.DataFrame(results).sort_values("mae", ascending=False)
    print(df.to_string(index=False))
    return df

# segmented_error_report(y_true_arr, y_pred_arr, np.array(decile))

# > Analogy: Overall MAE is like an airline's average on-time rate.
#   A 92% on-time average sounds great until you find that 40% of flights
#   from your home airport are delayed. Segmented analysis finds the
#   "home airport" problem.

# > Common mistake: Declaring the model production-ready after evaluating
#   on the full test set without running segmented analysis. In production,
#   the stakeholders who care about a specific segment will find the failure
#   whether you look for it or not.
```

### Why Employers Care

Regression is in the ML toolkit of nearly every data-heavy product. The ability to translate "mean absolute error = $180" into a business judgment -- is that good enough to deploy? -- is what separates a data scientist from a business-impacting ML engineer. Employers also want to know that you have moved past tutorial-style evaluation (just reporting R²) toward production-grade evaluation (segmented errors, baseline comparison, distribution of errors, operational impact).

### AI/ML-Specific Job Description Signals

- "experience building forecasting models in production"
- "ability to communicate model performance to non-technical stakeholders"
- "proficiency in regression analysis and model evaluation"
- "experience with demand forecasting / LTV modeling / pricing models"

### Weak Answer vs Strong Answer

**Q: "What evaluation metric would you use for a delivery time prediction model?"**

- **Weak:** "I would use RMSE because it is the standard metric."
- **Strong:** "For delivery time prediction, I would start with MAE in minutes because it is directly interpretable -- if MAE is 12 minutes, that means on average we're off by 12 minutes, which is a business statement the logistics team can act on. I would also report the 95th percentile error, because a 30-minute miss on a 45-minute delivery is a very different customer experience from a 12-minute miss on a 4-hour delivery, and MAPE would capture that relative accuracy. I would also compute these separately by delivery zone and delivery time window, because a model that's accurate for urban next-day deliveries but bad for rural same-day deliveries will cause problems even if the headline MAE looks fine."

### What Senior Interviewers Expect
- Immediate justification of metric choice by connecting it to what the business values
- Awareness that summary metrics hide segment failures
- The baseline check as standard practice, not an optional step
- Residual analysis as a first step in model debugging, not an afterthought

### Enough vs Overkill
- **Enough:** MAE/RMSE/R² definitions with business interpretations, baseline discipline, residual analysis, log-transform intuition, segmented evaluation
- **Overkill:** Full statistical inference theory, OLS assumptions and proofs, advanced time-series regression at this stage

## PART 5 -- SENIORITY LAYER

### Seniority Story 1 -- The Failure

> "A product team shipped a customer LTV regression model with R² of 0.89 and declared it production-ready. Three months later, the growth team found that their top-tier customer acquisition campaigns were dramatically underperforming. The model was systematically underestimating LTV for users who came through enterprise referral channels -- a small segment (8% of users) but the highest-value one. The overall R² looked great because that 8% got diluted in the aggregate. The segmented MAE for enterprise-referral users was five times the overall MAE. Nobody had looked at segmented error before deployment. The campaign team had been setting acquisition budgets based on a model that was wrong for exactly the users they cared about most."

**Lesson:** Aggregate metrics are averages. Averages hide the segment that will make someone angry in three months.

### Seniority Story 2 -- The Right Call

> "We were building a price recommendation model for a marketplace. Initial RMSE was 430. A colleague said 'we need a more complex model.' I plotted residuals first. The funnel shape was unmistakable -- heteroscedasticity. Items priced above $500 had five times the variance of items priced below $100. I log-transformed the target, retrained the same linear model, and RMSE dropped to 185 on the original scale. No model complexity change. No new features. One line of code. The lesson was: look at the residuals before reaching for a more complex algorithm. The data usually tells you what is wrong."

**Lesson:** A residual plot often reveals the fix before any algorithm change is needed.

### Tradeoff Language

> "Choosing between MAE and RMSE is a business decision about the cost of large errors relative to small ones. RMSE is not more 'correct' -- it penalizes large errors more. If your business cost structure has that same asymmetry -- one big miss costs more than ten small ones -- RMSE is the right optimization target. If errors are roughly equally costly regardless of size, MAE is better. I always ask the business stakeholder: is it worse to miss by $1,000 once, or by $100 ten times? Their answer determines the metric."

### Failure Vocabulary

| Phrase | Operational meaning |
|---|---|
| "the model regresses to the mean" | Model's predictions cluster near the training mean; it is not learning the extremes |
| "residual pattern / heteroscedasticity" | Errors are not uniformly distributed; suggests a transform or model is needed |
| "segment-level failure" | Aggregate metric is acceptable but one important slice is badly wrong |
| "the baseline was not beaten" | Model performs no better than predicting the training average; learned nothing |
| "prediction interval too narrow" | Model is overconfident; stated confidence does not match actual coverage |
| "the metric does not match the loss" | Model was optimized for MSE but the stakeholder cares about MAE; outputs are miscalibrated |

### Interview Q&A

**Q: "When would you use RMSE over MAE?"**

Strong answer:
> "RMSE and MAE differ in how they treat large errors. RMSE squares each error before averaging, which means a single large error contributes disproportionately to the score -- more than ten small errors of the same total magnitude. I choose RMSE when the business cost structure has that same asymmetry: when one very wrong prediction is far more costly than many slightly wrong ones. A concrete example is inventory forecasting -- a 1,000-unit miss on a single SKU creates a stockout that loses significant revenue, while ten 100-unit misses are manageable adjustments. In that case, training to minimize RMSE pushes the model to avoid large misses, which matches the actual cost structure. For problems where all errors are equally costly regardless of size -- like predicting delivery time where a 10-minute miss at 30 minutes is similar to a 10-minute miss at 240 minutes -- I use MAE because it is more interpretable and not dominated by outliers."

---

**Q: "How do you know when a regression model is actually good enough to deploy?"**

Strong answer:
> "I check four things. First, does it beat the baseline? If it doesn't outperform predicting the training mean, it has learned nothing and should not be deployed. Second, are the errors operationally acceptable in business units? Not R² or RMSE alone -- 'on average we're off by $180 on a $500 product, and the 95th percentile error is $420' is a statement the business can evaluate. Third, does the segmented error analysis show any critical segment where the model fails badly? An aggregate metric of $180 MAE can mask a $900 MAE on your most important customer tier. Fourth, does the residual plot look well-behaved? Systematic bias or heteroscedasticity means the model is structurally wrong for some range of values, regardless of the headline metric. Only when all four are acceptable do I consider the model deployment-ready."

## Summary Mental Model

```text
THE REGRESSION EVALUATION CHECKLIST

Step 1: Frame the task correctly
  Is the output truly continuous?
  Or should this be classification / ranking?

Step 2: Choose the right metric BEFORE training
  All errors equal cost?   -> MAE
  Large errors much worse? -> RMSE
  Scale-independent needed?-> MAPE
  Never R² alone.

Step 3: Build the baseline
  Predict training mean for all test examples.
  Every metric you report must beat this.
  If not: the model has learned nothing.

Step 4: Evaluate with full picture
  MAE (average error in business units)
  RMSE (error emphasizing large misses)
  95th percentile error (tail behavior)
  % within X% (operational tolerance)

Step 5: Segment before declaring success
  By value decile
  By key business segment
  By time period
  Find the slice that is broken.

Step 6: Plot residuals
  Random scatter? -> model is well-behaved
  Curved pattern? -> add non-linearity
  Funnel shape?   -> try log-transform
  Cluster far out? -> investigate that segment


THE ONE-SENTENCE RULE:
Report your metrics in business units, with a baseline comparison,
with a segment breakdown, before declaring a regression model ready.
```
