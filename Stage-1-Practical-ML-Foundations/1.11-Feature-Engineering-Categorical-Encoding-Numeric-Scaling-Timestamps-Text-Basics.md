## CONCEPT 1: Feature Engineering Is Representation Design, Not Random Preprocessing
<!-- cognitive job: establish that features are chosen views of reality and must reflect prediction-time availability -->

| Dimension | Answer |
|---|---|
| **What it is** | Feature engineering is the process of turning raw fields into representations the model can learn from at prediction time. |
| **Why it matters** | A model can only learn from the features you expose, and a bad representation can hide signal, leak future information, or make the wrong model look weak. |
| **When to use it** | Use it in every tabular ML workflow, especially when columns are messy, mixed-type, time-dependent, or partially useful only after transformation. |
| **When NOT to use it** | Do not treat feature engineering as endless feature invention without checking prediction-time availability, model fit, and leakage risk. |
| **What juniors miss** | They often think features are just existing columns, when in practice the representation is usually the real modeling decision. |
| **Senior signal** | "A feature is only valid if it is available at prediction time and encoded in a way the model can use without cheating." |

### Mental Model

Feature engineering is like designing the dashboard for a pilot. The plane already has raw signals underneath, but what you expose on the dashboard determines what decisions the pilot can make and how safely those decisions can be made.

## CONCEPT 2: Different Feature Types Need Different Engineering Choices
<!-- cognitive job: teach that categorical, numeric, time, and text columns each have their own safe transformation logic -->

| Dimension | Answer |
|---|---|
| **What it is** | Categorical, numeric, timestamp, and text fields need different representation strategies because they carry different kinds of structure. |
| **Why it matters** | The wrong transformation can erase signal, invent false order, waste memory, or make a model family underperform unfairly. |
| **When to use it** | Use type-aware engineering whenever a dataset mixes categories, continuous values, dates, and free text. |
| **When NOT to use it** | Do not force one universal transform onto every feature type just because it is convenient in code. |
| **What juniors miss** | They often apply transformations mechanically, such as scaling everything or ordinal-encoding categories with no true order. |
| **Senior signal** | "Feature engineering should respect both the data type and the model family: what helps a linear model may be unnecessary or harmful for a tree model." |

### Mental Model

Different feature types are like different materials in construction. You do not cut steel, wood, glass, and fabric with the same tool and expect the result to stay meaningful.

---

> **Before reading on:** Why is giving categories arbitrary numbers often dangerous for linear models?

```python
import pandas as pd

df = pd.DataFrame({
    "plan_type": ["basic", "pro", "enterprise"],
    "monthly_spend": [29.0, 99.0, 499.0],
    "event_time": pd.to_datetime(["2025-01-01", "2025-01-02", "2025-01-03"]),
})

df["days_since_signup"] = (pd.Timestamp("2025-01-10") - df["event_time"]).dt.days
df["is_enterprise"] = (df["plan_type"] == "enterprise").astype(int)
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `pd.to_datetime(...)` | Turns strings into real timestamp objects. | Time features are often only useful after being converted into measurable time structure. |
| `days_since_signup = ...` | Creates an elapsed-time numeric feature. | Relative time often matters more than raw calendar strings. |
| `plan_type == "enterprise"` | Creates a clean binary category feature. | Categories usually need explicit representation rather than raw labels or arbitrary numeric IDs. |
| `.astype(int)` | Converts a boolean signal into model-friendly numeric form. | Many practical features are meaningful flags rather than complex encodings. |

**Practice -- apply the concept:**

> You have a `city_id` column with 2,000 values, a `monthly_revenue` column, and a `created_at` timestamp. Name one likely good representation choice for each.

> **Real-world scenario:** A team fed raw date strings and naive integer category IDs into a linear model, then blamed the model family for poor performance. Once the data was represented appropriately by type, the baseline improved before any algorithm change.

## CONCEPT 3: Good Feature Pipelines Preserve Meaning, Prevent Leakage, and Stay Reproducible
<!-- cognitive job: connect engineered features to safe training/serving pipelines and production consistency -->

| Dimension | Answer |
|---|---|
| **What it is** | A feature pipeline is the repeatable path that transforms raw inputs into model-ready features the same way in training and serving. |
| **Why it matters** | If feature creation is inconsistent, leakage-prone, or manually repeated, model quality in development will not survive deployment. |
| **When to use it** | Use explicit feature pipelines whenever preprocessing must be learned, repeated, or deployed with the model. |
| **When NOT to use it** | Do not rely on ad hoc notebook transformations that are hard to reproduce or impossible to apply consistently in production. |
| **What juniors miss** | They often think of feature engineering as one-time analysis work instead of part of the deployed system contract. |
| **Senior signal** | "If the model depends on the transform, the transform is part of the model and must be versioned, fitted safely, and shipped with it." |

### Mental Model

A feature pipeline is like a mold used in manufacturing. The point is not only to shape the first part correctly, but to produce the same shape every time afterward. If training and serving use different molds, the model sees different realities.

---

> **Before reading on:** Why is scaling a numeric column before the train/validation split a feature-engineering mistake, not just a coding mistake?

---

```python
from sklearn.compose import ColumnTransformer
from sklearn.feature_extraction.text import TfidfVectorizer
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import OneHotEncoder, StandardScaler
from sklearn.linear_model import LogisticRegression

preprocess = ColumnTransformer(
    transformers=[
        ("num", StandardScaler(), ["monthly_spend", "days_since_signup"]),
        ("cat", OneHotEncoder(handle_unknown="ignore"), ["plan_type"]),
        ("txt", TfidfVectorizer(max_features=300), "support_note"),
    ]
)

model = Pipeline([
    ("preprocess", preprocess),
    ("clf", LogisticRegression(max_iter=1000)),
])
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `ColumnTransformer(...)` | Declares different transforms for different feature types. | This is the cleanest way to keep type-aware engineering consistent and explicit. |
| `StandardScaler()` on numeric columns | Learns scaling from training data only. | For linear models, scaling often matters; fitting it outside the train split risks leakage. |
| `OneHotEncoder(handle_unknown="ignore")` | Encodes categories safely. | Serving will eventually see unseen categories, and the pipeline must fail gracefully. |
| `TfidfVectorizer(max_features=300)` | Converts text into a sparse numeric representation. | "Text basics" in tabular ML often means using compact lexical features before jumping to embeddings. |
| `Pipeline([...])` | Couples preprocessing and model into one reproducible artifact. | The transform path becomes part of the deployed model contract, not just notebook history. |

**Practice -- apply the concept:**

> A model depends on scaling, one-hot encoding, and TF-IDF. Why should these transforms live inside the same fitted pipeline as the model?

> **Real-world scenario:** A support-intelligence classifier worked well in development but drifted in production because the deployed service was not applying the exact same preprocessing as training. Once the team packaged the feature transforms inside the model pipeline, training and serving finally matched.

## DECISION RULES

### Feature Engineering Framework

1. Start by asking whether each raw column is available at prediction time.
2. Classify each usable column by type: numeric, categorical, timestamp, text, or derived aggregate.
3. Choose transformations that respect the type and the model family.
4. For linear models, pay close attention to scaling and representation quality.
5. For tree models, avoid unnecessary scaling but still engineer useful time, text, and category representations.
6. Prefer elapsed-time, flags, grouped summaries, and simple lexical features over raw messy columns.
7. Fit learned transforms only on training data.
8. Package feature transforms with the model so training and serving use the same logic.

### Worked Example

| Step | Churn prediction example |
|---|---|
| Raw fields | `plan_type`, `monthly_spend`, `signup_date`, `support_note` |
| Availability check | all available at prediction time |
| Type handling | category -> one-hot; spend -> numeric; date -> elapsed days; note -> TF-IDF or embeddings |
| Model family | logistic regression baseline needs scaling for numeric features |
| Safety | fit scaler/vectorizer on train only |
| Deployment | store preprocessing with the model pipeline |

### Red Flags

| What someone says / does | Why it should raise a flag |
|---|---|
| "We just fed the raw category IDs directly into the linear model." | That may invent false numeric ordering. |
| "We scaled everything before splitting." | That risks leakage from validation/test into training transforms. |
| "The text field is there, but we left it as a raw string." | A potentially useful feature may be effectively unused. |
| "The model file is saved, but preprocessing is in a notebook." | Training and serving may not apply the same feature logic. |

### Use / Do Not Use

| Use feature engineering this way when | Do NOT use feature engineering this way when |
|---|---|
| you need representations that preserve meaning, match the model, and survive deployment | you are inventing transformations without checking prediction-time availability, leakage, or training/serving consistency |

## COMMON MISTAKES

### Mistake 1: Treating raw columns as automatically valid features

**Mistaken instinct:** "If the column exists in the table, the model should just use it."

**Why it feels right:** Using the raw column feels simpler and more objective than transforming it.

**Why it fails:** Raw fields often contain messy formats, hidden leakage, or weak representation that the model cannot use effectively.

**Better move:** Ask whether the raw field is prediction-safe and whether its current representation exposes the signal clearly.

### Mistake 2: Encoding categories in a way that invents false numeric meaning

**Mistaken instinct:** "Turning categories into integers is enough because the model just needs numbers."

**Why it feels right:** It feels efficient and avoids wider matrices.

**Why it fails:** Integer-coded categories can imply order or distance that does not exist, especially for linear models.

**Better move:** Use one-hot, frequency, target-aware methods with care, or model-native categorical handling depending on the model family and cardinality.

### Mistake 3: Fitting transforms before the split

**Mistaken instinct:** "Scaling or vectorizing first is cleaner because then the data is ready."

**Why it feels right:** It produces a tidy preprocessed dataset early.

**Why it fails:** The transform learns from information outside the training set, which leaks validation/test information into model development.

**Better move:** Fit learned transforms on training only, usually inside a pipeline.

### Mistake 4: Forgetting that text and time fields need representation choices too

**Mistaken instinct:** "Feature engineering mostly means numeric cleanup and categorical encoding."

**Why it feels right:** Structured columns feel more obviously feature-like than timestamps or text.

**Why it fails:** Time and text are often where major signal lives, and leaving them raw or naive wastes model capacity.

**Better move:** Engineer elapsed-time, cyclical, count, lexical, or embedding-style features where the task justifies them.

### Technique Landscape

| Concept family | Main technique families | What they are for |
|---|---|---|
| categorical features | one-hot, frequency encoding, target encoding, model-native categorical handling | turn discrete labels into learnable signal |

| time features | elapsed time, calendar parts, rolling windows, cyclical encoding | turn timestamps into patterns models can use |
| text basics | counts, TF-IDF, length features, embeddings | convert raw text into structured model inputs |

### When to Use What

| If your real need is... | Usually start with... | Not with... |
|---|---|---|
| low-cardinality categories for linear models | one-hot encoding | arbitrary integer IDs |
| skewed positive numeric values | log transform or clipping review | assuming raw scale is always ideal |
| timestamps with recency signal | elapsed-time features | raw date strings |
| simple tabular text signal | TF-IDF or targeted lexical features | leaving the text column raw |

### Coding Surface

| Concept family | Common libraries / tools | Canonical API shape | Typical output |
|---|---|---|---|

### Interview Q&A

**Q: Why does scaling matter for some models but not others?**

> "Scaling matters much more for linear models, distance-based methods, and gradient-sensitive optimization because feature magnitude directly affects how the model interprets or updates weights. Tree models split on thresholds and are usually much less sensitive to absolute scale, so scaling often adds little there."

---

**Q: Why can target encoding be risky?**

> "Because it uses label information to build the feature representation. If that encoding is fitted naively across the wrong data scope, it leaks target information into validation or test behavior. It can be powerful, but it needs careful train-only fitting and usually cross-fold discipline."
