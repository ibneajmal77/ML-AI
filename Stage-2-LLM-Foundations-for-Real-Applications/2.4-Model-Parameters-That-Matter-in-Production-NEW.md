## Core Mental Model

Parameters are control knobs on the probability distribution that next-token prediction (→ 2.1) samples from at each generation step. Temperature and top-p shape the distribution. Max tokens caps how long the generation runs. Stop sequences terminate it early on a signal you define.

Each parameter answers one question:
- **Temperature:** How freely should the model sample from less probable tokens?
- **Top-p:** What fraction of the probability mass should be eligible for sampling?
- **Max tokens:** How many tokens can the response be at most?
- **Stop sequences:** What string, when produced, ends generation immediately?

These are not model settings you tune once and forget. They are task-specific decisions: the right temperature for classifying a ticket is wrong for drafting a reply. Encode them as named presets — one per task type — not as a single global config.

## Main Concepts + Production Reality

### Parameter 1 — Temperature

**What it does:**

Temperature controls how the model samples from the probability distribution over possible next tokens at each generation step.

At temperature=0, the model always picks the single highest-probability token. Given the same input, you get the same output every time. The generation is deterministic (with a caveat — see below).

At temperature=1.0, the model samples from the full distribution without modification. High-probability tokens are still more likely to be chosen, but lower-probability tokens have a real chance. The same input can produce different outputs on different runs.

At temperature above 1.0, the distribution is flattened further — even low-probability tokens become more likely. Outputs become increasingly random and often incoherent.

```
Temperature   Behavior                    Production use case
──────────────────────────────────────────────────────────────────────
0.0           Near-deterministic          Classification, extraction,
                                          structured JSON output,
                                          routing decisions

0.1 – 0.3     Highly consistent with      Summarization where consistency
              minor variation             matters, factual Q&A

0.4 – 0.7     Balanced                    Drafting suggestions where some
                                          variation is acceptable

0.7 – 1.0     Creative, varied            Open-ended drafting, brainstorming,
                                          content generation

> 1.0         Unpredictable               Rarely useful in production
```

**The default trap:**

OpenAI's API default temperature is 1.0. This means if you call the API without specifying temperature, you are running with maximum sampling variance. For a classification task, this produces inconsistent labels. For a structured JSON extraction task, this increases the chance of malformed output. For a routing decision that triggers a tool call, this adds unnecessary variability to an action.

Never leave temperature unset in a production endpoint.

**The determinism caveat:**

Temperature=0 is not guaranteed to be perfectly deterministic across all calls. Floating-point parallelism in GPU computation can introduce tiny variations. To get reproducible outputs for testing and regression, use the `seed` parameter in addition to temperature=0. With `seed` set, the model will attempt to return consistent outputs for identical inputs. This matters for evaluation (→ 2.16) and A/B testing prompts (→ 2.14).

**Failure modes:**

*Too low for the wrong task:* A drafting endpoint at temperature=0 produces the same reply suggestion every time regardless of ticket context variation. Users notice it's not tailored.

*Too high for a structured task:* An extraction endpoint at temperature=1.0 occasionally returns responses where JSON keys are missing, values are fabricated, or the structure breaks entirely — sampling variance pushed the generation off the expected pattern.

*Not set at all:* Classification inconsistency that looks like a prompt problem, model regression, or race condition — but is actually default temperature=1.0 sampling differently each call.

### Parameter 2 — Top-p (Nucleus Sampling)

**What it does:**

At each generation step, top-p restricts the token sampling pool to the smallest set of tokens whose cumulative probability exceeds p.

With top-p=1.0 (the default), all tokens are eligible — the full vocabulary. With top-p=0.9, only tokens that together account for 90% of the probability mass are eligible. The long tail of low-probability tokens is excluded.

```
top-p=1.0   → full vocabulary eligible (same as no top-p restriction)
top-p=0.9   → exclude the bottom 10% of the probability mass
top-p=0.1   → only the very top of the distribution — highly concentrated
```

**The relationship to temperature:**

Temperature and top-p both control output diversity, but through different mechanisms. Temperature reshapes the distribution. Top-p restricts the pool after reshaping.

**The production rule: tune one, leave the other at default.**

Using both temperature and top-p at non-default values is counterproductive in most cases. The effects compound and interact in ways that are hard to reason about. Pick one as your primary control:

- Most teams use temperature and leave top-p at 1.0
- Some teams prefer top-p for tasks where they want consistent vocabulary but not full determinism

The only reliable approach: pick one, set a value, hold the other at default, and evaluate the output against your actual task criteria.

**When top-p is useful:**

Top-p shines when you want to prevent extremely unlikely tokens from appearing (which temperature alone does not fully prevent at moderate values) while still allowing meaningful variation. For example: a summarization task where you want readable variation but not bizarre word choices. top-p=0.9 with temperature=0.5 is a reasonable starting point.

**Failure modes:**

*Both temperature and top-p at restrictive values:* Outputs become stiff and repetitive. The model settles on a narrow vocabulary and structure.

*top-p very low on a creative task:* The model consistently uses the same phrases and structures, regardless of input variation.

### Parameter 3 — Max Tokens

**What it does:**

`max_tokens` sets the maximum number of tokens the model will generate in the response. It is a hard ceiling — the model stops generating at that token count, even mid-sentence.

It is not a target. Setting max_tokens=500 does not produce 500-token responses. It produces responses up to 500 tokens — the model stops when it determines the response is complete, or when it hits the ceiling.

**The context budget connection (→ 2.3):**

`max_tokens` is the output reserve in your token budget:

```
context_limit = input_tokens + max_tokens
128,000       = 126,000 prompt + 2,000 max_tokens

If you don't set max_tokens explicitly, some providers default to model maximum
(e.g. 4,096 or 16,384 tokens) — this reserves a large output budget
regardless of whether the task needs it.
```

Setting max_tokens too high on tasks with short expected outputs (classification, extraction) wastes reserved context budget on every call and increases the risk of verbose, padded responses.

**Per-task guidance:**

```
Task type          Expected output length    Recommended max_tokens
──────────────────────────────────────────────────────────────────────
Classification     1 word / short JSON       50 – 100
Extraction (JSON)  Structured fields         300 – 600
Summarization      2–5 sentences             200 – 400
Drafting           1–3 paragraphs            500 – 1,200
Tool call routing  Decision + reasoning      150 – 300
```

**Failure modes:**

*Too low — truncation:* The model is generating a structured JSON response and hits the token limit mid-object. Output is unparseable. This fails silently unless you check for truncation in the response.

```python
# Always check finish_reason in the response
response = client.chat.completions.create(...)
choice = response.choices[0]

if choice.finish_reason == "length":
    # Model hit max_tokens — output is truncated
    # Do NOT parse it as complete output
    logger.warning("Response truncated by max_tokens limit")
```

*Too high — padding and cost:* The model produces the answer in 50 tokens, then adds hedging, clarifications, and repetition to fill the available space. You pay for those tokens, and your parser has to handle extraneous content.

*Not set — surprise large bills:* A task that normally returns 100 tokens occasionally produces a 4,000-token verbose explanation when temperature is high and max_tokens is at default. Multiply by 10,000 daily calls.

### Parameter 4 — Stop Sequences

**What they are:**

Stop sequences are strings you specify that tell the model to stop generating the moment it produces them. The stop string itself is not included in the output. Generation ends cleanly at that boundary.

```python
# Example: Stop on newline — ensures a single-line classification output
response = client.chat.completions.create(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Classify this ticket: ..."}],
    stop=["\n"],        # stop when model produces a newline
    max_tokens=100,
)
# Output: "billing"   (not "billing\n\nThis ticket is about a billing issue...")
```

**Practical stop sequence patterns:**

```
Stop on newline:        stop=["\n"]
  Use for: single-label classification, one-line answers

Stop on double newline: stop=["\n\n"]
  Use for: single-paragraph responses

Stop on code fence:     stop=["```"]
  Use for: stopping after a code block without including the closing fence

Stop on section marker: stop=["###", "---"]
  Use for: multi-part generation where you process one section at a time

Stop on sentinel:       stop=["[END]"]
  Use for: prompts where you tell the model to output "[END]" when done
  → Reliable because "[END]" is not natural text
```

**Why stop sequences are underused:**

Most teams rely entirely on max_tokens to control output length. Stop sequences are more precise: they stop at a semantic boundary rather than an arbitrary token count. For classification tasks, `stop=["\n"]` is strictly better than `max_tokens=20` — it stops at the natural end of the label and handles multi-word labels correctly.

**Failure modes:**

*Stop sequence appears legitimately in the output:* You set `stop=["is"]` to stop after a single-sentence answer. The word "is" appears inside the answer. Generation terminates early.

*Stop sequence set on a task that needs to cross it:* A JSON extraction task with `stop=["\n"]` truncates every JSON response after the first field (which ends with a newline in pretty-printed JSON). Always verify that your stop sequence does not appear in normal expected output.

*Using stop sequences instead of proper prompt design:* Stop sequences control where generation ends — they do not fix poorly structured prompts. If the model is producing extraneous content before the useful output, the fix is prompt structure, not a stop sequence.

---

### Putting Parameters Together — Task-Based Configuration Design

The production pattern is not a single global config. It is a named preset per task type. Each preset encodes the right configuration for that task's requirements:

```
Task type     Required properties                   Config
──────────────────────────────────────────────────────────────────────
Classification  Deterministic, short output         temp=0, max=50, stop=["\n"]
Extraction      Deterministic, structured output    temp=0, max=500
Summarization   Consistent but not rigid            temp=0.3, max=400
Drafting        Variation acceptable                temp=0.7, max=800
Routing         Deterministic, action-triggering    temp=0, max=200
```

This design makes task requirements explicit in code rather than implicit in comments or tribal knowledge. When a new engineer joins and asks "why is classification temperature 0?" — the code answers by the task name on the preset.

    Raises KeyError if task is not registered — prevents silent misconfiguration.

**The LLM client wrapper — `app/services/llm.py`:**

```python
# app/services/llm.py

import logging
from openai import AzureOpenAI, BadRequestError, RateLimitError, APITimeoutError

from app.config import settings, LLMConfig
from app.utils.tokens import count_messages_tokens

logger = logging.getLogger(__name__)

_client = AzureOpenAI(
    azure_endpoint=settings.azure_openai_endpoint,
    api_key=settings.azure_openai_key,
    api_version=settings.azure_openai_api_version,
)


def chat_with_retry(
    messages: list[dict],
    config: LLMConfig,
    task: str = "unknown",
    max_retries: int = 2,
) -> str:
    """
    Call the LLM with the given messages and config.
    Logs token usage and cost on every call.
    Retries on transient errors (rate limit, timeout) up to max_retries times.
    Raises on context-length errors — caller must enforce budget before calling.
    """
    prompt_tokens = count_messages_tokens(messages)
    
    for attempt in range(max_retries + 1):
        try:
            response = _client.chat.completions.create(
                model=settings.azure_openai_deployment,
                messages=messages,
                temperature=config.temperature,
                max_tokens=config.max_tokens,
                top_p=config.top_p,
                stop=config.stop,
                seed=config.seed,
            )
        except BadRequestError as e:
            # Context length exceeded — do not retry, caller must fix
            if "maximum context length" in str(e):
                logger.error(
                    "Context length exceeded",
                    extra={"task": task, "prompt_tokens": prompt_tokens}
                )
                raise
            raise
        except (RateLimitError, APITimeoutError) as e:
            if attempt < max_retries:
                logger.warning(
                    f"Transient error on attempt {attempt + 1}, retrying",
                    extra={"task": task, "error": str(e)}
                )
                continue
            raise
        
        choice = response.choices[0]
        
        # Log finish_reason — critical for detecting truncation
        if choice.finish_reason == "length":
            logger.warning(
                "Response truncated — increase max_tokens or reduce input",
                extra={"task": task, "max_tokens": config.max_tokens}
            )
        
        usage = response.usage
        logger.info(
            "LLM call complete",
            extra={
                "task": task,
                "prompt_tokens": usage.prompt_tokens,
                "completion_tokens": usage.completion_tokens,
                "total_tokens": usage.total_tokens,
                "finish_reason": choice.finish_reason,
            }
        )
        
        return choice.message.content or ""
    
    raise RuntimeError("Unreachable")


# ── First endpoint: classify ─────────────────────────────────────────────────
# app/services/classify.py

CLASSIFY_SYSTEM_PROMPT = """You are a support ticket routing assistant.
Classify the support ticket into exactly one of these categories:
billing, technical, account, general

Return the category label only. No explanation."""

from app.config import get_task_config

def classify_ticket(ticket_text: str) -> str:
    """
    Classify a support ticket into one of four categories.
    Uses temperature=0 and stop=["\n"] for deterministic single-label output.
    """
    config = get_task_config("classification")
    messages = [
        {"role": "system", "content": CLASSIFY_SYSTEM_PROMPT},
        {"role": "user", "content": ticket_text},
    ]
    result = chat_with_retry(messages, config, task="classification")
    return result.strip().lower()
```

**Connects to:** `get_task_config()` is called by every service function in the project — classify, extract, summarize, route, chat. The `finish_reason == "length"` check becomes the standard pattern for detecting truncation. Token logging in `chat_with_retry()` feeds the cost reporting endpoint built in 2.15.

## Key Takeaways

- **Default parameters are not production parameters.** OpenAI's default temperature is 1.0. That is correct for creative generation and wrong for every structured, deterministic task in the copilot. Always set temperature explicitly on every endpoint.

- **Temperature=0 for any task where consistency matters.** Classification, extraction, routing, structured JSON output. The cost of wrong variation in these tasks (inconsistent labels, broken JSON, wrong routing decisions) is far higher than any benefit from sampling diversity.

- **Temperature > 0 only when variation adds value.** Drafting and summarization tasks where different users benefit from different angles. Even then, 0.3–0.7 is the useful range — above 0.7 starts adding noise without useful diversity.

- **Max tokens is a budget decision, not just a limit.** Set it to the expected output size for the task — not to the model maximum. Too low truncates responses mid-structure. Too high wastes money and encourages padding. Always check `finish_reason == "length"` in production.

- **Stop sequences are precise; max tokens is blunt.** For tasks with a natural output boundary (a label, a sentence, a JSON object), stop sequences give cleaner results than relying on max_tokens alone. Use both together: max_tokens as a safety ceiling, stop sequences as the semantic terminator.

- **Tune one of temperature or top-p — leave the other at default.** Using both at non-default values compounds effects that are hard to reason about. Pick temperature as your primary knob. Leave top-p at 1.0 unless you have a specific reason.

- **Task-based presets make configuration explicit and auditable.** A named preset answers "why is this task configured this way?" directly in the code. A single global config or inline parameters scattered across files makes that question unanswerable without archaeology.

## Optional Advanced Notes

**Frequency penalty and presence penalty:** Two additional parameters available on OpenAI models. Frequency penalty reduces the probability of tokens that have already appeared (proportional to how often they've appeared). Presence penalty applies a flat reduction to any token that has already appeared at least once. Both reduce repetition. In practice, most production systems find that a well-designed prompt and appropriate temperature handle repetition — these parameters are tuned when the model falls into repetitive loops despite correct temperature settings. Default (0.0) is appropriate for the copilot's task types.

**top_k:** Some models (particularly open-source, Anthropic Claude) expose `top_k` instead of or in addition to `top_p`. Top-k restricts the sampling pool to the k most probable tokens rather than a cumulative probability threshold. Functionally similar to top-p for production purposes. Claude's API accepts `top_k` and `top_p` as separate controls.

**seed and reproducibility:** Setting `seed` makes the model attempt to return consistent outputs for identical inputs. This is valuable for regression testing (→ 2.16) and prompt A/B testing (→ 2.14) where you need to isolate whether an output difference came from the prompt change or from sampling variance. Note: the seed guarantee is best-effort, not absolute — hardware and load variations can still produce minor differences.
