## Core Mental Model

### The Two-Layer Contract

Every API call to an LLM carries two structurally different types of content:

**Layer 1 — the operator specification.** Written by the team building the system. Contains everything that is true for every call: who the model is, what it does, what the output must look like, what it must never do. Stable across calls. Updated on deploy, not per request. Lives in the system message.

**Layer 2 — the per-request input.** Written by (or on behalf of) the end user. Contains only what changes call to call: the ticket to classify, the document to extract from, the draft to generate. Dynamic. Untrusted. Lives in the user message.

These are not "the first message" and "the second message." They are two different trust and stability layers with different architectural roles. The system message is the operator's contract with the model. The user message is a single instance of work to perform under that contract.

This distinction matters for three reasons:

1. **Attention position.** The model processes the system message first. Instructions at the beginning of the context receive the highest effective attention weight (→ 2.2). Standing instructions in the system message are attended to more reliably than instructions buried in a long user message.

2. **Trust separation.** Instructions in the system message are operator-authored. Instructions in the user message are user-authored. When a user submits text that says "ignore the above and do X instead," that text is in the user turn — architecturally below the system message. The boundary reduces the blast radius of injection attempts (→ 2.11).

3. **Update independence.** A system prompt stored separately from application code can be updated without a code deploy, versioned independently, and swapped per tenant. A system prompt hardcoded in a service file requires a code change, a code review, and a redeploy to change a single word.

The cognitive model for section order follows from point 1: because attention weight is highest at the beginning and end of the context (with a "lost in the middle" dip for long content), the most critical sections of the system prompt must be at the front. The section order that works is: role → task → output format → constraints → background (optional) → examples. Output format goes third — not fifth, not buried after background — because it is the most frequently violated section and must sit in the high-attention zone.

## Main Concepts + Production Reality

### Concept 1 — The System Message vs. User Message Boundary

**What to put where:**

| Belongs in system message | Belongs in user message |
|---|---|
| Role / persona definition | The actual document, ticket, or question |
| Task description | Session-specific context (if any) |
| Output format specification | Nothing else |
| Constraints and exclusions | |
| Background knowledge | |
| Few-shot examples | |

The user message should contain exactly one thing: the per-request input, wrapped in a delimiter tag.

**The wrong pattern — everything in one user message:**

```python
messages = [
    {
        "role": "user",
        "content": f"""You are a ticket routing system. Classify into: billing, technical, account.
Return JSON: {{"category": "...", "confidence": 0.0}}

Ticket: {ticket_text}"""
    }
]
```

This works in a demo. It breaks in three ways in production: (1) instructions at the bottom of a long message receive lower attention than instructions at the beginning, (2) user content like "Ignore the classification instruction" is in the same trust space as the actual instructions, (3) updating the classification instructions requires finding and changing application code.

**The correct pattern:**

```python
CLASSIFY_SYSTEM_PROMPT = """..."""  # stable spec, lives in system.py

messages = [
    {"role": "system", "content": CLASSIFY_SYSTEM_PROMPT},
    {"role": "user",   "content": f"<ticket>\n{ticket_text}\n</ticket>"},
]
```

The system message holds everything that is true for every call. The user message holds only this call's input, wrapped in a delimiter.

**Why the model respects the boundary:**

Instruction tuning (→ 2.1) trains the model to treat the system message as the operator's context-setting layer and the user message as the human's request within that context. This is not magic — it is what the training data reflected. A well-structured system prompt works with that trained behavior. Instructions in the user turn that look like operator instructions partially work, partially fail, and are susceptible to injection because the training signal for user-turn authority is weaker and mixed.

### Concept 2 — Section Order: Attention-Aware Prompt Structure

**The six-section order:**

```
1. Role           — who the model is; domain frame; 1–3 sentences
2. Task           — the single imperative; what success looks like
3. Output format  — exact shape: key names, types, exclusions    ← goes HERE, not later
4. Constraints    — bulleted do-not rules
5. Background     — optional; supporting context; keep short
6. Examples       — last position; 1–3 worked (input, output) pairs
```

**Why this order:**

The attention weight distribution from (→ 2.2) applies inside the system prompt itself, not just across the full context. The first ~500 tokens receive the highest attention. The middle of a long system prompt receives systematically less. The last ~500 tokens recover to second-highest.

Output format is the most commonly violated section in production. Engineers often place it after a long background section — which pushes it into the low-attention middle zone. Moving it to position 3 (immediately after role and task, before any background content) is one of the highest-leverage changes in prompt reliability. When tested in production: violations on long inputs dropped when output format moved from position 5 to position 3.

**Section weight rule:**

- Role and task: keep short — these are frame-setting, not the spec itself
- Output format: be exhaustive — enumerate key names, types, edge-case handling
- Constraints: bullet points, short per bullet, most critical constraint first
- Background: if it exceeds ~150 tokens, move it to RAG (→ 2.17) instead
- Examples: place at the very end; the last thing before the user turn is attended to strongly

---

### Concept 3 — Delimiter Patterns

Delimiters are explicit structural boundaries. They appear in two places:

**Within the system prompt — between sections:**

Markdown section headers (`## Output format`, `## Constraints`) prevent the model from interpreting the start of the output format block as part of the constraint list, or the start of the examples as part of the background. For system prompts with three or more sections, headers are required.

**At the boundary between examples and live input:**

Add `---` after the last example in the system prompt. Without it, the model may interpret the user turn as one more entry in the `Input:` / `Output:` pattern — producing `Output: {…}` before the actual JSON, or treating the user's XML tag as another example input. The `---` signals: "examples are over; what comes next is live input."

**In the user turn — around user-provided content:**

Wrap all user-supplied content in XML-style tags: `<ticket>`, `<document>`, `<contract>`, `<invoice>`. The tags signal: "this is data, not instruction." They make the boundary visible to the model and create a syntactic distinction between instruction space (system message, outside tags) and data space (inside tags).

The same tag format should appear in the examples block of the system prompt. When the examples show `<ticket>` wrapping, the live user turn's `<ticket>` tag pattern-matches to what the model has already seen — reinforcing the format rather than breaking it.

**The three-delimiter rule for production system prompts:**

```
□ Markdown ## headers between sections (for prompts with 3+ sections)
□ --- after the example block (for prompts with few-shot examples)
□ <tag>...</tag> around all user-supplied content in the user turn
```

All three cost near zero to add. Each prevents a class of production failure.

### Concept 4 — The Task Injection Pattern

The copilot serves four task types: classify, extract, summarize, draft. Each needs a different system prompt. The naive approach is four separate string constants in four separate files. The production approach is a single module that loads the right prompt by task key at call time.

**Why task injection:**

1. All system prompts live in one place — easy to find, easy to update, easy to version.
2. The application code refers to task keys, not prompt strings. Service files don't contain prompt text.
3. Adding a fifth task type means adding one entry to `system.py`, not creating a new file and updating imports.
4. The prompt module is independently testable — you can verify prompt structure without making API calls.

**The pattern:**

```python
# system.py holds all system prompts, keyed by task
SYSTEM_PROMPTS: dict[str, str] = {
    "classification": CLASSIFY_SYSTEM_PROMPT,
    "extraction":     EXTRACT_SYSTEM_PROMPT,
    "summarization":  SUMMARIZE_SYSTEM_PROMPT,
    "draft":          DRAFT_SYSTEM_PROMPT,
}

def get_system_prompt(task: str) -> str:
    if task not in SYSTEM_PROMPTS:
        raise KeyError(f"No system prompt registered for task: {task!r}")
    return SYSTEM_PROMPTS[task]
```

Service files call `get_system_prompt("classification")` — they never contain prompt text.

**Prompt caching implication:** Azure OpenAI and OpenAI cache input tokens when the same system prompt prefix appears in consecutive calls. The cache hit requires the system message content to be byte-identical across calls. This makes the task injection pattern cache-friendly: the same prompt object is returned for every call with the same task key, and it is never modified at call time. Do not inject dynamic content (timestamps, request IDs, session state) into the system message — that busts the cache on every call.

```python
# app/prompts/system.py
"""
System prompt module for ticket-ops-api.

All system prompts live here. Service files load prompts by task key via
get_system_prompt(). Prompts are structured with attention-aware section order:
Role → Task → Output format → Constraints → Background → Examples.
"""

from dataclasses import dataclass, field
from typing import Optional


# ---------------------------------------------------------------------------
# Builder — enforces section order; all prompts are built through this class
# ---------------------------------------------------------------------------

@dataclass
class SystemPromptBuilder:
    """
    Builds a system prompt in the correct section order for maximum
    instruction-following reliability.

    Section order is fixed: role → task → output_format → constraints →
    background (optional) → examples.

    Output format goes third — before background — because it is the
    most frequently violated section and must sit in the high-attention zone.
    """
    role: str
    task: str
    output_format: str
    constraints: list[str]
    background: Optional[str] = None
    examples: list[tuple[str, str]] = field(default_factory=list)

    def build(self) -> str:
        parts: list[str] = []

        # Section 1: Role (frame-setting, kept short)
        parts.append(self.role.strip())

        # Section 2: Task (single imperative)
        parts.append(f"## Task\n{self.task.strip()}")

        # Section 3: Output format (most critical — position 3, never buried)
        parts.append(f"## Output format\n{self.output_format.strip()}")

        # Section 4: Constraints
        if self.constraints:
            lines = "\n".join(f"- {c}" for c in self.constraints)
            parts.append(f"## Constraints\n{lines}")

        # Section 5: Background (optional; if long, move to RAG in 2.17)
        if self.background:
            parts.append(f"## Background\n{self.background.strip()}")

        # Section 6: Examples (last position = second-highest attention)
        if self.examples:
            blocks = "\n\n".join(
                f"Input: {inp}\nOutput: {out}"
                for inp, out in self.examples
            )
            parts.append(f"## Examples\n---\n{blocks}\n---")

        return "\n\n".join(parts)

# ---------------------------------------------------------------------------

SYSTEM_PROMPTS: dict[str, str] = {
    "classification": CLASSIFY_SYSTEM_PROMPT,
    "extraction":     EXTRACT_SYSTEM_PROMPT,
    "summarization":  SUMMARIZE_SYSTEM_PROMPT,

def get_system_prompt(task: str) -> str:
    """
    Load the system prompt for a given task key.

    Called by service files at request time. The returned string is
    byte-identical across calls for the same task key, enabling prompt
    caching on Azure OpenAI and OpenAI.

    Raises KeyError if the task key is not registered.
    """
    if task not in SYSTEM_PROMPTS:
        raise KeyError(
            f"No system prompt registered for task: {task!r}. "
            f"Available tasks: {sorted(SYSTEM_PROMPTS)}"
        )
    return SYSTEM_PROMPTS[task]
```

## Key Takeaways

- **The system message and user message are different trust layers, not just two slots.** Standing instructions belong in the system message — always. User-supplied content belongs in the user message — always. Mixing them into one user message creates injection surface and under-attended instructions.

- **Section order inside the system prompt is a reliability decision.** Output format goes third — immediately after role and task — before any background content. "Lost in the middle" applies inside your system prompt exactly as it applies in RAG. Move the most critical section to position 3 and violations drop.

- **Three delimiters, three problems prevented.** Markdown headers prevent section boundary ambiguity. `---` after the example block prevents the model from echoing `Output:` into the live response. XML tags in the user turn prevent data from bleeding into instruction space. All three cost one line each to add.

- **The task injection pattern decouples prompts from code.** Service files that call `get_system_prompt("classification")` never contain prompt text. Prompt changes are in one file, independently testable, independently versionable. Adding a new task type is one dict entry, not a new file.

- **Byte-identity is the prompt caching contract.** A system prompt that is modified at call time — even by injecting a timestamp — busts the cache on every request. Keep the system message static. Put per-request context in the user message, never in the system message.

- **The example block is the last thing the model sees before the user turn.** That position has second-highest attention weight. Use it for the edge cases that instructions alone cannot fully specify — they will be well-attended.

- **`system.py` is the start of prompt infrastructure.** The `SYSTEM_PROMPTS` registry and `get_system_prompt()` loader established here become the foundation for prompt versioning (→ 2.14), where the same registry gets tagged versions, A/B routing, and rollback logic.

## Optional Advanced Notes

**Multi-tenant system prompt loading:** In production, different tenants often need different personas or allowed-topic restrictions. The task injection pattern extends naturally: instead of `SYSTEM_PROMPTS[task]`, the key becomes `(tenant_id, task)`. The `get_system_prompt()` function fetches from a config store (database or blob storage) rather than a dict. The `SystemPromptBuilder` serializes to YAML, stored as a config artifact. This enables per-tenant prompt updates without code deploys and is the foundation for the prompt registry in 2.14.

**When dynamic context must live near instructions:** Sometimes genuinely important context — like a customer's account tier or an active incident — influences classification or drafting behavior. If it belongs near the standing instructions but must be dynamic, the pattern is to place it in a second `{"role": "system", "content": "..."}` message immediately before the user turn, not to inject it into the primary system prompt string. Some APIs support multiple system messages; others treat a second system message as a privileged user message. Test vendor behavior before relying on this — and note that any injection into the primary system prompt busts caching for all calls regardless of tenant or session.
