## Where This Fits in the System

**State:** After 2.7, `extract_ticket_fields()` returns a `ParseResult[ExtractionOutput]`. Pydantic validates the schema — correct key names, correct types. But schema-valid does not mean factually correct. A ticket says "TKT-0042"; the model might extract "TKT-0024". Pydantic passes both. The database accepts both. Only the business logic notices the wrong value — downstream, after damage.

**Gap:** The extraction pipeline has no mechanism to catch factually wrong but schema-valid output. Hallucinated numbers, wrong ticket IDs, invented urgency levels — all pass schema validation and reach the database silently.

**Unlocks:** Every extraction result now carries a signal: `is_grounded=True/False` and `requires_review=True/False`. Route handlers can act on these flags — log warnings, queue for human review, or return a confidence signal to callers. 2.11 extends the same `requires_review` routing pattern to handle injection-flagged inputs.

## Core Mental Model

### The Confidence Trap

The model speaks with the same fluency and confidence whether it is copying a fact from your document or generating one it does not actually know.

There is no change in tone. No hedging. No "I'm not sure but...". The mechanism that generates "The penalty is $50,000 per day" when it read it in the source is identical to the mechanism that generates "The penalty is $5,000 per day" when it didn't read anything — it's predicting plausible tokens either way.

**The engineering conclusion:** you cannot use the model's output style as a signal of accuracy. You need an external check — compare what the model said against what the source actually contains.

## Main Concepts + Production Reality

### Concept 1 — What Hallucination Is and Why It Happens

**Scenario:**
A contract says: *"Penalty: $50,000 per calendar day."*
You ask the model to extract the penalty amount.
The model returns: `{"penalty_per_day_usd": 5000}`.
Schema valid. Format perfect. Number wrong.

**What happened step by step:**

```
1. Model reads the contract clause
2. "50,000" and "5,000" both look like plausible penalty amounts
3. The model has seen many contracts in training — the average small-contract
   penalty is closer to $5,000 than $50,000
4. Training distribution pulls the prediction toward the more common value
5. Output is confident, fluent, schema-valid — and factually wrong
```

**What this is called:** hallucination — the model producing output that is plausible and fluent but not grounded in the provided source.

**Why there is no internal warning signal:**
```
Human expert doesn't know → feels uncertainty → hedges or says "I don't know"
LLM doesn't know         → generates the most probable next token → outputs confidently
```
The model has no gate between "I have evidence for this" and "this sounds right." Both produce identical fluent output.

**Four hallucination types in production — by risk level:**

```
HIGH RISK
  Factual fabrication   — invents a specific number, date, or name
                          e.g. "$5,000" when source says "$50,000"
  Source fabrication    — invents a citation, URL, or document reference
                          e.g. "See Section 12.4" when no Section 12.4 exists

MEDIUM RISK
  Extrapolation         — real fact, extended beyond what the source says
                          e.g. "90-day notice for termination" applied to renewals
                          when source only mentions termination
  Attribute confusion   — mixes properties of two similar entities
                          e.g. TKT-0042 vs TKT-0024, Party A vs Party B

LOW RISK (but not zero)
  Classification        — bounded output, source in context
  Summarization         — source in context, but extrapolation risk remains
```

**Engineering implication:**
```
Task has a source in context  →  lower risk (model copies/paraphrases)
Task asks from model memory   →  higher risk (model generates from distribution)
Output contains specific numbers, dates, citations  →  highest risk → verify
```

**Common trap:**
Adding prompt instructions like "only state facts you are certain of." The model has no certainty signal to consult — it will still generate confidently from its distribution. Instructions reduce rate marginally; they do not solve the problem.

### Concept 2 — Detection Heuristics

**Scenario:**
Your extraction pipeline returned `penalty_per_day_usd: 5000`. You have the source document. How do you know if that number is real or invented?

**What happens step by step:**

```
Source document:  "...penalty of $50,000 (fifty thousand US dollars) per calendar day..."
Extracted value:  5000

Check 1 — Verbatim grounding:
  Normalize both → strip "$", ",", lowercase
  Search "5000" in normalized source → not found → is_grounded = False
  → flag for review

Check 2 — Numeric presence:
  Extract all digit sequences from claim: ["5000"]
  Extract all digit sequences from source: ["50000", "10"]
  "5000" not in source numbers → confirms ungrounded
```

**What these checks are called:** grounding checks — comparing extracted claims against the source to verify support.

**Three heuristics that work in production:**

```
1. Verbatim grounding (cheapest — string search)
   → Extract numeric values, dates, quoted strings from the output
   → Search for each in the normalized source text
   → Not found → flag as ungrounded
   Cost: near zero. Best for: numbers, dates, ticket IDs.

2. LLM-as-judge (most accurate — second API call)
   → Send: source + extracted claim to a second model call
   → Ask: "is this claim supported, contradicted, or not mentioned?"
   → Returns: verdict + evidence quote
   Cost: one extra API call per check. Best for: paraphrased claims,
   inferences, high-stakes legal/financial extraction.

3. Self-consistency (detects instability — N API calls)
   → Run the same extraction 3–5 times at temperature > 0
   → Compare the extracted value across runs
   → High variance → hallucination risk signal
   → Low variance → stability (not proof of correctness)
   Cost: N extra API calls. Best for: generation without a source,
   when you have no document to check against.
```

**Engineering implication:**
```
Numeric/date field in extraction  →  run verbatim grounding (cheap, fast)
High-stakes field (legal, financial)  →  run LLM-as-judge on failures
No source document available  →  run self-consistency; flag high-variance results
```

**Common trap:**
Running verbatim grounding on inferred/boolean fields (`is_paid`, `has_arbitration_clause`). These are inferences — they will not appear verbatim in the source. Verbatim grounding produces false positives that destroy trust in the detection layer. Apply verbatim grounding only to specific factual values.

### Concept 3 — The Three-Tier Mitigation Stack

**Scenario:**
Your extraction pipeline is returning wrong values 4% of the time. You cannot eliminate hallucination — but you can make it a manageable, measurable risk instead of a silent one.

**What the three tiers do:**

```
Tier 1 — GROUNDING (before generation)
  Problem:  model doesn't have the fact → infers from training distribution
  Fix:      put the source in the prompt → model copies/paraphrases instead of inferring
  How:      RAG (retrieve relevant docs, inject before generation)
            Tool calling (fetch real data at generation time — 2.8)
  Effect:   dramatically reduces hallucination rate for knowledge-dependent tasks

Tier 2 — CONSTRAINING (at generation time)
  Problem:  model has freedom to invent values
  Fix:      reduce output degrees of freedom
  How:      Structured outputs / Pydantic schema (2.7) — model can't fabricate
              a field that doesn't exist in the schema
            Enums for constrained choices (2.8) — model can't invent team names
            Null-for-missing constraint — "if not present, return null, not a guess"
            temperature=0 — reduces run-to-run variance (not rate, but consistency)
  Effect:   reduces the surface area for hallucination

Tier 3 — VERIFICATION (after generation)
  Problem:  grounding + constraints still miss some failures
  Fix:      check the output against the source after generation
  How:      grounding check (Concept 2)
            LLM-as-judge
            Human review queue for flagged outputs
  Effect:   surfaces failures that tiers 1 and 2 didn't catch
            → routes to review instead of silently passing wrong values through
```

**The key principle for Tier 3:**
```
Flagged output → route to human review
              → NOT: reject and block

A detection layer that blocks all uncertain output
creates more friction than the hallucinations it prevents.
```

**Engineering implication:**
```
No single tier eliminates hallucination
Each tier catches what the previous missed
Start with Tier 1 (grounding) — it has the highest impact
Add Tier 2 (constraints) — no extra cost if you're already using structured outputs
Add Tier 3 (verification) — targeted at the highest-risk field types only
```

**Common trap:**
Treating hallucination as a prompt engineering problem and skipping to instructions like "do not fabricate." This is Tier 2 at best — and a weak version of it. The correct order is Tier 1 first. If the source is in the context, most hallucination problems reduce dramatically before you write a single constraint.

**Skeleton:**

```python
# app/services/extract.py  (updated from 2.7 version)
"""
Extraction service with grounding check.

After schema validation (2.7), a grounding check verifies that key extracted
values — ticket_id, urgency — appear in the source ticket text.

Every result carries:
  is_grounded: bool       — all checked fields found in source
  requires_review: bool   — True when is_grounded=False or high-risk flags present
  ungrounded_fields: list — which fields failed the check
"""

import logging
import re
from dataclasses import dataclass, field

from app.prompts.system import get_system_prompt
from app.prompts.templates import extract_user_message
from app.schemas import ExtractionOutput, ParseResult, parse_llm_output

logger = logging.getLogger(__name__)


# ---------------------------------------------------------------------------
# Grounding check utility
# ---------------------------------------------------------------------------

def _normalize(text: str) -> str:
    """Lowercase, collapse whitespace, strip punctuation."""
    text = text.lower()
    text = re.sub(r"[,.$%()'\":\-]", " ", text)
    return " ".join(text.split())


def grounding_check(value: str, source_text: str) -> dict:
    """
    Checks whether a specific value appears in the source text.

    Applies two checks:
      1. Normalized verbatim match — value present as substring after normalization
      2. Numeric token match — all digit sequences in value present in source
         (catches "$50,000" vs "50000" formatting differences)

    Returns:
        is_grounded: bool
        match_type: "verbatim" | "numeric" | "not_found"
    """
    norm_source = _normalize(source_text)
    norm_value  = _normalize(str(value))

    # Check 1: normalized verbatim match
    if norm_value and norm_value in norm_source:
        return {"is_grounded": True, "match_type": "verbatim"}

    # Check 2: all numeric tokens in value present in source
    nums_in_value  = re.findall(r"\d+", str(value))
    nums_in_source = re.findall(r"\d+", source_text)
    if nums_in_value and all(n in nums_in_source for n in nums_in_value):
        return {"is_grounded": True, "match_type": "numeric"}

    return {"is_grounded": False, "match_type": "not_found"}


# ---------------------------------------------------------------------------
# Grounded extraction result
# ---------------------------------------------------------------------------

@dataclass
class GroundedExtractionResult:
    """
    Wraps ParseResult[ExtractionOutput] with grounding verification.

    Callers use requires_review to decide routing:
      False → safe to write to database
      True  → log warning, queue for human review or retry
    """
    parse_result: ParseResult[ExtractionOutput]
    is_grounded: bool
    ungrounded_fields: list[str]
    requires_review: bool

    @property
    def ok(self) -> bool:
        return self.parse_result.ok

    @property
    def data(self) -> ExtractionOutput | None:
        return self.parse_result.data


# Fields to apply grounding check to — specific factual values, not inferences
GROUNDED_FIELDS = ["ticket_id"]


def extract_ticket_fields(ticket_text: str) -> GroundedExtractionResult:
    """
    Extract structured fields from a support ticket with grounding verification.

    Steps:
      1. Call LLM with extraction system prompt (via parse_llm_output from 2.7)
      2. Validate output against ExtractionOutput schema (Pydantic, from 2.7)
      3. Apply grounding check to specific factual fields
      4. Return GroundedExtractionResult with is_grounded and requires_review flags

    Returns GroundedExtractionResult — never raises.
    """
    parse_result = parse_llm_output(
        schema=ExtractionOutput,
        system_prompt=get_system_prompt("extraction"),
        user_message=extract_user_message(ticket_text),
        max_retries=1,
        temperature=0.0,
        max_tokens=300,
    )

    # If parsing failed, grounding is moot — but still return structured result
    if not parse_result.ok or parse_result.data is None:
        return GroundedExtractionResult(
            parse_result=parse_result,
            is_grounded=False,
            ungrounded_fields=["parse_failed"],
            requires_review=True,
        )

    data = parse_result.data
    ungrounded: list[str] = []

    for field_name in GROUNDED_FIELDS:
        value = getattr(data, field_name, None)
        if value is None:
            continue   # null is expected for absent fields — not a grounding failure
        result = grounding_check(str(value), ticket_text)
        if not result["is_grounded"]:
            ungrounded.append(field_name)
            logger.warning(
                "Grounding check failed: field=%s value=%r not found in source",
                field_name, value,
            )

    is_grounded = len(ungrounded) == 0
    requires_review = not is_grounded

    if requires_review:
        logger.warning(
            "Extraction requires review: ungrounded_fields=%s", ungrounded
        )

    return GroundedExtractionResult(
        parse_result=parse_result,
        is_grounded=is_grounded,
        ungrounded_fields=ungrounded,
        requires_review=requires_review,
    )
```

**Update callers** — route handlers now receive a `GroundedExtractionResult`:

```python
# In a FastAPI route handler:
result = extract_ticket_fields(ticket_text)

if not result.ok:
    return {"error": "extraction_failed", "detail": result.parse_result.error_detail}

if result.requires_review:
    # Log for human review queue — do not block, but flag
    logger.warning("Extraction flagged for review: %s", result.ungrounded_fields)

# Always return the data — review is routing, not rejection
return {
    "data": result.data.model_dump(),
    "is_grounded": result.is_grounded,
    "requires_review": result.requires_review,
}
```

**Connects to:** `parse_llm_output()` from 2.7 handles schema validation; grounding check is the layer on top. 2.11 uses the same `requires_review` routing pattern for injection-flagged inputs.

## Key Takeaways

- **Hallucination is the mechanism working correctly, not a bug.** The model predicts plausible tokens — and a plausible but wrong number is still the most probable token in many contexts. No prompt instruction changes this. Architecture does.

- **Schema-valid does not mean factually correct.** Pydantic catches wrong key names and wrong types. It does not catch `5000` where the source said `50000`. Grounding check is the layer that catches this.

- **The confidence trap is real.** Fluent, confident output carries no factual accuracy signal. High confidence language in the output ("As stated in the clause...") is generated like any other token — it is not correlated with correctness. Verify independently; never infer accuracy from tone.

- **Apply grounding checks to specific factual fields only.** Numbers, dates, IDs, quoted strings. Not inferences, not categories, not boolean flags. Verbatim grounding on inferred fields produces false positives that destroy trust in the detection layer.

- **`requires_review` is a routing signal, not a rejection gate.** A detection layer that blocks all flagged output creates more friction than the hallucinations it prevents. Flag → route to human review. Do not reject and return an error — partial results with a review flag are more useful than no result.

- **The three-tier order matters.** Grounding first (Tier 1 — highest impact), constraints second (Tier 2 — no extra cost), verification third (Tier 3 — targeted at highest-risk fields). Engineers who jump to Tier 3 without Tier 1 are building expensive detection for a problem they could have mostly prevented.

- **Self-consistency detects instability, not incorrectness.** High variance across N runs is a hallucination risk signal. Low variance means the model is stable — it does not mean the stable answer is right. A model can be consistently wrong.

## Optional Advanced Notes

**LLM-as-judge grounding for paraphrased claims:** Verbatim grounding misses cases where the model correctly paraphrased ("$50k" from "$50,000") or correctly inferred a derived value. For high-stakes extractions — legal penalties, financial terms, compliance requirements — use a second LLM call with an NLI (Natural Language Inference) prompt: provide the source and the claim, ask whether the claim is "supported", "contradicted", or "not mentioned." This catches true fabrications while reducing false positives from paraphrase. Cost: one extra API call per flagged field. Apply only to fields where verbatim grounding failed.

**Hallucination rate as a production metric:** Track `is_grounded=False` events in your logging pipeline as a ratio over total extractions. A rising grounding failure rate is a leading indicator of prompt regression — it often surfaces before output quality degrades enough for users to complain. Set an alert threshold (e.g., >3% grounding failures in a 1-hour window) and investigate prompt changes, model version changes, or input distribution shifts when it fires. This is covered in full in 2.16 (evaluation framework).
