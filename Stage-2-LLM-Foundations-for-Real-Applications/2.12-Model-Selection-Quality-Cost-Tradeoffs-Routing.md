## Objective

Design the task-to-model routing configuration for the copilot and implement `app/routing/model_map.py` — a central map that assigns each task type to the right model tier based on its quality requirements, cost constraints, and latency budget, and provides a fallback path when the primary model is unavailable.

---

## Where This Fits in the System

**State:** Every service file (`classify.py`, `extract.py`, `summarize.py`, `handlers.py`) currently uses the same hardcoded `DEPLOYMENT = "gpt-4o"` constant. One model handles everything. At low volume, this is invisible. At production volume, it means every simple ticket classification call — which needs almost no reasoning capability — pays full GPT-4o cost and latency.

**Gap:** No architectural decision was made about which tasks need the strongest model and which don't. Using the most capable (and most expensive) model for everything is not a quality strategy — it is the absence of a strategy. A copilot that classifies 1,000 tickets per hour at GPT-4o cost while a smaller model would handle classification at 80% lower cost with no quality difference has a cost structure problem, not a capability problem.

**Unlocks:** `model_map.py` decouples "which model" from "service logic." Services load their model assignment from the map. Changing a model for a specific task is a config change, not a code change. 2.13 uses this map to introduce multi-vendor routing. 2.15 uses it to apply caching and batching per model tier.

## Core Mental Model

### The Right Tool for the Right Job

A senior engineer on call at 2am uses a checklist to verify a deployment, not a full architectural review. The checklist is the right tool — fast, reliable, sufficient for the job. The architectural review is the right tool for a different job.

Model selection works the same way:

```
Simple, bounded task  →  small fast model  →  sufficient quality, low cost
Complex reasoning     →  large capable model →  necessary quality, higher cost
```

The mistake is defaulting to the most powerful model for everything — the equivalent of calling an architectural review for every checklist item. The quality ceiling you need depends on the task, not on a general preference for "the best."

## Main Concepts + Production Reality

### Concept 1 — The Size-Quality-Cost Triangle

**Scenario:**
Your copilot handles 1,000 tickets per day. 700 are classifications (billing vs. technical vs. account). 200 are extractions. 100 are complex routing decisions that need customer context reasoning. You're using GPT-4o for all 1,000. Your monthly API bill is $X. A colleague says you're overpaying.

**What's actually happening:**

```
GPT-4o handles a classification:
  Input tokens:  ~600   (system prompt + ticket)
  Output tokens: ~20    ({"category": "billing", "confidence": 0.95})
  Reasoning needed: minimal — bounded output, clear categories
  Cost per call:  ~$0.003

GPT-4o-mini handles the same classification:
  Same input, same output
  Reasoning needed: same (minimal)
  Cost per call:  ~$0.0003  (10x cheaper)
  Quality difference on bounded tasks: near zero
```

**The tradeoff in plain terms:**

```
Bigger model:
  + Better at multi-step reasoning
  + Better at ambiguous language
  + Lower hallucination rate on complex tasks
  + Better instruction-following on subtle constraints
  − 5–10x higher cost
  − 20–40% higher latency

Smaller model:
  + 5–10x cheaper
  + Lower latency
  + Sufficient quality on simple, bounded tasks
  − Degrades on complex reasoning
  − Higher hallucination rate when context is long or task is ambiguous
  − Less reliable instruction-following on nuanced prompts
```

**What this is called:** the model size vs. quality tradeoff — the relationship between model capability, cost, and latency across the model tier spectrum.

**Engineering implication:**

```
Match model tier to task complexity:

  Simple + bounded output    →  small model (GPT-4o-mini, Haiku)
  Medium complexity          →  mid-tier model (GPT-4o)
  Complex reasoning required →  large model (GPT-4o, Opus)

The boundary is defined by:
  Does quality visibly degrade when I run this task on a smaller model?
  Test it. Don't assume.
```

**Common trap:**
Assuming larger always means better for your use case. On bounded classification tasks with explicit output schemas, GPT-4o-mini often matches GPT-4o quality exactly — the reasoning ceiling is never reached. Run both, measure, decide.

### Concept 2 — Task-to-Model Matching

**Scenario:**
Your five copilot endpoints — classify, extract, summarize, route (with tools), and chat — all have different quality requirements. Which one actually needs the largest model?

**Walking through each task:**

```
/classify
  What it does:    output one of 4 categories + confidence float
  Reasoning need:  low — clear categories, structured output, bounded space
  Hallucination risk: low — classification from provided text (→ 2.10)
  Model needed:    small (GPT-4o-mini)

/extract
  What it does:    pull structured fields from a ticket
  Reasoning need:  low-medium — extract, don't infer
  Hallucination risk: medium — numeric fields need grounding check (→ 2.10)
  Model needed:    small-medium (GPT-4o-mini works, test edge cases)

/summarize
  What it does:    generate 3-5 sentences of prose
  Reasoning need:  medium — coherent synthesis across the full ticket
  Hallucination risk: medium — summarization from source, but extrapolation risk
  Model needed:    medium (GPT-4o-mini to GPT-4o depending on ticket complexity)

/route (with tool calling)
  What it does:    multi-step reasoning: fetch context, decide team, justify
  Reasoning need:  high — tool selection, multi-hop decision, justification prose
  Hallucination risk: medium — but wrong routing is a business impact
  Model needed:    large (GPT-4o)

/chat (multi-turn, Stage 2.17)
  What it does:    conversation over ticket history, context-aware responses
  Reasoning need:  high — maintain context, handle ambiguity, nuanced replies
  Model needed:    large (GPT-4o)
```

**What this is called:** task-to-model matching — assigning each task type to the model tier that meets its quality floor at the lowest acceptable cost.

**Engineering implication:**

```
Quality floor test:
  1. Run the task on the smaller model with your real ticket samples
  2. Measure: output quality, schema compliance rate, grounding check pass rate
  3. If quality is within acceptable bounds → use the smaller model
  4. If quality degrades on edge cases → upgrade that task only

Don't upgrade all tasks because one task needs a larger model.
```

**Common trap:**
Making model selection decisions from spec sheets and benchmarks instead of your own data. Public benchmarks measure performance on benchmark tasks — not your tickets, not your prompt design, not your schema. Always measure on your own representative sample before deciding.

### Concept 3 — The Model Routing Pattern

**Scenario:**
You've decided: classification uses GPT-4o-mini, routing uses GPT-4o. Right now that means editing the `DEPLOYMENT` constant in two different service files. Next month you want to test GPT-4o-mini on routing. You need to change code again.

**The problem with hardcoded deployments:**

```
Current state:
  classify.py    →  DEPLOYMENT = "gpt-4o"        ← needs to be "gpt-4o-mini"
  extract.py     →  DEPLOYMENT = "gpt-4o"        ← needs to be "gpt-4o-mini"
  summarize.py   →  DEPLOYMENT = "gpt-4o"        ← maybe "gpt-4o-mini"
  handlers.py    →  DEPLOYMENT = "gpt-4o"        ← stays "gpt-4o"

Every model change = find file → change constant → test → deploy
Every A/B test = create branch → change constants → merge → deploy
```

**The routing pattern:**

```
model_map.py holds the mapping:
  "classification" → "gpt-4o-mini"
  "extraction"     → "gpt-4o-mini"
  "summarization"  → "gpt-4o-mini"
  "routing"        → "gpt-4o"
  "chat"           → "gpt-4o"

Service files call: get_model("classification")
  → returns "gpt-4o-mini"
  → service uses that deployment name

Changing a model = change one line in model_map.py
A/B test = change model_map.py + check flag
```

**What this is called:** task-based model routing — a central configuration that maps task types to model deployments, decoupling "which model" from "service logic."

**Engineering implication:**

```
model_map.py also holds:
  Fallback model per task      → if primary deployment unavailable, use fallback
  Max tokens per task          → task-specific output budget
  Temperature per task         → (already in config.py from 2.4, but model can override)

Benefits:
  Single place to change models across all tasks
  Enables per-task A/B testing without code changes
  Fallback path prevents full outage when one deployment is down
  2.13 (vendor comparison) can swap entire provider here
```

**Common trap:**
Using environment variables per task (`CLASSIFY_DEPLOYMENT`, `EXTRACT_DEPLOYMENT`, ...) instead of a central map. Works for 2 tasks. Becomes unmanageable at 10 tasks across multiple environments. The central map is the right abstraction.

## Project Integration

**Component:** `app/routing/model_map.py` — task-to-model configuration and `get_model()` accessor.

**Build:** Model config dataclass, task map with primary and fallback deployments, `get_model()` function, and updates to all service files to load their deployment from the map.

**Skeleton:**

```python
# app/routing/model_map.py
"""
Task-to-model routing configuration for ticket-ops-api.

Each task is assigned a primary model tier and a fallback.
Services call get_model(task) instead of using a hardcoded DEPLOYMENT constant.

Changing a model for a task = change one entry in TASK_MODEL_MAP.
No service file changes required.
"""

import os
from dataclasses import dataclass


@dataclass(frozen=True)
class ModelConfig:
    """
    Model assignment for a single task type.

    primary:   the deployment to use under normal conditions
    fallback:  the deployment to use if primary is unavailable or returns an error
    max_tokens: output token budget for this task (overrides service-level default)
    """
    primary: str
    fallback: str
    max_tokens: int


# ---------------------------------------------------------------------------
# Model tier constants — edit these when deployments change
# ---------------------------------------------------------------------------

# Small model — sufficient for bounded, low-reasoning tasks
# ~10x cheaper than GPT-4o, similar quality on constrained outputs
_SMALL  = os.environ.get("MODEL_SMALL",  "gpt-4o-mini")

# Standard model — good balance of quality and cost for medium-complexity tasks
_MEDIUM = os.environ.get("MODEL_MEDIUM", "gpt-4o")

# Large model — required for multi-step reasoning, tool use, ambiguous tasks
_LARGE  = os.environ.get("MODEL_LARGE",  "gpt-4o")


# ---------------------------------------------------------------------------
# Task-to-model map
# ---------------------------------------------------------------------------

TASK_MODEL_MAP: dict[str, ModelConfig] = {
    # Classification: bounded output, 4 categories, schema-enforced
    # GPT-4o-mini matches GPT-4o quality on this task at 10x lower cost
    "classification": ModelConfig(
        primary=_SMALL,
        fallback=_MEDIUM,
        max_tokens=80,
    ),

    # Extraction: structured field pull from ticket text
    # Small model sufficient; grounding check (2.10) catches failures
    "extraction": ModelConfig(
        primary=_SMALL,
        fallback=_MEDIUM,
        max_tokens=300,
    ),

    # Summarization: prose generation, coherent synthesis
    # Start with small; upgrade if quality degrades on long tickets
    "summarization": ModelConfig(
        primary=_SMALL,
        fallback=_MEDIUM,
        max_tokens=400,
    ),

    # Routing: multi-step tool use, customer context reasoning, justification
    # Needs larger model — wrong routing has direct business impact
    "routing": ModelConfig(
        primary=_LARGE,
        fallback=_MEDIUM,
        max_tokens=600,
    ),

    # Draft: professional response generation, tone control
    # Medium model — quality of prose matters; small models drift on tone
    "draft": ModelConfig(
        primary=_MEDIUM,
        fallback=_SMALL,
        max_tokens=500,
    ),

    # Chat (added in 2.17): multi-turn, context-aware conversation
    # Largest available — context management and nuanced replies require it
    "chat": ModelConfig(
        primary=_LARGE,
        fallback=_MEDIUM,
        max_tokens=800,
    ),
}


def get_model(task: str) -> ModelConfig:
    """
    Return the ModelConfig for a given task.

    Raises KeyError if the task is not registered.
    Services use config.primary for the deployment name.
    Services fall back to config.fallback on primary deployment errors.

    Usage:
        config = get_model("classification")
        deployment = config.primary     # "gpt-4o-mini"
        max_tokens = config.max_tokens  # 80
    """
    if task not in TASK_MODEL_MAP:
        raise KeyError(
            f"No model config for task: {task!r}. "
            f"Registered tasks: {sorted(TASK_MODEL_MAP)}"
        )
    return TASK_MODEL_MAP[task]


def get_deployment(task: str) -> str:
    """Shortcut: return just the primary deployment name for a task."""
    return get_model(task).primary
```

## Key Takeaways

- **Bigger model ≠ better result for your task.** On bounded classification with explicit schemas, a small model often matches a large model exactly — the reasoning ceiling is never reached. The quality gap only appears when the task requires multi-step reasoning, handles long ambiguous context, or demands nuanced judgment.

- **Test on your own data, not on benchmarks.** Public benchmark scores reflect benchmark tasks. Your tickets, your prompt design, your schema constraints determine which model is sufficient. Run both on your representative sample. The answer often surprises.

- **Match cost to task complexity.** Simple bounded tasks (classify, extract with schema) → small model. Medium prose generation (summarize, draft) → test small first, upgrade only if quality degrades. Multi-step reasoning with tools (route, chat) → large model. This can cut API costs by 50–70% with no user-visible quality change.

- **Hardcoded deployment constants are a maintenance trap.** Every model change, A/B test, or vendor switch requires hunting through service files. A central `model_map.py` is the right abstraction — change one line, all services update.

- **Fallback is not optional.** Model deployments go down. A routing config without fallbacks turns a partial outage into a full one. Every task should have a fallback that degrades gracefully rather than hard-failing.

- **Model selection is a decision, not a default.** Defaulting to the largest model is not a quality strategy — it is the absence of a decision. Document it in an ADR: what task needs what model and why. When quality or cost requirements change, you have a record to update, not a codebase to grep.

## Optional Advanced Notes

**Model routing with feature flags:** In production, `get_model()` can check a feature flag store (LaunchDarkly, Unleash, or a simple Redis key) to return different models for different request cohorts — enabling A/B testing where 10% of classification requests use a new model while 90% continue on the current one. The flag check goes inside `get_model()` before reading `TASK_MODEL_MAP`. This is the same abstraction boundary exploited in 2.14 for prompt versioning.

**Cost estimation before routing:** For high-volume pipelines, you can estimate the input token count (using `tokens.py` from 2.3) before selecting a model and route to a smaller model when the input is short enough that the quality gap is minimal. Long, complex inputs with many extracted fields → route to the stronger model. Short single-field extractions → route to the smaller one. This is dynamic routing based on input characteristics rather than static task-type routing — it is more complex to implement but can reduce costs further on variable-length inputs.
