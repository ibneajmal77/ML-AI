## Objective

Build `app/prompts/registry.py` — a prompt version store for the Internal Copilot that registers named, versioned prompts, serves the active version to service calls, and supports one-line rollback to the previous version when a prompt change degrades output quality.

---

## Where This Fits in the System

**State:** After 2.13, the copilot has a working multi-provider pipeline. Model selection is driven by `model_map.py` and provider routing by `providers.py`. The remaining hardcoded element is the prompts: `CLASSIFY_SYSTEM_PROMPT` lives as a string literal in `classify.py`. Extraction and summarization prompts live the same way in their service files.

**Gap:** There is no way to change a prompt in production without a code deployment. There is no record of which prompt was active when a specific ticket was processed. If a prompt change degrades quality — classification accuracy drops, summaries become vague — the only recovery path is reverting a commit and redeploying. That takes minutes, rolls back everything in that commit (not just the prompt), and leaves no trace of the broken version for later diagnosis.

**Unlocks:** Services query the registry by name (`get_registry().get_active("classification_system")`). A prompt change is a registry update — no deployment required. If quality degrades, `rollback()` restores the previous version in milliseconds. 2.15 uses the `content_hash` of the active version as a stable prefix key for caching. 2.16 uses `get_history()` to enumerate versions when running regression tests.

## Core Mental Model

### Prompts Are Configuration, Not Code

Your database connection string lives in `.env`. Your model tier assignments live in `model_map.py`. Your API credentials are environment variables. These are all configuration — values that can change between environments and over time, stored separately from the code that uses them.

But your prompts are strings hardcoded in service files:

```
classify.py:  CLASSIFY_SYSTEM_PROMPT = "You are a support ticket..."
extract.py:   EXTRACT_SYSTEM_PROMPT  = "Extract the following fields..."
```

They behave like code because they live in code files. But they are not code. They are configuration — values that determine output behavior and change at a different rate and for different reasons than application logic.

The consequence of treating them as code:

```
Application logic change cadence:  weeks to months (features, refactors, bug fixes)
Prompt change cadence:             hours to days    (quality tuning, A/B tests, fixes)

When these are forced to be the same thing:
  Every prompt tweak   →  code review  →  deploy cycle   →  minutes of risk
  Every rollback       →  git revert   →  redeploy        →  rolls back unrelated changes
  Every incident       →  "which prompt was running at 2:47pm Tuesday?"  →  no answer
```

A prompt registry decouples these two cadences. Prompts live in the registry. Code imports names, not strings. The registry manages versions, active states, history, and rollback.

## Main Concepts + Production Reality

### Concept 1 — The Problem with Hardcoded Prompts

**Scenario:**
Your classification accuracy is at 94%. On Thursday morning you update `CLASSIFY_SYSTEM_PROMPT` in `classify.py` to add a few examples and tighten the output constraint. You deploy at 10am. By 11am, accuracy has dropped to 81% — monitoring shows a spike in `"general"` classifications for tickets that should be `"billing"`. You need to fix this immediately.

**What your options look like without a registry:**

```
Option 1: git revert and redeploy
  1. Identify the exact commit hash
  2. git revert → open PR → get review → merge
  3. Trigger deployment pipeline
  4. Wait 5–10 minutes for build + deploy
  ✗  Also reverts every other change bundled in that commit
  ✗  You might not have the exact original string saved anywhere
  ✗  No trace left of what v2 said — gone from the file

Option 2: manually restore the old string and redeploy
  1. Open the file, try to reconstruct the previous string from memory
  2. Another full deploy cycle
  ✗  You may not have the exact original text
  ✗  Another 5–10 minutes of degraded production
  ✗  Still no record of when the change happened
```

**What neither option gives you:**
- The verbatim text of the previous prompt
- A timestamp for when the change went live
- A log of which prompt version processed which specific ticket
- A rollback path that does not require touching source code

**What this pattern is called:** the "config as code" antipattern — embedding configuration (values that change at operational speed) inside source code (which changes at development speed).

**Engineering implication:**

```
Separate by change rate:
  Application logic   →  source code   (deploys with the app)
  Prompts             →  registry      (updates without a deploy)
  Model assignments   →  model_map.py  (config file, low change overhead)
  Credentials         →  .env          (never in source)

When configuration lives in code:
  Every config change  =  code change  =  deploy cycle  =  blast radius risk
  Every rollback       =  code rollback  =  rolls back unrelated work too
  Every incident       =  no version history  =  no clear evidence trail
```

**Common trap:**
Treating prompts as "just strings" that don't warrant operational discipline. Strings that determine LLM output behavior, change weekly, and directly affect user-facing quality are configuration. The format (Python string vs. JSON vs. database row) does not change this — the change cadence and operational impact do.

### Concept 2 — What Makes a Prompt Version

**Scenario:**
You want to store two versions of the classification system prompt. Three weeks after the incident, you need to answer: what was the exact text of v1, when did v2 go live, and did v1 and v2 use different wording or just different whitespace?

**The data you need per version:**

```
PromptVersion:
  name          →  which prompt ("classification_system", "extraction_system")
  version       →  integer, auto-incremented per name (1, 2, 3 ...)
  content       →  full prompt text, stored verbatim — no diff, no compression
  content_hash  →  SHA-256 of content (truncated to 16 hex chars)
  registered_at →  UTC timestamp — when this version was created
  is_active     →  bool — is this the version currently served to services

Why each field:

  name          →  services look up prompts by name, not by file path
                   "classification_system" survives file renames and refactors

  version       →  lets you reference and pin a version explicitly
                   "roll back to v1" is unambiguous

  content       →  verbatim storage makes restoration exact
                   diffs are unreadable and require reconstruction logic

  content_hash  →  SHA-256 of content = stable identity for the exact text
                   if two "different" versions share a hash, they are identical
                   used as a prefix cache key in 2.15 (→ 2.15)
                   detects silent in-file edits: stored hash ≠ current file hash → drift

  registered_at →  audit trail: "this version went live at 14:03 on Thursday"
                   combined with ticket logs: "ticket #4821 ran with v2"

  is_active     →  exactly one version per name should be active at a time
                   this flag is the operational state the registry manages
```

**What this is called:** a prompt version record — the complete, immutable identity of a specific prompt text at a specific point in time.

**Engineering implication:**

```
Store full content, not diffs:
  System prompts (→ 2.6) are typically under 2,000 tokens (→ 2.3)
  Full-text storage is cheap. Reconstruction from diffs is fragile.
  Verbatim content + hash = always recoverable

The content hash is more stable than the version number:
  If you rebuild the registry from scratch (migrate to a new store),
  version numbers may shift. The hash of "You are a support ticket..." 
  is always the same, regardless of which version slot it occupies.

Primary key is always (name, version):
  v3 of "classification_system" ≠ v3 of "extraction_system"
  Never use version number as a global primary key
```

**Common trap:**
Storing only the latest version. If you keep one row per prompt name and overwrite it on each update, you have no rollback target and no audit trail. Every version must be preserved, including deactivated ones.

### Concept 3 — The Registry Pattern

**Scenario:**
`classify.py` imports a hardcoded string. You want it to load the active classification prompt from a central store instead — so changing the prompt is a registry operation, not a file edit.

**What changes:**

```
Before (hardcoded):
  classify.py:
    CLASSIFY_SYSTEM_PROMPT = "You are a support ticket classification assistant..."

    def classify_ticket(text):
        messages = [{"role": "system", "content": CLASSIFY_SYSTEM_PROMPT}, ...]

After (registry-backed):
  classify.py:
    from app.prompts.registry import get_registry

    def classify_ticket(text, request_id=None):
        prompt_ver = get_registry().get_active("classification_system",
                                               request_id=request_id)
        messages = [{"role": "system", "content": prompt_ver.content}, ...]
```

**The registry's four core operations:**

```
register(name, content)
  →  creates next version for that name (v1, v2, ...)
  →  hashes the content
  →  deactivates previous active version for this name
  →  marks new version active
  →  returns PromptVersion

get_active(name)
  →  returns the PromptVersion where is_active=True for that name
  →  logs: name, version, hash, request_id  (prompt observability)
  →  raises PromptNotFoundError if no active version — fails loudly

set_active(name, version)
  →  deactivates all versions for this name
  →  activates the specified version number
  →  used for explicit rollback or for promoting a tested version

rollback(name)
  →  finds current active version → deactivates it
  →  activates the version immediately before it
  →  logs: "rolled back classification_system from v2 to v1"
  →  raises PromptNotFoundError if there is no previous version
```

**What this is called:** a prompt registry — a centralized store that manages named prompt versions and serves the current active version to every caller.

**Engineering implication:**

```
Services depend on names, not strings:
  "classification_system" → whatever version is currently active
  Updating the registry changes what every service receives on the next call
  No code change, no deploy, no restart required

The registry must fail loudly when a name is not registered:
  If get_active() silently falls back to a hardcoded string,
  the registry becomes optional → teams stop seeding it →
  the fallback becomes the real behavior → the registry is useless

Seed the registry at startup:
  scripts/seed_prompts.py loads all v1 prompts into the registry
  Idempotent: skips names that already have versions registered
  This is the first entry in version history — the v1 baseline
```

**Common trap:**
Building the registry but keeping the hardcoded string as a `try/except` fallback when `get_active()` raises. A fallback that is always available makes the registry optional. Services will silently use the hardcoded string the moment anyone forgets to seed the registry. Remove the fallback. Make missing prompt registration a startup failure, not a runtime default.

### Concept 4 — Rollback

**Scenario:**
You registered `classification_system` v2 with rewritten examples. Accuracy dropped from 94% to 81%. You need to restore v1 without touching source code or triggering a deployment.

**What rollback means in the registry:**

```
Before rollback:
  name = "classification_system"
  v1:  is_active = False   (was active before v2 was registered at 14:03)
  v2:  is_active = True    (current — causing degraded output)

registry.rollback("classification_system")
  →  scans versions for the current active (v2, at index 1)
  →  deactivates v2:  v2.is_active = False
  →  activates v1:    v1.is_active = True
  →  logs: "rolled back classification_system from v2 to v1"
  →  returns v1

After rollback:
  v1:  is_active = True    ← services now receive v1 on the next call
  v2:  is_active = False   ← preserved, not deleted — available for diagnosis
```

**Why you never delete deactivated versions:**

```
v2 may have been better on 90% of inputs, worse on a specific edge case
You need v2's content to understand what broke
The eval framework (→ 2.16) can replay golden tickets through both v1 and v2
  → "these 12 tickets regressed on v2: billing tickets with discount codes"
  → now you understand the failure, fix v3, register it, test before activating

The version history IS your incident record:
  v1 registered at: 2024-01-10 09:00
  v2 registered at: 2024-01-11 14:03
  v2 rolled back:   2024-01-11 14:47
  → v2 was active for 44 minutes
  → these are the ticket IDs processed in that window
  → run them through v1 and v2 side by side to find the delta
```

**What this is called:** prompt rollback — restoring the previous active version of a named prompt without a code change or deployment.

**Engineering implication:**

```
Rollback is immediate:
  set_active() or rollback() changes the active flag in-memory
  Services get the restored version on their next get_active() call
  No deploy, no PR, no waiting — recovery in < 1 second

Rollback does not fix the underlying problem:
  It restores the previous state while you diagnose the failed version
  After rollback:
    1. Pull all ticket IDs logged against v2 (from prompt observability logs)
    2. Run them through v1 and v2 with the eval framework (→ 2.16)
    3. Identify the failure pattern
    4. Register v3 with the fix
    5. Test v3 against golden dataset before activating in production

Never auto-rollback based on error rate alone:
  A bad prompt may return valid JSON with wrong category labels — no error raised
  Quality degradation needs eval-based detection (→ 2.16), not error rate thresholds
  Auto-rollback on error rate will miss quality regressions and trigger on unrelated errors
```

**Common trap:**
Rolling back without diagnosing the failed version. A rollback that is not followed by analysis means you will deploy the same broken prompt again later under a different version number. Treat every rollback as an incident — preserve v2, understand it, fix v3.

### Concept 5 — Prompt Observability

**Scenario:**
Support escalates a ticket processed on Tuesday at 2:47pm. The extracted `priority` field is `"low"` when it should be `"critical"`. You need to know: was this a model failure or a prompt failure? And which extraction prompt was active at 2:47pm?

**Without version logging:**

```
You know:
  The ticket text
  The wrong output

You do not know:
  Which prompt version ran on this ticket
  Whether the extraction prompt changed that day
  Whether this is isolated or systematic
  Whether reverting the prompt would fix it
```

**With version logging in get_active():**

```
Every get_active() call emits a structured log line:
  {
    "event":        "prompt_served",
    "name":         "extraction_system",
    "version":      2,
    "content_hash": "a3f9b2c1...",
    "request_id":   "req-4821",
    "ts":           "2024-01-09T14:47:23Z"
  }

Now you can answer in seconds:
  → ticket req-4821 ran with extraction_system v2
  → v2 was registered at 14:03 that day (44 min before the incident)
  → roll back to v1, re-run the same ticket, compare output
  → if output changes → the prompt caused the regression
  → if output is still wrong → investigate the model or the ticket data
```

**What this is called:** prompt observability — structured logging that links each LLM call to the exact prompt version that produced it.

**Engineering implication:**

```
Log at get_active() call time, not at service startup:
  Startup logs show what was registered when the process started
  Call-time logs show what ran on a specific request

Include at minimum: name, version, content_hash, request_id, timestamp
  content_hash is more stable than version number (→ Concept 2)
  request_id links the prompt log to the application request log

Prompt version logs feed directly into the eval framework (→ 2.16):
  "Re-run all requests that used extraction_system v2"
  → query logs for request_ids where name="extraction_system" and version=2
  → replay each ticket through v1 and v2
  → compare outputs systematically
```

**Common trap:**
Logging only the version number without the content hash. If the registry is rebuilt or migrated to a new backing store, version numbers may be reassigned. The content hash `"a3f9b2c1"` always identifies the same prompt text regardless of version number, environment, or when the registry was seeded.

```python
# app/prompts/registry.py
"""
Prompt version registry for ticket-ops-api.

Stores named, versioned prompt texts. Services query by name and receive
the currently active version. Rollback restores the previous version
without a code change or deployment.

Storage: in-memory dict. In production, swap the backing store by
subclassing PromptRegistry and overriding _store persistence.
Seed at startup via scripts/seed_prompts.py or seed_default_prompts().
"""

import hashlib
import logging
from dataclasses import dataclass, field
from datetime import datetime, timezone

logger = logging.getLogger(__name__)


class PromptNotFoundError(KeyError):
    """Raised when no active version exists for a prompt name,
    or when a requested version number does not exist."""


@dataclass
class PromptVersion:
    """
    A single versioned snapshot of a named prompt.

    content_hash: first 16 hex chars of SHA-256(content).
                  Stable identity across registry migrations.
    is_active:    True if this is the version currently served by get_active().
                  Exactly one version per name should be active at any time.
    """
    name:          str
    version:       int
    content:       str
    content_hash:  str
    registered_at: datetime
    is_active:     bool = False

    def __repr__(self) -> str:
        status = "ACTIVE" if self.is_active else "inactive"
        return f"<PromptVersion {self.name!r} v{self.version} [{status}] hash={self.content_hash}>"


def _hash_content(content: str) -> str:
    """Return first 16 hex chars of SHA-256 of the prompt content."""
    return hashlib.sha256(content.encode()).hexdigest()[:16]


class PromptRegistry:
    """
    In-memory prompt version store.

    _store: dict[name, list[PromptVersion]] — ordered by version number (ascending).
    """

    def __init__(self) -> None:
        self._store: dict[str, list[PromptVersion]] = {}

    # ------------------------------------------------------------------
    # Registration
    # ------------------------------------------------------------------

    def register(self, name: str, content: str) -> PromptVersion:
        """
        Register a new version of a named prompt.

        Auto-increments the version number for this name.
        Deactivates the previous active version and marks the new version active.
        Returns the new PromptVersion.
        """
        versions = self._store.setdefault(name, [])

        # Deactivate all existing versions for this name
        for v in versions:
            v.is_active = False

        new_version = PromptVersion(
            name=name,
            version=len(versions) + 1,
            content=content,
            content_hash=_hash_content(content),
            registered_at=datetime.now(timezone.utc),
            is_active=True,
        )
        versions.append(new_version)

        logger.info(
            "Registered prompt version: name=%s version=%d hash=%s",
            name, new_version.version, new_version.content_hash,
        )
        return new_version

    # ------------------------------------------------------------------
    # Query
    # ------------------------------------------------------------------

    def get_active(self, name: str, *, request_id: str | None = None) -> PromptVersion:
        """
        Return the active PromptVersion for the given name.

        Logs name, version, content_hash, and request_id for prompt observability.
        Raises PromptNotFoundError if no active version exists.

        Do NOT add a fallback to a hardcoded string — fail loudly.
        Missing prompt registration should surface immediately, not silently.
        """
        for v in reversed(self._store.get(name, [])):
            if v.is_active:
                logger.debug(
                    "Prompt served: name=%s version=%d hash=%s request_id=%s",
                    name, v.version, v.content_hash, request_id,
                )
                return v

        raise PromptNotFoundError(
            f"No active version for prompt {name!r}. "
            f"Seed the registry with: registry.register({name!r}, content)"
        )

    def get_version(self, name: str, version: int) -> PromptVersion:
        """
        Return a specific version of a named prompt (active or inactive).
        Used by the eval framework to test historical versions.
        """
        for v in self._store.get(name, []):
            if v.version == version:
                return v
        raise PromptNotFoundError(
            f"Version {version} not found for prompt {name!r}. "
            f"Available: {[v.version for v in self._store.get(name, [])]}"
        )

    def get_history(self, name: str) -> list[PromptVersion]:
        """
        Return all versions of a named prompt, sorted by version number ascending.
        Includes inactive versions — version history is never deleted.
        """
        return list(self._store.get(name, []))

    def list_prompts(self) -> list[str]:
        """Return all registered prompt names, sorted alphabetically."""
        return sorted(self._store.keys())

    # ------------------------------------------------------------------
    # Active version control
    # ------------------------------------------------------------------

    def set_active(self, name: str, version: int) -> PromptVersion:
        """
        Pin a specific version as the active version for a named prompt.

        Deactivates all other versions for that name.
        Use for explicit rollback, A/B test promotion, or pinning a tested version.
        """
        versions = self._store.get(name)
        if not versions:
            raise PromptNotFoundError(
                f"No versions registered for prompt {name!r}."
            )

        target = None
        for v in versions:
            if v.version == version:
                target = v
            else:
                v.is_active = False

        if target is None:
            raise PromptNotFoundError(
                f"Version {version} not found for prompt {name!r}. "
                f"Available: {[v.version for v in versions]}"
            )

        target.is_active = True
        logger.info(
            "Active version set: name=%s version=%d hash=%s",
            name, target.version, target.content_hash,
        )
        return target

    def rollback(self, name: str) -> PromptVersion:
        """
        Restore the immediately previous version of a named prompt.

        Deactivates the current active version and activates the one before it.
        Does NOT delete the deactivated version — it remains in history.

        Raises PromptNotFoundError if:
          - No versions are registered for this name
          - The current active version is already the earliest version
          - No active version can be found (inconsistent state)
        """
        versions = self._store.get(name, [])
        if len(versions) < 2:
            raise PromptNotFoundError(
                f"Cannot roll back {name!r}: "
                f"only {len(versions)} version(s) registered — need at least 2."
            )

        # Find the index of the current active version
        active_idx = None
        for i, v in enumerate(versions):
            if v.is_active:
                active_idx = i
                break

        if active_idx is None:
            raise PromptNotFoundError(
                f"No active version found for {name!r}. "
                f"Use set_active() to establish a baseline before rolling back."
            )

        if active_idx == 0:
            raise PromptNotFoundError(
                f"Cannot roll back {name!r}: version 1 is the earliest version."
            )

        from_version = versions[active_idx]
        to_version   = versions[active_idx - 1]

        from_version.is_active = False
        to_version.is_active   = True

        logger.warning(
            "Prompt rolled back: name=%s from_version=%d to_version=%d",
            name, from_version.version, to_version.version,
        )
        return to_version


# ---------------------------------------------------------------------------
# Module-level singleton accessor
# ---------------------------------------------------------------------------

_registry: PromptRegistry | None = None


def get_registry() -> PromptRegistry:
    """
    Return the module-level PromptRegistry singleton.

    The registry is empty until seeded. Call seed_default_prompts()
    (from scripts/seed_prompts.py) at application startup, or in test
    fixtures using a fresh PromptRegistry() instance directly.
    """
    global _registry
    if _registry is None:
        _registry = PromptRegistry()
    return _registry
```

```python
# scripts/seed_prompts.py
"""
Seed the prompt registry with v1 system prompts for all copilot tasks.

Run at startup or during initialization:
    python -m scripts.seed_prompts

Idempotent: skips names that already have at least one version registered.
Prompts defined here are the v1 source of record — edit and re-register
to create v2 rather than modifying these strings directly.
"""

from app.prompts.registry import get_registry

# v1 prompt content — one entry per named prompt used by the copilot.
# These match the quality bar from 2.5 (→ 2.5) and 2.6 (→ 2.6).
PROMPTS_V1: dict[str, str] = {
    "classification_system": (
        "You are a support ticket classification assistant. "
        "Classify the ticket into exactly one of: billing, technical, account, general. "
        "Return JSON only: {\"category\": \"...\", \"confidence\": 0.0}. "
        "No explanation. No prose."

        "Return only valid JSON matching the schema provided. "
        "If a field is not present in the ticket, return null for that field. "
        "Do not infer values not explicitly stated."
    ),
    "summarization_system": (
        "You are a support ticket summarization assistant. "
        "Summarize the ticket in 3–5 sentences. "
        "Cover: the core issue, any steps the user has already taken, "
        "and the user's expected outcome. "
        "Be direct. No filler phrases. No apologies."
    ),
    "routing_system": (
        "You are a support ticket routing assistant. "
        "Use the available tools to look up customer context, then route the ticket "
        "to the correct support team. "
        "Justify your routing decision in one sentence. "

def seed_default_prompts() -> None:
    """
    Register all v1 prompts into the registry for any name not yet registered.
    Safe to call multiple times — skips existing names.
    """
    registry = get_registry()
    existing = set(registry.list_prompts())

    for name, content in PROMPTS_V1.items():
        if name not in existing:
            pv = registry.register(name, content)
            print(f"  Seeded:  {name}  →  v{pv.version}  (hash={pv.content_hash})")
        else:
            print(f"  Skipped (already registered): {name}")


if __name__ == "__main__":
    print("Seeding prompt registry with v1 prompts...\n")
```

```python
# app/services/classify.py — updated to use prompt registry

import logging
from app.config import get_task_config
from app.services.llm import chat_with_retry
from app.prompts.registry import get_registry, PromptNotFoundError

logger = logging.getLogger(__name__)

_VALID_LABELS = {"billing", "technical", "account", "general"}


def classify_ticket(ticket_text: str, request_id: str | None = None) -> str:
    """
    Classify a support ticket using the active classification_system prompt.

    Loads the current active prompt version from the registry.
    Logs the version used at call time — for attribution if output degrades.
    Falls back to "general" and logs a warning on unexpected model output.
    """
    config = get_task_config("classification")

    # Load active prompt from registry — fails loudly if not seeded.
    # Do not add a try/except fallback here: a missing registry entry is a
    # configuration error that should surface immediately, not silently.
    prompt_ver = get_registry().get_active(
        "classification_system", request_id=request_id
    )

    messages = [
        {"role": "system", "content": prompt_ver.content},
        {"role": "user",   "content": ticket_text},
    ]
    raw = chat_with_retry(messages, config, task="classification")
    label = raw.strip().lower()

    if label not in _VALID_LABELS:
        logger.warning(
            "Unexpected classification label — falling back to 'general'",
            extra={
                "raw_output":     label[:100],
                "prompt_version": prompt_ver.version,
                "prompt_hash":    prompt_ver.content_hash,
            },
        )
        return "general"

    return label
```

## Key Takeaways

- **Prompts are configuration, not source code.** They change at operational speed (hours to days), not development speed (weeks to months). Configuration embedded in source code creates coupling: every prompt change requires a code change, a review, and a deployment. Separate them — the registry is the right boundary.

- **Rollback is the operational justification for versioning.** Version history is useful for audits. But the reason you build a registry is so that when a prompt change degrades production quality, you can restore in milliseconds — not minutes of deploy pipeline. Build rollback first; the rest follows.

- **Never delete a version.** Rollback deactivates a version — it does not remove it. The deactivated version is evidence. The eval framework (→ 2.16) can replay tickets through any historical version to isolate exactly where a regression appeared and why.

- **Content hash is more stable than version number.** Version 2 of the classification prompt in one environment is not the same as version 2 in another if they were seeded from different sources. The SHA-256 hash of the content is a stable identity. Use it for cache keys (→ 2.15) and cross-environment comparisons.

- **Log the version at call time, not at startup.** Startup logs tell you what was registered. They do not tell you what ran on ticket #4821 at 2:47pm Tuesday. Log `name`, `version`, and `content_hash` at every `get_active()` call. This is the data you need when a specific ticket produces wrong output hours after a prompt change.

- **Do not add a fallback to a hardcoded string when the registry raises.** A silent fallback makes the registry optional. Services will use the hardcoded string whenever the registry is not seeded, and no one will notice until there is an incident. Make missing registration a startup failure — fail loudly so the problem is found before the first request is processed.

## Optional Advanced Notes

**A/B prompt routing:** `get_active()` can be extended to accept a `variant` parameter and use weighted random selection to return v1 to 90% of requests and v2 to 10%. This enables controlled prompt experiments without a deployment: register v2, mark it as the "experiment" variant, observe quality metrics per cohort, then promote or discard. This pattern requires the eval framework from 2.16 to measure results objectively — without eval, you have no principled way to decide which variant wins.

**Prompt compilation vs. storage:** Store prompt templates with placeholders (`"Classify the ticket: {ticket_text}"`) — not compiled prompts with variables filled in. If you store compiled versions (post-substitution), every call produces a unique content string, the hash changes on every request, and prefix caching (→ 2.15) becomes impossible because no two requests share the same prefix. Always store templates; compile at call time by passing the ticket text in the user message, not by embedding it in the system prompt string.
