## CONCEPT 1: Sampling Parameters — Temperature and Top-P
<!-- cognitive job: establish what temperature and top-p do to the token probability distribution, correct the most common misconceptions, and give a clear decision rule for when each value is appropriate -->

| Dimension | Answer |
|---|---|
| **What it is** | Temperature and top-p are parameters that modify the probability distribution over tokens before sampling — temperature scales the sharpness of the distribution, top-p restricts sampling to the most probable token pool |
| **Why it matters** | These parameters directly control the consistency, creativity, and safety of outputs — a wrong setting makes a classification endpoint unreliable or a creative endpoint repetitive |
| **When to use it** | Set temperature explicitly on every production call — never rely on the API default; set top-p when you want an additional ceiling on output surprise beyond what temperature already provides |
| **When NOT to use it** | Do not combine very low temperature with very low top-p simultaneously — they both restrict output in similar ways and the combination makes outputs overly repetitive without providing additional quality benefit |
| **What juniors miss** | They believe temperature controls "how smart" the model is — that lower temperature makes the model more accurate. Temperature controls randomness in sampling, not factual accuracy. A model set to `temperature=0` will consistently produce the same wrong answer if the most probable token sequence is wrong. |
| **Senior signal** | "Temperature is a task-type decision, not a quality knob. For deterministic tasks — classification, extraction, structured output — I use zero. For generation tasks where variety is acceptable or desired — summarization, drafting — I use 0.2–0.5. I never set it above 0.7 in production unless the task explicitly requires creative variation." |

### Mental Model

Temperature is a volume dial on the probability distribution — not on the model's intelligence.

At `temperature=0`, the dial is turned to zero: the model always picks the single highest-probability token. The output is fully deterministic — the same input always produces the same output. This is not "smarter" — it is more consistent. If the highest-probability token happens to be wrong, the model will consistently produce the wrong answer.

At `temperature=1`, the dial is at neutral: the model samples from the raw probability distribution as learned during training. Some variation is expected.

Above `temperature=1`, the dial amplifies variance: the distribution is flattened so that lower-probability tokens become more viable. Outputs become more "surprising" and less coherent — rarely the right choice in production.

Top-p (nucleus sampling) works differently: instead of scaling the whole distribution, it finds the smallest set of tokens whose cumulative probability exceeds p, then restricts sampling to only that set. `top_p=0.9` means: only consider tokens that together account for 90% of the probability mass — ignore everything in the long tail.

### How Temperature Changes Token Selection

```python
import math
from collections import Counter


def softmax(logits: list[float], temperature: float) -> list[float]:
    """
    Apply temperature scaling to logits before softmax.
    temperature < 1: sharpen distribution (more deterministic)
    temperature = 1: raw distribution
    temperature > 1: flatten distribution (more random)
    """
    if temperature == 0:
        # Greedy decoding: return probability 1.0 to the highest logit only
        max_idx = logits.index(max(logits))
        return [1.0 if i == max_idx else 0.0 for i in range(len(logits))]

    scaled = [l / temperature for l in logits]
    max_scaled = max(scaled)
    exps = [math.exp(s - max_scaled) for s in scaled]   # numerical stability
    total = sum(exps)
    return [e / total for e in exps]


# Simplified: four candidate next tokens with their raw logit scores
tokens  = ["billing", "technical", "account", "general"]
logits  = [3.2, 1.8, 0.9, 0.4]   # model's raw scores before sampling

print(f"{'Token':<12} {'temp=0':>10} {'temp=0.3':>10} {'temp=1.0':>10} {'temp=2.0':>10}")
print("-" * 55)
for i, token in enumerate(tokens):
    p0   = softmax(logits, 0.0)[i]
    p03  = softmax(logits, 0.3)[i]
    p10  = softmax(logits, 1.0)[i]
    p20  = softmax(logits, 2.0)[i]
    print(f"{token:<12} {p0:>10.3f} {p03:>10.3f} {p10:>10.3f} {p20:>10.3f}")

# Token         temp=0   temp=0.3   temp=1.0   temp=2.0
# -------------------------------------------------------
# billing        1.000      0.990      0.588      0.389
# technical      0.000      0.010      0.166      0.222
# account        0.000      0.000      0.067      0.191
# general        0.000      0.000      0.036      0.198
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `temperature=0` → `[1.0, 0.0, 0.0, 0.0]` | Fully deterministic: only the top token is sampled | Classification and extraction tasks need this — the same input must always produce the same label. Any other setting introduces noise into a system that should be deterministic. |
| `temperature=0.3` → `[0.990, 0.010, ...]` | Very sharp: top token dominates, tiny chance of alternatives | A practical setting for structured output generation where slight variation is acceptable but the primary answer must be stable. |
| `temperature=1.0` → `[0.588, 0.166, ...]` | Full raw distribution: significant probability on alternatives | Appropriate for tasks where some variation adds value — open-ended QA, summarization. The model might choose "technical" 17% of the time for this input, which is wrong for classification. |
| `temperature=2.0` → `[0.389, 0.222, 0.191, 0.198]` | Near-flat: all options have substantial probability | The "general" token (least probable) now has almost 20% chance. Production classification at this temperature would be unusable — random variation exceeds the signal. |

## CONCEPT 2: Output Control — Max Tokens and Stop Sequences
<!-- cognitive job: establish max_tokens as the output budget enforcer and stop sequences as the structural delimiter — both are mandatory for predictable production outputs -->

| Dimension | Answer |
|---|---|
| **What it is** | `max_tokens` sets a hard ceiling on the number of output tokens the model can generate; stop sequences are strings that cause generation to halt immediately when they appear in the output |
| **Why it matters** | Without `max_tokens`, output length is unconstrained — a verbose model can produce thousands of tokens where you expected fifty, driving up cost and latency. Without stop sequences for structured output, text after the expected delimiter breaks downstream parsers. |
| **When to use it** | Set `max_tokens` on every production call — it is the output side of the token budget. Use stop sequences whenever the output has a known terminal delimiter: end of JSON, end of a code block, end of a list. |
| **When NOT to use it** | Do not set `max_tokens` too low for the task — a max_tokens limit that truncates a valid response mid-sentence produces broken output that is harder to handle than a long response. Set it above the p99 expected output length with a reasonable margin. |
| **What juniors miss** | They think the model naturally stops at a reasonable length. It does not — the model generates tokens until it produces an end-of-sequence token or hits the context window limit. Without `max_tokens`, a slightly verbose prompt can trigger a very long response. |
| **Senior signal** | "I set max_tokens based on measured p99 output length for this task, with a 20% buffer. For structured output endpoints, I also set a stop sequence on the closing delimiter so the model halts immediately — this saves tokens and makes truncation detection trivial." |

### Mental Model

`max_tokens` is a hard wall for the model's output. Think of it as the amount of paper you hand the model to write on. When the paper runs out, generation stops — mid-sentence if necessary. This is why setting it correctly matters: too low and you get truncated outputs; too high (or unset) and you pay for and wait for more tokens than you need.

Stop sequences are an emergency brake. You tell the model: "If you ever produce this exact string in your output, stop immediately." This is not about length — it is about structure. For JSON outputs, the stop sequence `}` or `}\n` tells the model to halt as soon as the object closes, preventing it from appending explanatory text after the JSON that would break your parser.

The two controls are complementary: `max_tokens` prevents runaway outputs, stop sequences enforce structural correctness.

### Max Tokens and Stop Sequences in Practice

```python
from openai import AzureOpenAI

client = AzureOpenAI(
    azure_endpoint="https://your-resource.openai.azure.com/",
    api_key="your-key",
    api_version="2024-02-01",
)

# ── Pattern 1: Classification endpoint ────────────────────────────────────────
# Task: one-label output from a fixed set
# Parameters: temperature=0 (deterministic), max_tokens tight, stop on newline

classification_response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {
            "role": "system",
            "content": (
                "Classify the support ticket into exactly one category. "
                "Reply with only the category name and nothing else.\n"
                "Categories: billing, technical, account, general"
            ),
        },
        {"role": "user", "content": "My invoice shows a charge I did not authorize."},
    ],
    temperature=0,          # deterministic — same input always produces same output
    max_tokens=10,          # "billing" is 1 token; 10 is generous for any label
    stop=["\n", "."],       # halt on newline or period — prevent appended text
)

label = classification_response.choices[0].message.content.strip()
# label = "billing"


# ── Pattern 2: JSON extraction endpoint ───────────────────────────────────────
# Task: extract structured fields as a JSON object
# Parameters: low temperature, enough max_tokens for schema, stop on closing brace

extraction_response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {
            "role": "system",
            "content": (
                "Extract the fields from the support ticket and return a JSON object.\n"
                "Return ONLY the JSON. No explanation before or after.\n"
                'Schema: {"issue_type": str, "urgency": "low"|"medium"|"high", "account_id": str|null}'
            ),
        },
        {
            "role": "user",
            "content": "Account #AC-9921 has been locked. We have a board meeting in two hours.",
        },
    ],
    temperature=0.1,        # very low — slight tolerance for phrasing variation in keys
    max_tokens=150,         # schema output is bounded; 150 is generous for this schema
    stop=["}\n\n", "```"],  # stop on JSON close + blank line, or on code fence end
)

import json
raw = extraction_response.choices[0].message.content.strip()
if not raw.endswith("}"):
    raw += "}"              # defensive: reattach closing brace if stop sequence consumed it
data = json.loads(raw)
# data = {"issue_type": "account_locked", "urgency": "high", "account_id": "AC-9921"}


# ── Pattern 3: Summarization endpoint ─────────────────────────────────────────
# Task: open-ended prose output of variable length
# Parameters: moderate temperature, generous max_tokens, no stop sequence needed

summary_response = client.chat.completions.create(
    model="gpt-4o",
    messages=[
        {
            "role": "system",
            "content": (
                "Summarize the following support conversation in 2–3 sentences. "
                "Focus on the core issue and the resolution, if any."
            ),
        },
        {"role": "user", "content": "[full conversation text here]"},
    ],
    temperature=0.3,        # low-moderate: consistent output, slight natural variation
    max_tokens=200,         # 2-3 sentences ≈ 60-100 tokens; 200 gives generous headroom
    # no stop sequences — natural sentence endings are fine for prose
)

summary = summary_response.choices[0].message.content.strip()
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `temperature=0, max_tokens=10, stop=["\n", "."]` | Classification: fully deterministic, output capped at 10 tokens, halts on punctuation | A classification endpoint that produces more than one word has almost certainly gone wrong. The stop sequence catches appended text before it reaches the parser. |
| `stop=["}\n\n", "```"]` | JSON extraction: stops at end of JSON object or end of code fence | Without this, the model frequently adds "Note: ..." or explanatory text after the JSON. The stop sequence makes the output boundary predictable without needing post-processing. |
| `if not raw.endswith("}"): raw += "}"` | Defensive: reattaches `}` if the stop sequence consumed it | When `}` is in the stop sequence list, the model halts but the `}` itself is NOT included in the output. This one-line check restores it. A robust parser always handles this edge case. |
| `temperature=0.3, max_tokens=200` | Summarization: slight variation allowed, generous output budget | A summary that gets cut at 80 tokens mid-sentence is worse than one that runs to 180 tokens. Set max_tokens above expected p99 output length — not right at the average. |

## CONCEPT 3: Parameter Strategy by Task Type — The Production Decision
<!-- cognitive job: synthesize all four parameters into a practical decision framework for the most common production task types so the learner can apply this to any new endpoint immediately -->

| Dimension | Answer |
|---|---|
| **What it is** | Each LLM task type has a parameter configuration that optimizes for its specific requirements — determinism, output length, structural correctness, and sampling behavior are all task-dependent |
| **Why it matters** | Copying a parameter set from one task type to another produces silent quality failures — the classification endpoint's `temperature=0` would make a creative drafting endpoint repetitive; the creative endpoint's `temperature=0.8` would make a classification endpoint unreliable |
| **When to use it** | Define the parameter configuration at the time you define the endpoint — before any code is written for the endpoint logic itself |
| **When NOT to use it** | Do not apply a universal "good enough" parameter set across all endpoints; each has different requirements that a single configuration cannot satisfy |
| **What juniors miss** | They pick parameters empirically by trial-and-error until outputs "look right." This produces fragile configurations that work in testing but break at the edges of the input distribution. Parameters should be chosen from the task type's requirements first, then validated empirically. |
| **Senior signal** | "The first question I ask about any new LLM endpoint is: deterministic or generative? That single answer gives me temperature, and the task output shape gives me max_tokens and stop sequences. I arrive at a starting configuration from task requirements, not from experimentation." |

### Mental Model

Think of parameter configurations as recipes for different cooking tasks. You would not use the same temperature, timing, and technique for braising meat as for making soufflé — they have fundamentally different requirements. Applying the braising recipe to a soufflé produces something inedible, but nobody would expect the techniques to be universal. Parameter sets for LLM tasks work the same way: the technique must match the task's requirements, not a universal default.

The four parameters — temperature, top-p, max_tokens, stop sequences — form a configuration profile. Four common profiles cover the majority of production task types: deterministic (classification/extraction), controlled generation (QA/summarization), structured output (JSON/code), and creative (drafting/brainstorming). Knowing which profile to start from and what to adjust gives you a working configuration before writing a single prompt.

### Parameter Profiles by Task Type

```python
from dataclasses import dataclass
from typing import Optional


@dataclass
class LLMParameterProfile:
    task_type: str
    temperature: float
    top_p: float
    max_tokens: int
    stop: list[str]
    reasoning: str


PARAMETER_PROFILES: dict[str, LLMParameterProfile] = {

    # ── Profile 1: Deterministic classification ──────────────────────────────
    "classification": LLMParameterProfile(
        task_type="classification",
        temperature=0.0,
        top_p=1.0,
        max_tokens=20,
        stop=["\n", ".", ","],
        reasoning=(
            "Must be fully deterministic — same input, same label, every time. "
            "Max tokens tight to prevent elaboration. "
            "Stop on punctuation to catch any appended text before the parser sees it."
        ),
    ),

    # ── Profile 2: Structured data extraction (JSON) ─────────────────────────
    "json_extraction": LLMParameterProfile(
        task_type="json_extraction",
        temperature=0.0,
        top_p=1.0,
        max_tokens=500,
        stop=["}\n\n", "```\n"],
        reasoning=(
            "Deterministic: field values must be consistent across identical inputs. "
            "max_tokens set to cover the expected schema size with headroom. "
            "Stop sequences enforce JSON boundary — prevent appended explanation text."
        ),
    ),

    # ── Profile 3: Factual Q&A (with retrieved context) ──────────────────────
    "factual_qa": LLMParameterProfile(
        task_type="factual_qa",
        temperature=0.1,
        top_p=1.0,
        max_tokens=400,
        stop=[],
        reasoning=(
            "Very low temperature: answer must be grounded in retrieved context, "
            "not creatively extended. Slight non-zero allows natural phrasing variation. "
            "No stop sequences: prose answers end naturally."
        ),
    ),

    # ── Profile 4: Summarization ──────────────────────────────────────────────
    "summarization": LLMParameterProfile(
        task_type="summarization",
        temperature=0.3,
        top_p=1.0,
        max_tokens=300,
        stop=[],
        reasoning=(
            "Low-moderate temperature: consistent summaries with natural language variation. "
            "max_tokens set above p99 expected summary length — truncation mid-sentence "
            "is worse than a slightly long summary. No stop sequences needed."
        ),
    ),

    # ── Profile 5: Code generation ────────────────────────────────────────────
    "code_generation": LLMParameterProfile(
        task_type="code_generation",
        temperature=0.2,
        top_p=0.95,
        max_tokens=1000,
        stop=["```\n", "\n\n#"],
        reasoning=(
            "Low temperature: code correctness is more important than novelty. "
            "top_p=0.95 as an additional ceiling — prevents very unlikely token choices "
            "that would introduce syntax errors. Stop on code fence close or on "
            "new top-level comment to capture complete functions without trailing text."
        ),
    ),

    # ── Profile 6: Creative generation / drafting ─────────────────────────────
    "creative_generation": LLMParameterProfile(
        task_type="creative_generation",
        temperature=0.8,
        top_p=0.95,
        max_tokens=800,
        stop=[],
        reasoning=(
            "Higher temperature: variety and surprise are desirable, not a bug. "
            "top_p=0.95 prevents fully incoherent token choices at the extreme tail. "
            "Generous max_tokens: creative outputs vary widely in length. "
            "No stop sequences — natural endings are acceptable."
        ),
    ),

    # ── Profile 7: Conversational assistant (general chat) ───────────────────
    "conversational": LLMParameterProfile(
        task_type="conversational",
        temperature=0.5,
        top_p=1.0,
        max_tokens=600,
        stop=[],
        reasoning=(
            "Moderate temperature: natural conversational variation without instability. "
            "max_tokens enough for multi-sentence responses without runaway generation. "
            "No stop sequences — conversation ends at natural sentence boundaries."
        ),
    ),
}


def get_profile(task_type: str) -> LLMParameterProfile:
    if task_type not in PARAMETER_PROFILES:
        raise ValueError(
            f"Unknown task type: {task_type!r}. "
            f"Available: {list(PARAMETER_PROFILES.keys())}"
        )
    return PARAMETER_PROFILES[task_type]


def build_api_call_kwargs(profile: LLMParameterProfile) -> dict:
    """Convert a parameter profile to OpenAI API call kwargs."""
    kwargs = {
        "temperature": profile.temperature,
        "top_p":        profile.top_p,
        "max_tokens":   profile.max_tokens,
    }
    if profile.stop:
        kwargs["stop"] = profile.stop
    return kwargs
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `PARAMETER_PROFILES` registry | Centralizes parameter decisions by task type | When a model update changes default behavior, you update the registry — not every individual call site in the codebase. Parameter decisions are explicit and reviewable, not scattered across files. |
| `classification` profile: `temperature=0.0, max_tokens=20` | Deterministic, tightly bounded output | A classification endpoint that produces long outputs or varying labels is broken by design — the parameters enforce the correct behavior contract. |
| `code_generation` profile uses both `temperature=0.2` and `top_p=0.95` | Both sampling restrictors active | For code, a slightly wrong token can introduce a syntax error — combining low temperature with top-p provides a double constraint. This is one of the few task types where using both together adds genuine value. |
| `creative_generation` profile: `temperature=0.8` | High variation is a feature | For drafting and brainstorming, predictable outputs are a problem — the model should produce different angles on each call. The parameter configuration explicitly enables this. |
| `get_profile` raises on unknown task type | Forces explicit task type assignment | Silently falling back to a default profile is how the wrong parameters get applied. Raising on unknown types forces the caller to make a deliberate choice. |

## DECISION RULES
<!-- cognitive job: support transfer — give a practical instrument for real decisions -->

### The Parameter Configuration Decision Framework

Use this sequence when configuring any new LLM endpoint. Takes five minutes. Prevents the parameter failures described in the hook.

```text
STEP 1: Is the output deterministic or generative?
        Deterministic (same input must always produce same output):
            → temperature = 0, top_p = 1.0
            Examples: classification, extraction, routing, scoring
        
        Generative (some variation is acceptable or desirable):
            → temperature = 0.2 to 0.8 depending on creativity requirement
            Examples: summarization, drafting, Q&A, conversation
                        │
                        ▼
STEP 2: What is the output structure?
        Fixed label or short phrase:
            → max_tokens = 10–50; stop = ["\n", "."]
        
        JSON or structured schema:
            → max_tokens = enough for full schema + 20% buffer
            → stop = ["}\n\n", "```\n"] or closing delimiter
        
        Bounded prose (summary, answer):
            → max_tokens = p99 observed output length + 20% buffer
            → stop = [] (natural sentence endings are fine)
        
        Unbounded creative output:
            → max_tokens = generous ceiling (500–2000)
            → stop = [] unless a known structural delimiter exists
                        │
                        ▼
STEP 3: Does output variety matter?
        No variety needed (auditable, reproducible, downstream parsing):
            → temperature = 0; consider top_p = 1.0
        
        Slight natural variation acceptable (summaries, factual QA):
            → temperature = 0.1–0.3
        
        Variety improves quality (drafts, recommendations, brainstorming):
            → temperature = 0.5–0.8; add top_p = 0.9–0.95 if needed
                        │
                        ▼
STEP 4: Validate on p99 inputs, not average inputs.
        Run the configuration on the longest, most complex inputs you expect.
        Check: does max_tokens truncate valid outputs?
        Check: do stop sequences fire too early on edge-case inputs?
        Check: is temperature producing consistent results for deterministic tasks?
                        │
                        ▼
STEP 5: Document the parameter choice and the reasoning.
        Store the configuration in a versioned registry or config file.
        Note the task type, the parameters chosen, and why.
        When the model updates and behavior changes, the registry is the
        single place to review and adjust.
```

### Red Flags in Code Review

| What you see | Why it should raise a flag |
|---|---|
| No `max_tokens` set in the API call | Output length is unconstrained — costs and latency are unpredictable; verbose responses in production are a matter of when, not if |
| Same temperature across all endpoints | Different task types have different determinism requirements — a single temperature setting is wrong for at least some of them |
| `temperature=0.7` on a classification endpoint | Classification must be deterministic — this will produce label inconsistency on borderline inputs |
| No stop sequences on a JSON extraction endpoint | The model will append text after the JSON object — the parser will fail on those responses |
| `max_tokens` set exactly to average expected output | p99 outputs will be truncated mid-sentence — set to p99 + 20%, not to average |
| Both `temperature` and `top_p` both set very low without justification | Double-restricting sampling causes repetitive, low-quality outputs — use one primary sampling control, not both simultaneously at extreme values |

## COMMON MISTAKES
<!-- cognitive job: inoculate against predictable errors by teaching error structure, not just correct answers -->

### Mistake 1: Using Temperature to Control Accuracy

**Mistaken instinct:** "If we lower the temperature, the model will give more accurate answers."

**Why it feels right:** Lower temperature makes outputs more consistent and confident-sounding. Consistency feels like accuracy. If the model always says the same thing, it sounds sure — and sure sounds accurate.

**Why it fails:** Temperature controls the randomness of token sampling — not the factual correctness of the content. At `temperature=0`, the model consistently selects the highest-probability token at each step. If the highest-probability sequence is factually wrong — which happens when the model lacks the relevant knowledge — lowering temperature makes the model consistently wrong. A model set to `temperature=0` will produce the same incorrect answer every single time. Accuracy is determined by training data and grounding (RAG, tools) — not by sampling parameters.

**Better move:** For factual accuracy, use retrieval or tool calling to provide the correct information as context. Set temperature based on whether the task requires determinism — not as a quality dial.

---

### Mistake 2: Not Setting Max Tokens on Any Endpoint

**Mistaken instinct:** "The model knows when to stop — it will end naturally at a reasonable length."

**Why it feels right:** LLMs do produce end-of-sequence tokens when a natural stopping point is reached. Most responses do end at a sensible length. This works in testing where inputs are carefully crafted.

**Why it fails:** In production, some inputs trigger unexpectedly verbose responses. Without `max_tokens`, those responses run until they hit the context window limit — potentially producing thousands of tokens where you expected dozens. The cost of one unconstrained verbose response can exceed the cost of hundreds of normal responses. Latency spikes accordingly. Downstream systems that have timeouts will start failing.

**Better move:** Set `max_tokens` on every production call. Measure the p99 output length on representative data, add a 20% buffer, and use that value. For fixed-format outputs (classification, extraction), set it tightly — 10–50 tokens. For prose outputs, set it generously above the expected maximum — truncation mid-sentence is more harmful than a slightly long response.

---

### Mistake 3: Forgetting That Stop Sequences Are Not Included in the Output

**Mistaken instinct:** "I added `}` as a stop sequence — the JSON object will be complete in the output."

**Why it feels right:** You added `}` as the stop sequence because you wanted generation to halt after the JSON closes. "Stop after `}`" sounds like "include `}` then stop."

**Why it fails:** Stop sequences cause generation to halt immediately when they are matched — but the stop sequence itself is NOT appended to the output. If you set `stop=["}"]`, the model generates `{"key": "value"` and then stops at the `}`, which is consumed by the stop mechanism and not included. The output is `{"key": "value"` — invalid JSON. Every parser that assumes the `}` will be present will fail.

**Better move:** Either use a stop sequence that comes *after* the closing brace (e.g., `"}\n\n"`, `"}\n---"`), or add a one-line post-processing step that appends the closing delimiter if it is missing. Both are production-standard patterns. The choice depends on whether the stop sequence you want to use precedes or follows the delimiter you want to preserve.

---

### Mistake 4: Applying a Development Parameter Set to Production

**Mistaken instinct:** "The parameters we used while building the feature work fine — we'll keep them for production."

**Why it feels right:** The development parameters produced outputs that looked good during testing. Changing parameters before deployment feels like introducing unnecessary risk.

**Why it fails:** Development parameters are often set for exploration, not production. Common development-phase habits: `temperature=0.7` for variety during prompt iteration, no stop sequences because the developer reads the full output, no `max_tokens` because lengthy outputs help during debugging. Each of these creates a production failure mode: inconsistent outputs for deterministic tasks, parser failures on appended text, unbounded latency and cost. The parameter set that is useful for building is almost never the correct set for operating.

**Better move:** Define the production parameter set before deployment using the task type framework. Apply it to a representative set of inputs and validate explicitly. Treat the transition from development parameters to production parameters as a required step, not an afterthought.

## SUMMARY

**Five things to carry out of this lesson:**

1. Temperature controls sampling randomness — not accuracy; `temperature=0` means consistent, not correct, and the correct value is determined by whether the task requires determinism or allows variation.
2. Top-p restricts sampling to the set of tokens that account for the top p% of probability mass — use it as a ceiling on surprise for code generation and creative tasks; leave it at 1.0 for deterministic tasks.
3. `max_tokens` is the output-side budget enforcer — always set it; set it to p99 expected output length plus 20%; tight for fixed-format outputs, generous for prose.
4. Stop sequences halt generation immediately when matched and are NOT included in the output — use them for JSON and structured outputs to prevent appended text; account for the missing delimiter in post-processing.
5. Each task type has a correct parameter profile: classification and extraction use `temperature=0`; summarization uses 0.2–0.3; code uses 0.2 with top-p; creative generation uses 0.7–0.8.

## FAST RECALL

### What This Lesson Is Really About

Parameters are not defaults you leave in place — they are the contract between your endpoint's task requirements and the model's sampling behavior, and each task type demands a different contract.

### 30-Second Recall

- Temperature = randomness control, not accuracy; 0 = deterministic, 1 = natural variation
- Top-p = restricts sampling pool to top-p probability mass; useful for code and creative tasks
- max_tokens = hard output ceiling; always set it; base it on p99 output length + buffer
- Stop sequences = structural delimiters; NOT included in output; defensive `+= delimiter` for JSON
- Task profile: classification → 0.0; summarization → 0.3; creative → 0.7–0.8

### Core Mental Model

```text
token logits
    ↓ ÷ temperature (0 = greedy, 1 = raw, >1 = flat)
    ↓ top-p filter  (keep tokens summing to top p% of mass)
    ↓ sample next token
    ↓ append to output
    ↓ if token in stop_sequences → halt (stop string NOT included)
    ↓ if output_tokens == max_tokens → halt (may be mid-sentence)
    ↓ if EOS token → halt naturally

accuracy ≠ temperature → accuracy = grounding (RAG, tools)
```

### Must Know Cold

- What temperature does to the token probability distribution and why it does not control accuracy
- When to use temperature=0 (deterministic tasks) vs higher values (generative tasks)
- Why max_tokens must always be set in production
- That stop sequences are NOT included in the output — and the defensive post-processing fix for JSON
- The approximate temperature value for each of the five main task types

### Code Patterns to Remember

```python
# Classification — deterministic, tight output, stop on punctuation
client.chat.completions.create(
    model="gpt-4o", messages=[...],
    temperature=0.0, max_tokens=20, stop=["\n", "."],
)

# JSON extraction — deterministic, schema-sized output, stop at JSON boundary
client.chat.completions.create(
    model="gpt-4o", messages=[...],
    temperature=0.0, max_tokens=300, stop=["}\n\n"],
)

# Summarization — low-moderate temp, generous output, no stop needed
client.chat.completions.create(
    model="gpt-4o", messages=[...],
    temperature=0.3, max_tokens=300,
)

# Creative drafting — higher temp, top-p ceiling, generous output
client.chat.completions.create(
    model="gpt-4o", messages=[...],
    temperature=0.8, top_p=0.95, max_tokens=800,
)

# Defensive JSON close — when stop=["}"] consumed the closing brace
raw = response.choices[0].message.content.strip()
if not raw.endswith("}"):
    raw += "}"
data = json.loads(raw)
```

### Decision Triggers

- Task output must be the same for identical inputs → temperature=0
- Task output is a fixed-set label → temperature=0, max_tokens=10–30, stop=["\n"]
- Task output is JSON → temperature=0–0.1, stop at closing delimiter, defensive `+= "}"` post-processing
- Task output is prose and must not be truncated → max_tokens at p99 length + 20%
- Task benefits from variety (drafts, options) → temperature=0.7–0.8, top_p=0.95
- Code generation → temperature=0.2, top_p=0.95 (double constraint for correctness)

### Output Shapes to Remember

| Parameter | Effect on output |
|---|---|
| `temperature=0` | Always the same output for the same input |
| `temperature=0.3` | Mostly consistent, natural phrasing variation |
| `temperature=0.8` | Different phrasing, structure, and angle on each call |
| `max_tokens=20` | Output cut at 20 tokens — mid-word if necessary |
| `stop=["\n"]` | Output halts at the first newline; newline not in output |
| `top_p=0.95` | Sampling restricted to tokens covering 95% of probability mass |

### Confusions to Avoid

| Easy confusion | Correct distinction |
|---|---|
| Lower temperature = more accurate | Temperature controls randomness, not correctness; accuracy requires grounding |
| Stop sequence is included in output | Stop sequences halt generation and are NOT included — account for the missing delimiter |
| max_tokens limits context window | max_tokens limits output length; context window limits total input + output |
| top_p and temperature do the same thing | Temperature scales the full distribution; top_p restricts which tokens are eligible — different mechanisms |
| Default parameters are safe for production | Defaults were chosen for general usability, not task-specific correctness — always set explicitly |

### Top Traps

- Using `temperature=0` to fix wrong answers — changes consistency, not correctness
- Not setting `max_tokens` — leaves output length and cost unbounded
- Setting `max_tokens` to average output length — p99 outputs get truncated mid-sentence
- Using stop sequences that consume the closing delimiter of structured output
- Copying development parameters to production without a deliberate review
- Using the same temperature for classification as for creative generation

### One Good Example to Keep

A content moderation classifier was deployed with `temperature=0.7` and no `max_tokens`. In production, identical messages received different labels on different calls (the temperature problem) and occasionally triggered 5-second response times when the model elaborated on its classification reasoning (the max_tokens problem). The fix: `temperature=0`, `max_tokens=10`, `stop=["\n"]`. Two lines. The root cause was copying parameters from the exploratory notebook into the production service without reviewing whether they were appropriate for a deterministic classification task.

## PRACTICAL LANDSCAPE

### Technique Landscape

| Parameter | What it controls | When to tune it |
|---|---|---|
| `temperature` | Sharpness of token probability distribution before sampling | Every endpoint — choose based on determinism vs. generation requirement |
| `top_p` | Maximum probability mass from which tokens are sampled | Code generation, creative tasks where a secondary ceiling adds value |
| `max_tokens` | Hard ceiling on output token count | Every endpoint — always set explicitly |
| `stop` | Strings that immediately halt generation | Any structured output endpoint (JSON, code fences, labels) |
| `frequency_penalty` | Penalizes tokens that have appeared frequently in the output | Long-form generation where repetition is a quality issue |
| `presence_penalty` | Penalizes any token that has appeared at all in the output | When topical variety across a response matters |
| `seed` | Sets a random seed for reproducible sampling (some models) | Testing and debugging — consistent outputs for the same seed |

### When to Use What

| If your real need is... | Set... | Not... |
|---|---|---|
| Same output every time | temperature=0 | temperature=0.3 "as a compromise" |
| Prevent model elaboration after structured output | stop sequence on closing delimiter | Rely on instructions alone |
| Cap response cost and latency | max_tokens at p99 + 20% | Leave max_tokens unset |
| Reduce output repetition in long drafts | frequency_penalty=0.3–0.6 | Higher temperature (different problem) |
| Code that avoids unlikely syntax tokens | top_p=0.95 + temperature=0.2 | Temperature alone |
| Reproducible outputs for testing | seed parameter (where supported) | temperature=0 alone (some variation still possible at API level) |

### Coding Surface

| Parameter | Type | Range | Default | Notes |
|---|---|---|---|---|
| `temperature` | float | 0.0–2.0 | 1.0 | 0.0 = greedy; >1.0 = rarely useful in production |
| `top_p` | float | 0.0–1.0 | 1.0 | 1.0 = no restriction; use 0.9–0.95 when needed |
| `max_tokens` | int | 1–model limit | None (unlimited) | Always set; base on p99 output length |
| `stop` | list[str] or str | up to 4 strings | None | Stop string excluded from output |
| `frequency_penalty` | float | -2.0–2.0 | 0.0 | Positive values reduce repetition |
| `presence_penalty` | float | -2.0–2.0 | 0.0 | Positive values encourage topical variety |
| `seed` | int | any int | None | Reproducibility hint (not guaranteed by all providers) |

| Parameter combination | Typical effect on output |
|---|---|
| `temperature=0` | Deterministic; same input → same output |

### Interview Q&A

**Q: What does temperature do, and how do you choose the right value for a production endpoint?**

> "Temperature scales the token probability distribution before sampling. At temperature=0 — greedy decoding — the model always picks the single highest-probability next token, producing fully deterministic output. At temperature=1, the model samples from the raw learned distribution. Above 1, the distribution flattens, making unlikely tokens more viable. I choose the value based on the task type. For deterministic tasks — classification, extraction, JSON output, routing — I use temperature=0. Same input must always produce same output for auditing, reproducibility, and downstream parsing. For tasks where some natural variation is acceptable — summarization, Q&A — I use 0.1 to 0.3: consistent but not robotic. For tasks where variety is a feature — creative drafting, generating multiple options — I use 0.7 to 0.8. The key thing I always clarify: temperature does not control accuracy. At temperature=0, the model consistently produces the highest-probability sequence — if that sequence is factually wrong, the model is consistently wrong. Factual accuracy requires grounding through retrieval or tools, not a lower temperature."

---

**Q: What are stop sequences and when do you use them?**

> "Stop sequences are strings you provide to the model that immediately halt generation when they appear in the output. The stop string itself is NOT appended to the output — generation stops before it is included. I use them whenever an endpoint has a known structural boundary. For JSON extraction, a stop sequence like `'}\n\n'` causes the model to halt as soon as the JSON object closes, preventing it from appending explanatory text that would break the parser. For classification endpoints, `'\n'` or `'.'` prevents elaboration after the label. Because the stop sequence is excluded from output, I always add a defensive post-processing step for JSON — check if the response ends with `}` and append it if not. This is the standard production pattern. I do not use stop sequences for prose outputs like summarization, where natural sentence endings are appropriate termination points and adding stop sequences would introduce edge-case truncation on specific phrases."

---

**Q: What happens if you do not set max_tokens on a production LLM call?**

> "Output length is unconstrained — the model generates tokens until it reaches a natural stopping point or the context window limit. In the average case this is fine. In the p99 case — complex inputs, ambiguous prompts, verbose model updates — the model may produce thousands of tokens where you expected dozens. This has two production consequences: cost spikes, because output tokens are billed at roughly 2x the input rate and unbounded output is unbounded cost; and latency spikes, because output generation is sequential — more tokens means more time, which means downstream timeouts on services expecting fast responses. I always set max_tokens explicitly, sized to p99 expected output length plus a 20% buffer. For classification and extraction endpoints, I set it very tightly — 10 to 50 tokens. For prose outputs, I set it generously above the expected maximum because truncating a response mid-sentence is worse than allowing a slightly long response."
