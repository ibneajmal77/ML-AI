## CONCEPT BLOCK 1 — What Streaming Is and How the Chunk Protocol Works

### A — The core idea in one sentence

Streaming mode sends each token to your code the moment it is generated instead of waiting for the full response — the API returns an iterator of small chunk objects rather than one large response object.

### B — Explain it plainly

LLMs generate text one token at a time, left to right. In non-streaming mode, the API waits until all tokens have been generated, assembles the full response, and sends it in one HTTP response body. Your code receives nothing until generation is complete.

In streaming mode, the API sends each token (or a small group of tokens) to your code immediately after it is generated, as a Server-Sent Event. Generation time is identical — the model is doing the same work. What changes is *when your code receives the output*.

The Python SDK returns a `Stream` (synchronous) or `AsyncStream` (asynchronous) iterator. Each iteration produces a `ChatCompletionChunk` object. A chunk has a different shape from the non-streaming `ChatCompletion` response:

- `chunk.choices[0].delta.content` — the text fragment generated in this step (a string, or `None` if this chunk carries no text)
- `chunk.choices[0].delta.role` — appears only in the first chunk (value `"assistant"`)
- `chunk.choices[0].finish_reason` — `None` for every chunk except the last, where it becomes `"stop"`, `"tool_calls"`, `"length"`, or `"content_filter"`
- `chunk.usage` — present only in the final chunk when `stream_options={"include_usage": True}` is set; contains token counts

The chunk delta pattern mirrors the full response structure but accumulates incrementally. The full content is the concatenation of all `delta.content` values across every chunk.

**First-token latency (TTFT — Time To First Token)** is the elapsed time from sending the API request to receiving the first non-null `delta.content`. TTFT is the metric that determines perceived responsiveness — users judge speed by when text starts appearing, not when it finishes. Total generation time matters for throughput budgets. TTFT matters for UX.

Rule of thumb: streaming is worth implementing for any output longer than 50–80 tokens. For short outputs (classification labels, single-sentence extractions), streaming adds complexity with no perceptible UX benefit — the entire response arrives in 1–2 chunks anyway.

### C — Concrete example with full working code

```python
import os
import time
import json
import logging
from openai import AzureOpenAI

client = AzureOpenAI(
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    api_key=os.environ["AZURE_OPENAI_API_KEY"],
    api_version="2024-02-01",
)
DEPLOYMENT = "gpt-4o"
logger = logging.getLogger(__name__)


def stream_summary(document_text: str) -> None:
    """
    Streams a summary to stdout, measuring and logging TTFT.
    Demonstrates the raw streaming loop before wrapping it in FastAPI.
    """
    messages = [
        {
            "role": "system",
            "content": (
                "You are a document summarizer. "
                "Produce a concise summary in 3–5 sentences. "
                "Plain prose, no bullet points."
            ),
        },
        {
            "role": "user",
            "content": f"<document>\n{document_text}\n</document>",
        },
    ]

    request_start = time.monotonic()
    first_token_time: float | None = None
    total_tokens_generated = 0
    full_content: list[str] = []

    # stream=True returns a Stream[ChatCompletionChunk] iterator
    stream = client.chat.completions.create(
        model=DEPLOYMENT,
        messages=messages,
        max_tokens=300,
        temperature=0.3,
        stream=True,
        stream_options={"include_usage": True},  # token counts in the final chunk
    )

    for chunk in stream:
        delta_content = chunk.choices[0].delta.content if chunk.choices else None
        finish_reason = chunk.choices[0].finish_reason if chunk.choices else None

        # Measure TTFT on the first non-null content chunk
        if delta_content is not None and first_token_time is None:
            first_token_time = time.monotonic()
            ttft_ms = (first_token_time - request_start) * 1000
            logger.info("TTFT: %.1f ms", ttft_ms)

        # Accumulate content
        if delta_content:
            full_content.append(delta_content)
            print(delta_content, end="", flush=True)   # immediate output

        # Usage is in the final chunk when stream_options include_usage is True
        if chunk.usage is not None:
            total_tokens_generated = chunk.usage.completion_tokens
            logger.info(
                "Stream complete — finish_reason=%s completion_tokens=%d total_ms=%.1f",
                finish_reason,
                total_tokens_generated,
                (time.monotonic() - request_start) * 1000,
            )

    print()  # newline after streamed output
    return "".join(full_content)


# Example output for a 200-word document:
# TTFT: 382.4 ms
# [text streams in progressively over ~3s]
# Stream complete — finish_reason=stop completion_tokens=87 total_ms=3241.7
```

### D — Streaming chunk anatomy

| Chunk field | First chunk | Middle chunks | Last chunk |
|---|---|---|---|
| `delta.role` | `"assistant"` | `None` | `None` |
| `delta.content` | First token(s) or `None` | Text fragment or `None` | `None` (content is in the second-to-last) |
| `finish_reason` | `None` | `None` | `"stop"` / `"tool_calls"` / `"length"` / `"content_filter"` |
| `usage` | `None` | `None` | Present only if `stream_options={"include_usage": True}` |

### E — Where this fits in the broader architecture

Streaming changes the delivery model, not the generation model. All of Lessons 2.4–2.8 still apply: the system prompt, tool definitions, and Pydantic validation are identical. What changes is how the response travels from the API to your client. The streaming loop is the point where you insert latency measurement, token counting, and — as Concept Block 3 covers — error detection.

### F — Why engineers get this wrong

Two mistakes. First: checking `chunk.choices[0].delta.content` without guarding for `None`. Every chunk arrives in the iterator, including intermediate chunks where `delta.content` is `None` (particularly the first chunk, which carries only `delta.role`). Concatenating `None` raises a `TypeError`. Always guard: `if delta_content is not None`. Second: treating the final `finish_reason` as always appearing on the same chunk as the last content. It does not — the last content chunk has `finish_reason=None`; the next chunk has `finish_reason="stop"` and `delta.content=None`. Treat them as separate events.

### G — Failure modes and their causes

| Failure | Root cause | Fix |
|---|---|---|
| `TypeError: can only concatenate str (not "NoneType")` | `delta.content` is `None` on non-text chunks | Guard: `if delta_content is not None` before using it |
| `finish_reason` never seen | Only checking `delta.content` chunks; skipping the final metadata chunk | Iterate until the stream is exhausted — the last chunk has content `None` and `finish_reason` set |
| Token count always zero | `stream_options={"include_usage": True}` not set | Add it to the API call; usage is only in the final chunk |
| TTFT measurement always zero | `first_token_time` reset on every chunk | Set it only once: `if first_token_time is None and delta_content is not None` |
| Streamed text is buffered by the HTTP layer | Response arrives all at once despite `stream=True` | Streaming is happening at the API layer; buffering may be at the network layer (Concept Block 3) |

### H — Reading the model's response as a signal

A `finish_reason="length"` in the final chunk means the model was cut off by `max_tokens` before completing the response. The streamed output is truncated mid-sentence. If this appears in production, either increase `max_tokens` or add a post-stream check: if `finish_reason == "length"`, append a truncation warning to the response rather than silently delivering incomplete output. A `finish_reason="content_filter"` means the Azure content policy blocked the output mid-stream — the streamed content up to that point may be partial or empty.

### I — Senior signal

> "TTFT is the metric I track for any streaming endpoint, not total generation time. A model with 600ms TTFT that takes 8 seconds to complete feels faster than a model with 1200ms TTFT that takes 5 seconds to complete. Users judge speed by when they see the first character. We alert if P95 TTFT exceeds 800ms — that's our UX threshold. Total generation time is a cost metric, not a UX metric."

## CONCEPT BLOCK 2 — SSE Endpoint Design in FastAPI

### A — The core idea in one sentence

A FastAPI streaming endpoint is an async generator that yields SSE-formatted strings — each yielded line is one `data:` event that the client's `EventSource` reads as it arrives.

### B — Explain it plainly

Server-Sent Events (SSE) is an HTTP protocol where the server keeps a connection open and pushes lines of text to the client as events occur. The client reads each event as it arrives without polling. It is simpler than WebSockets — unidirectional (server to client), built on plain HTTP, and natively supported by browser `EventSource` and by SSE client libraries in every major language.

The SSE wire format is simple:
- Each event is one line starting with `data:` followed by the payload
- Each event ends with a double newline (`\n\n`) — this is what signals the event boundary to the client
- The connection stays open until the server closes it or sends a terminal event

In FastAPI, a streaming endpoint returns a `StreamingResponse` with `media_type="text/event-stream"`. The body is an async generator function that yields one string per event. FastAPI sends each yielded string to the client immediately without buffering.

The OpenAI SDK has a synchronous streaming interface (`Stream`) and an asynchronous one (`AsyncStream`). In a FastAPI async context, you should use the async client (`AsyncAzureOpenAI`) with `async for chunk in stream` — this avoids blocking the event loop while waiting for the next token.

Three things every production SSE endpoint must do:
1. **Yield each content chunk as a `data:` event** — one event per token or token group
2. **Log first-token latency** — TTFT is a production metric; it must be measured at the endpoint level
3. **Send a terminal event** — a final `data: [DONE]\n\n` or similar sentinel so the client knows the stream is complete and can close the connection

### C — Concrete example with full working code

```python
import os
import time
import json
import logging
from openai import AsyncAzureOpenAI
from fastapi import FastAPI, Request
from fastapi.responses import StreamingResponse
from pydantic import BaseModel

app = FastAPI()
logger = logging.getLogger(__name__)

# Use the async client for FastAPI async endpoints
async_client = AsyncAzureOpenAI(
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    api_key=os.environ["AZURE_OPENAI_API_KEY"],
    api_version="2024-02-01",
)
DEPLOYMENT = "gpt-4o"

SUMMARIZE_SYSTEM_PROMPT = """\
## Role
You are a document summarizer.

## Task
Produce a concise summary of the document provided by the user.

## Output format
3–5 sentences of plain prose. No bullet points. No headers.

## Constraints
- Do not add information not present in the document.
- Do not begin with "This document..." or "The document..."."""


class SummarizeRequest(BaseModel):
    document_text: str
    max_tokens: int = 300


async def stream_summary_chunks(
    document_text: str,
    max_tokens: int,
    request_id: str,
) -> None:
    """
    Async generator that yields SSE-formatted events.

    Yields:
      - One "data: {...}\n\n" event per content chunk
      - One "data: {"event": "first_token", "ttft_ms": N}\n\n" at TTFT measurement
      - One "data: {"event": "done", "finish_reason": "stop"}\n\n" at stream end
      - One "data: {"event": "error", "detail": "..."}\n\n" on failure

    The client reads each event by splitting on "\n\n" and parsing the "data:" line.
    """
    messages = [
        {"role": "system", "content": SUMMARIZE_SYSTEM_PROMPT},
        {"role": "user", "content": f"<document>\n{document_text}\n</document>"},
    ]

    request_start = time.monotonic()
    first_token_seen = False
    finish_reason: str | None = None

    try:
        # AsyncAzureOpenAI.chat.completions.create with stream=True
        # returns AsyncStream[ChatCompletionChunk] — iterable with async for
        stream = await async_client.chat.completions.create(
            model=DEPLOYMENT,
            messages=messages,
            max_tokens=max_tokens,
            temperature=0.3,
            stream=True,
            stream_options={"include_usage": True},
        )

        async for chunk in stream:
            # Guard: choices may be empty on usage-only final chunk
            if not chunk.choices:
                continue

            delta_content = chunk.choices[0].delta.content
            chunk_finish = chunk.choices[0].finish_reason

            # Measure and emit TTFT on first content token
            if delta_content and not first_token_seen:
                first_token_seen = True
                ttft_ms = (time.monotonic() - request_start) * 1000
                logger.info(
                    "request_id=%s TTFT=%.1f ms", request_id, ttft_ms
                )
                # Emit TTFT as a metadata event — client can log it
                yield f"data: {json.dumps({'event': 'first_token', 'ttft_ms': round(ttft_ms, 1)})}\n\n"

            # Emit content chunk
            if delta_content:
                yield f"data: {json.dumps({'event': 'content', 'content': delta_content})}\n\n"

            # Capture finish_reason from the final chunk
            if chunk_finish is not None:
                finish_reason = chunk_finish

        # Stream exhausted — emit terminal event
        total_ms = (time.monotonic() - request_start) * 1000
        logger.info(
            "request_id=%s stream complete finish_reason=%s total_ms=%.1f",
            request_id, finish_reason, total_ms,
        )
        yield f"data: {json.dumps({'event': 'done', 'finish_reason': finish_reason})}\n\n"

    except Exception as e:
        # Error occurred during streaming — emit error event in-band
        # HTTP status code is already committed (200), so we cannot send a 500.
        logger.error(
            "request_id=%s stream error: %s: %s",
            request_id, type(e).__name__, e, exc_info=True,
        )
        yield f"data: {json.dumps({'event': 'error', 'detail': str(e)})}\n\n"


@app.post("/summarize")
async def summarize(body: SummarizeRequest, request: Request):
    """
    SSE streaming endpoint for document summarization.

    Response format: text/event-stream
    Each SSE event is a JSON object with an "event" field:
      - {"event": "first_token", "ttft_ms": 382.4}
      - {"event": "content", "content": "The contract..."}
      - {"event": "done", "finish_reason": "stop"}
      - {"event": "error", "detail": "..."} — only on failure

    Headers set:
      - Cache-Control: no-cache       — prevents proxy caching of the stream
      - X-Accel-Buffering: no         — disables nginx buffering
      - Connection: keep-alive        — keeps TCP connection open
    """
    import uuid
    request_id = str(uuid.uuid4())[:8]

    return StreamingResponse(
        stream_summary_chunks(body.document_text, body.max_tokens, request_id),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "X-Accel-Buffering": "no",    # critical: disables nginx buffering
            "Connection": "keep-alive",
        },
    )
```

### D — SSE wire format reference

```
# Each event on the wire looks like this:
data: {"event": "first_token", "ttft_ms": 382.4}\n\n
data: {"event": "content", "content": "The contract "}\n\n
data: {"event": "content", "content": "specifies a "}\n\n
data: {"event": "content", "content": "90-day notice period"}\n\n
data: {"event": "done", "finish_reason": "stop"}\n\n

# Double newline (\n\n) is the event boundary — clients split on this.
# Single newline (\n) continues the same event.
# "data:" prefix is required by the SSE specification.
# The payload after "data: " can be any string — JSON is the practical choice.
```

```python
# Client-side consumption (Python, httpx streaming):
import httpx
import json

with httpx.Client() as client:
    with client.stream("POST", "http://localhost:8000/summarize",
                        json={"document_text": doc}) as response:
        for line in response.iter_lines():
            if line.startswith("data: "):
                event = json.loads(line[len("data: "):])
                if event["event"] == "content":
                    print(event["content"], end="", flush=True)
                elif event["event"] == "done":
                    print()  # newline after completion
                    break
                elif event["event"] == "error":
                    print(f"\n[Error: {event['detail']}]")
                    break
```

### E — Where this fits in the broader architecture

The `stream_summary_chunks` generator is the LLM streaming layer. Everything downstream — the client receiving the events, the browser rendering the text progressively — is insulated from the raw OpenAI API by this generator. When you add cost logging (Lesson 2.15) or token counting, those additions go inside the generator after the stream is exhausted, before the terminal event is emitted. The endpoint function stays clean.

### F — Why engineers get this wrong

Three mistakes. First: using the synchronous `AzureOpenAI` client instead of `AsyncAzureOpenAI` inside an `async def` FastAPI endpoint. The synchronous stream iterator blocks the event loop while waiting for each token — other requests cannot be served during the wait. Use `AsyncAzureOpenAI` and `async for chunk in stream`. Second: missing the `X-Accel-Buffering: no` header. Without it, nginx (the most common reverse proxy in front of FastAPI) buffers the response until it exceeds the buffer size, then releases all buffered chunks at once. The client sees the same all-at-once delivery as non-streaming. This header disables that behavior. Third: not sending a terminal event. Without a `[DONE]` or similar sentinel, the client does not know when the stream is finished. It must wait for the connection to close — which may not happen immediately due to keep-alive settings.

### G — Failure modes and their causes

| Failure | Root cause | Fix |
|---|---|---|
| Event loop blocked, requests queue | Sync `AzureOpenAI` client used in async endpoint | Replace with `AsyncAzureOpenAI`; use `async for` |
| Streaming arrives all at once (proxy buffering) | `X-Accel-Buffering` header not set | Add `"X-Accel-Buffering": "no"` to response headers |
| Client hangs after last token | No terminal event sent | Yield `data: {"event": "done"}\n\n` after stream exhaustion |
| `finish_reason` never captured | Only content chunks checked; usage chunk skipped | Check `chunk_finish is not None` separately from `delta_content` check |
| TTFT never logged | Guard condition inverted or `first_token_seen` reset incorrectly | Set `first_token_seen = True` exactly once, inside the `not first_token_seen` branch |

### H — Reading the model's response as a signal

If your TTFT logs show consistently high values (800ms+) despite the model performing normally in non-streaming tests, the bottleneck is likely the time from your FastAPI server to the Azure OpenAI endpoint — network latency and regional routing. Switch to the Azure OpenAI endpoint in the region closest to your deployment. If TTFT is normal but the client receives the first byte much later, proxy buffering is the cause — verify `X-Accel-Buffering: no` is reaching the reverse proxy.

### I — Senior signal

> "The three headers on a streaming endpoint — `Cache-Control: no-cache`, `X-Accel-Buffering: no`, and `Connection: keep-alive` — are not optional polish. Miss `X-Accel-Buffering: no` behind nginx and your streaming endpoint silently becomes a non-streaming endpoint. We caught this in staging because our TTFT metric suddenly jumped to match total generation time. The proxy was buffering the entire response. One header fixed it, but the test that caught it — comparing TTFT to total generation time as a ratio — is now part of our deployment checklist."

## CONCEPT BLOCK 3 — Errors Mid-Stream and Production Patterns

### A — The core idea in one sentence

Once the first byte of a streaming response is sent, the HTTP status code is committed and cannot be changed — errors after that point must be signaled in-band as SSE error events, and production endpoints need a heartbeat, a hard timeout, and client-disconnect detection to remain reliable.

### B — Explain it plainly

In a normal HTTP request, if something goes wrong you return a 500 status code. In streaming, by the time an error occurs — mid-generation, after 40 tokens — the server has already sent HTTP 200 and the first bytes of the response. The HTTP protocol does not allow you to change the status code retroactively. The client is streaming; it already considers the response successful.

This means error handling in streaming is fundamentally different. Every error that can occur after the first byte must be signaled in-band: as a special event in the stream that the client is already reading. The error event must carry enough information for the client to decide what to do — show a partial result, retry, or show an error message.

Four production concerns beyond the happy path:

**In-band error events.** Any exception that occurs inside the streaming generator is caught, converted to a `{"event": "error", "detail": "..."}` SSE event, and yielded before the generator exits. The client reads this event and handles it gracefully. The alternative — letting the exception propagate — closes the connection abruptly. The client receives an incomplete stream with no indication of what happened.

**Heartbeat / keep-alive events.** Load balancers and reverse proxies have idle connection timeouts — typically 30–60 seconds. A streaming response that takes 90 seconds to generate will be killed by the load balancer at the 60-second mark if no bytes are sent during the generation. For long-running generations, you need to send periodic heartbeat events (SSE comment lines: `: keep-alive\n\n`) to keep the connection alive through the load balancer.

**Hard timeout on the stream.** If the model stalls — stops generating tokens mid-response — your generator will block indefinitely. An `asyncio.wait_for()` wrapper around the generation loop enforces a maximum generation time. If the timeout fires, yield an error event and close the generator cleanly.

**Client disconnect detection.** When a client navigates away or closes the browser tab, the TCP connection drops. In an async FastAPI generator, this surfaces as `asyncio.CancelledError` being raised inside the generator. Catching `CancelledError` lets you log the disconnect, cancel the upstream API call (to stop paying for tokens the client will never receive), and clean up resources.

### C — Concrete example with full working code

```python
import asyncio
import time
import json
import logging
from openai import AsyncAzureOpenAI

logger = logging.getLogger(__name__)

GENERATION_TIMEOUT_SECONDS = 60.0   # max time for the full generation
HEARTBEAT_INTERVAL_SECONDS = 15.0   # keep-alive ping interval


async def stream_summary_production(
    document_text: str,
    max_tokens: int,
    request_id: str,
) -> None:
    """
    Production-hardened streaming generator.

    Adds:
      - Hard generation timeout via asyncio.wait_for
      - Periodic heartbeat events (SSE comment lines) to prevent proxy timeouts
      - Client-disconnect detection via asyncio.CancelledError
      - In-band error events for all failure modes
    """
    messages = [
        {"role": "system", "content": SUMMARIZE_SYSTEM_PROMPT},
        {"role": "user", "content": f"<document>\n{document_text}\n</document>"},
    ]

    request_start = time.monotonic()
    first_token_seen = False
    last_heartbeat = time.monotonic()

    async def _generate():
        """Inner generator — wrapped with timeout in the outer function."""
        nonlocal first_token_seen, last_heartbeat

        stream = await async_client.chat.completions.create(
            model=DEPLOYMENT,
            messages=messages,
            max_tokens=max_tokens,
            temperature=0.3,
            stream=True,
            stream_options={"include_usage": True},
        )

        async for chunk in stream:
            if not chunk.choices:
                continue

            delta_content = chunk.choices[0].delta.content
            chunk_finish = chunk.choices[0].finish_reason

            # TTFT measurement
            if delta_content and not first_token_seen:
                first_token_seen = True
                ttft_ms = (time.monotonic() - request_start) * 1000
                logger.info("request_id=%s TTFT=%.1f ms", request_id, ttft_ms)
                yield f"data: {json.dumps({'event': 'first_token', 'ttft_ms': round(ttft_ms, 1)})}\n\n"

            # Content chunk
            if delta_content:
                yield f"data: {json.dumps({'event': 'content', 'content': delta_content})}\n\n"
                last_heartbeat = time.monotonic()   # reset heartbeat timer on content

            # Heartbeat: send a keep-alive comment if no content has been sent recently
            # SSE comment lines (": ...") are not dispatched as events — they only keep the connection alive
            now = time.monotonic()
            if now - last_heartbeat > HEARTBEAT_INTERVAL_SECONDS:
                yield ": keep-alive\n\n"            # SSE comment — invisible to event listeners
                last_heartbeat = now
                logger.debug("request_id=%s sent heartbeat", request_id)

            # Terminal event
            if chunk_finish is not None:
                total_ms = (time.monotonic() - request_start) * 1000
                logger.info(
                    "request_id=%s done finish_reason=%s total_ms=%.1f",
                    request_id, chunk_finish, total_ms,
                )
                yield f"data: {json.dumps({'event': 'done', 'finish_reason': chunk_finish})}\n\n"

    try:
        # Hard timeout: if generation exceeds GENERATION_TIMEOUT_SECONDS, raise TimeoutError
        async for event in asyncio.timeout(GENERATION_TIMEOUT_SECONDS, _generate()):
            yield event

    except asyncio.TimeoutError:
        elapsed = (time.monotonic() - request_start) * 1000
        logger.error(
            "request_id=%s stream timeout after %.1f ms (limit=%.0fs)",
            request_id, elapsed, GENERATION_TIMEOUT_SECONDS,
        )
        yield f"data: {json.dumps({'event': 'error', 'detail': 'generation_timeout', 'partial': True})}\n\n"

    except asyncio.CancelledError:
        # Client disconnected — log and exit cleanly without yielding (no client to receive it)
        elapsed = (time.monotonic() - request_start) * 1000
        logger.info(
            "request_id=%s client disconnected after %.1f ms first_token_seen=%s",
            request_id, elapsed, first_token_seen,
        )
        # Re-raise so FastAPI knows the generator was cancelled (it will close the stream)
        raise

    except Exception as e:
        logger.error(
            "request_id=%s unexpected stream error: %s: %s",
            request_id, type(e).__name__, e, exc_info=True,
        )
        yield f"data: {json.dumps({'event': 'error', 'detail': str(e)})}\n\n"
```

### D — Production streaming concerns and their solutions

| Concern | What happens without it | Solution |
|---|---|---|
| **In-band error events** | Connection closes abruptly mid-stream; client receives truncated content with no explanation | Catch all exceptions; yield `{"event": "error", "detail": "..."}` |
| **Heartbeat / keep-alive** | Load balancer kills the connection after idle timeout (typically 30–60s) | Yield SSE comment `: keep-alive\n\n` every 10–15s when no content is generated |
| **Hard timeout** | Model stall blocks the generator indefinitely; request never completes | `asyncio.timeout(N, generator())` with an in-band error event on `TimeoutError` |
| **Client disconnect handling** | Generator continues generating tokens and paying token cost after client left | Catch `asyncio.CancelledError`, re-raise, cancel the upstream API call |
| **`X-Accel-Buffering: no` header** | Nginx buffers the stream; client receives all-at-once delivery | Add header to `StreamingResponse` |
| **Terminal event** | Client does not know when the stream is finished | Yield `{"event": "done"}` after `finish_reason` is captured |

### E — Where this fits in the broader architecture

Error handling in streaming endpoints is architecturally different from error handling in standard REST endpoints because the HTTP response has already started. This asymmetry affects monitoring too: a streaming request that started successfully (200) but ended with an in-band error event will look like a success in your HTTP status-code metrics. To correctly track streaming failures, monitor the rate of `{"event": "error"}` events in your SSE stream, not the HTTP status code of the endpoint. This metric must be added explicitly to your observability layer.

### F — Why engineers get this wrong

Two mistakes. First: not sending heartbeat events and discovering this only in production when load balancer timeouts start appearing. In local development, there is no load balancer — the connection stays open indefinitely. In staging and production behind an ALB or nginx, long-running generations are killed silently. The failure looks like a client-side timeout, not a server-side problem. Add the heartbeat in development, before it bites in production. Second: not re-raising `CancelledError`. A `CancelledError` means the framework told the generator to stop. If you catch and swallow it, the generator continues running — continuing to send tokens to a closed connection and paying token cost for a client that is already gone. Always re-raise `CancelledError` after cleanup.

### G — Failure modes and their causes

| Failure | Root cause | Fix |
|---|---|---|
| Stream killed silently after 30–60s | Load balancer idle timeout; no heartbeat sent | Add `: keep-alive\n\n` SSE comment every 10–15s |
| Generator runs forever after client leaves | `CancelledError` caught and swallowed | Catch `CancelledError`, log the disconnect, re-raise |
| Hard timeout never fires | `asyncio.timeout` not applied | Wrap generator with `asyncio.timeout(GENERATION_TIMEOUT_SECONDS, ...)` |
| Error event not received by client | Exception raised after client disconnected | Treat `CancelledError` as disconnect — stop yielding, re-raise |
| Stream failure looks like success in metrics | In-band errors not tracked; only HTTP status monitored | Add explicit metric/log for `{"event": "error"}` in SSE stream |
| Partial content on truncated finish | `finish_reason="length"` not checked | After stream ends, check `finish_reason` — yield truncation warning if `"length"` |

### H — Reading the model's response as a signal

A rising rate of `finish_reason="length"` in your production logs means your `max_tokens` budget is regularly insufficient for the inputs you're receiving. Increase `max_tokens` or add dynamic token budgeting: estimate the required output length based on input length before calling the API, and set `max_tokens` per-request. A `finish_reason="content_filter"` mid-stream means the content policy triggered on generated output — the partial response up to that point may be fine, but the full generation was blocked. Log the input document and the truncation point for review.

### I — Senior signal

> "The HTTP status code for a streaming endpoint is committed on the first byte. That changes the entire error handling model. We do not monitor HTTP 5xx rates for our streaming endpoints — those only catch errors that occur before any content is sent. We monitor the rate of error events in the stream itself, the rate of client disconnects (CancelledError), and the rate of timeout events. Those three metrics together give the real picture of streaming endpoint health."

## DECISION RULES

```
IF output is longer than ~80 tokens AND user is waiting for it
→ use streaming — TTFT reduction is worth the implementation cost
→ set stream=True, use AsyncAzureOpenAI, yield via StreamingResponse

IF output is short (classification label, single-sentence extraction)
→ do NOT stream — response arrives in 1–2 chunks, no perceived benefit
→ synchronous response is simpler and avoids SSE client requirement

IF you are behind a reverse proxy (nginx, ALB, Caddy)
→ always set "X-Accel-Buffering": "no" on the streaming response
→ without this, the proxy may buffer the stream and deliver it all at once
→ test explicitly: if TTFT ≈ total generation time, proxy buffering is the cause

IF the generation may take longer than 30 seconds
→ add heartbeat events: yield ": keep-alive\n\n" every 10–15s when no content is sent
→ reset the heartbeat timer on every content chunk
→ load balancers kill idle connections — heartbeat prevents this

IF you need a hard bound on generation time
→ wrap the generator with asyncio.timeout(N, generator())
→ catch asyncio.TimeoutError inside the generator and yield an error event
→ set N conservatively: P99 generation time + 20% margin

IF the client disconnects mid-stream
→ asyncio.CancelledError is raised inside the async generator
→ catch it, log the disconnect, re-raise — do not swallow it
→ swallowing CancelledError lets the generator keep running and billing tokens

IF an exception occurs after streaming has started
→ you cannot change the HTTP status code — it is already 200
→ yield a {"event": "error", "detail": "..."} SSE event
→ monitor error event rate in the stream, not HTTP 5xx rates

IF finish_reason is "length" on a streaming response
→ the model was cut off by max_tokens — output is truncated mid-sentence
→ either increase max_tokens or emit a truncation warning to the client
→ do not silently deliver a truncated response as if it were complete

IF you need token counts from a streaming call
→ add stream_options={"include_usage": True} to the API call
→ usage appears only in the final chunk — process it after the loop
→ the final chunk has empty choices[] — guard with: if not chunk.choices: continue

IF tool calls are returned in a streaming response (finish_reason="tool_calls")
→ delta.tool_calls contains partial argument fragments — accumulate across chunks
→ arguments are only complete when finish_reason="tool_calls" is seen
→ parse json.loads() only after accumulation is complete
```

## COMMON MISTAKES

### Mistake 1: Using the synchronous client in an async FastAPI endpoint

**What it looks like:**
```python
# Wrong: synchronous client in an async endpoint
client = AzureOpenAI(...)   # sync client

@app.post("/summarize")
async def summarize(body: SummarizeRequest):
    stream = client.chat.completions.create(..., stream=True)  # sync iterator
    async def gen():
        for chunk in stream:   # blocking iteration in async context
            yield f"data: {chunk.choices[0].delta.content}\n\n"
    return StreamingResponse(gen(), media_type="text/event-stream")
```

**Why it happens:** Engineers use the sync client in all other endpoints, and streaming feels like a small addition.

**What goes wrong:** `for chunk in stream` blocks the asyncio event loop while waiting for each token. No other requests can be processed until the current token arrives. Under load, all concurrent requests queue behind each other. A service that handles 10 concurrent streaming requests becomes effectively single-threaded.

**The fix:** Use `AsyncAzureOpenAI` and `async for`:
```python
async_client = AsyncAzureOpenAI(...)
stream = await async_client.chat.completions.create(..., stream=True)
async for chunk in stream:
    ...
```

---

### Mistake 2: Missing the `X-Accel-Buffering: no` header behind nginx

**What it looks like:**
```python
return StreamingResponse(
    generator(),
    media_type="text/event-stream",
    # no X-Accel-Buffering header
)
```

**Why it happens:** In local development there is no nginx — streaming works perfectly. Engineers deploy to staging, test the endpoint, and it appears to work. The subtle sign is missed: TTFT and total generation time are identical.

**What goes wrong:** nginx buffers the proxy response until its `proxy_buffer_size` is full (default 8KB–16KB) or the upstream closes the connection, then releases the buffer at once. The client receives all chunks simultaneously at the end of generation. Streaming silently becomes non-streaming in production without any error.

**The fix:** Add the header to every streaming response:
```python
return StreamingResponse(
    generator(),
    media_type="text/event-stream",
    headers={"X-Accel-Buffering": "no", "Cache-Control": "no-cache"},
)
```
And add to the deployment checklist: "verify TTFT < 500ms in staging behind reverse proxy."

---

### Mistake 3: Swallowing `asyncio.CancelledError`

**What it looks like:**
```python
async def gen():
    try:
        async for chunk in stream:
            yield f"data: {chunk.choices[0].delta.content}\n\n"
    except Exception:   # catches CancelledError too
        pass            # silently swallow
```

**Why it happens:** The broad `except Exception` pattern is used to "make the generator resilient."

**What goes wrong:** `asyncio.CancelledError` is the signal that FastAPI sends when the client disconnects. Swallowing it means the generator continues iterating, receiving tokens, and yielding events to a connection that no longer exists. The upstream API call continues running. If you are paying per token, you pay for every token generated after the client left. On a high-traffic service, this can represent significant wasted cost.

**The fix:** Catch `Exception` separately from `CancelledError`. Log the disconnect and re-raise `CancelledError`:
```python
async def gen():
    try:
        async for chunk in stream:
            yield ...
    except asyncio.CancelledError:
        logger.info("Client disconnected mid-stream")
        raise            # re-raise — do not swallow
    except Exception as e:
        yield f"data: {json.dumps({'event': 'error', 'detail': str(e)})}\n\n"
```

---

### Mistake 4: No terminal event — client never knows the stream is done

**What it looks like:**
```python
async def gen():
    stream = await async_client.chat.completions.create(..., stream=True)
    async for chunk in stream:
        if chunk.choices[0].delta.content:
            yield f"data: {chunk.choices[0].delta.content}\n\n"
    # generator exits — no terminal event
```

**Why it happens:** The generator exits naturally when the stream is exhausted, which closes the HTTP connection. This seems sufficient.

**What goes wrong:** The client is reading SSE events. When the server closes the connection, the `EventSource` API in browsers automatically attempts to reconnect. The client interprets a clean connection close as a network interruption and starts making new requests. This creates a reconnect loop — the client keeps requesting new summaries, and each reconnect generates a new LLM call and incurs cost.

**The fix:** Always yield a terminal event before the generator exits. The client reads this event and explicitly closes the `EventSource` without reconnecting:
```python
yield f"data: {json.dumps({'event': 'done', 'finish_reason': finish_reason})}\n\n"
# generator exits cleanly after this — client receives the done event and closes
```

## RETRIEVAL PRACTICE

Answer from memory. Check after.

**Q1:** A streaming endpoint is deployed behind nginx. In local development, TTFT is 380ms. In staging behind nginx, TTFT is 6,200ms and matches total generation time exactly. What is the cause and what is the fix?

*Check:* Nginx is buffering the streaming response — it accumulates the full response before forwarding it to the client, making streaming equivalent to non-streaming. The fix is adding `"X-Accel-Buffering": "no"` to the `StreamingResponse` headers. This instructs nginx to disable proxy buffering for this response and forward each chunk immediately.

---

**Q2:** An exception is raised inside a streaming generator after 30 tokens have already been sent to the client. You want to inform the client something went wrong. Why can't you return a 500 HTTP status code, and what do you do instead?

*Check:* The HTTP status code (200) was committed on the first byte of the response — it cannot be changed after streaming has started. Instead, catch the exception inside the generator and yield an in-band error SSE event: `data: {"event": "error", "detail": "..."}\n\n`. The client reads this event and can display an error message or retry. Monitor these in-band error events explicitly — HTTP 5xx metrics will not capture them.

---

**Q3:** What does a `finish_reason="length"` in the final streaming chunk mean, and what should the endpoint do about it?

*Check:* The model was cut off by `max_tokens` before finishing its response — the output is truncated mid-sentence. The endpoint should either: increase `max_tokens`, implement dynamic `max_tokens` based on input length, or yield a truncation warning event to the client so it can display a partial result disclaimer. Silently delivering a truncated response as if complete is incorrect — it may mislead users.

---

**Q4:** What is a heartbeat event in SSE, why is it necessary, and what is the correct SSE wire format for it?

*Check:* A heartbeat is a periodic keep-alive signal sent during long-running streams to prevent load balancers from killing idle connections. Load balancers have idle timeout thresholds (typically 30–60 seconds); if no bytes are sent during generation for longer than the timeout, the connection is killed. The correct SSE wire format for a heartbeat is a comment line: `: keep-alive\n\n` — the colon prefix marks it as a comment, which is invisible to SSE event listeners but still transmits bytes to keep the connection alive.

---

**Q5:** You are reviewing a senior engineer's streaming endpoint. They use `except Exception: pass` inside the async generator. What specific production problem does this create and what is the correct pattern?

*Check:* `except Exception` catches `asyncio.CancelledError` (which inherits from `BaseException`, so it actually won't be caught by `except Exception` in Python 3.8+, but the engineer may not know this — the concern is valid for older patterns or if they use `except BaseException`). The real issue is: the generator continues running after the client disconnects, generating tokens and incurring API cost for a closed connection. The correct pattern: catch `asyncio.CancelledError` explicitly, log the disconnect, and re-raise it. Handle real exceptions (`except Exception`) by yielding an error event.

## SUMMARY

Streaming changes when data is delivered, not how it is generated. The model does the same work — streaming moves each token from "delivered at completion" to "delivered immediately."

**First-token latency (TTFT)** is the UX metric that streaming optimizes. Users perceive speed by when text starts appearing. Measure TTFT at the endpoint level, track it in production, and alert when P95 TTFT exceeds your threshold. Total generation time is a cost and throughput concern, not a UX concern.

**The SSE endpoint pattern** is an async generator that yields `data: {...}\n\n` events forwarded to a `StreamingResponse`. Use `AsyncAzureOpenAI` to avoid blocking the event loop. Set `X-Accel-Buffering: no` to prevent reverse proxy buffering. Always emit a terminal event so the client knows when the stream is complete.

**Error handling in streaming is in-band.** The HTTP status code is committed on the first byte. All errors that occur mid-stream must be yielded as error events inside the generator. Monitor error event rates in the stream, not HTTP 5xx metrics.

**Three production additions beyond the happy path:** heartbeat events to prevent load balancer timeouts on long generations, a hard timeout via `asyncio.timeout()` to prevent stalled generators from blocking indefinitely, and explicit `asyncio.CancelledError` handling to stop generating tokens when the client disconnects.

## PRACTICAL LANDSCAPE

### When to stream vs not stream

| Scenario | Stream? | Reason |
|---|---|---|
| Document summarization (200+ tokens) | Yes | TTFT matters; user waits while reading |
| Multi-paragraph content generation | Yes | Progressive rendering improves perceived speed |
| Ticket classification (single label) | No | 1–2 chunks; no perceptible benefit |
| Structured extraction (JSON output) | No | JSON must be complete before parsing; partial JSON is unusable |
| Tool calling (tool arguments) | Possible but complex | Arguments arrive in fragments; accumulate before parsing |
| Chat message completion | Yes | Primary use case; UX depends on progressive text |

### SSE vs WebSockets for streaming LLM responses

| Feature | SSE | WebSockets |
|---|---|---|
| Direction | Server → client (unidirectional) | Bidirectional |
| Protocol | Plain HTTP | Separate WS handshake |
| Browser support | Native `EventSource` API | Native `WebSocket` API |
| Reconnect on disconnect | Automatic (EventSource reconnects) | Manual |
| Load balancer compatibility | Standard HTTP — widely supported | Requires WS-aware proxy config |
| **When to use** | Streaming LLM responses — most cases | Real-time bidirectional chat, live collaboration |

SSE is the right choice for streaming LLM responses. It uses standard HTTP, needs no special proxy configuration, and the browser `EventSource` handles reconnection. WebSockets add complexity without benefit for the unidirectional server-to-client streaming pattern.
