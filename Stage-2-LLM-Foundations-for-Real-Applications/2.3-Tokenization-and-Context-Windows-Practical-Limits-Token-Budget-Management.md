## CONCEPT 1: What Tokenization Is — How Text Actually Becomes Tokens
<!-- cognitive job: establish that tokenization is not splitting on spaces, that subword units are the real unit, and that this has direct consequences for cost and context window usage -->

| Dimension | Answer |
|---|---|
| **What it is** | Tokenization is the process of splitting text into subword units using a learned vocabulary — not words, not characters, but fragments that balance vocabulary size against sequence length |
| **Why it matters** | Every token is a billing unit, a context window unit, and a latency unit — understanding how text tokenizes tells you exactly what a prompt will cost, whether it fits in context, and how long inference will take |
| **When to use it** | Use this understanding whenever you are estimating cost, checking context window fit, designing prompts for non-English content, or debugging unexpected token counts |
| **When NOT to use it** | Do not assume token counts from character or word counts — the relationship is content-dependent and varies enough to break cost models and context window assumptions |
| **What juniors miss** | They estimate tokens as "roughly one token per word." This approximation holds for plain English prose but fails badly for code, numbers, non-Latin scripts, and structured data like JSON — which are all common in production AI systems |
| **Senior signal** | "I never estimate token counts — I measure them. `tiktoken` is in every service that accepts variable-length input, and the token count check runs before the model call, not after the context error." |

### Mental Model

Tokenization is like a compression scheme for language. Instead of treating every character as a unit (too many units, very long sequences) or every word as a unit (too many unique words to handle, poor handling of new words), tokenizers use subword units — fragments that appear frequently enough to have their own slot in the vocabulary.

Common English words like "the," "is," "in" get their own single token. Long or rare words get split: "un-happy-ness" might become three tokens. Non-English scripts and technical content get split more aggressively because the tokenizer was trained primarily on English text — the subword vocabulary reflects English frequency patterns, not universal language patterns.

The result: the same semantic content expressed in different languages, formats, or styles will tokenize to very different lengths — which means very different costs and very different context window consumption.

### How the Tokenizer Actually Splits Text

```python
import tiktoken

enc = tiktoken.encoding_for_model("gpt-4o")


def show_tokens(text: str) -> None:
    tokens = enc.encode(text)
    decoded = [enc.decode([t]) for t in tokens]
    print(f"Text:   {text!r}")
    print(f"Tokens: {decoded}")
    print(f"Count:  {len(tokens)}")
    print()


# Common English words → usually one token each
show_tokens("the cat sat")
# Text:   'the cat sat'
# Tokens: [' the', ' cat', ' sat']       ← note leading space — part of the token
# Count:  3

# Longer words split into subword fragments
show_tokens("tokenization")
# Text:   'tokenization'
# Tokens: ['token', 'ization']
# Count:  2

show_tokens("unhappiness")
# Text:   'unhappiness'
# Tokens: ['un', 'happiness']
# Count:  2

# Numbers are expensive — each digit often its own token
show_tokens("123456789")
# Text:   '123456789'
# Tokens: ['123', '456', '789']           ← varies by model; can be 1 token per digit
# Count:  3

# Whitespace and indentation count — critical for code
show_tokens("    if x > 0:")
# Text:   '    if x > 0:'
# Tokens: ['    ', 'if', ' x', ' >', ' ', '0', ':']
# Count:  7                               ← 4 spaces = 1 token, but other whitespace adds up

# Emoji and special characters are expensive
show_tokens("Great job! 🎉")
# Text:   'Great job! 🎉'
# Tokens: ['Great', ' job', '!', ' 🎉']
# Count:  4                               ← emoji: 1–4 tokens depending on the character
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `enc = tiktoken.encoding_for_model("gpt-4o")` | Loads the tokenizer for the specific model | Different models use different tokenizers with different vocabularies — always use the tokenizer that matches the model you are calling |
| `enc.decode([t]) for t in tokens` | Decodes each token ID back to its text fragment | This is the only way to actually see how the tokenizer split the input — visual inspection of what each token is prevents surprises |
| `' the'` has a leading space | The space before a word is part of the token | This is a BPE (Byte Pair Encoding) convention — word-boundary spaces are absorbed into the following token. Stripping spaces before tokenizing changes the token count. |
| `'123456789'` → `['123', '456', '789']` | Numbers split into groups, not always digit-by-digit | Numeric content is expensive to tokenize and is handled differently across models. Financial data, IDs, and timestamps add up faster than prose. |
| `'    if x > 0:'` → 7 tokens for 12 characters | Code indentation consumes tokens | In code-heavy prompts, deeply nested code can produce token counts that are 2–3x higher than the character count suggests. |

## CONCEPT 2: How Token Counts Vary by Content Type — The Production Surprise
<!-- cognitive job: build a concrete mental map of which content types are token-expensive and why — this directly shapes cost modeling, context window planning, and truncation strategy -->

| Dimension | Answer |
|---|---|
| **What it is** | Different content types — English prose, code, JSON, non-Latin scripts, numbers, special characters — tokenize to very different lengths for the same amount of user-visible content |
| **Why it matters** | Token count is what you pay and what you fit into the context window — not character count, not word count, not line count. A 500-character input can be 80 tokens or 400 tokens depending on content type. |
| **When to use it** | Use this knowledge when cost modeling for a new use case, designing context window budgets for multilingual systems, and deciding what content types to truncate first when the budget is tight |
| **When NOT to use it** | Do not use these ratios as fixed constants — they are approximations that vary by tokenizer version and specific content. Always measure on representative samples of your actual data. |
| **What juniors miss** | They test in English, set cost budgets based on English token rates, then deploy in a multilingual context where non-Latin scripts tokenize at 3–5x the English rate — and the cost model collapses. |
| **Senior signal** | "Token efficiency is a content-type problem, not a model problem. Before I set a token budget, I run the actual tokenizer on a representative sample of every input type this service will receive." |

### Mental Model

Think of tokenization like exchanging currency. The tokenizer's vocabulary was built around English text — so English words have favorable "exchange rates" (one common word = one token). For other currencies (Arabic, Chinese, Japanese, code with unusual patterns), the exchange rate is less favorable — more tokens required to represent the same semantic content.

The exchange rate varies predictably: languages that share Latin roots and script with English are cheaper than languages with non-Latin scripts. Highly structured data (JSON, XML) is expensive because structural characters consume tokens. Code is variable — standard library calls are cheap, but deeply indented or heavily commented code is expensive.

| Code line / block | What it does | Why it matters |
|---|---|---|
| `arabic_text` ratio ≈ 0.611 vs `english_prose` ≈ 0.193 | Arabic is ~3.2x more token-expensive per character than English prose | A multilingual system with the same character budget for all languages will use 3x more tokens for Arabic content — tripling cost and shrinking available context for that user's session |

| `python_code` ratio ≈ 0.316 | Code is ~1.6x more expensive than prose per character | Indentation, punctuation, operators, and symbol-dense lines all add tokens. A 500-line Python file can use significantly more context than the line count implies. |
| `json_structured` ratio ≈ 0.338 | JSON is expensive per character | Every quote, colon, bracket, and comma is a token. Systems that pass large JSON objects in prompts (tool call results, API responses) should monitor this carefully. |
| `numeric_data` ratio ≈ 0.368 | Numbers and timestamps are expensive | Financial systems, analytics prompts, and data-heavy contexts that include raw numbers and timestamps will use more tokens than a prose description of the same information. |

## CONCEPT 3: Token Budget Management — Designing for the Limit Before You Hit It
<!-- cognitive job: establish token budget management as a production engineering discipline with concrete patterns, not just an awareness concern -->

| Dimension | Answer |
|---|---|
| **What it is** | Token budget management is the engineering practice of measuring, reserving, allocating, and monitoring token usage for every LLM call before the call is made — not after context errors appear |
| **Why it matters** | Without active budget management: context overflow causes silent truncation or hard errors, cost tracking is impossible, and latency is unpredictable because input length is uncontrolled |
| **When to use it** | Every endpoint or function that calls an LLM with variable-length input requires token budget management — this is every production LLM call except the most rigidly constrained fixed prompts |
| **When NOT to use it** | Fixed, unchanging prompts with no user-controlled content can skip per-call counting — but even then, store the known token count and alert if the prompt ever changes |
| **What juniors miss** | They add token counting after a context error appears in production. The correct pattern is to build it before the first production call — it is a cheaper fix than debugging silent truncation in a live system. |
| **Senior signal** | "Every LLM endpoint I ship has three things: a token count before the call, a defined behavior when the budget is exceeded, and a usage log for cost attribution. These are not optional — they are as mandatory as error handling." |

### Mental Model

Token budget management is like fuel planning for a long-haul flight. You know the tank size (context window). You know what must be in the tank before takeoff (system prompt, output reservation). You know the variable consumption (user message, retrieved content, conversation history). You calculate whether there is enough fuel before you leave the gate — not after you run out at 30,000 feet. And you log every flight's fuel consumption so the airline can accurately bill and forecast.

The analogy breaks down in one important place: when an airplane runs low on fuel, the pilot knows. When an LLM prompt exceeds the context window, neither the model nor the caller is automatically informed — the API may silently truncate the input and return a response that is based on partial information with no indication anything went wrong. This is why explicit pre-call measurement is not a nice-to-have.

### The Production Token Budget Pattern

```python
from __future__ import annotations

import logging
from dataclasses import dataclass, field
from typing import Literal

import tiktoken

logger = logging.getLogger(__name__)


# ── Model registry ─────────────────────────────────────────────────────────────

MODEL_CONTEXT_LIMITS: dict[str, int] = {
    "gpt-4o":                128_000,
    "gpt-4o-mini":           128_000,
    "gpt-4-turbo":           128_000,
    "gpt-3.5-turbo":          16_385,
    "gpt-3.5-turbo-16k":      16_385,
}

DEFAULT_OUTPUT_RESERVE = 2_000   # tokens held back for model output


# ── Token counting ──────────────────────────────────────────────────────────────

def count_tokens(text: str, model: str = "gpt-4o") -> int:
    """Count tokens for a string using the model's own tokenizer."""
    enc = tiktoken.encoding_for_model(model)
    return len(enc.encode(text))


def count_messages_tokens(messages: list[dict], model: str = "gpt-4o") -> int:
    """
    Count tokens for a messages list, including OpenAI's per-message overhead.
    OpenAI adds ~4 tokens per message for role/formatting metadata.
    """
    enc = tiktoken.encoding_for_model(model)
    total = 0
    for message in messages:
        total += 4                                    # per-message overhead
        total += len(enc.encode(message["role"]))
        total += len(enc.encode(message["content"]))
    total += 2                                        # reply priming overhead
    return total


# ── Budget dataclass ────────────────────────────────────────────────────────────

@dataclass
class TokenBudget:
    model: str
    system_tokens: int
    output_reserve: int = DEFAULT_OUTPUT_RESERVE
    context_limit: int = field(init=False)
    available_for_content: int = field(init=False)

    def __post_init__(self) -> None:
        if self.model not in MODEL_CONTEXT_LIMITS:
            raise ValueError(f"Unknown model: {self.model!r}. Add it to MODEL_CONTEXT_LIMITS.")
        self.context_limit = MODEL_CONTEXT_LIMITS[self.model]
        self.available_for_content = (
            self.context_limit
            - self.system_tokens
            - self.output_reserve
        )
        if self.available_for_content <= 0:
            raise ValueError(
                f"System prompt ({self.system_tokens} tokens) + output reserve "
                f"({self.output_reserve} tokens) exceeds context limit "
                f"({self.context_limit} tokens) for model {self.model!r}."
            )

    def fits(self, token_count: int) -> bool:
        return token_count <= self.available_for_content

    def remaining(self, used: int) -> int:
        return self.available_for_content - used


# ── Prompt assembler ────────────────────────────────────────────────────────────

TruncationStrategy = Literal["drop_oldest", "drop_middle", "summarize_flag"]


def assemble_prompt(
    system_prompt: str,
    user_message: str,
    retrieved_chunks: list[str],
    conversation_history: list[dict],
    model: str = "gpt-4o",
    truncation_strategy: TruncationStrategy = "drop_oldest",
) -> tuple[list[dict], dict]:
    """
    Assemble a messages list that fits within the model's context window.
    Returns the messages and a usage metadata dict for logging.

    Priority order (what gets cut first if budget is tight):
      1. Oldest conversation history turns
      2. Retrieved chunks from the end of the list
      3. User message and system prompt are NEVER cut — they are non-negotiable
    """
    system_tokens = count_tokens(system_prompt, model)
    budget = TokenBudget(model=model, system_tokens=system_tokens)

    user_tokens = count_tokens(user_message, model)
    if user_tokens > budget.available_for_content:
        raise ValueError(
            f"User message alone ({user_tokens} tokens) exceeds available content "
            f"budget ({budget.available_for_content} tokens). "
            "Cannot assemble a valid prompt."
        )

    used = user_tokens

    # ── Fit conversation history ───────────────────────────────────────────────
    fitted_history: list[dict] = []
    history = list(conversation_history)  # copy to avoid mutating caller's list

    if truncation_strategy == "drop_oldest":
        # Process from newest to oldest — keep the most recent turns
        history_reversed = list(reversed(history))
        for turn in history_reversed:
            turn_tokens = count_tokens(turn["content"], model) + 4  # +4 overhead
            if used + turn_tokens > budget.available_for_content:
                logger.warning(
                    "Conversation history truncated at %d tokens used "
                    "(budget: %d). Oldest turns dropped.",
                    used,
                    budget.available_for_content,
                )
                break
            fitted_history.insert(0, turn)
            used += turn_tokens

    # ── Fit retrieved chunks ───────────────────────────────────────────────────
    fitted_chunks: list[str] = []
    chunks_dropped = 0
    for chunk in retrieved_chunks:
        chunk_tokens = count_tokens(chunk, model)
        if used + chunk_tokens > budget.available_for_content:
            chunks_dropped += 1
            continue
        fitted_chunks.append(chunk)
        used += chunk_tokens

    if chunks_dropped:
        logger.warning(
            "%d retrieved chunk(s) dropped — token budget exhausted at %d tokens.",
            chunks_dropped,
            used,
        )

    # ── Build messages ─────────────────────────────────────────────────────────
    context_block = "\n\n---\n\n".join(fitted_chunks)
    user_content = (
        f"{context_block}\n\n{user_message}" if context_block else user_message
    )

    messages = [
        {"role": "system", "content": system_prompt},
        *fitted_history,
        {"role": "user",   "content": user_content},
    ]

    # ── Usage metadata for logging and cost attribution ────────────────────────
    usage_meta = {
        "model":                  model,
        "context_limit":          budget.context_limit,
        "system_tokens":          system_tokens,
        "history_turns_kept":     len(fitted_history),
        "history_turns_dropped":  len(conversation_history) - len(fitted_history),
        "chunks_kept":            len(fitted_chunks),
        "chunks_dropped":         chunks_dropped,
        "estimated_input_tokens": used + system_tokens,
        "output_reserve":         budget.output_reserve,
    }

    return messages, usage_meta
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `MODEL_CONTEXT_LIMITS` registry | Stores context limits per model in one place | Model limits change as vendors release updates. A central registry means one place to update when GPT-4o-mini's context window changes — not a scattered grep through the codebase. |
| `count_messages_tokens` adds 4 per message | Accounts for OpenAI's per-message overhead | The official `messages` format adds ~4 tokens per turn for role and formatting metadata. Ignoring this causes systematic undercount that accumulates over long conversation histories. |
| `TokenBudget.__post_init__` raises on negative budget | Fails explicitly if system prompt + reserve already exceeds limit | A system prompt that consumes the entire context window is a design flaw, not a runtime edge case. Raising at construction time catches it in development, not in production at 2am. |
| Priority: user message → history → chunks | Defines what gets cut when budget is exceeded | The user message is always included — cutting it makes the call meaningless. History is cut oldest-first — the most recent context is most relevant. Chunks are dropped from the end — the highest-ranked chunks appear first and are preserved. |
| `usage_meta` dict returned alongside messages | Captures token usage for logging and cost attribution | This dict goes into your observability layer. When the CTO asks why the AI budget doubled this month, you can answer: "Arabic-language support increased average input tokens by 3.1x — here is the per-language breakdown." |

### The Token Budget Design Checklist

Run this for every new LLM endpoint before the first production deployment.

```text
STEP 1: Identify all input components and their variability.
        [ ] System prompt — fixed? version-controlled?
        [ ] User message — what is the max realistic length? the p99 length?
        [ ] Conversation history — how many turns? does it grow unbounded?
        [ ] Retrieved content (RAG) — how many chunks? average chunk size?
        [ ] Tool call results — how large can API responses be?
        [ ] Any injected data (user profile, context records, JSON payloads)?
                        │
                        ▼
STEP 2: Measure token counts on representative samples of each component.
        Do NOT estimate from character or word counts.
        Use the actual tokenizer for the actual model.
        Measure p50, p90, p99 — not just average.
                        │
                        ▼
STEP 3: Calculate the token budget.
        available = context_limit - system_tokens - output_reserve
        Check: does p99 of total input fit within available?
        If not: you need a truncation strategy before deployment.
                        │
                        ▼
STEP 4: Define priority and truncation strategy.
        Non-negotiable (never cut): system prompt, current user message
        Cut first: oldest history turns, lowest-ranked retrieved chunks
        Cut second: middle retrieved chunks
        Never cut silently: always log what was dropped and why
                        │
                        ▼
STEP 5: Implement pre-call token count check.
        Count tokens → check against budget → truncate if needed → then call the model.
        The check must happen before the API call — never after.
                        │
                        ▼
STEP 6: Log token usage per call.
        Log: model, input token count, output token count, what was truncated.
        Aggregate: per-user, per-endpoint, per-language for cost attribution.
        Alert: when average input token count drifts more than 20% from baseline.
```

### Red Flags — Token Budget Warning Signs in Code Review and Architecture

| What you see | Why it should raise a flag |
|---|---|
| No token counting before LLM calls | Silent truncation or context errors in production — no visibility into which calls are affected |
| `len(text.split())` used as token estimate | Word count approximation — fails for code, JSON, and non-English content by up to 4x |

## COMMON MISTAKES
<!-- cognitive job: inoculate against predictable errors by teaching error structure, not just correct answers -->

### Mistake 1: Estimating Token Counts Instead of Measuring Them

**Mistaken instinct:** "One token is roughly one word for English — we can estimate from word count and add a 20% buffer."

**Why it feels right:** The one-token-per-word approximation works reasonably well for plain English prose. Adding a buffer feels like responsible conservatism. This has probably been accurate enough in past systems.

**Why it fails:** The approximation fails for the content types that dominate production AI systems: code (1.5–2x more than prose), JSON and structured data (1.7–2x), non-Latin scripts (3–5x), numbers and identifiers (1.5–2x), long compound words and technical jargon (more splits). The buffer that handles 20% English variance cannot handle 300% Arabic variance. Every production AI system eventually receives inputs that were not considered in the original estimate.

**Better move:** Use `tiktoken` (or the equivalent tokenizer for your model) on representative samples of every content type your system will actually receive. Measure p50, p90, and p99 — not just average. Set budgets from measured data, not from approximations.

---

### Mistake 2: Not Reserving Token Budget for Output

**Mistaken instinct:** "The context window is for input — the output just comes back. We don't need to account for it."

**Why it feels right:** Input and output feel like separate things — you send in, you get back. The context window limit sounds like an input limit.

**Why it fails:** For most LLM APIs, the context window limit covers both input and output tokens combined. If you fill 127,999 tokens with input on a 128K model and ask it to produce a 2,000-token response, one of two things happens: the response is truncated at 1 token, or the call returns an error. Either way, the output is not what you expected. The model also needs token budget to "think" — chain-of-thought reasoning that improves quality requires output space.

**Better move:** Always reserve explicit output space when calculating input budgets. A reasonable default is 1,500–2,500 tokens depending on expected response length. For endpoints that produce structured outputs (JSON responses, multi-step plans), measure the actual output length distribution and reserve accordingly.

---

### Mistake 3: Treating All Languages with the Same Token Budget

**Mistaken instinct:** "We have one token budget — it applies to all users."

**Why it feels right:** A unified policy is simpler to implement and maintain. The context window limit is the same regardless of language. If the budget works for English, why would it fail for other languages?

**Why it fails:** Non-Latin scripts tokenize at 3–5x the rate of English. A 500-token budget for user messages allows a detailed English question or a very short Arabic one. A system prompt that fits comfortably in 300 tokens in English occupies significantly more tokens if any of its text is translated or mixed-language. If the same conversation history retention policy applies to all languages, Arabic and Japanese conversations will fill the history budget 3–4x faster than English, leaving less room for retrieved context.

**Better move:** Detect the language of each input (or infer it from user settings) and apply per-language token budgets. For multilingual systems, measure actual token counts by language on representative data, then set budgets from that data — not from English-centric estimates.

---

### Mistake 4: Logging Token Usage Only When There Is an Error

**Mistaken instinct:** "We'll add token logging once we see a problem."

**Why it feels right:** Logging is overhead. "We'll cross that bridge when we come to it" is a reasonable engineering stance for edge cases. If there are no context errors, there is no problem to debug.

**Why it fails:** Silent truncation does not produce errors. A prompt that exceeds the context window produces a response — just one based on partial information. The response looks normal. Without per-call token logging, there is no way to know truncation is happening, no way to attribute cost increases to their sources, no way to detect drift when a prompt change doubles the system token count, and no way to answer questions from finance about why the AI budget changed.

**Better move:** Log token usage on every call — not just on errors. Capture: model, input token count, output token count, what (if anything) was truncated, user/session identifier for attribution. This data costs almost nothing to store and is essential for cost management and debugging.

1. Tokenization splits text into subword fragments using a learned vocabulary — not words, not characters — and the fragments reflect English frequency patterns, making non-Latin scripts 3–5x more token-expensive per character.
2. Token count is the billing unit and context window unit — never estimate from word or character count; always measure with the model's actual tokenizer on representative samples of your real input types.

| Easy confusion | Correct distinction |

### Interview Q&A

**Q: What is a token, and why does it matter for production LLM systems?**

> "A token is the atomic unit the LLM processes — a subword fragment produced by a learned tokenizer, typically 3–4 characters in common English words. It matters for production systems for three reasons: it is the billing unit, so cost scales directly with token count; it is the context window unit, so the amount of information you can include in one call is measured in tokens not characters; and it is the latency unit, because output latency scales with output token count. The practical complication is that token count is content-dependent in ways that surprise teams: a 100-character Arabic sentence uses 3–5x more tokens than a 100-character English sentence, and code with deep indentation uses 1.5–2x more than plain prose. This means estimates based on word or character counts produce incorrect budgets. I always measure with the actual tokenizer — `tiktoken` for OpenAI-compatible models — on representative samples of the actual content types the system will process."

---

**Q: What is silent truncation and why is it dangerous?**

> "Silent truncation happens when a prompt exceeds the model's context window limit and the excess content is removed before the model call, without raising an error. The model returns a response — it just does so based on partial information, with no indication that anything was missing. This is the most dangerous context window failure mode because it is invisible: the system logs no error, the response looks syntactically valid, and the caller has no signal that anything went wrong. The model's answer reflects only the content it received — which may have had critical sections removed. The production defense is pre-call token counting: measure the total input token count before every API call with variable-length content, compare against the budget, and truncate deliberately with a defined priority before the call — not reactively after an error. Token usage should also be logged per call so that patterns of high-token calls or frequent truncation are visible in monitoring dashboards."

---

**Q: How does tokenization affect the cost of a multilingual AI system?**

> "Dramatically, and in ways that are easy to underestimate if you only test in English. Tokenizers for the major LLMs were trained primarily on English text, so the vocabulary reflects English subword frequency patterns. Common English words map to one or two tokens. Arabic and CJK characters map to many more — the tokenizer has fewer fragments that match those scripts, so it falls back to more token IDs per character. Arabic text tokenizes at roughly 3–4x the token count of equivalent English content; Japanese and Chinese are similar. For a multilingual system, this means conversation history for Arabic users fills the context budget 3–4x faster, the same user message costs 3–4x more, and the same conversation-length limit in English allows far fewer turns in Arabic. The correct response is per-language token budgets, language detection at the request boundary, and measuring actual token counts by language on real production data — not applying English estimates to all markets."
