## Objective

Set up the infrastructure configuration for two providers (Azure OpenAI as primary, OpenAI direct as fallback), implement a provider client factory in `app/routing/providers.py` that returns the right client for each task, and wire the fallback path into `parse_llm_output()` so a provider outage degrades gracefully rather than failing completely.

## Where This Fits in the System

**State:** After 2.12, `model_map.py` has primary and fallback slots for each task — but both slots point to the same Azure OpenAI deployment. The fallback does nothing because the fallback is the same provider as the primary. If Azure OpenAI has an outage, every endpoint fails.

**Gap:** The copilot has no real redundancy. A provider incident — Azure OpenAI regional outage, rate limit exhaustion, API version deprecation — takes down the entire application. The `fallback` field in `ModelConfig` exists but is unused in practice.

**Unlocks:** Two providers are wired and tested. When the primary provider returns an error, `parse_llm_output()` retries on the fallback provider automatically. 2.15 uses the provider config to apply different caching strategies per provider (Azure OpenAI prompt caching vs. OpenAI prompt caching have different TTLs and prefix requirements).

## Core Mental Model

### The Primary-Fallback Contract

Every external dependency your system calls should have two properties:

```
1. A primary path  — the preferred provider under normal conditions
2. A fallback path — an alternative that activates automatically on primary failure
```

This is not a high-availability luxury. It is the minimum viable reliability contract for any production system that calls a third-party API. APIs have outages. Rate limits get hit. Regions go down. Without a fallback, every provider incident becomes a user-facing outage.

The key design decision is: **what triggers the fallback?**

```
Too aggressive: fallback on every slow response → unnecessary cost, cache misses
Too conservative: fallback only on complete provider outage → misses rate limit errors
Right: fallback on non-retryable provider errors — 429, 5xx, connection timeout
```

The fallback is not a retry on the same provider. It is a switch to a different one.

## Main Concepts + Production Reality

### Concept 1 — The Provider Landscape

**Scenario:**
You need to choose a primary LLM provider for a production B2B application. Three options are commonly evaluated. Which do you pick, and what do you wire as the fallback?

**The three main options and their tradeoffs:**

```
Azure OpenAI
  Models:     GPT-4o, GPT-4o-mini (same models as OpenAI, deployed on Azure)
  Strengths:  enterprise compliance (SOC2, HIPAA, GDPR), data residency control,
              private endpoints, SLA with financial backing, Azure AD integration
  Weaknesses: deployment setup required (not instant), regional availability varies,
              new model versions lag OpenAI direct by weeks
  Best for:   enterprise B2B, regulated industries, orgs already on Azure

OpenAI direct
  Models:     GPT-4o, GPT-4o-mini, o1, o3
  Strengths:  latest models first, simple API key setup, broader model selection
  Weaknesses: no enterprise SLA by default, data processed in OpenAI infrastructure,
              rate limits can be tighter on new accounts
  Best for:   startups, prototypes, teams needing latest models immediately

Anthropic (Claude)
  Models:     Claude Opus, Sonnet, Haiku (small/medium/large tiers)
  Strengths:  strong on long-context tasks, instruction following, safety defaults
  Weaknesses: different API shape (messages format differs slightly), no Azure hosting
  Best for:   long-document tasks, cases where Claude's reasoning style is a better fit
```

**What this means for the copilot:**

```
Primary:   Azure OpenAI
  Reason:  enterprise compliance, data residency, Azure AD auth
           consistent with corporate infrastructure

Fallback:  OpenAI direct (same model family, minimal prompt changes)
  Reason:  fastest path to a working fallback — same model, same API shape,
           only the endpoint and auth change
```

**Engineering implication:**

```
Same model family as fallback  →  zero prompt changes needed when switching
Different model family         →  prompts may need tuning for the fallback
                                  (Claude has different system prompt behavior
                                  than GPT models)

Rule: choose a fallback provider where the API shape is identical or near-identical
to the primary. OpenAI → Azure OpenAI (or reverse) is the lowest-friction fallback.
Anthropic as fallback to OpenAI requires prompt testing before you need it.
```

**Common trap:**
Choosing a fallback provider without testing your prompts against it. A fallback that produces degraded output under stress is not a real fallback — it is a second failure mode. Test prompts on both providers before declaring the fallback wired.

### Concept 2 — What Actually Differs Between Providers

**Scenario:**
You have Azure OpenAI working. You want to add OpenAI direct as a fallback. You assume you just swap the endpoint. Three things break.

**What differs between Azure OpenAI and OpenAI direct:**

```
1. Authentication
   Azure OpenAI:  api_key + azure_endpoint + api_version + deployment name
   OpenAI direct: api_key only (no endpoint, no api_version)

   Azure client:
     client = AzureOpenAI(
         azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
         api_key=os.environ["AZURE_OPENAI_API_KEY"],
         api_version="2024-02-01",
     )
     client.chat.completions.create(model="gpt-4o", ...)  # model = deployment name

   OpenAI client:
     client = OpenAI(api_key=os.environ["OPENAI_API_KEY"])
     client.chat.completions.create(model="gpt-4o", ...)  # model = model name

2. Model names
   Azure OpenAI:  model parameter = deployment name (you created: "gpt-4o-prod")
   OpenAI direct: model parameter = model name ("gpt-4o", "gpt-4o-mini")
   → your model_map.py needs provider-aware model name resolution

3. Prompt caching
   Azure OpenAI:  automatic prefix caching — no config needed, cache hits logged
   OpenAI direct: automatic prefix caching on supported models
   → behavior is similar but cache hit rates may differ; monitor separately
```

**What is identical:**
```
   API response shape    →  same ChatCompletion object
   streaming protocol   →  same SSE chunks
   tool call format     →  same tool_calls structure
   error types          →  similar (rate limit = 429, server error = 5xx)
```

**What this is called:** provider abstraction — a client factory that hides which provider is active behind a consistent interface.

**Engineering implication:**

```
Abstract the client creation behind a factory function
  get_client("azure")  →  returns AzureOpenAI client
  get_client("openai") →  returns OpenAI client

Services call the API the same way regardless of which client they hold
The provider switch happens in the factory, not in every service
```

**Common trap:**
Spreading provider-specific logic (Azure endpoint, api_version, deployment name mapping) across multiple service files. When the provider config changes, you're hunting through the codebase. All provider logic belongs in one place: `providers.py`.

### Concept 3 — Wiring the Fallback

**Scenario:**
Azure OpenAI returns a 429 (rate limit) at 2pm on a Tuesday. 300 tickets are queued. You want the system to automatically switch to OpenAI direct for those requests — without an engineer having to manually flip a switch.

**What needs to happen:**

```
Request arrives
     ↓
Try primary provider (Azure OpenAI)
     ↓
Success?   →  return result
     ↓
Non-retryable error? (429, 5xx, timeout)
     ↓
Try fallback provider (OpenAI direct)
     ↓
Success?   →  log "used fallback provider" + return result
     ↓
Fallback also failed?  →  return ParseResult(status=ERROR)
```

**Three error types and how to handle each:**

```
Retryable on same provider:
  RateLimitError with retry-after header  →  wait and retry same provider
  Transient 500                           →  retry same provider once

Switch to fallback:
  RateLimitError with no retry-after      →  switch to fallback immediately
  Persistent 5xx (2+ attempts)            →  switch to fallback
  Connection timeout                      →  switch to fallback

Do not retry either provider:
  400 Bad Request (your prompt/schema is wrong)  →  return error, fix the prompt
  401 Unauthorized (bad API key)                 →  return error, alert on-call
  Content filter (content_filter)               →  return error, log input
```

**What this is called:** provider fallback with error classification — routing API errors to the right response (retry same, switch provider, or surface error) based on whether the failure is transient, capacity-related, or a client-side error.

**Engineering implication:**

```
Only switch providers on capacity/availability failures (429, 5xx, timeout)
Never switch providers on client errors (400, 401) — fallback won't help
Log every fallback activation with: provider, task, error type, timestamp
Fallback activation rate is a monitoring metric — spike = primary provider issue
```

**Common trap:**
Retrying a 400 Bad Request on the fallback provider. If your request is malformed, the fallback returns the same 400. You've doubled your cost and latency for no benefit. Classify errors before deciding to retry or switch.

```python
# app/routing/providers.py
"""
Provider client factory for ticket-ops-api.

Abstracts Azure OpenAI (primary) and OpenAI direct (fallback) behind
a consistent interface. Services never import provider-specific clients
directly — they use get_client() and call_with_fallback().

Error classification determines whether a failure triggers a fallback:
  - Capacity/availability errors (429, 5xx, timeout) → try fallback
  - Client errors (400, 401, content_filter)         → surface immediately
"""

import logging
import os
from dataclasses import dataclass
from enum import Enum

from openai import AzureOpenAI, OpenAI, RateLimitError, APIStatusError, APIConnectionError

logger = logging.getLogger(__name__)


class Provider(str, Enum):
    AZURE  = "azure"
    OPENAI = "openai"


@dataclass(frozen=True)
class ProviderConfig:
    provider: Provider
    model_small:  str
    model_medium: str
    model_large:  str


# ---------------------------------------------------------------------------
# Provider configurations — loaded from environment
# ---------------------------------------------------------------------------

PRIMARY_CONFIG = ProviderConfig(
    provider=Provider.AZURE,
    model_small  = os.environ.get("MODEL_SMALL",  "gpt-4o-mini"),
    model_medium = os.environ.get("MODEL_MEDIUM", "gpt-4o"),
    model_large  = os.environ.get("MODEL_LARGE",  "gpt-4o"),
)

FALLBACK_CONFIG = ProviderConfig(
    provider=Provider.OPENAI,
    model_small  = os.environ.get("FALLBACK_MODEL_SMALL",  "gpt-4o-mini"),
    model_medium = os.environ.get("FALLBACK_MODEL_MEDIUM", "gpt-4o"),
    model_large  = os.environ.get("FALLBACK_MODEL_LARGE",  "gpt-4o"),
)


# ---------------------------------------------------------------------------
# Client factory
# ---------------------------------------------------------------------------

def get_client(provider: Provider) -> AzureOpenAI | OpenAI:
    """
    Return the API client for the given provider.
    Called once per request — clients are lightweight to construct.
    """
    if provider == Provider.AZURE:
        return AzureOpenAI(
            azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
            api_key=os.environ["AZURE_OPENAI_API_KEY"],
            api_version=os.environ.get("AZURE_OPENAI_API_VERSION", "2024-02-01"),
        )
    return OpenAI(api_key=os.environ["OPENAI_API_KEY"])


def resolve_model(deployment_name: str, config: ProviderConfig) -> str:
    """
    Map a deployment name (from model_map.py) to the correct model name
    for the given provider.

    Azure uses deployment names you created; OpenAI uses canonical model names.
    The mapping is approximate — both use the same model family here.
    """
    small  = os.environ.get("MODEL_SMALL",  "gpt-4o-mini")
    medium = os.environ.get("MODEL_MEDIUM", "gpt-4o")

    if deployment_name == small:
        return config.model_small
    if deployment_name in (medium, "gpt-4o"):
        return config.model_medium
    # Unknown deployment name — pass through and let the API reject it
    return deployment_name


# ---------------------------------------------------------------------------
# Error classification
# ---------------------------------------------------------------------------

def should_try_fallback(exc: Exception) -> bool:
    """
    Return True if the exception represents a capacity or availability failure
    that warrants switching to the fallback provider.

    Returns False for client errors (bad request, auth failure, content filter)
    where the fallback will return the same error.
    """
    if isinstance(exc, RateLimitError):
        return True   # 429 — capacity issue, fallback may have capacity

    if isinstance(exc, APIStatusError):
        if exc.status_code >= 500:
            return True   # 5xx — server-side failure, fallback may be healthy
        if exc.status_code in (400, 401, 403):
            return False  # client error — fallback won't help
        # 422 Unprocessable, etc. — treat as client error
        return False

    if isinstance(exc, APIConnectionError):
        return True   # network issue — fallback endpoint may be reachable

    return False  # unknown exception type — do not assume fallback helps


# ---------------------------------------------------------------------------
# Fallback-aware API call
# ---------------------------------------------------------------------------

def call_with_fallback(
    deployment: str,
    messages: list[dict],
    temperature: float,
    max_tokens: int,
    response_format: dict | None = None,
    tools: list[dict] | None = None,
    tool_choice: str = "auto",
    stream: bool = False,
) -> tuple:
    """
    Call the primary provider. On eligible errors, retry with the fallback.

    Returns: (response, provider_used: Provider)
    Raises: the original exception if both providers fail,
            or the primary exception if it is a client error.
    """
    kwargs = dict(
        messages=messages,
        temperature=temperature,
        max_tokens=max_tokens,
    )
    if response_format:
        kwargs["response_format"] = response_format
    if tools:
        kwargs["tools"] = tools
        kwargs["tool_choice"] = tool_choice
    if stream:
        kwargs["stream"] = stream

    # --- Primary provider attempt ---
    try:
        primary_client = get_client(PRIMARY_CONFIG.provider)
        primary_model  = resolve_model(deployment, PRIMARY_CONFIG)
        response = primary_client.chat.completions.create(
            model=primary_model, **kwargs
        )
        return response, PRIMARY_CONFIG.provider

    except Exception as primary_exc:
        if not should_try_fallback(primary_exc):
            logger.error(
                "Primary provider client error (no fallback): %s: %s",
                type(primary_exc).__name__, primary_exc,
            )
            raise

        logger.warning(
            "Primary provider failed (%s: %s) — trying fallback provider",
            type(primary_exc).__name__, primary_exc,
        )

    # --- Fallback provider attempt ---
    try:
        fallback_client = get_client(FALLBACK_CONFIG.provider)
        fallback_model  = resolve_model(deployment, FALLBACK_CONFIG)
        response = fallback_client.chat.completions.create(
            model=fallback_model, **kwargs
        )
        logger.info(
            "Fallback provider succeeded: deployment=%s fallback_model=%s",
            deployment, fallback_model,
        )
        return response, FALLBACK_CONFIG.provider

    except Exception as fallback_exc:
        logger.error(
            "Both providers failed. Primary: %s. Fallback: %s",
            primary_exc, fallback_exc,
        )
        raise fallback_exc
```

## Key Takeaways

- **Every external API call needs a fallback provider, not just a retry.** Retrying the same provider on a 429 or 5xx waits for the same degraded provider to recover. Switching to a different provider routes around the failure. These are different strategies for different failure types.

- **Same model family = lowest-friction fallback.** OpenAI ↔ Azure OpenAI uses the same GPT models, same API shape, same prompt format. No prompt changes needed when the provider switches. A different model family (e.g., Claude as fallback to GPT) requires prompt testing before you can trust it under pressure — test before you need it.

- **Classify errors before deciding to fall back.** 400 (bad request) on the fallback returns the same 400. 401 (auth failure) on the fallback returns the same 401. Only capacity and availability failures (429, 5xx, timeout) benefit from a provider switch. Blindly retrying client errors on the fallback doubles cost with no benefit.

- **All provider logic belongs in one file.** `providers.py` is the only place with Azure endpoints, api_version, provider-specific auth, and model name mapping. Services call `call_with_fallback()` and never import `AzureOpenAI` or `OpenAI` directly. When the provider config changes, one file changes.

- **Fallback activation rate is a monitoring signal.** A spike in fallback usage means the primary provider is degraded. Log every fallback activation with timestamp, provider, task, and error type. This data is more actionable than a generic "API error" log line.

- **Test your prompts on both providers before declaring the fallback ready.** A fallback that produces structurally different output is not a fallback — it is a second failure mode. Run your full test suite against the fallback provider in staging before treating it as production-ready.

## Optional Advanced Notes

**Provider routing beyond binary primary/fallback:** At high scale, some teams implement weighted routing — 90% of requests to the primary, 10% to the secondary — to keep the fallback warm and continuously validated. A cold fallback (never tested in production) may have silent issues that only surface during a real incident. Warm routing ensures the fallback is always exercised and its quality is always verified. Implement by adding a `weight` field to `ProviderConfig` and using `random.random()` to select the provider for each request.

**Anthropic as a fallback:** Claude's API uses `anthropic` SDK, not `openai`. The message format is similar but not identical — `system` is a top-level parameter, not a message in the list. Using Claude as a fallback requires a thin adapter that translates the OpenAI message format to the Anthropic format. Write this adapter before you need it in production. A fallback you've never tested is not a fallback. The adapter also needs prompt validation — some GPT-tuned prompts produce degraded results on Claude due to different RLHF training, and vice versa.
