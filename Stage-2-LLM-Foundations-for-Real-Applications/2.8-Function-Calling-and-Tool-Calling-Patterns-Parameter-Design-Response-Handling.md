A support routing team built an LLM pipeline to assign incoming tickets to the right team — billing, technical, account, or product. For simple tickets it worked well. ## CONCEPT BLOCK 1 — What Tool Calling Is and How the Two-Step Cycle Works

### A — The core idea in one sentence

The model does not execute tools — it outputs a structured description of a tool call, your code executes it, and the result is injected back into the conversation so the model can produce a final answer.

### B — Explain it plainly

When you call the Chat Completions API without tools, the model reads your messages and generates text. One API call, one response, done.

Tool calling adds a two-step cycle. In the first API call, you include a `tools` parameter: a list of function definitions telling the model what tools exist and what they do. The model reads the conversation and decides:

1. "I can answer this directly" → responds normally, `finish_reason="stop"`
2. "I need to call a tool to get information I don't have" → responds with `finish_reason="tool_calls"` and a structured description of what to call and with what arguments

When `finish_reason="tool_calls"`, the model has NOT called anything. It has produced a JSON object saying "I want to call `get_customer_context` with `{"customer_id": "C-1042"}` as the argument." Your code reads that, executes the actual function, and gets the real result. You then append both the model's tool call message and your tool result message to the conversation, and make a second API call. In the second call, the model sees the tool result and generates its final answer.

This is the complete cycle:

```
1st API call  →  finish_reason="tool_calls"
                 model says: "call get_customer_context({customer_id: 'C-1042'})"
                                    │
                    your code executes the function
                                    │
                              returns real data
                                    │
2nd API call  →  finish_reason="stop"
                 model says: "This customer has been active for 3 years and has 2 open tickets..."
```

The model never touches your database, your API, or your file system directly. It produces a structured intent. You execute the intent. You return the result. This design is intentional — it gives you complete control over what gets executed and with what permissions. The model cannot do anything you have not written a handler for.

### C — Concrete example with full working code

```python
import os
import json
from openai import AzureOpenAI

client = AzureOpenAI(
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    api_key=os.environ["AZURE_OPENAI_API_KEY"],
    api_version="2024-02-01",
)
DEPLOYMENT = "gpt-4o"

# --- Step 1: Define the tool ---
# The tool definition tells the model what the function does and what arguments it takes.
tools = [
    {
        "type": "function",
        "function": {
            "name": "get_customer_context",
            "description": (
                "Retrieve account context for a customer. "
                "Use this when routing a ticket requires knowing the customer's plan, "
                "account age, or number of open tickets."
            ),
            "parameters": {
                "type": "object",
                "properties": {
                    "customer_id": {
                        "type": "string",
                        "description": "The unique customer identifier, e.g. 'C-1042'.",
                    }
                },
                "required": ["customer_id"],
            },
        },
    }
]

# --- Step 2: First API call — model decides whether to use the tool ---
messages = [
    {
        "role": "system",
        "content": (
            "You are a ticket routing assistant. "
            "Route tickets to: billing, technical, account, or product. "
            "If you need customer account context to route correctly, use the tool."
        ),
    },
    {
        "role": "user",
        "content": "Ticket C-1042: 'My discount is not applying at checkout, third time this week.'",
    },
]

response = client.chat.completions.create(
    model=DEPLOYMENT,
    messages=messages,
    tools=tools,
    tool_choice="auto",    # model decides whether to call a tool
)

print(response.choices[0].finish_reason)
# → "tool_calls"   (model needs customer context before routing)

tool_calls = response.choices[0].message.tool_calls
print(tool_calls[0].function.name)        # → "get_customer_context"
print(tool_calls[0].function.arguments)  # → '{"customer_id": "C-1042"}'

# --- Step 3: Execute the tool in your code ---
def get_customer_context(customer_id: str) -> dict:
    """
    Real implementation: query your database.
    Here: stub returning example data.
    """
    # In production: SELECT * FROM customers WHERE id = %s
    return {
        "customer_id": customer_id,
        "plan": "growth",
        "account_age_days": 1247,
        "open_tickets": 2,
        "recent_purchases": ["Enterprise Add-on Pack", "Annual License"],
    }

tc = tool_calls[0]
args = json.loads(tc.function.arguments)       # arguments is a JSON string — always parse
tool_result = get_customer_context(**args)     # execute your function

# --- Step 4: Append assistant message + tool result, then second API call ---
messages.append(response.choices[0].message)  # assistant turn with tool_calls
messages.append({
    "role": "tool",
    "tool_call_id": tc.id,                     # must match the tool call id
    "content": json.dumps(tool_result),        # tool result as a string
})

final_response = client.chat.completions.create(
    model=DEPLOYMENT,
    messages=messages,
    tools=tools,
)

print(final_response.choices[0].finish_reason)  # → "stop"
print(final_response.choices[0].message.content)
# → "Route to billing. The customer is on the Growth plan with 1,247 days of history
#    and has 2 open tickets. A recurring discount failure pattern on an established paid
#    account is a billing team issue, not technical."
```

### D — The four `finish_reason` outcomes for tool-enabled calls

| `finish_reason` | What it means | What to do |
|---|---|---|
| `"stop"` | Model answered without needing a tool | Read `message.content` — final answer |
| `"tool_calls"` | Model wants to call one or more tools | Execute tools, inject results, call again |
| `"length"` | Response was cut off by `max_tokens` | Increase `max_tokens` or trim the conversation |
| `"content_filter"` | Azure content policy blocked the response | Log the input, do not retry blindly |

### E — Where this fits in the broader architecture

Tool calling is the boundary between language reasoning and code execution. The model handles: understanding the request, deciding whether a tool is needed, selecting the right tool, and constructing valid arguments. Your code handles: executing the action, enforcing permissions, accessing real data, and returning a result. Neither side does the other's job. This clean separation is why tool-calling agents are debuggable — when something goes wrong, you can inspect exactly what the model asked for and what your code returned.

### F — Why engineers get this wrong

Two mistakes. First: forgetting the second API call. Engineers execute the tool, print the result, and stop — the model never sees the result and never produces a final answer. The two-step cycle must close: execute → inject result → call again. Second: treating `finish_reason="tool_calls"` as a special case that bypasses normal conversation logic. It is not a special case — it is a normal turn in the conversation. The tool result is just another message in the list. The conversation context accumulates across both calls.

### G — Failure modes and their causes

| Failure | Root cause | Fix |
|---|---|---|
| Pipeline stops after tool execution | Second API call never made | Loop continues until `finish_reason == "stop"` |
| `tool_call_id` mismatch error | Tool message `tool_call_id` does not match `tc.id` | Copy `tc.id` exactly — do not generate a new ID |
| `json.loads()` fails on arguments | Model produced malformed JSON in arguments | Add `try/except json.JSONDecodeError`; log and escalate |
| Tool result not returned as string | Content field requires a string; dict passed directly | `json.dumps(result)` before setting `content` |
| Model ignores tool even when needed | Description too vague to trigger tool selection | Rewrite description with specific trigger conditions (Concept Block 2) |

### H — Reading the model's response as a signal

If the model consistently returns `finish_reason="stop"` on requests you expect it to use a tool for, the description is not specific enough about when to call it. A description like "fetches customer data" is too broad — the model does not know *when* it needs customer data. A description like "Use this when routing a ticket requires knowing the customer's plan, account age, or open ticket count" gives the model the trigger conditions it needs to decide correctly.

### I — Senior signal

> "The model does not call tools — it generates a structured intent that says 'call this function with these arguments.' Our code executes the intent. This distinction matters a lot operationally: it means we can add rate limiting, permission checks, audit logging, and input validation at the tool handler layer before anything touches the database. The model never has direct access to anything."

## CONCEPT BLOCK 2 — Tool Schema Design: Descriptions and Parameter Contracts

### A — The core idea in one sentence

The `description` field is what tells the model *when* to use a tool and *what it is for* — it is the most important part of the schema; the `parameters` block is the contract that tells the model *what arguments to supply* and in what format.

### B — Explain it plainly

A tool schema has two distinct jobs. The model uses them differently.

**The `function.description`** is read at decision time. Before the model produces any output, it reads the description of each available tool and decides: "Does this request require information or actions that any of these tools can provide?" A vague description leads to missed calls (model does not realize the tool applies) or phantom calls (model calls a tool when it should not). A precise description that includes trigger conditions, use cases, and examples gives the model enough information to make correct decisions.

**The `function.parameters`** is a JSON Schema object read at argument-generation time. Once the model has decided to call a tool, it uses the parameter schema to know what arguments to produce and what format they should be in. Parameter descriptions, enums for constrained choices, and required/optional designations directly control the quality of the arguments the model generates.

The description is the intelligence layer. The parameters block is the contract layer.

Three parameter design rules that matter in production:

**Use `enum` for constrained string choices.** If an argument can only be `"billing"`, `"technical"`, `"account"`, or `"product"`, declare that as an enum. Without an enum, the model may freely invent a string value — `"billing_escalation"`, `"technical_priority"` — that no handler will recognize.

**Write parameter descriptions as instructions, not labels.** `"description": "the customer ID"` is a label. `"description": "The unique customer identifier in the format C-NNNN, e.g. 'C-1042'. Found in the ticket metadata."` is an instruction. The model uses the instruction to produce the right value.

**Keep `required` honest.** If the model can always infer the argument from context, require it. If the argument is genuinely optional and the function works without it, make it optional. Do not mark optional arguments as required — the model will hallucinate a value rather than omit the field.

### C — Concrete example with full working code

```python
# --- Well-designed tool schema ---
# Every field has a description that tells the model what to put there.
# Enums constrain the routing_reason argument.
# Required vs optional reflects the actual function signature.

tools_well_designed = [
    {
        "type": "function",
        "function": {
            "name": "route_ticket",
            "description": (
                "Assign a support ticket to a team and record the routing reason. "
                "Use this after you have gathered enough context to make a routing decision. "
                "Do NOT use this if you still need customer context — call get_customer_context first."
            ),
            "parameters": {
                "type": "object",
                "properties": {
                    "ticket_id": {
                        "type": "string",
                        "description": (
                            "The ticket identifier from the conversation. "
                            "Format: T-NNNN or the raw ID string from the user message."
                        ),
                    },
                    "team": {
                        "type": "string",
                        "enum": ["billing", "technical", "account", "product"],
                        "description": (
                            "The team to route the ticket to. "
                            "billing: payment, invoice, subscription, discount issues. "
                            "technical: bugs, errors, outages, API failures. "
                            "account: login, access, permissions, user management. "
                            "product: feature requests, feedback, roadmap questions."
                        ),
                    },
                    "confidence": {
                        "type": "number",
                        "description": (
                            "Your routing confidence as a float between 0.0 and 1.0. "
                            "Use 1.0 for unambiguous cases, lower values when the ticket "
                            "could fit multiple teams."
                        ),
                    },
                    "routing_reason": {
                        "type": "string",
                        "description": (
                            "One sentence explaining why this team was chosen. "
                            "Reference specific ticket content, not generic reasoning."
                        ),
                    },
                    "escalate": {
                        "type": "boolean",
                        "description": (
                            "Set to true if this ticket shows signs of urgency: "
                            "repeat issue (third+ occurrence), data loss risk, or enterprise customer. "
                            "Defaults to false."
                        ),
                    },
                },
                "required": ["ticket_id", "team", "confidence", "routing_reason"],
                # escalate is optional — the model includes it only when relevant
            },
        },
    },
    {
        "type": "function",
        "function": {
            "name": "get_customer_context",
            "description": (
                "Retrieve account metadata for a customer. "
                "Use this when the ticket routing decision depends on the customer's plan, "
                "account age, or number of currently open tickets. "
                "Do NOT call this for every ticket — only when plan or history is relevant."
            ),
            "parameters": {
                "type": "object",
                "properties": {
                    "customer_id": {
                        "type": "string",
                        "description": "The customer identifier, e.g. 'C-1042'. Extract from the ticket.",
                    }
                },
                "required": ["customer_id"],
            },
        },
    },
]

# --- Poorly-designed version for contrast ---
tools_poorly_designed = [
    {
        "type": "function",
        "function": {
            "name": "route_ticket",
            "description": "Routes a ticket.",         # ← no trigger conditions, no examples
            "parameters": {
                "type": "object",
                "properties": {
                    "ticket_id": {"type": "string"},   # ← no description — model guesses format
                    "team": {"type": "string"},         # ← no enum — model invents "technical_priority"
                    "reason": {"type": "string"},       # ← field name differs from handler ("routing_reason")
                },
                "required": ["ticket_id", "team", "reason", "confidence"],  # ← confidence required but not defined
            },
        },
    },
]
```

The poorly-designed version will produce routing failures because: the model does not know when to call the tool, it can invent team names that no handler recognizes, the field name mismatch (`reason` vs `routing_reason`) means handler code crashes on `KeyError`, and requiring `confidence` while not defining it in `properties` causes the model to hallucinate its format.

### D — Tool schema quality checklist

| Schema element | Weak | Strong |
|---|---|---|
| `function.description` | "Gets customer data" | "Use when routing requires knowing plan, age, or open tickets. Do NOT call for every ticket." |
| `parameter.description` | `"the team"` | `"billing: payment/invoice/discount. technical: bugs/errors. account: login/access. product: features."` |
| Constrained string field | `"type": "string"` | `"type": "string", "enum": ["billing", "technical", "account", "product"]` |
| Optional field | Listed in `required` | Absent from `required`, present in `properties` with a default hint in description |
| Numeric field | `"type": "number"` | `"type": "number", "description": "Float 0.0–1.0. Use 1.0 for unambiguous cases."` |

### E — Where this fits in the broader architecture

Tool descriptions are prompt engineering, not configuration. They should be version-controlled, tested against a golden set of tool-selection decisions (does the model call the right tool for these inputs?), and updated when the model's tool selection behavior drifts. A tool selection regression — where a model update causes the model to stop calling `get_customer_context` on context-dependent tickets — looks like a routing accuracy drop, not a parsing error. The description is the signal that drives that decision.

### F — Why engineers get this wrong

Two mistakes. First: treating the description as documentation for developers, not instructions for the model. Developers know what "route ticket" means. The model needs to know *when* routing is appropriate and *what signals* should trigger the call. Write for the model, not for humans reading the schema. Second: not using enums for constrained choices. A string field without an enum gives the model complete freedom to invent values. The model will occasionally invent plausible-sounding values — `"billing_urgent"`, `"technical_bug"` — that your dispatch handler will not recognize. Every constrained choice should be an enum.

### G — Failure modes and their causes

| Failure | Root cause | Fix |
|---|---|---|
| Model never calls the tool | Description does not state when to use it | Add explicit trigger conditions: "Use when X is needed" |
| Model calls tool on every request | Description too permissive ("always use this") | Add "Do NOT use this when Y" to the description |
| Tool argument uses wrong string value | No enum, model invents a value | Add `"enum": [...]` to constrained string parameters |
| Handler crashes on `KeyError` | Field name in schema differs from handler expectation | Schema field names must exactly match handler argument names |
| Optional field always null | Field not described; model does not know when to set it | Add description explaining when to include it and what value to use |

### H — Reading the model's response as a signal

When you inspect `tc.function.arguments` and find the model is consistently using the wrong team name or hallucinating the format of an ID, the issue is almost always in the parameter description, not in the model. The model follows the description it was given. If `customer_id` has no description and the ticket contains both `C-1042` and a user email, the model will sometimes use the email, sometimes the ID. Add the description: "The customer identifier in the format C-NNNN. Do not use the email address." The ambiguity is in the schema, not in the model.

### I — Senior signal

> "The description field is where most tool calling bugs come from, not the code. Before adding any error handling, I read the tool description out loud and ask: if I were the model, would I know when to call this tool and what arguments to provide? If the answer is no, no amount of retry logic will fix it. Fix the description first."

## CONCEPT BLOCK 3 — Response Handling: Parsing, Dispatching, and Completing the Cycle

### A — The core idea in one sentence

A production tool call handler reads `finish_reason`, safely parses the arguments JSON string, dispatches to a named handler registry, injects the result into the conversation, and loops until `finish_reason="stop"` — with explicit error handling for bad arguments, unknown tools, and handler failures.

### B — Explain it plainly

The happy path for tool calling is short: read `tool_calls`, execute each one, append results, call again. Production tool call handling has four additional concerns that the happy path does not cover.

**Parallel tool calls.** The model can request multiple tools in a single turn — `message.tool_calls` is a list. A handler that only processes `tool_calls[0]` will silently drop all other calls. The handler must process all of them before making the second API call.

**Argument validation.** `tc.function.arguments` is a JSON string. `json.loads()` can fail — the model may produce malformed arguments under edge cases. Even if parsing succeeds, the arguments may not match your function signature. Validate arguments before passing them to the handler.

**Unknown tools.** The model may call a tool name that does not exist in your registry — this can happen after a tool is renamed, removed, or if the model hallucinates a tool name. Your dispatch logic must handle unknown tool names without crashing.

**Handler failures.** The handler function itself can raise an exception — database unavailable, permission denied, downstream API timeout. A handler failure should produce a structured error result, not crash the pipeline. The model receives the error as a tool result and can adapt its response.

**The loop pattern.** In simple single-tool scenarios, two API calls always complete the cycle. In multi-tool scenarios (the model calls tool A, reads the result, and then decides it needs tool B), the cycle requires more iterations. A robust handler wraps the entire cycle in a loop that continues until `finish_reason="stop"` with a hard iteration cap to prevent infinite loops.

### C — Concrete example with full working code

```python
import os
import json
import logging
from typing import Any, Callable

logger = logging.getLogger(__name__)

# --- Tool handler registry ---
# Maps tool name → handler function.
# Each handler takes **kwargs matching the tool's parameter schema.

def get_customer_context(customer_id: str) -> dict:
    """Real implementation: query your database by customer_id."""
    # Stub for demonstration
    return {
        "customer_id": customer_id,
        "plan": "growth",
        "account_age_days": 1247,
        "open_tickets": 2,
    }

def route_ticket(
    ticket_id: str,
    team: str,
    confidence: float,
    routing_reason: str,
    escalate: bool = False,
) -> dict:
    """Real implementation: write to your routing database."""
    logger.info(
        "Routing ticket=%s to team=%s confidence=%.2f escalate=%s reason=%r",
        ticket_id, team, confidence, escalate, routing_reason,
    )
    return {"status": "routed", "ticket_id": ticket_id, "team": team}

TOOL_REGISTRY: dict[str, Callable[..., Any]] = {
    "get_customer_context": get_customer_context,
    "route_ticket": route_ticket,
}


def execute_tool_call(tc) -> str:
    """
    Parse and execute a single tool call.
    Returns the result as a JSON string (tool message content must be a string).
    Never raises — returns a structured error string on failure.
    """
    tool_name = tc.function.name

    # 1. Parse arguments — model produces a JSON string
    try:
        args: dict = json.loads(tc.function.arguments)
    except json.JSONDecodeError as e:
        logger.error(
            "Tool call %s: failed to parse arguments JSON: %s | raw: %r",
            tool_name, e, tc.function.arguments,
        )
        return json.dumps({
            "error": "argument_parse_failed",
            "detail": str(e),
            "raw_arguments": tc.function.arguments,
        })

    # 2. Dispatch — look up the handler
    handler = TOOL_REGISTRY.get(tool_name)
    if handler is None:
        logger.error("Tool call %s: no handler registered.", tool_name)
        return json.dumps({
            "error": "unknown_tool",
            "tool_name": tool_name,
            "available_tools": list(TOOL_REGISTRY.keys()),
        })

    # 3. Execute — catch handler failures
    try:
        result = handler(**args)
        return json.dumps(result)
    except TypeError as e:
        # Arguments don't match the function signature — schema/handler mismatch
        logger.error(
            "Tool call %s: argument mismatch: %s | args: %r", tool_name, e, args,
        )
        return json.dumps({
            "error": "argument_mismatch",
            "detail": str(e),
            "args_received": args,
        })
    except Exception as e:
        logger.error(
            "Tool call %s: handler raised %s: %s", tool_name, type(e).__name__, e,
            exc_info=True,
        )
        return json.dumps({
            "error": "handler_failed",
            "error_type": type(e).__name__,
            "detail": str(e),
        })


def run_tool_loop(
    messages: list[dict],
    tools: list[dict],
    max_iterations: int = 5,
) -> str:
    """
    Execute the full tool calling loop until finish_reason="stop".

    - Handles parallel tool calls (processes all tool_calls in one turn).
    - Returns the model's final text response.
    - Raises RuntimeError if max_iterations is exceeded (loop guard).
    """
    for iteration in range(max_iterations):
        response = client.chat.completions.create(
            model=DEPLOYMENT,
            messages=messages,
            tools=tools,
            tool_choice="auto",
        )
        choice = response.choices[0]

        if choice.finish_reason == "stop":
            # Model produced a final answer — return it
            return choice.message.content

        if choice.finish_reason == "tool_calls":
            tool_calls = choice.message.tool_calls  # list — handle all of them

            # Append assistant message (with tool_calls) to conversation
            messages.append(choice.message)

            # Execute all tool calls, append each result
            for tc in tool_calls:
                result_str = execute_tool_call(tc)
                logger.info(
                    "Iteration %d: executed tool=%s id=%s",
                    iteration + 1, tc.function.name, tc.id,
                )
                messages.append({
                    "role": "tool",
                    "tool_call_id": tc.id,         # must match the tool call id exactly
                    "content": result_str,
                })

            # Loop continues — second (or subsequent) API call made at top of loop
            continue

        # Unexpected finish_reason (length, content_filter)
        logger.warning(
            "Unexpected finish_reason=%s at iteration %d",
            choice.finish_reason, iteration + 1,
        )
        raise RuntimeError(
            f"Tool loop terminated with unexpected finish_reason={choice.finish_reason!r}"
        )

    raise RuntimeError(
        f"Tool loop exceeded max_iterations={max_iterations}. "
        "Check for a tool that always returns data that triggers another tool call."
    )


# --- Usage ---

def route_support_ticket(ticket_text: str) -> str:
    """
    Route a support ticket using tool-augmented reasoning.
    The model may call get_customer_context and then route_ticket.
    """
    messages = [
        {
            "role": "system",
            "content": (
                "You are a ticket routing assistant. "
                "If you need customer context to route correctly, use get_customer_context. "
                "Once you have enough information, call route_ticket to assign the ticket. "
                "Always call route_ticket before responding — do not just say which team verbally."
            ),
        },
        {
            "role": "user",
            "content": ticket_text,
        },
    ]

    final_answer = run_tool_loop(
        messages=messages,
        tools=tools_well_designed,
        max_iterations=5,
    )
    return final_answer


# Test
result = route_support_ticket(
    "Ticket T-8821, Customer C-1042: My discount is not applying, this is the third time."
)
print(result)
# → Model calls get_customer_context(customer_id="C-1042")
#   Receives: {plan: "growth", account_age_days: 1247, open_tickets: 2}
#   Model calls route_ticket(ticket_id="T-8821", team="billing",
#                             confidence=0.95, routing_reason="...", escalate=True)
#   Receives: {status: "routed", ticket_id: "T-8821", team: "billing"}
#   Model responds: "Ticket T-8821 has been routed to billing with escalation..."
```

### D — `tool_choice` values and when to use each

| `tool_choice` value | What it does | When to use |
|---|---|---|
| `"auto"` (default) | Model decides whether to call a tool | Most cases — let the model decide based on need |
| `"required"` | Model must call at least one tool | When you need the model to always call a tool, e.g. structured data extraction via tools |
| `"none"` | Model cannot call any tools | When you want a plain text response even though tools are defined |
| `{"type": "function", "function": {"name": "X"}}` | Model must call tool X | When a specific tool call is always required for this turn |

### E — Where this fits in the broader architecture

`run_tool_loop()` is the primitive that Stage 4 (LangGraph) extends. A LangGraph node is effectively `run_tool_loop()` with state management, conditional routing between nodes, and persistence. Understanding this loop at the raw API level makes every LangGraph abstraction transparent — when a LangGraph agent breaks, you know exactly what API call to inspect.

The tool handler registry (`TOOL_REGISTRY`) is also your permission enforcement layer. You add authentication checks, rate limits, and audit logging in `execute_tool_call()` before any handler runs. The model cannot bypass this layer.

### F — Why engineers get this wrong

Three mistakes. First: processing only `tool_calls[0]` and ignoring parallel tool calls. The model can request multiple tools in one turn. If you only handle the first one, you append only one tool result, and the second API call fails because the conversation has an unresolved tool call ID with no result. Second: not appending the assistant message before the tool messages. The conversation must contain `assistant (tool_calls) → tool (result) → tool (result)` in that order. Omitting the assistant message causes an API error. Third: no iteration cap on the tool loop. An agent that calls tool A, which returns data that always triggers tool B, which returns data that always triggers tool A, will loop forever and consume your entire token budget.

### G — Failure modes and their causes

| Failure | Root cause | Fix |
|---|---|---|
| API error: "tool message without preceding assistant message" | Tool message appended without first appending the assistant `choice.message` | Always `messages.append(choice.message)` before appending tool results |
| API error: "tool_call_id not found" | `tool_call_id` in the tool message does not match `tc.id` | Copy `tc.id` exactly; never generate a new ID |
| Silent dropped tool call | Code only processes `tool_calls[0]` | Loop over `choice.message.tool_calls` — process all of them |
| Infinite tool loop | No `max_iterations` cap | Hard cap at 5–10 iterations; raise `RuntimeError` if exceeded |
| Handler crash propagates up | No `try/except` in `execute_tool_call` | Return structured error JSON from handler; model adapts its response |
| Model invents tool arguments | Schema parameter has no description | Add description with format example for every parameter |

### H — Reading the model's response as a signal

When the model receives a tool result containing `{"error": "handler_failed", ...}` and still produces a reasonable response — "I was unable to retrieve the customer context, routing based on ticket content only" — this is working as intended. The model can adapt to error results. Design your error return values to be informative but not alarming to the model. An error result that says `{"error": "database connection timeout after 3s"}` gives the model enough information to tell the user something useful. An unhandled exception that crashes the pipeline tells the model nothing.

### I — Senior signal

> "The tool loop has exactly one invariant: every tool call ID that appears in the assistant message must have a corresponding tool message before the next API call. Everything else — error handling, parallel execution, the iteration cap — is about making the loop robust. When debugging a tool calling pipeline, I first check that invariant. Nine times out of ten, the issue is a missing tool message or a mismatched ID, not anything subtle."

## DECISION RULES

```
IF your prompt requires information that is not in the model's training data
→ define a tool that retrieves that information
→ write the description with specific trigger conditions ("use when X is needed")
→ do NOT use tool calling just to add structure — Lesson 2.7 (JSON mode) is cheaper for that

IF a parameter can only take a fixed set of values
→ always add "enum": [...] to that parameter's schema
→ never leave constrained choices as a free string — the model will hallucinate values

IF you need the model to always call a tool (e.g. always write to a database before responding)
→ set tool_choice to {"type": "function", "function": {"name": "tool_name"}}
→ or use tool_choice="required" if any tool is acceptable

IF the model should decide whether a tool is needed
→ use tool_choice="auto" (the default)
→ write the description with both when-to-use AND when-not-to-use conditions

IF multiple tools are available
→ loop over all entries in choice.message.tool_calls — not just tool_calls[0]
→ append all tool results before making the second API call

BEFORE making the second API call
→ verify: assistant message with tool_calls is appended
→ verify: one tool message per tool call, each with matching tool_call_id
→ these two conditions must both be true — API will reject the call otherwise

IF a tool call happens more than 3–5 times in a loop
→ cap iterations (max_iterations=5) and raise RuntimeError
→ infinite tool loops will exhaust your token budget silently without a cap

IF a handler raises an exception
→ catch it in execute_tool_call, return structured error JSON as the tool content
→ do not let handler exceptions propagate to the top-level caller
→ the model can read an error result and adapt — it cannot adapt to a Python crash

IF tc.function.arguments fails json.loads()
→ log the raw string and return a structured error as the tool content
→ do not crash — return {"error": "argument_parse_failed", "raw": ...}
→ this is rare with gpt-4o but possible under extreme prompt conditions

IF you want to know why the model is choosing (or not choosing) a tool
→ inspect the description — it is the primary driver of tool selection
→ add explicit "Use this when..." and "Do NOT use this when..." clauses
→ test tool selection with a golden set of inputs before shipping
```

## COMMON MISTAKES

### Mistake 1: Processing only the first tool call

**What it looks like:**
```python
if choice.finish_reason == "tool_calls":
    tc = choice.message.tool_calls[0]    # only first tool call
    result = execute_tool(tc)
    messages.append({"role": "tool", "tool_call_id": tc.id, "content": result})
    # second API call — but tool_calls[1] has no matching tool message
```

**Why it happens:** Most tutorial examples show a single tool, so engineers hardcode index `[0]`.

**What goes wrong:** When the model issues two tool calls in parallel (e.g. `get_customer_context` and `check_ticket_history` simultaneously), only the first gets a tool message. The second API call includes an assistant message with two tool call IDs but only one tool result. The API returns an error: one tool call ID has no corresponding result.

**The fix:** Always loop over all entries in `choice.message.tool_calls`:
```python
for tc in choice.message.tool_calls:
    result = execute_tool_call(tc)
    messages.append({"role": "tool", "tool_call_id": tc.id, "content": result})
```

---

### Mistake 2: Forgetting to append the assistant message before tool results

**What it looks like:**
```python
if choice.finish_reason == "tool_calls":
    for tc in choice.message.tool_calls:
        result = execute_tool_call(tc)
        messages.append({"role": "tool", "tool_call_id": tc.id, "content": result})
    # assistant message never appended
```

**Why it happens:** Engineers think about the data flow — "I need to give the model the tool result" — and skip the step of appending the assistant turn that generated the tool call.

**What goes wrong:** The API requires that a `tool` message is preceded by an `assistant` message that contains the matching `tool_calls`. Omitting the assistant message produces an API error: "messages with role 'tool' must be a response to a preceding message with role 'assistant'."

**The fix:** Always append `choice.message` (the assistant message with tool_calls) before appending any tool results:
```python
messages.append(choice.message)       # ← assistant message first
for tc in choice.message.tool_calls:
    messages.append({"role": "tool", "tool_call_id": tc.id, "content": result})
```

---

### Mistake 3: No iteration cap on the tool loop

**What it looks like:**
```python
while True:
    response = client.chat.completions.create(...)
    if response.choices[0].finish_reason == "stop":
        return response.choices[0].message.content
    # execute tools, append results, continue
```

**Why it happens:** In the happy path, two iterations always terminate the loop. The `while True` feels safe because it always exits in testing.

**What goes wrong:** Two tools can create a cycle — tool A returns data that triggers tool B, tool B returns data that triggers tool A. In production, under certain inputs, this cycle can run for hundreds of iterations before your token budget is exhausted or the API rate limit fires. The bill for a single request becomes catastrophic. With a gpt-4o deployment at $5/M input tokens, a 100-iteration loop on a 2K-token conversation costs $1.00 per request.

**The fix:** Hard cap at 5–10 iterations. Raise an error if exceeded:
```python
for iteration in range(max_iterations):   # max_iterations=5
    ...
raise RuntimeError(f"Tool loop exceeded {max_iterations} iterations.")
```

---

### Mistake 4: Trusting arguments without validation before execution

**What it looks like:**
```python
args = json.loads(tc.function.arguments)
result = handler(**args)     # no validation — args passed directly
```

**Why it happens:** The model was told to use a specific schema, so engineers assume the arguments will always match.

**What goes wrong:** The model may produce correct JSON but with wrong types, extra fields, or missing optional fields that the handler does not default gracefully. A `TypeError` from `handler(**args)` crashes the tool loop. More dangerously, if the tool writes to a database, wrong argument values may be written without error — `customer_id="C-1042-invalid"` may not raise, but it may silently fail to match any database record.

**The fix:** At minimum, wrap handler execution in `try/except TypeError` to catch argument mismatches. For critical tools (writes, deletions, external API calls), validate arguments against a Pydantic model before calling the handler:
```python
from pydantic import BaseModel

class RouteTicketArgs(BaseModel):
    ticket_id: str
    team: str
    confidence: float
    routing_reason: str
    escalate: bool = False

args = RouteTicketArgs.model_validate(json.loads(tc.function.arguments))
result = route_ticket(**args.model_dump())
```

**Q1:** You call the API with `tools=` defined and `tool_choice="auto"`. The model returns `finish_reason="tool_calls"`. What are the exact steps you must take before making the second API call?

*Check:* (1) Append `choice.message` (the assistant message with tool_calls) to the messages list. (2) For every entry in `choice.message.tool_calls`, execute the tool and append a `{"role": "tool", "tool_call_id": tc.id, "content": result_string}` message. Both steps are required. Omitting the assistant message causes an API error. Missing a tool result for any tool call ID causes an API error.

---

**Q2:** Your tool schema has `"team": {"type": "string"}` without an enum. What is the specific failure this causes, and what is the fix?

*Check:* Without an enum, the model can produce any string value — including invented values like `"billing_urgent"` or `"technical_priority"` that your handler does not recognize. The fix is `"enum": ["billing", "technical", "account", "product"]`. With an enum, the model's token sampling is constrained to only those values, and hallucinated team names cannot appear.

---

**Q3:** A tool loop runs correctly in all your tests but occasionally gets stuck in production, consuming 40+ iterations before hitting a rate limit. What is the root cause and what prevents it?

*Check:* Two tools are creating a cycle — tool A returns data that triggers the model to call tool B, and tool B returns data that triggers tool A again. In testing, the input data was always simple enough to terminate in two iterations. In production, certain inputs activate the cycle. Prevention: hard cap at `max_iterations=5` (or similar), raise `RuntimeError` if exceeded, and inspect which two tools are in the cycle to break it by adding a "Do NOT use this if you already have X" clause to the description.

---

**Q4:** `execute_tool_call()` catches all exceptions and returns a structured error JSON string. The model receives `{"error": "handler_failed", "detail": "database connection timeout"}` as the tool result. What does the model do with this, and why is this better than letting the exception propagate?

*Check:* The model reads the error result and adapts — it may say "I was unable to retrieve the customer context due to a system error. Routing based on ticket content only." The pipeline continues and the user gets a degraded but useful response. If the exception propagates, the entire request handler crashes, the user gets a 500 error, and no information is logged about which tool failed and with what arguments. The structured error approach gives the model agency to recover while keeping the pipeline running and the failure observable.

---

**Q5:** A team is building a tool that searches an internal knowledge base. They write the description as: "Searches the knowledge base." After deploying, the model almost never calls the tool, even on queries that clearly require knowledge base content. What is the most likely cause and how do you fix it?

*Check:* The description "Searches the knowledge base" gives the model no trigger conditions — it does not know when to call the tool vs answering from its own knowledge. The model defaults to answering from its own training data because the description does not specify when external search is needed. Fix: rewrite the description with explicit triggers: "Use this when the question requires information about internal policies, product documentation, or procedures not likely to be in general knowledge. Do NOT use this for general knowledge questions the model can answer confidently from training data."

## SUMMARY

Function calling gives the model the ability to request real data and real actions from your code, rather than inferring or hallucinating from training knowledge alone.

**The two-step cycle** is the core primitive: first call returns `finish_reason="tool_calls"` with a structured description of what to execute; your code executes it; second call receives the result and produces the final answer. The model never touches your database or APIs directly — it generates an intent, and your code decides whether and how to execute it.

**Tool schema design** determines whether the model calls the right tool at the right time. The `description` field must state explicit trigger conditions — "Use this when X is needed; do NOT use this when Y" — because the model's tool selection decision is driven almost entirely by the description. Parameter `enum` fields for constrained choices and per-parameter descriptions that give format examples prevent the model from inventing argument values.

**Response handling** in production requires: looping over all entries in `tool_calls` (not just the first), appending the assistant message before tool results, catching all handler exceptions and returning structured error JSON, and capping the loop at 5–10 iterations with a hard error on overflow.

**The monitoring frame:** Track which tools are called, how often, with what arguments, and whether handler failures are rising. A tool that is never called despite relevant inputs has a description problem. A tool that is always called even when unnecessary has a description that is too permissive. Tool selection quality is a prompt engineering metric, not just an infrastructure metric.

## PRACTICAL LANDSCAPE

### Tool calling vs other structured output methods

| Approach | What it solves | When to use |
|---|---|---|
| **Tool calling** | Model needs external data or must trigger an action | Routing decisions, database lookups, action execution |
| **JSON mode (Lesson 2.7)** | Model output must be parseable JSON | Data extraction from unstructured text — no external data needed |
| **Structured Outputs (Lesson 2.7)** | Model output must conform to a specific schema | Same as JSON mode but with API-level key enforcement |
| **System prompt instructions** | Model behavior must follow stable rules | Formatting, tone, domain constraints — not dynamic data |

Use tool calling when the decision or output depends on data that is not in the model's context. Use structured outputs when the task is pure text transformation with known input and output schema.

### Tool calling API surface (openai SDK v1.x)

```python
# Define tools
tools = [{"type": "function", "function": {"name": ..., "description": ..., "parameters": ...}}]

# Call with tools
response = client.chat.completions.create(
    model=DEPLOYMENT,
    messages=messages,
    tools=tools,
    tool_choice="auto",              # "auto" | "required" | "none" | {"type": "function", ...}
    parallel_tool_calls=True,        # default True — model may return multiple tool calls
)

# Read result
finish_reason = response.choices[0].finish_reason   # "stop" | "tool_calls" | "length" | ...
tool_calls = response.choices[0].message.tool_calls  # list[ChatCompletionMessageToolCall] | None
tc = tool_calls[0]
tc.id                   # unique ID for this tool call — must appear in the tool result message
tc.function.name        # "get_customer_context"
tc.function.arguments   # '{"customer_id": "C-1042"}' — JSON string, parse with json.loads()

# Inject result
messages.append(response.choices[0].message)         # assistant message with tool_calls
messages.append({
    "role": "tool",
    "tool_call_id": tc.id,                            # must match tc.id exactly
    "content": json.dumps(result),                    # string — JSON-encode dicts
})
```

| Approach | Advantages | Disadvantages | Use when |

### Interview questions this lesson prepares you for

**Q: How does function calling work at the API level — walk me through the full cycle.**

> "When you pass a `tools` list to the Chat Completions API, the model can respond in two ways: with `finish_reason='stop'` if it can answer directly, or with `finish_reason='tool_calls'` if it needs to call a tool. In the tool_calls case, the model returns a structured JSON object — not the actual function result — describing which function to call and with what arguments. Your code reads `tc.function.name` and `json.loads(tc.function.arguments)`, executes the real function, and appends both the assistant message and a tool result message back to the conversation. Then you call the API again. The model reads the tool result and produces the final response. The model never touches your database directly — it generates an intent; your code decides whether and how to execute it."

**Q: Your tool is defined correctly but the model never calls it, even on queries that clearly need it. What do you check?**

> "First thing I check is the description. The model's tool selection decision is driven almost entirely by the description field — if it doesn't contain specific trigger conditions, the model defaults to answering from its own training data. I rewrite the description to say 'Use this when X is needed' with concrete examples of X, and add 'Do NOT use this when Y' to prevent over-triggering. If the description looks correct, I check whether tool_choice is set to 'none' accidentally, or whether there's a system prompt instruction conflicting with tool use. I'd also test with tool_choice='required' to confirm the tool itself works — if it works when forced, the issue is definitely in the description."

**Q: How do you prevent a tool-calling agent from running indefinitely and consuming your entire token budget?**

> "Hard iteration cap on the tool loop — we set max_iterations to 5 and raise a RuntimeError if it's exceeded. In our experience, legitimate multi-tool calls complete in 2–3 iterations. If a loop exceeds 5, it means two tools are in a cycle — tool A's output is triggering tool B, and tool B's output is triggering tool A. We alert on this, inspect which tools are cycling, and break the cycle by adding explicit 'Do NOT use this if you already have X' clauses to one of the descriptions. The cost of a 100-iteration loop on a gpt-4o deployment is significant — this cap has saved us multiple times."

### Key vocabulary

| Term | Definition |
|---|---|
| **Tool calling / function calling** | API feature where the model produces a structured description of a function call; your code executes it |
| **`finish_reason="tool_calls"`** | Indicates the model wants to call one or more tools — has not executed anything |
| **`tool_calls`** | List of tool call objects on the assistant message — each has `id`, `function.name`, `function.arguments` |
| **`tc.function.arguments`** | JSON string of the arguments the model wants to call the function with — always parse with `json.loads()` |
| **Tool message** | Conversation message with `role="tool"`, `tool_call_id`, and `content` (the execution result) |
| **`tool_choice`** | Controls whether and which tools the model must call: `"auto"`, `"required"`, `"none"`, or a specific tool |
| **Parallel tool calls** | Model requests multiple tool calls in a single turn — `message.tool_calls` has more than one entry |
| **Tool handler registry** | Dict mapping tool name → handler function — your dispatch layer |
| **`enum` in parameters** | JSON Schema constraint that restricts a string parameter to a fixed set of allowed values |
| **Iteration cap** | Hard limit on tool loop iterations — prevents infinite cycles from exhausting the token budget |
