## 1. Why This Matters in Industry

A team ships an LLM-powered support assistant. It works in demo mode. In production, the model sometimes gives the right answer, sometimes changes tone, sometimes invents details, and sometimes breaks the expected output format. The team says, "the model is inconsistent." The real problem is simpler: they never built the right mental model of what the model is actually doing.

By the end of this lesson, you will be able to:
- explain what an LLM is in simple and correct terms
- distinguish pretrained knowledge from real-time information
- decide when an LLM alone is enough and when extra system design is required
- design the first safe service boundary for the stage project
- talk about LLM behavior in a professional, industry-credible way

## 2. Core Idea

### The one-line mental model

An LLM is a probabilistic token generator that predicts what text should come next based on the text it has already seen.

### What it is

- A model that generates output one token at a time
- A system trained on large amounts of text to continue patterns in language
- A useful reasoning-and-generation component for drafting, classification, extraction, summarization, and tool selection

### What it is not

- Not a database
- Not a search engine
- Not a guaranteed fact source
- Not a real-time system unless you connect it to real-time data

### Why it matters in production

If you misunderstand the model, you will build the wrong system. In real work, this causes:
- hallucinated facts
- brittle prompts
- wrong trust assumptions
- poor output reliability
- broken downstream automation

## 3. Professional Decision Rules

### Use an LLM directly when

- the task is drafting, rewriting, summarizing, classifying, or extracting from provided text
- perfect factual precision is not required
- the output can be validated before it reaches users or downstream systems

### Do not rely on the LLM alone when

- the answer must be factually exact
- the information is time-sensitive
- the output will trigger business logic, database writes, payments, or legal/medical actions
- the model needs private company data it was never trained on

### Industry rule of thumb

- If the model must know current or private facts, add retrieval or tools
- If the output must be machine-readable, add structure and validation
- If the answer must be correct, add verification
- If the result drives action, never trust raw model output without controls

## 4. Real-World Example

### Problem

You are building a support-ticket assistant for a SaaS company. The product manager wants one endpoint that can:
- classify the ticket
- summarize it
- mention the customer plan
- tell the agent whether this customer already has open issues

### Weak implementation

The team sends this prompt:

```text
Classify this support ticket, summarize it, and mention the customer's plan and open issues.
```

They assume the model already knows:
- the company customer database
- the customer's account status
- current ticket history

### Why it fails in industry

- Classification and summarization may work
- Customer plan and open issue count may be hallucinated
- Output shape may vary
- Different runs may produce different wording and confidence
- Nobody can defend the system in code review because the trust boundary is undefined

### Better implementation

Split the task by what the model can do from language alone and what requires external data.

```python
from dataclasses import dataclass


@dataclass
class TicketTaskDecision:
    can_use_llm_directly: bool
    needs_external_data: bool
    reason: str


def decide_llm_architecture(task: str) -> TicketTaskDecision:
    task = task.lower()

    if "summarize" in task or "classify" in task:
        return TicketTaskDecision(
            can_use_llm_directly=True,
            needs_external_data=False,
            reason="The model can work from the provided ticket text."
        )

    if "customer plan" in task or "open issues" in task or "account history" in task:
        return TicketTaskDecision(
            can_use_llm_directly=False,
            needs_external_data=True,
            reason="The model needs real account data that is not inside its weights."
        )

    return TicketTaskDecision(
        can_use_llm_directly=False,
        needs_external_data=True,
        reason="Assume external grounding is required until proven otherwise."
    )


tasks = [
    "classify support ticket",
    "summarize ticket",
    "mention customer plan",
    "mention open issues",
]

for task in tasks:
    print(task, "->", decide_llm_architecture(task))
```

### What makes this professional

- It separates language tasks from data tasks
- It defines trust boundaries before coding
- It prevents fake certainty about customer/account facts
- It gives you a defensible architecture decision in design review

## 5. Build It in the Real Project

### Feature added in this lesson

Create the first project artifact: an architecture contract that defines what the LLM is allowed to do directly and what must come from external systems.

### What you will build

In the stage project, create:
- a short `docs/llm-service-boundary.md`
- a first `POST /classify` scope note
- a task table separating direct LLM tasks from grounded tasks

### Suggested file contents

```md
# LLM Service Boundary

## LLM can do directly
- classify ticket text
- summarize ticket text
- rewrite agent response drafts

## LLM must not do alone
- state customer plan
- state billing balance
- state open tickets count
- state account permissions

## Requires external data
- CRM lookup
- billing system lookup
- ticket history lookup

## Rule
If the answer depends on current or private company data, the LLM must use retrieval or tools.
```

### Definition of done

- the project has a written LLM boundary document
- the team can point to what the model may and may not do alone
- the first endpoint scope is limited to something the LLM can do safely
- no private or real-time facts are assumed to come from the model itself

## 6. Professional Engineering Layer

### What to validate

- Is the task based only on provided text, or on missing external data?
- Is correctness required, or is best-effort generation acceptable?
- Is the output going to a human reader, or directly into software?

### What to log

Even in early project stages, log:
- task type
- model used
- prompt version
- whether external data was required
- whether output was used for display or automation

### What to test

For early-stage LLM services:
- same ticket returns a valid category every time
- unsupported tasks are rejected or routed to a grounded path
- no endpoint claims access to data it does not have

### What to monitor in production

- hallucination rate on factual claims
- format compliance
- classification consistency
- escalation frequency for unsupported requests

## 7. Common Production Failures

1. Treating the LLM like a database  
Why it happens: fluent outputs feel like retrieval.  
How to fix it: require retrieval, tools, or verification for specific facts.

2. Letting one endpoint do too much  
Why it happens: teams want a "smart assistant" fast.  
How to fix it: narrow the first endpoint to one safe job, such as classification or summarization.

3. Trusting confident wording as evidence  
Why it happens: the response sounds certain and polished.  
How to fix it: judge truth from system design and evidence, not from model tone.
