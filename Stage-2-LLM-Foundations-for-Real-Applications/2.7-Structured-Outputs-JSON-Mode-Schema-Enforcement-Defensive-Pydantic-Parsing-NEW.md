Implement `app/schemas/` — typed Pydantic output schemas for the copilot's four task types, field validators for date and numeric fields, and a `parse_llm_output()` wrapper with one retry on `ValidationError` — so that all LLM output is validated at the application boundary before reaching any downstream code.

## §2 Where This Fits in the System

**State:** After 2.6, the extraction endpoint calls `json.loads()` on the raw model output and accesses fields directly: `result["category"]`, `result["urgency"]`. No schema validation exists. The system prompt specifies the correct output format — but nothing enforces that the model actually returned it.

**Gap:** `json.loads()` only guarantees syntactic validity. The model can return valid JSON with wrong key names (`"date"` instead of `"invoice_date"`), wrong types (`"4750"` string instead of `4750.0` float), or missing required fields — all silently, all reaching the database with no error. These failures are rare enough on clean inputs to pass development testing. They are frequent enough at production volume to become incidents.

**Unlocks:** Every service that calls the LLM returns a validated, typed Pydantic object — not a raw dict. Downstream code works with typed fields. A `ValidationError` surfaces at the LLM boundary with the exact field name and failure reason. 2.8 reuses the same schema pattern for tool parameter validation.

## §3 Core Mental Model + Decision Rule

### The Three-Layer Defense

When LLM output must match a schema, three layers can catch failures — and they catch different things.

**Layer 1 — Prompt specification** (→ 2.6): The output format section lists exact key names, types, and null-handling rules. This reduces the violation rate. It does not prevent violations — the model follows the spec most of the time, not all of the time.

**Layer 2 — API enforcement** (JSON mode): The API constrains token sampling to produce syntactically valid JSON. Eliminates `JSONDecodeError`. Does not enforce key names or types.

**Layer 3 — Application validation** (Pydantic): The application validates parsed JSON against a typed schema. Catches anything that escaped Layers 1 and 2 — wrong key names, type mismatches, format violations, missing required fields.

Each layer catches what the others miss. Teams that run only Layer 1 see 3–5% failure rates. Teams that add Layer 2 see near-zero `JSONDecodeError` but still see key-name drift. Teams that add all three see failures surface as typed `ValidationError` exceptions with diagnostic detail — not as silent corrupted records in the database.

The implementation cost of all three is low. The cost of running only one is silent data corruption at scale.

---

### Decision Rule

**USE all three layers when:**
- LLM output touches a database, API response, or downstream business logic.
- The endpoint runs at production volume.
- A silent wrong value would cause a production incident.

**AVOID adding Pydantic validation before the schema is stable:**
- While iterating on which fields to return, a strict schema slows iteration.
- Get the output format right in the prompt first, then lock it down with Pydantic.

**SIGNAL:** If a key-name drift or wrong type would go undetected and reach the database → add all three layers now. If you're still iterating on what fields to return → defer Pydantic until the schema stabilizes.

## §4 Main Concepts + Production Reality

### Concept 1 — JSON Mode: What It Guarantees and What It Doesn't

Your `/classify` endpoint works in local testing. You deploy it. Two weeks in, the support team reports intermittent 500 errors. You check the logs. The raw model output for the failing tickets looks like this:

```
```json
{"category": "billing", "confidence": 0.97}
```
```

The model wrapped the JSON in a markdown code fence. `json.loads()` raises `JSONDecodeError`. The endpoint returns 500.

Here's what happened:
1. The model generated a code-fenced response — common when prompted without JSON mode.
2. Your code called `json.loads()` on the full raw string.
3. The backticks are not valid JSON → `JSONDecodeError`.
4. The exception propagated unhandled → 500.

The fix is **JSON mode**:

```python
response = client.chat.completions.create(
    model=DEPLOYMENT,
    temperature=0.0,
    max_tokens=300,
    response_format={"type": "json_object"},   # ← JSON mode
    messages=[...]
)
raw = response.choices[0].message.content
result = json.loads(raw)   # guaranteed not to raise JSONDecodeError
```

**What JSON mode guarantees:**
- `json.loads()` will not raise `JSONDecodeError`.
- No markdown fences, no prose before or after the JSON, no single-quoted keys.

**What JSON mode does NOT guarantee:**

| Failure type | JSON mode prevents it? |
|---|---|
| Markdown-wrapped JSON | Yes |
| Wrong key names (`"date"` vs `"invoice_date"`) | No |
| Wrong types (`"4750"` string vs `4750.0` float) | No |
| Missing required fields | No |
| Semantically wrong values | No |

One caveat: if the system prompt does not mention JSON, JSON mode may cause the model to return `{}` instead of content. JSON mode must always be paired with an explicit output format instruction in the system prompt (→ 2.6). The two layers work together — neither is sufficient alone.

**Engineering implication:**
- Enable JSON mode on every production endpoint that expects JSON output.
- Treat `JSONDecodeError` in production as a signal that JSON mode is not active.
- JSON mode eliminates one failure class. It does not replace schema validation.

**Common trap:** Assuming JSON mode handles everything. It only guarantees syntactic validity — not that the model returned the keys your prompt specified.

### Concept 2 — Pydantic Schema Validation for LLM Output

JSON mode is active. Your extraction endpoint runs for a week with no errors. Then you notice 4% of extracted records have `null` in every field — despite the tickets containing clear information. You add some logging. The raw JSON looks like this:

```json
{"cat": "billing", "urg": "high", "summary": "Customer was double-charged."}
```

The model returned abbreviated key names. Your schema is all-optional. Here's what happened:
1. Model returned valid JSON with abbreviated keys.
2. `json.loads()` parsed it successfully.
3. `ExtractionOutput.model_validate(data)` was called.
4. All schema fields are `Optional` with `= None`.
5. Unknown keys (`"cat"`, `"urg"`) were ignored; expected keys were absent.
6. Pydantic returned `ExtractionOutput(category=None, urgency=None, ...)`.
7. No `ValidationError` was raised — the object looked valid.
8. Database wrote all-null records silently.

This is the **all-optional schema trap**. The fix is **Pydantic schema validation** with required fields and typed constraints.

**Four patterns that matter for LLM output:**

**Required fields — raise on absence:**
```python
category: Literal["billing", "technical", "account", "general"]
```
No default means absence raises `ValidationError: Field required`. Key-name drift now surfaces immediately.

**Optional fields with null — for legitimately absent data:**
```python
ticket_id: str | None = None
```
Use when the model may legitimately not find the field. Maps to the null-for-missing constraint in the system prompt.

**Type coercion — automatic for common model behavior:**
```python
confidence: float
```
Pydantic V2 coerces before raising. A `float` field receiving `"0.97"` (string) becomes `0.97`. This handles a common model pattern without raising. Add a `@field_validator` if you need to reject coercible values.

**Field validators — for format constraints beyond type:**
```python
@field_validator("invoice_date")
@classmethod
def validate_iso_date(cls, v: str | None) -> str | None:
    if v is None:
        return v
    try:
        date.fromisoformat(v)
        return v
    except ValueError:
        raise ValueError(f"invoice_date must be ISO 8601 (YYYY-MM-DD), got: {repr(v)}")
```

The model can return a valid string that violates the format — `"15/03/2024"` instead of `"2024-03-15"`. Without a `@field_validator`, this passes validation and breaks the database write downstream.

**Engineering implication:**
- Make required fields required — use `Literal` types for enums.
- Add `@field_validator` for dates, ranges, lengths, and regex patterns.
- Accept that `ValidationError` on a required field is the correct signal, not an error to suppress.

**Common trap:** Making every field `Optional` to "be flexible." An all-optional schema never raises. Wrong key names silently produce all-null objects.

### Concept 3 — Defensive Parsing with Retry

You add Pydantic validation. Now `ValidationError` surfaces correctly when the model returns a wrong format. But your route handler doesn't catch it. The first time a ticket with a non-ISO date triggers `ValidationError`, the request fails with an unhandled exception. The ticket is not processed — it's just lost.

Here's what happened:
1. Model returned `{"invoice_date": "15/03/2024"}` — valid JSON, wrong format.
2. `ExtractionOutput.model_validate(data)` raised `ValidationError`.
3. No exception handler existed in the route.
4. Request returned 500 and the ticket was dropped.

The fix is a **defensive parsing wrapper with retry** — a function that handles `ValidationError` at the LLM boundary, retries once with the error detail injected, and returns a typed result object instead of raising.

**What goes in a retry prompt:**
```python
messages.append({"role": "assistant", "content": raw_output})
messages.append({
    "role": "user",
    "content": (
        f"Your previous output failed schema validation:\n{error_detail}\n\n"
        f"Return the corrected JSON only. Fix the field(s) listed above."
    )
})
```

The retry must be specific — not "try again" but "fix `invoice_date`: must be ISO 8601 (YYYY-MM-DD), got '15/03/2024'." The model uses the error detail to target the correction. Specific retries succeed roughly 80% of the time on format errors.

**The return type is always typed:**
```
ParseResult(status=SUCCESS, data=ExtractionOutput, ...)  # first attempt succeeded
ParseResult(status=RETRY_OK, data=ExtractionOutput, ...) # retry succeeded
ParseResult(status=FAILED, data=None, error_detail=...)  # all retries exhausted
ParseResult(status=ERROR, data=None, error_detail=...)   # unexpected exception
```

Callers always check `result.ok` before accessing `result.data`. The LLM boundary is the only place where `ValidationError` is caught.

**Engineering implication:**
- One retry is usually enough. More than two retries on the same input likely means a prompt or schema problem — not a recoverable model error.
- For real-time user-facing endpoints: set `max_retries=0`. Return a typed failure immediately rather than adding retry latency.
- For background extraction pipelines: `max_retries=1` is standard. Log failures for review.

**Common trap:** `except Exception: return {}` — an empty dict that bypasses Pydantic is silent data corruption. A typed `ParseResult(status=FAILED)` is always the correct alternative.

```python
# app/schemas/outputs.py
"""
Typed Pydantic schemas for all LLM output types in ticket-ops-api.

All LLM outputs pass through model_validate() before reaching any
application code. Downstream services work with typed objects, not raw dicts.
"""

import json
import logging
import os
from dataclasses import dataclass
from datetime import date
from enum import Enum
from typing import Generic, Literal, Type, TypeVar

from openai import AzureOpenAI
from pydantic import BaseModel, ValidationError, field_validator

logger = logging.getLogger(__name__)

T = TypeVar("T", bound=BaseModel)

client = AzureOpenAI(
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    api_key=os.environ["AZURE_OPENAI_API_KEY"],
    api_version="2024-02-01",
)
DEPLOYMENT = os.environ.get("AZURE_OPENAI_DEPLOYMENT", "gpt-4o")


# ---------------------------------------------------------------------------
# Output schemas — one per task type
# ---------------------------------------------------------------------------

class ClassificationOutput(BaseModel):
    category: Literal["billing", "technical", "account", "general"]
    confidence: float

    @field_validator("confidence")
    @classmethod
    def validate_confidence_range(cls, v: float) -> float:
        if not 0.0 <= v <= 1.0:
            raise ValueError(f"confidence must be between 0.0 and 1.0, got: {v}")
        return round(v, 4)


class ExtractionOutput(BaseModel):
    category: Literal["billing", "technical", "account", "general"] | None = None
    urgency: Literal["low", "medium", "high", "critical"] | None = None
    issue_summary: str | None = None
    ticket_id: str | None = None
    missing_info: list[str] = []

    @field_validator("issue_summary")
    @classmethod
    def validate_summary_length(cls, v: str | None) -> str | None:
        if v is not None and len(v.split()) > 25:
            words = v.split()[:20]
            logger.warning(
                "issue_summary exceeded word limit, truncated",
                extra={"original_length": len(v.split())},
            )
            return " ".join(words)
        return v


class SummarizationOutput(BaseModel):
    summary: str
    action_items: list[str] = []
    priority: Literal["low", "medium", "high", "critical"]


class DraftOutput(BaseModel):
    subject: str
    body: str
    tone: Literal["empathetic", "informational", "escalation"]

    @field_validator("subject")
    @classmethod
    def validate_subject_length(cls, v: str) -> str:
        if len(v) > 60:
            logger.warning(
                "Draft subject exceeded 60 chars, truncating",
                extra={"subject": v},
            )
            return v[:60]
        return v

    @field_validator("body")
    @classmethod
    def validate_body_not_empty(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("Draft body must not be empty")
        return v.strip()


# ---------------------------------------------------------------------------
# Retry wrapper — typed result, never raises unhandled exceptions
# ---------------------------------------------------------------------------

class ParseStatus(str, Enum):
    SUCCESS  = "success"
    RETRY_OK = "retry_ok"     # first attempt failed; retry succeeded
    FAILED   = "failed"       # all retries exhausted; structured failure returned
    ERROR    = "error"        # unexpected non-validation exception


@dataclass
class ParseResult(Generic[T]):
    """
    Typed result of an LLM call + schema validation.

    Callers always receive a ParseResult — never a naked exception or raw dict.

    Usage:
        result = parse_llm_output(ClassificationOutput, system_prompt, user_msg)
        if result.ok:
            output: ClassificationOutput = result.data
        else:
            log and handle result.error_detail
    """
    status: ParseStatus
    data: T | None
    raw: str | None
    error_detail: str | None
    attempts: int

    @property
    def ok(self) -> bool:
        return self.status in (ParseStatus.SUCCESS, ParseStatus.RETRY_OK)


def parse_llm_output(
    schema: Type[T],
    system_prompt: str,
    user_message: str,
    max_retries: int = 1,
    temperature: float = 0.0,
    max_tokens: int = 400,
) -> ParseResult[T]:
    """
    Call the LLM, validate the output against a Pydantic schema, and retry
    once on ValidationError with the error detail injected back into the prompt.

    Returns a ParseResult — never raises. Callers inspect result.ok before
    accessing result.data.
    """
    messages = [
        {"role": "system", "content": system_prompt},
        {"role": "user",   "content": user_message},
    ]
    raw: str | None = None
    attempts = 0

    for attempt in range(max_retries + 1):
        attempts += 1
        try:
            response = client.chat.completions.create(
                model=DEPLOYMENT,
                temperature=temperature,
                max_tokens=max_tokens,
                response_format={"type": "json_object"},
                messages=messages,
            )
            raw = response.choices[0].message.content.strip()
            validated = schema.model_validate_json(raw)
            status = ParseStatus.SUCCESS if attempt == 0 else ParseStatus.RETRY_OK
            return ParseResult(
                status=status, data=validated, raw=raw,
                error_detail=None, attempts=attempts,
            )

        except ValidationError as exc:
            error_detail = str(exc)
            logger.warning(
                "LLM output schema validation failed",
                extra={
                    "schema": schema.__name__,
                    "attempt": attempt + 1,
                    "error": error_detail,
                    "raw": (raw or "")[:300],
                },
            )
            if attempt < max_retries:
                messages.append({"role": "assistant", "content": raw or ""})
                messages.append({
                    "role": "user",
                    "content": (
                        f"Your previous output failed schema validation:\n{error_detail}\n\n"
                        f"Return the corrected JSON only. Fix the field(s) listed above."
                    ),
                })
            else:
                return ParseResult(
                    status=ParseStatus.FAILED, data=None, raw=raw,
                    error_detail=error_detail, attempts=attempts,
                )

        except json.JSONDecodeError as exc:
            # JSON mode should prevent this — if it occurs, do not retry
            logger.error(
                "JSONDecodeError despite JSON mode — raw output logged",
                extra={"raw": (raw or "")[:300], "error": str(exc)},
            )
            return ParseResult(
                status=ParseStatus.ERROR, data=None, raw=raw,
                error_detail=f"JSONDecodeError: {exc}", attempts=attempts,
            )

        except Exception as exc:
            logger.error(
                "Unexpected error in parse_llm_output",
                extra={"schema": schema.__name__, "error": str(exc)},
                exc_info=True,
            )
            return ParseResult(
                status=ParseStatus.ERROR, data=None, raw=raw,
                error_detail=str(exc), attempts=attempts,
            )

    return ParseResult(
        status=ParseStatus.FAILED, data=None, raw=raw,
        error_detail="max_retries exceeded", attempts=attempts,
    )
```

## §7 Interview Drill

### Q1 — Concept: Explain the three validation layers for LLM output and what each one catches.

**Model answer:**
Three layers catch different failure classes. The prompt's output format section — in the system prompt — reduces the violation rate by making the schema explicit. But the model follows specs most of the time, not all of the time. JSON mode constrains token sampling so the output is syntactically valid JSON — no markdown fences, no code blocks, no prose wrappers. But it doesn't enforce key names or types. Pydantic schema validation catches what escaped the first two layers: wrong key names, type mismatches, invalid formats, missing required fields. Each layer is cheap to add. Running only one leaves systematic gaps.

---

### Q2 — System Design: Your extraction pipeline processes 10,000 tickets per day. Validation fails on 3% of inputs. You're considering adding a retry. What factors determine your decision and what is the latency impact?

**Model answer:**
The key question is: what is the cost of a failed extraction vs. the cost of retry latency? For background batch processing — tickets queued overnight — one retry per 3% failure rate adds one extra API call on 300 tickets per day. The latency hit is invisible to users. Worth it. For real-time user-facing endpoints — a user waiting for a response — adding one API call's latency for 3% of requests may be acceptable, but you should measure it. Set `max_retries=0` for endpoints where latency matters and handle failures by returning a typed error response to the user. The failure rate also tells you something: 3% is high. It suggests the output format section in the system prompt is under-specified. Fix the prompt first — reduce the failure rate — then decide if retry is needed at all.

---

### Q3 — Debug: Your `/extract` endpoint returns `ExtractionOutput(category=None, urgency=None, issue_summary=None, ticket_id=None, missing_info=[])` on inputs that clearly contain ticket fields. No `ValidationError` is raised. What is the cause and what is the fix?

**Model answer:**
This is the all-optional schema trap. The model is returning valid JSON with key names that don't match the schema's field names — for example, `{"cat": "billing", "urg": "high"}` instead of `{"category": "billing", "urgency": "high"}`. Pydantic ignores unknown keys by default. All expected keys are absent, all fields default to `None`, and `model_validate()` returns without error. First confirm by printing the raw JSON before validation. If the keys don't match the field names, you have two problems: the model is drifting from the key names specified in the output format section, and the schema has no required fields to catch it. Fix both: update the system prompt's output format section with the exact key names, and make required fields required — remove their `Optional` defaults. The first `ValidationError` on a wrong key is the correct signal that the prompt and schema are misaligned.

## §10 Key Takeaways

- **Three layers, three failure classes.** Prompt specification reduces violation rate. JSON mode eliminates `JSONDecodeError`. Pydantic catches key-name drift, type mismatches, and format violations. Each layer catches what the others miss. Running only one leaves systematic gaps.

- **JSON mode removes syntax failures, not semantic failures.** A model with JSON mode active can still return `{"date": "2024-03-15"}` when you expected `{"invoice_date": "2024-03-15"}`. Valid JSON. Wrong keys. Pydantic is the fix.

- **All-optional schemas never raise.** A schema where every field is `Optional` with `= None` will silently accept wrong key names and return all-null objects. Make required fields required. `ValidationError` on a missing required field is the correct signal for key-name drift.

- **Pydantic coerces before raising.** A `float` field receiving `"0.97"` (string) returns `0.97`. This handles common model behavior. Add a `@field_validator` if you need to reject coercible values.

- **Retry with error detail, not blindly.** Injecting the `ValidationError` message back into the conversation gives the model specific correction targets. "Fix `invoice_date`: must be ISO 8601 (YYYY-MM-DD), got '15/03/2024'" succeeds far more often than a blind retry.

- **Never use `except Exception: return {}`.** An empty dict that bypasses Pydantic is silent data corruption. Return a typed `ParseResult(status=FAILED)`. The caller gets diagnostic information; the pipeline keeps running.

- **`ParseResult` is the typed interface between the LLM and application code.** Service files return `ParseResult[T]`. Route handlers check `result.ok` before accessing `result.data`. The LLM boundary is the only place where `ValidationError` is caught.

## §12 Optional Advanced Notes

**Structured Outputs with JSON Schema (stronger than JSON mode):** Azure OpenAI and OpenAI support `response_format={"type": "json_schema", "json_schema": {"name": "...", "strict": True, "schema": ...}}`. This constrains token sampling to match the declared schema — key names and types are enforced at the API level. Use `pydantic`'s `.model_json_schema()` to generate the schema object. Requires model version `gpt-4o-2024-08-06` or later. With Structured Outputs active, most key-name and type `ValidationError`s disappear. Retries drop to format-only failures. Pydantic field validators still run — they handle semantic constraints the JSON Schema cannot express.

**Async wrappers for production throughput:** The `parse_llm_output()` function uses the synchronous OpenAI client. For high-throughput pipelines with concurrent ticket processing, replace with the async client (`AsyncAzureOpenAI`) and `async def parse_llm_output_async(...)`. The retry logic is identical — replace `client.chat.completions.create` with `await client.chat.completions.create`. Covered in streaming and batch processing (→ 2.9, → 2.15).
