## Where This Fits in the System

**State:** After 2.6, the extraction endpoint calls `json.loads()` on the raw model output and accesses fields directly: `result["category"]`, `result["urgency"]`. No schema validation exists. The system prompt specifies the correct output format, but nothing enforces that the model actually returned what was specified.

**Gap:** `json.loads()` only guarantees syntactic validity. The model can return valid JSON with the wrong key names (`"date"` instead of `"invoice_date"`), wrong types (`"4750"` string instead of `4750.0` number), or missing required fields — all silently, all reaching the database or downstream service with no error. These failures are rare enough on well-formed inputs that they pass development testing, and frequent enough at production volume that they become incidents.

**Unlocks:** Every service that calls the LLM returns a validated, typed Pydantic object — not a raw dict. Downstream code (database writes, API responses, routing logic) works with typed fields. A `ValidationError` surfaces at the LLM boundary with the exact field name and failure reason, not six layers deep as a `KeyError`. 2.8 (function calling) reuses the same schema pattern for tool parameter validation.

## Core Mental Model

### The Three-Layer Defense

When an LLM returns output that is supposed to match a schema, three different layers can catch failures — and they catch different things:

**Layer 1 — Prompt specification** (→ 2.5, 2.6): The system prompt's output format section lists exact key names, types, and null-handling rules. This reduces the rate of schema violations by making the spec explicit. It does not prevent them — the model follows the spec most of the time, not all of the time.

**Layer 2 — API enforcement** (JSON mode / Structured Outputs): The API constrains token sampling so the output is syntactically valid JSON, or, with Structured Outputs, matches the declared schema at the token level. JSON mode eliminates `JSONDecodeError`. Structured Outputs eliminate key-name and type failures for supported models.

**Layer 3 — Application validation** (Pydantic): The application validates the parsed JSON against a typed schema. Catches anything that escaped Layers 1 and 2 — wrong key names, type mismatches, invalid formats (date strings that aren't ISO 8601), missing required fields.

The point of understanding three layers is this: **each one catches what the others miss.** Teams that run only Layer 1 see 3–5% failure rates. Teams that add Layer 2 see near-zero `JSONDecodeError` but still see key-name drift. Teams that add all three see failures surface as typed `ValidationError` exceptions with diagnostic detail — not silent wrong values in the database.

You always need all three. The implementation cost is low. The cost of running only one is silent data corruption at scale.

## Main Concepts + Production Reality

### Concept 1 — JSON Mode: What It Does and Does Not Do

**The problem it solves:** Without JSON mode, the model generates free text. Asking it to "return JSON" produces valid JSON most of the time and fails in several ways the rest of the time: JSON wrapped in a markdown code fence, a Python dict literal with single quotes, a prose sentence explaining why it cannot answer, or truncated JSON at a token limit boundary.

**What JSON mode guarantees:**

```python
response = client.chat.completions.create(
    model=DEPLOYMENT,
    temperature=0.0,
    max_tokens=300,
    response_format={"type": "json_object"},   # ← JSON mode
    messages=[...]
)
raw = response.choices[0].message.content
result = json.loads(raw)   # guaranteed not to raise JSONDecodeError
```

- `json.loads()` will not raise `JSONDecodeError`
- No markdown fences (`json {...}`)
- No prose before or after the JSON
- No single-quoted keys

**What JSON mode does NOT guarantee:**

- The returned object has the keys your prompt specified
- Key names match exactly (not synonyms or variants)
- Values have the correct types (`"4750"` string vs. `4750.0` float)
- Required fields are present rather than absent
- Values are semantically correct (not hallucinated)

**The caveat that causes silent failures:** If the prompt does not explicitly mention JSON, enabling JSON mode may cause the model to return an empty object `{}` rather than produce content. JSON mode must always be paired with an explicit output format instruction in the system prompt. The two layers work together — neither is sufficient alone.

**Structured Outputs (stronger variant):** Azure OpenAI and OpenAI also offer `response_format={"type": "json_schema", ...}` with a JSON Schema object. This constrains token sampling to match the declared schema — key names and types are enforced at the API level, not just the application level. This eliminates the key-name drift problem for supported model versions (gpt-4o-2024-08-06+). It does not eliminate semantic validation (correct formats, positive numbers, non-future dates) — Pydantic still does that work.

| Layer | Syntactic validity | Key names enforced | Type enforcement | Format validation |
|---|---|---|---|---|
| No mode | Not guaranteed | Not guaranteed | Not guaranteed | Not guaranteed |
| JSON mode | Guaranteed | Not guaranteed | Not guaranteed | Not guaranteed |
| Structured Outputs (json_schema) | Guaranteed | Guaranteed | Guaranteed | Not guaranteed |
| + Pydantic validation | Guaranteed | Guaranteed | Guaranteed + coercion | Field validators |

**Production rule:** Use JSON mode on all deployments. Use Structured Outputs when the model version supports it. Run Pydantic validation on all outputs regardless of which API mode is active.

### Concept 2 — Pydantic Schema Definition for LLM Output

**What Pydantic adds:** A Pydantic model turns a raw dict into a typed Python object with validated fields. `model_validate(data)` either returns a fully typed object or raises `ValidationError` with the exact field that failed and why.

**Four patterns that matter for LLM output:**

**Optional fields with null:**

```python
invoice_number: str | None = None
```

Use when the model may legitimately not find the field. Maps to the null-for-missing constraint in the system prompt. Optional fields do not raise on absence — they return `None`.

**Required fields (no default):**

```python
vendor_name: str
```

Use for fields that must always be present. Absence raises `ValidationError: Field required`. This is the correct signal: the model returned a key-name mismatch or a structurally wrong output.

**Type coercion:**

Pydantic V2 coerces before raising. A `float` field receiving `"4750"` (string) is coerced to `4750.0`. This handles a common model behavior — returning numbers as strings — without a `ValidationError`. If you need to reject string numbers, add a `@field_validator` that checks `isinstance(v, (int, float))` before coercion runs.

**Field validators for format-sensitive fields:**

```python
@field_validator("invoice_date")
@classmethod
def validate_iso_date(cls, v: str | None) -> str | None:
    if v is None:
        return v
    try:
        date.fromisoformat(v)
        return v
    except ValueError:
        raise ValueError(f"invoice_date must be ISO 8601 (YYYY-MM-DD), got: {repr(v)}")
```

Use when the field has a format constraint beyond type — dates, currency codes, email addresses, regex patterns. The model can return a valid string that violates the format (`"15/03/2024"` instead of `"2024-03-15"`). Without a `@field_validator`, this passes validation silently and breaks the database write downstream.

**The silent-failure trap with optional fields:**

```python
# All optional — wrong key names silently produce all-null object
class BadSchema(BaseModel):
    invoice_number: str | None = None
    invoice_date: str | None = None
    vendor_name: str | None = None
    total_amount: float | None = None

# Model returns wrong keys: {"number": ..., "date": ..., "vendor": ..., "amount": ...}
result = BadSchema.model_validate_json(wrong_keys_json)
# → BadSchema(invoice_number=None, invoice_date=None, vendor_name=None, total_amount=None)
# → No ValidationError raised. All nulls. Looks valid. Silent data corruption.
```

The fix: make required fields required. Accept that a `ValidationError` on a required field is the correct signal to act on — not an error to suppress.

### Concept 3 — Defensive Parsing Wrapper with Retry and Fallback

**The problem:** Even with JSON mode and Pydantic schemas, some outputs will fail validation. The system prompt reduces the rate; it does not eliminate it. When validation fails, there are three possible responses:

1. **Retry with error context.** Inject the `ValidationError` back into the conversation and call the API again. The model sees what it produced and what was wrong. This resolves format errors (wrong date format, wrong key name) about 80% of the time on the retry. Cost: one extra API call.

2. **Return a typed failure object.** After N retries, return a structured `ParseResult(status=FAILED, ...)` — not a raw exception. The caller decides how to handle it: queue for human review, retry later, return an error response. The pipeline continues running.

3. **Never swallow the error.** `except Exception: return {}` is the anti-pattern. An empty dict that bypasses Pydantic reaches the database as all-null fields — silent corruption. A typed failure object is always preferable to a silently empty result.

**What belongs in a retry prompt:**

```python
# On ValidationError, add the error detail to the conversation
messages.append({"role": "assistant", "content": raw_output})
messages.append({
    "role": "user",
    "content": (
        f"Your previous output failed schema validation:\n{error_detail}\n\n"
        f"Return the corrected JSON only. Fix the field(s) listed above."
    )
})
```

The retry should be specific — not "try again" but "fix invoice_date: must be ISO 8601 format." The model uses the error detail to target the correction.

**Latency and cost:** One retry adds one API call's latency and cost. For extraction pipelines, this is acceptable for a 3–5% validation failure rate. For real-time user-facing responses, set `max_retries=0` and handle failures by returning a typed failure object to the user immediately rather than waiting for a retry.

# app/schemas/outputs.py
"""
Typed Pydantic schemas for all LLM output types in ticket-ops-api.

All LLM outputs pass through model_validate() before reaching any
application code. Downstream services work with typed objects, not raw dicts.
"""

import json
import logging
import os
from dataclasses import dataclass
from datetime import date
from enum import Enum
from typing import Generic, Literal, Type, TypeVar

from openai import AzureOpenAI
from pydantic import BaseModel, ValidationError, field_validator

logger = logging.getLogger(__name__)

T = TypeVar("T", bound=BaseModel)

client = AzureOpenAI(
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    api_key=os.environ["AZURE_OPENAI_API_KEY"],
    api_version="2024-02-01",
)
DEPLOYMENT = os.environ.get("AZURE_OPENAI_DEPLOYMENT", "gpt-4o")


# ---------------------------------------------------------------------------
# Output schemas — one per task type
# ---------------------------------------------------------------------------

class ClassificationOutput(BaseModel):
    category: Literal["billing", "technical", "account", "general"]
    confidence: float

    @field_validator("confidence")
    @classmethod
    def validate_confidence_range(cls, v: float) -> float:
        if not 0.0 <= v <= 1.0:
            raise ValueError(f"confidence must be between 0.0 and 1.0, got: {v}")
        return round(v, 4)


class ExtractionOutput(BaseModel):
    category: Literal["billing", "technical", "account", "general"] | None = None
    urgency: Literal["low", "medium", "high", "critical"] | None = None
    issue_summary: str | None = None
    ticket_id: str | None = None
    missing_info: list[str] = []

    @field_validator("issue_summary")
    @classmethod
    def validate_summary_length(cls, v: str | None) -> str | None:
        if v is not None and len(v.split()) > 25:
            # Truncate and log — model exceeded the 20-word limit
            words = v.split()[:20]
            logger.warning(
                "issue_summary exceeded word limit, truncated",
                extra={"original_length": len(v.split())},
            )
            return " ".join(words)
        return v


class SummarizationOutput(BaseModel):
    summary: str
    action_items: list[str] = []
    priority: Literal["low", "medium", "high", "critical"]


class DraftOutput(BaseModel):
    subject: str
    body: str
    tone: Literal["empathetic", "informational", "escalation"]

    @field_validator("subject")
    @classmethod
    def validate_subject_length(cls, v: str) -> str:
        if len(v) > 60:
            logger.warning(
                "Draft subject exceeded 60 chars, truncating",
                extra={"subject": v},
            )
            return v[:60]
        return v

    @field_validator("body")
    @classmethod
    def validate_body_not_empty(cls, v: str) -> str:
        if not v.strip():
            raise ValueError("Draft body must not be empty")
        return v.strip()


# ---------------------------------------------------------------------------
# Retry wrapper — typed result, never raises unhandled exceptions
# ---------------------------------------------------------------------------

class ParseStatus(str, Enum):
    SUCCESS  = "success"
    RETRY_OK = "retry_ok"     # first attempt failed; retry succeeded
    FAILED   = "failed"       # all retries exhausted; structured failure returned
    ERROR    = "error"        # unexpected non-validation exception


@dataclass
class ParseResult(Generic[T]):
    """
    Typed result of an LLM call + schema validation.

    Callers always receive a ParseResult — never a naked exception or raw dict.

    Usage:
        result = parse_llm_output(ClassificationOutput, system_prompt, user_msg)
        if result.ok:
            output: ClassificationOutput = result.data
        else:
            log and handle result.error_detail
    """
    status: ParseStatus
    data: T | None
    raw: str | None
    error_detail: str | None
    attempts: int

    @property
    def ok(self) -> bool:
        return self.status in (ParseStatus.SUCCESS, ParseStatus.RETRY_OK)


def parse_llm_output(
    schema: Type[T],
    system_prompt: str,
    user_message: str,
    max_retries: int = 1,
    temperature: float = 0.0,
    max_tokens: int = 400,
) -> ParseResult[T]:
    """
    Call the LLM API, parse the output against a Pydantic schema, and retry
    once on ValidationError with the error detail injected back into the prompt.

    Returns a ParseResult — never raises. Callers inspect result.ok before
    accessing result.data.

    Args:
        schema:       Pydantic model class to validate against.
        system_prompt: Full system prompt string (from get_system_prompt()).
        user_message:  User turn content (from templates.py constructors).
        max_retries:   Number of retries after the first attempt. Default 1.
        temperature:   Sampling temperature. Default 0.0 for deterministic output.
        max_tokens:    Max tokens for the response.

    Returns:
        ParseResult with status SUCCESS, RETRY_OK, FAILED, or ERROR.
    """
    messages = [
        {"role": "system", "content": system_prompt},
        {"role": "user",   "content": user_message},
    ]
    raw: str | None = None
    attempts = 0

    for attempt in range(max_retries + 1):
        attempts += 1
        try:
            response = client.chat.completions.create(
                model=DEPLOYMENT,
                temperature=temperature,
                max_tokens=max_tokens,
                response_format={"type": "json_object"},
                messages=messages,
            )
            raw = response.choices[0].message.content.strip()
            validated = schema.model_validate_json(raw)
            status = ParseStatus.SUCCESS if attempt == 0 else ParseStatus.RETRY_OK
            return ParseResult(
                status=status,
                data=validated,
                raw=raw,
                error_detail=None,
                attempts=attempts,
            )

        except ValidationError as exc:
            error_detail = str(exc)
            logger.warning(
                "LLM output schema validation failed",
                extra={
                    "schema": schema.__name__,
                    "attempt": attempt + 1,
                    "error": error_detail,
                    "raw": (raw or "")[:300],
                },
            )
            if attempt < max_retries:
                # Inject error into conversation and retry
                messages.append({"role": "assistant", "content": raw or ""})
                messages.append({
                    "role": "user",
                    "content": (
                        f"Your previous output failed schema validation:\n{error_detail}\n\n"
                        f"Return the corrected JSON only. Fix the field(s) listed above."
                    ),
                })
            else:
                return ParseResult(
                    status=ParseStatus.FAILED,
                    data=None,
                    raw=raw,
                    error_detail=error_detail,
                    attempts=attempts,
                )

        except json.JSONDecodeError as exc:
            # JSON mode should prevent this; if it occurs, do not retry
            logger.error(
                "JSONDecodeError despite JSON mode — raw output logged",
                extra={"raw": (raw or "")[:300], "error": str(exc)},
            )
            return ParseResult(
                status=ParseStatus.ERROR,
                data=None,
                raw=raw,
                error_detail=f"JSONDecodeError: {exc}",
                attempts=attempts,
            )

        except Exception as exc:
            logger.error(
                "Unexpected error in parse_llm_output",
                extra={"schema": schema.__name__, "error": str(exc)},
                exc_info=True,
            )
            return ParseResult(
                status=ParseStatus.ERROR,
                data=None,
                raw=raw,
                error_detail=str(exc),
                attempts=attempts,
            )

    # Unreachable, but satisfies the type checker
    return ParseResult(
        status=ParseStatus.FAILED, data=None, raw=raw,
        error_detail="max_retries exceeded", attempts=attempts,
    )
```

    ("March 15, 2024",  "2024-03-15"),   # human readable → ISO
    ("not a date",      "raises"),       # should raise ValueError
]

for input_val, expected in cases:
    data = {
        "category": "technical", "urgency": "medium",
        "issue_summary": "Test.", "ticket_id": None,
        "missing_info": [], "ticket_created": input_val,
    }
    if expected == "raises":
        try:
            ExtractionOutput.model_validate(data)
            print(f"FAIL: {input_val!r} should have raised")
        except ValidationError as e:
            print(f"PASS: {input_val!r} raised correctly")
    else:
        result = ExtractionOutput.model_validate(data)
        assert result.ticket_created == expected, f"Got {result.ticket_created!r}"
        print(f"PASS: {input_val!r} → {result.ticket_created!r}")
```

## Key Takeaways

- **Three layers, three failure classes.** Prompt specification reduces violation rate. JSON mode eliminates `JSONDecodeError`. Pydantic catches key-name drift, type mismatches, and format violations. Each layer catches what the others miss. Running only one layer leaves systematic gaps that reach production.

- **JSON mode removes syntax failures, not semantic failures.** A model running in JSON mode can still return `{"date": "2024-03-15"}` when you expected `{"invoice_date": "2024-03-15"}`. Syntactically valid. Semantically wrong. Pydantic is the fix.

- **Optional fields with no required fields is a trap.** A schema where every field is `Optional` with `= None` will never raise `ValidationError`. Wrong key names silently produce all-null objects that reach the database. Make required fields required. Accept `ValidationError` as the correct signal for a key-name mismatch.

- **Pydantic coerces before raising.** A `float` field receiving `"4750"` (string) returns `4750.0` without error. This handles a common model behavior automatically. Add a `@field_validator` if you need to reject coercible values.

- **Retry with error detail, not blind retry.** Injecting the `ValidationError` message back into the conversation gives the model specific correction targets. "Fix invoice_date: must be ISO 8601 (YYYY-MM-DD), got '15/03/2024'" succeeds far more often than a blind retry.

- **Never swallow with `except Exception: return {}`**. An empty dict that bypasses Pydantic is silent data corruption. Return a typed `ParseResult(status=FAILED)` instead. The caller gets diagnostic information; the pipeline keeps running.

- **`ParseResult` is the typed interface between the LLM and application code.** Service files return `ParseResult[T]`. Route handlers check `result.ok` before accessing `result.data`. The LLM boundary is the only place where `ValidationError` is caught — not inside database write functions or business logic.

## Optional Advanced Notes

**Structured Outputs with JSON Schema (stronger than JSON mode):** Azure OpenAI and OpenAI support `response_format={"type": "json_schema", "json_schema": {"name": "...", "strict": True, "schema": ...}}`. This constrains token sampling to match the schema — key names and types are enforced at the API level, before the output reaches your application. Use `pydantic`'s `.model_json_schema()` to generate the schema object. Requires model version `gpt-4o-2024-08-06` or later. When available, this eliminates most key-name and type `ValidationError`s from `parse_llm_output()`, reducing retries to format-only failures (`invoice_date` format, `confidence` range). Even with Structured Outputs enabled, Pydantic field validators still run — they validate semantic correctness that the JSON Schema cannot express.

**Async wrappers for production throughput:** The `parse_llm_output()` function uses the synchronous OpenAI client. For high-throughput pipelines where multiple tickets are processed concurrently, replace with the async client (`AsyncAzureOpenAI`) and `async def parse_llm_output_async(...)`. The retry logic is identical; replace `client.chat.completions.create` with `await client.chat.completions.create`. Covered in more detail in the streaming and batch processing lessons (→ 2.9, → 2.15).
