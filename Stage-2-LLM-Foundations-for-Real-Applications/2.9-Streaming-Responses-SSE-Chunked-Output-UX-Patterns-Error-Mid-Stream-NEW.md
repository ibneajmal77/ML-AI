## §2 Where This Fits in the System

**State:** Every endpoint built so far — `/classify`, `/extract`, `/route` — waits for the full model response, then returns it in one HTTP response body. For short outputs (a label, a JSON object) that works fine. For `/summarize`, which generates 200–400 tokens of prose, it means the user sees nothing for 4–8 seconds, then the full text appears at once.

**Gap:** Users do not experience total generation time. They experience the wait before anything appears. A 7-second blank screen followed by instant full text feels slower than 400ms to first character followed by 7 seconds of progressive text — even though the total time is the same. Without streaming, the summarization endpoint has a blank-screen wait that drives abandonment regardless of how fast the model actually runs.

**Unlocks:** `/summarize` becomes a first-class UX feature — text appears immediately and builds progressively. The SSE generator pattern established here is reused in 2.17's `/chat` endpoint. The TTFT metric logged here becomes a production monitoring datapoint in 2.15.

## §3 Core Mental Model + Decision Rule

### The Pipe vs. The Bucket

Two ways to deliver water from a tank to a person waiting:

**Bucket:** Fill the entire bucket, carry it over, hand it to them. They wait the full time. They get everything at once.

**Pipe:** Open the tap immediately. Water flows as fast as it is produced. They feel it the moment the first drop arrives.

The model generates tokens one at a time — it is always producing output incrementally. The only question is whether you make the user wait for the bucket to fill, or open the pipe immediately.

Streaming opens the pipe. The model does the same work. The generation time is identical. What changes is *when the user sees the first token* — and that is the metric that determines whether the feature feels fast or slow.

---

### Decision Rule

**USE streaming when:**
- The endpoint generates more than ~100 tokens of prose that users read as it arrives.
- Users wait for the response in real time and blank-screen latency is noticeable.

**AVOID streaming when:**
- Output is short — a classification label, a JSON object. Streaming adds SSE overhead with no UX benefit.
- The caller needs the complete output before doing anything — batch processing, Pydantic validation, downstream function calls.

**SIGNAL:** If users describe the endpoint as "slow" despite a fast P95 total generation time → they are experiencing blank-screen wait. Add streaming. If the output feeds into another parse or function call before reaching the user → use non-streaming.

## §4 Main Concepts + Production Reality

### Concept 1 — What Streaming Is and How Chunks Work

**Scenario:**
Your summarization endpoint generates a 300-token summary. Without streaming, the user stares at a spinner for 5 seconds. With streaming, text starts appearing after 380ms.

**What happens step by step:**

```
Non-streaming path:
  Model generates token 1 → internal buffer
  Model generates token 2 → internal buffer
  ...
  Model generates token 300 → sends ALL 300 tokens at once
  User sees text: 5 seconds after request

Streaming path:
  Model generates token 1 → sends immediately → user sees first word: 380ms
  Model generates token 2 → sends immediately
  ...
  Model generates token 300 → sends immediately
  Total generation time: still 5 seconds — but user sees text the whole time
```

**What this is called:** streaming mode (`stream=True`). The API returns a `Stream` iterator instead of a single response object. Each iteration gives you a `ChatCompletionChunk`.

**The chunk shape** — three fields matter:

```python
for chunk in stream:
    chunk.choices[0].delta.content    # the new text fragment — or None
    chunk.choices[0].finish_reason    # None on most chunks; "stop" on the last one
    chunk.usage                        # token counts — only on the final chunk
                                       # (requires stream_options={"include_usage": True})
```

**Engineering implication:**
- Full content = concatenate all `delta.content` values where content is not `None`
- `finish_reason` appears on its own separate final chunk — not the same chunk as the last content
- Token counts only appear at the very end if you explicitly request them

**Common trap:**
Concatenating `delta.content` without checking for `None` first — several chunks carry no content (role chunk, final metadata chunk). `"text" + None` raises `TypeError`.

---

**First-token latency (TTFT — Time To First Token):**

```
Request sent → ... → first non-null delta.content arrives

That elapsed time = TTFT
```

TTFT is the UX metric. Total generation time is the cost/capacity metric. They are different numbers serving different purposes:

```
TTFT 380ms, total 5s   →  user feels: fast (sees text almost immediately)
TTFT 1200ms, total 3s  →  user feels: slow (long blank wait even though total is shorter)
```

Measure TTFT on every streaming request. Alert when P95 exceeds ~800ms.

### Concept 2 — SSE Endpoint Design in FastAPI

**Scenario:**
You return text to a browser as it generates. The browser needs to display each fragment the moment it arrives — not wait for the connection to close.

**What happens step by step:**

```
1. FastAPI endpoint returns StreamingResponse with media_type="text/event-stream"
2. Connection stays open (not closed after first response body)
3. Server yields one formatted line per token:  data: {"content": "The contract "}\n\n
4. Client reads each line as it arrives — no polling, no waiting
5. Server yields a final line:  data: {"event": "done"}\n\n
6. Client reads done event → closes connection
```

**What this is called:** SSE (Server-Sent Events). A standard HTTP protocol for server-to-client push over a persistent connection.

**The SSE wire format:**

```
data: {"event": "content", "content": "The contract "}\n\n
data: {"event": "content", "content": "specifies a 90-day"}\n\n
data: {"event": "done", "finish_reason": "stop"}\n\n
```

- Each event = one line starting with `data:` + payload
- Events are separated by `\n\n` (double newline) — this is the event boundary
- Payload can be any string — JSON is the practical choice for structured events

**Engineering implication:**

Three headers are required on the response — miss any one and streaming silently breaks:

```
Cache-Control: no-cache      → prevents proxy caching the entire stream before sending
X-Accel-Buffering: no        → disables nginx buffering (without this, nginx holds the
                                whole response and releases it all at once — streaming
                                becomes non-streaming silently)
Connection: keep-alive       → keeps the TCP connection open during generation
```

Also: use `AsyncAzureOpenAI` (not `AzureOpenAI`) in async FastAPI endpoints. The sync client blocks the event loop while waiting for each token — no other requests can be served during that time.

**Common trap:**
Using the sync client inside `async def`. Works in development (one request at a time). Breaks under load — each streaming request blocks the entire server.

### Concept 3 — Mid-Stream Errors

**Scenario:**
Your streaming endpoint has sent 40 tokens to the user when the database goes down. The model stops mid-sentence. The user sees "The contract specifies a 90-day notice period for—" and then nothing. No error. No explanation. The spinner just keeps spinning.

**What happens (the problem):**

```
HTTP 200 sent with first token              ← status code committed, cannot change it
40 tokens streamed to client
Database times out → exception raised in generator
Generator crashes → connection drops abruptly
Client receives: incomplete text + no signal about what happened
```

**What should happen instead:**

```
HTTP 200 sent with first token
40 tokens streamed to client
Exception occurs
Generator catches it → yields error event in-band
data: {"event": "error", "detail": "upstream timeout"}\n\n
Client reads error event → shows "partial result — generation interrupted"
Generator exits cleanly
```

**What this is called:** in-band error signaling. Because the HTTP status code is already committed (200), errors after the first byte must be communicated as events inside the stream itself — not as HTTP error codes.

**Engineering implication — three production concerns beyond happy path:**

```
1. In-band errors
   Wrap the entire generator in try/except
   On any exception → yield error event → generator exits
   Never let the exception propagate — that drops the connection silently

2. Heartbeat events (for long generations)
   Load balancers kill idle connections after ~30-60 seconds
   A 90-second generation with no bytes sent → connection killed at 60s
   Fix: yield "keep-alive" comment events every 15 seconds during generation
   SSE comment format: ": keep-alive\n\n"  (colon prefix = comment, client ignores it)

3. Hard timeout
   If the model stalls (stops generating mid-response) → generator blocks forever
   Fix: wrap generation loop with asyncio.wait_for(timeout=60)
   On timeout → yield error event → exit cleanly
```

**Common trap:**
Letting an exception inside the generator propagate unhandled. FastAPI catches it and closes the connection — but the client receives a truncated SSE stream with no `done` or `error` event. The client has no way to distinguish "generation complete" from "server crashed."

## §5 Project Integration

**Component:** `app/services/summarize.py` — full streaming summarization service and `/summarize` endpoint.

**Build:** Async generator with TTFT measurement, SSE event formatting, in-band error handling, heartbeat, timeout, and client-disconnect detection.

**Skeleton:**

```python
# app/services/summarize.py
"""
Streaming summarization service for ticket-ops-api.

Implements the /summarize endpoint as an SSE stream.
Each generated token is sent to the client immediately.
TTFT is measured and logged per request.
Errors after stream start are sent as in-band SSE error events.
"""

import asyncio
import json
import logging
import os
import time
import uuid

from fastapi import FastAPI
from fastapi.responses import StreamingResponse
from openai import AsyncAzureOpenAI
from pydantic import BaseModel

from app.prompts.system import get_system_prompt
from app.prompts.templates import summarize_user_message

logger = logging.getLogger(__name__)

async_client = AsyncAzureOpenAI(
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    api_key=os.environ["AZURE_OPENAI_API_KEY"],
    api_version="2024-02-01",
)
DEPLOYMENT = os.environ.get("AZURE_OPENAI_DEPLOYMENT", "gpt-4o")

GENERATION_TIMEOUT_SECONDS = 60.0
HEARTBEAT_INTERVAL_SECONDS = 15.0


# ---------------------------------------------------------------------------
# Streaming generator
# ---------------------------------------------------------------------------

async def stream_summary_chunks(
    document_text: str,
    max_tokens: int,
    request_id: str,
):
    """
    Async generator that yields SSE-formatted events.

    Event types:
      {"event": "first_token", "ttft_ms": 382.4}   — emitted once at TTFT
      {"event": "content", "content": "The ..."}   — one per token fragment
      {"event": "done", "finish_reason": "stop"}   — stream complete
      {"event": "error", "detail": "..."}          — in-band error (any failure after start)
    """
    messages = [
        {"role": "system", "content": get_system_prompt("summarization")},
        {"role": "user",   "content": summarize_user_message(document_text)},
    ]

    request_start = time.monotonic()
    first_token_seen = False
    last_heartbeat = time.monotonic()
    finish_reason: str | None = None

    try:
        stream = await async_client.chat.completions.create(
            model=DEPLOYMENT,
            messages=messages,
            max_tokens=max_tokens,
            temperature=0.3,
            stream=True,
            stream_options={"include_usage": True},
        )

        async for chunk in asyncio.wait_for(
            _iterate_stream(stream), timeout=GENERATION_TIMEOUT_SECONDS
        ):
            # Guard: some chunks (usage-only) have empty choices
            if not chunk.choices:
                continue

            delta_content = chunk.choices[0].delta.content
            chunk_finish  = chunk.choices[0].finish_reason

            # TTFT — measure and emit once on the first content token
            if delta_content and not first_token_seen:
                first_token_seen = True
                ttft_ms = (time.monotonic() - request_start) * 1000
                logger.info("request_id=%s TTFT=%.1f ms", request_id, ttft_ms)
                yield f"data: {json.dumps({'event': 'first_token', 'ttft_ms': round(ttft_ms, 1)})}\n\n"

            # Content event — one per token fragment
            if delta_content:
                yield f"data: {json.dumps({'event': 'content', 'content': delta_content})}\n\n"

            # Capture finish_reason from the final chunk
            if chunk_finish is not None:
                finish_reason = chunk_finish

            # Heartbeat — keeps load balancer connection alive during long generations
            now = time.monotonic()
            if now - last_heartbeat > HEARTBEAT_INTERVAL_SECONDS:
                yield ": keep-alive\n\n"   # SSE comment — clients ignore it
                last_heartbeat = now

        # Stream exhausted — terminal event
        total_ms = (time.monotonic() - request_start) * 1000
        logger.info(
            "request_id=%s done finish_reason=%s total_ms=%.1f",
            request_id, finish_reason, total_ms,
        )
        yield f"data: {json.dumps({'event': 'done', 'finish_reason': finish_reason})}\n\n"

    except asyncio.CancelledError:
        # Client disconnected — log and exit cleanly (do not yield — connection is gone)
        logger.info("request_id=%s client disconnected", request_id)
        return

    except asyncio.TimeoutError:
        logger.error("request_id=%s generation timeout after %.0fs", request_id, GENERATION_TIMEOUT_SECONDS)
        yield f"data: {json.dumps({'event': 'error', 'detail': 'generation timeout'})}\n\n"

    except Exception as exc:
        # Any other error after streaming started — send in-band, cannot change HTTP status
        logger.error(
            "request_id=%s stream error %s: %s",
            request_id, type(exc).__name__, exc, exc_info=True,
        )
        yield f"data: {json.dumps({'event': 'error', 'detail': str(exc)})}\n\n"


async def _iterate_stream(stream):
    """
    Helper: async iterator wrapper so asyncio.wait_for can apply to the whole loop.
    Yields chunks from the AsyncStream.
    """
    async for chunk in stream:
        yield chunk


# ---------------------------------------------------------------------------
# FastAPI endpoint
# ---------------------------------------------------------------------------

app = FastAPI()


class SummarizeRequest(BaseModel):
    document_text: str
    max_tokens: int = 300


@app.post("/summarize")
async def summarize(body: SummarizeRequest):
    """
    SSE streaming endpoint — yields summary tokens as they are generated.

    Response format: text/event-stream
    See stream_summary_chunks for event schema.
    """
    request_id = str(uuid.uuid4())[:8]
    return StreamingResponse(
        stream_summary_chunks(body.document_text, body.max_tokens, request_id),
        media_type="text/event-stream",
        headers={
            "Cache-Control": "no-cache",
            "X-Accel-Buffering": "no",     # disables nginx buffering — required for streaming
            "Connection": "keep-alive",
        },
    )
```

## §7 Interview Drill

### Q1 — Concept: Explain the difference between TTFT and total generation time to a senior engineer. Why does TTFT matter more for perceived UX?

**Model answer:**
TTFT is how long the user waits before seeing any text. Total generation time is how long the full response takes. Streaming doesn't change total generation time — the model does the same work at the same speed. What changes is TTFT: without streaming, TTFT equals total generation time. With streaming, TTFT drops to 300–500ms on most models. Users judge perceived speed by when text starts appearing, not when it ends. A 5-second total with a 400ms TTFT feels fast. A 3-second total with a 2-second blank screen feels slow. Optimize TTFT for perceived quality. Optimize total time for cost and capacity.

---

### Q2 — System Design: You have three endpoints — `/classify` (returns a category label), `/extract` (returns a 10-field JSON object), and `/summarize` (returns 200–400 tokens of prose). Which should use streaming, and what breaks if you apply streaming to the wrong one?

**Model answer:**
Only `/summarize`. `/classify` returns one or two tokens. Streaming a label adds SSE overhead with zero UX benefit — the first event arrives before the user can even see it. `/extract` returns JSON — you need the complete object before Pydantic can validate it. Streaming a partial JSON object means reassembling it server-side before parsing, adding latency and implementation complexity. The output never reaches the user incrementally anyway because the downstream code blocks on the full object. `/summarize` returns prose the user reads as it arrives. That is exactly the use case streaming solves. Getting it wrong on `/extract` means either broken validation (if you try to parse chunks) or rebuilding non-streaming behavior inside the streaming path — all the complexity with none of the benefit.

---

### Q3 — Debug: Your `/summarize` endpoint worked in development. After deploying behind nginx, users report text appears all at once after a 6-second wait — identical to the old non-streaming behavior. Your server logs show TTFT at 380ms per request. What is the cause and what is the fix?

**Model answer:**
The server is streaming correctly — TTFT 380ms confirms it. Nginx is buffering the entire SSE stream before sending it to the client. The connection reaches the browser only after generation completes. The missing header is `X-Accel-Buffering: no`. This disables nginx's proxy buffering for that specific response. `Cache-Control: no-cache` is also required to prevent caching layers from holding the response, and `Connection: keep-alive` keeps the TCP connection open during generation. The diagnostic signal for this exact problem is: TTFT in server logs is fast (correct), but the user experiences a blank screen for the full generation duration (nginx holding the buffer). The fix is one header line on the `StreamingResponse`.

## §10 Key Takeaways

- **Streaming doesn't make the model faster — it makes the wait invisible.** Total generation time is the same. TTFT is what changes. Users judge speed by when text starts appearing, not when it finishes. Optimize TTFT, not total time, for perceived UX quality.

- **`delta.content` is `None` on most chunks.** The first chunk carries only the role. The last few carry finish_reason and usage but no content. Always guard with `if delta_content` before using it. `"text" + None` raises `TypeError`.

- **`finish_reason` and last content arrive on separate chunks.** The last content chunk has `finish_reason=None`. The next chunk has `finish_reason="stop"` and `delta.content=None`. Check them independently.

- **Three headers are not optional.** `X-Accel-Buffering: no` is the critical one — miss it behind nginx and streaming silently becomes non-streaming. Your TTFT metric jumping to match total generation time is the diagnostic signal.

- **Use `AsyncAzureOpenAI` in async endpoints.** The sync client blocks the event loop. One streaming request blocks all other requests. In development with one user this is invisible. In production it is a throughput disaster.

- **Errors after the first byte must be in-band.** HTTP status code is committed once streaming starts. You cannot send a 500. Every post-start failure goes into a `{"event": "error", "detail": "..."}` SSE event. Never let the generator crash silently — the client gets a truncated stream with no signal about what happened.

- **Always send a terminal `done` event.** Without it, the client cannot distinguish "generation complete" from "server crashed mid-stream." The terminal event is what closes the loop on the client side cleanly.

## §12 Optional Advanced Notes

**Token counting on streaming calls:** `chunk.usage` is `None` on all chunks except the last, and only when you set `stream_options={"include_usage": True}`. The last chunk has `delta.content=None` and `finish_reason` but the usage object. Check `if chunk.usage is not None` separately from the content check to capture input and completion token counts for cost logging (→ 2.15).

**Streaming and tool calling:** Tool call results arrive in streaming mode as `delta.tool_calls` instead of `delta.content`. The chunk shape is different — you accumulate the tool call arguments string across multiple chunks, then parse and execute once the stream reaches `finish_reason="tool_calls"`. For the copilot's current endpoints, tool calling and streaming are on separate endpoints (tool calling on `/route`, streaming on `/summarize`), so this pattern is not needed here — but it matters in Stage 4 agentic endpoints.
