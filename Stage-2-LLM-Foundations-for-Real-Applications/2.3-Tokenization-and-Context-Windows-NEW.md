## Core Mental Model + Decision Rule

### Part A — Mental Model: The Unit Economics Frame

Your first internationalized user submits a support ticket in Thai. The same ticket in English: 420 tokens. In Thai: 1,100 tokens. Same problem. Same resolution time. But 2.6x the cost, and 2.6x the context space consumed — every single call.

That's the central problem this lesson solves: token counts are not obvious from raw text, and you cannot manage what you cannot measure.

Everything that matters operationally — cost, latency, capacity, limits — is denominated in tokens. Not words. Not characters. Not lines. The 128k context window is 128,000 tokens. API pricing is per 1,000 tokens. The `max_tokens` parameter you set is a token count. The budget you plan for each endpoint is a token budget.

The engineering consequence: you cannot manage a token budget by estimating from word or character count. You must measure. The `tokens.py` utility you build in this lesson is the measurement instrument for every other module in the project. Every cost number, every budget table, every "this fits" claim downstream depends on it being correct.

The context window is a hard limit. Exceeding it does not produce a warning or a truncated response — it fails the entire request with a 400 error. Enforcement must happen inside your application, before the API call, with a deliberate truncation or rejection strategy. "Hope we stay under the limit" is not a production system.

### Part B — Decision Rule

```
USE tiktoken measurement when:  you are assembling any prompt that includes variable
                                 content (user input, tool results, history)
AVOID guessing from word count: word count is directionally useful but wrong enough
                                 to cause real failures at scale — especially for
                                 non-English content and JSON payloads
SIGNAL that you need this:      if you are writing code that calls the LLM API
                                 and have not yet called count_tokens() on the
                                 assembled prompt, you are flying blind

USE tail truncation when:       ticket content is too long and front-loads the problem
AVOID middle truncation for:    ticket text — losing the middle is acceptable;
                                 losing the end of a description is also acceptable here
SIGNAL to truncate vs. reject:  if the user controls the input length (free-form text),
                                 truncate silently with a log warning;
                                 if a document feed controls it, reject and alert
```

## Main Concepts + Production Reality

### Concept 1 — How Tokenization Works

You write a function to estimate prompt length: `len(text.split())`. It looks reasonable. In testing, your prompts average 400 words and cost you about 530 tokens — close enough. Then a user submits a ticket that's 400 words of JSON-heavy log output. Your estimate says 400. The actual token count: 780. Budget enforcement passes. The API request fails with a 400 error.

Here's what happened:

1. Your text was split into sub-word pieces before the model ever saw it
2. JSON brackets, quotes, and colons each cost tokens — they're not absorbed into nearby words
3. Your word-count estimate couldn't know this — the split was done by the tokenizer, not by spaces
4. The model never saw raw characters — it saw a sequence of integer IDs, one per token
5. Those IDs are what the API counts for pricing, limits, and latency

**This process is tokenization.** The standard algorithm for OpenAI models is Byte Pair Encoding (BPE): start with individual characters, then repeatedly merge the most frequent adjacent pairs into new tokens. The result:

- Common words → single tokens (`"the"`, `"is"`, `"cat"`)
- Less common words → subword pieces (`"unbelievable"` → `["un", "believ", "able"]`)
- JSON structure characters → individual tokens (brackets, colons, quotes each cost one)

**Common trap:** Assuming word count is a reliable token estimate. It works for clean English prose but breaks badly for code, JSON, non-English text, and anything with special characters.

You do not need to know BPE in depth. What you need to know:

**Different models use different tokenizers.** Even within OpenAI's model family, tokenizers have changed:

```
cl100k_base   →   GPT-3.5-turbo, GPT-4, GPT-4-turbo
o200k_base    →   GPT-4o, GPT-4o-mini
```

The library you will use is `tiktoken` — OpenAI's tokenizer library. Always initialize it with the correct model name so you get the right encoding:

```python
import tiktoken

# Correct — model-specific encoding
enc = tiktoken.encoding_for_model("gpt-4o")

# Also valid — encoding by name if you know it
enc = tiktoken.get_encoding("o200k_base")

tokens = enc.encode("Hello, how can I help you today?")
# → list of integer IDs, e.g. [13225, 11, 1495, 649, 358, ...]
print(len(tokens))  # → 9
```

The integer IDs are what the model processes. The count of those IDs is what determines cost, latency, and whether you fit within the context window.

**For non-OpenAI models:** tiktoken only covers OpenAI models. Claude (Anthropic) uses a similar BPE-based tokenizer but with a different vocabulary. Gemini uses SentencePiece. For rough estimation across providers, the 4 characters-per-token rule of thumb works. For exact counts with non-OpenAI models, use the provider's own tokenizer library or API tokenization endpoint.

### Concept 2 — Token Rates by Content Type

You plan a 1,000-token budget for ticket content based on your testing with English tickets — 500-word average, roughly 670 tokens, with a comfortable margin. A new enterprise customer starts submitting tickets in Arabic. The first one: 480 words, 1,840 tokens. Budget enforcement fires on every single request. The budget table was correct for English. It was never validated for anything else.

The rate at which text converts to tokens varies significantly by content type — and planning around one content type then deploying across others is a common source of production surprises:

**Reference rates (approximate, English, GPT-4o tokenizer):**

```
Content type              Approx. rate          Notes
──────────────────────────────────────────────────────────────────
English prose             ~4 chars/token        750 words ≈ 1,000 tokens
                                                1 page (~500 words) ≈ 670 tokens

Python code               ~3.5 chars/token      Keywords are efficient;
                                                long identifiers split

JSON (structured)         ~3 chars/token        Brackets, quotes, colons,
                                                keys all cost tokens

Plain numbers             variable              "100" = 1 token
                                                "1,234,567" = 5 tokens
                                                Long floats can be 3-5 tokens

Non-English (CJK)         ~1.5 chars/token      Each CJK character ≈ 1 token
                                                (better than earlier models)

Non-English (Arabic,      ~1 char/token or      Can be 2-3x the token count
Thai, some SEA langs)     worse                 of equivalent English content

Whitespace/newlines       count as tokens       "\n\n" = 1-2 tokens
                                                Excessive whitespace adds up

Repeated patterns         efficient             BPE merges common sequences
```

**Why this matters for the copilot specifically:**

The `/extract` endpoint receives raw ticket text. Tickets submitted by non-English speakers will cost more tokens than equivalent English tickets — sometimes significantly more. If your budget is planned around English-language tickets, internationalization can push you over the limit.

JSON tool definitions (used in 2.8) cost tokens too — every function schema you pass to the API as part of `tools=` is part of the input token count. A single tool definition with a detailed description and parameter docs can be 150–300 tokens. Five tools = potentially 1,000+ tokens of overhead on every call.

**The practical test:** never assume. Before finalizing a prompt or system configuration, measure the actual token count using `count_tokens()`.

### Concept 3 — Token Budget Management

A ticket comes in. The user pasted an entire email thread — 4,000 words. Normal tickets average 400 tokens. This one: 5,300 tokens. Your system prompt is 600 tokens. Output reserve is 2,000. Total: 7,900 tokens — well within the 128k limit. Request succeeds.

Six months later, your `/chat` endpoint has been live for two weeks. A power user is on turn 18 of a conversation. History alone: 11,000 tokens. System prompt: 600. Current message: 400. Output reserve: 2,000. Total: 14,000 tokens — still fine. Turn 30: total hits 34,000 tokens. Turn 45:

```
openai.BadRequestError: Error code: 400
"This model's maximum context length is 128000 tokens.
 However, your messages resulted in 131423 tokens."
```

No warning. No partial response. The user's session crashes. Here's what went wrong:

1. Each `/chat` turn appends user + assistant messages to history
2. History grew by ~800 tokens per turn
3. No budget check enforced the limit before sending
4. The API never truncates — it rejects

**This is the core problem token budget management solves.** The budget model:

```
Context limit (model property)
    = Input budget + Output reserve

Input budget
    = System prompt tokens
    + Instructions tokens
    + Content tokens    (user input, documents)
    + Tool definitions  (each tool schema costs tokens — 150–300 per tool)
    + Tool results
    + Conversation history

Output reserve
    = max_tokens on the API call
    = Space you hold back for the model's response
```

Every token in one component is unavailable to another. If your system prompt grows from 500 to 1,000 tokens during a refactor, the content budget shrinks by 500 — silently, until a real request hits the new limit.

**The four enforcement practices:**

**1. Reserve upfront.** Decide the token allocation per component before assembling the prompt — not after. Fill each component within its allocation.

**2. Measure before sending.** Call `count_tokens()` on the assembled prompt before the API call. If it exceeds the budget, apply the truncation strategy. Never let it reach the API unchecked.

**3. Truncate deliberately.** When content exceeds its allocation:
- **Tail truncation:** keep the first N tokens — good for ticket text (problem is front-loaded)
- **Middle truncation:** keep first N/2 + last N/2 — preserves start and end, loses middle (→ 2.2: lost-in-middle already weakens middle content anyway)
- **Semantic chunking:** split into chunks, process each (→ Stage 3)
- **Rejection:** return a user-facing error asking them to shorten input

For the copilot at this stage: tail truncation for ticket content, rejection for anything structural (tool definitions, system prompts). Always log when truncation fires — that's your early warning signal.

**4. Reserve output tokens separately.** `max_tokens` is not a ceiling on cost — it's space subtracted from the context limit upfront. Always verify: `prompt_tokens + max_tokens ≤ context_limit`.

**What this means financially:**

Output tokens are 3–5x more expensive than input tokens:

```
GPT-4o (approximate):
  Input:  ~$2.50 per 1M tokens
  Output: ~$10.00 per 1M tokens

GPT-4o-mini:
  Input:  ~$0.15 per 1M tokens
  Output: ~$0.60 per 1M tokens
```

The classify endpoint: ~500 input tokens + ~10 output tokens. At 10,000 requests/day: $0.30/day on GPT-4o-mini vs. $3.00/day on GPT-4o. Budget management is the foundation of cost optimization (→ 2.15). You cannot optimize costs you have not measured.

**Common trap:** Only checking the total token count at request time, not per component. When the request fails, you know the total exceeded the limit — but you don't know which component is oversized. Track token counts per component from the start.

**Engineering implication:**
```
Without per-component tracking:   error fires → you don't know what to trim
With per-component tracking:      error fires → you know exactly which budget to adjust
Rule: log system_prompt_tokens, content_tokens, history_tokens separately, every call
```

## Project Integration

**Component:** `app/utils/tokens.py` — the token utility module that all endpoints depend on.

**What it implements:** Four functions covering everything the copilot needs for token management.

```python
# app/utils/tokens.py

import tiktoken
from dataclasses import dataclass


# ── Encoding cache ──────────────────────────────────────────────────────────
# tiktoken.encoding_for_model() is fast but creates a new object each call.
# Cache the encoder to avoid repeated initialization.

_ENCODING_CACHE: dict[str, tiktoken.Encoding] = {}

def _get_encoding(model: str) -> tiktoken.Encoding:
    if model not in _ENCODING_CACHE:
        try:
            _ENCODING_CACHE[model] = tiktoken.encoding_for_model(model)
        except KeyError:
            # Fallback for unknown models — use the most recent OpenAI encoding
            _ENCODING_CACHE[model] = tiktoken.get_encoding("o200k_base")
    return _ENCODING_CACHE[model]


# ── Core functions ───────────────────────────────────────────────────────────

def count_tokens(text: str, model: str = "gpt-4o") -> int:
    """
    Return the number of tokens in text for the given model's tokenizer.

    Use this before every API call to verify the prompt fits within budget.
    """
    if not text:
        return 0
    enc = _get_encoding(model)
    return len(enc.encode(text))


def count_messages_tokens(
    messages: list[dict],
    model: str = "gpt-4o",
) -> int:
    """
    Return the token count for a messages list as the API receives it.

    The API adds per-message overhead (~4 tokens per message for role/structure).
    This function accounts for that overhead so counts match what the API charges.
    """
    enc = _get_encoding(model)
    total = 0
    for message in messages:
        total += 4  # per-message overhead: role + structure tokens
        for value in message.values():
            total += len(enc.encode(str(value)))
    total += 2  # reply priming tokens (the model starts its response)
    return total


def fits_in_budget(text: str, budget: int, model: str = "gpt-4o") -> bool:
    """
    Return True if text fits within the given token budget.

    Use to check individual components (user content, tool results)
    before assembling the full prompt.
    """
    return count_tokens(text, model) <= budget


def truncate_to_token_budget(
    text: str,
    budget: int,
    model: str = "gpt-4o",
    truncation: str = "tail",
) -> tuple[str, bool]:
    """
    Truncate text to fit within the given token budget.

    Returns (truncated_text, was_truncated).
    Caller should log when was_truncated=True for monitoring.

    truncation options:
      "tail"   — keep the first `budget` tokens (default, good for ticket text)
      "middle" — keep first budget//2 + last budget//2 tokens
                 (preserves start and end, loses middle — 
                  note: middle content may already get less attention, → 2.2)
    """
    enc = _get_encoding(model)
    token_ids = enc.encode(text)

    if len(token_ids) <= budget:
        return text, False

    if truncation == "tail":
        truncated_ids = token_ids[:budget]
    elif truncation == "middle":
        half = budget // 2
        truncated_ids = token_ids[:half] + token_ids[-half:]
    else:
        raise ValueError(f"Unknown truncation strategy: {truncation!r}")

    return enc.decode(truncated_ids), True


# ── Budget planning ──────────────────────────────────────────────────────────

@dataclass
class TokenBudget:
    """
    Explicit token budget for one API call.

    Plan this before building the prompt — not after.
    Fill each component within its allocation.
    Validate before calling the API.

    Example:
        budget = TokenBudget(
            context_limit=128_000,
            output_reserve=1_000,
            system_prompt=600,
            instructions=200,
            content=800,
            tool_results=0,
        )
        assert budget.is_valid(), f"Budget exceeds context limit: {budget.total}"
    """
    context_limit: int
    output_reserve: int
    system_prompt: int
    instructions: int
    content: int
    tool_results: int = 0
    history: int = 0

    @property
    def input_total(self) -> int:
        return (
            self.system_prompt
            + self.instructions
            + self.content
            + self.tool_results
            + self.history
        )

    @property
    def total(self) -> int:
        return self.input_total + self.output_reserve

    @property
    def content_remaining(self) -> int:
        """How many tokens are available for additional content beyond planned."""
        return self.context_limit - self.total

    def is_valid(self) -> bool:
        return self.total <= self.context_limit

    def summary(self) -> str:
        return (
            f"Budget: system={self.system_prompt} | instr={self.instructions} | "
            f"content={self.content} | tools={self.tool_results} | "
            f"history={self.history} | output={self.output_reserve} | "
            f"total={self.total}/{self.context_limit}"
        )
```

**Connecting to config.py:**

The `Settings` class from 2.3's build step (defined in `app/config.py`) stores `context_limit` and `output_reserve`. The `TokenBudget` class takes those values at construction time:

## Interview Drill

**Q1 — Concept question:** "Why can't you estimate token count from word count, and why does it matter in production?"

Model answer: The tokenizer splits text into subword pieces using Byte Pair Encoding — not by words or spaces. Content type changes the rate dramatically: English prose runs ~4 chars per token, JSON and code run ~3, Arabic or Thai can hit 1 char per token or worse. In production, this matters because the context window is a hard limit — exceed it and the API returns a 400 error with no partial response. Budget planning based on word count will be correct for your test data and wrong for the first user who submits in a different format or language.

**Q2 — System design question:** "You're building three endpoints: one for ticket classification (short output, deterministic), one for ticket summarization (longer output), and one for a multi-turn support chat (grows with history). How do you allocate the token budget differently for each, and what breaks if you get it wrong?"

Key points to cover:
- Classification: small output reserve (20–50 tokens), most budget goes to content — wrong: wasting budget on output reserve inflates cost per call
- Summarization: larger output reserve (500–1000 tokens), content budget must account for full ticket — wrong: truncating too aggressively loses key details that affect summary quality
- Multi-turn chat: dynamic input budget — history grows each turn and must be measured and compressed before it hits the limit — wrong: no compression means session crash at turn N with no warning to the user
- All three: the total must always satisfy `prompt_tokens + max_tokens ≤ context_limit` — wrong: an unchecked accumulation causes 400 errors in production that are invisible in development

1. Add `count_tokens(ticket_text)` before the API call
2. If it exceeds the content budget, call `truncate_to_token_budget()` with tail truncation

## Key Takeaways (Section 10)

- **Count tokens — don't guess from word count.** "Hello world" is 2 tokens. A 400-word JSON payload can be 780 tokens. Word count feels close enough until the first non-English user breaks your budget. Always measure with tiktoken.

- **Token rates vary a lot by content type.** English prose: ~4 chars/token. JSON/code: ~3 chars/token. Arabic or Thai: can be 2–3× the token count of the same message in English. Build your budget tables with the content type your users will actually send.

- **Exceeding the context limit fails the whole request.** The API doesn't truncate for you. It returns a 400 error and no response. Your budget enforcement has to happen in your code, before the API call — not as error recovery after the fact.

- **`max_tokens` subtracts from your input budget.** The formula is `prompt_tokens + max_tokens ≤ context_limit`. Output tokens are also 3–5× more expensive than input tokens. Leave `max_tokens` unset and you're burning money on every call.

- **Plan the allocation before you fill it.** Decide upfront how many tokens each component gets — system prompt, instructions, content, history. Fill within those limits. If you fill first and check after, you'll discover the problem at 2 AM when a real user hits the edge case.

- **When content is too long, truncate deliberately.** Pick a strategy: tail truncation (keep the start), middle truncation (keep start and end), or rejection. Log whenever truncation fires — it tells you when real users are hitting your limits.

- **`tokens.py` is the foundation everything else measures from.** Every budget table, cost estimate, and "this fits" check in this project goes through this module. Get it right and test it thoroughly.

## Optional Advanced Notes (Section 12)

**Message overhead in chat APIs:** The OpenAI chat completions API adds approximately 4 tokens per message for role and structural framing (`{"role": "user", "content": ...}`), plus 2 tokens for reply priming. For a 5-turn conversation, this overhead is ~22 tokens — negligible for most use cases, but worth knowing when you're counting precisely at high context lengths. The `count_messages_tokens()` function in `tokens.py` accounts for this.

**tiktoken and non-OpenAI models:** tiktoken is accurate for OpenAI models only. For Claude (Anthropic), the character-to-token ratio is similar to GPT-4o (~4 chars/token for English) but not identical. Anthropic provides a token counting endpoint in their API. For budgeting purposes, a 10–15% buffer above tiktoken estimates provides reasonable coverage when switching providers.

**Streaming and token counts:** When using streaming responses (→ 2.9), the full output token count is not known until the stream completes. For cost logging with streaming, accumulate the chunk count during the stream, or use the `usage` object in the final chunk if the provider returns it. The `TokenBudget` planning approach remains the same — you are reserving output tokens upfront regardless of whether you use streaming.
