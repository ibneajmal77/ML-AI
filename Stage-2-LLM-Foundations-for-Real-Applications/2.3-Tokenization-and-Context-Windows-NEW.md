## Core Mental Model + Decision Rule

### Part A — Mental Model

Imagine you get your first user who writes support tickets in Thai. You check their first ticket — 480 words, the same length as a typical English ticket. But the token count is 1,840 instead of 670. Your budget was sized for English. Every Thai ticket blows it.

That's the core problem: **you can't guess token counts from word count or character count.** The only way to know is to measure.

Here's why: everything you pay for and every limit you run into is measured in tokens — not words, not characters. The 128k context window means 128,000 tokens. API pricing is per 1,000 tokens. The `max_tokens` setting on every API call is a token count. Your entire budget is a token budget.

And the context window is a hard cutoff. Go over it and the API fails the whole request with a 400 error. There's no "it'll just truncate a bit." It fails. Your code needs to catch this before the request goes out — not after.

**`tokens.py` is the tool that lets you measure instead of guess.** Every cost estimate, every "will this fit?" check, and every budget enforcement in the project goes through it.

### Part B — Decision Rule

**Always measure before sending.** Any time your prompt includes variable content — user input, tool results, conversation history — call `count_tokens()` before the API call. If you haven't done that, you're guessing.

**Truncate or reject when content is too long.** If the ticket text is over budget:
- Truncate from the tail (keep the first N tokens) — works well for tickets because the problem description is usually at the top
- Reject and ask the user to shorten — better when the input comes from an automated feed, not a human

**Don't use word count as a proxy.** It's close enough for clean English prose, but it breaks on JSON, code, and non-English text. Use it for rough sanity checks only — never for actual enforcement.

## Main Concepts + Production Reality

### Concept 1 — How Tokenization Works

You write a quick helper to estimate prompt length: `len(text.split())`. In testing it works fine — your prompts average 400 words and cost around 530 tokens. Close enough.

Then a user submits a 400-word ticket that's mostly JSON log output. You estimate 400 tokens. The real count: 780. Your budget check passes. The API call fails with a 400 error.

Here's what went wrong:

1. Before the model reads your text, it gets split into small pieces called **tokens**
2. JSON brackets, quotes, colons — each one is a separate token. They don't get merged into nearby words
3. Your word-count estimate had no way to know this. It just counted spaces
4. The model never sees raw text — it sees a list of integer IDs, one per token
5. The API counts those IDs to determine cost, speed, and whether you're within the limit

**This splitting process is called tokenization.** OpenAI models use an algorithm called Byte Pair Encoding (BPE). It starts with individual characters and repeatedly merges the most common pairs into single tokens:
- Common words become one token: `"the"`, `"is"`, `"cat"`
- Less common words split into pieces: `"unbelievable"` → `["un", "believ", "able"]`
- JSON special characters each get their own token: `{`, `"`, `:`, `}`

**You don't need to understand BPE deeply.** What matters:

Different model versions use different tokenizers:
```
cl100k_base  →  GPT-3.5-turbo, GPT-4, GPT-4-turbo
o200k_base   →  GPT-4o, GPT-4o-mini
```

The library is called `tiktoken`. Always use it with the right model name:

```python
import tiktoken

enc = tiktoken.encoding_for_model("gpt-4o")

tokens = enc.encode("Hello, how can I help you today?")
# → [13225, 11, 1495, 649, 358, ...]
print(len(tokens))  # → 9
```

Those integers are what the model processes. The count of them is what determines your cost and whether you fit in the context window.

**Common trap:** Using word count to estimate tokens. It works for English prose but breaks badly on JSON, code, and non-English text. Measure, don't estimate.

### Concept 2 — Token Rates by Content Type

Your budget table says each ticket gets up to 1,000 tokens. You built and tested it entirely with English tickets. Then an enterprise customer starts sending Arabic tickets. First one: 480 words, 1,840 tokens. Budget check fires on every single request.

The budget table was right for English. It was never tested for anything else.

Different types of content convert to tokens at very different rates:

```
Content type          Rate              Example
─────────────────────────────────────────────────────────────────
English prose         ~4 chars/token    500 words ≈ 670 tokens
Python / code         ~3.5 chars/token  Keywords tokenize well;
                                        long variable names split
JSON (structured)     ~3 chars/token    Every bracket, quote,
                                        and colon costs a token
Numbers               varies            "100" = 1 token
                                        "1,234,567" = 5 tokens
Chinese / Japanese    ~1.5 chars/token  Each character ≈ 1 token
Arabic / Thai         ~1 char/token     Can be 2–3x more tokens
                      or worse          than the same text in English
Whitespace / newlines count too         "\n\n" = 1–2 tokens
```

**Two things this affects in the copilot:**

First: the `/extract` endpoint takes raw ticket text. If your users write in multiple languages, your token budget needs to account for the worst case, not just English.

Second: tool definitions (used in 2.8) cost tokens. Every function schema you pass in `tools=` is part of your input token count. One detailed tool definition: 150–300 tokens. Five tools: potentially 1,000+ tokens added to every call.

**The rule:** before finalizing any prompt or configuration, measure the actual token count with `count_tokens()`. Don't assume.

### Concept 3 — Token Budget Management

A long ticket comes in — a user pasted an entire email thread, 4,000 words, 5,300 tokens. Your system prompt is 600 tokens. Output reserve is 2,000. Total: 7,900 tokens. Well within 128k. Request works fine.

Six months later, your `/chat` endpoint is live. A user has been chatting for 45 turns. Each turn added roughly 800 tokens of history. By turn 45, history alone is 35,000 tokens. System prompt: 600. Current message: 400. Output reserve: 2,000. Total: 38,000 tokens — still fine. Turn 90:

```
openai.BadRequestError: Error code: 400
"This model's maximum context length is 128000 tokens.
 However, your messages resulted in 131423 tokens."
```

No warning. No partial response. The user's session crashes.

What went wrong:
1. Each turn added messages to history
2. History grew by ~800 tokens per turn with no ceiling
3. Nobody checked the total before sending the request
4. The API doesn't truncate — it just rejects

**Token budget management is how you prevent this.** The budget for any API call looks like this:

```
Context limit  =  Input budget  +  Output reserve

Input budget   =  system prompt
               +  instructions
               +  content (ticket text, documents)
               +  tool definitions (150–300 tokens each)
               +  tool results
               +  conversation history

Output reserve =  max_tokens   (the space you hold back for the model's reply)
```

Every token you spend in one part is unavailable to another. If your system prompt grows from 500 to 1,000 tokens after a refactor, your content budget shrinks by 500 — silently, until a real user hits it.

**Four things to do:**

**1. Plan the budget before filling it.** Decide how many tokens each component gets before you assemble the prompt. Don't fill first and check after — you'll discover the edge case at the worst moment.

**2. Measure before sending.** Call `count_tokens()` on the assembled prompt before making the API call. If it's over budget, apply your truncation strategy. Never let it reach the API unchecked.

**3. When content is too long, truncate with a plan:**
- **Tail truncation** — keep the first N tokens. Good for ticket text because the problem is usually described at the top.
- **Middle truncation** — keep the first N/2 tokens and the last N/2. Preserves the opening and the end, loses the middle. (The model already pays less attention to the middle anyway → 2.2.)
- **Rejection** — tell the user their input is too long and ask them to shorten it.

For tickets: use tail truncation and log a warning every time it fires. That log tells you when real users are hitting your limits.

**4. Account for output tokens in the budget.** `max_tokens` isn't just a cap on output length — it's space you reserve from the context limit upfront. Always check: `prompt_tokens + max_tokens ≤ context_limit`.

**What this costs financially:**

Output tokens are 3–5x more expensive than input tokens:

```
GPT-4o:      Input ~$2.50/1M tokens,  Output ~$10.00/1M tokens
GPT-4o-mini: Input ~$0.15/1M tokens,  Output ~$0.60/1M tokens
```

At 10,000 requests/day, GPT-4o costs about 10x more than GPT-4o-mini. Budget management is the foundation for cost optimization later (→ 2.15). You can't optimize what you haven't measured.

**Common trap:** Only logging the total token count, not the breakdown by component. When a request fails, you know the total was too high — but you don't know which part caused it. Log `system_prompt_tokens`, `content_tokens`, and `history_tokens` separately from day one.

**What you're building:** `app/utils/tokens.py` — the token counting module every endpoint imports.

```python
# app/utils/tokens.py

import tiktoken
from dataclasses import dataclass


# Cache the encoder so we don't recreate it on every call
_ENCODING_CACHE: dict[str, tiktoken.Encoding] = {}

def _get_encoding(model: str) -> tiktoken.Encoding:
    if model not in _ENCODING_CACHE:
        try:
            _ENCODING_CACHE[model] = tiktoken.encoding_for_model(model)
        except KeyError:
            # Unknown model — fall back to the latest OpenAI encoding
            _ENCODING_CACHE[model] = tiktoken.get_encoding("o200k_base")
    return _ENCODING_CACHE[model]


def count_tokens(text: str, model: str = "gpt-4o") -> int:
    """Return the token count for text using the correct tokenizer for that model."""
    if not text:
        return 0
    enc = _get_encoding(model)
    return len(enc.encode(text))


def count_messages_tokens(messages: list[dict], model: str = "gpt-4o") -> int:
    """
    Return the token count for a messages list, including per-message overhead.

    The API adds ~4 tokens per message for role and structure, plus 2 tokens
    for reply priming. This function accounts for that so your count matches
    what the API actually charges.
    """
    enc = _get_encoding(model)
    total = 0
    for message in messages:
        total += 4  # per-message overhead
        for value in message.values():
            total += len(enc.encode(str(value)))
    total += 2  # reply priming
    return total


def fits_in_budget(text: str, budget: int, model: str = "gpt-4o") -> bool:
    """Return True if the text fits within the given token budget."""
    return count_tokens(text, model) <= budget


def truncate_to_token_budget(
    text: str,
    budget: int,
    model: str = "gpt-4o",
    truncation: str = "tail",
) -> tuple[str, bool]:
    """
    Cut text down to fit within the given token budget.

    Returns (truncated_text, was_truncated).
    Log when was_truncated=True — it means a real user hit your limit.

    truncation:
      "tail"   — keep the first `budget` tokens (best for ticket text)
      "middle" — keep first half + last half (preserves start and end)
    """
    enc = _get_encoding(model)
    token_ids = enc.encode(text)

    if len(token_ids) <= budget:
        return text, False

    if truncation == "tail":
        truncated_ids = token_ids[:budget]
    elif truncation == "middle":
        half = budget // 2
        truncated_ids = token_ids[:half] + token_ids[-half:]
    else:
        raise ValueError(f"Unknown truncation strategy: {truncation!r}")

    return enc.decode(truncated_ids), True


@dataclass
class TokenBudget:
    """
    Plan the token budget for one API call before you assemble the prompt.

    Fill each component within its allocation.
    Call is_valid() before sending — if it returns False, something is over budget.

    Example:
        budget = TokenBudget(
            context_limit=128_000,
            output_reserve=1_000,
            system_prompt=600,
            instructions=200,
            content=800,
        )
        assert budget.is_valid(), f"Over budget: {budget.total} tokens"
    """
    context_limit: int
    output_reserve: int
    system_prompt: int
    instructions: int
    content: int
    tool_results: int = 0
    history: int = 0

    @property
    def input_total(self) -> int:
        return (
            self.system_prompt
            + self.instructions
            + self.content
            + self.tool_results
            + self.history
        )

    @property
    def total(self) -> int:
        return self.input_total + self.output_reserve

    @property
    def content_remaining(self) -> int:
        """How many more tokens are available beyond what's already planned."""
        return self.context_limit - self.total

    def is_valid(self) -> bool:
        return self.total <= self.context_limit

    def summary(self) -> str:
        return (
            f"system={self.system_prompt} | instructions={self.instructions} | "
            f"content={self.content} | tools={self.tool_results} | "
            f"history={self.history} | output={self.output_reserve} | "
            f"total={self.total}/{self.context_limit}"
        )
```

**How endpoints will use this:**

```python

budget = TokenBudget(
```

## Interview Drill

**Q1 — Concept question:** "Why can't you just use word count to estimate tokens, and why does it matter in production?"

Model answer: The tokenizer splits text into sub-word pieces — not by spaces. A JSON bracket costs a token. An Arabic word might cost 4–5 tokens while the same word in English costs 1. Word count is close enough for clean English prose but wrong enough to cause real failures once your users send anything else. In production it matters because the context window is a hard limit — go over it and the API returns a 400 error with no partial response, while your users see a crash.

---

**Q2 — System design question:** "You have three endpoints: `/classify` (short, deterministic output), `/summarize` (a paragraph of output), and `/chat` (multi-turn, history grows over time). How do you plan the token budget differently for each?"

What a strong answer covers:
- `/classify`: small output reserve (10–20 tokens), most of the budget goes to the system prompt and ticket content. Risk: wrong if content is longer than tested (e.g. pasted email threads).
- `/summarize`: larger output reserve (100–200 tokens), content budget sized for full ticket. Risk: truncating the ticket too aggressively loses the details the summary needs.
- `/chat`: history grows each turn and must be measured and capped before sending. Without this, the session eventually crashes at turn N with no warning. Cover `compress_history()` from 2.17.
- All three: `prompt_tokens + max_tokens ≤ context_limit`, checked in code, not handled as an exception.

---

**Q3 — Debugging question:** "Your `/classify` endpoint works fine in testing but fails on about 8% of production requests with a 400 error. Logs show: failing requests have ~4,200 input tokens. Passing requests average ~480 input tokens. What's the cause and how do you fix it?"

Diagnosis:
- The 8% are outlier-length inputs — users pasting long documents or email threads
- Budget enforcement isn't happening before the API call, or the budget threshold is too high
- The content budget was sized for average tickets, not worst-case inputs

Fix:
1. Add `count_tokens(ticket_text)` before the API call
2. If it exceeds the content budget, call `truncate_to_token_budget(ticket_text, budget.content)`
3. Log every time truncation fires — the 8% that were crashing should now be truncated instead
4. Check if you want to notify users that their input was shortened

## Key Takeaways

- **Count tokens, don't guess from word count.** English prose is roughly 4 chars per token, but JSON, code, and non-English text all tokenize differently. A 400-word JSON payload can easily be 780 tokens. Measure with tiktoken.

- **Token rates vary a lot by content type.** English ~4 chars/token. JSON/code ~3. Arabic or Thai can be 2–3× the tokens of the same content in English. Build your budgets around the content your users will actually send.

- **Exceeding the context limit fails the whole request.** No graceful degradation. No truncation. A 400 error. Your enforcement needs to happen in your code before the API call.

- **`max_tokens` subtracts from your input budget.** The formula: `prompt_tokens + max_tokens ≤ context_limit`. Output tokens also cost 3–5× more than input tokens — leave `max_tokens` unset and you're burning money.

- **Plan the budget before you fill it.** Decide how many tokens each component gets first — system prompt, instructions, content, history. Filling first and checking after is how you discover the edge case at the worst time.

- **When content is too long, truncate with a strategy.** Tail truncation keeps the start (good for tickets). Middle truncation keeps the start and end. Rejection tells the user to shorten. Pick one, implement it, and log every time it fires.

- **`tokens.py` is the foundation everything else relies on.** Every cost estimate, every budget check, every "will this fit?" call in this project goes through this module. Test it carefully.

## Optional Advanced Notes

**Message overhead in chat APIs:** The OpenAI chat API adds about 4 tokens per message for role and structure, plus 2 tokens for reply priming. For a 5-turn conversation, that's ~22 tokens of overhead — small but worth knowing if you're counting precisely near the limit. The `count_messages_tokens()` function in `tokens.py` already accounts for this.

**tiktoken only covers OpenAI models.** For Claude (Anthropic), the rate is similar (~4 chars/token for English) but not identical. Anthropic has a token counting API endpoint. For rough estimates when switching providers, add a 10–15% buffer on top of tiktoken counts.

**Streaming and token counts:** When using streaming responses (→ 2.9), you don't know the final output token count until the stream finishes. For cost logging during streaming, either count chunks as they arrive or use the `usage` field in the final chunk if the provider sends it. The budget planning works the same way — you still reserve output tokens upfront.
