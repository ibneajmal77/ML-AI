## Core Mental Model

### The Unit Economics Frame

Every LLM API interaction has a unit: the token. Not words. Not characters. Not lines. Tokens.

Everything that matters operationally — cost, latency, capacity, limits — is denominated in tokens. A 128k context window is 128,000 tokens. API pricing is per 1,000 tokens. Latency scales with tokens generated. The budget you plan for each endpoint is a token budget.

The problem is that token counts are not obvious from raw text. "Hello" is 1 token. "unbelievable" is 3 tokens. A 500-word English email is roughly 650 tokens. The same email in Thai might be 2,000 tokens. A block of JSON with nested keys and quoted values tokenizes very differently than prose of equivalent character length.

**The engineering consequence:** you cannot manage a token budget by estimating from character or word count. You must measure. The utility you build in this lesson is the measurement instrument for every other piece of the project.

The context window is a hard limit. When you exceed it, the API does not truncate gracefully — it fails the entire request with a 400 error. Budget enforcement must happen inside your application, before the API call, with deliberate truncation logic when needed. "Hope we stay under the limit" is not a strategy.

## Main Concepts + Production Reality

### Concept 1 — How Tokenization Works

Tokenization is the process of splitting input text into tokens before it enters the model. The model does not see raw characters — it sees a sequence of integer IDs, each corresponding to a token in the model's vocabulary.

The standard algorithm used by OpenAI models (and most modern LLMs) is **Byte Pair Encoding (BPE)**. BPE works by starting with individual characters, then repeatedly merging the most frequent pairs of symbols into new tokens. The result is a vocabulary where:

- Common words are single tokens (`"the"`, `"is"`, `"cat"`)
- Less common words split into subword pieces (`"unbelievable"` → `["un", "believ", "able"]`)
- Very rare character sequences fall back to individual bytes

You do not need to know BPE in depth to work with it. What you need to know:

**Different models use different tokenizers.** Even within OpenAI's model family, tokenizers have changed:

```
cl100k_base   →   GPT-3.5-turbo, GPT-4, GPT-4-turbo
o200k_base    →   GPT-4o, GPT-4o-mini
```

The library you will use is `tiktoken` — OpenAI's tokenizer library. Always initialize it with the correct model name so you get the right encoding:

```python
import tiktoken

# Correct — model-specific encoding
enc = tiktoken.encoding_for_model("gpt-4o")

# Also valid — encoding by name if you know it
enc = tiktoken.get_encoding("o200k_base")

tokens = enc.encode("Hello, how can I help you today?")
# → list of integer IDs, e.g. [13225, 11, 1495, 649, 358, ...]
print(len(tokens))  # → 9
```

The integer IDs are what the model processes. The count of those IDs is what determines cost, latency, and whether you fit within the context window.

**For non-OpenAI models:** tiktoken only covers OpenAI models. Claude (Anthropic) uses a similar BPE-based tokenizer but with a different vocabulary. Gemini uses SentencePiece. For rough estimation across providers, the 4 characters-per-token rule of thumb works. For exact counts with non-OpenAI models, use the provider's own tokenizer library or API tokenization endpoint.

### Concept 2 — Token Rates by Content Type

Token counts are not proportional to character or word count in a uniform way. The rate varies significantly by content type, and this directly affects how you plan budgets.

**Reference rates (approximate, English, GPT-4o tokenizer):**

```
Content type              Approx. rate          Notes
──────────────────────────────────────────────────────────────────
English prose             ~4 chars/token        750 words ≈ 1,000 tokens
                                                1 page (~500 words) ≈ 670 tokens

Python code               ~3.5 chars/token      Keywords are efficient;
                                                long identifiers split

JSON (structured)         ~3 chars/token        Brackets, quotes, colons,
                                                keys all cost tokens

Plain numbers             variable              "100" = 1 token
                                                "1,234,567" = 5 tokens
                                                Long floats can be 3-5 tokens

Non-English (CJK)         ~1.5 chars/token      Each CJK character ≈ 1 token
                                                (better than earlier models)

Non-English (Arabic,      ~1 char/token or      Can be 2-3x the token count
Thai, some SEA langs)     worse                 of equivalent English content

Whitespace/newlines       count as tokens       "\n\n" = 1-2 tokens
                                                Excessive whitespace adds up

Repeated patterns         efficient             BPE merges common sequences
```

**Why this matters for the copilot specifically:**

The `/extract` endpoint receives raw ticket text. Tickets submitted by non-English speakers will cost more tokens than equivalent English tickets — sometimes significantly more. If your budget is planned around English-language tickets, internationalization can push you over the limit.

JSON tool definitions (used in 2.8) cost tokens too — every function schema you pass to the API as part of `tools=` is part of the input token count. A single tool definition with a detailed description and parameter docs can be 150–300 tokens. Five tools = potentially 1,000+ tokens of overhead on every call.

**The practical test:** never assume. Before finalizing a prompt or system configuration, measure the actual token count using `count_tokens()`.

### Concept 3 — What Happens at the Context Limit

When input tokens + output token reserve exceed the model's context limit, the API returns an error. Not a warning. Not a truncated response. A failure.

```python
# What you get when you exceed the context limit (OpenAI)
openai.BadRequestError: Error code: 400
  "This model's maximum context length is 128000 tokens. 
   However, your messages resulted in 131423 tokens (131023 in the messages, 
   400 in the completion). Please reduce the length of the messages."
```

The error message is accurate but not actionable at the moment it fires — you're already in the middle of a request handler, and the user is waiting. The right place to catch this is before the API call, in your own code.

**What the error message tells you:**
- Total input token count for that request
- The max_tokens (output reserve) you requested
- Combined total that exceeded the limit

**What it does not tell you:**
- Which component of your prompt is oversized
- How to fix it

That's your job — and it requires knowing the token breakdown of each component (system prompt, instructions, content, tool results, history) independently. If you only know the total when it fails, you cannot fix it systematically.

**The three failure scenarios you will encounter:**

```
Scenario 1 — Long user input
  Normal tickets: 200–500 tokens
  A user pastes an entire email thread: 4,000 tokens
  Your budget assumed 500 → request fails

Scenario 2 — Context accumulation in multi-turn
  Each /chat turn adds user + assistant messages to history
  By turn 15, history alone is 8,000 tokens
  Combined with system prompt + current message → over limit
  (→ 2.17 covers the management strategy)

Scenario 3 — Tool results are larger than expected
  ticket_lookup returns a ticket with long resolution history
  Tool result: 2,000 tokens instead of estimated 800
  Combined context → over limit
```

In all three cases, the fix is the same: enforce the budget in your code before the API call, with a defined truncation or rejection strategy for when the budget is exceeded.

### Concept 4 — Token Budget Management

Budget management is the practice of planning, measuring, and enforcing token allocations for every API call before it is made.

**The budget model:**

```
Total context limit (model property)
    = Input budget + Output reserve

Input budget
    = System prompt tokens
    + Instructions tokens
    + Content tokens (user input, documents)
    + Tool definition tokens
    + Tool result tokens
    + Conversation history tokens

Output reserve
    = max_tokens you set on the API call
    = The space you're holding for the model's response
```

Every token spent on one component is unavailable for another. If your system prompt grows from 500 to 1,000 tokens, you either shrink another component or increase the total budget — which means fewer tokens remain for content and history.

**The four practices:**

**1. Reserve upfront.** Before assembling the prompt, decide the token allocation for each component. Do not fill components and then check if it fits — plan the allocation first, then fill within the allocation.

**2. Measure before sending.** Call `count_tokens()` on the assembled prompt before calling the API. If it exceeds the budget, apply the truncation strategy — never let it reach the API unchecked.

**3. Truncate deliberately.** When user content exceeds its budget allocation:
- Tail truncation: take the first N tokens of the content (loses the end)
- Middle truncation: take the first N/2 and last N/2 tokens (loses the middle — but see 2.2 on lost-in-middle)
- Semantic chunking: split into chunks, process each (→ Stage 3 for full patterns)
- Rejection: return an error to the user asking them to shorten input

For the copilot at this stage, tail truncation is acceptable for ticket content — tickets front-load the problem description. Apply it with a logged warning so you can monitor how often it fires.

**4. Reserve output tokens separately.** The output reserve (`max_tokens` parameter) is not a suggestion — it's the space you hold back from the context limit. If your prompt uses 126,000 tokens and you request `max_tokens=4000`, the total is 130,000 — which exceeds a 128k limit. Always calculate: `total = prompt_tokens + max_tokens ≤ context_limit`.

**Cost structure — what budget management means financially:**

Most providers charge separately for input and output tokens. Output tokens are 3–5x more expensive per token than input tokens on most models:

```
GPT-4o (approximate, subject to change):
  Input:  ~$2.50 per 1M tokens
  Output: ~$10.00 per 1M tokens

GPT-4o-mini:
  Input:  ~$0.15 per 1M tokens
  Output: ~$0.60 per 1M tokens
```

For the copilot's classification endpoint, a typical request might cost ~0.003 cents with GPT-4o-mini or ~0.03 cents with GPT-4o. At 10,000 requests/day, that's $0.30 vs. $3.00 per day — a 10x cost difference. Budget management is the foundation of cost optimization (→ 2.15). You cannot optimize costs you have not measured.

## Project Integration

**Component:** `app/utils/tokens.py` — the token utility module that all endpoints depend on.

**What it implements:** Four functions covering everything the copilot needs for token management.

```python
# app/utils/tokens.py

import tiktoken
from dataclasses import dataclass


# ── Encoding cache ──────────────────────────────────────────────────────────
# tiktoken.encoding_for_model() is fast but creates a new object each call.
# Cache the encoder to avoid repeated initialization.

_ENCODING_CACHE: dict[str, tiktoken.Encoding] = {}

def _get_encoding(model: str) -> tiktoken.Encoding:
    if model not in _ENCODING_CACHE:
        try:
            _ENCODING_CACHE[model] = tiktoken.encoding_for_model(model)
        except KeyError:
            # Fallback for unknown models — use the most recent OpenAI encoding
            _ENCODING_CACHE[model] = tiktoken.get_encoding("o200k_base")
    return _ENCODING_CACHE[model]


# ── Core functions ───────────────────────────────────────────────────────────

def count_tokens(text: str, model: str = "gpt-4o") -> int:
    """
    Return the number of tokens in text for the given model's tokenizer.

    Use this before every API call to verify the prompt fits within budget.
    """
    if not text:
        return 0
    enc = _get_encoding(model)
    return len(enc.encode(text))


def count_messages_tokens(
    messages: list[dict],
    model: str = "gpt-4o",
) -> int:
    """
    Return the token count for a messages list as the API receives it.

    The API adds per-message overhead (~4 tokens per message for role/structure).
    This function accounts for that overhead so counts match what the API charges.
    """
    enc = _get_encoding(model)
    total = 0
    for message in messages:
        total += 4  # per-message overhead: role + structure tokens
        for value in message.values():
            total += len(enc.encode(str(value)))
    total += 2  # reply priming tokens (the model starts its response)
    return total


def fits_in_budget(text: str, budget: int, model: str = "gpt-4o") -> bool:
    """
    Return True if text fits within the given token budget.

    Use to check individual components (user content, tool results)
    before assembling the full prompt.
    """
    return count_tokens(text, model) <= budget


def truncate_to_token_budget(
    text: str,
    budget: int,
    model: str = "gpt-4o",
    truncation: str = "tail",
) -> tuple[str, bool]:
    """
    Truncate text to fit within the given token budget.

    Returns (truncated_text, was_truncated).
    Caller should log when was_truncated=True for monitoring.

    truncation options:
      "tail"   — keep the first `budget` tokens (default, good for ticket text)
      "middle" — keep first budget//2 + last budget//2 tokens
                 (preserves start and end, loses middle — 
                  note: middle content may already get less attention, → 2.2)
    """
    enc = _get_encoding(model)
    token_ids = enc.encode(text)

    if len(token_ids) <= budget:
        return text, False

    if truncation == "tail":
        truncated_ids = token_ids[:budget]
    elif truncation == "middle":
        half = budget // 2
        truncated_ids = token_ids[:half] + token_ids[-half:]
    else:
        raise ValueError(f"Unknown truncation strategy: {truncation!r}")

    return enc.decode(truncated_ids), True


# ── Budget planning ──────────────────────────────────────────────────────────

@dataclass
class TokenBudget:
    """
    Explicit token budget for one API call.

    Plan this before building the prompt — not after.
    Fill each component within its allocation.
    Validate before calling the API.

    Example:
        budget = TokenBudget(
            context_limit=128_000,
            output_reserve=1_000,
            system_prompt=600,
            instructions=200,
            content=800,
            tool_results=0,
        )
        assert budget.is_valid(), f"Budget exceeds context limit: {budget.total}"
    """
    context_limit: int
    output_reserve: int
    system_prompt: int
    instructions: int
    content: int
    tool_results: int = 0
    history: int = 0

    @property
    def input_total(self) -> int:
        return (
            self.system_prompt
            + self.instructions
            + self.content
            + self.tool_results
            + self.history
        )

    @property
    def total(self) -> int:
        return self.input_total + self.output_reserve

    @property
    def content_remaining(self) -> int:
        """How many tokens are available for additional content beyond planned."""
        return self.context_limit - self.total

    def is_valid(self) -> bool:
        return self.total <= self.context_limit

    def summary(self) -> str:
        return (
            f"Budget: system={self.system_prompt} | instr={self.instructions} | "
            f"content={self.content} | tools={self.tool_results} | "
            f"history={self.history} | output={self.output_reserve} | "
            f"total={self.total}/{self.context_limit}"
        )
```

**Connecting to config.py:**

The `Settings` class from 2.3's build step (defined in `app/config.py`) stores `context_limit` and `output_reserve`. The `TokenBudget` class takes those values at construction time:

```python

    context_limit=settings.context_limit,      # 128_000

)

    ticket_text, truncated = truncate_to_token_budget(ticket_text, budget.content)
    if truncated:
        logger.warning("Ticket text truncated", extra={"original_tokens": ticket_tokens})
```

## Key Takeaways

- **The unit of everything is the token, not the word.** Cost, latency, and context limits are all in tokens. Estimate by content type; always measure with tiktoken before relying on an estimate.

- **Token rates vary significantly by content type.** English prose: ~4 chars/token. JSON and code: ~3 chars/token. Non-English languages: can be 2–3x the token count of equivalent English. Never assume — measure.

- **The context window is a hard limit.** Exceeding it fails the entire request with a 400 error. No graceful degradation. Enforcement must happen in your code before the API call — not in an exception handler.

- **Output tokens are not free.** The output reserve is subtracted from the context limit before you even have space for input. Always calculate: input + max_tokens ≤ context_limit. And output tokens are 3–5x more expensive than input tokens — unconstrained `max_tokens` burns money on every call.

- **Plan the budget before filling it.** Decide the token allocation for each component — system prompt, instructions, content, tool results, history — before assembling the prompt. Filling first and checking after is how you get surprised at 2am by a request that exceeds the limit.

- **Truncation must be deliberate.** When content exceeds its budget allocation, have an explicit strategy: tail truncation, middle truncation, or rejection with a user-facing message. Letting the API truncate or fail without a defined strategy is not a production system.

- **`tokens.py` is the measurement instrument for the entire project.** Every cost number, every budget table, every "this fits" claim depends on this utility being correct. Test it thoroughly — it is foundational.

## Optional Advanced Notes

**Message overhead in chat APIs:** The OpenAI chat completions API adds approximately 4 tokens per message for role and structural framing (`{"role": "user", "content": ...}`), plus 2 tokens for reply priming. For a 5-turn conversation, this overhead is ~22 tokens — negligible for most use cases, but worth knowing when you're counting precisely at high context lengths. The `count_messages_tokens()` function in `tokens.py` accounts for this.

**tiktoken and non-OpenAI models:** tiktoken is accurate for OpenAI models only. For Claude (Anthropic), the character-to-token ratio is similar to GPT-4o (~4 chars/token for English) but not identical. Anthropic provides a token counting endpoint in their API. For budgeting purposes, a 10–15% buffer above tiktoken estimates provides reasonable coverage when switching providers.

**Streaming and token counts:** When using streaming responses (→ 2.9), the full output token count is not known until the stream completes. For cost logging with streaming, accumulate the chunk count during the stream, or use the `usage` object in the final chunk if the provider returns it. The `TokenBudget` planning approach remains the same — you are reserving output tokens upfront regardless of whether you use streaming.
