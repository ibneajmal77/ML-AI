## CONCEPT 1: Embeddings — How the Model Converts Text Into Something It Can Reason About
<!-- cognitive job: establish that meaning is represented as vectors and that semantic similarity is a geometric property — foundation for understanding retrieval, search, and why models can generalize -->

| Dimension | Answer |
|---|---|
| **What it is** | An embedding is a list of numbers — a vector — that represents the meaning of a token in a high-dimensional space where semantically similar tokens produce geometrically similar vectors |
| **Why it matters** | Embeddings are how the model converts raw text into a form it can compute with — every downstream operation in the transformer (attention, layer transformations, output prediction) operates on these vectors, not on words |
| **When to use it** | Use this mental model when reasoning about semantic search, retrieval quality, why embedding distance is meaningful, and why similar words produce similar model behavior |
| **When NOT to use it** | Do not treat embeddings as encoding objective meaning — they encode statistical co-occurrence patterns from training data; concepts that appear in similar contexts will be embedded similarly, regardless of real-world truth |
| **What juniors miss** | They conflate tokens and embeddings. A token is the text fragment ("Paris", "##ization", "▁the"). An embedding is the learned numerical representation of that token. The model never processes text directly — it processes embeddings. |
| **Senior signal** | "The model's concept of similarity is defined by its embedding space, which is shaped by training data co-occurrence — not semantic ground truth. That is why embedding models trained on different corpora produce different similarity judgments for the same pair of concepts." |

### Mental Model

An embedding is like GPS coordinates for meaning. Geographic coordinates place a city in physical two-dimensional space — nearby coordinates mean physically nearby places. Embeddings place a word in a very high-dimensional meaning space — nearby coordinates mean semantically related concepts.

"Paris" and "London" are close in both geographic and embedding space. "Paris" and "contract law" are far apart in both. The geometry of the embedding space encodes relationships: the vector from "king" to "queen" is approximately the same as the vector from "man" to "woman" — because those relationships appear consistently in training text.

Unlike geographic coordinates, embedding space is hundreds or thousands of dimensions — which is why the model can encode thousands of distinct types of semantic relationship simultaneously.

### What an Embedding Looks Like and Why Distance Is Meaningful

```python
import numpy as np

# A real embedding vector has 768–3072 dimensions.
# This example uses 4 dimensions to show the structure.
# In practice you get these from an embedding model, not by hand.

# Conceptual embeddings — what the geometry looks like:
embeddings = {
    "cat":         np.array([0.91,  0.87,  0.03,  0.12]),
    "kitten":      np.array([0.89,  0.84,  0.04,  0.11]),  # close to cat
    "dog":         np.array([0.85,  0.76,  0.05,  0.15]),  # same neighborhood
    "automobile":  np.array([0.10,  0.09,  0.91,  0.88]),  # very different region
}


def cosine_similarity(a: np.ndarray, b: np.ndarray) -> float:
    """Measures how similar two vectors are, regardless of their magnitude.
    1.0 = identical direction, 0.0 = unrelated, -1.0 = opposite meaning."""
    return float(np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b)))


pairs = [
    ("cat", "kitten"),
    ("cat", "dog"),
    ("cat", "automobile"),
]

for word_a, word_b in pairs:
    sim = cosine_similarity(embeddings[word_a], embeddings[word_b])
    print(f"{word_a!r:12} ↔ {word_b!r:12}  similarity = {sim:.3f}")

# cat          ↔ kitten        similarity = 0.999   ← very similar meaning
# cat          ↔ dog           similarity = 0.993   ← same semantic neighborhood
# cat          ↔ automobile    similarity = 0.207   ← semantically distant
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `np.array([0.91, 0.87, ...])` | Represents a token as a vector of numbers | This is what the model actually processes — not the word "cat," but a high-dimensional vector learned during training that encodes where "cat" sits in meaning space |
| `cosine_similarity(a, b)` | Measures the angle between two vectors | Cosine similarity is the standard metric for embedding comparison because magnitude varies — direction (angle) is what encodes semantic relationship |
| `cat ↔ kitten = 0.999` | Extremely close in embedding space | This is why embedding-based search finds "kitten care tips" when you search for "cat advice" — the vectors are so similar that retrieval treats them as nearly equivalent |
| `cat ↔ automobile = 0.207` | Very different regions of embedding space | Different enough that a retrieval system will not confuse them — the embedding space has learned to separate unrelated concepts |

## CONCEPT 2: Attention — How the Model Decides What Each Token Means in Context
<!-- cognitive job: build the engineering intuition for why context matters, why position matters, and why attention patterns explain real production phenomena like "lost in the middle" -->

| Dimension | Answer |
|---|---|
| **What it is** | Attention is the mechanism that lets the model look at every other token in the sequence when computing the representation of any single token — it assigns a weight to how much each other token should influence the current one |
| **Why it matters** | Attention is why context resolves ambiguity. The same word produces a different internal representation depending on what it attends to — which is why "bank" means different things in different sentences, and why the model can handle pronoun references and long-range dependencies |
| **When to use it** | Use this mental model when reasoning about why prompt structure affects output quality, why information position within a long context matters, why the "lost in the middle" degradation happens, and why the model handles some reference patterns better than others |
| **When NOT to use it** | Do not think of attention as the model "reading carefully" — it is a mathematical weighted aggregation over the full token sequence, computed simultaneously for all tokens, not sequentially like a human reader |
| **What juniors miss** | They assume attention is uniform — that the model pays equal attention to all parts of the input. Attention weights are not uniform. Tokens near the beginning and end of a context typically receive stronger attention weights than tokens in the middle, which explains why critical information buried in the middle of a long prompt is more likely to be missed. |
| **Senior signal** | "Attention is why context window position is a design decision. If the most critical information for a task is placed in the middle of a 100K-token context, it will receive less attention weight on average than the same information placed at the beginning or end. We structure prompts to put the highest-priority content where attention is strongest." |

### Mental Model

Imagine every word in a sentence holding a sign that says "pay attention to me" — and every other word in the sentence deciding how much to look at that sign. When the model processes the word "it" in the sentence "The trophy did not fit in the suitcase because it was too large," every other word in the sentence votes on what "it" should refer to. "Trophy" and "suitcase" get high votes because they are the candidate referents. "Did" and "because" get low votes because they do not help resolve the reference. The model uses these votes — the attention weights — to build a representation of "it" that incorporates the most relevant context.

The key production insight: this happens for all tokens simultaneously, not sequentially. And the vote distribution is not uniform — it is shaped by the learned patterns of training data. That non-uniformity is why position in the context window affects how much influence a piece of information has.

### How Attention Works — The Conceptual View

```python
# This represents the attention mechanism conceptually.
# Real attention operates in high-dimensional vector space.
# This simplified version shows the structure of the idea.

sentence = ["The", "trophy", "did", "not", "fit", "because", "it", "was", "too", "large"]

# When the model processes the token "it" (index 6),
# it assigns an attention weight to every other token.
# Higher weight = more influence on what "it" comes to mean.

attention_weights_for_it = {
    "The":     0.01,
    "trophy":  0.61,   # "it" most likely refers to the trophy
    "did":     0.02,
    "not":     0.01,
    "fit":     0.08,   # "fit" is relevant — it constrains the referent
    "because": 0.02,
    "it":      0.07,   # self-attention — the token always attends to itself
    "was":     0.05,
    "too":     0.03,
    "large":   0.10,   # "large" is directly relevant to resolving the reference
}

# The resulting representation of "it" is a weighted combination
# of the embeddings of all tokens, using these weights.
# result = sum(weight * embedding[token] for token, weight in weights.items())
#
# This weighted combination is what the next layer of the transformer sees —
# not the raw embedding of "it", but "it enriched by its context."

assert abs(sum(attention_weights_for_it.values()) - 1.0) < 1e-9  # weights sum to 1
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `"trophy": 0.61` | Highest attention weight for resolving "it" | The model learned from training data that in this syntactic pattern, "it" refers to a noun earlier in the sentence — the largest noun is the most probable referent |
| `"large": 0.10` | Moderate weight even though "large" comes after "it" | Transformers can attend to tokens in both directions — unlike RNNs which were strictly left-to-right. This bidirectional attention is a core transformer capability. |
| `"did": 0.02, "not": 0.01` | Very low weights for function words | Function words carry little semantic content relevant to reference resolution — the model learns to down-weight them for this type of task |
| `sum(weights) == 1.0` | Attention weights are a probability distribution over the sequence | This is the softmax operation — every token's representation is a convex combination of all other token embeddings, weighted by relevance |

## CONCEPT 3: Layers and Context Window — What Depth and Size Mean for Engineering
<!-- cognitive job: establish the context window as a hard engineering constraint and connect transformer depth to capability and cost — both are production design variables -->

| Dimension | Answer |
|---|---|
| **What it is** | A transformer stacks many attention layers — each one refining token representations toward more abstract meaning — and the context window is the hard limit on how many tokens all of these layers can process in a single call |
| **Why it matters** | Layers determine model capability and inference cost. The context window is a non-negotiable architectural limit: content beyond it is silently excluded, and designing without accounting for it causes production failures |
| **When to use it** | Use this when estimating request cost and latency, designing prompt structures for variable-length inputs, chunking documents for RAG, and reasoning about why larger models cost more per token |
| **When NOT to use it** | Do not treat the context window as a target to fill — sending the maximum token count every call maximizes cost and latency. Design for the minimum context that preserves task quality. |
| **What juniors miss** | They treat the context window as a soft limit — assuming the model handles content "less well" when it is long. The limit is hard. Content that exceeds the window is not processed at all: it is either truncated (silently, in many APIs) or the call returns an error. There is no graceful degradation — there is truncation. |
| **Senior signal** | "Context window is a production constraint I design around from the start, not a number I check when things break. Every endpoint that accepts variable-length input needs a token count check before the model call, with a defined behavior for when the limit is approached — truncation strategy, chunking, or error." |

### Mental Model

The context window is like a workbench with a fixed surface area. Everything you want the model to work with — the system prompt, conversation history, retrieved documents, the user's message — must fit on that bench simultaneously. If your inputs exceed the bench size, something gets pushed off the edge before the model ever sees it. The bench does not expand because the task is important. The model has no awareness that something was excluded.

Layers are like the depth of processing — a shallow model does one pass of pattern matching; a deep model does many passes, each one building more abstract representations on top of the previous. More layers means the model can encode more complex relationships, but every layer adds compute cost per token. This is why GPT-4 costs more per token than GPT-3.5 — the architecture is deeper and wider, not just "smarter."

### Context Window in Practice — The Token Budget Pattern

```python
import tiktoken
from openai import AzureOpenAI

MODEL = "gpt-4o"
CONTEXT_LIMIT = 128_000   # GPT-4o context window in tokens
OUTPUT_RESERVE = 2_000    # tokens reserved for model output
MAX_INPUT_TOKENS = CONTEXT_LIMIT - OUTPUT_RESERVE   # 126,000

encoder = tiktoken.encoding_for_model(MODEL)


def count_tokens(text: str) -> int:
    return len(encoder.encode(text))


def build_prompt_within_budget(
    system_prompt: str,
    retrieved_chunks: list[str],
    user_message: str,
) -> list[dict]:
    """
    Assemble a prompt that fits within the context window.
    Drops retrieved chunks from the end if the budget is exceeded.
    Critical information (system prompt, user message) is always preserved.
    """
    system_tokens = count_tokens(system_prompt)
    user_tokens = count_tokens(user_message)
    overhead = system_tokens + user_tokens

    if overhead > MAX_INPUT_TOKENS:
        raise ValueError(
            f"System prompt + user message alone ({overhead} tokens) "
            f"exceeds input budget ({MAX_INPUT_TOKENS} tokens)."
        )

    budget_for_chunks = MAX_INPUT_TOKENS - overhead
    included_chunks: list[str] = []
    used_tokens = 0

    for chunk in retrieved_chunks:
        chunk_tokens = count_tokens(chunk)
        if used_tokens + chunk_tokens > budget_for_chunks:
            break   # stop adding chunks — budget exhausted
        included_chunks.append(chunk)
        used_tokens += chunk_tokens

    context_text = "\n\n".join(included_chunks)

    return [
        {"role": "system", "content": system_prompt},
        {"role": "user",   "content": f"{context_text}\n\n{user_message}"},
    ]


# Usage
messages = build_prompt_within_budget(
    system_prompt="You are a contract analysis assistant. Answer based only on the provided clauses.",
    retrieved_chunks=["Clause 4.2: Payment terms are net-30...", "Clause 7.1: Termination requires 60-day notice..."],
    user_message="What are the payment and termination terms?",
)
```

| Code line / block | What it does | Why it matters |
|---|---|---|
| `MAX_INPUT_TOKENS = CONTEXT_LIMIT - OUTPUT_RESERVE` | Reserves token budget for the model's output | A common mistake is filling the entire context window with input, leaving no room for output — the model either truncates its response or the call errors. Always reserve for output. |
| `count_tokens(text)` | Measures prompt size before sending | Counting tokens before every call with variable-length inputs is the production standard. Silent truncation by the API is worse than an explicit error — the model gives an answer without the missing context, and nothing in the output signals that information was lost. |
| `if overhead > MAX_INPUT_TOKENS: raise ValueError(...)` | Fails explicitly if fixed content already exceeds budget | The system prompt and user message are non-negotiable. If they alone exceed the budget, the task cannot be completed correctly — an early explicit error is better than a silent degraded result. |
| `for chunk in retrieved_chunks: if ... > budget: break` | Fills remaining budget with retrieved chunks, stopping when full | This is the standard RAG context packing pattern: fixed structure first, variable content second, stop when the limit is reached. The most important retrieved chunks should be first in the list. |

## DECISION RULES
<!-- cognitive job: support transfer — give a practical instrument for real decisions -->

### The Transformer Architecture Sanity Check for System Design

Use this when designing any LLM-powered system that handles variable-length inputs or needs to reason over long content.

```text
STEP 1: What is the token budget for this request?
        context_window_limit
        - tokens for system prompt
        - tokens reserved for output
        - tokens for conversation history (if multi-turn)
        = available tokens for user content + retrieved context
        
        If available_tokens <= 0 before any user content is added → redesign the prompt.
                        │
                        ▼
STEP 2: Is the input length bounded or unbounded?
        Bounded (always under limit) → count tokens in testing, add a guard in production
        Unbounded (documents, long conversations, web pages) → chunking or truncation strategy required
                        │
                        ▼
STEP 3: Where does the most critical information live in the prompt?
        Critical instructions → system prompt (always first, always inside window)
        Critical facts → beginning or end of context (stronger attention weights)
        Optional context → middle (weaker attention weights, first to be cut)
                        │
                        ▼
STEP 4: What is the truncation strategy if the limit is hit?
        [ ] Truncate the least important content (middle of retrieved chunks)
        [ ] Raise an explicit error (for debugging / high-stakes calls)
        [ ] Chunk and process in multiple calls
        [ ] Switch to a model with a larger context window
                        │
                        ▼
STEP 5: What model size is appropriate for this task?
        Simple, bounded tasks (classification, short QA) → smaller model, lower cost, lower latency
        Complex reasoning, synthesis across long content → larger model accepted
        Cost-sensitive, high-volume → always test whether a smaller model meets quality bar first
```

### Worked Example

**Scenario:** A support team wants an LLM that reads full conversation threads (sometimes 20+ messages) and classifies the ticket into one of eight support categories.

| Step | Question | Answer for this case |
|---|---|---|
| 1 | Token budget? | System prompt: 300 tokens. Output: 50 tokens (category label + brief reason). Input budget: context_window - 350. |
| 2 | Input length bounded? | Conversation threads vary. Long threads can be thousands of tokens. Unbounded — needs a strategy. |
| 3 | Critical info position? | The first and last user message are most diagnostic. Middle messages are context. Structure: first message, last message, abbreviated middle. |
| 4 | Truncation strategy? | For classification: keep newest and oldest messages, summarize the middle if thread exceeds budget. |
| 5 | Model size? | Classification with 8 categories on conversation text is not complex — a smaller/cheaper model should be tested first before committing to GPT-4o. |

### Red Flags in System Design Conversations

| What someone says | Why it should raise a flag |
|---|---|
| "The model has a 128K context window — we'll just put the whole document in" | Maximum context does not mean optimal context. 128K tokens is expensive to process and the model's attention degrades in the middle. Chunk and retrieve instead. |
| "If it's over the limit, the model will just cut off gracefully" | Many APIs truncate silently without an error. The model answers without the missing content. Silent truncation produces wrong answers with no error signal. |
| "We'll put the critical instructions at the end so the model sees them last" | Recency is not always better. System prompts at the beginning receive high attention weight by convention. Instructions buried in the middle of a long context are at risk regardless of where they appear relative to documents. |
| "We're using the biggest model because it's smarter" | Model size is a cost and latency multiplier. Always establish the minimum model that meets the quality bar for the specific task. |
| "Embeddings are just word vectors — they're all pretty similar" | Different embedding models trained on different data produce meaningfully different similarity judgments. Choosing the wrong embedding model for a domain (e.g., a general model for legal or medical text) degrades retrieval quality measurably. |

### Architecture Summary: Transformers and What They Mean for Production

| Architectural property | What it produces | Production implication |
|---|---|---|
| Embeddings encode semantic similarity | Similar concepts have similar vectors | Enables semantic search; quality depends on embedding model training data |
| Attention is context-sensitive | Same token produces different representations in different contexts | Context window position matters; prompt structure affects output quality |
| Attention weights are non-uniform | Middle of long context receives less weight | Put critical instructions at the start; use RAG instead of stuffing long docs |
| Context window is a hard limit | Content beyond limit is excluded | Count tokens before every variable-length call; have a truncation strategy |
| Layers add abstraction at compute cost | More layers = more capability and more cost | Match model depth to task complexity; test smaller models first |

## COMMON MISTAKES
<!-- cognitive job: inoculate against predictable errors by teaching error structure, not just correct answers -->

### Mistake 1: Treating the Context Window as a Soft Quality Preference

**Mistaken instinct:** "If the document is too long, the model just handles it less well — it still sees everything."

**Why it feels right:** Software systems often degrade gracefully under load. You expect the model to do something reasonable with content that is "too much." The mental model is a person reading a long document — they get tired, but they still read it.

**Why it fails:** The context window is a hard architectural limit. The transformer's attention mechanism operates over a fixed maximum number of tokens. Content beyond that limit is not processed at all — it is truncated before the model call, either by the API silently or by the client library. The model has no awareness that anything was excluded. Its output reflects only the content it received.

**Better move:** Count tokens before every call where input length is variable. Define a truncation strategy in advance. Test what your truncation does to output quality. Never assume content is included unless you verified the token count.

---

### Mistake 2: Putting the Most Critical Information in the Middle of a Long Context

**Mistaken instinct:** "It does not matter where in the prompt we put the instructions — the model reads it all."

**Why it feels right:** When you read a document, position matters less than content. A well-trained reader attends to important information regardless of where it sits in the document.

**Why it fails:** Attention weights are not uniform across the token sequence. Research and production experience consistently show that tokens near the beginning and end of a long context receive stronger average attention weights than tokens in the middle. Critical instructions or facts placed in the middle of a large context are more likely to be underweighted in the model's final output representation. This is the documented "lost in the middle" phenomenon.

**Better move:** Put the system prompt and critical instructions at the beginning. Put the user message and the most important context at the end. If using RAG, rank retrieved chunks and put the highest-relevance chunks closest to the user message. Use the middle for supplementary context that is helpful but not decision-critical.

---

### Mistake 3: Confusing Embedding Models with LLM Models

**Mistaken instinct:** "We are using GPT-4o, so the embeddings are GPT-4o embeddings."

**Why it feels right:** It sounds natural that a GPT-4o-powered system would use GPT-4o for every operation. The same vendor often offers both.

**Why it fails:** Embedding models and generation models are separate systems with separate training objectives and separate APIs. GPT-4o is a text generation model. `text-embedding-3-large` is an embedding model. You call them separately. Mixing them up leads to either calling the wrong API (expensive generation call instead of cheap embedding call) or misunderstanding why similarity scores do not match your expectations.

**Better move:** Keep the mental distinction clear: embedding models convert text to vectors and are optimized for semantic similarity. Generation models predict next tokens and are optimized for text quality. Both are transformers, but they are trained for different objectives and used for different steps in the pipeline.

---

### Mistake 4: Assuming All Embedding Models Are Equivalent

**Mistaken instinct:** "Any embedding model will work for our retrieval system — they all convert text to vectors."

**Why it feels right:** The idea of "converting text to a vector" sounds like a commodity operation. If vectors represent meaning, and all models were trained on text, surely they produce similar quality.

**Why it fails:** Embedding quality varies significantly across models and is domain-sensitive. An embedding model trained on general web text will produce poor semantic similarity for specialized legal, medical, or code content because those domains are underrepresented in its training data. Additionally, embedding models vary in dimensions (384 to 3072), multilingual capability, and the specific definition of "similar" they optimize for. Switching embedding models invalidates your existing vector index — all stored embeddings must be regenerated.

**Better move:** Select embedding models using benchmark data for your domain. For multilingual systems, use models explicitly trained on multilingual corpora. Test retrieval precision on a sample of your actual data before committing. Treat the embedding model as a critical architectural choice — changing it later is expensive.

For each situation, identify the primary architectural cause and whether it is an embedding problem, an attention/position problem, or a context window problem:
- a) The model gives the wrong answer about a contract clause that was on page 35 of a 40-page document injected into the prompt
- b) The model retrieves documents about "feline care" when the user asks about "cat health" — and it should
- c) The model correctly answers a question about the first and last paragraphs of an essay, but misses a key claim from the middle
- d) The model returns an error or truncated output when a 200,000-word document is submitted

*Check:* a) Context window problem — page 35 of a 40-page document likely exceeds or approaches the limit, and content may have been truncated; b) Embedding property — "feline" and "cat" have similar embeddings, so cosine similarity correctly finds the match (this is the desired behavior); c) Attention/position problem — "lost in the middle" phenomenon, middle content receives weaker attention weights; d) Context window hard limit — the document exceeds the model's maximum token capacity.

**3. Apply**
You are designing a RAG system for a legal firm. Each legal brief is 30–80 pages. The model has a 128,000-token context window. The system prompt is 600 tokens, and output is reserved at 2,000 tokens. A page is approximately 400 tokens. Walk through the token budget, explain whether you can fit the largest briefs in context, and describe the chunking strategy you would use if not.

*Check:* Available input budget = 128,000 - 600 - 2,000 = 125,400 tokens. Largest brief (80 pages × 400 tokens) = 32,000 tokens — fits in budget. However, using 32,000 tokens of input when only a small portion is relevant is expensive and degrades attention quality. Better approach: chunk the brief into sections, embed each section, retrieve the 5–10 most relevant sections for the user's query (roughly 2,000–4,000 tokens), and pack only those into the context. This is faster, cheaper, and produces better results than stuffing the full document.

**4. Error diagnosis**
A document Q&A system worked correctly in testing with 5-page documents. After deployment, users submit 100-page documents and report that the model "ignores" information from certain sections. The system does not return any errors. Name three hypotheses and what you would check for each.

*Check:* 1) Silent truncation — the document exceeds the context window and the API is truncating without an error; check token count of a representative 100-page document against the model's context limit; 2) Lost in the middle — the document fits but critical sections are in the middle, receiving lower attention weights; check which page ranges produce correct answers and whether the failures cluster in the middle third; 3) Chunking not applied — the team is injecting the full document as one block, which even if it fits is inefficient; check whether a RAG chunking approach improves middle-section answers.

**5. Senior framing**
A product manager asks: "We have GPT-4o with a 128K context window — why do we need RAG? Can't we just put all our company documents in the context every time?" How do you respond?

*Check:* Three reasons to use RAG even with a large context window: 1) Cost — 128K tokens of input per request multiplied by request volume is expensive; retrieving the 10 most relevant chunks (2K–5K tokens) is 25–60x cheaper per call; 2) Attention quality — the model's attention is not uniformly distributed; burying important information in 128K tokens of noise degrades answer quality compared to providing 5K tokens of precisely relevant context; 3) Freshness and scale — a vector index can store millions of documents and retrieve the relevant ones at inference time; a 128K context window cannot hold all company knowledge simultaneously and has no mechanism for updating dynamically.

## SUMMARY

**Five things to carry out of this lesson:**

1. Embeddings convert tokens into numerical vectors where semantic similarity is geometric proximity — similar meaning, similar vector direction.
2. Attention allows every token to "look at" every other token in the sequence simultaneously, so the same word produces different internal representations depending on context.
3. Attention weights are not uniform — tokens at the beginning and end of a long context receive stronger weights on average; critical information in the middle is at systematic risk of being underweighted.
4. The context window is a hard architectural limit, not a soft preference — content beyond it is silently excluded and the model has no awareness that anything was missing.
5. Larger models cost more because they have more layers and wider attention — match model depth to task complexity; always test whether a smaller model meets the quality bar first.

```text
TEXT → [TOKENIZER] → tokens → [EMBEDDING LAYER] → vectors

                    ↓ attention ↓
each token attends to all others → context-enriched representations
    (non-uniform weights: start/end > middle for long contexts)

                    ↓ stacked layers ↓
early layers:  syntax, local patterns
later layers:  semantics, world knowledge, reasoning structure

                    ↓ output layer ↓
probability distribution over next token → sample → repeat

HARD LIMIT: all of the above must fit in [CONTEXT WINDOW] tokens
            anything beyond the limit → silently excluded
```

## FAST RECALL

### What This Lesson Is Really About

The transformer converts text into geometry, uses attention to let every token influence every other, and operates within a hard token budget that is a production constraint you must design around from the start.

### 30-Second Recall

- Embeddings = numbers that place tokens in meaning space — similar meaning, nearby vectors
- Attention = every token influences every other, weighted by relevance; weights are non-uniform
- "Lost in the middle" = content in the middle of a long context gets systematically lower attention weight
- Context window = hard limit; content beyond it is silently excluded, not gracefully degraded
- Layers = depth of abstraction and capability; more layers = more cost per token

           ↓

### Decision Triggers

- Input length is variable → count tokens before every call; define truncation strategy
- Document exceeds context window → chunk and retrieve; do not try to fit everything in
- Critical instruction or fact → place at start or end of context, not in the middle
- Semantic search or retrieval needed → embedding model (separate from generation model)
- Task is simple and bounded → test a smaller/cheaper model before defaulting to the largest

### Output Shapes to Remember

| Need | Typical output |
|---|---|
| `tiktoken.encode(text)` | List of integers (one per token); `len(...)` gives token count |
| `np.dot(a, b) / (norms)` | Float in [-1, 1]; 1.0 = identical direction, 0.0 = orthogonal |
| Embedding API call | List of floats (e.g., 1,536 for `text-embedding-3-small`) |
| Context window exceeded | Silent truncation (common) or `context_length_exceeded` error (explicit) |

### Confusions to Avoid

| Easy confusion | Correct distinction |
|---|---|
| Token = embedding | A token is the text fragment; an embedding is the learned numerical representation of that token |
| Context window is a soft quality limit | It is a hard architectural limit — content beyond it is simply not processed |
| Attention means the model reads carefully | Attention is a mathematical weighted aggregation computed simultaneously for all tokens — not sequential reading |
| All embedding models produce the same similarity judgments | Quality, domain coverage, and multilingual capability vary significantly across embedding models |
| Larger context window → always better | Larger context costs more, degrades attention quality in the middle, and rarely outperforms targeted retrieval |

### Top Traps

- Not counting tokens for variable-length inputs — silent truncation with no error signal
- Placing critical instructions in the middle of a long context prompt
- Using a general embedding model for a specialized domain (legal, medical, code) without testing retrieval quality
- Switching embedding models without regenerating the full vector index
- Assuming the model "handled" content that exceeded the context window
- Filling the entire context window on every call — maximum context = maximum cost + maximum latency

### One Good Example to Keep

A legal document assistant was built to answer questions about contract terms. For short contracts it worked. For long contracts (60+ pages), it missed clauses from the middle sections. The team debugged by testing: pages 1–10 worked, pages 50–60 worked, pages 25–40 failed consistently. The fix was not a better prompt or a bigger model. It was chunked retrieval — split the contract into sections, embed each section, retrieve the 5–8 most relevant sections for each question, and pass only those into the prompt. The model now saw 4,000 tokens of highly relevant content instead of 50,000 tokens with poor attention coverage of the critical middle.

### Interview Q&A

**Q: What is an embedding and why is it used in LLM systems?**

> "An embedding is a numerical vector that represents the semantic meaning of a piece of text — a token, a sentence, or a passage — in a high-dimensional space where geometrically similar vectors represent semantically similar meanings. Embeddings are foundational to LLM systems for two reasons. First, the transformer architecture processes numbers, not text — the embedding layer converts raw tokens into numerical representations that the attention mechanism and layers can operate on. Second, embeddings enable semantic retrieval: you can embed a user query and a collection of documents, then find the documents most similar to the query by measuring cosine distance between their vectors. This is the core of every RAG system. The key engineering consideration is that embedding quality is domain-sensitive — a model trained on general web text may not produce high-quality semantic similarity for specialized domains like legal, medical, or code content."

---

**Q: What is the context window and why does it matter for production system design?**

> "The context window is the hard maximum number of tokens — input plus output combined — that the model can process in a single call. It is not a quality preference or a soft limit: content beyond the window is excluded before the model ever sees it. Many APIs truncate silently without raising an error, which means the model gives an answer based on incomplete information with no indication that anything was missing. This makes it one of the most dangerous silent failure modes in LLM integration. For production systems, I treat the context window as a constraint I design around from day one: count tokens for every call with variable-length input, reserve budget for output, and define a truncation or chunking strategy before deployment. There is also a secondary effect within long contexts — attention weights are non-uniform, and the middle of a long prompt receives systematically lower weight than the beginning and end. This is why RAG — retrieving a small amount of precisely relevant content — outperforms stuffing a long document into context even when the document fits within the window."

---

**Q: Why does position within a long prompt affect model output quality?**

> "Attention is not uniform across the sequence. The transformer's attention mechanism assigns a weight to how much each token influences every other token — and these weights are learned from training data patterns. For very long contexts, empirical research and production experience consistently show that tokens near the beginning and end receive higher average attention weights than tokens in the middle. This is called the 'lost in the middle' effect. It means that instructions or facts placed in the middle of a 100K-token context are at systematic risk of being underweighted in the model's final representations, even though they technically fit within the context window. The production response to this is threefold: put critical instructions in the system prompt at the beginning; put the most important retrieved context closest to the user message at the end; and use targeted retrieval rather than long-document injection so that all the content the model sees is relevant and there is no expendable middle to lose things in."
