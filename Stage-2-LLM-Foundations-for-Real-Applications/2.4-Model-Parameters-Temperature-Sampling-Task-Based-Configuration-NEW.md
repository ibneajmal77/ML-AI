## Core Mental Model

### Parameters as Behavioral Contracts Per Task

Think of `LLMConfig` not as "API options to pass" but as a behavioral contract that defines how the model should act for a specific task.

Different tasks need different behavioral contracts:

```
Classification  →  determinism contract   →  same ticket must return same label every time
Extraction      →  precision contract     →  JSON must be stable and tightly bounded
Summarization   →  coherence contract     →  fluent, moderate-length, some variation acceptable
Drafting        →  creativity contract    →  variation between runs is desirable, not a bug
```

The four parameters you'll configure — temperature, top-p, max_tokens, stop sequences — are the levers that implement these contracts. Understanding each lever means understanding *why* a classification task and a drafting task must be configured differently, not just memorizing the "right" values.

## Main Concepts + Production Reality

### Concept 1 — Temperature

**Scenario:**
You run your ticket classifier on the same billing complaint five times. You get back: `billing`, `billing`, `general`, `billing`, `account`. The ticket hasn't changed. The model hasn't changed. The answers are inconsistent.

**What's happening internally:**

1. The model reads the ticket and scores every possible next token by probability:
   - `"billing"` → 62%
   - `"general"` → 21%
   - `"account"` → 11%
   - everything else → remaining 6%

2. **temperature = 1.0** (default): the model samples from those probabilities exactly as they are
   → 62% of runs return `"billing"`, 21% return `"general"`, 11% return `"account"`
   → Non-deterministic. Bad for any automated pipeline.

3. **temperature = 0.0**: the model always picks the single highest-probability token
   → Always returns `"billing"`
   → Deterministic. Reproducible. Right for classification.

4. **temperature = 1.5**: the distribution is flattened — probabilities spread more evenly
   → `"billing"` drops to ~45%, `"general"` rises to ~30%, `"account"` to ~20%
   → High variance. Useful for brainstorming; destructive for anything parsed or dispatched automatically.

**This process is called temperature** — a scalar that controls how sharply the model commits to its highest-probability prediction vs. samples from the full distribution.

**Engineering implication:**

```
Task type         Temperature range   Behavioral contract
──────────────────────────────────────────────────────────────────
Classification    0.0 – 0.2           Same input → same label, every time
Extraction        0.0 – 0.2           JSON structure must be stable across runs
Summarization     0.3 – 0.6           Fluent prose, moderate variation acceptable
Drafting          0.6 – 1.0           Different phrasings per run is desired
```

**Common trap:** Setting `temperature=0` and assuming outputs will be bit-identical across all API calls. Near-equal probability tokens can still produce variation due to floating-point precision and server-side batching. Temperature minimizes variance — it does not guarantee identical outputs. For evaluation reproducibility, also use the `seed` parameter (→ Optional Advanced Notes).

### Concept 2 — Top-p (Nucleus Sampling)

**Scenario:**
You're generating a ticket summary. You want natural variation — "The customer reported a billing issue" vs. "The user flagged a payment problem" — but you don't want the model writing "The customer invoked a spectral anomaly" because `"spectral"` had a 0.001% probability in the distribution.

**What's happening internally:**

1. The model scores all ~100,000 tokens in its vocabulary by probability for the next position
2. **top_p = 1.0**: every token in the vocabulary is eligible — including the bizarre low-probability tail
3. **top_p = 0.9**: find the smallest set of tokens whose cumulative probability reaches 90%
   - Example: `"The"` (40%) + `"A"` (25%) + `"Customer"` (15%) + `"User"` (10%) = 90%
   - Those 4 tokens form the candidate nucleus → everything else (including `"spectral"`) is excluded
   - The model samples from just those 4 → natural variation without bizarre outputs
4. **top_p = 0.1**: only the very top token reaches 10% alone — extremely narrow, almost greedy

**This filtering is called nucleus sampling (top-p)** — the model samples only from the "nucleus" of high-probability tokens that together account for `p` probability mass.

**Engineering implication:**
- Top-p and temperature work together. Temperature controls how spread the distribution is; top-p controls how wide the candidate pool is.
- When `temperature = 0.0`, top-p has no practical effect — greedy decoding always picks the single highest token regardless of pool size.
- For most production tasks, `top_p = 0.9` or `0.95` is a safe default: it filters the low-probability tail without restricting useful vocabulary variation.
- Tune temperature first for the variance level you want. Adjust top-p only if you see bizarre low-probability tokens in the output.

**Common trap:** Tuning both temperature and top-p simultaneously without understanding they interact. They compound. Start with temperature and treat top-p as a secondary dial you rarely need to move.

### Concept 3 — max_tokens

**Scenario:**
Your `/extract` endpoint is supposed to return a small JSON object — roughly 80 tokens. One morning, an unusually complex ticket causes the model to write 1,200 tokens of explanation before generating the JSON. The JSON parser fails. The API call costs 15× more than expected. This happens at 3 AM while your pipeline is running unattended.

**What's happening internally:**

1. The model generates tokens one by one indefinitely until it emits a stop token — and on some inputs, "stopping" means writing a long preamble first
2. **max_tokens = 200**: generation halts after 200 output tokens regardless of what the model "wanted" to write next
3. Too low — `max_tokens = 50` for a JSON that needs 80 tokens: the JSON gets cut off mid-object → your parser crashes with `JSONDecodeError`
4. Too high — `max_tokens = 4000` for a single-label classification: you're reserving 4,000 tokens of API capacity on every call, even though the output is 8 characters

**max_tokens is the hard output cap** — it protects your cost budget and forces concise outputs by design.

**Engineering implication:**

```
Task              Expected output size    Recommended max_tokens
─────────────────────────────────────────────────────────────────
Classification    1 word label            10 – 20
Extraction        JSON object, 5-7 fields 150 – 300
Summarization     2–3 sentences           100 – 200
Drafting          1–2 paragraphs          300 – 600
```

**Budget math — linking back to 2.3:**

```
Rule: prompt_tokens + max_tokens ≤ context_limit

Example (extraction endpoint):
  System prompt:   180 tokens
  User message:    350 tokens
  max_tokens set:  300
  Total:           830 tokens  →  well within 128k limit ✓

Mistake (max_tokens left at model maximum = 4096):
  Same prompt:   530 tokens
  max_tokens:    4096
  Total:         4626 tokens  →  still fits, but 4000 tokens reserved per call
  At 10,000 calls/day: 40M output tokens reserved daily  →  cost multiplied ~10x
```

**Common trap:** Leaving `max_tokens` unset (or at the model maximum) for tasks with short, predictable outputs. Defaults are tuned for generality, not for your use case. Set it per task, sized to actual expected output with a 2× buffer.

### Concept 4 — Stop Sequences

**Scenario:**
Your classification prompt uses a few-shot pattern: `Input: <ticket>` → `Output: <label>`. You want the model to fill in the label and stop. Instead, the model returns:

```
Output: billing

Input: Can I change my email address?
Output: account
```

The model continued the pattern and generated the next fake example — because that's what statistically follows `Output: billing` in a few-shot context.

**What's happening internally:**

1. Few-shot prompts work by establishing a pattern — the model sees Input/Output pairs and predicts what comes next
2. After generating `"billing\n"`, the pattern predicts `"Input:"` should follow — so the model generates it
3. **stop=["\n"]**: as soon as the model emits a newline after `"billing"`, generation halts immediately
   → You get: `"billing"` — clean, parseable, no continuation
4. Multiple stop sequences: model stops at the first string it generates that matches any entry in the list

**Stop sequences are strings that, when generated, immediately terminate output** — the matching string itself is not included in the response.

**Engineering implication:**

```
Prompt pattern                        Stop sequences to use
─────────────────────────────────────────────────────────────────────
Single-label output (one word)        ["\n"]
Few-shot Input:/Output: pairs         ["Input:", "\n\nInput:"]
JSON output                           None needed — max_tokens handles cutoff
Multi-turn with a sentinel            ["---END---", "DONE"]
```

**Common trap:** Forgetting stop sequences on few-shot classification prompts. The model generates the next fake Input/Output pair, inflating token usage, returning unparseable multi-block output, and failing the downstream label validator.

## Project Integration

**Component:** `app/config.py` — the task-based configuration registry all service modules import, plus `app/services/llm.py` — the API wrapper every service module uses. These two files form the foundation every subsequent module builds on.

**What gets implemented:**
1. `Settings` — global config loaded from the environment (model, API key, context limits)
2. `LLMConfig` — per-task parameter dataclass (temperature, top_p, max_tokens, stop)
3. `TASK_CONFIGS` — maps task name to its behavioral contract (documented inline)
4. `get_task_config()` — the function all service modules call
5. `app/services/llm.py` — `chat_with_retry()` with token logging and retry on rate limit

```python
# app/config.py

from dataclasses import dataclass, field
from pydantic_settings import BaseSettings


# ── Global settings ───────────────────────────────────────────────────────────
# Loaded from environment / .env file once at startup.
# Required: OPENAI_API_KEY

class Settings(BaseSettings):
    """
    Global application configuration.

    Set these in .env or environment variables before running the app.
    """
    openai_api_key: str
    model: str = "gpt-4o-mini"       # Default model for most tasks
    strong_model: str = "gpt-4o"     # Higher quality model for complex tasks (→ 2.12)
    context_limit: int = 128_000     # Max context window in tokens
    output_reserve: int = 2_000      # Default token reserve for output budget planning (→ 2.3)

    class Config:
        env_file = ".env"


settings = Settings()


# ── Per-task parameter configuration ─────────────────────────────────────────
# Each task type has its own behavioral contract.
# Parameters are set per task — not globally.
# Add new task configs to TASK_CONFIGS below; never hardcode parameters in service files.

@dataclass
class LLMConfig:
    """
    Parameter set for one task type.

    Pass this to chat_with_retry() — it unpacks into the API call.
    model=None means use settings.model (resolved in get_task_config).
    """
    temperature: float = 0.0
    top_p: float = 1.0
    max_tokens: int = 200
    stop: list[str] = field(default_factory=list)
    model: str | None = None  # None → resolved to settings.model at call time


# ── Task configuration registry ───────────────────────────────────────────────
# One entry per task type. Document the rationale for each parameter choice.
# When a parameter changes, change it here — not in every service file that uses it.

TASK_CONFIGS: dict[str, LLMConfig] = {
    "classification": LLMConfig(
        temperature=0.0,    # Deterministic — same ticket must return same label every run
        top_p=1.0,          # Irrelevant at temp=0 but explicit for clarity
        max_tokens=10,      # Output is a single label: "billing", "technical", etc.
        stop=["\n"],        # Stop after the label — prevents few-shot continuation
    ),
    "extraction": LLMConfig(
        temperature=0.0,    # JSON must be structurally stable across runs
        top_p=1.0,
        max_tokens=300,     # JSON object with ~5–7 fields + string values
        stop=[],            # No stop sequences — JSON structure handles termination
    ),
    "summarization": LLMConfig(
        temperature=0.4,    # Some variation is acceptable — summaries can be paraphrased
        top_p=0.9,          # Filter extreme tail; allow natural vocabulary variation
        max_tokens=150,     # 2–3 sentence target
        stop=[],
    ),
    "drafting": LLMConfig(
        temperature=0.7,    # Variation is desirable — different tone each run
        top_p=0.95,
        max_tokens=400,     # 1–2 paragraph reply
        stop=[],
    ),
}


def get_task_config(task: str) -> LLMConfig:
    """
    Return the LLMConfig for a task type with model resolved to settings.model
    if no task-specific model is set.

    Raises KeyError for unregistered tasks — never silently falls back.
    """
    if task not in TASK_CONFIGS:
        raise KeyError(
            f"No config registered for task {task!r}. "
            f"Available: {list(TASK_CONFIGS.keys())}"
        )
    cfg = TASK_CONFIGS[task]
    if cfg.model is None:
        return LLMConfig(
            temperature=cfg.temperature,
            top_p=cfg.top_p,
            max_tokens=cfg.max_tokens,
            stop=cfg.stop,
            model=settings.model,
        )
    return cfg
```

```python
# app/services/llm.py
"""
OpenAI client wrapper.

All service modules call chat_with_retry() — they do not construct the OpenAI
client or set parameters directly. This is the single place where the API client
is constructed, retry logic lives, and per-call token usage is logged.

Retry strategy, caching, and cost optimization are expanded in 2.15.
"""

import logging
import time

from openai import OpenAI, APIError, RateLimitError

from app.config import settings, LLMConfig
from app.utils.tokens import count_messages_tokens

logger = logging.getLogger(__name__)

# ── Client singleton ──────────────────────────────────────────────────────────
# Constructed once at module import. All service calls use this instance.
_client = OpenAI(api_key=settings.openai_api_key)


def chat_with_retry(
    messages: list[dict],
    config: LLMConfig,
    task: str = "unknown",
    retries: int = 2,
) -> str:
    """
    Call the chat completions API with task-specific parameters.

    Returns the assistant message content as a string.
    Retries up to `retries` times on rate limit or transient server errors.
    Logs prompt tokens, completion tokens, and task name on every call.

    Args:
        messages:  OpenAI message format — [{"role": "...", "content": "..."}, ...]
        config:    LLMConfig from get_task_config() — never hardcode params here
        task:      Task name for log correlation
        retries:   Retry attempts on transient errors (default 2)
    """
    prompt_tokens = count_messages_tokens(messages, model=config.model)

    for attempt in range(retries + 1):
        try:
            response = _client.chat.completions.create(
                model=config.model,
                messages=messages,
                temperature=config.temperature,
                top_p=config.top_p,
                max_tokens=config.max_tokens,
                stop=config.stop if config.stop else None,
            )

            content = response.choices[0].message.content or ""
            completion_tokens = response.usage.completion_tokens

            logger.info(
                "LLM call complete",
                extra={
                    "task": task,
                    "model": config.model,
                    "prompt_tokens": prompt_tokens,
                    "completion_tokens": completion_tokens,
                    "total_tokens": prompt_tokens + completion_tokens,
                    "temperature": config.temperature,
                },
            )

            return content

        except RateLimitError:
            if attempt < retries:
                wait = 2 ** attempt  # 1s then 2s
                logger.warning(
                    f"Rate limit hit (attempt {attempt + 1}/{retries + 1}), "
                    f"retrying in {wait}s",
                    extra={"task": task},
                )
                time.sleep(wait)
            else:
                raise

        except APIError as e:
            logger.error(f"API error on task {task!r}: {e}", extra={"task": task})
            raise
```

```python
# app/services/classify.py — minimal version
# Prompt is upgraded in 2.5 (few-shot examples + constraints from templates.py)

import logging

from app.config import get_task_config
from app.services.llm import chat_with_retry

logger = logging.getLogger(__name__)

# Minimal system prompt — upgraded to production quality in 2.5
CLASSIFY_SYSTEM_PROMPT = (
    "You are a support ticket classification assistant. "
    "Classify support tickets into exactly one of these categories: "
    "billing, technical, account, general. "
    "Return the category label only. No explanation."
)

_VALID_LABELS = {"billing", "technical", "account", "general"}


def classify_ticket(ticket_text: str) -> str:
    """
    Classify a support ticket into one of four categories.

    Returns the category label as a lowercase string.
    Falls back to "general" and logs a warning on unexpected model output.
    Prompt is upgraded with few-shot examples and constraints in 2.5.
    """
    config = get_task_config("classification")
    messages = [
        {"role": "system", "content": CLASSIFY_SYSTEM_PROMPT},
        {"role": "user", "content": ticket_text},
    ]
    raw = chat_with_retry(messages, config, task="classification")
    label = raw.strip().lower()

    if label not in _VALID_LABELS:
        logger.warning(
            "Unexpected classification label — falling back to 'general'",
            extra={"raw_output": label[:100]},
        )
        return "general"

    return label
```

```python
# app/routers/tickets.py — /classify endpoint

from fastapi import APIRouter, HTTPException
from pydantic import BaseModel

from app.services.classify import classify_ticket
from app.utils.tokens import count_tokens, fits_in_budget

router = APIRouter(prefix="/tickets", tags=["tickets"])

# Upper bound for a single ticket — anything longer gets rejected before the API call
_MAX_TICKET_INPUT_TOKENS = 2_000


class TicketRequest(BaseModel):
    text: str


class ClassifyResponse(BaseModel):
    category: str
    input_tokens: int  # Logged for cost monitoring


@router.post("/classify", response_model=ClassifyResponse)
def classify(request: TicketRequest) -> ClassifyResponse:
    """
    Classify a support ticket into billing, technical, account, or general.

    Returns the category label and input token count for cost monitoring.
    Rejects input over 2,000 tokens with a 422 before making an API call.
    """
    if not fits_in_budget(request.text, _MAX_TICKET_INPUT_TOKENS):
        raise HTTPException(
            status_code=422,
            detail=(
                f"Ticket text exceeds {_MAX_TICKET_INPUT_TOKENS} token limit. "
                "Please shorten the input."
            ),
        )

    category = classify_ticket(request.text)
    return ClassifyResponse(
        category=category,
        input_tokens=count_tokens(request.text),
    )
```

## Key Takeaways

- **Parameters are behavioral contracts, not settings.** Classification needs determinism. Drafting needs variance. Extraction needs brevity. Each task type encodes a different contract — register it in `TASK_CONFIGS` and never scatter parameter values across service files.

- **Temperature is a risk dial.** Low temperature (`0.0`–`0.2`) gives reproducibility at the cost of variety. High temperature gives variety at the cost of consistency. For any task that is parsed or dispatched automatically, use `temperature ≤ 0.2`.

- **Temperature reduces variance — it does not eliminate it.** At `temperature=0`, near-equal probability tokens can still produce variation across API calls due to floating-point and batching. If you need evaluation-level reproducibility, combine `temperature=0` with the `seed` parameter.

- **max_tokens is a cost and reliability control, not a suggestion.** Uncapped or over-generous `max_tokens` inflates cost on every call and reserves context window space you will never use. Set it per task, sized to actual expected output plus a 2× buffer.

- **Stop sequences are the completion boundary for few-shot patterns.** Without `stop=["\n"]` on a classification prompt, the model generates the next fake Input/Output example. The fix is one line. Add it by default whenever your prompt uses `Input:`/`Output:` patterns.

- **Top-p rarely needs tuning if temperature is set correctly.** Set temperature first. Use `top_p=0.9`–`0.95` as a production default and leave it there unless you observe bizarre low-probability tokens appearing in output.

- **Log token usage on every call from day one.** The `chat_with_retry()` wrapper logs `prompt_tokens`, `completion_tokens`, and task name on every call. This data drives cost optimization in 2.15 — you cannot optimize costs you have never measured.

## Optional Advanced Notes

**Seed parameter for evaluation reproducibility:** The OpenAI API exposes an integer `seed` parameter that makes sampling more repeatable across calls with the same input. Combined with `temperature=0`, seed produces near-identical outputs for the same input — useful for evaluation and regression testing (→ 2.16). Add it to `LLMConfig` and to `chat_with_retry()` when building the evaluation framework. It is not needed in the production serving path.

**Presence and frequency penalties:** The OpenAI API also exposes `presence_penalty` and `frequency_penalty` — scalars that reduce the probability of tokens based on whether they have already appeared in the output. Useful for suppressing repetitive phrasing in long-form generation. For the copilot's task types — short JSON, labels, brief summaries — these parameters are rarely needed. If you observe repetitive output in summarization or drafting, add them to `LLMConfig` before tuning temperature.

**Temperature behavior differs across model sizes:** `temperature=0.7` on GPT-4o produces different effective variance than the same value on GPT-4o-mini or Claude Sonnet. Larger models tend to have higher-confidence probability distributions (sharper peaks), so the same temperature value produces less variance on a stronger model. When switching models (→ 2.12), re-evaluate temperature settings — they do not transfer directly.
