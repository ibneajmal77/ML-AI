## CONCEPT BLOCK 1 — What Hallucination Is and Why It Happens

### A — The core idea in one sentence

Hallucination is the model producing fluent, confident output that is factually unsupported — not because something went wrong, but because next-token prediction has no gate that distinguishes "I have reliable evidence for this" from "this sounds like what would come next."

### B — Explain it plainly

When a human expert does not know something, they have a clear internal signal: uncertainty. They hedge, qualify, or say "I don't know." An LLM does not have this gate. At every step of generation, the model picks the most probable next token from its learned distribution. When the correct answer is strongly represented in training data, the most probable tokens are correct. When it is not — because the fact is post-cutoff, niche, proprietary, or simply not well-covered — the most probable tokens are still *something*. That something is statistically plausible, coherent with the surrounding text, and often formatted exactly like a correct answer. It is just wrong.

This is not a failure mode. It is the mechanism working as designed. The model optimizes for "what tokens are likely to follow what came before" — not for "is this true." These are different objectives, and the training process does not collapse them into one.

**Three root causes in production:**

**Knowledge gap.** The fact does not exist in training data — it is post-cutoff, internal to an organization, or from a niche domain with low representation in the training corpus. The model has no signal to draw on, so it completes from pattern.

**Context gap.** The fact exists somewhere in the world but was not provided in the prompt. The model is asked to extract or recall something that is not in the conversation. Without retrieval, it infers.

**Calibration failure.** The model assigns high confidence to inferences, not just to facts it has strong evidence for. Confidence in the output is determined by the smoothness of the probability distribution over tokens at generation time — not by the quality of the underlying knowledge.

**Four hallucination types that appear in production:**

**Factual fabrication** — the model invents a specific fact: a number, a date, a statistic. The invented value is plausible and formatted correctly. The contract penalty example is this type.

**Source fabrication** — the model invents a citation, URL, paper title, or named source. These look exactly like real citations. They do not exist.

**Attribute confusion** — the model mixes properties of two similar entities. It knows that Entity A has property X and Entity B has property Y, and assigns X to B or Y to A. Common with similar-sounding names, organizations, or products.

**Extrapolation** — the model extends a real fact beyond its actual scope. The contract says "30-day notice period for early termination"; the model extracts "30-day notice period" and applies it to renewal as well, which the contract does not say.

### D — Hallucination risk by task type

| Task type | Hallucination risk | Why |
|---|---|---|
| **Extraction from provided text** | Low | Source is in context; model copies or paraphrases |
| **Summarization of provided text** | Low-Medium | Source in context; risk is extrapolation and omission |
| **Classification of provided text** | Low | Bounded output; source in context |
| **Generation from knowledge** | High | No ground truth in context; model infers |
| **Specific facts (dates, numbers, names)** | High | Specificity requires precision the model may lack |
| **Citations and sources** | Very high | Model fabricates plausible-looking references |
| **Multi-hop reasoning** | Medium-High | Error compounds at each reasoning step |
| **Code generation** | Medium | Syntax correct; logic and API usage may be wrong |

### E — Where this fits in the broader architecture

Hallucination is not one problem — it is a family of problems with different root causes, risk profiles, and mitigations. Lesson 2.8 already applied one mitigation: tool calling grounds the model in real data before it decides. This lesson adds the detection and verification layer that operates *after* generation, catching the failures that grounding does not prevent.

### F — Why engineers get this wrong

The most common mistake is treating hallucination as a prompt engineering problem. Engineers add instructions like "only state facts you are certain of" or "do not fabricate information." These reduce hallucination rate marginally on some tasks. They do not eliminate it — the model has no reliable internal certainty signal to consult. The correct frame is architectural: put the source in the context (grounding), constrain the output space (structured outputs), and verify the output against the source (detection). Instructions are the last resort, not the first.

### G — Failure modes and their causes

| Failure | Root cause | Fix |
|---|---|---|
| Extraction returns plausible but wrong value | Source not in context; model generates from distribution | Provide source text in the prompt |
| Citations look real but don't exist | No factual source; model fabricates citation-shaped tokens | Never trust unverified citations; require source URL in the prompt |
| Model says "I am confident" before a wrong answer | Confidence language is generated like all other tokens | Treat all confidence language as noise; verify independently |
| Same fact, different value across two runs | Knowledge gap with no stable training signal | Self-consistency check; flag variance as hallucination signal |
| Extrapolation — claim goes beyond the source | Model completes pattern beyond what text says | Grounding check: verify claim is present in source text |

### H — Reading the model's response as a signal

High linguistic confidence ("As specified in the contract...", "The penalty is clearly stated as...") is not a signal of factual accuracy — it is a signal that the model generated confident-register tokens. In extraction tasks, if you see confident attribution language and you cannot verify the claim in the source, treat it as a hallucination risk regardless of how certain the model sounds. Conversely, hedging language ("typically", "generally", "approximately") is a weak signal the model is generating from distribution rather than from a specific source. Both observations inform risk assessment; neither is definitive.

### I — Senior signal

> "Hallucination is not random noise — it is structured. The model is more likely to hallucinate specific numbers than category labels. It is more likely to hallucinate citations than it is to hallucinate that a document exists. It is more likely to extrapolate from a nearby fact than to invent something with no connection to the source. Understanding the structure of hallucination lets you build targeted detection: check numerics against the source, check citations for existence, check extrapolations for textual support. A blanket 'did the model hallucinate?' check is less useful than three specific checks each targeting one hallucination type."

## CONCEPT BLOCK 2 — Hallucination Patterns and Detection Heuristics

### A — The core idea in one sentence

Different tasks have predictably different hallucination risk profiles, and the most useful detection heuristics are targeted — grounding checks for extraction, self-consistency for generation, and specific claim flagging for high-stakes output types.

### B — Explain it plainly

You cannot ask the model "did you hallucinate?" and trust the answer — the model has no reliable introspective access to whether its output is factually supported. Detection must be external: compare the output against a ground truth, compare multiple outputs against each other, or flag structural patterns associated with high hallucination risk.

Three heuristics that work in practice:

**Grounding check.** For extraction and summarization tasks — where a source document is in the prompt — verify that every factual claim in the output is supported by the source text. The simplest form: check whether the extracted value appears verbatim or near-verbatim in the source. The more sophisticated form: use a second LLM call to classify whether the claim is entailed by, contradicted by, or not mentioned in the source. This is NLI (Natural Language Inference) applied as a post-processing step.

**Self-consistency check.** Run the same prompt N times (typically 3–5) at non-zero temperature. Compare the factual claims across responses. If the model has a strong, stable factual basis — extraction from a provided source — the responses will agree. If the model is generating from a weak or uncertain distribution — knowledge-based generation — the responses will diverge on specific values. High variance across runs is a hallucination risk signal. Low variance is weak evidence of reliability (the model can be consistently wrong if the error is in the training distribution).

**Specific claim flagging.** Certain content types carry structurally higher hallucination risk regardless of task: exact numbers and percentages, specific dates, named citations and URLs, quotes attributed to specific people, and regulatory or legal references. A detection layer that flags these types for review — before writing to a database or presenting to a user — provides coverage without running full verification on every output.

### C — Concrete example with full working code

```python
import re
import json
import logging
from dataclasses import dataclass, field
from openai import AzureOpenAI

logger = logging.getLogger(__name__)


# ── Heuristic 1: Verbatim Grounding Check ────────────────────────────────────

def normalize(text: str) -> str:
    """Lowercase, collapse whitespace, strip punctuation for comparison."""
    text = text.lower()
    text = re.sub(r"[,.$%()'\"]", " ", text)   # remove common punctuation
    return " ".join(text.split())


def grounding_check(claim: str, source_text: str) -> dict:
    """
    Checks whether a claim (or its key terms) appears in the source text.

    Returns:
        is_grounded: bool — True if claim is verbatim-supported in source
        match_snippet: str | None — the matching text fragment if found
        normalized_claim: str — what was actually searched for
    """
    norm_source = normalize(source_text)
    norm_claim = normalize(claim)

    if norm_claim in norm_source:
        # Find the position and extract a small snippet for evidence
        start = norm_source.find(norm_claim)
        snippet = norm_source[max(0, start - 20): start + len(norm_claim) + 20]
        return {"is_grounded": True, "match_snippet": snippet, "normalized_claim": norm_claim}

    # Partial match: check if all numeric tokens in the claim appear in source
    # Catches "50,000" → "50000" normalization mismatches
    numbers_in_claim = re.findall(r"\d+", claim)
    numbers_in_source = re.findall(r"\d+", source_text)
    if numbers_in_claim and all(n in numbers_in_source for n in numbers_in_claim):
        return {"is_grounded": True, "match_snippet": None, "normalized_claim": norm_claim,
                "note": "numeric_partial_match"}

    return {"is_grounded": False, "match_snippet": None, "normalized_claim": norm_claim}


# ── Heuristic 2: LLM-as-Judge Grounding (NLI-style) ─────────────────────────

NLI_SYSTEM_PROMPT = """\
You are a fact-checking assistant. Given a source document and a claim,
determine whether the claim is supported by the source.

Respond with JSON only:
{
  "verdict": "supported" | "contradicted" | "not_mentioned",
  "evidence": "<exact quote from source that supports or contradicts, or null>",
  "confidence": <float 0.0–1.0>
}

Rules:
- "supported": the claim is directly stated or clearly entailed by the source
- "contradicted": the source explicitly states something that disagrees with the claim
- "not_mentioned": the source neither supports nor contradicts the claim
- Do not use your own knowledge — base your verdict only on the source provided."""


def llm_grounding_check(claim: str, source_text: str) -> dict:
    """
    Uses a second LLM call to classify whether the claim is supported by source.
    More accurate than verbatim matching for paraphrased or inferred claims.
    Use for high-stakes extractions where verbatim check is too strict.
    """
    response = client.chat.completions.create(
        model=DEPLOYMENT,
        temperature=0.0,
        max_tokens=200,
        response_format={"type": "json_object"},
        messages=[
            {"role": "system", "content": NLI_SYSTEM_PROMPT},
            {"role": "user", "content": (
                f"<source>\n{source_text}\n</source>\n\n"
                f"<claim>\n{claim}\n</claim>"
            )},
        ],
    )
    return json.loads(response.choices[0].message.content)


# ── Heuristic 3: Self-Consistency Check ──────────────────────────────────────

import asyncio
from openai import AsyncAzureOpenAI

async_client = AsyncAzureOpenAI(
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    api_key=os.environ["AZURE_OPENAI_API_KEY"],
    api_version="2024-02-01",
)


async def self_consistency_check(
    system_prompt: str,
    user_message: str,
    field_to_check: str,
    n: int = 3,
    temperature: float = 0.7,
) -> dict:
    """
    Runs the same prompt N times and checks if a specific field is consistent.
    High variance → hallucination risk signal.
    Low variance → weak evidence of stability (not proof of correctness).

    Returns:
        values: list of observed values across runs
        is_consistent: bool — all runs agreed on this field
        unique_values: int — number of distinct values seen
    """
    tasks = [
        async_client.chat.completions.create(
            model=DEPLOYMENT,
            temperature=temperature,
            max_tokens=200,
            response_format={"type": "json_object"},
            messages=[
                {"role": "system", "content": system_prompt},
                {"role": "user", "content": user_message},
            ],
        )
        for _ in range(n)
    ]
    responses = await asyncio.gather(*tasks)

    values = []
    for r in responses:
        try:
            data = json.loads(r.choices[0].message.content)
            values.append(str(data.get(field_to_check, "MISSING")))
        except json.JSONDecodeError:
            values.append("PARSE_ERROR")

    unique = set(values)
    return {
        "values": values,
        "is_consistent": len(unique) == 1,
        "unique_values": len(unique),
        "field_checked": field_to_check,
    }


# ── Heuristic 4: Specific Claim Flagging ─────────────────────────────────────

# Patterns that indicate high-risk specific claims
HIGH_RISK_PATTERNS = [
    (re.compile(r"\$[\d,]+(?:\.\d{2})?"), "monetary_value"),
    (re.compile(r"\b\d{1,3}(?:,\d{3})*(?:\.\d+)?\s*(?:%|percent)\b", re.I), "percentage"),
    (re.compile(r"\b\d{4}-\d{2}-\d{2}\b"), "iso_date"),
    (re.compile(r"\b(?:January|February|March|April|May|June|July|August|"
                r"September|October|November|December)\s+\d{1,2},?\s+\d{4}\b"), "written_date"),
    (re.compile(r"https?://\S+"), "url"),
    (re.compile(r'"[^"]{10,}"'), "quoted_text"),
    (re.compile(r"\b(?:section|clause|article)\s+\d+(?:\.\d+)*\b", re.I), "document_reference"),
]


def flag_specific_claims(text: str) -> list[dict]:
    """
    Scans output text for high-risk content types that warrant verification.
    Returns a list of flagged items with their type and value.
    """
    flags = []
    for pattern, claim_type in HIGH_RISK_PATTERNS:
        for match in pattern.finditer(text):
            flags.append({
                "type": claim_type,
                "value": match.group(),
                "position": match.start(),
            })
    return flags


# ── Combined extraction with detection layer ─────────────────────────────────

@dataclass
class ExtractionResult:
    data: dict
    is_grounded: bool
    grounding_details: dict
    high_risk_flags: list[dict]
    requires_review: bool
    raw_output: str


def extract_with_grounding(
    source_text: str,
    system_prompt: str,
) -> ExtractionResult:
    """
    Runs extraction then immediately applies grounding check and claim flagging.
    Adds is_grounded and requires_review to every extraction result.
    """
    response = client.chat.completions.create(
        model=DEPLOYMENT,
        temperature=0.0,
        max_tokens=400,
        response_format={"type": "json_object"},
        messages=[
            {"role": "system", "content": system_prompt},
            {"role": "user", "content": f"<source>\n{source_text}\n</source>"},
        ],
    )
    raw = response.choices[0].message.content
    data = json.loads(raw)

    # Grounding check: verify all string values appear in source
    ungrounded_fields = []
    for field_name, value in data.items():
        if value is None:
            continue
        check = grounding_check(str(value), source_text)
        if not check["is_grounded"]:
            ungrounded_fields.append(field_name)
            logger.warning(
                "Grounding check failed: field=%s value=%r not found in source",
                field_name, value,
            )

    is_grounded = len(ungrounded_fields) == 0
    flags = flag_specific_claims(raw)

    return ExtractionResult(
        data=data,
        is_grounded=is_grounded,
        grounding_details={"ungrounded_fields": ungrounded_fields},
        high_risk_flags=flags,
        requires_review=not is_grounded or len(flags) > 0,
        raw_output=raw,
    )
```

### D — Detection heuristic comparison

| Heuristic | What it catches | Cost | Precision | Best for |
|---|---|---|---|---|
| **Verbatim grounding check** | Extracted values not in source text | Negligible (string search) | High for exact claims; misses paraphrase | Numeric extraction, date extraction |
| **LLM-as-judge grounding** | Paraphrased or inferred claims not supported by source | 1 extra API call | High | Legal, compliance, high-stakes extraction |
| **Self-consistency check** | Knowledge-based generation with unstable output | N extra API calls | Catches variance, not systematic errors | Facts without a source document |
| **Specific claim flagging** | High-risk content types in any output | Negligible (regex) | Catches type, not correctness | Routing to human review |

### E — Where this fits in the broader architecture

Detection heuristics are a middle layer between generation and delivery. Generation produces output; detection classifies it; the calling code decides what to do with flagged results — route to human review, retry with a grounding-focused prompt, or surface a confidence flag to the user. The `ExtractionResult.requires_review` flag in the code above is that decision signal. It does not block delivery — it routes delivery. This is important: a detection layer that blocks all flagged output creates more friction than it prevents.

### F — Why engineers get this wrong

Two mistakes. First: running grounding checks on fields that cannot be grounded. Boolean flags (`is_paid`, `has_arbitration_clause`) may not appear verbatim in the source — they are inferences. Applying verbatim grounding to inference fields produces false positives that destroy trust in the detection layer. Apply verbatim grounding only to specific factual claims: numbers, dates, names, quotes. Apply LLM-as-judge grounding for inferences. Second: using self-consistency as proof of correctness. If the model is consistently wrong — e.g., it learned a wrong fact during training — all N runs will agree on the wrong value. Self-consistency is evidence of stability, not accuracy. It is most useful for flagging *instability* as a risk signal, not for confirming *correctness*.

### G — Failure modes and their causes

| Failure | Root cause | Fix |
|---|---|---|
| Grounding check flags inferred fields as ungrounded | Verbatim check applied to inference-type fields | Apply verbatim grounding only to numeric, date, and quoted fields |
| Self-consistency shows all runs agree but value is wrong | Systematic error in training distribution | Self-consistency cannot fix systematic errors — pair with source grounding |
| Claim flagging produces too many false positives | Pattern too broad, flagging every number | Tune patterns to domain; flag only high-precision risk types |
| LLM judge cost is prohibitive at scale | Every extraction runs a second LLM call | Apply LLM judge only to high-stakes fields; use verbatim check for others |
| Detection layer blocks all uncertain output | `requires_review` used as a hard gate | Use `requires_review` as a routing signal, not a block — route to human, not to rejection |

### H — Reading the model's response as a signal

When the verbatim grounding check fails on a numeric field, there are three possibilities: the model hallucinated the number (most dangerous), the number exists in the source but was formatted differently (e.g., "$50,000" vs "50000"), or the model correctly paraphrased a calculation (e.g., "total: $150,000" when the source says "3 items at $50,000 each"). Before routing to human review, run a normalized numeric check — extract all digit sequences from both the claim and the source and compare. This reduces false positives from formatting differences while still catching true fabrications.

### I — Senior signal

> "Detection heuristics are not a replacement for getting the generation right. They are a safety net for the failures that good grounding and prompt design do not prevent. We apply the cheapest check first — verbatim grounding, which is just a string search — and escalate to LLM-as-judge only when the verbatim check fails on a high-stakes field. We track the ratio of grounding failures to total extractions as a quality metric. If it climbs above our threshold, that is a prompt regression signal, not an infrastructure issue."

## CONCEPT BLOCK 3 — Mitigation Strategies: Three Tiers

### A — The core idea in one sentence

Hallucination mitigation works in three tiers — ground the model with real data so it does not need to infer, constrain the output space so it cannot express unsupported claims, and verify the output after generation to catch what the first two tiers miss.

### B — Explain it plainly

No single mitigation eliminates hallucination. The three-tier approach applies defenses in sequence, each catching what the previous tier did not prevent.

**Tier 1 — Grounding: put the source in the context.**
The most effective mitigation. If the fact the model needs is in the prompt, it can copy or paraphrase from the source rather than generating from the training distribution. Two mechanisms: RAG (retrieval-augmented generation — retrieve the relevant documents and inject them into the prompt before generating) and tool calling (use a tool to fetch real data at generation time rather than asking the model to recall it from weights). Grounding does not eliminate hallucination — the model can still extrapolate beyond the source or confuse two nearby facts — but it dramatically reduces the rate for knowledge-dependent tasks.

**Tier 2 — Constraining the output space.**
Reducing the degrees of freedom in the model's output reduces the opportunity for hallucination. Structured outputs (Lesson 2.7) force the output into a schema — the model cannot produce a fabricated citation if the schema has no field for it. Enums for constrained choices (Lesson 2.8) prevent the model from inventing string values. System prompt constraints — "If a field is not present in the document, set it to null. Do not infer or estimate values." — reduce extrapolation. `temperature=0` reduces run-to-run variance (not hallucination rate, but consistency). None of these are perfect. Each reduces the problem surface.

**Tier 3 — Verification: check the output after generation.**
Even with grounding and constraints, some outputs will be wrong. Verification is the last layer — grounding checks, LLM-as-judge, human review queues, and downstream validation. Concept Block 2 covers the detection heuristics. This tier is where you decide what to do with flagged outputs: retry with an explicit correction prompt, route to human review, or surface a confidence flag to the user. The key principle: a flagged output is not automatically rejected. It is routed appropriately based on the business cost of a wrong answer.

### C — Concrete example with full working code

```python
import json
import logging
from pydantic import BaseModel, field_validator
from typing import Literal

logger = logging.getLogger(__name__)


# ── Tier 2: Constrained schema + null-for-missing system prompt ──────────────

EXTRACTION_SYSTEM_PROMPT = """\
## Role
You are a contract extraction assistant.

## Task
Extract the fields listed in the Output format section from the contract clause
provided by the user.

## Output format
Return a JSON object with exactly these fields:
  "penalty_per_day_usd"  : number | null   — the per-day penalty amount in USD
  "cure_period_days"     : number | null   — the number of days in the cure period
  "cure_period_unit"     : "calendar" | "business" | null — type of days
  "applies_to"           : string | null   — what event triggers the penalty

## Constraints
- If a field is not explicitly stated in the clause, set its value to null.
- Do not infer, estimate, or calculate values not directly present in the text.
- Do not extrapolate from similar clauses you may know from other contracts.
- Use null, not 0 or "unknown", for missing fields."""


class ContractExtraction(BaseModel):
    """
    Tier 2 constraint: required fields + enum for cure_period_unit.
    Model cannot invent a new value for cure_period_unit — only 3 options.
    """
    penalty_per_day_usd: float | None = None
    cure_period_days: int | None = None
    cure_period_unit: Literal["calendar", "business"] | None = None
    applies_to: str | None = None

    @field_validator("penalty_per_day_usd")
    @classmethod
    def validate_positive_penalty(cls, v: float | None) -> float | None:
        if v is not None and v <= 0:
            raise ValueError(f"penalty_per_day_usd must be positive, got {v}")
        return v

    @field_validator("cure_period_days")
    @classmethod
    def validate_positive_days(cls, v: int | None) -> int | None:
        if v is not None and v <= 0:
            raise ValueError(f"cure_period_days must be positive, got {v}")
        return v


# ── Tier 1 + Tier 3: Grounding + Verification pipeline ──────────────────────

def extract_contract_clause(clause_text: str) -> dict:
    """
    Full mitigation pipeline:
      Tier 1: source in prompt (model extracts from text, not from memory)
      Tier 2: Pydantic schema + constrained prompt (limits output space)
      Tier 3: grounding check + requires_review flag (detects unsupported claims)
    """
    # Tier 1: source is in the prompt — model extracts, does not recall
    response = client.chat.completions.create(
        model=DEPLOYMENT,
        temperature=0.0,                             # Tier 2: reduce variance
        max_tokens=200,
        response_format={"type": "json_object"},     # Tier 2: force JSON
        messages=[
            {"role": "system", "content": EXTRACTION_SYSTEM_PROMPT},
            {"role": "user", "content": f"<clause>\n{clause_text}\n</clause>"},
        ],
    )
    raw = response.choices[0].message.content

    # Tier 2: Pydantic validation — catches type errors, invalid enums
    from pydantic import ValidationError
    try:
        extraction = ContractExtraction.model_validate_json(raw)
    except ValidationError as e:
        logger.error("Extraction schema validation failed: %s | raw: %r", e, raw)
        return {
            "data": None,
            "is_grounded": False,
            "requires_review": True,
            "error": str(e),
        }

    # Tier 3: grounding check — verify numeric values appear in source
    ungrounded = []
    for field_name in ["penalty_per_day_usd", "cure_period_days"]:
        value = getattr(extraction, field_name)
        if value is not None:
            check = grounding_check(str(int(value)), clause_text)
            if not check["is_grounded"]:
                ungrounded.append(field_name)
                logger.warning(
                    "Grounding check failed: field=%s extracted=%s not in source",
                    field_name, value,
                )

    # Tier 3: high-risk claim flagging
    flags = flag_specific_claims(raw)

    is_grounded = len(ungrounded) == 0
    requires_review = not is_grounded

    logger.info(
        "Extraction complete: is_grounded=%s ungrounded=%s flags=%d requires_review=%s",
        is_grounded, ungrounded, len(flags), requires_review,
    )

    return {
        "data": extraction.model_dump(),
        "is_grounded": is_grounded,
        "grounding_details": {"ungrounded_fields": ungrounded},
        "high_risk_flags": flags,
        "requires_review": requires_review,
        "raw_output": raw,
    }


# --- Test on the contract snippet from Concept Block 1 ---

result = extract_contract_clause(CONTRACT_SNIPPET)
print(json.dumps(result, indent=2))
```

### D — Mitigation tier comparison

| Tier | Mechanism | What it prevents | What it misses | Cost |
|---|---|---|---|---|
| **Tier 1 — Grounding** | Source in prompt; RAG; tool calling | Knowledge-gap hallucination; context-gap hallucination | Extrapolation; attribution confusion within the source | Token cost of injecting source |
| **Tier 2 — Constraining** | Structured output; enums; null-for-missing prompt; `temperature=0` | Output format fabrication; invented string values; variance | Factual errors within constrained fields | Negligible |
| **Tier 3 — Verification** | Grounding check; LLM-as-judge; human review | Extrapolation; miscopy from source; calibration failure | Systematic errors in training data | API call (LLM judge) or reviewer time |

### E — Where this fits in the broader architecture

The three-tier structure applies to every LLM feature, not just extraction. For a summarization endpoint (Lesson 2.9), Tier 1 is the document in the prompt, Tier 2 is a max-length constraint and null-for-missing instruction, and Tier 3 is a post-generation faithfulness check — does the summary make claims not present in the source? For a tool-calling agent (Lesson 2.8), Tier 1 is the tool that retrieves real data, Tier 2 is the tool parameter schema with enums, and Tier 3 is argument validation before the handler executes. The tiers are always present; only their implementation changes per task type.

### F — Why engineers get this wrong

The most common mistake is skipping to Tier 2 without Tier 1. Engineers add structured outputs and null-for-missing constraints, measure a reduced hallucination rate, and consider the problem solved. What they measured is reduced format-level hallucination — the model no longer invents string values or wrong key names. But factual hallucination in the values themselves is not reduced by schema enforcement. Pydantic cannot check whether "$5,000" is the right number. Only grounding can. Tier 2 without Tier 1 produces correctly formatted wrong answers.

A second mistake: treating Tier 3 as the primary mitigation. Verification catches failures — it does not reduce them. Relying on human review as the main defense creates a review bottleneck and trains the team to expect a high error rate rather than engineering it down. The correct sequence: minimize failures with Tier 1 and Tier 2, then verify the residual with Tier 3.

### G — Failure modes and their causes

| Failure | Root cause | Fix |
|---|---|---|
| Correctly formatted but wrong value | Tier 2 only; no grounding | Add Tier 1 — source in prompt |
| Grounding added but model still extrapolates | Source in context but model extends beyond it | Add Tier 3 — verify claim against source |
| Human review queue overwhelmed | Tier 3 flags too much; routing calibration wrong | Tune detection thresholds; apply LLM judge only to high-stakes fields |
| Null-for-missing instruction ignored | Instruction buried at end of long system prompt | Move null constraint to output format section near the top; repeat inline |
| Model invents values despite null instruction | Model fills null with hallucinated value | Add `temperature=0`; use `enum` where possible; add Pydantic required fields |

### H — Reading the model's response as a signal

When Tier 3 grounding checks consistently fail on the same field type across many documents, the failure is not random — it is systematic. Investigate: is that field type rarely stated explicitly in the source (making null the correct answer most of the time)? Is the model being asked to infer something that cannot be extracted? If so, the field definition is wrong — remove it, or change the constraint to "set to null unless the exact value is stated verbatim." Persistent grounding failures on a specific field are a specification problem, not a model problem.

### I — Senior signal

> "The three-tier structure is how I explain hallucination mitigation to stakeholders. Tier 1 is architectural — we ground the model so it does not have to guess. That alone cuts hallucination rate by 60–80% on knowledge-dependent tasks. Tier 2 is constraint — we limit what the model can express. That cuts another 10–15% by removing the degrees of freedom hallucination needs. Tier 3 is verification — we catch the residual. If our Tier 3 review queue is growing, that is a signal that Tier 1 or Tier 2 needs strengthening, not that we need more reviewers."

## DECISION RULES

```
IF the task requires a specific fact (date, number, name, citation)
→ apply Tier 1: put the source document in the prompt
→ do NOT ask the model to recall facts from training weights
→ if source is not available at inference time: use RAG (Stage 3) or tool calling (Lesson 2.8)

IF the output is extraction from a provided source
→ add "If not present, set to null. Do not infer or estimate." to the system prompt
→ make required fields truly required in the Pydantic schema (no default = None)
→ run verbatim grounding check on all numeric and date fields after extraction

IF the output contains specific numbers, dates, citations, or quotes
→ flag them with specific claim detection regardless of task type
→ these content types have structurally higher hallucination risk
→ route flagged outputs to human review or LLM-as-judge verification

IF grounding check fails on a numeric field
→ first check: is it a formatting mismatch ("$50,000" vs "50000")? — normalize and recheck
→ if still fails: route to human review — do not silently write to database
→ log: field name, extracted value, source text — preserve evidence for diagnosis

IF grounding checks consistently fail on the same field type
→ this is a specification problem, not a model problem
→ investigate: is this field type rarely explicit in the source?
→ fix: change field to null-for-missing, or remove it from the schema

IF running self-consistency check on a knowledge-based task
→ use temperature 0.5–0.8 to get genuine variance (temperature=0 gives consistent output)
→ high variance (different values across runs) → hallucination risk signal; route to review
→ low variance is NOT proof of correctness — the model can be consistently wrong

IF the model produces confident-sounding attribution language ("as stated in...", "clearly...")
→ treat it as noise, not as evidence
→ verify the claim against the source independently of what the model says about it

IF human review queue is growing faster than capacity
→ the detection threshold is too broad, OR Tier 1/2 mitigations need strengthening
→ tighten detection: apply LLM judge only to high-stakes fields, not all fields
→ investing in better grounding (Tier 1) reduces the volume that reaches Tier 3

IF cost of LLM-as-judge verification is prohibitive
→ apply it selectively: only to ungrounded fields from the verbatim check
→ or: apply to a sampled percentage (e.g., 5%) for quality monitoring rather than every call
→ verbatim grounding check + claim flagging covers most production risk at near-zero cost
```

## COMMON MISTAKES

### Mistake 1: Using prompt instructions as the primary mitigation

**What it looks like:**
```python
system_prompt = """
You are a precise extraction assistant.
Only state facts you are certain of.
Do not fabricate or hallucinate information.
If you are unsure, say "unknown."
"""
```

**Why it happens:** Telling the model not to hallucinate feels like the obvious first step, and it does reduce hallucination rate slightly on some benchmarks.

**What goes wrong:** The model has no reliable internal certainty signal. The instruction "only state facts you are certain of" cannot be followed in the way a human would follow it — the model does not have a "certainty check" to run before generating each token. The instruction reduces verbose speculation but does not prevent confident-sounding fabrication of specific values. A model that returns `"penalty_per_day_usd": 5000` when the source says 50,000 is following a plausible pattern — not ignoring the instruction.

**The fix:** Instructions are Tier 2, not a replacement for Tier 1. Start with the source in the prompt. Let instructions handle edge cases after the main mitigation is in place.

---

### Mistake 2: Treating schema validation as hallucination prevention

**What it looks like:**
```python
# "We have Pydantic validation — hallucination is handled"
class ExtractedData(BaseModel):
    penalty_per_day_usd: float
    cure_period_days: int
    cure_period_unit: Literal["calendar", "business"]
```

**Why it happens:** Pydantic validation catches wrong types and invalid enums. It visibly reduces one class of error. Engineers conclude the hallucination problem is solved.

**What goes wrong:** Pydantic validates *format*, not *truth*. `penalty_per_day_usd: 5000.0` passes Pydantic. It is also wrong by a factor of 10. The `Literal["calendar", "business"]` constraint prevents `"business days"` — but it cannot prevent the model from choosing `"calendar"` when the source says `"business"`. Schema enforcement eliminates structural hallucination. It does nothing for value-level hallucination.

**The fix:** Pair schema validation (Tier 2) with grounding checks (Tier 3). After Pydantic validates the shape, verify that the numeric values appear in the source text.

---

### Mistake 3: Applying verbatim grounding to all field types

**What it looks like:**
```python
# Checking whether "has_arbitration_clause: true" appears in the source
check = grounding_check("true", source_text)
# → always fails because "true" is a Pydantic bool, not in the text
```

**Why it happens:** Engineers apply grounding checks uniformly to all extracted fields, including booleans and inferences.

**What goes wrong:** Boolean flags and inferred fields are not verbatim — they are conclusions drawn from the text. "Arbitration is required under this agreement" supports `has_arbitration_clause: true`, but "true" does not appear in the source. The grounding check always fails, `requires_review` is always true, and the human review queue is flooded with correct extractions. Trust in the detection layer is destroyed.

**The fix:** Apply verbatim grounding only to specific factual claims: numbers, dates, monetary values, quoted text, and named references. Apply LLM-as-judge for inferences. Do not apply grounding to boolean or categorical fields.

---

### Mistake 4: Using `temperature=0` as a hallucination fix

**What it looks like:**
```python
# "Temperature zero means the model is deterministic and accurate"
response = client.chat.completions.create(
    model=DEPLOYMENT,
    temperature=0.0,   # "fixes" hallucination
    ...
)
```

**Why it happens:** `temperature=0` makes the output deterministic and often more precise-sounding. Engineers observe fewer obviously wrong answers and attribute this to accuracy.

**What goes wrong:** `temperature=0` reduces variance — the model always produces the same output for the same input. If the model hallucinated a value at `temperature=0.7`, it will still hallucinate the same value at `temperature=0`. Determinism is not accuracy. The model's most-probable token sequence at `temperature=0` is its best guess — not a verified fact. The hallucination rate on knowledge-based tasks is nearly identical at `temperature=0` and `temperature=0.3`.

**The fix:** Use `temperature=0` for consistency in extraction tasks — it removes run-to-run variance which is separately undesirable. But do not treat it as a hallucination mitigation. The grounding and verification tiers are required regardless of temperature.

Answer from memory. Check after.

**Q1:** Name the three root causes of hallucination and give one example of each.

*Check:* Knowledge gap — the fact is not in training data (e.g., a post-cutoff event, an internal company process). Context gap — the fact exists but is not in the prompt (e.g., asking the model to recall a specific contract value without providing the contract). Calibration failure — the model expresses high confidence on inferred output it has no factual basis for (e.g., "the penalty is $5,000 per day" stated confidently when no source was provided).

---

**Q2:** A grounding check on a numeric field returns `is_grounded: False`. Before routing to human review, what two alternative explanations should you check, and how?

*Check:* First — formatting mismatch: the value exists in the source but in a different format (e.g., model returns `50000` but source has `$50,000`). Fix: normalize both by extracting digit sequences and comparing. Second — paraphrase or calculation: the source says "3 items at $50,000 each" and the model correctly computed `total: 150000`. Fix: apply LLM-as-judge grounding check, which handles entailment, not just verbatim match. Only route to human review after both alternatives are ruled out.

---

**Q3:** What is the difference between self-consistency confirming a value and that value actually being correct?

*Check:* Self-consistency (all N runs produce the same value) only confirms the output is stable — not that it is accurate. The model can be consistently wrong if the error is a stable pattern in the training distribution. For example, if the model learned to associate "SaaS enterprise contract" with "$5,000 per day penalty," it will produce that value consistently across all runs at any temperature. Self-consistency is useful for flagging instability (high variance = hallucination signal), but low variance does not confirm correctness.

---

**Q4:** You are reviewing a system prompt for a contract extraction endpoint. It contains: "Do not hallucinate. Only extract information explicitly stated in the document." Where does this instruction fit in the three-tier mitigation hierarchy, and what is missing?

*Check:* This is a Tier 2 constraint — it attempts to limit what the model expresses. It is not a replacement for Tier 1 (the source document must be in the prompt for the model to extract from rather than recall) or Tier 3 (grounding checks verify that extracted values actually appear in the source). The instruction alone does not prevent value-level hallucination. What is missing: Tier 1 confirmation that the full source text is injected into the user message, and Tier 3 grounding checks on numeric and date fields after extraction.

---

**Q5:** A senior engineer describes their hallucination mitigation approach: "We run the LLM-as-judge on every single extracted field across all documents." What is wrong with this approach architecturally, and how would you redesign it?

*Check:* LLM-as-judge costs one extra API call per field check — on large-volume extraction this is prohibitive and the cost scales with document volume, not with actual risk. It also adds latency to every extraction regardless of whether the output is high-risk. Redesign: apply a cheap verbatim grounding check first (string search, negligible cost) on all numeric and date fields; apply LLM-as-judge only to fields that fail the verbatim check — because those are the ones where paraphrase or entailment needs to be evaluated. For low-stakes fields, apply LLM-as-judge on a sampled percentage (e.g., 5%) for quality monitoring rather than every call.

## SUMMARY

Hallucination is not a model defect to be fixed by the next model release — it is a structural consequence of next-token prediction operating without a factual verification gate. Every production LLM system must architect around it.

**The root cause** is simple: the model produces statistically plausible tokens. When the correct answer has strong training signal, the plausible token is correct. When it does not — knowledge gap, context gap, calibration failure — the most plausible token is still something, not silence.

**Detection heuristics** work best when targeted: verbatim grounding for numerics and dates (string search, negligible cost), LLM-as-judge for paraphrased claims (one extra API call), self-consistency for knowledge-based generation (N calls, catches instability), specific claim flagging for routing (regex). Do not apply the expensive heuristic uniformly — apply it where the cheap heuristic flags a problem.

**The three-tier mitigation hierarchy** is the engineering answer: Tier 1 grounds the model in real data so it extracts rather than infers; Tier 2 constrains the output space so it cannot express unsupported claims; Tier 3 verifies the residual and routes failures to the right destination. Each tier is necessary. None is sufficient alone. Tier 2 without Tier 1 produces correctly formatted wrong answers.

**The monitoring frame:** track grounding failure rate per field type as a quality metric. A rising rate on a specific field is a specification problem — the field definition is asking the model to infer something that cannot be extracted. Fix the specification, not the detection threshold.

### Interview questions this lesson prepares you for

**Q: Why do LLMs hallucinate, and what is the correct engineering response?**

> "Hallucination is structural — it is not a bug but a consequence of how next-token prediction works. The model picks the most probable token at each step. When the correct answer has strong training signal, the most probable token is right. When it does not — knowledge gap, context gap, calibration failure — the most probable token is still something plausible-sounding. There is no internal 'I don't know' gate. The engineering response is architectural, not prompt-based. Three tiers: ground the model with real data so it extracts rather than infers (RAG, tool calling), constrain the output space so it cannot express unsupported claims (schema, enums, null-for-missing), and verify the output after generation to catch the residual (grounding checks, LLM-as-judge, human review routing)."

**Q: How do you detect hallucination in production without manually reviewing every output?**

> "Automated detection in layers by cost. First layer: verbatim grounding check — string search to verify that extracted numeric and date values appear in the source text. Negligible cost, catches the most dangerous class of extraction error. Second layer: specific claim flagging — regex scan for monetary values, percentages, dates, URLs, and quoted text in any output. These content types have structurally higher hallucination risk and get routed to the next layer. Third layer: LLM-as-judge grounding — a second API call that classifies whether the claim is supported, contradicted, or not mentioned in the source. Applied only to flagged fields from layer two, not uniformly. We track grounding failure rate per field type as a quality metric — a rising rate on a specific field is a specification problem, not a model problem."

**Q: What is the difference between temperature=0 and hallucination prevention?**

> "`temperature=0` makes the output deterministic — the model always produces the same most-probable token sequence for the same input. It does not reduce hallucination rate. The model's most-probable completion at `temperature=0` is its best guess — not a verified fact. If the model hallucinated a value in its training distribution, it will produce that same hallucinated value consistently at `temperature=0`. We use `temperature=0` for extraction tasks to eliminate run-to-run variance, which is separately undesirable. But the grounding and verification layers are required regardless of temperature setting."

### Key vocabulary

| Term | Definition |
|---|---|
| **Hallucination** | LLM output that is fluent and confident but factually unsupported — a structural property of next-token prediction |
| **Knowledge gap** | Hallucination cause: the fact does not exist in training data (post-cutoff, niche, proprietary) |
| **Context gap** | Hallucination cause: the fact exists but was not provided in the prompt |
| **Calibration failure** | Hallucination cause: the model expresses high confidence on inferences it has no factual basis for |
| **Factual fabrication** | Inventing a specific fact: number, date, statistic — formatted correctly but wrong |
| **Source fabrication** | Inventing citations, URLs, or named sources that do not exist |
| **Attribute confusion** | Mixing properties of two similar entities (assigning Entity A's property to Entity B) |
| **Extrapolation** | Extending a real fact beyond its actual scope in the source text |
| **Grounding** | Tier 1 mitigation: providing source material in the prompt so the model extracts rather than infers |
| **Verbatim grounding check** | Tier 3 detection: verifying extracted values appear in the source text (string search) |
| **LLM-as-judge** | Using a second LLM call to classify whether a claim is supported, contradicted, or not mentioned |
| **Self-consistency check** | Running the same prompt N times and comparing factual claims — high variance signals hallucination risk |
| **NLI (Natural Language Inference)** | Classification task: does text A entail, contradict, or not mention text B? Used in LLM-as-judge pattern |
| **`requires_review`** | Routing flag on extraction output — signals the result needs verification before use, not rejection |
