## Where This Fits in the System

**State:** After 2.7, the copilot can classify, extract, summarize, and draft with typed validated output. Every response is text: a category label, a structured extraction, a summary. The model has no mechanism to retrieve real data — account plans, open ticket counts, customer history — before making a decision. When context-dependent routing decisions are made without real data, the model fills the gap by inferring from training distributions. That inference is sometimes right and frequently wrong in ways that look confident.

**Gap:** The `/classify` endpoint routes tickets into categories using only the ticket text. For tickets where routing accuracy depends on account state — "is this customer on a paid plan?", "how many open tickets do they already have?", "is this the third occurrence of this issue?" — the model has no mechanism to get that information. It either routes without it (low accuracy on context-dependent tickets) or hallucinates it (confident but fabricated).

**Unlocks:** The `/route` endpoint, built in this lesson, gives the model two real tools: one to fetch customer context from the database, one to write a routing decision. The model calls them in sequence. The pipeline produces verified routing decisions backed by real account data, not inference. 2.11 extends the handler layer with injection detection and audit logging.

## Core Mental Model

### Intent → Execution → Result

The most important thing to understand about function calling is what the model actually does and does not do:

**The model does not execute code. It produces a structured intent.**

When you define tools and include them in an API call, the model reads the conversation and decides: "I need to call `get_customer_context` with `{"customer_id": "C-1042"}` before I can answer this." It then outputs a structured JSON object describing that intent — and stops. Your code reads the intent, executes the actual function, gets the real result, and returns that result to the model. The model then produces a final answer using the real data.

This is not an implementation detail. It is the design. The model cannot access your database, your APIs, or your file system directly. It can only describe what it wants to call and what arguments to pass. Every execution happens in code you write and control.

```
1st API call  →  finish_reason="tool_calls"
                 model produces: {name: "get_customer_context", args: {customer_id: "C-1042"}}
                                   ↓
                   YOUR CODE executes get_customer_context("C-1042")
                   returns: {plan: "growth", account_age: 1247, open_tickets: 2}
                                   ↓
2nd API call  →  finish_reason="stop"
                 model produces final response using the real data
```

The clean boundary between "model describes intent" and "code executes intent" is why tool-calling systems are debuggable and auditable. When something goes wrong, you can inspect exactly what the model asked for, what your code returned, and how the model used the result.

## Main Concepts + Production Reality

### Concept 1 — The Two-Step Turn Cycle

**The turn structure:**

A standard API call: one request, one response, done. A tool-enabled call adds a step when the model requests a tool.

```python
# Step 1: first API call — model may or may not request a tool
response = client.chat.completions.create(
    model=DEPLOYMENT,
    messages=messages,
    tools=tools,
    tool_choice="auto",
)

finish_reason = response.choices[0].finish_reason

# finish_reason="stop"      → model answered directly, no tool needed
# finish_reason="tool_calls" → model produced a tool call intent
```

When `finish_reason="tool_calls"`, the response contains `message.tool_calls` — a list of structured tool call objects. Each has:
- `tc.id` — a unique ID for this tool call (you must echo it back)
- `tc.function.name` — the tool to call
- `tc.function.arguments` — a JSON **string** (not a dict) of the arguments

```python
# Step 2: execute the tool
tc = response.choices[0].message.tool_calls[0]
args = json.loads(tc.function.arguments)   # always parse — it's a string
result = your_function(**args)

# Step 3: inject the result and call again
messages.append(response.choices[0].message)   # assistant turn with tool_calls first
messages.append({
    "role": "tool",
    "tool_call_id": tc.id,                     # must match exactly
    "content": json.dumps(result),             # content must be a string
})

# Step 4: second API call — model sees the result and produces the final answer
final = client.chat.completions.create(
    model=DEPLOYMENT,
    messages=messages,
    tools=tools,
)
# → finish_reason="stop" → final.choices[0].message.content is the answer
```

**Two sequencing rules that the API enforces:**

1. The assistant message with `tool_calls` must be appended **before** any tool messages. Appending tool messages first causes an API error.
2. Every `tool_call_id` in a tool message must exactly match the `id` of a `tool_calls` entry in the preceding assistant message. A mismatch causes an API error.

These are not soft conventions. The API validates the conversation structure and rejects malformed sequences.

**`finish_reason` outcomes to handle:**

| `finish_reason` | Meaning | Action |
|---|---|---|
| `"stop"` | Model answered; no tool needed or tool loop complete | Return `message.content` |
| `"tool_calls"` | Model produced one or more tool call intents | Execute all, inject results, call again |
| `"length"` | Response cut off at `max_tokens` | Increase `max_tokens` or trim conversation |
| `"content_filter"` | Azure content policy blocked response | Log input, do not retry without investigation |

### Concept 2 — Tool Schema Design: Descriptions and Parameter Contracts

A tool schema has two jobs the model uses at different times:

**`function.description`** — read at decision time. Before producing any output, the model reads all tool descriptions and decides which tools, if any, apply to the current situation. A description like `"Gets customer data"` is useless for this decision — it does not tell the model when to call the tool or what situation triggers it. A description like `"Use when routing a ticket requires knowing the customer's plan, account age, or open ticket count. Do NOT call this for every ticket — only when history or plan affects the routing decision"` gives the model the trigger conditions it needs.

**`function.parameters`** — read at argument-generation time. Once the model decides to call a tool, it uses the parameter schema to know what arguments to produce. Parameter descriptions, enums for constrained choices, and the `required` list directly control argument quality.

**Four rules that matter:**

**1. Write descriptions for the model, not for developers.**
Developers know what "route ticket" means. The model needs to know *when* routing is appropriate and what signals should trigger the call. Write for the model — include trigger conditions, include when-not-to-use conditions.

**2. Always use `enum` for constrained string choices.**
A `"type": "string"` field with no enum gives the model freedom to invent values: `"billing_urgent"`, `"technical_escalated"`, `"account-access"`. None of those exist in your dispatch handler. Every constrained string must be an enum.

**3. Write parameter descriptions as instructions, not labels.**
`"description": "the customer ID"` is a label. `"description": "The unique customer identifier in the format C-NNNN. Extract from the ticket metadata. Example: 'C-1042'."` is an instruction the model uses to produce the right value.

**4. Keep `required` honest.**
If the model can always determine the argument from context, require it. If the argument is genuinely optional and the function works without it, omit it from `required`. Marking optional arguments as required causes the model to hallucinate a value rather than omit the field.

**Before / after comparison:**

```python
# BAD — model will hallucinate team names, miss trigger conditions
{
    "name": "route_ticket",
    "description": "Routes a ticket.",          # no trigger conditions
    "parameters": {
        "type": "object",
        "properties": {
            "ticket_id": {"type": "string"},    # no description — format unknown
            "team": {"type": "string"},          # no enum — model invents values
            "reason": {"type": "string"},        # field name differs from handler
        },
        "required": ["ticket_id", "team", "reason", "confidence"],  # confidence not defined
    },
}

# GOOD — trigger conditions, enums, instructive descriptions
{
    "name": "route_ticket",
    "description": (
        "Assign a support ticket to a team and record the routing reason. "
        "Use this after you have gathered enough context to make a routing decision. "
        "Do NOT use this if you still need customer context — call get_customer_context first."
    ),
    "parameters": {
        "type": "object",
        "properties": {
            "ticket_id": {
                "type": "string",
                "description": "Ticket identifier from the conversation. Format: T-NNNN.",
            },
            "team": {
                "type": "string",
                "enum": ["billing", "technical", "account", "product"],
                "description": (
                    "billing: payment, invoice, subscription, discount issues. "
                    "technical: bugs, errors, API failures, outages. "
                    "account: login, access, permissions, user management. "
                    "product: feature requests, feedback, roadmap questions."
                ),
            },
            "confidence": {
                "type": "number",
                "description": "Routing confidence as a float 0.0–1.0. Use 1.0 for unambiguous cases.",
            },
            "routing_reason": {
                "type": "string",
                "description": "One sentence explaining why this team was chosen. Reference specific ticket content.",
            },
            "escalate": {
                "type": "boolean",
                "description": (
                    "True if the ticket shows urgency: third+ occurrence, data loss risk, "
                    "or enterprise customer. Defaults to false."
                ),
            },
        },
        "required": ["ticket_id", "team", "confidence", "routing_reason"],
        # escalate is optional — model includes it only when relevant
    },
}
```

**`tool_choice` values:**

| Value | What it does | When to use |
|---|---|---|
| `"auto"` (default) | Model decides whether to call a tool | Most cases |
| `"required"` | Model must call at least one tool | When a structured action is always required |
| `"none"` | Model cannot call any tools | When you want plain text despite tools being defined |
| `{"type": "function", "function": {"name": "X"}}` | Model must call tool X | When one specific tool call is always required |

### Concept 3 — Response Handling: Dispatch, Parallel Calls, and the Loop

**Four concerns the happy path misses:**

**Parallel tool calls.** The model can request multiple tools in one turn — `message.tool_calls` is a list. Processing only `tool_calls[0]` silently drops all others. The next API call then fails because unresolved tool call IDs exist in the conversation with no matching tool messages.

**Argument parsing.** `tc.function.arguments` is a JSON string. `json.loads()` can fail. Even when it succeeds, the arguments may not match the function signature. Both failures need structured error returns, not unhandled exceptions.

**Unknown tool names.** The model may call a tool name not in your registry — after a tool is renamed, removed, or in rare cases of hallucination. Your dispatch must handle unknown names without crashing.

**Handler exceptions.** The actual handler function can raise — database unavailable, permission denied, downstream timeout. Handler exceptions must produce structured error JSON, not crash the pipeline. The model receives the error as a tool result and can adapt its response gracefully.

**The loop pattern:**

```
while finish_reason == "tool_calls":
    execute all tool calls in this turn
    append assistant message + all tool result messages
    make next API call

# loop exits when finish_reason == "stop"
```

In simple scenarios, the loop runs twice (one tool call turn + one final answer turn). In multi-tool scenarios, it may run more. A hard iteration cap prevents the edge case where a tool always returns data that triggers another tool call.

# app/tools/handlers.py
"""
Tool handler registry and execution loop for ticket-ops-api.

All tool execution passes through execute_tool_call():
  - Parses the arguments JSON string
  - Dispatches to the registered handler
  - Returns structured error JSON on any failure — never raises

run_tool_loop() closes the two-step cycle and handles parallel tool calls.
"""

import json
import logging
import os
from typing import Any, Callable

from openai import AzureOpenAI

from app.tools.definitions import TOOLS

logger = logging.getLogger(__name__)

client = AzureOpenAI(
    azure_endpoint=os.environ["AZURE_OPENAI_ENDPOINT"],
    api_key=os.environ["AZURE_OPENAI_API_KEY"],
    api_version="2024-02-01",
)

# ---------------------------------------------------------------------------
# Handler registry — maps tool name → handler function
# ---------------------------------------------------------------------------

TOOL_REGISTRY: dict[str, Callable[..., Any]] = {
    "get_customer_context": get_customer_context,
    "route_ticket": route_ticket,
}


# ---------------------------------------------------------------------------
# Execution layer
# ---------------------------------------------------------------------------

def execute_tool_call(tc) -> str:
    """
    Parse and execute a single tool call from the model.

    Returns the result as a JSON string — tool message content must be a string.
    Never raises. Returns structured error JSON on any failure so the model can
    adapt its response rather than crashing the pipeline.
    """
    tool_name = tc.function.name

    # 1. Parse arguments — tc.function.arguments is always a JSON string
    try:
        args: dict = json.loads(tc.function.arguments)
    except json.JSONDecodeError as exc:
        logger.error(
            "tool=%s argument parse failed: %s | raw=%r",
            tool_name, exc, tc.function.arguments,
        )
        return json.dumps({
            "error": "argument_parse_failed",
            "detail": str(exc),
            "raw_arguments": tc.function.arguments,
        })

    # 2. Dispatch — look up the handler in the registry
    handler = TOOL_REGISTRY.get(tool_name)
    if handler is None:
        logger.error("tool=%s: no handler registered. Available: %s", tool_name, list(TOOL_REGISTRY))
        return json.dumps({
            "error": "unknown_tool",
            "tool_name": tool_name,
            "available_tools": sorted(TOOL_REGISTRY.keys()),
        })

    # 3. Execute — catch handler failures
    try:
        result = handler(**args)
        return json.dumps(result)
    except TypeError as exc:
        # Arguments don't match the function signature — schema/handler mismatch
        logger.error("tool=%s argument mismatch: %s | args=%r", tool_name, exc, args)
        return json.dumps({
            "error": "argument_mismatch",
            "detail": str(exc),
            "args_received": list(args.keys()),
        })
    except Exception as exc:
        logger.error(
            "tool=%s handler raised %s: %s",
            tool_name, type(exc).__name__, exc,
            exc_info=True,
        )
        return json.dumps({
            "error": "handler_failed",
            "error_type": type(exc).__name__,
            "detail": str(exc),
        })


def run_tool_loop(
    messages: list[dict],
    max_iterations: int = 5,
    temperature: float = 0.0,
) -> str:
    """
    Execute the full tool-calling loop until finish_reason="stop".

    Handles:
      - Parallel tool calls (processes all entries in message.tool_calls)
      - Structured error returns from execute_tool_call
      - Hard iteration cap to prevent infinite loops

    Returns the model's final text response.
    Raises RuntimeError if max_iterations is exceeded or an unexpected
    finish_reason is encountered.
    """
    for iteration in range(max_iterations):
        response = client.chat.completions.create(
            model=DEPLOYMENT,
            messages=messages,
            tools=TOOLS,
            tool_choice="auto",
            temperature=temperature,
        )
        choice = response.choices[0]

        if choice.finish_reason == "stop":
            return choice.message.content

        if choice.finish_reason == "tool_calls":
            tool_calls = choice.message.tool_calls  # list — all must be handled

            # Append assistant message first (API requirement)
            messages.append(choice.message)

            # Execute every tool call in this turn, append each result
            for tc in tool_calls:
                result_str = execute_tool_call(tc)
                logger.info(
                    "iteration=%d tool=%s id=%s",
                    iteration + 1, tc.function.name, tc.id,
                )
                messages.append({
                    "role": "tool",
                    "tool_call_id": tc.id,    # must match tc.id exactly
                    "content": result_str,    # must be a string
                })
            # Loop continues — next API call at top of iteration
            continue

        # Unexpected finish_reason
        logger.warning(
            "Unexpected finish_reason=%r at iteration=%d",
            choice.finish_reason, iteration + 1,
        )
        raise RuntimeError(
            f"Tool loop ended with unexpected finish_reason={choice.finish_reason!r}"
        )

    raise RuntimeError(
        f"Tool loop exceeded max_iterations={max_iterations}. "
        "A tool may be returning data that always triggers another tool call."
    )
```

# app/routes/route.py  (or app/main.py — wherever the FastAPI app is defined)
"""
/route endpoint — tool-augmented ticket routing.

The model may call get_customer_context and then route_ticket.
The endpoint returns the model's final response after the tool loop closes.
"""

from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

from app.tools.handlers import run_tool_loop

app = FastAPI()

ROUTE_SYSTEM_PROMPT = (
    "You are a ticket routing assistant for a B2B SaaS support platform. "
    "If the routing decision depends on the customer's plan or account history, "
    "use get_customer_context first. "
    "Always call route_ticket to record the decision — do not just state the team verbally. "
    "Be concise in your final response: confirm the routing and escalation status only."
)


class RouteRequest(BaseModel):
    ticket_text: str


class RouteResponse(BaseModel):
    response: str


@app.post("/route", response_model=RouteResponse)
def route_ticket_endpoint(req: RouteRequest) -> RouteResponse:
    messages = [
        {"role": "system", "content": ROUTE_SYSTEM_PROMPT},
        {"role": "user",   "content": req.ticket_text},
    ]
    try:
        final_response = run_tool_loop(messages, max_iterations=5)
    except RuntimeError as exc:
        raise HTTPException(status_code=500, detail=str(exc))
    return RouteResponse(response=final_response)
```

## Key Takeaways

- **The model produces intent. Your code executes it.** The model never touches your database or APIs directly. It outputs a structured description of what to call. You execute, validate, and return the result. This is why tool-calling systems are auditable — you see exactly what the model asked for and exactly what you returned.

- **The conversation ordering rule is an API constraint.** The assistant message with `tool_calls` must be appended before any tool result messages. Every tool call ID in that assistant message must have a matching tool result message before the next API call. These are not conventions — the API rejects calls that violate them.

- **Always loop over all `tool_calls` entries.** The model can request multiple tools in one turn. Processing only `tool_calls[0]` silently drops the rest, leaves unresolved tool IDs in the conversation, and causes the next API call to fail.

- **The description field controls tool selection — it is prompt engineering.** A description like "Gets customer data" gives the model no trigger conditions. The model over-triggers or under-triggers. Write descriptions for the model: include when to use, when not to use, and what signals should trigger the call.

- **Always use `enum` for constrained string choices.** A `"type": "string"` field with no enum lets the model invent values: `"billing_urgent"`, `"technical-bug"`. None of those exist in your handler. Every constrained string must be an enum.

- **`execute_tool_call()` never raises.** Handler failures — database timeouts, permission errors, bad arguments — return structured error JSON. The model receives the error as a tool result and can produce a graceful response. An unhandled exception crashes the pipeline and returns nothing useful to anyone.

- **Put a hard cap on `run_tool_loop()`.** A tool that always returns data triggering another tool call loops forever. `max_iterations=5` is sufficient for the vast majority of single-session tool use. Raise `RuntimeError` when the cap is hit — that is the right signal that something unexpected is happening.

## Optional Advanced Notes

**Parallel tool execution for latency:** When the model requests multiple tools in one turn, the skeleton above executes them sequentially. In production, if the handlers are I/O-bound (database queries, external API calls), executing them concurrently with `asyncio.gather()` can reduce latency proportionally. Switch to the async OpenAI client (`AsyncAzureOpenAI`) and `async def execute_tool_call_async(tc)` with `await asyncio.gather(*[execute_tool_call_async(tc) for tc in tool_calls])`. The sequencing rule (append assistant message before tool messages) still applies — collect all results first, then append them.

**Tool use and injection surface:** The tool parameter that accepts free text from user input — `routing_reason` uses model-generated text, but a future tool accepting user-submitted text directly — creates injection surface. A user who controls the input to a tool call can attempt to inject arguments that your handler was not designed for. Lesson 2.11 covers sanitizing tool call arguments and validating them against an allowlist before the handler runs. For the tools defined here, both inputs are model-generated (not raw user text), so the surface is limited — but the pattern in `execute_tool_call()` already has the right hook: argument parsing and validation happen before dispatch.
